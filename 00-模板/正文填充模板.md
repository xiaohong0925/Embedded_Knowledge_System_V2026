# 正文填充模板

> 用于逐小点生成《嵌入式 Linux 知识体系（V2025）》教材正文。
> 风格：严谨教科书 + 平易近人。定义精确、推导严密、类比为盐、场景具体。

---

## 核心原则（3 条，不可妥协）

1. **推导优先于结论**
   每个机制必须从工程痛点或设计矛盾出发，回答"为什么需要它"。
   每个设计决策必须回答：如果不这样设计，会出现什么问题？

2. **代码必须带读**
   所有代码片段必须注明文件路径 + 函数名 + 行号范围（如 `kernel/irq/manage.c:__setup_irq() 第 1200-1250 行`）。
   关键行必须有注释（右侧或下方），禁止只贴代码不解释。

3. **深度优先，量化服从深度**
   字数、代码行数、图数均不设上限。如果讲清楚需要 8000 字、120 行代码、6 张图，那就写这么多。
   禁止为了凑字数而车轱辘话，也禁止因为"怕太长"而砍掉必要推导。

---

## 八条红线（违反即作废）

1. **严禁跳步**
   禁止从类比直接跳到源码不给定义，禁止从概念直接跳到 API 调用不给背景。
   概念 → 代码之间必须有推导桥梁。

2. **严禁抽象定义堆砌**
   禁止连续 3 段"xxx 是 yyy"的纯定义内容。每个定义必须配类比、场景或反例。

3. **严禁工业化流水线**
   禁止连续 2 个小节使用完全相同的叙述节奏（如不能每个小点都"类比开头→表格→代码→验证"）。
   允许直接从问题出发、从反例出发、从对比出发。

4. **严禁认知阶段标题**
   正文禁止出现 ①②③④⑤ 阶段编号，禁止用"### 类比引入""### 正式定义"等标题切分认知阶段。
   五要素（场景→定义→图示→代码→验证）须融合为流畅叙事。

5. **严禁无边界混写**
   类型 A（内核机制章）禁止写驱动 API 实现细节（如 `request_irq` 的驱动用法留给第 5 章）。
   涉及驱动 API 时标注"驱动编程细节详见第 X 章"，涉及硬件架构时标注"具体 SoC 实现详见第 X 章"。

6. **严禁口语化表达**
   禁止："说白了"、"说实话"、"我当年"、"你可能会问"、"其实"等博客口语。
   必须：使用"因此"、"由于"、"由此可知"、"换言之"等逻辑连接词。

7. **严禁类比替代定义**
   类比最多出现 1 次/小点，且必须在正式定义之后，作为"辅助理解"而非"核心定义"。
   禁止用"就像..."来解释一个尚未正式定义的概念。

8. **严禁未定义术语前置**
   任何术语在首次解释前，禁止出现在定义句中。
   错误："CPU 通过 irq_desc 找到 irqaction 链表"（irq_desc 和 irqaction 未定义）。
   正确："CPU 通过中断描述符（`irq_desc`）找到挂接的动作描述符链表（`irqaction`）"。

## 语言风格规范（教科书 + 平易近人）

### 1. 开头：一句话引出，立刻切定义

优先用一句话对比或场景引出主题，然后立刻给出正式定义。

**示例**：
> 在嵌入式系统中，CPU 与外设交换数据只有两种基本方式：要么 CPU 主动问，要么外设主动喊。最原始的做法是**轮询**（Polling）……

### 2. 类比：盐，不是主食

- 每小点最多 1 处类比
- 必须在正式定义之后
- 最多 2 句话
- 可用 `>` 引用块包裹，或放在括号内

**示例**：
> 💡 **直观理解**：中断把"CPU 主动查"变成了"外设主动叫"……就像餐厅里给每桌装一个服务铃——客人需要时按铃，服务员只在听到铃声时才行动。

### 3. 推导：问题 → 答案，用工程化比喻

- 每段开头优先抛出一个**具体的工程问题**，然后回答
- 用"天花板"、"陷阱"、"矛盾"、"困境"等工程化比喻替代纯学术语言
- 场景必须**具体到数字和设备**：不说"功耗高"，说"手持设备待机从数周暴跌到数小时"

**示例**：
> **第一道天花板：CPU 空转与功耗失控**
> 假设一个工业传感器每 1ms 产生一次有效数据……

### 4. 过渡：逻辑连接，不是闲聊

- 小点之间用逻辑连接词过渡："由于上述缺陷，中断机制应运而生"
- 禁止："接下来我们来看"、"下面我们介绍"等无意义过渡句
- 最后一句必须是**钩子**，预告下一个问题："但外设发出的这条'异步请求'，在硬件层面到底意味着什么？"

### 5. 结尾：钩子收束

- 每个小点最后 1 句必须是钩子，引导读者继续读（最后一个小点除外）
- 钩子格式：提出一个未回答的问题，或指出下一个自然延伸的方向

## 排版规范

### 段落与换行

单段完成一个完整意思，不硬性规定行数。但以下场景**必须换段**（空行分隔）：

1. **从"引出/背景"切换到"正式定义"时**
   > 最原始的做法是轮询（Polling）——CPU 以固定周期反复读取外设的状态寄存器，直到发现数据就绪。
   >
   > 轮询的第一道工程天花板是 CPU 空转与功耗失控。

2. **从"问题/痛点"切换到"具体场景/数据"时**
   > 轮询的第一道工程天花板是 CPU 空转与功耗失控。
   >
   > 假设一个工业传感器每 1ms 产生一次有效数据……

3. **从"场景推导"切换到"结论/定义"时**
   > 如果不引入中断，手持设备待机时间将从数周暴跌到数小时。
   >
   > 中断（Interrupt，异步通知机制）正是为了打破这个僵局而设计的……

4. **从"定义"切换到"类比/直观理解"时**
   > CPU 收到后，保存当前执行现场……
   >
   > 💡 **直观理解**：中断把"CPU 主动查"变成了"外设主动叫"……

5. **从"文字论述"切换到"表格/代码/图示"时**
   > 两种范式的工程差异可通过下表直观对比：
   >
   > | 维度 | 轮询 | 中断 |

6. **从"原理"切换到"验证/命令"时**
   > 由此可知，中断机制具有不可替代的工程价值。
   >
   > **验证：你的系统里谁在发中断？**

**禁止**：
- 禁止用 `<br>` 标签或 HTML 换行
- 禁止把一句话硬拆成两段凑"短段落"
- 禁止连续 3 段超过 6 行（出现文字墙）

### 术语
- 术语**首次出现时**加括号解释（如 ISR（Interrupt Service Routine，中断服务程序））
- 禁止在一句内连续倾泻多个未解释术语
- 不要直呼"新手""小白"
