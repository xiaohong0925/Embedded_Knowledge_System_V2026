## 驱动开发环境搭建与基础流程

**交叉编译环境配置：针对ARM/RISC-V的驱动编译脚本编写
**驱动运行在嵌入式硬件（如ARM架构开发板、RISC-V开发板），但开发通常在x86架构的PC上进行，因此需要交叉编译环境（在一种架构的机器上编译出另一种架构可执行的代码）。
这部分核心是“获取工具链→配置环境→验证有效性→编写通用编译脚本”。

1.  工具链获取与安装（以ARM为例，RISC-V流程类似）

    嵌入式ARM架构最常用的交叉编译工具链是GNU Arm Embedded Toolchain，需根据开发板的ARM架构（32位/64位）选择版本：
    - 32位ARM（如ARM9、Cortex-A7）：选择`arm-linux-gnueabihf`（硬浮点支持）
    - 64位ARM（如Cortex-A53/A72）：选择`aarch64-linux-gnu`

    安装步骤（Ubuntu 20.04系统）：
    ```bash
    # 1. 下载工具链（以32位为例，官网地址可替换为国内镜像）
    wget https://developer.arm.com/-/media/Files/downloads/developer-tools/gnu-arm-embedded/10-2020-q4-major/gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2

    # 2. 解压到指定目录（通常放在/opt下，需root权限）
    sudo tar -xjvf gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2 -C /opt/

    # 3. 重命名简化路径（可选，方便后续配置）
    sudo mv /opt/gcc-arm-none-eabi-10-2020-q4-major /opt/arm-linux-gnueabihf
    ```

    RISC-V架构工具链获取：
    选择`riscv64-linux-gnu`工具链，下载地址为https://github.com/riscv-collab/riscv-gnu-toolchain/releases，解压和安装步骤与ARM一致，路径建议设为`/opt/riscv64-linux-gnu`。

2.  环境变量配置（让系统识别工具链命令）
    环境变量`PATH`用于指定系统查找可执行文件的路径，需将工具链的`bin`目录添加到`PATH`中：
    ```bash
    # 1. 临时配置（仅当前终端生效，用于测试）
    export PATH=/opt/arm-linux-gnueabihf/bin:$PATH

    # 2. 永久配置（所有终端生效，推荐）
    # 编辑用户级配置文件（仅当前用户生效）
    echo "export PATH=/opt/arm-linux-gnueabihf/bin:$PATH" >> ~/.bashrc
    # 立即生效配置
    source ~/.bashrc
    ```

3.  验证环境有效性
    执行工具链的编译器命令，查看版本信息，若输出正常则说明环境配置成功：
    ```bash
    # ARM工具链验证
    arm-linux-gnueabihf-gcc -v
    # 预期输出：包含"Target: arm-linux-gnueabihf"和版本信息（如gcc version 10.2.1 20201103）

    # RISC-V工具链验证
    riscv64-linux-gnu-gcc -v
    # 预期输出：包含"Target: riscv64-linux-gnu"和版本信息
    ```

4.  通用编译脚本（Makefile）编写

    驱动编译依赖内核源码，且需指定交叉编译工具链，编写通用Makefile可避免每次编译都输入复杂命令。
以ARM架构为例，Makefile模板如下（关键变量已标注说明）：
    ```makefile
    # 1. 目标驱动模块名（需修改为自己的驱动名，如hello_drv）
    obj-m += hello_drv.o

    # 2. 内核源码路径（需修改为自己的开发板内核源码路径，绝对路径）
    KERNELDIR := /home/user/linux-5.15.71-sunxi

    # 3. 当前目录（自动获取，无需修改）
    PWD := $(shell pwd)

    # 4. 编译命令（指定架构和交叉编译工具链，无需修改）
    all:
        make -C $(KERNELDIR) M=$(PWD) ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- modules

    # 5. 清理命令（删除编译生成的临时文件，无需修改）
    clean:
        make -C $(KERNELDIR) M=$(PWD) ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- clean
    ```
    变量说明：
    - `obj-m`：指定要编译的驱动模块（.o文件），内核会将其编译为.ko模块文件；
    - `KERNELDIR`：开发板对应的内核源码路径（必须是已配置过的内核，后续步骤会讲）；
    - `ARCH`：目标硬件架构（arm/riscv64）；
    - `CROSS_COMPILE`：交叉编译工具链前缀（与前面安装的工具链对应）。

**内核源码准备：配置`CONFIG_MODULES`与驱动相关配置项
**驱动模块（.ko文件）是内核的“扩展组件”，编译驱动必须依赖与开发板内核版本一致、配置一致的内核源码。核心步骤是“获取源码→配置内核→编译模块依赖”。

1.  内核源码获取
    内核源码有两个主要来源，优先选择厂商定制版（兼容性更好）：
    - 厂商定制版（推荐）：开发板厂商官网下载（如全志、瑞芯微），例如全志H3开发板的内核源码通常包含`soc`相关驱动，兼容性更高；
    - 官方原版：Linux内核官网（https://www.kernel.org/）下载对应版本（如5.15.71），需自行适配开发板硬件（难度较高，适合有经验者）。

    源码解压后放在固定路径（如`/home/user/linux-5.15.71-sunxi`），后续编译驱动时会用到这个路径。

2.  配置内核：启用模块支持（`CONFIG_MODULES`）
    驱动模块依赖内核的“模块加载机制”，需通过内核配置工具启用`CONFIG_MODULES`选项（默认可能已启用，需确认）：
    ```bash
    # 1. 进入内核源码目录
    cd /home/user/linux-5.15.71-sunxi

    # 2. 加载开发板默认配置（关键！厂商源码通常提供默认配置文件）
    # 以全志H3为例，默认配置文件在arch/arm/configs/sunxi_defconfig
    make sunxi_defconfig
    # 若为RISC-V架构，配置文件可能在arch/riscv/configs/xxx_defconfig

    # 3. 进入图形化配置界面（修改配置项）
    make menuconfig
    ```
    图形化界面操作步骤：
    1.  在界面中按“/”键，输入`CONFIG_MODULES`搜索选项；
    2.  找到“Enable loadable module support (MODULES)”选项，确保其被选中（前面有“*”或“M”，“*”表示编译进内核，“M”表示编译为模块，此处只要启用即可）；
    3.  若未选中，按“Y”键选中，然后按“ESC”键退出搜索，回到主界面；
    4.  按“Save”保存配置，默认保存到`.config`文件（内核编译时会读取此文件），然后按“Exit”退出。

    关键说明：`CONFIG_MODULES`是驱动开发的基础选项，未启用则无法编译和加载.ko模块。

3.  编译内核与模块依赖
    配置完成后，需编译内核和模块，生成驱动编译依赖的文件（如`modules.order`、`Module.symvers`）：
    ```bash
    # 1. 编译内核（可选，若仅编译驱动模块，可跳过此步，但建议编译一次确保配置正确）
    make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- -j4
    # -j4：使用4个线程编译，加速编译过程，根据CPU核心数调整（如-j8）

    # 2. 编译内核模块（必须执行！生成驱动编译依赖）
    make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- modules -j4
    ```
    编译完成后，内核源码目录会生成模块相关的依赖文件，后续编译驱动时会自动引用这些文件。

**基础流程：驱动源码→Makefile编写→编译→加载→调试
**这是驱动开发的“标准流程”，以一个最简单的“Hello驱动”为例，完整演示从代码编写到调试的全流程。

1.  步骤1：编写驱动源码（hello_drv.c）
    这是一个最简化的驱动模块，仅包含“模块加载”和“模块卸载”两个核心函数，用于验证流程：
    ```c
    #include <linux/init.h>   // 包含模块初始化相关函数（如module_init）
    #include <linux/module.h> // 包含模块相关宏（如MODULE_LICENSE）

    // 模块加载函数：当执行insmod加载模块时，内核自动调用此函数
    static int __init hello_drv_init(void)
    {
        // printk：内核态打印函数（类似用户态printf，输出到内核日志）
        printk("Hello driver loaded!\n");
        return 0; // 返回0表示加载成功
    }

    // 模块卸载函数：当执行rmmod卸载模块时，内核自动调用此函数
    static void __exit hello_drv_exit(void)
    {
        printk("Hello driver unloaded!\n");
    }

    // 注册模块加载/卸载函数（内核识别的宏）
    module_init(hello_drv_init);
    module_exit(hello_drv_exit);

    // 模块许可声明：必须添加，否则加载时会报"tainted kernel"警告
    MODULE_LICENSE("GPL");
    // 模块作者信息（可选）
    MODULE_AUTHOR("Embedded Linux Learner");
    // 模块描述信息（可选）
    MODULE_DESCRIPTION("A Simple Hello Driver");
    ```
    关键术语解释：
    - 模块（Module）：内核的可动态加载/卸载组件，无需重新编译内核即可扩展功能；
    - `module_init`：宏定义，指定模块加载时执行的函数；
    - `printk`：内核态打印函数，输出内容默认存放在内核日志缓冲区，需用`dmesg`命令查看。

2.  步骤2：编写Makefile（复用前面的通用模板）
    将前面编写的通用Makefile复制到驱动源码目录，修改`obj-m`为`hello_drv.o`，确保`KERNELDIR`路径正确：
    ```makefile
    obj-m += hello_drv.o
    KERNELDIR := /home/user/linux-5.15.71-sunxi
    PWD := $(shell pwd)

    all:
        make -C $(KERNELDIR) M=$(PWD) ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- modules

    clean:
        make -C $(KERNELDIR) M=$(PWD) ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- clean
    ```

3.  步骤3：编译驱动模块
    在驱动源码目录执行`make`命令，编译生成.ko模块文件：
    ```bash
    # 执行编译
    make
    # 编译成功后，查看生成的文件
    ls -l
    ```
    预期输出：会生成`hello_drv.ko`（目标驱动模块）、`hello_drv.o`（编译中间文件）等，其中`hello_drv.ko`是需要拷贝到开发板的文件。

    常见错误排查：
    - 若提示“内核路径不存在”：检查`KERNELDIR`是否为内核源码的绝对路径；
    - 若提示“交叉编译工具链未找到”：检查`CROSS_COMPILE`前缀是否与安装的工具链一致，且环境变量已配置。

4.  步骤4：加载驱动模块到开发板
    将`hello_drv.ko`通过SSH或U盘拷贝到开发板，执行加载命令：
    ```bash
    # 加载驱动模块（需root权限，开发板通常默认root用户）
    insmod hello_drv.ko

    # 查看已加载的模块，确认加载成功
    lsmod | grep hello_drv
    # 预期输出：hello_drv  16384  0  # 模块名、大小、引用计数
    ```

5.  步骤5：调试驱动（查看内核日志）
    用`dmesg`命令查看`printk`的输出，验证模块加载函数是否执行：
    ```bash
    # 查看内核日志（默认显示所有日志，可加过滤条件）
    dmesg | grep "Hello"
    # 预期输出：[  123.456789] Hello driver loaded!  # 时间戳可能不同
    ```

6.  步骤6：卸载驱动模块
    执行`rmmod`命令卸载模块，再次查看日志确认卸载函数执行：
    ```bash
    # 卸载驱动模块
    rmmod hello_drv

    # 查看日志，验证卸载函数输出
    dmesg | grep "Hello"
    # 预期输出：
    # [  123.456789] Hello driver loaded!
    # [  456.789012] Hello driver unloaded!
    ```

**实操：编译并加载第一个自定义字符设备驱动模块
**前面的“Hello驱动”仅验证流程，实际驱动需实现硬件控制或数据交互。
这里以“字符设备驱动”（最常用的驱动类型，如LED、按键）为例，实现一个“支持用户态读写”的简单驱动，用户态可通过设备文件控制驱动输出信息。

1.  字符设备驱动核心概念
    - 字符设备（Character Device）：
按字节流顺序读写的设备，驱动通过`file_operations`结构体向内核提供“打开、关闭、读写”等接口，
用户态通过设备文件（如`/dev/char_drv`）访问。
    - 设备号（Device Number）：
每个字符设备对应唯一的设备号（主设备号+次设备号），用于内核识别设备。

2.  编写字符设备驱动源码（char_drv.c）
    ```c
    #include <linux/init.h>
    #include <linux/module.h>
    #include <linux/fs.h>       // 包含文件操作相关结构体（file_operations）
    #include <linux/cdev.h>     // 包含字符设备相关结构体（cdev）
    #include <linux/uaccess.h>  // 包含用户态与内核态数据拷贝函数（copy_to_user等）

    #define DEV_NAME "char_drv" // 设备名
    #define DEV_MAJOR 240       // 主设备号（建议选240以上，避免与系统设备冲突）
    #define DEV_MINOR 0         // 次设备号
    #define BUF_SIZE 128        // 数据缓冲区大小

    static char dev_buf[BUF_SIZE] = "Hello from char driver!\n"; // 内核态缓冲区
    static struct cdev char_cdev;                               // 字符设备结构体

    // 1. 设备打开函数：用户态执行open("/dev/char_drv")时调用
    static int char_drv_open(struct inode *inode, struct file *file)
    {
        printk("Char driver opened!\n");
        return 0;
    }

    // 2. 设备读取函数：用户态执行read()时调用，内核态数据拷贝到用户态
    static ssize_t char_drv_read(struct file *file, char __user *buf, size_t count, loff_t *loff)
    {
        int ret;
        // copy_to_user：内核态数据拷贝到用户态（参数：用户态缓冲区、内核态缓冲区、长度）
        ret = copy_to_user(buf, dev_buf, count);
        if (ret > 0) {
            printk("Read data failed, %d bytes not copied\n", ret);
            return -EFAULT;
        }
        return count; // 返回读取的字节数
    }

    // 3. 设备写入函数：用户态执行write()时调用，用户态数据拷贝到内核态
    static ssize_t char_drv_write(struct file *file, const char __user *buf, size_t count, loff_t *loff)
    {
        int ret;
        // 清空内核缓冲区
        memset(dev_buf, 0, BUF_SIZE);
        // copy_from_user：用户态数据拷贝到内核态
        ret = copy_from_user(dev_buf, buf, count);
        if (ret > 0) {
            printk("Write data failed, %d bytes not copied\n", ret);
            return -EFAULT;
        }
        printk("Char driver wrote: %s\n", dev_buf);
        return count; // 返回写入的字节数
    }

    // 4. 设备关闭函数：用户态执行close()时调用
    static int char_drv_release(struct inode *inode, struct file *file)
    {
        printk("Char driver closed!\n");
        return 0;
    }

    // 字符设备操作集：关联驱动函数与内核接口
    static const struct file_operations char_fops = {
        .owner = THIS_MODULE,    // 模块所有者，防止模块在使用时被卸载
        .open = char_drv_open,   // 关联打开函数
        .read = char_drv_read,   // 关联读取函数
        .write = char_drv_write, // 关联写入函数
        .release = char_drv_release, // 关联关闭函数
    };

    // 模块加载函数：注册字符设备
    static int __init char_drv_init(void)
    {
        int ret;
        dev_t dev_num; // 设备号（主设备号+次设备号）

        // 1. 静态申请设备号（主设备号DEV_MAJOR，次设备号从DEV_MINOR开始，1个设备）
        dev_num = MKDEV(DEV_MAJOR, DEV_MINOR);
        ret = register_chrdev_region(dev_num, 1, DEV_NAME);
        if (ret < 0) {
            printk("Register device number failed!\n");
            return ret;
        }

        // 2. 初始化字符设备结构体
        cdev_init(&char_cdev, &char_fops);
        char_cdev.owner = THIS_MODULE;

        // 3. 向内核注册字符设备
        ret = cdev_add(&char_cdev, dev_num, 1);
        if (ret < 0) {
            printk("Add char device failed!\n");
            unregister_chrdev_region(dev_num, 1); // 注册失败，释放设备号
            return ret;
        }

        printk("Char driver initialized successfully!\n");
        return 0;
    }

    // 模块卸载函数：注销字符设备
    static void __exit char_drv_exit(void)
    {
        dev_t dev_num = MKDEV(DEV_MAJOR, DEV_MINOR);
        cdev_del(&char_cdev); // 注销字符设备
        unregister_chrdev_region(dev_num, 1); // 释放设备号
        printk("Char driver exited successfully!\n");
    }

    module_init(char_drv_init);
    module_exit(char_drv_exit);
    MODULE_LICENSE("GPL");
    MODULE_DESCRIPTION("A Simple Character Device Driver");
    ```

3.  编译与加载驱动
    复用前面的Makefile（修改`obj-m += char_drv.o`），编译后拷贝到开发板，执行加载：
    ```bash
    # 开发板上加载驱动
    insmod char_drv.ko

    # 查看设备号是否注册成功
    cat /proc/devices | grep char_drv
    # 预期输出：240 char_drv  # 主设备号240，设备名char_drv

    # 创建设备文件（用户态访问驱动的入口）
    mknod /dev/char_drv c 240 0
    # 说明：mknod创建设备文件，c表示字符设备，240是主设备号，0是次设备号
    ```

4.  用户态测试驱动
    通过`cat`和`echo`命令测试驱动的读写功能：
    ```bash
    # 1. 读取驱动数据（调用read函数）
    cat /dev/char_drv
    # 预期输出：Hello from char driver!  # 读取内核态缓冲区数据

    # 2. 写入数据到驱动（调用write函数）
    echo "Test write to char driver" > /dev/char_drv

    # 3. 再次读取，验证写入是否生效
    cat /dev/char_drv
    # 预期输出：Test write to char driver

    # 4. 查看内核日志，验证驱动函数执行
    dmesg | grep "Char driver"
    # 预期输出：
    # [  789.123456] Char driver initialized successfully!
    # [  800.456789] Char driver opened!
    # [  800.456890] Char driver closed!
    # [  810.789012] Char driver opened!
    # [  810.789123] Char driver wrote: Test write to char driver
    # [  810.789234] Char driver closed!
    ```

5.  卸载驱动
    ```bash
    rmmod char_drv
    # 查看日志确认卸载
    dmesg | grep "Char driver exited"
    # 预期输出：[  820.345678] Char driver exited successfully!
    ```

## 设备树与驱动模型的匹配实战

### 设备树节点编写规范：compatible、reg、interrupts等关键属性

设备树（Device Tree，DT）是Linux内核中“描述硬件信息的树形结构文件”，核心作用是替代传统板级代码（board.c），将硬件配置与驱动代码解耦

——驱动通过解析设备树获取硬件参数（如寄存器地址、中断号），无需硬编码。
编写设备树的核心是按规范定义“节点”和“属性”，其中`compatible`（匹配标识）、`reg`（地址信息）、`interrupts`（中断信息）是驱动匹配的关键属性。

1.  设备树基本结构与节点命名规范
    设备树以`.dts`为源文件，经`dtc`（设备树编译器）编译为`.dtb`（二进制文件）供内核解析，基本结构是“根节点→子节点→属性”，节点命名需遵循固定规则：
    ```dts
    /* 根节点（所有设备节点的父节点，固定命名为/） */
    / {
        model = "Vendor Board Name";  // 开发板型号（可选，用于内核识别板型）
        compatible = "vendor,board";  // 板级兼容标识（可选）
        
        /* 子节点：LED设备节点（命名规则：设备类型@基地址） */
        led@300B0000 {  // @后为reg属性的基地址，用于区分同类型设备
            compatible = "vendor,led-dt";  // 驱动匹配核心属性（必须）
            reg = <0x300B0000 0x1000>;     // GPIO控制器基地址+地址长度（必须）
            interrupts = <25 0x04>;        // 中断号+触发方式（可选，若用中断）
            led-gpio = <&pio 1 5 GPIO_ACTIVE_HIGH>;  // 自定义属性（可选）
        };
    };
    ```
    关键规则：
    - 节点命名：`设备类型@基地址`，基地址需与`reg`属性的第一个值一致（如`led@300B0000`对应`reg = <0x300B0000 ...>`），避免同类型设备重名；
    - 属性格式：`属性名 = <值1 值2 ...>;`，值的类型由硬件需求决定（如地址用十六进制，GPIO用引脚编号）。

2.  三大核心属性解析（驱动匹配的关键）
    驱动与设备树的匹配及硬件参数获取，完全依赖以下三大核心属性，必须严格按规范编写：

属性：`compatible`
核心作用：驱动匹配"身份证"，驱动通过此属性识别设备
格式规范与示例：字符串列表，格式：`"厂商名,设备名"`，如  
```dts
compatible = "vendor,led-dt";
```
关键说明：驱动的`of_match_table`需包含相同字符串才能匹配；可写多个值（如`"vendor,led-dt", "linux,led"`），实现兼容旧驱动

属性：`reg`
核心作用：描述硬件寄存器的物理地址范围
格式规范与示例：数组，格式：`<基地址 地址长度>`，如  
```dts
reg = <0x300B0000 0x1000>;
```
关键说明：基地址是硬件手册中标注的寄存器起始地址，长度是寄存器组的总字节数；多个地址范围用逗号分隔（如`reg = <0x1000 0x4, 0x2000 0x8>`）

属性：`interrupts`
核心作用：描述设备的中断信息
格式规范与示例：数组，格式：`<中断号 触发方式>`，如  
```dts
interrupts = <25 IRQF_TRIGGER_FALLING>;
```
关键说明：中断号是硬件分配的中断编号（需查硬件手册）；触发方式用内核宏（如`IRQF_TRIGGER_FALLING`表示下降沿触发，也可写十六进制`0x04`）

    补充：自定义属性（如`led-gpio`）用于传递非标准参数（如GPIO引脚编号），格式需与驱动约定一致（如`&pio 1 5`表示引用`pio`节点的第1组第5号引脚）。

3.  实战：LED设备树节点完整编写（基于全志H3开发板）
    结合全志H3硬件手册（GPIO控制器基地址`0x300B0000`，LED接GPIO1_5，中断号25），完整节点如下：
    ```dts
    /dts-v1/;  // 设备树版本（固定）
    #include "sun8i-h3.dtsi"  // 引入芯片级设备树（包含pio等节点定义）

    / {
        aliases {
            led0 = &led;  // 别名，方便引用（可选）
        };

        /* LED设备节点（挂在根节点下） */
        led: led@300B0000 {
            compatible = "vendor,led-dt";  // 驱动匹配标识
            status = "okay";  // 启用此节点（若为"disabled"则内核不解析）
            reg = <0x300B0000 0x1000>;     // GPIO控制器地址范围
            interrupts = <GIC_SPI 25 IRQF_TRIGGER_FALLING>;  // 中断信息（GIC_SPI表示共享中断）
            led-gpio = <&pio 1 5 GPIO_ACTIVE_HIGH>;  // GPIO1_5，高电平点亮
        };
    };
    ```
    说明：`#include "sun8i-h3.dtsi"`是芯片厂商提供的基础设备树，已定义`pio`（GPIO控制器）、`gic`（中断控制器）等节点，自定义节点可直接引用。

**驱动中设备树属性解析API：`of_property_read_u32`/`of**_find_node_by_name`

### 调试技巧：用`dtc`编译设备树与`debugfs`查看设备树节点信息

设备树与驱动匹配失败时，需先验证“设备树是否正确编译”“内核是否正确解析节点”，
核心工具是`dtc`（编译验证）和`debugfs`（运行时查看），配合`dmesg`（内核日志）定位问题。

1.  `dtc`工具：编译与反编译设备树（验证语法错误）
    `dtc`是设备树编译器，用于将`.dts`源文件编译为内核可识别的`.dtb`二进制文件，同时可反编译`.dtb`验证编译结果，是设备树编写的基础工具。

    常用命令（Ubuntu环境，需先安装`dtc`：`sudo apt install device-tree-compiler`）：
    ```bash
    # 1. 编译.dts为.dtb（检查语法错误，生成二进制文件）
    dtc -I dts -O dtb -o led.dtb led.dts
    # 选项说明：-I（输入格式dts）、-O（输出格式dtb）、-o（输出文件名）
    # 若有语法错误（如括号不匹配、属性名错误），会直接报错，如"Syntax error at line 10"

    # 2. 反编译.dtb为.dts（验证编译后的内容是否与源码一致）
    dtc -I dtb -O dts -o led_decompile.dts led.dtb
    # 对比led.dts和led_decompile.dts，确认属性值未被篡改

    # 3. 查看.dtb的详细信息（包含节点结构和属性值）
    dtc -I dtb -O dts -o - led.dtb  # -o - 表示输出到终端
    ```
    关键场景：编译时若报错“Undefined reference to 'pio'”，说明未引入包含`pio`节点的芯片级设备树（需添加`#include "sun8i-h3.dtsi"`）。

2.  `debugfs`：运行时查看设备树节点与属性（内核已解析的信息）
    内核启动后会解析`.dtb`文件，将节点和属性信息存入`debugfs`文件系统，通过挂载`debugfs`可直观查看“内核实际解析到的硬件信息”，是验证设备树是否生效的核心手段。

    操作步骤（开发板上执行）：
    ```bash
    # 1. 挂载debugfs（临时挂载，重启后失效）
    mount -t debugfs none /sys/kernel/debug

    # 2. 进入设备树节点目录（所有解析后的节点都在该路径下）
    cd /sys/kernel/debug/devicetree/base

    # 3. 查看节点结构（ls查看子节点，对应设备树中的节点）
    ls
    # 预期输出：led@300B0000  pio  gic  ...（包含我们定义的LED节点）

    # 4. 查看LED节点的属性（进入节点目录，cat查看属性值）
    cd led@300B0000
    cat compatible  # 查看compatible属性
    # 预期输出：vendor,led-dt（与设备树一致）
    cat reg  # 查看reg属性（十六进制输出，需转换为十进制）
    # 预期输出：300b0000 00001000（对应0x300B0000和0x1000）
    cat led-gpio  # 查看自定义GPIO属性
    # 预期输出：00000001 00000005 00000001（对应pio组1、引脚5、高电平有效）

    # 5. 查看节点状态（status属性，确认节点是否启用）
    cat status
    # 预期输出：okay（启用），若为disabled则内核不解析此节点
    ```
    关键场景：若`debugfs`中没有自定义的`led@300B0000`节点，说明设备树未被内核加载（需确认`.dtb`文件是否正确烧录到开发板）。

3.  `dmesg`：查看设备树解析与驱动匹配日志
    内核解析设备树和驱动匹配时会输出日志，通过`dmesg`过滤关键词可快速定位问题：
    ```bash
    # 1. 查看设备树解析日志（过滤"dtb"或节点名）
    dmesg | grep -i "dtb"
    # 预期输出：[    0.000000] OF: fdt: Machine model: Vendor Board Name（设备树加载成功）

    dmesg | grep -i "led@300B0000"
    # 预期输出：[    1.234567] of: registered device node /led@300B0000（节点注册成功）

    # 2. 查看驱动匹配日志（过滤驱动名或probe）
    dmesg | grep -i "led-dt-driver"
    # 预期输出：[  100.789012] led-dt-driver: probe of led@300B0000 succeeded（驱动匹配并probe成功）
    ```

### 问题排查：设备树与驱动不匹配的常见原因（compatible字段错误等）

设备树与驱动匹配失败的核心表现是“驱动`probe`函数不执行”，
常见原因集中在“设备树编写错误”“驱动匹配表错误”“内核配置错误”三类，以下按排查优先级排序：

1.  原因1：compatible字段不匹配（最常见）
    现象：`dmesg`中无驱动`probe`成功日志，`lsmod`显示驱动已加载但无设备关联；
    排查步骤：
    ```bash
    # 步骤1：查看驱动的of_match_table（确认驱动支持的compatible）
    # 方法：驱动源码中找到of_match_table，记录compatible值（如"vendor,led-dt"）

    # 步骤2：查看设备树的compatible（通过debugfs）
    cat /sys/kernel/debug/devicetree/base/led@300B0000/compatible
    # 若输出与驱动的of_match_table不一致（如多写一个空格、厂商名错误），修改设备树或驱动

    # 步骤3：验证匹配结果（内核日志）
    dmesg | grep -i "of_match"
    # 若显示"no matching of_node for driver led-dt-driver"，确认compatible字段完全一致（区分大小写）
    ```
    修复方案：确保驱动`of_match_table`中的`compatible`与设备树节点的`compatible`完全一致（字符串无拼写错误、无多余空格）。

2.  原因2：设备树节点未启用（status属性错误）
    现象：`debugfs`中找不到自定义节点，驱动无法匹配；
    排查步骤：
    ```bash
    # 步骤1：查看设备树源码中的status属性
    # 确认节点是否有status = "okay"; 若为status = "disabled"，内核不解析

    # 步骤2：若节点在子目录下，检查父节点状态
    # 例：若LED节点挂在"peripheral@10000000"下，需确认父节点status也是"okay"
    cat /sys/kernel/debug/devicetree/base/peripheral@10000000/status
    ```
    修复方案：将节点及父节点的`status`属性设为`"okay"`，重新编译烧录设备树。

3.  原因3：reg/interrupts属性格式错误（驱动解析失败）
    现象：`dmesg`中显示“read reg failed”或“invalid irq”，驱动`probe`返回错误；
    排查步骤：
    ```bash
    # 步骤1：查看设备树的reg属性格式（debugfs）
    cat /sys/kernel/debug/devicetree/base/led@300B0000/reg
    # 预期输出为两个值（基地址+长度），若只有一个值，说明格式错误

    # 步骤2：对比硬件手册，确认reg地址正确
    # 全志H3的GPIO控制器基地址是0x300B0000，若写为0x300B0001则错误

    # 步骤3：查看中断属性格式
    cat /sys/kernel/debug/devicetree/base/led@300B0000/interrupts
    # 预期输出为中断号+触发方式（如25 4），若格式错误需修改设备树
    ```
    修复方案：按硬件手册修正`reg`的基地址和长度、`interrupts`的中断号和触发方式，确保与驱动解析的数组长度一致（如`reg`需两个值，驱动用`of_property_read_u32_array`读取2个元素）。

4.  原因4：内核未启用设备树支持（配置错误）
    现象：所有设备树节点都无法解析，`dmesg`显示“dtb not found”；
    排查步骤：
    ```bash
    # 步骤1：查看内核配置（需内核支持设备树）
    zcat /proc/config.gz | grep CONFIG_OF
    # 预期输出：CONFIG_OF=y（启用设备树支持）

    # 步骤2：确认开发板启动时加载了正确的.dtb文件
    # 查看启动日志（如U-Boot日志），确认"Loading Device Tree to 87000000, size 28000 bytes"
    ```
    修复方案：重新配置内核，启用`CONFIG_OF=y`（设备树支持）和`CONFIG_OF_DEVICE=y`（设备树设备支持），重新编译内核和设备树。

## 驱动核心接口
与数据交互

**字符设备驱动核心接口：`cdev_init`/`cdev_add`与文件操作集`**file_operations`

字符设备是嵌入式Linux最常用的设备类型（如LED、按键、串口），
其驱动的核心是通过字符设备结构体（`cdev`）向内核注册设备，通过文件操作集（`file_operations`） 向用户态提供标准化接口（如`open`/`read`/`write`）。
这两个接口是字符设备驱动的“骨架”，所有功能都围绕它们展开。

1.  核心结构体解析：`cdev`与`file_operations`
    内核通过`cdev`抽象字符设备，通过`file_operations`定义设备的“操作行为”，二者必须关联才能实现完整功能：
    - `struct cdev`（字符设备抽象）：定义于`include/linux/cdev.h`，核心作用是“将设备与操作集绑定，并注册到内核”，
  关键成员：
      ```c
      struct cdev {
          struct kobject kobj;          // 内核对象（用于设备模型管理，无需手动赋值）
          struct module *owner;         // 模块所有者（必须设为THIS_MODULE，防止模块使用时被卸载）
          const struct file_operations *ops;  // 关联的文件操作集（核心，需手动赋值）
          struct list_head list;        // 内核用于管理设备的链表（无需手动赋值）
          dev_t dev;                    // 设备号（主设备号+次设备号，注册时分配）
          unsigned int count;           // 设备数量（单个设备设为1即可）
      };
      ```
    - `struct file_operations`（文件操作集）：定义于`include/linux/fs.h`，是“用户态系统调用与内核态驱动函数的映射表”，用户态调用`open("/dev/xxx")`时，内核会自动调用表中对应的`open`函数。
  常用成员（仅列核心）：
      ```c
      struct file_operations {
          struct module *owner;         // 模块所有者（与cdev的owner一致）
          int (*open) (struct inode *, struct file *);  // 打开设备（必须实现）
          ssize_t (*read) (struct file *, char __user *, size_t, loff_t *);  // 读取设备
          ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);  // 写入设备
          int (*release) (struct inode *, struct file *);  // 关闭设备（必须实现）
          long (*unlocked_ioctl) (struct file *, unsigned int, unsigned long);  // 控制设备（可选）
      };
      ```
    核心关系：`cdev.ops = &file_operations` → 驱动通过`cdev_add`将`cdev`注册到内核 → 用户态通过设备文件访问驱动时，内核通过`cdev`找到`file_operations`并调用对应函数。

2.  核心接口使用流程：初始化→注册→注销
    字符设备驱动的核心流程是“初始化`cdev`→ 关联`file_operations`→ 注册到内核→ 卸载时注销”，结合设备号管理（下一节讲），完整步骤如下：
    ```c
    #include <linux/init.h>
    #include <linux/module.h>
    #include <linux/fs.h>
    #include <linux/cdev.h>
    #include <linux/uaccess.h>

    #define DEV_NAME "demo_char"  // 设备名
    #define DEV_MAJOR 240        // 静态主设备号（240+，避免冲突）
    #define DEV_MINOR 0          // 次设备号
    #define DEV_COUNT 1          // 设备数量

    // 1. 定义全局变量（cdev和file_operations）
    static struct cdev demo_cdev;
    static char dev_buf[128] = "init data\n";  // 内核缓冲区（数据交互用）

    // 2. 实现file_operations的核心函数
    // 打开设备：用户态open时调用，可做初始化（如清空缓冲区）
    static int demo_open(struct inode *inode, struct file *file)
    {
        dev_info(file->f_inode->i_sb->s_dev, "device opened\n");
        return 0;
    }

    // 读取设备：用户态read时调用，内核→用户态传数据
    static ssize_t demo_read(struct file *file, char __user *buf, size_t count, loff_t *loff)
    {
        int ret;
        // copy_to_user：内核态→用户态拷贝（参数：用户缓冲区、内核缓冲区、长度）
        ret = copy_to_user(buf, dev_buf, count);
        if (ret > 0) {  // ret>0表示有ret字节未拷贝成功
            dev_err(file->f_inode->i_sb->s_dev, "read failed: %d bytes uncopy\n", ret);
            return -EFAULT;
        }
        return count;  // 返回实际读取字节数
    }

    // 写入设备：用户态write时调用，用户态→内核态传数据
    static ssize_t demo_write(struct file *file, const char __user *buf, size_t count, loff_t *loff)
    {
        int ret;
        memset(dev_buf, 0, sizeof(dev_buf));  // 清空内核缓冲区
        // copy_from_user：用户态→内核态拷贝（参数：内核缓冲区、用户缓冲区、长度）
        ret = copy_from_user(dev_buf, buf, count);
        if (ret > 0) {
            dev_err(file->f_inode->i_sb->s_dev, "write failed: %d bytes uncopy\n", ret);
            return -EFAULT;
        }
        dev_info(file->f_inode->i_sb->s_dev, "write data: %s", dev_buf);
        return count;  // 返回实际写入字节数
    }

    // 关闭设备：用户态close时调用，可做清理（如释放临时资源）
    static int demo_release(struct inode *inode, struct file *file)
    {
        dev_info(file->f_inode->i_sb->s_dev, "device closed\n");
        return 0;
    }

    // 3. 定义file_operations结构体并赋值
    static const struct file_operations demo_fops = {
        .owner = THIS_MODULE,    // 模块所有者，必须设为THIS_MODULE
        .open = demo_open,       // 关联打开函数
        .read = demo_read,       // 关联读取函数
        .write = demo_write,     // 关联写入函数
        .release = demo_release, // 关联关闭函数
    };

    // 4. 模块加载函数：初始化cdev并注册到内核
    static int __init demo_char_init(void)
    {
        int ret;
        dev_t dev_num;  // 设备号（主设备号+次设备号）

        // 步骤1：构造设备号（MKDEV宏：主设备号<<20 | 次设备号）
        dev_num = MKDEV(DEV_MAJOR, DEV_MINOR);

        // 步骤2：静态申请设备号（下一节讲动态申请）
        ret = register_chrdev_region(dev_num, DEV_COUNT, DEV_NAME);
        if (ret < 0) {
            dev_err(NULL, "register dev num failed: %d\n", ret);
            return ret;
        }

        // 步骤3：初始化cdev并关联file_operations
        cdev_init(&demo_cdev, &demo_fops);  // 核心接口1：绑定cdev与操作集
        demo_cdev.owner = THIS_MODULE;      // 设为当前模块，防止模块被意外卸载

        // 步骤4：将cdev注册到内核（核心接口2：内核开始管理该设备）
        ret = cdev_add(&demo_cdev, dev_num, DEV_COUNT);
        if (ret < 0) {
            dev_err(NULL, "add cdev failed: %d\n", ret);
            unregister_chrdev_region(dev_num, DEV_COUNT);  // 注册失败，释放设备号
            return ret;
        }

        dev_info(NULL, "char dev init success: major=%d, minor=%d\n", DEV_MAJOR, DEV_MINOR);
        return 0;
    }

    // 5. 模块卸载函数：注销cdev并释放设备号
    static void __exit demo_char_exit(void)
    {
        dev_t dev_num = MKDEV(DEV_MAJOR, DEV_MINOR);
        cdev_del(&demo_cdev);  // 核心接口3：从内核注销cdev
        unregister_chrdev_region(dev_num, DEV_COUNT);  // 释放设备号
        dev_info(NULL, "char dev exit success\n");
    }

    module_init(demo_char_init);
    module_exit(demo_char_exit);
    MODULE_LICENSE("GPL");
    MODULE_DESCRIPTION("Demo Character Device Driver");
    ```
    关键接口说明：
    - `cdev_init(struct cdev *cdev, const struct file_operations *fops)`：绑定`cdev`与`file_operations`，是驱动功能的“关联纽带”；
    - `cdev_add(struct cdev *cdev, dev_t dev, unsigned int count)`：将设备注册到内核，注册成功后内核会将设备与设备号关联，用户态可通过设备号访问；
    - `cdev_del(struct cdev *cdev)`：从内核注销设备，必须在模块卸载时调用，否则会导致资源泄漏。

## 驱动调试与问题定位基础

**驱动开发中“调试”的核心目标是“定位代码逻辑错误、硬件交互异常、内核接口使用不当**”三类问题，基础调试手段虽不涉及复杂工具，但能解决80%以上的入门级问题。
本节从“最基础的打印调试”到“灵活的模块参数”，再到“高频问题排查流程”，层层递进落地实操。

### 打印调试：`printk`日志级别配置与`dmesg`过滤技巧

**模块参数：通过`module_param`传递调试参数
**模块参数允许“加载驱动时传递参数”或“运行时修改参数”，
无需重新编译驱动即可调整调试逻辑，是“灵活调试”的核心手段。

**常见问题：驱动加载失败（依赖缺失、符号未导出）的排查流程
**
驱动加载失败是最高频的问题，
核心表现为`insmod`返回“Invalid module format”等错误
需按“加载日志→依赖检查→符号检查→版本检查”的流程排查。

1.  核心排查工具与前置准备
    需掌握三个核心工具：
    - `insmod`：加载驱动（需指定完整路径，不自动解决依赖）；
    - `modprobe`：加载驱动并自动解决依赖（推荐，需驱动在`/lib/modules/<内核版本>/`目录）；
    - `dmesg`：查看加载失败的内核日志（核心排查依据）；
    - `lsmod`：查看已加载模块及依赖关系；
    - `modinfo`：查看驱动模块信息（如内核版本、依赖符号）。

2.  高频问题1：依赖模块缺失（`modprobe`加载失败）
    现象：`insmod`加载时无错误，但`modprobe`加载失败；或驱动使用了其他模块的接口（如`gpio_request`依赖`gpio_core`模块），未加载依赖模块导致初始化失败。
    排查步骤：
    ```bash
    # 步骤1：用modprobe查看依赖关系（--show-depends显示驱动依赖的模块）
    modprobe --show-depends led_param_drv
    # 预期输出：insmod /lib/modules/5.15.71/kernel/drivers/gpio/gpio_core.ko
    #          insmod /lib/modules/5.15.71/kernel/drivers/led/led_param_drv.ko
    # 若输出缺失依赖模块（如gpio_core.ko），需先加载依赖

    # 步骤2：手动加载依赖模块
    modprobe gpio_core
    # 再加载目标驱动
    modprobe led_param_drv

    # 步骤3：验证依赖是否加载成功
    lsmod | grep -E "led_param_drv|gpio_core"
    # 预期输出两个模块均已加载
    ```
    修复方案：将依赖模块加入驱动加载脚本（如`/etc/init.d/led_drv`），或通过`MODULE_DEPEND`宏在驱动中声明依赖（内核自动加载）：
    ```c
    // 驱动源码中声明依赖gpio_core模块
    MODULE_DEPEND(led_param_drv, gpio_core, 1, 0, 0);  // 模块名、依赖模块名、最小版本等
    ```

3.  高频问题2：内核符号未导出（`Unknown symbol`错误）
    现象：`insmod`加载时返回“Unknown symbol in module”，`dmesg`日志显示“Unknown symbol xxx (err 0)”，原因是驱动使用了内核或其他模块的未导出符号（接口函数）。
    排查步骤：
    ```bash
    # 步骤1：提取错误日志中的未导出符号（如日志显示"Unknown symbol devm_gpio_request"）
    dmesg | grep "Unknown symbol"
    # 输出：[  123.456789] led_param_drv: Unknown symbol devm_gpio_request (err 0)

    # 步骤2：检查符号是否已导出（内核符号存于/proc/kallsyms，带"T"表示导出）
    cat /proc/kallsyms | grep "devm_gpio_request"
    # 若输出为空或带"t"（小写，未导出），说明符号未导出；若带"T"（大写，已导出），则是其他问题

    # 步骤3：确认驱动使用的符号是否属于内核导出接口
    # 内核导出符号需用EXPORT_SYMBOL或EXPORT_SYMBOL_GPL宏，如gpio_core模块导出devm_gpio_request
    ```
    修复方案：
    - 若符号属于内核模块：确保依赖模块已加载且符号已导出（如`gpio_core`模块导出`devm_gpio_request`）；
    - 若符号属于自定义模块：在提供符号的模块中用`EXPORT_SYMBOL`导出（如自定义`my_gpio`模块导出`my_gpio_request`）：
      ```c
      // 提供符号的模块（my_gpio.c）
      void my_gpio_request(int gpio) { /* 实现逻辑 */ }
      EXPORT_SYMBOL(my_gpio_request);  // 导出符号，供其他模块使用
      ```

4.  高频问题3：驱动与内核版本不匹配（`Invalid module format`）
    现象：`insmod`加载时返回“Invalid module format”，`dmesg`日志显示“version magic '5.15.71 SMP mod_unload' should be '5.15.71 SMP preempt mod_unload'”。
    原因：驱动编译时使用的内核源码版本/配置（如是否开启`PREEMPT`抢占）与开发板运行的内核不一致。
    排查步骤：
    ```bash
    # 步骤1：查看开发板运行的内核版本信息
    uname -a
    # 输出：Linux board 5.15.71 #1 SMP PREEMPT Tue Dec 5 10:00:00 CST 2025 armv7l GNU/Linux
    # 关键信息：5.15.71、PREEMPT（抢占模式）

    # 步骤2：查看驱动编译时的内核版本信息（modinfo）
    modinfo led_param_drv.ko | grep "vermagic"
    # 输出：vermagic:       5.15.71 SMP mod_unload armv7l
    # 对比发现：驱动编译时未开启PREEMPT，与开发板内核不一致

    # 步骤3：确认编译驱动的内核配置
    cat /home/user/linux-5.15.71/.config | grep CONFIG_PREEMPT
    # 输出：# CONFIG_PREEMPT is not set（驱动编译内核未开启抢占）
    ```
    修复方案：
    - 用与开发板内核“版本一致、配置一致”的内核源码重新编译驱动（核心：开发板内核的`.config`文件需复制到编译用的内核源码目录）；
    - 编译时指定内核配置文件：
      ```bash
      # 将开发板的/config文件复制到内核源码目录
      cp /proc/config.gz /home/user/linux-5.15.71/
      gunzip config.gz
      mv config .config
      # 重新配置并编译内核和驱动
      make menuconfig  # 确认配置一致
      make modules -j4
      ```

5.  高频问题4：硬件资源冲突（GPIO/中断被占用）
    现象：驱动`probe`函数返回`-EBUSY`，`dmesg`日志显示“gpio-12 already in use”或“interrupt 25 already allocated”。
    排查步骤：
    ```bash
    # 步骤1：检查GPIO占用情况（debugfs）
    mount -t debugfs none /sys/kernel/debug
    cat /sys/kernel/debug/gpio | grep -i "used"
    # 输出：gpio-12 (                    ): out  hi    [used by i2c1]
    # 说明GPIO12被i2c1模块占用

    # 步骤2：检查中断占用情况
    cat /proc/interrupts | grep 25
    # 输出：25:        123  sunxi-gpio  12 Edge      i2c1_irq
    # 说明中断25被i2c1占用

    # 步骤3：确认占用模块是否必要
    lsmod | grep i2c1
    # 若i2c1模块未使用，可卸载
    ```
    修复方案：
    - 卸载占用资源的无关模块（如`rmmod i2c_dev`）；
    - 修改驱动使用的硬件资源（如更换GPIO引脚为13，修改设备树或驱动源码）；
    - 在设备树中禁用占用资源的模块（如将i2c1节点的`status`设为`disabled`）。
