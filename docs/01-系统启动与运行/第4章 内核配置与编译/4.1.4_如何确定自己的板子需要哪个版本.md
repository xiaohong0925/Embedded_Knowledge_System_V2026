# 4.1.4 如何确定自己的板子需要哪个版本

> 所属章节：第4章 内核移植与编译 > 4.1 内核版本与源码获取
> 难度：[B→I] | 预计阅读时间：15分钟

## 本节导读
拿到一块新开发板后，面对 kernel.org 上几十个内核版本，你该怎么选？本节教你从厂商 BSP、主线 LTS 和社区支持三个维度做出判断，并通过检查设备树和驱动支持情况验证你的选择是否可行。

## 知识点1：版本选择策略 [I] ~800字

嵌入式开发板的内核版本选择，从来不是"选最新的"那么简单。你需要在**稳定性**、**功能需求**和**维护周期**之间做权衡。以下三个维度是做出决策的核心依据。

### 1. 厂商 BSP 推荐版本

绝大多数开发板出厂时，厂商都会提供一个板级支持包（BSP）。这个 BSP 的内核版本通常是厂商验证过、驱动最完整的版本。如果你是第一次接触这块板子，**优先使用厂商 BSP** 是最稳妥的做法。

拿到 BSP 后，第一件事就是确认它基于哪个内核版本：

```bash
# 进入内核源码目录，查看 Makefile 头部
head -5 Makefile
# 输出类似：
# VERSION = 5
# PATCHLEVEL = 10
# SUBLEVEL = 65
# EXTRAVERSION =
# NAME = Dare Coyote
```

这表示厂商 BSP 基于 **Linux 5.10.65**。如果这个数字是 4.x 甚至 3.x，就要警惕了——厂商可能已经停止维护，你需要考虑升级到更新的 LTS 版本。

### 2. 主线最新 LTS 版本

LTS（Long Term Support）版本是内核社区承诺维护至少2~6年的稳定版本，适合产品化项目。截至本教材编写时，推荐的 LTS 版本包括 **6.1**、**6.6**、**6.12** 等。你可以随时在 kernel.org 查看最新 LTS 状态。

[图1：kernel.org 主页标注 LTS 版本位置]

选择主线 LTS 的好处是：安全补丁持续更新、社区文档丰富、第三方驱动兼容性最好。但代价也很明显：如果厂商 BSP 已经包含大量闭源或专用驱动，迁移到主线可能需要重新移植这些驱动。

### 3. 社区支持状态

并非所有开发板都能完美运行主线内核。你需要检查社区是否活跃维护这块板子。一个简单的方法是搜索内核源码树中的设备树文件：

```bash
# 在内核源码根目录下搜索你的板子型号
grep -r "your_board_name" arch/arm/boot/dts/ arch/arm64/boot/dts/
```

如果能搜到 `.dts` 或 `.dtsi` 文件，说明主线内核已经包含这块板子的基础支持。接下来还要确认这个设备树首次出现在哪个内核版本，可以通过 git log 追溯：

```bash
cd arch/arm64/boot/dts/your_vendor/
git log --oneline your-board.dts
# 查看第一行提交，确定首次引入版本
```

### 版本选择决策流程

面对上述信息，按以下流程做出决策：

```mermaid
flowchart TD
    A[获取开发板与BSP] --> B{厂商BSP是否存在?}
    B -->|是| C[提取BSP内核版本号]
    B -->|否| D[检索主线内核设备树]
    C --> E{BSP版本是否≥4.19?}
    E -->|是| F[评估安全更新需求]
    E -->|否| G[BSP过旧建议升级]
    D --> H{主线是否已有设备树?}
    H -->|是| I[选择对应LTS版本]
    H -->|否| J[优先使用厂商BSP或社区fork]
    F --> K[最终决策：厂商BSP或主线LTS]
    G --> K
    I --> K
    J --> K
```

[图2：内核版本选择决策流程图]

⚠️ **陷阱**：不要凭直觉选择"最新稳定版"（如 6.x 的最新非 LTS 小版本）。非 LTS 版本只维护几个月，产品量产半年后可能就没有安全补丁了。

💡 **提示**：如果你是为了**学习内核移植**而不是做产品，反而建议从**最新主线版本**开始。因为主线代码最新、社区讨论最多，遇到问题更容易搜到答案。

## 知识点2：兼容性检查 [I] ~600字

选定版本后，必须验证这块板子的硬件能在该版本上正常工作。兼容性检查主要围绕**设备树**和**驱动支持**两个层面展开。

### 1. 设备树兼容性检查

设备树（Device Tree）是 ARM/ARM64 架构用来描述硬件信息的机制。内核启动时，会根据设备树来初始化外设。如果设备树文件缺失或版本不匹配，板子根本起不来。

设备树文件存放在内核源码的以下路径：

| 架构 | 设备树路径 |
|------|-----------|
| ARM 32位 | `arch/arm/boot/dts/` |
| ARM 64位 | `arch/arm64/boot/dts/` |
| RISC-V | `arch/riscv/boot/dts/` |

检查步骤：

```bash
# 以 ARM64 的 Rockchip 平台为例
ls arch/arm64/boot/dts/rockchip/ | grep -i rk3568
# 输出示例：
# rk3568-evb.dtb
# rk3568-nanopi-r5s.dtb
# rk3568-xxx.dtsi
```

如果能找到与你的板子完全匹配（或高度相关）的 `.dts` 或 `.dtsi` 文件，说明主线内核至少**知道这块硬件的存在**。接下来要检查这个设备树是否包含你需要的所有外设节点（网卡、USB、HDMI、SPI 等）。

⚠️ **陷阱**：文件名相近不代表完全兼容。例如 `rk3568-evb.dts` 是厂商评估板配置，如果你的板子是第三方定制板，GPIO 分配、电源树（Power Tree）可能完全不同，直接套用会导致外设不工作。

### 2. 驱动支持情况检查

设备树只是"硬件说明书"，真正干活的是驱动。你需要确认内核配置中是否启用了对应驱动。最简单的方法是通过 Kconfig 和 `make menuconfig` 检查。

以 Rockchip GMAC 网卡驱动为例：

```bash
# 搜索驱动对应的 Kconfig 配置项
grep -r "ROCKCHIP_DWMAC" drivers/net/ethernet/stmicro/stmmac/Kconfig
# 输出：
# config DWMAC_ROCKCHIP
#     tristate "Rockchip dwmac support"

# 检查当前 .config 是否启用了该驱动
grep CONFIG_DWMAC_ROCKCHIP .config
# 输出：
# CONFIG_DWMAC_ROCKCHIP=y
```

如果结果是 `=y`（编译进内核）或 `=m`（编译为模块），说明驱动已启用；如果结果为空或 `is not set`，说明驱动未启用，需要在 `make menuconfig` 中手动打开。

💡 **提示**：对于 GPU、VPU 这类复杂多媒体模块，主线驱动往往不完整。如果你的产品依赖硬件视频编解码，**务必提前在厂商社区确认主线驱动成熟度**，否则选错版本会导致产品上市前返工。

🔴 **危险**：千万不要在验证兼容性之前就把内核刷入产品的主存储（eMMC/NAND）。先用 TFTP + NFS 或 SD 卡启动做验证，刷坏主存储可能让板子变砖，需要用调试器恢复。

## 本节总结

| 决策因素 | 要点 | 推荐操作 |
|---------|------|---------|
| 厂商 BSP | 驱动最完整，版本可能较旧 | 首次使用优先采用；低于 4.19 建议升级 |
| 主线 LTS | 安全补丁持续，社区文档丰富 | 产品化首选 6.1/6.6/6.12 等 LTS 版本 |
| 社区支持 | 设备树存在 ≠ 所有外设可用 | 搜索 `.dts` 并用 `git log` 确认首次引入版本 |
| 设备树检查 | 硬件描述文件决定能否启动 | 在 `arch/xxx/boot/dts/` 目录下搜索板子型号 |
| 驱动检查 | Kconfig 决定驱动是否编译 | 用 `grep` 搜索 `.config` 中的 `CONFIG_xxx` 项 |
