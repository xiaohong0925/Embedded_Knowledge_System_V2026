# 4.2.6 .config文件：配置的物理载体

## 本节导读

本节揭开内核配置的"幕后文件"——`.config`的神秘面纱，你将学会读懂它的语法格式、安全地手动修改它，并掌握文件存放与备份的最佳实践。

## 知识点1：.config文件格式 [B] ~800字

当你在内核源码目录执行 `make menuconfig` 并勾选了若干选项后，配置结果究竟存在哪里？答案是内核源码根目录下的一个隐藏文件——`.config`。这个文件名前面有个点号，在Linux系统中表示它是隐藏文件，执行 `ls -a` 才能看到它。

`.config` 本质上是一个纯文本文件，你可以用任何文本编辑器打开它。它的语法极其简单，只有三种行：

### 1. 生效的配置项（启用）

```
CONFIG_SMP=y
CONFIG_MODULES=y
CONFIG_ARM64=y
```

所有配置项名称都以 `CONFIG_` 前缀开头，这是硬编码的规则。等号后面的值有三种可能：

| 值 | 含义 | 类比 |
|:---|:---|:---|
| `y` | 编译进内核（built-in） | 把家具固定在墙上，不可拆卸 |
| `m` | 编译为可加载模块（module） | 像U盘，用时插入，不用时拔出 |
| `n` | 未启用（不写入文件） | 这项功能不存在 |

⚠️ **陷阱**：如果某个选项未被选中，它在 `.config` 文件中**不会出现**（或者被注释掉），而不是写成 `CONFIG_XXX=n`。这是新手最容易误解的地方。

### 2. 字符串/数值型配置项

```
CONFIG_LOCALVERSION="-custom"
CONFIG_DEFAULT_HOSTNAME="(none)"
CONFIG_LOG_BUF_SHIFT=18
```

这类配置项在 menuconfig 中通常对应"输入框"，值可以是字符串或数字。

### 3. 注释行

```
# CONFIG_IPV6 is not set
# Linux/arm64 5.15.0 Kernel Configuration
#
# CONFIG_CGROUPS=y
```

以 `#` 开头的行是注释。特别值得注意的是 `# CONFIG_XXX is not set` 这种形式——它表示该功能被**明确禁用**，与"从未配置过"有微妙的区别。menuconfig 生成的 `.config` 会用这种方式记录你主动取消勾选的项。

### .config与menuconfig的关系

menuconfig 和 `.config` 的关系，就像 Word 软件和 docx 文件的关系：前者是编辑器，后者是存储格式。

```mermaid
flowchart LR
    A[开发者] -->|交互操作| B[make menuconfig]
    B -->|读取/解析| C[.config]
    B -->|更新| C
    C -->|被读取| D[内核构建系统]
    D -->|生成| E[include/config/auto.conf]
    D -->|生成| F[include/generated/autoconf.h]
```

[图1：menuconfig与.config的关系流程图]

当你启动 menuconfig 时，它会先读取 `.config` 中的已有配置，展示在界面上；当你修改并保存后，menuconfig 把新结果写回 `.config`。随后执行 `make` 时，构建系统会进一步把 `.config` 转换成两个关键文件：

- `include/config/auto.conf`：供 Makefile 使用的配置变量
- `include/generated/autoconf.h`：供 C 代码使用的宏定义

💡 **提示**：你可以直接查看 `include/generated/autoconf.h` 来确认某个 CONFIG 选项最终是否生效。例如 `grep CONFIG_SMP include/generated/autoconf.h` 能看到 `#define CONFIG_SMP 1`。

## 知识点2：手动编辑.config [I] ~600字

既然 `.config` 是纯文本，能否跳过 menuconfig，直接用 Vim 或 VS Code 编辑它？**答案是肯定的**，但必须了解其中的风险。

### 为什么可以手动编辑

1. **格式简单**：只有 `CONFIG_XXX=y/m/值` 和 `# 注释` 两种语法
2. **速度快**：如果你知道确切要改哪个选项，直接搜索替换比层层进入 menuconfig 菜单更高效
3. **自动化友好**：脚本生成 `.config` 是实现自动化编译的必要手段

### 手动编辑的三大注意事项

#### 注意事项一：依赖关系

内核配置项之间存在复杂的依赖链。例如 `CONFIG_USB_STORAGE` 依赖 `CONFIG_USB` 和 `CONFIG_SCSI`。如果你手动打开前者却未打开后者，配置处于**不一致状态**。menuconfig 会自动处理这些依赖，但纯文本编辑不会。

#### 注意事项二：无效值

假设你写成：

```
CONFIG_XXXXXXX=y
```

如果 `CONFIG_XXXXXXX` 根本不是一个合法的内核配置项名称，menuconfig 会直接忽略它，但构建系统不会报错——它只是白白占了一行，让你误以为配置成功了。

#### 注意事项三：格式陷阱

```
CONFIG_SMP = y        # 错误！等号两侧不能有空格
CONFIG_SMP=y          # 正确
```

⚠️ **陷阱**：`.config` 的解析器对空格敏感，`CONFIG_SMP = y` 会被视为无效行而被忽略。

### 修改后的恢复：make oldconfig

手动编辑后，最安全的做法是执行：

```bash
cd linux-5.15/
cp .config .config.backup        # 先备份
vim .config                       # 手动修改
make oldconfig                    # 恢复一致性
```

`make oldconfig` 的作用是：以内核源码中最新的 Kconfig 文件为基准，检查 `.config` 中的每个选项。如果发现新增选项或依赖缺失，它会逐个问你（或按默认值处理），最终生成一个**完整且一致**的新 `.config`。

💡 **提示**：如果你不想被交互式提问打断，可以用 `make olddefconfig`，它会自动对所有新选项使用默认值。

🔴 **危险**：永远不要在内核构建过程中手动编辑 `.config`。构建系统会缓存配置状态，运行时修改可能导致不可预知的行为。正确流程是：停掉编译 → 修改 `.config` → 执行 `make oldconfig` → 重新编译。

## 知识点3：.config文件位置 [B] ~400字

### 默认位置

在内核源码树中，`.config` 的默认位置就是**内核源码根目录**：

```bash
$ ls -a ~/linux-5.15/
.config          <--- 就在这里
Makefile
arch/
drivers/
Kconfig
...
```

这个位置是硬编码的。执行 `make` 时，构建系统会自动读取当前目录下的 `.config`。

### 自定义位置

如果你需要维护多个针对不同硬件平台的配置，可以通过环境变量指定别的路径：

```bash
export KCONFIG_CONFIG=/path/to/my_defconfig
make menuconfig      # 会读取指定路径的配置
make                 # 同样使用指定路径的配置
```

💡 **提示**：交叉编译时，很多开发者会把配置文件命名为 `config-imx6ull`、`config-rk3588` 等，放在内核源码外单独管理，通过 `KCONFIG_CONFIG` 切换。

### 备份策略

`.config` 是辛辛苦苦配置出来的成果，一旦丢失就得重新勾选几百个选项。推荐以下备份策略：

| 场景 | 备份方式 | 命令示例 |
|:---|:---|:---|
| 日常修改前 | 本地副本 | `cp .config .config.bak.$(date +%Y%m%d)` |
| 发布给团队 | 导出为 defconfig | `make savedefconfig`（生成 `defconfig` 文件） |
| 归档保存 | 版本控制 | `git add -f .config` 或单独存放到 configs/ 目录 |

`make savedefconfig` 是一个非常实用的命令，它会把当前 `.config` 中与默认值不同的选项提取出来，生成一个精简的 `defconfig` 文件。这个文件体积很小，适合作为"配置快照"分享给同事或保存到版本控制系统。

⚠️ **陷阱**：执行 `make distclean` 或 `make mrproper` 会**删除 `.config`**！这是内核清理命令的默认行为。在执行这类清理操作前，务必确认已备份。

## 本节总结

| 概念 | 要点 | 操作 |
|:---|:---|:---|
| `CONFIG_` 前缀 | 所有内核配置项的统一命名前缀 | `grep CONFIG_SMP .config` |
| `=y` / `=m` | 内置编译 vs 模块编译 | 根据硬件需求选择 |
| 注释行 `#` | `# CONFIG_XXX is not set` 表示明确禁用 | 不要手写 `=n` |
| 手动编辑 | 可以但需谨慎，可能破坏依赖一致性 | 编辑后执行 `make oldconfig` |
| 文件位置 | 默认在内核根目录，可经 `KCONFIG_CONFIG` 自定义 | `export KCONFIG_CONFIG=...` |
| 备份策略 | `cp` 备份、`savedefconfig` 导出、版本控制归档 | `make savedefconfig` |
