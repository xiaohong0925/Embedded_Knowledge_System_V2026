### 五大核心概念速查

**1. 内核版本号** [B]

三段式 `主版本.次版本.修订号`（如 `5.15.32`），LTS版本适合嵌入式产品。选择原则：用板子厂商验证过的版本，不盲目追新。

**2. Kconfig 与三态选择** [B]

Kconfig 是内核的配置描述语言，menuconfig 是它的可视化前端。每个选项有三种状态：

| 状态 | 符号 | 含义 | 产物 |
|------|------|------|------|
| **Y**（编译进内核） | `CONFIG_XXX=y` | 代码直接链接到内核镜像 | 体积增大，启动即用 |
| **M**（编译为模块） | `CONFIG_XXX=m` | 生成 `.ko` 文件，按需加载 | 灵活，可动态装卸 |
| **N**（不编译） | `# CONFIG_XXX is not set` | 代码不参与编译 | 镜像最小，功能缺失 |

⚠️ **陷阱**：嵌入式设备若无模块自动加载机制（如 `depmod`/`modprobe`），把关键驱动编译为 **Y** 更安全，否则启动后找不到 `.ko` 文件会导致设备失联。

**3. 镜像格式的族谱** [I]

内核编译产出一系列镜像文件，它们的"进化路线"是：

`vmlinux`（ELF调试文件）→ `objcopy` 剥离头 → `Image`（原始二进制）→ 可选压缩+自解压头 → `zImage` → 加U-Boot头 → `uImage`。设备树则是独立的 `*.dtb` 二进制，描述板级硬件信息。

💡 **提示**：ARM嵌入式最常用的启动组合是 `zImage + *.dtb`，U-Boot下用 `bootz` 命令启动。

**4. 编译的命门：ARCH 与 CROSS_COMPILE** [B]

交叉编译必须明确告诉内核"我在为谁编译"：

```bash
# 典型的 ARM 交叉编译命令
$ make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- menuconfig
$ make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- -j$(nproc)
```

🔴 **危险**：漏写 `ARCH` 或 `CROSS_COMPILE` 会导致内核编译成 x86 格式，或调用本机 gcc 而非交叉编译器，浪费大量时间后才发现镜像根本无法在 ARM 板上运行。

**5. 编译流程的里程碑** [B]

一个完整的内核编译流程包含五个里程碑：① 环境检查 → ② 导入默认配置（`make xxx_defconfig`）→ ③ 定制化配置（`make menuconfig`）→ ④ 编译内核与模块（`make`）→ ⑤ 安装模块与拷贝镜像到目标机。

## 本节总结

| 主题 | 核心要点 | 关键命令/文件 |
|------|---------|--------------|
| **内核版本** | 三段式编号；优先选LTS或厂商验证版本 | `uname -r` |
| **获取源码** | 官方kernel.org、发行版仓库、芯片厂商BSP | `git clone / wget` |
| **Kconfig配置** | 用 menuconfig 可视化勾选；/ 搜索关键词 | `make menuconfig` |
| **三态选择** | Y=编入内核，M=模块，N=不编译 | `.config` 中的 `=y` / `=m` |
| **默认配置** | 用 `arch/arm/configs/xxx_defconfig` 快速起步 | `make xxx_defconfig` |
| **镜像格式** | vmlinux→Image→zImage→uImage；dtb单独加载 | `vmlinux`, `zImage`, `*.dtb` |
| **交叉编译** | 必须指定 ARCH 和 CROSS_COMPILE | `make ARCH=arm CROSS_COMPILE=...` |
| **编译产物** | 内核镜像、System.map、模块.ko、设备树.dtb | `make modules_install` |
| **错误排查** | 先看错误类型→检查PATH/ARCH/依赖→clean重编 | `make clean`, `make distclean` |
