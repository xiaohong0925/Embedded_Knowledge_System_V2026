## <span class="blue">串口线连接细节——TX/RX交叉与GND共地

串口通信只需要三根线就能工作：**TX（发送）、RX（接收）、GND（地）**。但这三根线的接法有严格规则，接错轻则没输出，重则烧芯片。

### TX↔RX 交叉连接原则

串口通信是**双向的**：设备A说话（从TX发出去），设备B收听（从RX收进来）。因此两个设备的TX和RX必须**交叉连接**：

```
PC（USB转串口模块）        开发板
┌─────────┐               ┌─────────┐
│   TX    │──────────────→│   RX    │  ← 开发板接收来自PC的数据
│   RX    │←──────────────│   TX    │  ← PC接收来自开发板的数据
│   GND   │───────────────│   GND   │  ← 共地，必须接
└─────────┘               └─────────┘
```

[图2：TX/RX 交叉连接示意图——记住"我说你听，你说我听"]

用一句话记住：**"TX永远找对方的RX"**。这是串口连接的铁律。

> ⚠️ **陷阱**：如果错误地把 TX→TX、RX→RX 直连（"同名相连"），开发板的TX发出的信号送到了USB转串口模块的TX（而TX是输出引脚，不会读信号），结果双方都在"说"、没人"听"，屏幕上永远 blank 一片。

> 💡 **提示**：有些高级的USB转串口模块（如部分FT232模块）会把丝印标成"从模块视角出发"的RXD/TXD，容易混淆。如果接交叉没输出，就试试直连——少数模块的丝印标注确实不统一。但**默认先接交叉**。

### GND 必须共地——电平的参考点 [B]

GND（地线）不是"可有可无"的第三根线，它是整个通信的**电平参考基准**。没有共地，开发板输出的高电平（比如3.3V）在USB转串口模块看来可能是个不确定的电压，因为它不知道"0V在哪里"。

> 🔴 **危险**：不连GND时，通信可能"时好时坏"或出现大量乱码，误导你以为是波特率设置错误，排查方向完全跑偏。

> 💡 **提示**：如果串口线是三芯线（TX、RX、GND），务必三根全接。如果是排针方式连接，GND排针往往有好几个，接任意一个即可——它们在板子上是通的。

### VCC 接还是不接？ [B]

很多USB转串口模块有四根线：**TX、RX、GND、VCC（3.3V或5V）**。

| 线名 | 是否连接开发板 | 原因 |
|------|-------------|------|
| TX | ✅ 接开发板RX | 数据交叉发送 |
| RX | ✅ 接开发板TX | 数据交叉接收 |
| GND | ✅ 必须接 | 共地参考 |
| VCC | ❌ 一般**不接** | 避免与开发板独立电源冲突 |

[表1：串口线四芯连接规则表——VCC一般悬空或只接模块供电]

> ⚠️ **陷阱**：如果把USB转串口模块的VCC（比如5V）接到开发板的3.3V系统上，可能直接烧毁开发板的SoC！即使电压相同（都是3.3V），两个独立电源并联也可能因微小压差产生"倒灌电流"，长期工作不稳定。

> 💡 **提示**：只有一种情况需要连接VCC——**开发板本身没有独立电源**，需要从USB转串口模块取电（常见于最小系统板）。但即便如此，也要先确认模块的VCC输出电压和开发板的工作电压一致，且模块供电能力足够（通常<500mA）。

## <span class="blue">确认连接成功的检查清单 [B] 

接上电后，如何快速判断"连接成功了"？下面这张检查清单，按顺序逐条验证，任何一条不通过都能立即定位问题。

### 检查清单

| 序号 | 检查项 | 操作方法 | 预期结果 | 如果失败 |
|------|--------|---------|---------|---------|
| 1 | 开发板电源LED | 观察板子上的LED | 至少有一颗电源灯常亮 | 检查电源适配器、极性、电压 |
| 2 | PC识别串口设备 | `dmesg \| grep tty` 或设备管理器 | 出现 `/dev/ttyUSB0` 或 `COM3` | 换USB口、换线、检查驱动 |
| 3 | 串口工具能打开 | 用minicom/PuTTY打开端口 | 端口打开成功，无"Permission denied" | 检查权限（Linux加`sudo`或改`udev`规则） |
| 4 | 上电有输出 | 观察终端窗口 | 出现字符流（哪怕是乱码） | 检查波特率、TX/RX是否接反、GND是否连接 |

[表2：硬件连接成功检查清单——四步定位法]

### 详细操作说明

**检查项2：PC是否识别串口设备**

```bash
# Linux 下查看内核日志，找 USB 转串口设备的识别信息
dmesg | grep -i tty

# 典型输出示例：
# [  123.456] usb 1-2: cp210x converter now attached to ttyUSB0
# [  123.457] usbcore: registered new interface driver cp210x

# 也可以用 lsusb 查看芯片型号
lsusb | grep -i "serial\|UART\|cp210x\|ch340\|ftdi"
```

代码2：Linux 下查看串口设备是否被识别

> 💡 **提示**：常见USB转串口芯片有 CH340、CP2102、FT232、PL2303。不同芯片需要不同的驱动，Linux内核通常自带，Windows可能需要手动安装驱动。

**检查项4：上电有输出（哪怕乱码）**

如果终端出现了字符，但看起来是乱码（如 `~}` ），这**反而是好消息**——说明物理连接是成功的，只是波特率或编码不对。常见波特率尝试顺序：115200 → 38400 → 57600 → 9600。

> ⚠️ **陷阱**：如果终端完全空白，不要急于怀疑波特率。先确认 TX/RX 没有接反。一个简单的测试方法：用镊子或跳线帽短接USB转串口模块的TX和RX，然后在终端键盘上敲几个字符。如果回显正常，说明模块和PC是好的；如果不回显，问题在PC端或模块端。

> 💡 **提示**：有些开发板上电后需要按一下复位键（Reset）或电源键（Power Key）才开始输出启动日志。如果上电后LED亮了但终端没反应，按一下板子上的 RESET 键试试。

## 本节总结

| 概念 | 要点 | 操作 |
|------|------|------|
| 三步连接法 | 先串口、后电源 | 断电→接串口→开终端→接电源 |
| TX/RX 交叉 | "我说你听" | 模块TX → 板子RX；模块RX → 板子TX |
| GND 共地 | 电平参考基准 | 必须连接，不接会乱码或无输出 |
| VCC 不连 | 避免电源冲突 | 一般悬空，除非板子需要从串口取电 |
| 检查清单 | 四步定位 | LED亮 → 设备识别 → 端口打开 → 有输出 |

本节把"接几根线"这样一个看似简单的事，拆解成了一组**安全规范**和**排查逻辑**。<BR>嵌入式开发中，硬件连接是第一道门槛，也是最容易因为"想当然"而浪费大量时间的环节。

## 配套资源

### 关键术语速查
| 术语 | 含义 |
|------|------|
| TX | Transmit，发送端，数据从这儿出去 |
| RX | Receive，接收端，数据从这儿进来 |
| GND | Ground，地线，电平0V参考点 |
| VCC | 芯片供电电压（常指3.3V或5V） |
| TTL | Transistor-Transistor Logic，晶体管逻辑电平（这里指3.3V/5V电平标准） |
| 波特率 | 每秒传输的比特数，常见115200 |
| dmesg | Linux 内核环形缓冲区日志，可查看硬件识别信息 |
| 交叉连接 | TX接RX、RX接TX的接线方式 |
| 共地 | 两个独立设备的GND连在一起，确保电平基准一致 |
