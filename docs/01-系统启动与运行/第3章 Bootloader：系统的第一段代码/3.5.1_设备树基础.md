# 3.5.1 设备树基础

> 所属章节：第3章 嵌入式Linux启动流程 > 3.5 设备树
> 难度：[B→B] | 预计阅读时间：15分钟

## 本节导读
本节将带你认识嵌入式Linux中描述硬件的"通用语言"——设备树。学完后，你能理解设备树存在的意义，看懂简单的`.dts`源文件，并能在U-Boot中查看设备树信息。

---

## 知识点1：设备树是什么 [B][M] ~1,000字

在嵌入式Linux的世界里，每块开发板的硬件配置都不同：有的用了GPIO口控制LED，有的接了I2C温度传感器，有的外接了SPI显示屏。这些硬件信息，内核启动时必须知道，否则它就像一个蒙着眼睛的司机，根本不知道怎么操作板子上的器件。

### 历史包袱：硬编码的噩梦

早期的Linux内核（2.6时代之前），硬件信息是直接写死在内核代码里的。以ARM平台为例，每来一款新开发板，内核里就要增加几百甚至上千行的C代码来描述这块板子的CPU型号、内存大小、外设连接方式等。这带来两个致命问题：

1. **内核膨胀**：支持100款开发板，内核里就有100份硬件描述代码，互相搅在一起，牵一发而动全身。
2. **无法通用**：同一款芯片，换块开发板就要重新编译内核。内核镜像和硬件强耦合。

💡 **提示**：这就像一个餐厅把每桌客人的忌口都写进厨房的操作手册里——客人一换，手册就得重写。

### PowerPC的启示：Device Tree

IBM在PowerPC架构上早已用一种叫**设备树（Device Tree）**的机制来描述硬件。它的核心思想非常优雅：**把硬件描述从内核代码中抽离出来，做成一份独立的数据文件**。ARM社区在2011年前后引入了这个方案，从此ARM Linux内核进入了"设备树时代"。

### 设备树是硬件的"身份证"

你可以把设备树想象成一份**硬件身份证**或**硬件说明书**：

- 它用树状结构描述板子上有什么硬件（CPU、内存、外设）
- 它说明每个硬件的型号、连接方式、配置参数
- 它不包含任何驱动逻辑，只陈述"事实"

内核启动时，会读取这份"身份证"，然后拿着它去找合适的驱动程序来操作对应的硬件。这样一来：

- **内核通用化**：同一个内核镜像，配上不同的设备树文件，就能在不同板子上跑起来。
- **硬件描述独立化**：改硬件描述不需要重新编译整个内核，只需要改设备树文件。

### 设备树的两张面孔：.dts 和 .dtb

设备树在开发阶段以**文本源文件**的形式存在，后缀名是`.dts`（Device Tree Source）。这个文件人类可以阅读和编辑。

但内核启动时不可能去解析一段文本，那太慢了。所以需要把`.dts`编译成**二进制的扁平化格式**，后缀名是`.dtb`（Device Tree Blob）。这个二进制文件内核可以直接加载到内存中解析。

[图1：设备树编译流程图]

```mermaid
flowchart LR
    A[.dts 源文件<br/>人类可读的文本] --> B[dts编译器<br/>dtc]
    B --> C[.dtb 二进制文件<br/>内核可直接加载]
    C --> D[U-Boot加载dtb到内存]
    D --> E[Linux内核读取<br/>并解析设备树]
    E --> F[内核匹配驱动<br/>初始化硬件]

    style A fill:#e1f5e1
    style C fill:#fff3cd
    style F fill:#f8d7da
```

编译命令很简单：

```bash
# 使用dtc（Device Tree Compiler）编译
# 从.dts 生成 .dtb
dtc -I dts -O dtb -o myboard.dtb myboard.dts

# 反编译：从.dtb 还原 .dts（方便调试查看）
dtc -I dtb -O dts -o myboard.dts myboard.dtb
```

⚠️ **陷阱**：初学者常犯的错误是直接在目标板上编辑`.dtb`文件——千万别这么做！`.dtb`是二进制格式，用hex编辑器也看不出所以然。始终编辑`.dts`源文件，重新编译生成`.dtb`。

💡 **提示**：Linux内核源码的`arch/arm64/boot/dts/`（或`arch/arm/boot/dts/`）目录下存放了数千款开发板的`.dts`文件，是学习设备树语法最好的参考资料。

🔴 **危险**：设备树只描述硬件，不包含任何业务逻辑。不要在设备树里写if-else、循环等逻辑。如果看到有人这么做，那是错误的做法。

## 知识点2：设备树基本语法 [B] ~800字

设备树源文件（`.dts`）的语法类似C语言和JSON的结合体，非常直观。我们只需要掌握三个核心概念：**节点（node）**、**属性（property）**和**compatible**。

### 节点（Node）：硬件的"抽屉"

设备树用树状结构组织，每个硬件对应一个**节点**。节点用大括号`{ }`包裹，类似C语言的结构体。

```dts
/* 这是一个最简单的设备树示例 */
/dts-v1/;

/ {
    model = "My Development Board";
    compatible = "myvendor,myboard";

    /* CPU节点 */
    cpus {
        cpu@0 {
            compatible = "arm,cortex-a53";
        };
    };

    /* 内存节点 */
    memory@80000000 {
        device_type = "memory";
        reg = <0x80000000 0x20000000>;  /* 起始地址0x80000000, 大小512MB */
    };

    /* UART串口节点 */
    uart0: serial@101f0000 {
        compatible = "ns16550a";
        reg = <0x101f0000 0x100>;       /* 寄存器基地址和长度 */
        interrupts = <1>;
        clock-frequency = <1843200>;
    };
};
```

代码说明：
- `/dts-v1/;`：声明设备树版本号，必须放在第一行。
- `/ { }`：根节点，整棵树的起点，所有硬件都从根节点往下挂。
- `memory@80000000`：`memory`是节点名，`@`后面的数字是硬件地址，方便人类阅读时识别。
- `uart0: serial@101f0000`：`uart0:`是给节点取的**别名**，后续可以用`&uart0`来引用。

### 属性（Property）：节点的"标签"

节点内部用`属性 = 值;`的格式来描述硬件细节。属性值有四种常见类型：

| 属性名 | 作用 | 示例 |
|--------|------|------|
| `compatible` | 驱动匹配的关键标识，内核靠它找驱动 | `"myvendor,myboard"` |
| `reg` | 描述硬件寄存器地址和大小 | `<0x101f0000 0x100>` |
| `interrupts` | 描述中断号 | `<1>` |
| `#address-cells` / `#size-cells` | 定义子节点`reg`用几个32位数字表示地址/大小 | `<1>` / `<1>` |

### compatible：驱动匹配的"暗号"

`compatible`是设备树中**最重要**的属性，它是内核和设备驱动之间的"接头暗号"。

内核启动时，会遍历设备树中的每个节点，读取它的`compatible`值，然后去内核已注册的驱动列表中找"能对上暗号"的驱动。一旦匹配成功，驱动就会加载并初始化这个硬件。

```dts
compatible = "ns16550a";
```

上面这个值意味着：内核会找`compatible`包含`"ns16550a"`的UART驱动来处理这个串口。

💡 **提示**：`compatible`常常写多个值，从具体到通用，越靠前越具体。例如：`compatible = "myvendor,myuart", "ns16550a";`。这样如果"myvendor,myuart"没有专用驱动，还能 fallback 到通用的"ns16550a"驱动。

⚠️ **陷阱**：如果`compatible`写错了，或者内核里没有对应的驱动，内核启动时会静默忽略这个设备（没有报错！）。这会让初学者非常困惑——硬件明明接了，内核怎么没反应？遇到这种问题，先检查`compatible`是否和内核驱动匹配。

🔴 **危险**：`compatible`的字符串必须和内核驱动的`of_device_id`表完全一致，包括大小写。`"ns16550a"`和`"ns16550A"`会被视为两个不同的值！

### 节点的引用：`&别名`

设备树允许在一个地方定义节点，在另一个地方补充或修改它。这用到了**引用（reference）**机制：

```dts
/* 在根节点外，修改/追加 uart0 节点的属性 */
&uart0 {
    status = "okay";          /* 启用这个设备 */
    current-speed = <115200>;  /* 追加波特率信息 */
};
```

`&uart0`表示"我要修改前面别名叫`uart0`的那个节点"。这种写法在大型设备树中非常常见——芯片厂商提供一份基础`.dtsi`文件（类似头文件），开发板厂商再用`&引用`的方式做针对性修改。

## 知识点3：U-Boot中的设备树操作 [B] ~600字

U-Boot作为 bootloader，负责把Linux内核和设备树文件加载到内存中。它提供了一组`fdt`命令（fdt = Flattened Device Tree），让你在启动前查看和修改设备树。

### 加载dtb到内存

典型的启动流程中，U-Boot会把`.dtb`文件从Flash/SD卡/网络加载到RAM的某个地址：

```bash
# 从TFTP服务器下载dtb到内存地址0x88000000
U-Boot> tftp 0x88000000 myboard.dtb

# 或者从MMC（SD卡）读取
U-Boot> load mmc 0:1 0x88000000 /boot/myboard.dtb
```

加载完成后，你需要告诉U-Boot"设备树在内存的哪个位置"：

```bash
# 设置设备树在内存中的地址
U-Boot> fdt addr 0x88000000
```

这条命令是后续所有fdt操作的前提，不执行它，`fdt list`、`fdt print`等命令都会失效。

⚠️ **陷阱**：`fdt addr`设置的地址必须是你实际加载dtb的地址。如果填错了，U-Boot会解析一块随机的内存内容，输出一堆乱码或者直接挂死。

### 查看设备树结构：fdt list

想看设备树的"骨架"，用`fdt list`列出某个节点下的子节点：

```bash
# 列出根节点下的所有一级子节点
U-Boot> fdt list /
```

输出类似这样：

```
/ {
    cpus {
    };
    memory@80000000 {
    };
    serial@101f0000 {
    };
};
```

想看某个节点下的完整属性？加上节点路径：

```bash
# 查看 /serial@101f0000 节点的所有属性
U-Boot> fdt list /serial@101f0000
```

### 查看属性值：fdt print

`fdt print`用于查看某个节点的具体属性值，或者不带参数打印整棵树：

```bash
# 打印整棵设备树（输出很长！）
U-Boot> fdt print

# 只打印某个节点的属性
U-Boot> fdt print /serial@101f0000

# 只打印某个属性的值
U-Boot> fdt print /serial@101f0000 compatible
```

典型的输出：

```
compatible = "ns16550a";
reg = <0x101f0000 0x00000100>;
interrupts = <0x00000001>;
clock-frequency = <0x001c2000>;
```

💡 **提示**：设备树里的数值默认是十六进制显示的。上面`clock-frequency`的`0x001c2000`就是十进制的1843200。

### 启动时传递dtb给内核

最后一步，在启动内核时把dtb地址传给内核：

```bash
# booti 用于启动ARM64内核，三个参数分别是：内核地址、initrd地址（-表示无）、dtb地址
U-Boot> booti 0x80080000 - 0x88000000
```

内核启动后，会从`0x88000000`读取设备树，完成后续的设备初始化和驱动匹配。

🔴 **危险**：如果在U-Boot中用`fdt set`或`fdt rm`修改了设备树，这些修改只存在于内存中，不会写回存储介质。重启后修改会丢失。要永久修改，必须去编辑`.dts`源文件并重新编译烧录。

## 本节总结

设备树把硬件描述从内核代码中分离出来，让内核变得通用、可移植。本节从概念到语法再到实操，建立了对设备树的初步认知。

| 概念 | 要点 | 操作 |
|------|------|------|
| 设备树 | 硬件的"身份证"，树状数据结构 | 编辑`.dts` → `dtc`编译 → 生成`.dtb` |
| `.dts` | 人类可读的文本源文件 | 用文本编辑器修改 |
| `.dtb` | 内核加载的二进制格式 | 用`dtc`编译生成 |
| 节点 | 描述一个硬件设备 | 语法：`节点名@地址 { ... };` |
| `compatible` | 驱动匹配暗号，最关键属性 | 必须和内核驱动严格一致 |
| `fdt addr` | 告诉U-Boot设备树在哪 | `fdt addr 0x88000000` |
| `fdt print` | 查看设备树属性 | `fdt print /节点路径` |
