# 5.2.3 虚拟文件系统：/proc与/sys的挂载

> 所属章节：第5章 根文件系统构建 > 5.2 核心目录与挂载点

## 本节导读

本节带你认识Linux中三个"看不见摸不着"却无处不在的虚拟文件系统——/proc、/sys和/dev。你将理解为什么`ps`命令能看到进程信息、`mdev`能自动创建设备节点，并能亲手完成挂载操作。

[图1：虚拟文件系统与内核关系图]

上图揭示了虚拟文件系统的本质：**它们不是存储在磁盘上的真实文件，而是内核向用户空间开放的"窗口"**。当你读取`/proc/loadavg`时，内核实时计算并返回当前平均负载；当你向`/sys/class/leds/.../brightness`写入数值时，内核直接控制硬件LED的亮度。

## 知识点1：/proc文件系统——进程信息窗口 [I]

`/proc`是"process filesystem"的缩写，但它展示的内容远不止进程。它像一个**24小时直播的内核情报站**，把内核内部的各种运行时信息翻译成人类可读的文本文件。

### /proc里有什么

进入`/proc`目录，你会看到两类内容：

| 内容类型 | 命名规律 | 示例 | 说明 |
|---------|---------|------|------|
| 进程目录 | 纯数字命名 | `1/`, `245/` | 每个运行中的进程都有一个以PID命名的目录 |
| 系统信息文件 | 英文单词命名 | `cpuinfo`, `meminfo`, `loadavg` | 反映CPU、内存、系统负载等全局状态 |

以PID为名的子目录里，存放着该进程的详细信息。比如`/proc/1/`是init进程的情报中心，里面有`cmdline`（启动命令行参数）、`status`（进程状态和内存占用）、`fd/`（打开的文件描述符）等文件。

### 为什么`ps`命令能工作

💡 **提示**：`ps`命令并不调用什么神秘接口，它的底层实现就是**遍历`/proc`目录下的数字文件夹**，读取每个目录里的`stat`文件，然后整理成表格展示。如果`/proc`没有挂载，`ps`会输出为空或报错。

```bash
# 手动模拟ps的核心逻辑：列出所有进程PID
ls /proc | grep -E '^[0-9]+$' | sort -n
```

### 挂载/proc

在嵌入式设备的启动脚本中，挂载/proc是早期初始化步骤之一：

```bash
# 手动挂载proc文件系统
mount -t proc proc /proc
```

命令拆解：`mount`是挂载命令；`-t proc`指定文件系统类型为proc；第一个`proc`是虚拟设备名（语法占位）；`/proc`是挂载目标目录。

也可以把这一行写入`/etc/fstab`，让启动时自动挂载：

```
proc  /proc  proc  defaults  0  0
```

```bash
# 查看CPU信息、内存使用、系统负载
cat /proc/cpuinfo
cat /proc/meminfo
cat /proc/loadavg
```

⚠️ **陷阱**：`/proc`里的文件**不是普通文件**，它们没有固定大小（`ls -l`显示为0字节），是内核动态生成的。不要把它们当作日志文件来存储数据——系统重启后`/proc`里的内容全部消失。

🔴 **危险**：`/proc/sys/`目录下的某些文件涉及内核参数（如`/proc/sys/kernel/sysrq`），修改不当可能导致系统不稳定。嵌入式设备中通常会把这些关键文件设为只读或受权限保护。

## 知识点2：/sys文件系统——设备驱动展示厅 [I]

如果`/proc`是进程情报站，那`/sys`就是**设备与驱动的展览馆**。`/sys`全称为sysfs，专门用来展示内核设备模型（Device Model），把原本混杂在`/proc`里的设备信息分离出来，结构更清晰。

### /sys的目录结构

| 目录 | 内容 |
|------|------|
| `/sys/class/` | 按**功能类别**组织设备，如`leds/`、`gpio/`、`net/` |
| `/sys/block/` | 块设备（磁盘、闪存分区） |
| `/sys/bus/` | 总线类型，如`platform/`、`usb/`、`i2c/` |
| `/sys/devices/` | 设备的物理层级树，反映硬件拓扑关系 |
| `/sys/module/` | 已加载的内核模块信息 |

💡 **提示**：同一个设备在`/sys`中可能有多个"入口"。例如一个LED设备，既在`/sys/class/leds/`下按功能分类出现，也在`/sys/devices/`下按物理拓扑出现。它们本质上是指向同一内核对象的**符号链接**。

### 为什么mdev/udev依赖/sys

嵌入式系统中，热插拔设备的自动管理依靠`mdev`（BusyBox提供）或`udev`。其工作原理是：

1. 内核检测到新设备，在`/sys`下创建对应目录和属性文件
2. `mdev/udev`监听内核的**uevent**通知
3. 根据`/sys`中设备的属性（如主次设备号），在`/dev`下创建设备节点

没有`/sys`，`mdev/udev`就成了"瞎子"——它不知道有哪些设备存在，也不知道该创建设备节点。

### 挂载/sys与实用操作

```bash
# 手动挂载sysfs
mount -t sysfs sysfs /sys

# 查看系统有哪些LED
ls /sys/class/leds/

# 控制LED亮度：0为关闭，1为开启
echo 0 > /sys/class/leds/heartbeat/brightness
echo 1 > /sys/class/leds/heartbeat/brightness
```

⚠️ **陷阱**：嵌入式设备的`/sys`内容因内核配置和设备树差异而不同。在板A上存在的`/sys/class/leds/`，在板B上可能没有——这完全取决于内核驱动是否加载了对应的LED设备。

## 知识点3：/dev目录——设备节点大门 [I]

`/dev`目录是**用户空间与硬件设备沟通的必经之路**。所有需要与硬件交互的操作（读串口、写Flash、访问传感器），最终都要通过`/dev`下的某个设备节点来完成。

### 设备节点是什么

设备节点是一种**特殊文件**，它不代表磁盘上的数据，而是代表一个内核管理的设备。每个设备节点有**主设备号**（标识设备类别）和**次设备号**（标识同类设备中的具体实例）。

```bash
# 查看/dev下的设备节点属性
ls -l /dev/ttyS0 /dev/null
# crw-rw---- 1 root root 4, 64 Jan 1 00:00 /dev/ttyS0
# crw-rw-rw- 1 root root 1,  3 Jan 1 00:00 /dev/null
```

权限位前的`c`表示**字符设备**，`b`表示**块设备**。`4, 64`就是主/次设备号。

### /dev的两种管理模式

| 模式 | 机制 | 适用场景 |
|------|------|---------|
| 静态 | 启动前用`mknod`预先创建所有节点 | 极简嵌入式系统 |
| 动态 | 通过`mdev`/`udev`按需创建 | 大多数现代嵌入式系统 |

现代嵌入式Linux普遍采用动态模式。启动脚本里通常只需要一行配置：`echo /sbin/mdev > /proc/sys/kernel/hotplug`，然后执行`mdev -s`扫描`/sys`并创建当前已存在设备对应的节点。

⚠️ **陷阱**：不要手动删除`/dev`下的设备节点然后困惑为什么设备"不见了"。动态管理模式下，这些节点是由`mdev`根据`/sys`状态维护的。如果节点消失，首先检查`/sys`里对应设备是否还存在，以及`mdev`服务是否正常运行。

`/dev`的深入内容（手动`mknod`创建设备节点、主次设备号分配规则）将在**5.4节**详细讲解。

## 知识点4：挂载验证——确认虚拟文件系统就位 [B]

挂载完虚拟文件系统后，如何确认它们真的在工作？

### 查看当前挂载状态

```bash
# 显示所有已挂载的文件系统
mount

# 只查看proc类型的挂载
mount -t proc

# 查看指定挂载点是否已挂载
mountpoint /proc   # 输出: /proc is a mountpoint
mountpoint /sys    # 输出: /sys is a mountpoint
```

在正确挂载的系统上，`mount`输出中应包含：

```
proc on /proc type proc (rw,relatime)
sysfs on /sys type sysfs (rw,relatime)
```

### 快速有效性检查

```bash
# 检查/proc是否可用（/proc/1是init进程目录）
[ -d /proc/1 ] && echo "/proc OK" || echo "/proc NOT mounted"

# 检查/sys是否可用
[ -d /sys/class ] && echo "/sys OK" || echo "/sys NOT mounted"

# 检查ps命令能否正常工作（依赖/proc）
ps | head -3
```

如果`/proc`没有挂载，`ps`通常会报错或返回空列表；如果`/sys`没有挂载，`ls /sys/class/`会提示目录不存在。

⚠️ **陷阱**：`mount`命令显示挂载成功，不代表文件系统真正可用。内核配置中如果**没有启用CONFIG_PROC_FS或CONFIG_SYSFS**，挂载命令会失败或挂载后目录为空。嵌入式开发中裁剪内核时，务必确认这两个配置项已勾选。

## 本节总结

这三棵虚拟文件系统树是Linux内核与用户空间之间的**三座桥梁**：/proc让你看见系统在做什么，/sys让你控制系统能做什么，/dev让你真正与硬件打交道。缺少任何一个，嵌入式Linux系统都无法正常运转。

**表1：三大虚拟文件系统对比**

| 特性 | /proc | /sys | /dev |
|------|-------|------|------|
| 全称 | procfs | sysfs | devtmpfs |
| 核心内容 | 进程信息、内核参数 | 设备层级、驱动属性 | 设备访问节点 |
| 内核配置项 | CONFIG_PROC_FS | CONFIG_SYSFS | CONFIG_DEVTMPFS |
| 典型使用者 | `ps`, `top`, 系统监控 | `mdev`, `udev`, 硬件控制 | 应用程序、驱动 |
| 数据持久性 | 不持久（内存中动态生成） | 不持久 | 节点不持久，指向的硬件存在 |
| 挂载命令 | `mount -t proc proc /proc` | `mount -t sysfs sysfs /sys` | 通常由内核自动挂载 |

| 概念 | 核心要点 | 关键操作 |
|------|---------|---------|
| **/proc** | 进程与内核运行时信息的实时窗口，`ps`命令依赖它 | `mount -t proc proc /proc` |
| **/sys** | 设备驱动的结构化展示，mdev/udev的热插拔基础 | `mount -t sysfs sysfs /sys` |
| **/dev** | 用户空间访问硬件的节点入口，支持静态/动态管理 | `mdev -s` 扫描创建节点 |
| **验证** | 用`mount`、`mountpoint`和目录测试确认挂载有效 | `mount -t proc` 查看状态 |
