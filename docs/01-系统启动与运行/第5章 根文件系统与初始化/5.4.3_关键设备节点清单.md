## 本节导读

设备节点是用户空间程序与内核驱动沟通的"门牌号"。本节将介绍嵌入式Linux中最不可或缺的5类设备节点，讲解它们的作用、创建方法，以及如何在实际系统中验证它们是否正确存在。学完本节，你将能够独立检查最小根文件系统中的`/dev`目录，确保关键设备不缺失。

## 知识点1：必备设备节点 [B][M] ~1,000字

在嵌入式系统中，`/dev`目录通常需要精简——毕竟Flash空间有限。但以下5类设备节点是**"绝对不可删除"**的核心成员。它们就像是家里的水、电、燃气接口，少了任何一个，系统要么无法启动，要么功能残缺。

### 1. /dev/console —— 系统"嘴和耳朵" [B]

**作用**：`console`是系统启动阶段和单用户模式下的**唯一交互通道**。内核启动早期的`printk`消息、init进程的输出、紧急模式下的shell提示符，全部依赖这个节点。

🔴 **危险**：如果`/dev/console`缺失，内核启动后可能**没有任何输出**，开发者会误以为系统死机，其实它只是在"哑口无言"地运行。

创建方式：
```bash
mknod /dev/console c 5 1
```

`console`会映射到实际的串口（如UART0），具体映射由内核启动参数`console=ttyS0,115200`指定。

### 2. /dev/null —— 数字"黑洞" [B]

**作用**：任何写入`null`的数据都会被**静默丢弃**，读取`null`则立即得到EOF（文件结束）。

💡 **提示**：这是Shell脚本中最常用的重定向目标。例如将某个命令的标准输出和错误输出全部丢弃：
```bash
my_noisy_command > /dev/null 2>&1
```

没有`/dev/null`，很多脚本会因无法重定向而报错退出。

创建方式：
```bash
mknod /dev/null c 1 3
```

### 3. /dev/zero —— 零字节"打印机" [B]

**作用**：读取`zero`会得到**无限长的全零字节流**；写入的数据则被丢弃。

常见用途：快速生成指定大小的空文件：`dd if=/dev/zero of=blank.bin bs=1M count=10`。

创建方式：
```bash
mknod /dev/zero c 1 5
```

⚠️ **陷阱**：不要对`/dev/zero`执行`cat`，它会无限输出零字节，瞬间占满终端或磁盘！

### 4. /dev/random 与 /dev/urandom —— 随机数"源泉" [I]

**作用**：这两个节点从内核的**熵池（entropy pool）**中提取随机数据，供加密、密钥生成、协议栈使用。

| 节点 | 阻塞行为 | 用途 | 嵌入式建议 |
|------|----------|------|-----------|
| `/dev/random` | **会阻塞**，直到熵池足够 | 高强度加密密钥 | 谨慎使用，启动早期可能长期阻塞 |
| `/dev/urandom` | **不阻塞**，伪随机 | 普通随机数、会话ID、TCP序列号 | **推荐**，适合大多数场景 |

⚠️ **陷阱**：在熵匮乏的嵌入式设备上（如缺少键盘、鼠标、磁盘抖动等熵源），读取`/dev/random`可能**永久阻塞**。例如SSH服务在生成主机密钥时卡住，往往就是这个原因。解决方案：使用`/dev/urandom`替代，或引入硬件随机数发生器驱动。

创建方式：
```bash
mknod /dev/random  c 1 8
mknod /dev/urandom c 1 9
```

### 5. /dev/tty* —— 终端与串口 [B]

**作用**：`tty`代表**电传打字机（Teletype）**的缩写，如今泛指字符终端设备。

在嵌入式场景中，最常用的是：
- `/dev/tty` —— 当前进程的终端别名
- `/dev/tty0` ~ `/dev/tty63` —— 虚拟控制台
- `/dev/ttyS0`、`/dev/ttyS1`… —— 物理串口（UART）

如果你通过串口线连接开发板调试，那么`getty`或`login`程序正是在`/dev/ttyS0`上等待你的输入。

创建方式示例（假设UART0为主调试串口）：
```bash
mknod /dev/ttyS0 c 4 64
mknod /dev/tty    c 5 0
```

⚠️ **陷阱**：串口主次设备号在不同内核版本中可能变化。在较新内核中，`ttyS0`的设备号可能是`4:64`，但某些发行版使用动态分配（`udev`）。手动创建节点时，务必通过`Documentation/devices.txt`或`/proc/devices`确认当前内核的设备号。

### 6. 主设备号与次设备号的概念 [I]

每个设备节点的文件名旁，内核关心的是**一对数字**——主设备号（Major）标识驱动类型，次设备号（Minor）标识该驱动下的具体实例。

例如：`/dev/console c 5 1` 表示
- `c` = 字符设备（character）
- `5` = 主设备号 → "tty驱动"类
- `1` = 次设备号 → 该类的第1个实例，即控制台

可以把主设备号理解为"小区楼号"，次设备号理解为"房间号"。

## 知识点2：设备节点验证 [B] ~400字

创建了设备节点后，必须验证它们是否正确可用。最基础的检查方法如下。

### 操作步骤

1. **列出设备节点详细信息**
```bash
ls -l /dev/
```

典型输出片段：
```
crw-r--r-- 1 root root 5, 1 Jan  1 00:00 console
crw-rw-rw- 1 root root 1, 3 Jan  1 00:00 null
crw-r--r-- 1 root root 1, 5 Jan  1 00:00 zero
crw-rw-rw- 1 root root 1, 8 Jan  1 00:00 random
crw-rw-rw- 1 root root 1, 9 Jan  1 00:00 urandom
crw-rw---- 1 root root 4, 64 Jan  1 00:00 ttyS0
```

2. **解读输出格式**

| 字段 | 示例值 | 含义 |
|------|--------|------|
| 文件类型 | `c` / `b` | `c`=字符设备，`b`=块设备 |
| 权限 | `rw-rw-rw-` | 所有者/组/其他的读写权限 |
| 主次设备号 | `5, 1` | 主设备号5，次设备号1 |

3. **确认内核已注册对应驱动**
```bash
cat /proc/devices
```

搜索主设备号。例如看到`5 tty`，说明主设备号5的驱动确实已加载。

4. **功能验证**
```bash
# 测试 null：写入后不应产生任何输出或文件
echo "hello" > /dev/null && echo "null正常"

# 测试 zero：读取10字节应得到全零
dd if=/dev/zero of=/tmp/ten_zeros bs=1 count=10
xxd /tmp/ten_zeros | head
# 期望输出：00000000: 0000 0000 0000 0000 0000 ............
```

💡 **提示**：如果`ls -l`显示设备节点是普通文件（没有`c`或`b`标识，也没有主次设备号），说明创建失败——可能是用`touch`而不是`mknod`创建的。

## 本节总结

| 设备节点 | 主:次 | 作用 | 缺失后果 | 验证命令 |
|----------|-------|------|----------|----------|
| `/dev/console` | 5:1 | 内核与用户交互通道 | 启动无输出、无法单用户调试 | `echo hi > /dev/console` |
| `/dev/null` | 1:3 | 丢弃数据、提供EOF | 脚本重定向失败 | `echo hi > /dev/null` |
| `/dev/zero` | 1:5 | 提供无限零字节流 | 无法清零/生成空文件 | `dd if=/dev/zero of=/tmp/z bs=1 count=4` |
| `/dev/random` | 1:8 | 阻塞式真随机数 | 加密操作卡住 | `head -c 4 /dev/random \| xxd` |
| `/dev/urandom` | 1:9 | 非阻塞伪随机数 | 安全协议无法工作 | `head -c 4 /dev/urandom \| xxd` |
| `/dev/ttyS0` | 4:64 | 串口终端 | 无法串口登录调试 | `stty -F /dev/ttyS0` |
