## 知识点1：gcc的角色——C代码到汇编代码 [B] ~500字

### 1.1 为什么叫gcc"厨师"？

gcc把C源代码（原材料）一步步加工成机器能吃的"汇编代码"（半成品）。就像餐厅做菜有四道工序：

| 工序 | 厨房操作 | gcc阶段 | 输出 |
|------|----------|---------|------|
| 洗菜切菜 | 番茄切块、鸡蛋打散 | **预处理** | `.c` → `.i` |
| 烹饪调味 | 下锅翻炒、加盐加糖 | **编译** | `.i` → `.s` |
| 装盘定型 | 盛入盘中、摆好造型 | **汇编** | `.s` → `.o` |
| 搭配配菜 | 配米饭、配汤上桌 | **链接** | `.o` → 可执行文件 |

**gcc只做前两步**（预处理+编译到汇编），后面的"下锅烧制"（`.s→.o`）交给`as`，"搭配上桌"（`.o→a.out`）交给`ld`。

### 1.2 gcc四阶段流程图

```mermaid

    C -->|"gcc -c<br/>汇编"| D[hello.o<br/>目标文件]

```

### 1.3 动手验证：查看gcc生成的汇编代码

创建测试文件并生成汇编：

```bash
$ cat > hello.c << 'EOF'
#include <stdio.h>
int main(void)
{
    printf("Hello, Embedded Linux!\n");
    return 0;
}
EOF

$ gcc -S hello.c -o hello.s && cat hello.s
```

你会看到类似输出（x86-64平台）：

```asm
    .section    .rodata
.LC0:
    .string "Hello, Embedded Linux!"
    .text
    .globl  main
main:
    pushq   %rbp
    leaq    .LC0(%rip), %rax
    movq    %rax, %rdi
    call    puts@PLT
    movl    $0, %eax
    popq    %rbp
    ret
```

[图2：gcc生成的汇编代码示例截图——展示了如何从C代码翻译成汇编指令]

这就是gcc的核心成果：把人类能读懂的C代码，翻译成CPU能理解的汇编指令。

### 1.4 gcc编译阶段选项速查表

| 选项 | 阶段 | 输出文件 | 作用说明 |
|------|------|----------|----------|
| `-E` | 预处理 | `.i`（标准输出） | 展开`#include`和`#define`，保留注释 |
| `-S` | 编译 | `.s` | **只编译到汇编**，是gcc的核心工作 |
| `-c` | 汇编 | `.o` | 调用`as`汇编器，生成目标文件 |
| （无） | 链接 | `a.out` | 调用`ld`链接器，生成可执行文件 |
| `-o` | 通用 | 自定义 | 指定输出文件名，可以和任何阶段搭配 |

💡 **提示**：`-S`是区分"编译器前端"和"后端"的关键分水岭。gcc到`-S`为止，后面由`as`和`ld`完成。

### 常见错误

⚠️ **错误1**：用`gcc -c hello.c -o hello.s`试图生成汇编文件
> 原因：`-c`会调用汇编器`as`，输出的是`.o`目标文件。想生成汇编代码用`-S`。

⚠️ **错误2**：以为`-o`可以单独使用
> 原因：`-o`只是指定输出文件名，必须配合编译动作（`-E`、`-S`、`-c`或不加选项）。

## 知识点2：交叉gcc的特殊性 [B] ~400字

### 2.1 交叉gcc是什么？

普通gcc是"本地厨师"——在x86电脑上编译x86程序。交叉gcc则是"带着菜谱去外地开分店的厨师"——在x86电脑上编译ARM/RISC-V程序。

交叉gcc命名格式：`<架构>-<厂商>-<操作系统>-gcc`

- `aarch64-linux-gnu-gcc` —— 64位ARM Linux
- `arm-linux-gnueabihf-gcc` —— 32位ARM Linux（硬浮点）
- `riscv64-linux-gnu-gcc` —— 64位RISC-V Linux

### 2.2 查看交叉gcc的"内置配置"

交叉gcc最大的区别：**它在编译gcc自身时就被植入了目标架构的基因**：

```bash
$ aarch64-linux-gnu-gcc -v
...
Target: aarch64-linux-gnu          ← 目标架构已固定！
Configured with: ... --target=aarch64-linux-gnu
...
```

交叉gcc"出厂"时就知道为谁服务，不用每次编译都告诉它目标架构。

### 2.3 嵌入式调优三板斧

交叉gcc知道目标架构，你还可以进一步"精调"：

| 参数 | 作用 | 示例 |
|------|------|------|
| `-march=armv8-a` | 指定指令集架构版本 | 决定可用哪些指令 |
| `-mcpu=cortex-a53` | 指定具体CPU型号 | gcc知道流水线特性，优化调度 |
| `-mfloat-abi=hard` | 指定浮点调用约定 | `hard`=FPU寄存器传参，`soft`=整数寄存器 |

```bash
# ARM 32位：为Cortex-A9（硬浮点）优化
$ arm-linux-gnueabihf-gcc -march=armv7-a -mcpu=cortex-a9 \
    -mfloat-abi=hard -mfpu=vfpv3 hello.c -o hello

# ARM 64位：为Cortex-A53优化
$ aarch64-linux-gnu-gcc -march=armv8-a -mcpu=cortex-a53 \
    hello.c -o hello
```

💡 **提示**：`-mcpu`比`-march`更具体。gcc看到`-mcpu=cortex-a53`，还知道L1缓存大小、流水线深度，能做更聪明的调度。

### 常见错误

🔴 **危险**：32位ARM交叉gcc有`gnueabi`（软浮点）和`gnueabihf`（硬浮点）两个版本，选错会导致浮点运算慢10倍以上。确认目标板有FPU就用`gnueabihf`。

⚠️ **错误3**：交叉编译时直接用本地gcc，生成x86可执行文件，放到ARM板报"Exec format error"。编译前用`which gcc`确认路径。

## 本节总结

| 概念 | 要点 |
|------|------|
| gcc的"厨师"角色 | 把C代码编译成汇编（`.s`），后面不归它管 |
| 四阶段编译 | 预处理→编译→汇编→链接，对应`-E`/`-S`/`-c`/默认 |
| 交叉gcc | 编译时已固定目标架构（`--target=...`），不用每次指定 |
| 嵌入式调参 | `-march`选指令集，`-mcpu`选核，`-mfloat-abi`选浮点方式 |

💡 **一句话记忆**：gcc是切配厨师（C→汇编），交叉gcc是带着外地菜谱的厨师。
