## 知识点1：GCC 头文件搜索路径 [I] ~700字

当你写下一行 `#include <stdio.h>`，GCC 并不会满硬盘乱找。它有一张**固定的搜索顺序表**，按优先级依次查找，找到即止。

### 搜索优先级总览

GCC 的头文件搜索遵循严格的优先级顺序，可以用下面的流程理解：

```mermaid
flowchart TD
    A[遇到 #include 指令] --> B{是 #include "..." 还是 <...>?}
    B -->|双引号 ""| C[1. 当前文件所在目录<br/>2. -I 指定的目录<br/>3. 系统路径]
    B -->|尖括号 <>| D[跳过当前目录<br/>直接按系统顺序查]
    C --> E[查到即停止]
    D --> F[1. -I 指定的目录]
    F --> G[2. 环境变量 C_INCLUDE_PATH]
    G --> H[3. --sysroot/usr/include]
    H --> I[4. 编译器内置系统路径]
    I --> E
    E --> J[编译继续]
    E -->|全部未找到| K[报错：No such file]
    style F fill:#fff3e0
    style H fill:#e1f5fe
    style K fill:#ffcdd2
```

> **图1：GCC 头文件搜索流程图**——双引号和尖括号的搜索策略不同，`-I`优先级最高，sysroot 路径是交叉编译的关键。

### 各优先级详解

| 优先级 | 路径来源 | 说明 | 交叉编译中的特殊性 |
|:------:|----------|------|------------------|
| 1 | `-I/dir` 指定的目录 | 用户自定义路径，优先级最高 | 常用于第三方库头文件 |
| 2 | `C_INCLUDE_PATH` 环境变量 | 通过环境变量追加 | 不推荐，易污染全局环境 |
| 3 | `--sysroot/usr/include` | 交叉编译器的"虚拟根目录" | **交叉编译时取代 `/usr/include`** |
| 4 | 编译器内置系统路径 | 编译器编译时硬编码的路径 | 对交叉编译器指向 sysroot 内部 |

⚠️ **陷阱**：`-I` 的顺序很重要！GCC 按**从左到右**的顺序搜索 `-I` 目录，如果多个目录中有同名头文件，**先找到的那个会被使用**，后面的即使版本更新也会被静默忽略。

### 实操：打印编译器的真实搜索路径

GCC 内部藏得很深的搜索路径，可以用一条命令"诱骗"它吐出来：

```bash
# 对于交叉编译器
arm-linux-gnueabihf-gcc -xc -E -v /dev/null

# 对于本地 GCC
gcc -xc -E -v /dev/null
```

这条命令的含义拆解：
- `-xc`：把输入当作 C 语言处理（即使 `/dev/null` 没有后缀）
- `-E`：只运行预处理阶段（正是找头文件的阶段）
- `-v`：显示详细过程（verbose）
- `/dev/null`：空输入文件，我们只是要触发预处理来"看过程"

在输出中，找到这两段关键内容：

```
#include "..." search starts here:
#include <...> search starts here:
 /usr/lib/gcc-cross/arm-linux-gnueabihf/10/include
 /usr/arm-linux-gnueabihf/include
 /usr/include/arm-linux-gnueabihf
 /usr/include
End of search list.
```

💡 **提示**：交叉编译器的输出**不会**出现 `/usr/include`（宿主机 x86 头文件目录），取而代之的是 `/usr/arm-linux-gnueabihf/include` 或 sysroot 下的路径。如果出现 `/usr/include`，说明你的交叉编译器配置有问题！

### 验证 sysroot 路径是否生效

结合上一节（2.3.1）的知识，可以验证 sysroot 是否被正确纳入搜索：

```bash
SYSROOT=$(arm-linux-gnueabihf-gcc -print-sysroot)
arm-linux-gnueabihf-gcc -xc -E -v /dev/null 2>&1 | grep -A5 "search starts here"

# 确认 sysroot/usr/include 在列表中
ls -d "$SYSROOT/usr/include"
```

## 知识点2：常见"头文件找不到"排错 [I] ~500字

嵌入式开发中，"`fatal error: xxx.h: No such file or directory`"是最让人抓狂的错误之一。下面三个场景覆盖了 90% 的真实案例。

### 场景1：`-I` 忘了加或路径写错

**现象**：
```
fatal error: mylib.h: No such file or directory
   3 | #include "mylib.h"
```

**诊断**：头文件在当前目录或某个子目录里，但你没告诉 GCC 去哪里找。

**修复**：
```bash
# 如果 mylib.h 在 ./include/ 下
gcc main.c -I./include -o main

# 如果 mylib.h 在 ../common/inc/ 下
gcc main.c -I../common/inc -o main
```

⚠️ **陷阱**：`-I` 的顺序很重要！GCC 按**从左到右**的顺序搜索 `-I` 目录，如果多个目录中有同名头文件，**先找到的那个会被使用**，后面的即使版本更新也会被静默忽略。Makefile 中如果路径以变量传入（如 `-I$(INC_DIR)`），请务必确认变量已定义且不为空。

### 场景2：头文件不在 sysroot 里

**现象**：交叉编译时提示找不到标准头文件，如 `stdio.h`。

**诊断**：你的交叉编译器 sysroot 是**最小化**的，可能只包含基础 libc 头文件。第三方库（如 `libxml2`、`openssl`）的头文件需要额外安装到 sysroot 中。

**修复**：
```bash
# 查看 sysroot 里到底有哪些头文件
SYSROOT=$(arm-linux-gnueabihf-gcc -print-sysroot)
ls "$SYSROOT/usr/include/" | grep -i xml
# 如果为空，说明 libxml2 头文件没装
```

💡 **提示**：在嵌入式开发中，正确的做法是把第三方库**交叉编译安装到 sysroot**，而不是用 `-I` 指向源码目录。这样链接阶段也能找到对应的库文件。

### 场景3：交叉编译器偷偷找了 PC 的头文件 [🔴 经典陷阱]

**现象**：程序编译通过了，但在目标板上运行异常；或者编译时提示某些结构体/宏定义不存在，但你知道目标系统应该有。

**诊断**：编译器混入了**宿主机的头文件**！这通常发生在以下情况：

1. 你手动加了 `-I/usr/include`
2. 编译器没有正确配置 sysroot，回退到了宿主机路径
3. 某个 Makefile 硬编码了绝对路径

**验证**：
```bash
# 打印实际搜索路径，检查是否有 /usr/include
arm-linux-gnueabihf-gcc -xc -E -v /dev/null 2>&1 | grep "/usr/include"

# 如果有输出，说明确实在搜索 PC 的头文件！
```

**修复**：
- 删除 Makefile 中硬编码的 `-I/usr/include`
- 用 `--sysroot=` 强制指定正确的 sysroot
- 用 `-nostdinc` 禁用所有标准路径，只保留 `-I` 指定的路径（高级用法）

🔴 **危险**：用 x86 的头文件编译 ARM 程序是**静默杀手**——它不会报错，但会导致结构体大小、字节序、宏定义全部错乱，最终产物在板子上崩溃或行为异常。务必用 `-v` 定期检查搜索路径！

## 本节总结

| 场景 | 错误现象 | 根因 | 排查命令 | 修复方法 |
|------|----------|------|----------|----------|
| 自定义头文件找不到 | `fatal error: mylib.h: ...` | 未加 `-I` 或路径错误 | `ls -R \| grep mylib.h` | `gcc -I/path/to/include` |
| 系统头文件找不到 | 缺少 `stdio.h` 等标准头 | sysroot 不完整 | `gcc -print-sysroot` 后检查 | 补装库到 sysroot |
| 运行时异常/编译通过但行为怪 | 结构体大小不符、字节序错误 | 混入了 PC 的 `/usr/include` | `gcc -xc -E -v \| grep include` | 移除 `-I/usr/include`，指定 `--sysroot` |
| `-I` 顺序导致旧版本头文件被使用 | 函数声明不匹配 | 多个 `-I` 目录有同名文件 | `gcc -H` 显示实际 include 的文件 | 调整 `-I` 从左到右的顺序 |
