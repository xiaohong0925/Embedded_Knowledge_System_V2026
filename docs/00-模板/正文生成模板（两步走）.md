# 正文生成模板（两步走：先地图，后填空）

> 用于指导 AI 生成《嵌入式 Linux 知识体系（V2025）》教材正文。
> 核心设计：先输出结构地图，再逐方向填充内容，用户直接复制拼接。

---

## 使用流程（两步走）

### 第一步：生成本节地图

将下方"第一步提示词"完整复制给 AI，AI 只输出**纯结构**（无正文）。

你将得到：
- 文件开头的导语 + 本节地图表格
- 每个方向的"小点标题 + 必须包含 + 禁止写入"
- 本节自检清单

**操作**：直接复制 AI 输出，作为 `.md` 文件的**开头部分**。

---

### 第二步：逐方向填空

将下方"第二步提示词"完整复制给 AI，每次只填**一个方向**（2~3 个小点）。

你将得到：
- 带 `## 方向X` 标题的完整正文
- 每个小点带 `### 小点X` 标题
- 方向开头有导语，结尾有过渡句
- 可直接拼接在地图后面

**操作**：复制 AI 输出，粘贴在地图后方。填完方向1，再填方向2，直到本节完成。

## 第二步提示词（逐方向填空）

```text
[角色] 你是《嵌入式 Linux 知识体系（V2025）》的教材作者。
[风格] 严谨教科书 + 平易近人。定义精确、推导严密、类比为盐、场景具体。

[任务] 填充 [ 中断基础与硬件抽象 方向1：中断的工程背景与核心概念 ] 的完整正文。
[深度] [___]

[核心原则]
1. 推导优先于结论：每个机制必须从"为什么需要"开始，回答工程痛点。
2. 代码必须带读：shell 命令需注明适用场景，逐字段解读输出。
3. 深度优先：字数服从"讲清楚"，不硬性压缩。

[五条红线]
1. 严禁跳步：禁止从类比直接跳到源码，概念→代码必须有推导桥梁。
2. 严禁抽象定义堆砌：禁止连续 3 段"xxx 是 yyy"，每个定义配类比或场景。
3. 严禁工业化流水线：禁止连续 2 个小节使用相同叙述节奏。
4. 严禁认知阶段标题：禁止出现 ①②③④⑤ 或 "### 类比引入" 等阶段切分。
5. 严禁未定义术语前置：术语首次出现必须加括号解释，禁止一句内倾泻多个术语。

[语言风格]
- 开头：一句话对比/场景引出，立刻切定义
- 类比：盐不是主食。每小点最多 1 处，定义之后，最多 2 句话，用 > 💡 引用块
- 推导：问题 → 答案，用"天花板""陷阱""矛盾"等工程比喻，场景具体到数字
- 过渡：逻辑连接词（"由于""因此"），禁止"接下来我们来看"
- 结尾：钩子收束，预告下一个问题（最后一个小点除外）

[排版规范]
- 认知切换时换段（场景→定义、原理→代码、文字→表格），空行分隔
- 禁止用 <br> 标签，段落分隔必须用空行
- 单段完成一个完整意思，禁止把一句话硬拆两段
- 禁止连续 3 段超过 6 行（文字墙）
- 术语首次出现：**术语**（英文全称，中文解释）
- 重点概念：**加粗**，技术术语：`反引号`，警告：> ⚠️ 引用块

[输出格式]
直接输出可拼接的 Markdown 正文，包含：
- ## 方向X：... [深度]（方向标题）
- 方向导语（1-2 句承上启下）
- ### 小点1：...（小点标题）
- ...（正文）
- ### 小点2：...
- ...
- 方向结尾过渡句（衔接下一个方向）

[内容要求]
（根据地图填写，如：）
- 小点1：...
- 小点2：...
- 小点3：...

[类型边界]
（根据章节类型填写，如：）
- 本节属于类型 A（内核机制），禁止写驱动 API 实现细节（留给第 5 章）
- 涉及驱动 API 时标注"驱动编程细节详见第 X 章"
```

## 拼接指南

### 文件结构

```text
01-中断基础与硬件抽象.md
├── 第一步输出（地图）
│   ├── # 标题 + 导语
│   ├── ## 本节地图
│   ├── ## 方向1 结构（小点标题 + 必须包含 + 禁止写入）
│   └── ## 方向2 结构...
│
├── 第二步输出（方向1 正文）
│   ├── ## 方向1：... [B→I]
│   ├── ### 小点1：...（正文）
│   ├── ### 小点2：...（正文）
│   └── ### 小点3：...（正文 + 过渡句）
│
├── 第二步输出（方向2 正文）
│   ├── ## 方向2：... [I]
│   ├── ### 小点1：...（正文）
│   ├── ### 小点2：...（正文）
│   └── ### 小点3：...（正文 + 过渡句）
│
└── 自检清单（地图自带，写完后逐项打勾）
```

### 操作步骤

1. **新建文件**：`01-中断基础与硬件抽象.md`
2. **粘贴地图**：复制第一步输出，作为文件开头
3. **粘贴方向1**：复制第二步（方向1）输出，贴在地图后面
4. **粘贴方向2**：复制第二步（方向2）输出，贴在方向1后面
5. **完成**：push 到 GitHub

### 关键原则

- **不要手动改格式**：AI 输出已带正确的 `##` `###` 层级，直接粘贴
- **不要手动加内容**：如果 AI 漏了某个小点，重新生成该方向，不要自己补
- **不要合并方向**：每个方向独立生成，独立粘贴，保持质量

### 小点1：为什么需要中断——轮询的困境与事件驱动的必然

在嵌入式系统中，CPU 与外设交换数据只有两种基本方式：要么 CPU 主动问，要么外设主动喊。

最原始的做法是**轮询**（Polling，主动查询模型）——CPU 以固定周期反复读取外设的状态寄存器，直到发现数据就绪。

轮询的第一道工程天花板是 CPU 空转与功耗失控。

假设一个工业传感器每 1ms 产生一次有效数据，CPU 为了不错过它，必须以毫秒级间隔查询状态寄存器。这意味着即使外设 99% 的时间处于空闲，CPU 也无法进入低功耗状态，空转功耗直接吃掉电池续航。如果不引入中断，手持设备或物联网节点的待机时间将从数周暴跌到数小时。

## 自检清单（模板级）

使用本模板时，确认以下事项：

- [ ] 第一步提示词是否完整复制给 AI（包含角色 + 风格 + 格式要求）？
