# 嵌入式Linux知识体系 层级命名与呈现统一规范
> 版本：V2026.05.04 双人协作最终版
> 生效日期：2026-05-04
> 适用范围：双人协作内容编写、mkdocs部署渲染
> 核心对齐：层级命名严格遵循「1级模块→2级大章→3级章节→4级小节→5级知识点」的五级体系，兼顾灵活性与风格统一性

## 一、五级层级命名与文件结构强制规范
### 1.1 层级总表（100%对齐约定命名）
| 层级序号 | 官方固定名称 | 文件系统载体 | 内容标题规则 | 编号规则 | 核心强制要求 |
|----------|--------------|--------------|--------------|----------|--------------|
| 1级 | 模块（Module） | `docs/0N-模块名/` 一级文件夹 | 模块概览 `README.md` 内唯一标题 `# 模块名` | 2位数字前缀，格式 `0N-模块名`，个位数前补0 | 必须包含根README.md，作为模块总览，明确整体难度、学习路径 |
| 2级 | 大章（Chapter） | `NN-大章名/` 二级文件夹（模块内的子文件夹） | 大章概览 `README.md` 内唯一标题 `# 大章名` | 2位数字前缀，格式 `NN-大章名`，个位数前补0；`00-`固定为「模块学习路径与前置基础」大章 | 所有大章统一用文件夹承载，无论内容多少，禁止单md文件作为大章；必须包含大章内README.md |
| 3级 | 章节（Section） | `NN-章节名.md` 单文件（大章文件夹内的md文件） | 文件内唯一一级标题 `# 章节名` | 2位数字前缀，格式 `NN-章节名.md`，个位数前补0 | 一个章节对应一个独立md文件，禁止多章节内容合并到一个文件；正文内标题无任何序号 |
| 4级 | 小节（Subsection） | 章节md文件内的正文结构 | 正文内 `## <strong>小节名</strong>` | 正文内无任何序号 | 小节标题同行末尾必须添加极简模式BIEM难度徽章；小节之间用 `---` 水平线分隔 |
| 5级 | 知识点（Point） | 章节md文件内的正文结构 | 正文内 `### <strong>知识点名</strong>` | 正文内无任何序号 | 禁止使用 `####` 及更低级标题，知识点下用加粗子标题/列表承载细分内容；继承上级小节难度，不单独标记 |

---

### 1.2 各层级详细规则与落地示例
#### 1级：模块（Module）
- **核心定位**：全书顶层知识板块，对应10大核心领域，是知识体系的一级导航入口
- **文件系统规则**：`docs/` 下的一级文件夹，命名固定为 `0N-模块名`，2位数字编号
- **强制内容**：文件夹根目录必须包含 `README.md`，内容包含模块定位、整体难度范围、学习路径、前置基础要求
- **落地示例**：
  ```
  docs/
  ├── 01-硬件层/
  │   └── README.md
  ├── 02-Bootloader与系统启动/
  │   └── README.md
  ├── 03-Linux内核深度解析/
  │   └── README.md
  └── ... 其余7个模块，统一格式
  ```

#### 2级：大章（Chapter）
- **核心定位**：模块下的核心知识单元，对应单领域的完整主题，是内容组织的核心载体
- **文件系统规则**：模块文件夹内的二级子文件夹，命名固定为 `NN-大章名`，2位数字编号
- **强制规则**：
  1.  每个模块下，`00-模块学习路径与前置基础` 为固定大章文件夹，作为模块整体学习指南
  2.  每个大章文件夹内，必须包含 `README.md`，内容包含大章定位、难度范围、学习目标、章节导航、与其他模块的边界说明
- **落地示例（硬件层模块内的大章）**：
  ```
  docs/01-硬件层/
  ├── README.md （1级模块总览）
  ├── 00-模块学习路径与前置基础/ （2级大章文件夹）
  │   └── README.md
  ├── 01-原理图阅读与板级硬件基础/ （2级大章文件夹）
  │   └── README.md
  ├── 02-CPU架构/ （2级大章文件夹）
  │   └── README.md
  ├── 03-时钟电源与复位管理/ （2级大章文件夹）
  │   └── README.md
  └── ... 其余大章，统一格式
  ```

#### 3级：章节（Section）
- **核心定位**：大章下的逻辑分支，单大章内容的模块化拆分，是内容写作的独立文件载体
- **文件系统规则**：大章文件夹内的独立md文件，命名固定为 `NN-章节名.md`，2位数字编号
- **强制规则**：
  1.  一个章节对应一个独立md文件，禁止多章节内容合并到一个文件
  2.  文件内唯一一级标题为 `# 章节名`，正文内无任何序号
  3.  章节开头必须用引用块标注本节难度、前置基础、关联内容
- **落地示例（CPU架构大章内的章节）**：
  ```
  docs/01-硬件层/02-CPU架构/
  ├── README.md （2级大章总览）
  ├── 01-核心架构基础认知.md
  ├── 02-ARMv7-v8架构深度解析.md
  ├── 03-RISC-V架构核心特性.md
  ├── 04-MIPS架构嵌入式适配.md
  └── 05-嵌入式CPU架构选型实战.md
  ```

#### 4级：小节（Subsection）
- **核心定位**：章节下的最小可独立讲解单元，是知识点的核心承载
- **正文规则**：用 `## <strong>小节名</strong>`，正文内无任何序号
- **强制规则**：
  1.  小节标题**同行末尾**必须添加极简模式BIEM难度徽章
  2.  小节之间必须用 `---` 水平线分隔
  3.  一小节仅讲解一个独立的小知识点，保证内容独立性
- **落地示例**：
  ```markdown
  # ARMv7/v8架构深度解析
  > 📊 本节难度：<span class="badge-i">I</span> → <span class="badge-e">E</span>
  > 📚 前置基础：核心架构基础认知
  ---

  ## <strong>ARMv8架构核心特权等级</strong> <span class="badge-i">I</span>
  正文内容...<br>

  ---

  ## <strong>ARMv8 MMU与缓存硬件机制</strong> <span class="badge-e">E</span>
  正文内容...<br>
  ```

#### 5级：知识点（Point）
- **核心定位**：小节下的最小内容单元，承载具体的原理、公式、步骤、场景拆解
- **正文规则**：用 `### <strong>知识点名</strong>`，禁止使用 `####` 及更低级标题
- **强制规则**：
  1.  每个知识点只讲一个细节点，禁止一个知识点内混杂多个内容
  2.  继承上级小节的难度等级，不单独标记难度
  3.  细分内容用有序/无序列表、加粗子标题承载，禁止再拆分子层级
- **落地示例**：
  ```markdown
  ## <strong>ARMv8架构核心特权等级</strong> <span class="badge-i">I</span>
  <span class="red">ARMv8特权等级</span>是ARMv8架构的核心安全隔离机制。<br>

  ### <strong>EL0 用户级</strong>
  对应Linux用户态，无硬件特权操作权限，仅可访问受限的硬件资源<br>

  ### <strong>EL1 内核级</strong>
  对应Linux内核态，可操作核心硬件寄存器，管理内存、中断等核心资源<br>
  ```

## 二、mkdocs页面呈现硬标准
### 2.1 排版红线（违反即调整）
| 元素 | 强制规则 |
|------|----------|
| 段落长度 | ≤4行，超出必须拆段 |
| 行内换行 | 用 `<br>` 标签 |
| 段落/小节分隔 | 用 `---` 水平线 |
| 标题序号 | 正文内所有标题**绝对不加序号**，编号仅出现在文件/文件夹命名中 |
| 章节标题 | 单文件内唯一一级标题 `# 标题`，无序号 |
| 小节标题 | `## <strong>标题</strong>`，无序号 |
| 知识点标题 | `### <strong>标题</strong>`，无序号 |
| 子标题 | 禁止使用 `####` 及更低级标题，用 `<strong>子标题</strong>` 承载 |
| 表格 | 纯Markdown格式，**严禁包含任何 `<span>` 标签** |
| 代码块 | 必须带语言标识，内部用真实换行，禁止用 `<br>` |
| 框图 | 所有ASCII框图**必须替换为Mermaid** |

---

### 2.2 四色标记系统（严格执行）
```html
<span class="red">核心概念</span>     ← 段首标注，每段最多1次
<span class="green">技术术语/API</span>  ← 术语首次出现时标注
<span class="blue">关键结论/金句</span>  ← 定义句、过渡句、核心结论
<span class="orange">列表项标题</span>    ← 有序列表每个item的开头标题
```
- 表格内、代码块内**严禁任何 `<span>` 标签**
- 每段仅可标注1个红色核心概念，禁止多红标

## 三、双人协作版 引用块专属规范
### 3.1 3条核心必守红线（绝对不能碰）
1.  **正文核心内容绝对不能用引用块承载**：核心定义、原理讲解、流程步骤必须用普通段落+`<br>`，引用块仅用来放「非核心的补充、提示、元信息」，不喧宾夺主。
2.  **绝对不打断正文认知流**：引用块只能放在两个固定位置——「标题之后、正文/水平线之前」，或者「完整段落的末尾」，禁止在正文段落中间穿插引用块。
3.  **禁止嵌套、禁止放表格/代码块**：嵌套引用块、引用块内放表格/代码块会导致mkdocs移动端渲染异常，这类内容用原生Markdown格式承载。

---

### 3.2 3类放开的推荐用法
#### 用法1：元信息类（固定位置：章节/小节开头，核心主场景）
**用途**：统一承载章节/小节的元信息，全文档格式100%对齐，读者打开页面一眼就能获取关键信息。
**固定位置**：标题之后、`---`水平线之前
**示例**：
```markdown
# U-Boot 移植与定制
> 📊 难度：<span class="badge-b">B</span> → <span class="badge-e">E</span>
> 🎯 定位：讲解U-Boot完整移植流程、定制化开发、嵌入式场景适配
> 📚 前置基础：硬件层DDR/存储原理、Bootloader核心认知
> 🔗 关联章节：内核移植见04模块，安全启动见08模块
---
```
```markdown
## <strong>DDR训练与宽温适配</strong> <span class="badge-e">E</span>
> 📚 前置依赖：硬件层DDR内存硬件原理章节
> ⚠️ 适用范围：仅工业级宽温设备，消费级设备可跳过
---
```

#### 用法2：提示补充类（固定位置：完整段落末尾）
**用途**：放避坑提醒、补充说明、实战踩坑经验、额外扩展内容，不打断正文核心认知流，比单纯的加粗/上色视觉冲击力更强。
**示例**：
```markdown
<span class="red">Bootloader</span>生命周期极短，仅在上电后几百毫秒到几秒内运行。<br>
一旦将控制权交给内核，Bootloader完全退出运行，占用内存被内核回收覆盖。<br>

> ⚠️ 【实战避坑】Bootloader退出后，其运行时的所有临时数据、栈空间都会被内核完全覆盖，无法通过内核回溯Bootloader的运行日志，启动调试必须提前开启串口全量日志输出。
{: .warning }
```

#### 用法3：核心结论类（固定位置：小节开头/末尾）
**用途**：放小节的核心金句、关键结论、底层规律，帮读者快速抓住小节核心，完全贴合「每段1个核心结论」的规则。
**示例**：
```markdown
## U-Boot 与其他Bootloader的核心优势
---

> <span class="blue">核心结论：对于99%的嵌入式Linux商用场景，U-Boot是唯一最优解，其他Bootloader仅适用于小众特殊需求场景。</span>
{: .conclusion }

正文内容对比...<br>
```

## <strong>ARMv8架构核心特权等级</strong> <span class="badge-i">I</span>
<span class="red">ARMv8特权等级</span>是ARMv8架构的核心安全隔离机制。<br>

<span class="blue">ARMv8架构通过特权等级实现了硬件级的安全隔离，是嵌入式系统安全启动的核心基础。</span><br>

### <strong>EL0 用户级</strong>
对应Linux用户态，无硬件特权操作权限，仅可访问受限的硬件资源<br>

### <strong>EL1 内核级</strong>
对应Linux内核态，可操作核心硬件寄存器，管理内存、中断等核心资源<br>

### <strong>EL2 虚拟化级</strong>
