# VFS核心原理与数据结构

> 📊 **本节难度等级：** <span class="badge-ie">**IE级**</span>

---

### <strong>VFS的抽象能力本质是通过四个核心数据结构（对象）实现的——这四个对象构成了VFS的“骨架”，定义了文件系统的全局属性、文件物理特征、目录映射关系和进程访问实例。
嵌入式场景中，这四个对象的内存占用和效率直接影响系统性能，因此需同时掌握“原理定义”和“嵌入式精简策略”。</strong>


### <strong>1. super_block：文件系统全局元数据</strong>

核心定义：super_block（超级块）是单个文件系统的全局描述符

——当一个文件系统（如ext4、tmpfs）被挂载后，内核会在内存中创建一个super_block实例，存储该文件系统的全局信息，直至文件系统被卸载。
整个文件系统的所有操作，都需要通过super_block关联到具体的文件系统驱动。

核心字段解析（基于Linux 5.10内核，仅列嵌入式关键字段）：
```c
struct super_block {
    struct list_head    s_list;        // 所有super_block的链表（内核统一管理）
    const struct super_operations    *s_op;  // 超级块操作集（如卸载、同步）
    struct dentry       *s_root;       // 文件系统的根目录dentry（关键！挂载点对应此目录）
    struct block_device *s_bdev;       // 关联的块设备（如eMMC分区/dev/mmcblk0p2）
    unsigned long       s_blocksize;   // 文件系统块大小（嵌入式常见4KB/8KB，影响IO效率）
    const char          *s_type;       // 文件系统类型（如"ext4"、"tmpfs"）
    atomic_t            s_active;      // 引用计数（确保未使用时才卸载）
    struct list_head    s_inodes;      // 该文件系统的所有inode链表
    struct shrinker     *s_shrink;     // 内存紧张时的缓存回收回调（嵌入式关键）
    // 其他字段省略...
};
```
- 关键字段作用：`s_root`是文件系统挂载的“锚点”
——挂载时内核将`s_root`与挂载点目录的dentry关联，使应用能通过路径访问；
    `s_blocksize`需与Flash的物理擦除块大小匹配（如<span class="green">NAND</span> Flash擦除块为128KB时，块大小设为4KB可减少读写浪费）。

嵌入式精简策略：  
super_block占用内存虽小（约几百字节），但嵌入式设备常挂载多个文件系统（ext4+tmpfs+<span class="green">procfs</span>等），需通过内核配置和使用规范减少冗余：
1.  关闭文件系统冗余特性：
编译内核时关闭ext4的“日志校验和”（`CONFIG_EXT4_FS_SECURITY`）、tmpfs的“配额管理”（`CONFIG_TMPFS_QUOTA`）等非必需功能，减少super_block字段占用；
2.  减少无效挂载：
避免挂载未使用的<span class="red">虚拟文件系统</span>（如cgroupfs、<span class="green">sysfs</span>子目录），每个挂载的文件系统都会创建super_block，多余挂载会浪费内存；
3.  利用共享超级块：
对相同设备的只读挂载（如多个进程挂载同一ext4分区为只读），内核会复用super_block，避免重复创建。<br>

### <strong>2. inode：文件物理属性（权限、大小、数据块指针）及嵌入式inode数量优化</strong>

核心定义：inode（索引节点）是文件物理属性的唯一载体
——每个文件（包括目录、设备文件）对应一个唯一的inode，存储文件的权限、大小、创建时间、数据块地址等“元数据”，
但不存储文件名（文件名由dentry管理）。

核心字段解析（嵌入式关键字段）：
```c
struct inode {
    umode_t             i_mode;        // 文件权限（如0644表示所有者可读可写，其他只读）
    loff_t              i_size;        // 文件大小（字节）
    struct inode_operations  *i_op;    // inode操作集（如创建、删除子文件）
    const struct file_operations  *i_fop;  // 默认文件操作集（关联到具体文件系统）
    struct list_head    i_dentry;      // 关联的所有dentry（一个inode可对应多个文件名）
    struct address_space *i_mapping;   // 页缓存映射（关联文件数据块与内存页）
    blkcnt_t            i_blocks;      // 文件占用的磁盘块数
    struct block_device *i_bdev;       // 关联的块设备（如eMMC分区）
    // 其他字段省略...
};
```
- 关键区分：inode与文件的关系是“一对一”，但与文件名是“一对多”
——比如通过`ln /a/file /b/link`创建硬链接后，两个文件名对应同一个inode，修改其中一个文件内容，另一个会同步变化。

嵌入式inode数量优化：  
嵌入式设备的存储（如<span class="green">eMMC</span>、NOR Flash）容量有限，而每个inode约占用128-512字节，若inode数量过多会浪费存储空间；
若数量不足，会导致“磁盘有空间但无法创建文件”的问题，需针对性优化：
1.  格式化时指定inode数量：
通过`mkfs.ext4`的`-i`参数设置“每多少字节分配一个inode”，公式为“inode总数≈分区容量/(-i参数值)”。
例如8GB <span class="green">eMMC</span>分区，若存储大量小文件（如日志文件），可设置`mkfs.ext4 -i 4096 /dev/mmcblk0p2`（每4KB分配一个inode，总inode数约200万）；若存储大文件（如固件镜像），可设置`-i 32768`（每32KB分配一个inode，总inode数约25万）；
2.  动态调整inode缓存：
通过内核参数`/proc/sys/vm/vfs_cache_pressure`调整inode缓存回收优先级
——嵌入式内存紧张时，可将值设为150（默认100），提升inode缓存的回收效率；
3.  选择合适的文件系统：
ubifs、f2fs等嵌入式专用文件系统支持“动态inode分配”，无需在格式化时固定数量，更适配存储容量较小的场景。<br>

### <strong>3. dentry：目录项缓存（文件名→inode映射）及LRU淘汰机制</strong>

核心定义：dentry（目录项）是文件名与inode的映射载体
——用于解析文件路径（如`/a/b/c`），每个目录或文件都会对应一个dentry，形成树形结构。
同时dentry也是<span class="red">VFS</span>缓存的核心，通过缓存常用路径的dentry，避免每次路径解析都读取磁盘。

核心字段解析（缓存相关关键字段）：
```c
struct dentry {
    struct inode        *d_inode;      // 关联的inode（文件名→inode的映射核心）
    struct dentry       *d_parent;     // 父目录dentry（构成路径树结构）
    struct qstr         d_name;        // 文件名（如"c"是路径/a/b/c的最后一个dentry名称）
    struct list_head    d_hash;        // 哈希表链表（用于快速查找dentry）
    struct list_head    d_lru;         // LRU链表（缓存淘汰时使用）
    atomic_t            d_count;       // 引用计数（被引用时不被淘汰）
    const struct dentry_operations *d_op;  // dentry操作集
    // 其他字段省略...
};
```
- 核心作用：路径解析的“加速器”
——当执行`cat /a/b/c`时，内核会从根目录dentry（`/`）开始，依次查找`a`、`b`、`c`的dentry，
若dentry缓存命中，直接通过`d_inode`获取inode，无需读取磁盘；若未命中，才会从磁盘读取目录数据并创建dentry。

LRU淘汰机制：  
dentry缓存占用的内存属于“可回收内存”，
当系统内存紧张时，内核通过LRU（Least Recently Used，最近最少使用）算法淘汰不常用的dentry，流程如下：
1.  缓存组织：所有未被引用的dentry通过`d_lru`链表串联，分为“活跃链表”和“非活跃链表”；
2.  访问更新：当dentry被访问时，从非活跃链表移至活跃链表；
3.  淘汰触发：内存紧张时，内核先扫描非活跃链表，淘汰最久未被访问的dentry；若仍不足，再淘汰活跃链表中久未访问的dentry；
4.  引用保护：若dentry的`d_count`（引用计数）大于0（如进程正打开该文件），则不会被淘汰。

嵌入式优化：
通过`/proc/sys/vm/dentry_cache_limit`（部分内核版本支持）设置dentry缓存的最大内存占用，
例如嵌入式设备内存为512MB时，可设为`echo 33554432 > /proc/sys/vm/dentry_cache_limit`（限制缓存不超过32MB）。<br>

### <strong>4. file：进程打开的文件实例（文件指针、访问模式、私有数据）</strong>

核心定义：file（文件对象）是进程打开文件后的“会话实例”
——当进程执行`open`命令打开一个文件时，内核会创建一个file对象，存储该进程对文件的访问状态（如文件指针位置、访问模式），进程关闭文件后，file对象被销毁。

核心字段解析（进程访问相关关键字段）：
```c
struct file {
    union {
        struct llist_node    fu_llist;  // 文件对象链表
        struct rcu_head     fu_rcuhead; // RCU回收头
    } f_u;
    struct path         f_path;        // 关联的dentry和vfsmount
    const struct file_operations  *f_op;  // 该文件的操作集（如read/write）
    loff_t              f_pos;         // 文件指针位置（下次读写的偏移量）
    umode_t             f_mode;        // 访问模式（如O_RDONLY只读、O_WRONLY只写）
    struct file_ra_state    f_ra;      // 预读状态（提升连续读写效率）
    void                *private_data; // 私有数据（驱动可存储设备上下文）
    // 其他字段省略...
};
```
- 关键区分：inode与file的关系是“一对多”
——一个文件（对应一个inode）可被多个进程打开，每个进程对应一个独立的file对象，例如两个进程同时打开`/etc/config.txt`，会创建两个file对象，各自的`f_pos`（文件指针）独立，修改不会相互影响。

嵌入式场景特点：  
嵌入式设备常存在“多进程共享硬件设备文件”的场景（如多个进程访问同一串口`/dev/ttyS0`），
file对象的`private_data`字段是核心适配点
——驱动程序可将设备的硬件上下文（如串口波特率、校验位配置）存储在`private_data`中，每个file对象对应独立的上下文，实现多进程安全访问。<br>

### <strong>四大核心对象关联图示
四个对象通过指针形成“全局→局部→进程实例”的层级关联，关系如下：</strong>

- 关联逻辑总结：  
  1.  一个super_block对应一个文件系统，包含该文件系统的所有inode和根dentry；  
  2.  一个inode可对应多个dentry（硬链接场景），一个dentry对应一个inode；  
  3.  一个dentry可被多个file对象关联（多进程打开同一文件），每个file对象对应一个进程的访问状态。

super_block（文件系统全局）
    ↓ s_root
dentry树（目录路径缓存）
    ├── dentry(根目录)
    │   └── d_child → dentry(目录a)
    │       └── d_child → dentry(文件1) ←┐
    │                                         │
    └── dentry(文件1的硬链接) ←──────┤
                                               │
     d_inode ↖────────────────┘
              inode(文件1的元数据) ←──────┐
                                                 │
            进程1 → file对象1(f_pos=100) → f_path → dentry(文件1)
            进程2 → file对象2(f_pos=200) → f_path → dentry(文件1)
简化为三句话：
- `dentry` 负责找路（路径名→inode映射，支持硬链接）
- `inode` 负责存数据（文件属性+数据块位置）
- `file` 负责记录读写到哪了（每个进程打开文件时创建，互不影响）
