### <strong>E级核心目标：掌握“编译阶段→运行阶段”的基础排错逻辑，无需深入内核原理，通过工具与日志即可定位80%的入门级问题。</strong>


### <strong>1. DTS语法错误：dtc编译报错实战排查</strong>

DTS语法错误是研发初期最高频问题，`dtc`（<span class="red">设备树</span>编译器）会输出明确的报错信息，
核心是“定位报错行→匹配错误类型→针对性修正”。

以下为5类高频错误及解决方案：
1. 节点未闭合错误
- dtc报错信息：`Error: test.dts:20.1: syntax error, unexpected $end, expecting '}'`
- 错误原因：第20行节点缺少闭合符号`}`，如`led@0 { compatible = "gpio-leds"; `未结尾
- 解决步骤：
  1. 定位报错行附近的节点（第20行前后）
  2. 检查节点定义的`{`与`}`数量是否一致
  3. 用`vim`的`%`键匹配括号位置

2. 属性值格式错误
- dtc报错信息：`Error: test.dts:25.3: invalid property value (must be <...> for array)`
- 错误原因：reg、interrupts等数组类型属性未用`<>`包裹，如`reg = 0x12345678`
- 解决步骤：
  1. 确认属性类型（数组型需`<>`，字符串型需`""`）
  2. 修正格式为`reg = <0x12345678 0x100>`

3. 引用未定义节点
- dtc报错信息：`Error: test.dts:30.3: reference to non-existent node or label "vcc3v3"`
- 错误原因：属性中引用的电源节点`&vcc3v3`未在DTS/DTSI中定义
- 解决步骤：
  1. 检查是否遗漏包含电源节点的DTSI（如`#include "power.dtsi"`）
  2. 确认节点标签`vcc3v3:`是否存在

4. 关键字拼写错误
- dtc报错信息：`Error: test.dts:35.3: unknown property 'compatibe' in node /soc/led@0`
- 错误原因：核心属性`compatible`拼写错误为`compatibe`
- 解决步骤：
  1. 对照常见属性列表（compatible、status、reg等）修正拼写
  2. 用IDE（如VS Code+<span class="red">设备树</span>插件）开启语法提示

5. 节点地址重复
- dtc报错信息：`Warning: test.dts:40.3: duplicate unit address 0x0 in /soc`
- 错误原因：同一父节点下存在两个地址为0x0的子节点（如`led@0`和`button@0`）
- 解决步骤：
  1. 修改子节点地址为唯一值（如`button@1`）
  2. 确认地址与硬件原理图一致

E级高效排错技巧
编译时添加参数显示警告信息，提前规避潜在问题：
```bash
dtc -I dts -O dtb -o test.dtb test.dts -W no-unit_address_duplicate
```

`-W`参数可显示警告信息（如属性冗余），帮助发现非致命但可能影响稳定性的问题。<br>

### <strong>2. 节点匹配失败：驱动无设备（E级实战）</strong>

节点匹配失败表现为“驱动加载成功但无对应设备”，
核心原因是`compatible`属性与驱动`of_match_table`不匹配，排查流程如下（以4G模块EC20为例）：

1. 第一步：查看驱动加载与匹配日志
`# 1. 查看驱动是否加载
lsmod | grep ec20  # 输出ec20 16384 0 说明加载成功
# 2. 查看匹配失败日志（关键）
<span class="green">dmesg</span> | grep -i "of: platform: no matching driver"
# 报错示例：[    1.234567] OF: platform: no matching driver for '/soc/ec20@0'
# 说明节点存在但未匹配到驱动`

2. 第二步：验证<span class="red">设备树</span>节点信息
`# 挂载<span class="green">debugfs</span>查看节点实际解析结果（复用前文工具，不扩展原理）
mount -t <span class="green">debugfs</span> none /mnt/debug
cat /mnt/debug/device-tree/soc/ec20@0/compatible
# 输出示例：quectel,ec20-v2 （实际为compatible属性值）`

3. 第三步：核对驱动匹配表
`// 驱动代码中的of_match_table（核心匹配依据）
static const struct of_device_id ec20_of_match[] = {
    { .compatible = "quectel,ec20" },  // 驱动只匹配该值
    { /* Sentinel */ }
};
MODULE_DEVICE_TABLE(of, ec20_of_match);

# 问题定位：节点compatible为"quectel,ec20-v2"，与驱动不匹配
# 解决方案：1. 修改设备树为"quectel,ec20"；2. 驱动添加"quectel,ec20-v2"兼容项`

4. 第四步：验证匹配结果
`# 修正后重启，查看设备是否正常识别
ls /sys/bus/platform/devices/ec20@0  # 存在该目录说明匹配成功
<span class="green">dmesg</span> | grep "ec20: probe successful"  # 驱动probe成功日志`<br>

### <strong>E级侧重“硬件属性配置→实际运行异常”的排查，核心覆盖GPIO和中断两大高频适配场景，结合硬件原理图与工具验证，定位“设备树与硬件不匹配”问题。</strong>


### <strong>1. GPIO引脚配置错误：设备无响应（实战案例）</strong>

场景：LED灯通过<span class="green">GPIO</span>0_17控制，<span class="red">设备树</span>配置后灯不亮，排查步骤：

1. 核对<span class="red">设备树</span><span class="green">GPIO</span>配置
`// <span class="red">设备树</span>LED节点
led_red: led@0 {
    compatible = "gpio-leds";
    gpios = <&gpio0 17 <span class="green">GPIO</span>_ACTIVE_HIGH>;  // 引脚0_17，高电平点亮
    label = "red:led";
    status = "okay";
};`

2. 验证硬件原理图关键检查：
1. LED正极是否通过电阻连接GPIO0_17；2. 负极是否接地；3. 确认GPIO0_17未被其他外设占用（如<span class="green">UART</span>）。
若原理图中LED连接<span class="green">GPIO</span>0_18，则<span class="red">设备树</span>引脚号错误。

3. 用工具验证<span class="green">GPIO</span>实际状态
`# 1. 查看<span class="green">GPIO</span>0_17的注册状态
cat /sys/class/gpio/gpio49/direction  # <span class="green">GPIO</span>0_17对应编号=0*32+17=17？需确认芯片GPIO编码规则
# 注意：不同芯片编码不同（如全志H3 GPIO0_17对应编号17，TI AM335x对应49）
# 2. 手动控制GPIO点亮LED（验证硬件连通性）
echo 17 > /sys/class/gpio/export  # 导出<span class="green">GPIO</span>0_17
echo out > /sys/class/gpio/gpio17/direction  # 设为输出
echo 1 > /sys/class/gpio/gpio17/value  # 输出高电平
# 若灯亮：说明硬件正常，设备树配置错误；若不亮：硬件故障（如电阻烧毁）`<br>

### <strong>2. 中断号冲突：设备工作异常（E级重点）</strong>

中断号冲突表现为“设备偶尔响应”或“系统崩溃”，排查方法：
```bash

# 1. 查看系统中断分配情况
cat /proc/interrupts
# 输出示例（关键部分）：
#            CPU0       
#  32:        123  gpio-mxc  17 Edge      button@0  # 按键中断占32号
#  33:        456  gpio-mxc  18 Edge      ec20@0    # 4G模块中断占33号
# 若发现两个设备占用同一中断号（如32号），则存在冲突

# 2. 核对设备树中断配置
// 冲突场景：按键和4G模块均使用中断32
button@0 {
    compatible = "gpio-keys";
    interrupts = <32 IRQ_TYPE_EDGE_RISING>;  // 中断32
};
ec20@0 {
    compatible = "quectel,ec20";
    interrupts = <32 IRQ_TYPE_EDGE_FALLING>;  // 同一中断32
};

# 3. 解决方案：修改其中一个设备的中断号（需与硬件原理图一致）
# 如4G模块改用中断33，设备树改为：
interrupts = <33 IRQ_TYPE_EDGE_FALLING>;
```<br>
