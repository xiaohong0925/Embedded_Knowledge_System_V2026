# 设备树入门实战：编写与编译

> 📊 **本节难度等级：** <span class="badge-bi">**BI级**</span>

---

### <strong>设备树的编译依赖设备树编译器（dtc），而嵌入式开发中dtc通常集成在交叉编译工具链中（针对ARM、RISC-V等架构），无需单独安装。
本小节以最常用的ARM架构为例，讲解交叉编译工具链的安装与dtc工具的验证，操作环境为Ubuntu 22.04（其他Linux发行版步骤类似）。</strong>


### <strong>1. 交叉编译工具链选择与下载</strong>

嵌入式ARM开发常用的交叉编译工具链为GNU Arm Embedded Toolchain（官方维护，免费开源），
根据开发板CPU架构选择对应版本：
- 32位ARM架构（如ARMv7）：选择“arm-none-linux-gnueabihf”版本；
- 64位ARM架构（如ARMv8/AArch64）：选择“aarch64-none-linux-gnu”版本。

下载步骤：
1. 访问GNU Arm官方下载页（[官网链接](https://developer.arm.com/tools-and-software/open-source-software/developer-tools/gnu-toolchain/gnu-arm-embedded/downloads)），
选择对应版本（例：32位ARM的“9-2020-q2-update”版本）；

2. 下载后得到压缩包（例：`gcc-arm-9-2020-q2-update-x86_64-arm-none-linux-gnueabihf.tar.xz`），
存放至Ubuntu的`/opt`目录（系统级工具常规存放路径）。<br>

### <strong>2. 工具链安装与环境变量配置</strong>

通过解压与环境变量配置，让系统能全局调用工具链命令，具体操作命令如下：
```bash

# 1. 解压压缩包至/opt目录
sudo tar -xvf /opt/gcc-arm-9-2020-q2-update-x86_64-arm-none-linux-gnueabihf.tar.xz -C /opt/

# 2. 配置环境变量（临时生效，重启终端后失效）
export PATH=$PATH:/opt/gcc-arm-9-2020-q2-update-x86_64-arm-none-linux-gnueabihf/bin

# 3. 永久配置环境变量（推荐）：编辑.bashrc文件
echo "export PATH=\$PATH:/opt/gcc-arm-9-2020-q2-update-x86_64-arm-none-linux-gnueabihf/bin" >> ~/.bashrc
source ~/.bashrc  # 使配置立即生效
```<br>

### <strong>3. dtc工具验证</strong>

交叉编译工具链中的dtc命令命名格式为“*架构前缀*-dtc”，通过以下命令验证是否安装成功：
```bash

# 32位ARM工具链的dtc验证
arm-none-linux-gnueabihf-dtc --version

# 64位ARM工具链的dtc验证（若安装对应版本）
aarch64-none-linux-gnu-dtc --version
```

成功输出示例（版本信息可能不同，核心为“DTC version”字段）：
```log

Version: DTC 1.6.0-g27d17f3f
```

常见问题：若执行命令时提示“command not found”，检查两点：
① 工具链解压路径是否正确；
② 环境变量配置的路径是否与解压路径一致（可通过`ls /opt/`查看实际解压后的目录名）。<br>

### <strong>本小节以“ARM开发板控制LED灯”为场景，编写最小化DTS文件（Hello World级），覆盖设备树核心节点结构、关键属性和简单外设描述。
选择LED灯是因为其硬件连接简单（仅需<span class="green">GPIO</span>引脚），能快速理解“<span class="red">设备树</span>描述硬件”的核心逻辑。
硬件前提：假设开发板上的LED灯连接情况为——正极通过电阻连接<span class="green">GPIO</span>1组的第3个引脚（GPIO1_3），负极接地，低电平点亮。</strong>


### <strong>1. 最小DTS的核心结构</strong>

<span class="red">设备树</span>以“根节点（/）”为起点，所有硬件设备都作为根节点的子节点存在，最小DTS需包含三大核心部分：

1. 根节点：所有设备的父节点，必含`compatible`属性（描述系统兼容性）；
2. CPU节点：描述CPU核心信息（嵌入式系统必备，可复用芯片厂商提供的基础配置）；
3. 外设节点：本案例为LED灯节点，描述其挂载总线、引脚、激活电平等信息。<br>

### <strong>2. 完整最小DTS代码（test_led.dts）</strong>

```dts
// 注释：设备树支持单行注释（//）和多行注释（/* ... */）
/dts-v1/;  // 声明设备树版本（固定为v1）
/plugin/;  // 声明为可加载片段（简化示例，实际量产可省略）

// 根节点：所有设备的顶级节点
/ {
    // 核心属性：描述当前系统（开发板）的兼容性，供内核匹配
    compatible = "test,led-dev-board"; // compatible 属性错误：内核无法识别开发板类型，可能无法启动或加载错误的板级支持包
    model = "Test LED Development Board";  // 开发板型号（可选，用于标识）
                                           // 仅影响 cat /proc/device-tree/model 的信息显示
    // CPU节点：描述ARM CPU核心（此处以双核心为例，简化配置）
    cpus {
        #address-cells = <1>;  // 地址单元格数量（CPU编号为1个32位整数）
        #size-cells = <0>;     // 大小单元格数量（CPU无“大小”属性，为0）

        // CPU核心0节点（编号为0）
        cpu@0 {
            compatible = "arm,cortex-a7";  // CPU架构（Cortex-A7为例）
            device_type = "cpu";
            reg = <0>;  // CPU编号
        };

        // CPU核心1节点（编号为1）
        cpu@1 {
            compatible = "arm,cortex-a7";
            device_type = "cpu";
            reg = <1>;
        };
    };

    // GPIO控制器节点：LED依赖的GPIO控制器（需与硬件实际GPIO控制器匹配）
    gpio1: gpio@0x01C20800 {  // 标签：gpio1；节点名：gpio@基地址（实际基地址需查芯片手册）
        compatible = "allwinner,sunxi-gpio";  // GPIO控制器兼容驱动名（芯片厂商定义）
        reg = <0x01C20800 0x400>;  // 寄存器基地址+地址长度（查芯片手册获取）
        #gpio-cells = <2>;  // GPIO属性单元格数量（<2>; 引脚号 + 标志位）
                            // ← 声明：用我，得给2个参数
        status = "okay";  // 设备状态：okay（启用）、disabled（禁用）
    };

    // 外设节点：LED灯节点（挂载在根节点下，依赖gpio1控制器）
    leds {
        compatible = "gpio-leds";  // 兼容“GPIO类LED”通用驱动（内核自带）
        label = "system-leds";     // 节点标识（可选）

        // 具体LED子节点（led0）
        led0: led@0 {
            label = "user-led";  // LED标识（应用层可通过此名访问）
            // GPIO属性：引用gpio1控制器的第3个引脚，输出方向，低电平激活
            gpios = <&gpio1 3 1>;  // 引脚号 | 激活电平(0=low,1=high) 
            default-state = "off";  // 默认状态：熄灭
            linux,default-trigger = "none";  // 触发方式：无（手动控制）
        };
    };
};
```

### <strong>3. 关键代码解析（新手必懂）</strong>

- 标签（label）：
如`gpio1:`和`led0:`，是<span class="red">设备树</span>中的“节点别名”，用于快速引用节点
（后续其他节点可通过`&gpio1`引用<span class="green">GPIO</span>1控制器节点，避免重复写长地址）；

- compatible属性：
核心匹配属性，格式为“厂商名,设备名”，内核驱动通过此属性匹配对应的驱动程序。
例如LED节点的`"gpio-leds"`对应内核自带的`gpio_leds`驱动，无需自定义驱动即可使用；

- gpios属性：
<span class="green">GPIO</span>外设的核心配置，`<&gpio1 3 1>`含义分解：
- `&gpio1`：引用标签为gpio1的<span class="green">GPIO</span>控制器节点；
- `3`：<span class="green">GPIO</span>控制器的第3个引脚（GPIO1_3）；
- `1`：激活电平为低电平（0为高电平，1为低电平，具体由驱动定义）。

status属性：
控制设备是否启用，`"okay"`表示内核会解析该节点并初始化设备，`"disabled"`表示忽略该设备。

### <strong>DTS文件为文本格式，内核无法直接解析，需通过dtc工具编译为二进制的DTB文件。
本小节讲解完整编译流程、参数详解、反编译验证和常见错误排查，从B级基础命令到I级参数调优。</strong>


### <strong>1. 基础编译命令（B级）</strong>

使用交叉编译工具链中的dtc命令，将上一小节编写的`test_led.dts`编译为`test_led.dtb`，命令如下：
```bash

# 32位ARM架构编译命令
arm-none-linux-gnueabihf-dtc -I dts -O dtb -o test_led.dtb test_led.dts

# 64位ARM架构编译命令（若对应工具链）
aarch64-none-linux-gnu-dtc -I dts -O dtb -o test_led.dtb test_led.dts
```

命令执行后，若无任何输出，通过`ls`命令可看到生成的`test_led.dtb`文件，说明编译成功。<br>

### <strong>2. dtc核心参数详解（I级）</strong>

基础命令中的参数是最常用的，实际开发中需根据需求调整参数，核心参数如下表：

|参数         |含义                                             |示例
|-------------|-------------------------------------------------|---------------------------------
|-I <格式>    |指定输入文件格式（dts：文本源文件；dtb：二进制文件）|-I dts（输入为DTS）
|-O <格式>    |指定输出文件格式（常用dtb，也支持dts反编译）        |-O dtb（输出为DTB）
|-o <文件名>  |指定输出文件名                                    |-o test_led.dtb（输出为该文件）      
|-W <警告类型>|启用警告（如unused-node：未使用节点）              |-W unused-node（提示未使用的节点）
|-R <版本>    |指定DTB的版本号（需与内核支持版本匹配）            |-R 17（版本17，对应内核3.10+）
|-S <大小>    |指定DTB文件最大大小（单位：字节）                  |-S 65536（最大64KB）

进阶编译示例（启用警告+指定版本）：
```bash
arm-none-linux-gnueabihf-dtc -I dts -O dtb -o test_led.dtb -W unused-node -R 17 test_led.dts
```<br>

### <strong>3. 反编译验证（I级）</strong>

为确认编译后的DTB文件与原DTS一致（或排查内核解析问题），可将DTB反编译为DTS文件进行对比，命令如下：
```bash

# 反编译命令
arm-none-linux-gnueabihf-dtc -I dtb -O dts -o test_led_decompile.dts test_led.dtb
```

反编译后生成`test_led_decompile.dts`，
通过`diff test_led.dts test_led_decompile.dts`命令对比原文件与反编译文件，
核心内容（节点、属性）应完全一致，仅可能增加编译自动补充的默认属性（如`#address-cells`），属于正常现象。<br>

### <strong>4. 常见编译错误排查（I级）</strong>

编译时若DTS存在语法错误，dtc会输出明确的报错信息，以下是两类高频错误及解决方法：


1. 属性值格式错误（单元格数量不匹配）
典型报错：`test_led.dts:20.16-17 syntax error: unexpected <, expecting >`  
根因：<span class="green">GPIO</span>属性`gpios = <&gpio1 3>`中，单元格数量与控制器定义不符。
例如`gpio1`节点的`#gpio-cells`值为2或3，但只提供了1个参数。  

处理：查阅目标控制器的`#gpio-cells`定义，补全缺失字段。如改为`<&gpio1 3 1>`（增加标志位或第2个参数）。

2. 节点命名规范错误（缺少单元地址）
典型报错：`test_led.dts:15.5-6: ERROR (node_name): Node name 'gpio1' must have unit address`  
根因：带寄存器映射的节点（如<span class="green">GPIO</span>控制器）必须使用`节点类型@基地址`格式，纯名称`gpio1`不符合规范。  

处理：将节点名改为带物理基地址的形式，如`gpio@0x01C20800`（具体地址查芯片手册<span class="green">GPIO</span>章节）。

3. 标签引用未定义（phandle指向不存在节点）
典型报错：`test_led.dts:30.9: ERROR (reference_not_found): Reference to non-existent node or label 'gpio2'`  
根因：`&gpio2`引用的标签在DTS中不存在，通常是拼写错误或节点被删除/重命名。  
处理：核对根节点下的实际标签定义，修正为存在的标签（如`&gpio1`），或检查是否遗漏了包含该节点的`#include`文件。


排查建议：编译时加`-@`参数生成符号信息，配合`dtc -I dts -O dts test_led.dts`反编译检查展开后的节点结构，可快速定位引用和单元格数量问题。<br>

### <strong>设备树有严格的语法规则，掌握以下核心规则是编写正确DTS的基础，本小节结合前面的LED示例，系统讲解节点、属性、标签、引用的语法及数据类型。</strong>


### <strong>1. 节点语法规则</strong>

节点是<span class="red">设备树</span>的核心组成部分，用于描述一个硬件设备，
语法格式为：`标签: 节点名@地址 { 子节点; 属性; };`

- 节点名：格式为  设备类型@地址
“地址”是设备在父总线中的地址（如<span class="green">GPIO</span>控制器的寄存器基地址），
无地址时可省略“@地址”（如LED节点`led@0`）；

- 层级关系：节点通过嵌套实现层级，
如LED节点`led0`是`leds`节点的子节点，`leds`是根节点的子节点，
对应硬件的“根→LED组→单个LED”层级；

/dts-v1/;
/plugin/;

/ {
    // 根节点（无标签，无@地址）
    compatible = "fsl,imx6ull-evk";

    // 层级1：SOC节点（有基地址）
    soc: soc@02000000 {
        compatible = "simple-bus";
        reg = <0x02000000 0x100000>;
        #address-cells = <1>;
        #size-cells = <1>;

        // 层级2：<span class="green">GPIO</span>控制器（有寄存器基地址）
        gpio5: gpio@020ac000 {
            compatible = "fsl,imx6ul-gpio";
            reg = <0x020ac000 0x4000>;
            #gpio-cells = <2>;
        };
    };

    // 层级1：LED父节点（无寄存器地址，省略@）
    leds {
        compatible = "gpio-leds";

        // 层级2：LED子节点（@后为单元编号，非寄存器地址）
        led0: led@0 {
            label = "red";
            gpios = <&gpio5 2 <span class="green">GPIO</span>_ACTIVE_LOW>;
        };

        led1: led@1 {
            label = "green";
            gpios = <&gpio5 3 <span class="green">GPIO</span>_ACTIVE_LOW>;
        };
    };
};<br>

### <strong>2. 属性语法规则</strong>

属性用于描述节点的具体参数，格式为：`属性名 = <值>;
`（数值类型）或`属性名 = "值";`（字符串类型），核心规则如下：

（1）常见核心属性
|属性名        |作用                               |示例
|compatible    |驱动匹配标识，“厂商名,设备名”格式|`compatible = "gpio-leds";`
|reg           |设备地址信息，“基地址 长度”格式  |`reg = <0x01C20800 0x400>;`
|#address-cells|子节点地址的单元格数量             |`#address-cells = <1>;`
|#size-cells   |子节点地址长度的单元格数量         |`#size-cells = <0>;`
|status        |设备状态（okay/disabled）          |`status = "okay";`

（2）数据类型
属性值支持多种数据类型，开发中常用的有4种：
- 字符串类型：用于描述名称、兼容性等，用双引号包裹，如`label = "user-led";`；
- 32位整数类型：用尖括号`<>`包裹，单个整数或多个整数（单元格），如`#gpio-cells = <3>;`；
- 数组类型：多个整数的集合，如`reg = <0x01C20800 0x400>;`（基地址+长度数组）；
- 引用类型：通过标签引用其他节点，如`gpios = <&gpio1 3 1>`中的`&gpio1`。 

    soc: soc@f7000000 {
        compatible = "simple-bus";
        
        /* reg数组：基地址(64位) + 长度(32位) */
        reg = <0x0 0xf7000000 0x0 0x1000000>;  
        
        /* #cells数组：子节点地址/长度单元格数 */
        #address-cells = <2>;
        #size-cells = <1>;

        /* <span class="green">GPIO</span>控制器节点 */
        gpio0: gpio@ffd04000 {
            /* 字符串：驱动匹配标识 */
            compatible = "intel,socfpga-agilex-gpio";
            
            /* 数组：寄存器范围（64位基地址 + 32位长度） */
            reg = <0x0 0xffd04000 0x0 0x1000>;
            
            /* 32位整数：<span class="green">GPIO</span>参数单元格数 */
            #gpio-cells = <2>;  // 引脚号 + 标志位
            
            /* 字符串：设备状态枚举值 */
            status = "okay";  // 或 "disabled"
        };
    };<br>

### <strong>3. 标签与引用规则</strong>

- 标签：为节点起的别名，

格式为“标签名:*”，紧跟节点名前（如`gpio1: gpio@0x01C20800`），
作用是简化节点引用，避免写长节点路径；

- 引用：通过“`&标签名`”引用节点，
如引用gpio1控制器节点时，直接用`&gpio1`，无需写完整路径`&/gpio@0x01C20800`；

- 注意：标签名在整个DTS中必须唯一，否则编译时会报错“duplicate label”。

    /* ========= 定义带标签的节点 ========= */

    /* 标签：main_i2c0 - <span class="green">I2C</span>控制器 */
    main_i2c0: i2c@ffd02000 {
        compatible = "intel,socfpga-agilex-i2c";
        reg = <0x0 0xffd02000 0x0 0x100>;
        #address-cells = <1>;
        #size-cells = <0>;
        status = "okay";
    };

    /* ========= 通过&标签引用节点 ========= */

    /* EEPROM节点：引用main_i2c0标签 */
    eeprom@50 {
        compatible = "atmel,24c02";
        reg = <0x50>;
        /* 引用<span class="green">I2C</span>总线，无需写完整路径 */
        i2c-parent = <&main_i2c0>;  
    };

### <strong>4. 注释规范</strong>

<span class="red">设备树</span>支持两种注释方式，用于标注代码含义（量产DTS必备，提升可维护性）：
- 单行注释：`// 注释内容`，适用于短注释；
- 多行注释：`/* 注释内容 */`，适用于长段说明（如硬件连接描述）。<br>
