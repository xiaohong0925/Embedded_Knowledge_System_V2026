# 设备树底层原理：内核解析流程

> 📊 **本节难度等级：** <span class="badge-ie">**IE级**</span>

### <strong>设备树二进制文件（DTB）的生命周期始于启动阶段，其核心流程是“Bootloader加载DTB到内存→传递给Linux内核→内核确认DTB有效性”。
这一过程是<span class="red">设备树</span>解析的前提，若DTB加载或传递失败，内核将无法识别硬件，直接导致启动失败。本小节以嵌入式领域最常用的U-Boot（Bootloader）和ARM架构为例，详解全流程机制。</strong>


### <strong>1. DTB的存储与加载逻辑</strong>

DTB文件通常与内核镜像（zImage）、根文件系统镜像一同存储在开发板的非易失性存储设备（如eMMC、<span class="green">SPI</span> Flash）中，
且需划分独立的“<span class="red">设备树</span>分区”（如命名为dtb partition），分区地址需在U-Boot的配置文件（如`include/configs/rk3399.h`）中预先定义。

U-Boot启动后，加载DTB的方式分为“从本地存储加载”和“从网络加载”两种，实际开发中常用网络加载调试，本地加载用于量产：

- 本地存储加载（量产场景）：
U-Boot通过读取存储设备的分区表，定位DTB分区的起始地址和大小，再通过存储控制器驱动将DTB数据读取到指定的内存地址。

例如瑞芯微RK3399开发板的U-Boot命令：
`# 从<span class="green">eMMC</span>的dtb分区加载DTB到内存地址0x01f00000
mmc read 0x01f00000 0x800 0x400`
其中“0”为<span class="green">eMMC</span>设备号，“0x01f00000”为目标内存地址，“0x800”为DTB分区的起始扇区，“0x400”为读取的扇区数。

- 网络加载（调试场景）：
通过TFTP协议从PC端的TFTP服务器下载DTB到内存，适合频繁修改DTB的调试阶段，命令如下：
`# 1. 设置开发板IP和TFTP服务器IP
setenv ipaddr 192.168.1.100
setenv serverip 192.168.1.200
# 2. 下载DTB到内存地址0x01f00000
tftp 0x01f00000 test_led.dtb`<br>

### <strong>2. Bootloader到内核的传递机制</strong>

DTB加载到内存后，U-Boot需通过特定机制将“DTB在内存中的起始地址”传递给内核，

不同架构的传递方式不同，ARM架构遵循ARM Linux启动规范，核心依赖三个寄存器：
- `r0`：固定设置为0（历史兼容要求，标识启动模式）；
- `r1`：设置为机器码（Machine ID），早期无<span class="red">设备树</span>时用于内核匹配板级文件，引入设备树后可忽略，但需非0值；
- `r2`：设置为DTB在内存中的起始地址（核心传递参数）。

U-Boot通过`bootz`（启动压缩内核）命令触发传递流程，命令格式为`bootz 内核地址 根文件系统地址 DTB地址`，示例：
```bash

# 启动内核：内核地址0x01000000，无单独根文件系统（用“-”占位），DTB地址0x01f00000
bootz 0x01000000 - 0x01f00000
```

执行该命令后，U-Boot会自动配置r0、r1、r2寄存器，
然后通过汇编指令`bl kernel_entry`跳转到内核启动入口，完成DTB地址的传递。<br>

### <strong>3. 内核对DTB的接收与有效性校验</strong>

内核启动入口位于`arch/arm/kernel/head.S`（ARM架构），
汇编代码首先接收r2寄存器中的DTB地址，然后执行两项关键操作：

1. 有效性校验：DTB文件头部包含魔数（Magic Number）`0xd00dfeed`，内核通过校验该魔数判断DTB是否有效，源码片段如下：
`/* 从r2寄存器获取DTB地址，存入dtb_ptr */
ldr r1, =dtb_ptr
str r2, [r1]
/* 校验DTB魔数 */
ldr r0, [r2, #0]
ldr r1, =0xd00dfeed
cmp r0, r1
bne invalid_dtb  /* 魔数不匹配则跳转到无效处理流程 */`若魔数不匹配，内核会打印错误日志`Error: invalid device tree blob`并启动失败。

2. DTB重定位（可选）：
若DTB所在的内存区域与内核镜像重叠，内核会将DTB复制到空闲内存区域，确保后续解析时不被覆盖。<br>

### <strong>内核接收DTB后，由“设备树子系统”负责解析，该子系统的核心是`of_core`（Open Firmware Core）模块，它提供了一套通用的解析接口和数据结构，适配所有支持设备树的架构。
子系统架构可分为三层，每层职责明确，且通过函数接口解耦。</strong>


### <strong>1. of_core子系统三层架构</strong>

三层架构从下到上依次为“数据管理层”“解析引擎层”“总线适配层”，其关系和职责如下流程图所示：
┌───────────────────────────────┐
│                   内核<span class="red">设备树</span>子系统                           │
├───────────────────────────────┤
│  ┌─────┐      ┌─────┐      ┌─────┐      │
│  │总线适配层│─┬→│解析引擎层│─┬→│数据管理层│      │
│  └──┬──┘  │  ──────┘  │  └───┬─┘      │
│        │        │                  │          │          │
│        │        └─────────┴─────┘          │
│        │                           │                       │
│        ↓                           ↓                       │
│   ┌─────┐              ┌─────┐                 │
│   │各类总线  │              │          │                 │
│   │驱动      │              │ DTB内存  │                 │
│   │(platform/│              │ 数据     │                 │
│   │i2c/spi等)│              └─────┘                 │
│   └──┬──┘                                             │
│         │                                                   │
│         ↓                                                   │
│   ┌─────┐                                             │
│   │外设驱动  │                                             │
│   │(LED/传感 │                                             │
│   │ 器等)    │                                             │
│   └─────┘                                             │
└───────────────────────────────┘

- 数据管理层（底层）：
负责DTB数据的<span class="red">内存管理</span>和结构抽象，将二进制的DTB转换为内核可操作的树形数据结构（节点、属性），
核心数据结构为`struct device_node`（描述设备节点）和`struct property`（描述节点属性），
定义在`include/linux/of.h`中。

- 解析引擎层（中间层）：
提供核心解析接口，包括节点查找（如`<span class="green">of_find_node_by_name</span>`）、属性读取（如`of_property_read_u32`）、GPIO解析（如`of_get_named_gpio`）等，
这些接口对上层总线适配层透明，确保不同架构的解析逻辑一致。

- 总线适配层（上层）：
将解析后的节点信息转换为对应总线的设备结构体，例如将根节点下的子节点转换为`platform_device`（平台设备），
将<span class="green">I2C</span>控制器节点下的子节点转换为`i2c_client`（I2C从设备），使<span class="red">设备树</span>节点与内核的设备模型无缝对接。<br>

### <strong>2. 核心解析流程四步骤</strong>

of_core子系统的解析流程从内核初始化阶段开始，到生成设备结构体结束，共分为四个关键步骤，流程如下：
内核启动早期 (start_kernel)
    ↓
步骤1: of_core 初始化
└── 初始化<span class="red">设备树</span>子系统，准备解析DTB
    ↓
步骤2: DTB 解析与节点树构建 (of_parse_dt / unflatten_device_tree)
└── 将Bootloader传入的DTB二进制数据
    展开为内核内存中的 struct device_node 树
    ↓  
步骤3: 总线设备生成 (of_platform_populate)
└── 遍历 device_node 树，识别有 compatible 属性的节点
    创建对应的 platform_device (或 i2c_client, spi_device 等)
    ↓
步骤4: 设备注册 (platform_device_register)
└── 将 platform_device 添加到 platform_bus 总线
    触发驱动匹配，执行 driver->probe()<br>

### <strong>设备树解析生成设备结构体后，内核的核心任务是“为设备匹配到对应的驱动程序”，这一过程由总线的匹配机制完成。
其中`compatible`属性是匹配的“钥匙”，不同总线的匹配逻辑略有差异，但核心都基于<span class="red">设备树</span>节点的属性与驱动的匹配表比对。本小节以最常用的platform总线为例，详解匹配全流程。</strong>


### <strong>1. compatible属性的匹配规则（核心）</strong>

`compatible`属性（兼容属性）是<span class="red">设备树</span>节点与驱动绑定的核心依据，
其格式为“厂商名,设备名”的字符串列表（支持多个值，用于兼容不同驱动），
例如test_led.dts中`leds`节点的定义：
```dts

leds {
    compatible = "gpio-leds", "test,user-led";  // 支持两个兼容值
    /* 其他属性... */
};
```

驱动程序中需定义“匹配表”（`of_match_table`），存储驱动支持的`compatible`值，示例如下（LED驱动的匹配表）：
```c

#include <linux/of.h>
#include <linux/platform_device.h>

// 驱动支持的compatible值列表
static const struct of_device_id led_of_match[] = {
    { .compatible = "gpio-leds" },  // 与设备树第一个值匹配
    { .compatible = "test,user-led" },  // 与设备树第二个值匹配
    { /* Sentinel */ }  // 哨兵值，标识列表结束
};
MODULE_DEVICE_TABLE(of, led_of_match);  // 将匹配表注册到内核
```

匹配规则为“按顺序比对，找到第一个匹配项即停止”，具体逻辑：
1. 内核遍历设备结构体的`dev.of_node`指针（指向对应<span class="red">设备树</span>节点），读取`compatible`属性的字符串列表；
2. 遍历驱动的`of_match_table`列表，将每个列表项的`compatible`值与设备节点的列表逐一比对；
3. 若找到完全一致的字符串，匹配成功；若遍历结束仍无匹配项，匹配失败，驱动无法绑定设备。<br>

### <strong>2. platform总线的匹配流程（实战解析）</strong>

platform总线是嵌入式Linux的“通用总线”，用于连接无明确硬件总线（如<span class="green">I2C</span>、SPI）的外设（如LED、按键），
其匹配流程是<span class="red">设备树</span>与驱动绑定的典型场景，完整流程分为“设备注册”和“驱动注册”两个阶段，
核心依赖`platform_match`函数（匹配逻辑入口）。

（1）设备注册阶段：从节点到platform_device

该阶段在<span class="red">设备树</span>解析的步骤3和4完成，`of_platform_populate`函数递归遍历节点树时，对每个无明确总线属性的节点执行：
```c

// 简化后的核心逻辑
static int of_platform_device_create(struct device_node *np, const char *bus_id,
                                     struct device *parent)
{
    struct platform_device *pdev;
    // 1. 分配platform_device结构体
    pdev = platform_device_alloc("", -1);
    if (!pdev) return -ENOMEM;
    // 2. 关联设备树节点（关键：将node指针存入dev.of_node）
    pdev->dev.of_node = of_node_get(np);
    // 3. 注册到platform总线
    return platform_device_add(pdev);
}
```

注册完成后，设备结构体`platform_device`被添加到platform总线的设备链表中，
其`dev.of_node`字段指向对应的<span class="red">设备树</span>节点，为后续匹配提供依据。<br>

### <strong>设备与驱动匹配成功后，驱动需要从设备树节点中读取具体的硬件参数（如GPIO引脚号、中断号、寄存器基地址），并将这些参数映射到内核的设备结构体或硬件寄存器中，完成“软件参数→硬件控制”的流转。
这一过程依赖`of_core`提供的接口函数，且需严格遵循“先解析参数，后初始化硬件”的原则。</strong>


### <strong>1. 核心of接口函数分类与实战</strong>

of接口函数按功能可分为“节点操作”“属性读取”“硬件资源解析”三类

1. 节点操作函数
- `<span class="green">of_find_node_by_name</span>(parent, name)`
  - 功能：根据节点名查找子节点
  - 实战：从根节点查找"leds"节点  
  <span class="green">of_find_node_by_name</span>(NULL, "leds");
- `of_get_child_by_name(parent, name)`
  - 功能：根据子节点名查找直接子节点
  - 实战：从"leds"节点查找"led0"子节点  
  of_get_child_by_name(leds_node, "led0");

2. 属性读取函数
- `of_property_read_u32(np, propname, &out)`
  - 功能：读取32位整数属性
  - 实战：读取LED的默认亮度（若有"brightness"属性）
  of_property_read_u32(np, "cells", &val)
- `of_property_read_string(np, propname, &out)`
  - 功能：读取字符串属性
  - 实战：读取LED的标签  
  of_property_read_string(led0_node, "label", &led_label);
- `of_property_count_elems_of_size(np, propname, elem_size)`
  - 功能：统计属性值的个数
  - 实战：统计"gpios"属性的单元格数量  
  of_property_count_elems_of_size(led0_node, "gpios", sizeof(u32));

3. 硬件资源解析函数
- `of_get_named_gpio(np, propname, index)`
  - 功能：解析<span class="green">GPIO</span>属性，返回GPIO编号
  - 实战：解析"gpios"属性，获取<span class="green">GPIO</span>编号  
  gpio_num = of_get_named_gpio(led0_node, "gpios", 0);
- `of_irq_get_byname(np, name)`
  - 功能：根据名称解析中断号
  - 实战：若LED支持中断（如按键联动），解析"interrupts"属性
  key_irq = of_irq_get_byname(key_node, "key0");<br>

### <strong>3. 内核设备结构体与设备树节点的关联</strong>

数据流转的核心是“<span class="red">设备树</span>节点与内核设备结构体的强关联”，
这种关联通过两个关键指针实现，确保驱动在任何时候都能通过设备结构体找到对应的<span class="red">设备树</span>节点：

- 设备结构体→<span class="red">设备树</span>节点：
`struct device`结构体中的`of_node`指针，在设备生成时由`of_platform_<span class="green">device_create</span>`函数赋值，如前面的platform设备生成逻辑；

- <span class="red">设备树</span>节点→设备结构体：
`struct device_node`结构体中的`dev`指针，在设备注册时由内核自动关联，用于通过节点反向查找设备。

通过这两个指针，内核实现了“<span class="red">设备树</span>节点”与“内核设备模型”的双向映射，为数据流转提供了底层支撑。<br>
