## 驱动开发环境搭建与基础流程

---

### <strong>交叉编译环境配置：针对ARM/RISC-V的驱动编译脚本编写</strong>

<span class="red">驱动开发</span>运行在嵌入式硬件（如ARM架构开发板、RISC-V开发板），<br>
但开发通常在x86架构的PC上进行，<br>
因此需要<span class="green">交叉编译环境</span>（在一种架构的机器上编译出另一种架构可执行的代码）。<br>

这部分核心是<span class="blue">"获取工具链→配置环境→验证有效性→编写通用编译脚本"</span>。<br>

<span class="orange"><strong>1. 工具链获取与安装（以ARM为例，RISC-V流程类似）</strong></span><br>

嵌入式ARM架构最常用的交叉编译工具链是<span class="green">GNU Arm Embedded Toolchain</span>，<br>
需根据开发板的ARM架构（32位/64位）选择版本：<br>
* <span class="green">32位ARM</span>（如ARM9、Cortex-A7）：选择<span class="green">`arm-linux-gnueabihf`</span>（硬浮点支持）<br>
* <span class="green">64位ARM</span>（如Cortex-A53/A72）：选择<span class="green">`aarch64-linux-gnu`</span><br>

安装步骤（Ubuntu 20.04系统）：<br>

```bash
# 1. 下载工具链（以32位为例，官网地址可替换为国内镜像）
wget https://developer.arm.com/-/media/Files/downloads/developer-tools/gnu-arm-embedded/10-2020-q4-major/gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2

# 2. 解压到指定目录（通常放在/opt下，需root权限）
sudo tar -xjvf gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2 -C /opt/

# 3. 重命名简化路径（可选，方便后续配置）
sudo mv /opt/gcc-arm-none-eabi-10-2020-q4-major /opt/arm-linux-gnueabihf
```

RISC-V架构工具链获取：<br>
选择<span class="green">`riscv64-linux-gnu`</span>工具链，<br>
下载地址为https://github.com/riscv-collab/riscv-gnu-toolchain/releases，<br>
解压和安装步骤与ARM一致，<br>
路径建议设为<span class="green">`/opt/riscv64-linux-gnu`</span>。<br>

### <strong>内核源码准备：配置CONFIG_MODULES与驱动相关配置项</strong>

<span class="red">驱动模块</span>（<span class="green">`.ko`</span>文件）是内核的"扩展组件"，<br>
编译驱动必须依赖与开发板内核版本一致、配置一致的内核源码。<br>
核心步骤是<span class="blue">"获取源码→配置内核→编译模块依赖"</span>。<br>

<span class="orange"><strong>1. 内核源码获取</strong></span><br>

内核源码有两个主要来源，<br>
优先选择厂商定制版（兼容性更好）：<br>
* <span class="blue">厂商定制版（推荐）</span>：开发板厂商官网下载（如全志、瑞芯微），<br>
  例如全志H3开发板的内核源码通常包含<span class="green">`soc`</span>相关驱动，兼容性更高；<br>
* <span class="blue">官方原版</span>：Linux内核官网（https://www.kernel.org/）下载对应版本（如5.15.71），<br>
  需自行适配开发板硬件（难度较高，适合有经验者）。<br>

源码解压后放在固定路径（如<span class="green">`/home/user/linux-5.15.71-sunxi`</span>），<br>
后续编译驱动时会用到这个路径。<br>

### <strong>基础流程：驱动源码→Makefile编写→编译→加载→调试</strong>

这是驱动开发的<span class="red">"标准流程"</span>，<br>
以一个最简单的<span class="green">"Hello驱动"</span>为例，<br>
完整演示从代码编写到调试的全流程。<br>

<span class="orange"><strong>1. 步骤1：编写驱动源码（hello_drv.c）</strong></span><br>

这是一个最简化的驱动模块，<br>
仅包含"模块加载"和"模块卸载"两个核心函数，<br>
用于验证流程：<br>

```c
#include <linux/init.h>   // 包含模块初始化相关函数（如module_init）
#include <linux/module.h> // 包含模块相关宏（如MODULE_LICENSE）

// 模块加载函数：当执行insmod加载模块时，内核自动调用此函数
static int __init hello_drv_init(void)
{
    // printk：内核态打印函数（类似用户态printf，输出到内核日志）
    printk("Hello driver loaded!\n");
    return 0; // 返回0表示加载成功
}

// 模块卸载函数：当执行rmmod卸载模块时，内核自动调用此函数
static void __exit hello_drv_exit(void)
{
    printk("Hello driver unloaded!\n");
}

// 注册模块加载/卸载函数（内核识别的宏）
module_init(hello_drv_init);
module_exit(hello_drv_exit);

// 模块许可声明：必须添加，否则加载时会报"tainted kernel"警告
MODULE_LICENSE("GPL");
// 模块作者信息（可选）
MODULE_AUTHOR("Embedded Linux Learner");
// 模块描述信息（可选）
MODULE_DESCRIPTION("A Simple Hello Driver");
```

关键术语解释：<br>
* <span class="green">模块（Module）</span>：内核的可动态加载/卸载组件，<br>
  无需重新编译内核即可扩展功能；<br>
* <span class="green">`module_init`</span>：宏定义，指定模块加载时执行的函数；<br>
* <span class="green">`printk`</span>：内核态打印函数，<br>
  输出内容默认存放在内核日志缓冲区，<br>
  需用<span class="green">`dmesg`</span>命令查看。<br>

### <strong>实操：编译并加载第一个自定义字符设备驱动模块</strong>

前面的"Hello驱动"仅验证流程，<br>
实际驱动需实现硬件控制或数据交互。<br>
这里以<span class="red">"字符设备驱动"</span>（最常用的驱动类型，如LED、按键）为例，<br>
实现一个"支持<span class="red">用户态</span>读写"的简单驱动，<br>
<span class="red">用户态</span>可通过设备文件控制驱动输出信息。<br>

<span class="orange"><strong>1. 字符<span class="red">设备驱动</span>核心概念</strong></span><br>

* <span class="blue">字符设备（Character Device）</span>：<br>
  按字节流顺序读写的设备，<br>
  驱动通过<span class="green">`file_operations`</span>结构体向内核提供"打开、关闭、读写"等接口，<br>
  用户态通过设备文件（如<span class="green">`/dev/char_drv`</span>）访问。<br>
* <span class="blue">设备号（Device Number）</span>：<br>
  每个字符设备对应唯一的设备号（主设备号+次设备号），<br>
  用于内核识别设备。<br>

## 设备树与驱动模型的匹配实战

---

### <strong>设备树节点编写规范：compatible、reg、interrupts等关键属性</strong>

<span class="red">设备树（Device Tree，DT）</span>是Linux内核中"描述硬件信息的树形结构文件"，<br>
核心作用是替代传统板级代码（<span class="green">`board.c`</span>），<br>
将硬件配置与驱动代码解耦——<br>
驱动通过解析<span class="red">设备树</span>获取硬件参数（如寄存器地址、中断号），<br>
无需硬编码。<br>

编写<span class="red">设备树</span>的核心是按规范定义"节点"和"属性"，<br>
其中<span class="green">`compatible`</span>（匹配标识）、<span class="green">`reg`</span>（地址信息）、<span class="green">`interrupts`</span>（中断信息）<br>
是驱动匹配的关键属性。<br>

---

### <strong>驱动中设备树属性解析API：of_property_read_u32/of_find_node_by_name</strong>

驱动与<span class="red">设备树</span>匹配后，<br>
需通过内核提供的<span class="green">OF（Open Firmware）API</span>解析节点中的属性值<br>
（如<span class="green">`reg`</span>的地址、<span class="green">`led-gpio`</span>的引脚），<br>
核心流程是<span class="blue">"找到节点→解析属性→初始化硬件"</span>。<br>

常用API分为"节点查找"和"属性解析"两类，<br>
以下结合LED驱动实战讲解。<br>

---

### <strong>调试技巧：用dtc编译设备树与debugfs查看设备树节点信息</strong>

<span class="red">设备树</span>与驱动匹配失败时，<br>
需先验证"<span class="red">设备树</span>是否正确编译""内核是否正确解析节点"，<br>
核心工具是<span class="green">`dtc`</span>（编译验证）和<span class="green">`debugfs`</span>（运行时查看），<br>
配合<span class="green">`dmesg`</span>（内核日志）定位问题。<br>

<span class="orange"><strong>1. dtc工具：编译与反编译<span class="red">设备树</span>（验证语法错误）</strong></span><br>

<span class="green">`dtc`</span>是设备树编译器，<br>
用于将<span class="green">`.dts`</span>源文件编译为内核可识别的<span class="green">`.dtb`</span>二进制文件，<br>
同时可反编译<span class="green">`.dtb`</span>验证编译结果，<br>
是<span class="red">设备树</span>编写的基础工具。<br>

常用命令（Ubuntu环境，需先安装<span class="green">`dtc`</span>：<span class="green">`sudo apt install device-tree-compiler`</span>）：<br>

```bash
# 1. 编译.dts为.dtb（检查语法错误，生成二进制文件）
dtc -I dts -O dtb -o led.dtb led.dts
# 选项说明：-I（输入格式dts）、-O（输出格式dtb）、-o（输出文件名）
# 若有语法错误（如括号不匹配、属性名错误），会直接报错，如"Syntax error at line 10"

# 2. 反编译.dtb为.dts（验证编译后的内容是否与源码一致）
dtc -I dtb -O dts -o led_decompile.dts led.dtb
# 对比led.dts和led_decompile.dts，确认属性值未被篡改

# 3. 查看.dtb的详细信息（包含节点结构和属性值）
dtc -I dtb -O dts -o - led.dtb  # -o - 表示输出到终端
```

关键场景：编译时若报错"Undefined reference to 'pio'"，<br>
说明未引入包含<span class="green">`pio`</span>节点的芯片级设备树<br>
（需添加<span class="green">`#include "sun8i-h3.dtsi"`</span>）。<br>

<span class="orange"><strong>2. <span class="green">debugfs</span>：运行时查看<span class="red">设备树</span>节点与属性（内核已解析的信息）</strong></span><br>

内核启动后会解析<span class="green">`.dtb`</span>文件，<br>
将节点和属性信息存入<span class="green">`debugfs`</span>文件系统，<br>
通过挂载<span class="green">`debugfs`</span>可直观查看"内核实际解析到的硬件信息"，<br>
是验证<span class="red">设备树</span>是否生效的核心手段。<br>

操作步骤（开发板上执行）：<br>

```bash
# 1. 挂载debugfs（临时挂载，重启后失效）
mount -t debugfs none /sys/kernel/debug

# 2. 进入设备树节点目录（所有解析后的节点都在该路径下）
cd /sys/kernel/debug/devicetree/base

# 3. 查看节点结构（ls查看子节点，对应设备树中的节点）
ls
# 预期输出：led@300B0000  pio  gic  ...（包含我们定义的LED节点）

# 4. 查看LED节点的属性（进入节点目录，cat查看属性值）
cd led@300B0000
cat compatible  # 查看compatible属性
# 预期输出：vendor,led-dt（与设备树一致）
cat reg  # 查看reg属性（十六进制输出，需转换为十进制）
# 预期输出：300b0000 00001000（对应0x300B0000和0x1000）
cat led-gpio  # 查看自定义GPIO属性
# 预期输出：00000001 00000005 00000001（对应pio组1、引脚5、高电平有效）

# 5. 查看节点状态（status属性，确认节点是否启用）
cat status
# 预期输出：okay（启用），若为disabled则内核不解析此节点
```

关键场景：若<span class="green">`debugfs`</span>中没有自定义的<span class="green">`led@300B0000`</span>节点，<br>
说明<span class="red">设备树</span>未被内核加载<br>
（需确认<span class="green">`.dtb`</span>文件是否正确烧录到开发板）。<br>

<span class="orange"><strong>3. <span class="green">dmesg</span>：查看<span class="red">设备树</span>解析与驱动匹配日志</strong></span><br>

内核解析<span class="red">设备树</span>和驱动匹配时会输出日志，<br>
通过<span class="green">`dmesg`</span>过滤关键词可快速定位问题：<br>

```bash
# 1. 查看设备树解析日志（过滤"dtb"或节点名）
dmesg | grep -i "dtb"
# 预期输出：[    0.000000] OF: fdt: Machine model: Vendor Board Name（设备树加载成功）

dmesg | grep -i "led@300B0000"
# 预期输出：[    1.234567] of: registered device node /led@300B0000（节点注册成功）

# 2. 查看驱动匹配日志（过滤驱动名或probe）
dmesg | grep -i "led-dt-driver"
# 预期输出：[  100.789012] led-dt-driver: probe of led@300B0000 succeeded（驱动匹配并probe成功）
```

### <strong>问题排查：设备树与驱动不匹配的常见原因（compatible字段错误等）</strong>

设备树与驱动匹配失败的核心表现是"驱动<span class="green">`probe`</span>函数不执行"，<br>
常见原因集中在"<span class="red">设备树</span>编写错误""驱动匹配表错误""内核配置错误"三类，<br>
以下按排查优先级排序：<br>

<span class="orange"><strong>1. 原因1：compatible字段不匹配（最常见）</strong></span><br>

现象：<span class="green">`dmesg`</span>中无驱动<span class="green">`probe`</span>成功日志，<br>
<span class="green">`lsmod`</span>显示驱动已加载但无设备关联；<br>
排查步骤：<br>

```bash
# 步骤1：查看驱动的of_match_table（确认驱动支持的compatible）
# 方法：驱动源码中找到of_match_table，记录compatible值（如"vendor,led-dt"）

# 步骤2：查看设备树的compatible（通过debugfs）
cat /sys/kernel/debug/devicetree/base/led@300B0000/compatible
# 若输出与驱动的of_match_table不一致（如多写一个空格、厂商名错误），修改设备树或驱动

# 步骤3：验证匹配结果（内核日志）
dmesg | grep -i "of_match"
# 若显示"no matching of_node for driver led-dt-driver"，确认compatible字段完全一致（区分大小写）
```

修复方案：确保驱动<span class="green">`of_match_table`</span>中的<span class="green">`compatible`</span><br>
与设备树节点的<span class="green">`compatible`</span>完全一致<br>
（字符串无拼写错误、无多余空格）。<br>

<span class="orange"><strong>2. 原因2：<span class="red">设备树</span>节点未启用（status属性错误）</strong></span><br>

现象：<span class="green">`debugfs`</span>中找不到自定义节点，驱动无法匹配；<br>
排查步骤：<br>

```bash
# 步骤1：查看设备树源码中的status属性
# 确认节点是否有status = "okay"; 若为status = "disabled"，内核不解析

# 步骤2：若节点在子目录下，检查父节点状态
# 例：若LED节点挂在"peripheral@10000000"下，需确认父节点status也是"okay"
cat /sys/kernel/debug/devicetree/base/peripheral@10000000/status
```

修复方案：将节点及父节点的<span class="green">`status`</span>属性设为<span class="green">`"okay"`</span>，<br>
重新编译烧录<span class="red">设备树</span>。<br>

<span class="orange"><strong>3. 原因3：reg/interrupts属性格式错误（驱动解析失败）</strong></span><br>

现象：<span class="green">`dmesg`</span>中显示"read reg failed"或"invalid irq"，<br>
驱动<span class="green">`probe`</span>返回错误；<br>
排查步骤：<br>

```bash
# 步骤1：查看设备树的reg属性格式（debugfs）
cat /sys/kernel/debug/devicetree/base/led@300B0000/reg
# 预期输出为两个值（基地址+长度），若只有一个值，说明格式错误

# 步骤2：对比硬件手册，确认reg地址正确
# 全志H3的GPIO控制器基地址是0x300B0000，若写为0x300B0001则错误

# 步骤3：查看中断属性格式
cat /sys/kernel/debug/devicetree/base/led@300B0000/interrupts
# 预期输出为中断号+触发方式（如25 4），若格式错误需修改设备树
```

修复方案：按硬件手册修正<span class="green">`reg`</span>的基地址和长度、<br>
<span class="green">`interrupts`</span>的中断号和触发方式，<br>
确保与驱动解析的数组长度一致<br>
（如<span class="green">`reg`</span>需两个值，<br>
驱动用<span class="green">`of_property_read_u32_array`</span>读取2个元素）。<br>

<span class="orange"><strong>4. 原因4：内核未启用<span class="red">设备树</span>支持（配置错误）</strong></span><br>

现象：所有<span class="red">设备树</span>节点都无法解析，<br>
<span class="green">`dmesg`</span>显示"dtb not found"；<br>
排查步骤：<br>

```bash
# 步骤1：查看内核配置（需内核支持设备树）
zcat /proc/config.gz | grep CONFIG_OF
# 预期输出：CONFIG_OF=y（启用设备树支持）

# 步骤2：确认开发板启动时加载了正确的.dtb文件
# 查看启动日志（如U-Boot日志），确认"Loading Device Tree to 87000000, size 28000 bytes"
```

修复方案：重新配置内核，<br>
启用<span class="green">`CONFIG_OF=y`</span>（设备树支持）<br>
和<span class="green">`CONFIG_OF_DEVICE=y`</span>（设备树设备支持），<br>
重新编译内核和<span class="red">设备树</span>。<br>

## 驱动核心接口与数据交互

---

### <strong>字符设备驱动核心接口：cdev_init/cdev_add与文件操作集file_operations</strong>

<span class="red">字符设备</span>是嵌入式Linux最常用的设备类型（如LED、按键、串口），<br>
其驱动的核心是通过<span class="green">字符设备结构体（`cdev`）</span>向内核注册设备，<br>
通过<span class="green">文件操作集（`file_operations`）</span><br>
向用户态提供标准化接口（如<span class="green">`open`/`read`/`write`</span>）。<br>

这两个接口是字符<span class="red">设备驱动</span>的<span class="blue">"骨架"</span>，<br>
所有功能都围绕它们展开。<br>

---

### <strong>设备号管理：静态申请与动态申请的适用场景（register_chrdev_region等）</strong>

<span class="red">设备号</span>是内核识别字符设备的"唯一标识"，<br>
分为<span class="green">主设备号</span>（高20位，标识设备类型，如串口主设备号是4）<br>
和<span class="green">次设备号</span>（低12位，标识同类型的具体设备，如串口1的次设备号是0）。<br>

设备号的申请方式分<span class="blue">"静态"和"动态"</span>，<br>
需根据场景选择。<br>

| 申请方式 | 适用场景 | 核心API | 特点 |
| --- | --- | --- | --- |
| 静态申请 | 已知主设备号，需固定分配 | `register_chrdev_region` | 需确保设备号未被占用，适合量产产品 |
| 动态申请 | 未知主设备号，由内核分配 | `alloc_chrdev_region` | 避免冲突，适合开发调试 |

### <strong>数据交互方式：copy_to_user/copy_from_user的使用与效率考量</strong>

用户态与<span class="red">内核态</span>运行在不同的地址空间<br>
（用户态地址是虚拟地址，<span class="red">内核态</span>无法直接访问），<br>
数据交互必须通过内核提供的安全拷贝函数，<br>
核心是<span class="green">`copy_to_user`</span>（内核→用户态）<br>
和<span class="green">`copy_from_user`</span>（用户态→内核态）。<br>

此外，还需关注拷贝效率，<br>
避免频繁小批量拷贝导致性能瓶颈。<br>

| 函数 | 方向 | 核心作用 | 返回值含义 |
| --- | --- | --- | --- |
| `copy_to_user` | 内核→用户态 | 将内核缓冲区数据拷贝到用户态缓冲区 | 0=成功，非0=剩余未拷贝字节数 |
| `copy_from_user` | 用户态→内核 | 将用户态缓冲区数据拷贝到内核缓冲区 | 0=成功，非0=剩余未拷贝字节数 |

### <strong>驱动开发中"调试"的核心目标与三类问题定位</strong>

<span class="red">驱动开发中"调试"</span>的核心目标是<br>
"定位代码逻辑错误、硬件交互异常、内核接口使用不当"三类问题，<br>
基础调试手段虽不涉及复杂工具，<br>
但能解决<span class="blue">80%以上的入门级问题</span>。<br>

本节从"最基础的打印调试"到"灵活的模块参数"，<br>
再到"高频问题排查流程"，<br>
层层递进落地实操。<br>

### <strong>打印调试：printk日志级别配置与dmesg过滤技巧</strong>

<span class="red">`printk`</span>是内核态最基础的调试工具（类似用户态<span class="green">`printf`</span>），<br>
但内核对日志有"级别管控"，<br>
需通过配置级别确保关键日志输出，<br>
再用<span class="green">`dmesg`</span>过滤提取有效信息——<br>
这是驱动开发的<span class="blue">"第一调试手段"</span>。<br>

<span class="green">`printk`</span>支持8个日志级别，<br>
从<span class="green">`KERN_EMERG`</span>（紧急）到<span class="green">`KERN_DEBUG`</span>（调试），<br>
常用级别如下：<br>

| 级别宏 | 级别值 | 含义 | 典型使用场景 |
| --- | --- | --- | --- |
| `KERN_EMERG` | 0 | 系统崩溃 | 内核panic时 |
| `KERN_ALERT` | 1 | 需立即处理 | 硬件故障 |
| `KERN_CRIT` | 2 | 严重错误 | 关键服务失败 |
| `KERN_ERR` | 3 | 错误 | 驱动加载失败 |
| `KERN_WARNING` | 4 | 警告 | 非致命异常 |
| `KERN_NOTICE` | 5 | 普通但重要 | 设备插拔通知 |
| `KERN_INFO` | 6 | 信息 | 正常状态报告 |
| `KERN_DEBUG` | 7 | 调试 | 详细运行信息 |

默认情况下，<br>
级别低于<span class="green">`KERN_WARNING`</span>（即<span class="green">`KERN_ERR`</span>及以上）的日志会输出到控制台，<br>
级别高的（<span class="green">`KERN_INFO`</span>、<span class="green">`KERN_DEBUG`</span>）需显式配置才能显示。<br>

配置方法（动态调整，无需重启）：<br>

```bash
# 查看当前日志级别
cat /proc/sys/kernel/printk
# 输出示例：4 4 1 7（分别对应：当前级别、默认级别、最小级别、boot默认级别）

# 临时修改级别（如改为8，显示所有级别日志）
echo 8 > /proc/sys/kernel/printk

# 过滤特定驱动的日志（以led_drv为例）
dmesg | grep "led_drv"
# 仅显示led_drv驱动的日志，排除其他干扰

# 实时跟踪新日志（类似tail -f）
dmesg -w | grep "led_drv"
```

### <strong>模块参数：通过module_param传递调试参数</strong>

<span class="red">模块参数</span>允许"加载驱动时传递参数"或"运行时修改参数"，<br>
无需重新编译驱动即可调整调试逻辑，<br>
是<span class="blue">"灵活调试"的核心手段</span>。<br>

<span class="orange"><strong>1. 核心接口：module_param与module_param_array</strong></span><br>

内核提供宏定义用于声明模块参数，<br>
支持整数、字符串、数组等常见类型，<br>
核心宏定义如下：<br>

* <span class="green">单个参数</span>：<span class="green">`module_param(name, type, perm)`</span><br>
  * <span class="green">`name`</span>：参数名（用户态传递时使用的名称）；<br>
  * <span class="green">`type`</span>：参数类型（如<span class="green">`int`</span>、<span class="green">`charp`</span>字符串、<span class="green">`bool`</span>布尔值）；<br>
  * <span class="green">`perm`</span>：参数文件权限<br>
    （<span class="green">`/sys/module/<模块名>/parameters/<参数名>`</span>的权限，<br>
    如<span class="green">`0644`</span>表示读写权限）。<br>

* <span class="green">数组参数</span>：<span class="green">`module_param_array(name, type, nump, perm)`</span><br>
  * <span class="green">`nump`</span>：指针，用于存储用户态传入的数组元素个数。<br>

### <strong>常见问题：驱动加载失败（依赖缺失、符号未导出）的排查流程</strong>

<span class="red">驱动加载失败</span>是最高频的问题，<br>
核心表现为<span class="green">`insmod`</span>返回"Invalid module format"等错误，<br>
需按<span class="blue">"加载日志→依赖检查→符号检查→版本检查"</span>的流程排查。<br>

<span class="orange"><strong>1. 核心排查工具与前置准备</strong></span><br>

需掌握三个核心工具：<br>
* <span class="green">`insmod`</span>：加载驱动（需指定完整路径，不自动解决依赖）；<br>
* <span class="green">`modprobe`</span>：加载驱动并自动解决依赖（推荐，需驱动在<span class="green">`/lib/modules/<内核版本>/`</span>目录）；<br>
* <span class="green">`dmesg`</span>：查看加载失败的内核日志（核心排查依据）；<br>
* <span class="green">`lsmod`</span>：查看已加载模块及依赖关系；<br>
* <span class="green">`modinfo`</span>：查看驱动模块信息（如内核版本、依赖符号）。<br>

### <strong>实战：LED驱动加载失败的全流程排查案例</strong>

以"LED驱动加载失败"为例，<br>
串联前文调试技巧，<br>
完整演示排查流程：<br>

<span class="orange"><strong>1. 问题现象</strong></span><br>

执行<span class="green">`insmod led_param_drv.ko`</span>后返回"Invalid module format"，<br>
LED不亮，<span class="green">`lsmod`</span>无驱动信息。<br>

<span class="orange"><strong>2. 排查步骤</strong></span><br>

```bash
# 步骤1：查看内核日志（定位错误类型）
dmesg | tail -20
# 关键错误示例：
# "led_param_drv: version magic '5.15.0-sunxi SMP preempt mod_unload ARMv7 p2v8 ' should be '5.15.0-sunxi SMP preempt mod_unload ARMv7 p2v7 '"
# → 结论：驱动编译的内核版本与目标板不一致

# 步骤2：检查驱动依赖（modinfo查看依赖符号）
modinfo led_param_drv.ko
# 查看depends字段，确认依赖模块是否已加载

# 步骤3：检查符号导出（grep查看未定义符号）
nm led_param_drv.ko | grep " U "
# "U"表示未定义符号，需确认这些符号是否由其他模块导出

# 步骤4：验证内核版本一致性
modinfo led_param_drv.ko | grep vermagic
# 对比目标板内核版本（uname -r），必须完全一致
```

<span class="orange"><strong>3. 修复方案</strong></span><br>

* <span class="blue">版本不匹配</span>：使用与目标板完全一致的内核源码重新编译驱动；<br>
* <span class="blue">依赖缺失</span>：先加载依赖模块（如<span class="green">`insmod dependency.ko`</span>），<br>
  再用<span class="green">`modprobe`</span>加载目标驱动；<br>
* <span class="blue">符号未导出</span>：确认依赖模块已加载且符号已导出<br>
  （检查<span class="green">`/proc/kallsyms`</span>），<br>
  或在驱动源码中添加<span class="green">`EXPORT_SYMBOL`</span>导出符号。<br>
