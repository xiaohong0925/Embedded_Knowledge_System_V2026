### <strong>传输路径：外设→DMA通道→内存（总线级数据流向）</strong>

<span class="red">DMA</span>的“外设→内存”传输并非“直接直连”，而是通过系统总线架构实现的高效数据流转
——这就像城市里的“物流运输网络”，外设是“工厂”，<span class="red">DMA</span>是“物流枢纽”，内存是“仓库”，总线是“高速公路”，数据传输的效率不仅取决于DMA本身，还与总线带宽、拓扑结构直接相关。

中级开发者必须理解总线级路径，才能排查“传输速度不达预期”等架构级问题。

（1）系统总线架构中的<span class="red">DMA</span>位置
嵌入式SoC（如STM32MP157、S3C2440）的总线通常分为“外设总线”和“系统总线”两级，<span class="red">DMA</span>控制器是连接两级总线的核心枢纽，具体拓扑如下（以STM32MP157为例）：

1.  <span class="red">DMA</span>与外设的总线匹配性：
低速外设（如<span class="green">UART</span>）挂在APB总线，只能对接支持APB的<span class="red">DMA</span>控制器（如DMA1）；高速外设（如SDIO）挂在AHB总线，必须对接AHB兼容的DMA控制器（如DMA2）
——若错配（如SDIO对接<span class="red">DMA</span>1），会因总线速率不匹配导致传输效率骤降（类似“高速货车走乡村小路”）。
2.  <span class="red">DMA</span>的总线仲裁权：
当多个<span class="red">DMA</span>通道同时传输（如<span class="green">UART</span>接收+ADC采样），DMA控制器会通过“总线仲裁器”分配AHB3总线的使用权，仲裁规则通常为“固定优先级”或“轮转优先级”（可通过寄存器配置）
——实战中需将高速外设（如SDIO）的通道优先级设为最高，避免被低速传输阻塞。
3.  内存访问的权限控制：
DMA访问DDR/SRAM前需经过MPU（内存保护单元），若MPU未配置“DMA访问权限”，会导致传输失败（类似“物流车进仓库被保安拦下”）——驱动开发时需确认<span class="red">设备树</span>中MPU对目标内存区域的权限配置。<br>

### <strong>分散-聚合（Scatter-Gather）：非连续内存块的高效传输原理</strong>

“分散-聚合”是<span class="red">DMA</span>的高级传输模式，解决“非连续内存块的批量传输”问题
——传统<span class="red">DMA</span>只能传输“连续内存”，若要传输多块分散内存（如应用层分配的3个独立缓冲区），需CPU先将数据复制到连续内存，再启动DMA；而分散-聚合模式允许DMA直接遍历多块非连续内存，自动完成“分散读取”或“聚合写入”，彻底解放CPU。

（1）核心价值：为什么需要分散-聚合？
举一个工业场景：摄像头采集1024x768的图像数据，应用层为了<span class="red">内存管理</span>方便，将缓冲区分成3块（256KB、256KB、256KB），若用传统DMA传输：
1.  CPU需先将3块分散内存的数据复制到1块768KB的连续内存（耗时5ms）；
2.  <span class="red">DMA</span>将连续内存的数据传输到LCD控制器（耗时3ms）；
3.  传输完成后CPU释放连续内存（耗时0.5ms）。

总耗时8.5ms，且CPU复制过程占用大量资源；而用分散-聚合模式：
1.  <span class="red">DMA</span>直接遍历3块分散内存，依次传输到LCD控制器（耗时3ms）；
2.  传输完成后触发中断，CPU仅做状态校验（耗时0.1ms）。

总耗时3.1ms，CPU开销降低98%——这就是分散-聚合的核心价值：**避免CPU数据拼接/拆分，提升非连续内存传输效率**。<br>

### <strong>缓存一致性：DMA与CPU缓存的同步问题（dma_sync_*系列函数）</strong>

缓存一致性是<span class="red">DMA</span>开发中“最隐蔽的陷阱”
——CPU为了提升内存访问速度，会将常用数据缓存到“CPU缓存”（如L1/L2缓存），而<span class="red">DMA</span>直接访问“物理内存”（如DDR），若两者数据不同步，会导致“CPU读不到DMA写入的数据”或“DMA读取到CPU缓存的旧数据”。中级开发者必须理解问题根源，并掌握`dma_sync_*`系列函数的使用方法。

（1）问题根源：CPU缓存与物理内存的“数据孤岛”
CPU访问内存的流程是“CPU→缓存→物理内存”，而<span class="red">DMA</span>访问流程是“DMA→物理内存”，两者没有直接交互，导致两种典型的不一致场景：

场景1：<span class="red">DMA</span>写内存，CPU读不到最新数据（<span class="green">UART</span>接收场景）
1.  CPU首次读取内存缓冲区（0x80000000），数据被缓存到L1缓存；
2.  驱动启动<span class="red">DMA</span>，<span class="green">UART</span>接收数据后，DMA将最新数据写入物理内存（0x80000000）；
3.  CPU再次读取该地址时，直接从L1缓存读取旧数据，忽略了<span class="red">DMA</span>写入的新数据——导致“<span class="green">UART</span>已接收数据，但应用层读不到”。

场景2：CPU写内存，<span class="red">DMA</span>读不到最新数据（<span class="green">UART</span>发送场景）
1.  CPU将待发送数据写入内存缓冲区（0x80000000），数据仅存于L1缓存，未同步到物理内存；
2.  驱动启动<span class="red">DMA</span>，DMA从物理内存读取旧数据并发送——导致“应用层发送新数据，但<span class="green">UART</span>发送旧数据”。<br>

### <strong>DMA引擎核心API：dma_request_channel（申请通道）、dma_slave_config（配置）</strong>

Linux内核为不同厂商的<span class="red">DMA</span>控制器（如STM32 DMA、TI EDMA、NVIDIA Tegra DMA）提供了统一的DMA引擎框架
——核心价值是“屏蔽硬件差异，提供标准化接口”：驱动开发者无需关注STM32 <span class="red">DMA</span>的CCR寄存器、TI EDMA的PARAMSET寄存器等硬件细节，只需调用内核封装的标准API，即可完成DMA通道申请、参数配置、传输控制，大幅提升驱动的可移植性。

### <strong>DMA-BUF机制：跨设备数据共享（如摄像头→GPU的图像数据传输）</strong>

传统<span class="red">DMA</span>传输中，不同设备的缓冲区相互隔离（如摄像头的采集缓冲区、GPU的渲染缓冲区），若要实现“摄像头采集的图像直接由GPU渲染”，需CPU将摄像头缓冲区的数据复制到GPU缓冲区
——这种“CPU中转”方式会占用大量CPU资源，且复制过程耗时（如1080P图像复制耗时>10ms）。
<span class="red">DMA</span>-BUF机制通过“零拷贝（Zero-Copy）”实现跨设备数据共享，多个设备直接访问同一块物理内存，彻底消除CPU复制开销。

（1）<span class="red">DMA</span>-BUF的核心组成要素
<span class="red">DMA</span>-BUF是内核提供的“跨设备共享内存对象”，核心由3个组件构成：
| 组件                | 核心作用                                  | 类比（物流场景）                          |
|---------------------|-------------------------------------------|-------------------------------------------|
| dma_buf             | 核心对象，代表共享的物理内存缓冲区，包含fd（文件描述符） | 共享仓库，有唯一的“仓库编号（fd）”        |
| dma_buf_attachment  | 设备与dma_buf的绑定关系                    | 仓库的“准入凭证”，设备需绑定后才能访问    |
| sg_table            | 分散-聚合表，描述dma_buf的物理内存布局    | 仓库的“货架分布图”，告知设备数据存放在哪  |<br>

### <strong>通道管理：DMA通道的独占与共享策略（设备树dma-names配置）</strong>

<span class="red">DMA</span>通道是嵌入式系统的“稀缺硬件资源”（如STM32MP157仅提供16个DMA通道），多个设备可能竞争同一个通道。
内核通过“独占”和“共享”两种策略管理通道资源，<span class="red">设备树</span>是配置通道管理策略的核心入口（通过dma-names、dma-shared等属性）。

（1）独占策略：高实时性设备的专属通道
① 核心特点
- 通道被一个设备“独占使用”，其他设备无法申请该通道；
- 适用场景：高实时性、低延迟要求的设备（如工业串口<span class="green">UART</span>、电机控制ADC、医疗设备的传感器）；
- 优势：无通道竞争，传输延迟可预测；
- 劣势：通道利用率低，若设备未使用通道，资源会被闲置。

② <span class="red">设备树</span>配置示例（STM32MP157 <span class="green">UART</span>4独占DMA1通道4）
```dts
uart4: serial@40010800 {
    compatible = "st,stm32h7-uart";
    reg = <0x40010800 0x400>;
    interrupts = <GIC_SPI 53 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&rcc UART4_K>;
    dmas = <&dma1 4 0x400 0x0>; // 指定DMA1通道4
    dma-names = "rx", "tx";     // 通道名称：rx对应接收，tx对应发送
    status = "okay";
};

dma1: dma-controller@48000000 {
    compatible = "st,stm32mp1-dma";
    reg = <0x48000000 0x1000>;
    interrupts = <GIC_SPI 11 IRQ_TYPE_LEVEL_HIGH>;
    #dma-cells = <3>;
    dma-channels = <8>;
    status = "okay";
};
```

③ 实战注意点
驱动卸载时必须调用`dma_release_channel`释放独占通道，否则内核会将通道标记为“忙”，即使驱动已卸载，其他设备也无法申请：
```c
// 驱动remove函数中释放通道
static int uart4_remove(struct platform_device *pdev) {
    struct uart_dma_dev *dma_dev = platform_get_drvdata(pdev);
    dmaengine_terminate_all(dma_dev->rx_chan);
    dma_release_channel(dma_dev->rx_chan); // 关键：释放独占通道
    return 0;
}
```<br>

### <strong>地址约束：DMA可访问的内存范围（物理地址vs虚拟地址）
DMA控制器是“硬件外设”，其地址访问逻辑与CPU完全不同——CPU通过MMU将虚拟地址映射为物理地址，且可访问内核/<span class="red">用户态</span>的所有内存；而DMA直接访问物理内存，且受硬件架构限制（如地址总线宽度、内存映射范围），这是DMA地址约束的核心根源。</strong>

（1）核心约束：<span class="red">DMA</span>的地址访问限制
嵌入式SoC的<span class="red">DMA</span>控制器受两类硬件限制，决定了其可访问的内存范围：

| 限制类型                | 具体表现                                  | 典型示例（STM32MP157/ARMv7）|
|-------------------------|-------------------------------------------|-----------------------------------|
| 地址总线宽度限制        | 32位DMA控制器仅能访问0x00000000~0xFFFFFFFF（4GB）物理地址；64位DMA可访问更大范围 | STM32MP157的DMA控制器为32位，无法访问4GB以上的物理内存 |
| 内存映射范围限制        | DMA仅能访问“外设可访问的内存区域”（片内SRAM、DDR），无法访问CPU私有缓存、MMU映射的虚拟内存 | 无法直接访问CPU的L1缓存，也无法访问仅存在虚拟地址的用户态内存 |
| 内存对齐限制            | 源/目标地址需按“总线宽度对齐”（如32位总线需4字节对齐），否则触发硬件错误 | UART DMA传输的内存地址若为0x80000001（1字节偏移），会导致传输失败 |<br>

### <strong>长度约束：单次传输的最大长度（受DMA控制器硬件限制）
<span class="red">DMA</span>的单次传输长度并非“无限大”，而是受长度寄存器位数的硬件限制
——长度寄存器是<span class="red">DMA</span>控制器的核心硬件单元，其位数决定了单次可传输的最大字节数，这是长度约束的核心根源。</strong>

（1）硬件根源：长度寄存器的位数限制
不同厂商的<span class="red">DMA</span>控制器长度寄存器位数不同，直接决定单次传输的最大长度：
| DMA控制器类型        | 长度寄存器位数 | 单次最大传输长度（字节） | 典型芯片                          |
|---------------------|----------------|--------------------------|-----------------------------------|
| STM32F103 DMA       | 16位           | 65535（0xFFFF）| STM32F103C8T6                     |
| STM32MP157 DMA      | 24位           | 16777215（0xFFFFFF）| STM32MP157AA                      |
| TI AM335x EDMA      | 32位           | 4294967295（0xFFFFFFFF） | TI AM3358（BeagleBone Black）|

关键注意点：
- 长度寄存器的“计数单位”需匹配地址宽度：若地址宽度为4字节（32位），16位寄存器的最大传输长度为65535×4=262140字节（而非65535字节）；
- 部分<span class="red">DMA</span>控制器的长度寄存器“从0开始计数”，实际最大长度为“位数最大值-1”（如16位寄存器最大65534字节）。<br>

### <strong>并发约束：多通道传输的优先级调度规则
当多个<span class="red">DMA</span>通道同时申请总线资源时，DMA控制器的**仲裁器**会按预设规则分配总线使用权，这是并发约束的核心根源
——若调度规则配置不当，会导致“低优先级通道被饿死”“高优先级抢占导致数据不完整”等问题。</strong>

（1）硬件仲裁器的核心调度规则
<span class="red">DMA</span>控制器的仲裁器支持两类核心调度规则，可通过寄存器配置：

| 调度规则        | 核心逻辑                                                                 | 适用场景                          | 
|-----------------|--------------------------------------------------------------------------|-----------------------------------|
| 固定优先级      | 通道优先级固定（如通道0>通道1>…>通道7），高优先级通道可抢占低优先级通道 | 工业控制（关键通道需优先传输）| 
| 轮转优先级      | 同优先级通道轮流使用总线，无抢占机制                                     | 多媒体传输（多通道需公平调度）|

硬件配置示例（STM32MP157 <span class="red">DMA</span>1仲裁规则）：
通过`devmem`命令修改<span class="red">DMA</span>1仲裁寄存器（地址0x48000000），配置固定优先级：
```bash
# 读取DMA1仲裁寄存器（bit0-1：通道0优先级，bit2-3：通道1优先级，以此类推）
devmem 0x48000000 32
# 写入0x00003333：将通道0-3的优先级设为最高（3级），通道4-7设为最低（0级）
devmem 0x48000000 32 0x00003333
```<br>

### <strong>总结</strong>

1. 地址约束：
<span class="red">DMA</span>仅能访问物理地址，且受地址总线宽度、内存对齐限制，需通过`dma_alloc_coherent`分配可访问内存，避免直接使用CPU虚拟地址；
2. 长度约束：
单次传输长度受<span class="red">DMA</span>长度寄存器位数限制，超大传输需拆分为多次单次传输，可通过描述符链批量提交提升效率；
3. <span class="red">并发</span>约束：
多通道传输的优先级由硬件仲裁器决定，可通过<span class="red">设备树</span>独占配置、内核优先级标记、抢占阈值设置，平衡关键通道实时性与低优先级通道公平性。<br>
