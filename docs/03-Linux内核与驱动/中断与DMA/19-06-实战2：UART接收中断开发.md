# 实战2：UART接收中断开发

> 📊 **本节难度等级：** <span class="badge-e">**E级**</span>

---

### <strong>UART（串口）是工业控制中最常用的通信接口（如PLC、传感器、工控屏通信），工业场景中常需接收1024字节以上的大报文，且要求“低延迟、无丢包”。
本实战以STM32MP157的<span class="green">UART</span>4为例，实现“UART接收中断触发→环形缓冲区暂存数据→底半部解析报文”的核心功能，解决“接收缓冲区溢出”“中断嵌套导致丢包”“低延迟与高吞吐量平衡”三大工业级难点。</strong>


### <strong>核心难点</strong>

1. 接收缓冲区溢出：
<span class="green">UART</span>硬件FIFO（先进先出缓冲区）容量有限（如8字节），若<span class="red">中断处理</span>不及时，新接收的数据会覆盖旧数据，导致丢包；
2. 中断嵌套问题：
<span class="green">UART</span>接收中断优先级若低于其他中断（如电机控制），会导致中断响应延迟，进而引发缓冲区溢出；
3. 低延迟与高吞吐量平衡：
单次接收1字节触发中断会增加CPU开销，需配置FIFO触发阈值（如4字节触发中断），平衡延迟与开销。<br>

### <strong>驱动开发要点</strong>

1. 双缓冲区设计：中断接收数据→环形缓冲区暂存
环形缓冲区（Circular Buffer）是解决<span class="green">UART</span>接收数据“实时暂存+无溢出”的经典方案，核心是“中断上下文写入数据，进程上下文读取数据”，避免硬中断上下文处理复杂逻辑。

（1）环形缓冲区核心设计（数据结构+操作函数）
```c
// 环形缓冲区数据结构
#define UART_BUF_SIZE 2048  // 缓冲区大小（大于最大报文1024字节）
struct uart_ring_buf {
    unsigned char buf[UART_BUF_SIZE];
    unsigned int head;      // 写指针（中断上下文修改）
    unsigned int tail;      // 读指针（进程上下文修改）
    spinlock_t lock;        // 自旋锁（保护缓冲区，软中断上下文可用）
};

// 初始化环形缓冲区
static void uart_ring_buf_init(struct uart_ring_buf *rb) {
    rb->head = 0;
    rb->tail = 0;
    spin_lock_init(&rb->lock);
}

// 写入数据到缓冲区（中断上下文调用）
static int uart_ring_buf_write(struct uart_ring_buf *rb, unsigned char data) {
    unsigned long flags;
    unsigned int next_head;

    spin_lock_irqsave(&rb->lock, flags);  // 关中断+自旋锁，保护临界区
    next_head = (rb->head + 1) % UART_BUF_SIZE;
    if (next_head == rb->tail) {  // 缓冲区满
        spin_unlock_irqrestore(&rb->lock, flags);
        return -ENOMEM;
    }
    rb->buf[rb->head] = data;
    rb->head = next_head;
    spin_unlock_irqrestore(&rb->lock, flags);
    return 0;
}

// 读取数据从缓冲区（进程上下文调用）
static int uart_ring_buf_read(struct uart_ring_buf *rb, unsigned char *data) {
    unsigned long flags;
    int ret = -EAGAIN;

    spin_lock_irqsave(&rb->lock, flags);
    if (rb->head != rb->tail) {  // 缓冲区有数据
        *data = rb->buf[rb->tail];
        rb->tail = (rb->tail + 1) % UART_BUF_SIZE;
        ret = 0;
    }
    spin_unlock_irqrestore(&rb->lock, flags);
    return ret;
}
```

### <strong>故障排查：报文丢包问题（中断优先级过低/缓冲区过小）</strong>

工业场景中<span class="green">UART</span>报文丢包是高频故障，需按“硬件→中断→缓冲区”的顺序排查，核心排查步骤如下：

1. 硬件层面排查（物理层问题）
- 测量串口电平：使用示波器测量TX/RX引脚，确认波特率、奇偶校验匹配（工业场景常见“波特率错配导致乱码/丢包”）；
- 检查FIFO溢出：读取<span class="green">UART</span> LSR寄存器的OE位（Overrun Error），若置1表示FIFO溢出，命令：
  ```bash
  devmem 0x40010805 8  # 读取UART4 LSR寄存器，bit1为1表示OE
  ```
2. 中断层面排查（优先级/响应延迟）
- 检查中断优先级：<span class="green">UART</span>接收中断优先级需高于非关键中断（如LED控制），配置命令：
  ```bash
  # 查看当前中断优先级（irq=88）
  cat /proc/irq/88/priority
  # 设置优先级为50（数值越小优先级越高，GICv2范围0-255）
  echo 50 > /proc/irq/88/priority
  ```
- 查看中断响应延迟：使用`perf`工具统计中断响应时间，命令：
  ```bash
  perf record -e irq:irq_handler_entry -g
  perf report  # 查看uart_rx_irq_handler的响应延迟
  ```

3. 缓冲区层面排查（容量/读写速度）
- 监控环形缓冲区使用率：在驱动中添加打印，统计head与tail的差值，若频繁接近满（>90%），需增大缓冲区大小；
- 检查底半部处理速度：若工作队列处理报文速度慢于中断写入速度，需优化解析逻辑（如简化校验、异步通知应用层）。

4. 丢包排查清单（工业级）
| 丢包现象               | 可能原因                     | 解决方案                                  |
|------------------------|------------------------------|-------------------------------------------|
| 偶发丢1-2字节          | FIFO触发阈值过低，中断响应延迟 | 提高触发阈值（如4→8字节），提升中断优先级  |
| 连续丢包，OE位置1      | FIFO溢出                     | 增大环形缓冲区，优化中断处理函数执行时间  |
| 报文截断（后半部分丢失） | 环形缓冲区过小               | 将缓冲区从2048字节增大至4096字节          |
| 随机丢包，无OE位       | 自旋锁持有时间过长           | 简化中断处理函数，减少临界区代码          |<br>
