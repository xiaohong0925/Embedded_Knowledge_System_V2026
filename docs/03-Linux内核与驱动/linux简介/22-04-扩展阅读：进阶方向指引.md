### <strong>内核版本选型不是“选最新”或“选最老”的简单判断，而是嵌入式项目的“顶层技术决策”
——选对版本能让项目少走80%的后期维护弯路，选错则可能面临“上线后频繁崩溃”“硬件驱动不兼容”“维护期被迫重构”等致命问题。比如工业控制项目选了短期支持的主线版本，3年后版本停止更新，遭遇安全漏洞却无补丁可打；
物联网新品选了老旧LTS版本，新研发的传感器没有对应驱动，被迫额外投入3个月开发驱动。本节核心是帮你建立“版本特性→场景需求→决策依据”的选型逻辑，而非给出唯一答案。</strong>


### <strong>LTS版本与主线版本的差异
Linux内核版本分为两大分支：LTS版本（Long-Term Support，长期支持版本） 和主线版本（Mainline Version） ，两者的设计目标、维护策略完全不同，直接决定了适配的项目场景。</strong>

在选型前，必须先明确两者的核心差异，避免因“版本类型误判”导致选型失误。

1. 核心定义与版本命名规则
首先要能快速识别版本类型——Linux内核版本采用“主版本.次版本.修订版本”的命名规则，通过“次版本奇偶性”和“官方标注”可直接区分类型：
- 主版本：重大架构迭代，如从4.x到5.x增加了BPF架构、原生RAID支持等核心特性；
- 次版本：偶数为LTS候选（如5.14→5.15，5.14是主线，5.15是LTS），奇数为纯主线（如5.16、5.17）；
- 修订版本：累加更新（如5.15.100，100表示第100次修订）；
- 官方标注：LTS版本会在内核官网（https://www.kernel.org/）标注“longterm”，主线标注“mainline”。

举个直观例子：
| 版本号    | 类型 | 官方标注 | 核心特征 |
|----------|------|----------|----------|
| 5.15.148 | LTS  | longterm | 2021年10月发布，支持至2026年12月，累计修订148次 |
| 6.1.60   | LTS  | longterm | 2022年12月发布，支持至2026年12月，面向新硬件适配 |
| 6.6.30   | 主线 | mainline | 2023年10月发布，无长期支持，2024年3月停止更新 |<br>

### <strong>嵌入式场景版本选型的核心考量（稳定性、驱动支持、实时性）
嵌入式场景差异极大（工业控制、物联网、汽车电子、消费电子），选型不能“一刀切”，需围绕“稳定性、驱动支持、实时性”三大核心维度，结合场景特性权衡。以下是每个维度的深度解析及决策逻辑，均来自嵌入式项目实战总结。</strong>

1. 核心考量1：稳定性——嵌入式项目的“生命线”
稳定性是嵌入式选型的“第一优先级”，比“新功能”更重要——工业设备停机1小时可能损失数十万，医疗设备故障可能危及生命。内核版本的“稳定性”不是“绝对无bug”，而是“已知bug可修复、无致命漏洞、运行行为可预测”。

稳定性的两大核心判断标准：
- 维护类型与频率：LTS版本的更新仅含“安全补丁（CVE修复）”和“关键bug修复”，不会引入新功能导致兼容性风险；主线版本的更新含新功能，可能触发“新bug”。例如5.15 LTS的更新日志中，90%以上是“fix: xxx crash”“CVE-2024-xxxx: buffer overflow”类修复，无“add: new xxx driver”类功能新增。
- 社区验证程度：选择“被大量嵌入式发行版采用的LTS版本”更稳定。例如：
  - 树莓派OS、Armbian等主流嵌入式系统采用5.15 LTS；
  - 工业级发行版（如Wind River Linux）采用4.19 LTS、6.1 LTS；
  - 若某LTS版本被汽车电子（如特斯拉车载系统）采用，说明其稳定性经过极端场景验证。

实操验证方法（M级进阶技巧）：
1.  查社区bug反馈：访问Linux内核bugzilla（https://bugzilla.kernel.org/），搜索目标版本的“open bug”数量，5.15 LTS的open bug通常比同期主线少60%以上；
2.  跑稳定性测试工具：在目标版本上运行`stress-ng`（压力测试）、`kmemleak`（内存泄漏检测），连续运行72小时无崩溃或泄漏即可初步判定稳定：
    ```bash
    # 安装stress-ng（嵌入式系统需提前编译）
    opkg install stress-ng
    # 对CPU、内存、IO进行压力测试，持续72小时
    stress-ng --cpu 4 --mem 2G --io 4 --timeout 72h --metrics
    ```<br>

1. 四步决策流程（通用适配所有嵌入式场景）<br>

### <strong>嵌入式开发的进阶之路，必然绕不开“开源社区协作”——从修复内核驱动的小Bug，到为U-Boot添加新开发板支持，再到给Buildroot提交自定义软件包，每一次贡献都是技术能力的认证。
但开源社区有严格的协作规范，盲目提交往往会被直接退回（如补丁格式错误、描述模糊）。本节以“内核补丁提交”为基础，延伸至U-Boot、Buildroot两大嵌入式核心项目的贡献逻辑，帮你建立“规范先行、精准落地”的协作思维。</strong>


### <strong>内核补丁提交的基本规范
Linux内核作为嵌入式系统的“根基”，其社区协作规范被多数嵌入式开源项目（如U-Boot、BusyBox）借鉴，掌握内核补丁提交逻辑，可无缝迁移至其他项目。核心原则是“清晰传递信息、降低维护成本”，具体拆解为“准备工作、开发规范、提交流程、自检验证”四步。</strong>

1. 前置准备：环境配置与工具依赖
内核社区不接受GitHub Pull Request，所有协作通过邮件列表（Mailing List） 完成，需提前配置Git与邮件客户端，确保补丁可被社区工具解析。

核心工具配置（实操命令）
1.  Git基础配置（确保提交者信息可追溯）：
    ```bash
    # 配置提交者姓名和邮箱（必须与社区注册邮箱一致）
    git config --global user.name "Your Full Name"
    git config --global user.email "your.email@example.com"
    # 配置默认编辑器（用于编写提交信息，推荐vim）
    git config --global core.editor vim
    ```
2.  邮件客户端配置（以sendmail为例，内核社区推荐工具）：
    ```bash
    # 安装sendmail（Ubuntu/Debian系统）
    sudo apt install sendmail git-email
    # 配置Git使用sendmail发送邮件
    git config --global sendemail.transport sendmail
    # 可选：配置邮件签名（避免被识别为垃圾邮件）
    git config --global sendemail.signature "Your Name\nEmbedded Developer"
    ```
3.  获取内核源码（建议从稳定分支克隆，避免主线版本冲突）：
    ```bash
    # 克隆Linux内核稳定分支（以6.1 LTS为例，嵌入式常用）
    git clone -b linux-6.1.y git://git.kernel.org/pub/scm/linux/kernel/git/stable/linux-stable.git
    cd linux-stable
    ```<br>

### <strong>嵌入式相关开源项目（U-Boot、Buildroot）贡献入门
嵌入式开发者最常接触的开源项目除了内核，还有U-Boot（引导加载程序） 和Buildroot（根文件系统构建工具） ，其贡献逻辑与内核相似，但有项目专属的规范和渠道。</strong>

1. U-Boot贡献：聚焦板级支持与驱动
U-Boot是嵌入式系统的“启动基石”，贡献场景多为“添加新开发板支持”“修复驱动Bug”“优化启动流程”，核心协作渠道是邮件列表（u-boot@lists.denx.de）。

核心差异与实操要点
1.  规范差异：
    - 提交信息前缀为`[U-Boot]`，如`[U-Boot] board: stm32mp1: Add support for custom board`。
    - 板级代码需放在`board/`目录（如`board/stm32mp1/custom_board/`），需包含`Kconfig`、`Makefile`、`board.c`三个核心文件。
2.  添加新开发板案例：
    - 步骤1：复制同类开发板的板级目录（如复制`board/stm32mp1/evk/`到`custom_board/`）。
    - 步骤2：修改`board.c`中的硬件初始化逻辑（如串口、DDR配置），修改`Kconfig`中的板级名称。
    - 步骤3：生成补丁<span class="red">并发</span>送，接收者参考`MAINTAINERS`文件中`STM32`相关维护者和`u-boot@lists.denx.de`列表。
3.  自检工具：使用U-Boot自带的`tools/checkpatch.pl`检查补丁格式，与内核工具用法一致。<br>
