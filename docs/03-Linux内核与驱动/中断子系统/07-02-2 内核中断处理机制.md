### <strong>中断向量表的跳转逻辑</strong>

我们可以将中断向量表，比作轨道交通线网调度中心的紧急事件标准化通讯录。
通讯录中每一个编号，对应唯一的紧急事件类型与专属处置负责人。
事件触发后，调度中心可直接跳转至对应负责人的处置流程，无需逐级上报。

中断向量表（Interrupt Vector Table, IVT），是CPU内核中预先定义的、固定地址的连续内存数组。
数组中的每一个元素，称为一个中断向量，对应一个异常/中断类型的处理函数入口地址。
它是CPU响应中断后，找到对应处理逻辑的唯一硬件级路由表，是<span class="red">中断处理</span>流程的起点。

ARMv8-A架构的中断向量表，固定按异常级别（EL）、异常类型分为16个固定入口。
每个入口对应一类异常/中断，包括同步异常、IRQ普通中断、FIQ快速中断、系统错误异常。
CPU收到GIC转发的中断信号后，会自动根据中断类型，跳转到向量表中对应的固定入口地址。

| 向量表偏移| 异常级别 | 中断类型       | 核心作用                             |
| :-------- | :--------| :------------- | :----------------------------------- |
| 0x000     | EL1      | 同步异常       | 处理指令执行错误、系统调用等同步事件 |
| 0x080     | EL1      | IRQ普通中断    | 处理外设触发的普通硬件中断，最常用   |
| 0x100     | EL1      | FIQ快速中断    | 处理高优先级的实时中断               |
| 0x180     | EL1      | SError系统错误 | 处理硬件内存访问错误、总线错误等异常 |

以下为Linux 5.15 LTS内核中，ARM64架构中断向量表的核心定义源码：
```c
// 源码路径：linux-5.15/arch/arm64/kernel/entry.S
// ARM64架构Linux内核中断向量表，硬件强制要求对齐到2KB地址边界
.align 11
SYM_CODE_START(vectors)
    kernel_ventry   1, sync_invalid         // EL1同步异常无效入口
    kernel_ventry   1, irq                  // EL1 IRQ普通中断核心入口
    kernel_ventry   1, fiq_invalid          // EL1 FIQ中断无效入口
    kernel_ventry   1, error_invalid        // EL1 SError错误无效入口

    // 省略其他异常级别入口，核心外设中断均走0x080偏移的irq入口
SYM_CODE_END(vectors)
```
代码关键说明：
- `SYM_CODE_START/END`：内核符号起止宏，标记向量表的内存地址范围
- `.align 11`：向量表必须对齐到2^11=2048字节，是ARMv8架构的硬件强制要求
- `kernel_ventry`：内核向量入口宏，自动完成中断现场初步保存与跳转逻辑

在嵌入式Linux开发板上，执行以下命令可验证内核中断向量表的实际地址：
```bash
cat /proc/kallsyms | grep -w "vectors"
```
典型输出与解读：
```
ffff000008001000 T vectors
```
输出中的`ffff000008001000`，即为当前内核中断向量表的起始物理地址，完全符合ARMv8架构的对齐要求。<br>

### <strong>中断上下文的进入与退出</strong>

我们可以将中断上下文的进入与退出，比作调度员切换紧急处置模式的完整流程。
调度员接到紧急事件后，先将当前手头的工作文件、进度记录完整封存。随后进入专属的紧急处置模式，仅处理当前紧急事件，不介入日常调度工作。
事件处置完成后，调度员解封之前封存的文件，回到之前的工作进度继续执行。

中断上下文（Interrupt Context），是CPU响应中断后，执行<span class="red">中断处理</span>程序时的专属运行环境。
它与进程上下文严格隔离，有明确的执行约束，不可执行可能导致阻塞、睡眠的操作。
中断上下文的生命周期，仅存在于中断进入到中断退出的完整周期内。
```
┌────────────────────────┐
│       中断上下文进入与退出完整执行流           │
├────────────────────────┤
│  CPU收到GIC转发的中断信号，停止当前指令执行    │
├────────────────────────┤
│  硬件自动保存核心寄存器，跳转到中断向量表入口  │
├────────────────────────┤
│  【进入中断上下文】内核保存完整执行现场        │
│  调用irq_enter()标记进入中断上下文环境         │
├────────────────────────┤
│  执行中断处理程序(ISR)，完成中断事件核心响应   │
├────────────────────────┤
│  【退出中断上下文】调用irq_exit()标记退出环境  │
│  内核恢复之前保存的完整执行现场                │
├────────────────────────┤
│  硬件恢复程序计数器，回到被打断的主程序继续执行│
└────────────────────────┘
```

中断上下文的核心执行约束，是保障<span class="red">中断处理</span>实时性与稳定性的核心规则：
- 不可调用可能导致睡眠、阻塞的内核函数，如<span class="green">mutex_lock</span>、msleep等
- 不可执行耗时过长的操作，<span class="red">中断处理</span>总时长应控制在微秒级
- 不可直接访问用户空间内存，中断上下文无用户空间内存映射
- 不可主动放弃CPU调度，非RT内核中中断上下文不可被<span class="red">进程调度</span>器抢占

以下为Linux 5.15 LTS内核中，标记中断上下文进入与退出的核心源码：
```c
// 源码路径：linux-5.15/include/linux/hardirq.h
// 中断上下文进入核心宏，标记当前CPU进入硬中断环境
#define __irq_enter()                                        \
    do {                                                      \
        account_hardirq_enter(current);                      \
        // 增加硬中断计数，标记进入中断上下文
        __preempt_count_add(HARDIRQ_OFFSET);                 \
        lockdep_hardirq_enter();                              \
    } while (0)

// 中断上下文退出核心宏，清除硬中断环境标记
#define __irq_exit()                                         \
    do {                                                      \
        lockdep_hardirq_exit();                               \
        // 减少硬中断计数，标记退出中断上下文
        __preempt_count_sub(HARDIRQ_OFFSET);                 \
        account_hardirq_exit(current);                       \
    } while (0)
```
代码关键说明：
- 内核通过`preempt_count`变量的高位，标记当前CPU的运行上下文类型
- `HARDIRQ_OFFSET`是硬中断上下文的标记位，非0时代表当前处于中断上下文
- 该变量是内核判断当前是否可睡眠、是否可调度的核心依据

执行以下命令，可验证系统中断上下文的总执行次数：
```bash
cat /proc/stat | grep intr
```
典型输出与解读：
```
intr 52894021 23894021 34021 1892 120384 0 0 0 0 0 ...
```
输出中第一个数字`52894021`，是系统启动以来所有CPU的硬中断总触发次数，每一次触发都对应一次完整的中断上下文进入与退出流程。<br>

### <strong>中断栈的切换与保护</strong>

我们可以将中断栈的切换与保护，比作调度员使用专属的紧急事件处置台账。
处理紧急事件时，必须使用独立的专用台账，不可与日常调度工作的台账混用。避免日常工作数据被紧急处置操作污染，保障紧急事件处置的独立性。
事件处置完成后，立即切回日常工作台账，专用台账封存留档。

中断栈（Interrupt Stack），是内核为<span class="red">中断处理</span>程序分配的、独立的专属内存栈空间。
它用于中断上下文执行时的函数调用、局部变量存储、执行现场保存。
它与进程的用户栈、内核栈完全隔离，是<span class="red">中断处理</span>流程稳定性的核心保障。
```
┌──────────────────────────────────┐
│         ARM64 Linux 中断栈内存布局                                 │
├──────────────────────────────────┤
│  高地址  ┌─────────────────┐ 每个CPU核心独立拥有│
│          │  CPU0中断栈                      │  固定大小：16KB    │
│          ├─────────────────┤                    │
│          │  CPU1中断栈                      │  栈增长方向：向下  │
│          ├─────────────────┤                    │
│          │  CPU2中断栈                      │与进程内核栈完全隔离│
│          ├─────────────────┤                    │
│          │  CPU3中断栈                      │  带栈溢出防护页    │
│  低地址  └─────────────────┘                    │
└──────────────────────────────────┘
```

ARM64架构的Linux内核，为每个CPU核心分配了独立的中断栈，固定大小为16KB。
中断栈底部设置了不可访问的防护页，用于检测栈溢出，避免内存污染。CPU响应中断后，会自动从进程内核栈，切换到当前CPU的专属中断栈。
<span class="red">中断处理</span>完成后，再切回之前的进程内核栈，保障进程栈的数据完整性。

以下为Linux 5.15 LTS内核中，ARM64架构中断栈切换的核心汇编代码：
```c
// 源码路径：linux-5.15/arch/arm64/kernel/entry.S
// 中断栈切换核心宏，IRQ中断入口处执行
.macro  irq_stack_entry
    // 读取中断栈数组起始地址，存入x1寄存器
    adrp    x1, irq_stack
    add     x1, x1, :lo12:irq_stack
    // 读取当前CPU编号，定位对应CPU的中断栈
    mrs     x2, tpidr_el1
    add     x1, x1, x2
    // 计算中断栈顶地址，每个CPU栈大小为16KB
    mov     x2, #IRQ_STACK_SIZE
    add     x1, x1, x2
    // 切换栈指针SP到中断栈，完成栈切换
    mov     sp, x1
.endm
```
代码关键说明：
- `irq_stack`是内核定义的中断栈数组起始地址，每个CPU对应16KB的独立栈空间
- `tpidr_el1`是ARMv8系统寄存器，存储当前CPU编号，用于定位对应中断栈
- `IRQ_STACK_SIZE`固定为16KB，是ARM64 Linux中断栈的标准大小
- 该宏在IRQ中断入口处执行，完成从进程栈到中断栈的无缝切换

中断栈的核心保护机制，包括：
- 每个CPU独立栈，避免多核<span class="red">并发</span>访问的竞态问题
- 栈底不可访问防护页，触发栈溢出时直接抛出异常，避免内存污染
- 固定大小限制，避免<span class="red">中断处理</span>程序过度消耗栈空间
- 栈切换过程自动保存原栈指针，中断退出时可完整恢复

执行以下命令，可验证当前内核中断栈的配置大小：
```bash
zcat /proc/config.gz | grep IRQ_STACK_SIZE
```
典型输出与解读：
```
CONFIG_IRQ_STACK_SIZE=16384
```
输出中的`16384`字节（即16KB），与ARM64 Linux中断栈的标准大小完全匹配。<br>
