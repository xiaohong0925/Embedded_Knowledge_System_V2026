### <strong>中断的定义与核心价值</strong>

我们可以将嵌入式系统中的CPU核心，比作城市轨道交通的行车调度室。
调度员日常执行的列车运行监控、站点停靠调度，对应CPU串行执行的主程序任务流。
轨道区间的异物告警、列车紧急制动信号、突发客流调度指令，对应向CPU发出的中断信号。

从技术层面，中断是硬件外设或软件逻辑，向CPU核心发出的异步紧急事件信号。
它是嵌入式系统实现事件驱动架构、保障实时响应能力的核心基础机制。

异步，指事件的发生与CPU当前执行的指令流无固定时序关联，可在程序执行的任意时刻触发。
程序计数器（PC，Program Counter），用于记录CPU下一条待执行指令的内存地址，是程序执行的核心指针。
<span class="red">中断处理</span>程序（ISR，Interrupt Service Routine），是预先定义的、用于响应特定中断事件的专用代码段。

中断触发后，CPU的完整执行流程如下：
```
┌────────────────────────┐
│  主程序正常执行流                              │
│  指令1 → 指令2 → 指令3 → 【指令4（被打断）】│
└────────────┬───────────┘
                          ↓
                    ┌───────┐
                    │ 中断信号触发 │
                    └───┬───┘
                            ↓
            ┌───────────────┐
            │ 保存当前CPU执行现场          │
            │ 跳转到对应中断处理程序(ISR)  │
            │ 执行ISR完成事件响应          │
            │ 恢复之前保存的执行现场       │
            └───────┬───────┘
                            ↓
┌─────────────────────────┐
│  主程序恢复执行：指令4 → 指令5 → 指令6...      │
└─────────────────────────┘
```

在Linux系统中，所有已注册的中断及其运行状态，都可通过`/proc`文件系统原生接口直接观测。
无需额外工具，执行以下命令即可查看系统中断的基础统计信息：
```bash
cat /proc/interrupts | head -10
```

该命令的典型输出如下：
```
           CPU0       CPU1       
  0:   23894021          0  GICv2  30 Level     arch_timer
  1:      34021          0  GICv2  44 Level     uart-pl011
  2:       1892          0  GICv2  45 Level     gpio-mxc
  3:     120384          0  GICv2  46 Level     eth0
```

输出各字段的核心含义如下：
- 第1列：中断号（IRQ，Interrupt Request），系统为每个中断分配的唯一标识
- 第2-3列：对应CPU核心上，该中断的累计触发次数
- 第4-6列：中断控制器型号、硬件中断号、触发方式
- 最后一列：该中断对应的外设名称

在嵌入式场景中，中断机制的核心价值，体现在三个不可替代的优势：
- 极致的CPU利用率：无事件发生时，CPU可执行主任务或进入低功耗休眠，无额外资源消耗
- 确定的实时响应能力：事件触发后立即打断主程序，处理延迟可控制在微秒级，满足强实时场景需求
- 天然的多外设并行支持：可为多个外设注册独立中断，实现多事件的并行响应与优先级调度<br>

### <strong>中断与轮询的场景对比</strong>

延续轨道交通调度的场景，可清晰区分两种事件处理机制的核心差异。
中断驱动模式下，只有真实触发紧急告警时，调度员才会暂停常规工作处置。
轮询（Polling）模式下，调度员必须每隔固定周期，全量巡检所有线路与列车状态。

从技术层面，轮询是一种同步周期性的事件处理机制。
程序会按照预设的固定时间周期，循环读取外设的状态寄存器，判断是否有待处理事件。

两种机制的核心差异与嵌入式场景选型边界，可通过下表清晰区分：
| 对比维度       | 中断驱动机制                                                 | 轮询查询机制                                                |
| :---           | :---                                                         | :---                                                        |
| CPU资源占用    | 事件触发时才占用CPU，无事件时空闲占用趋近于0                 | 需按周期循环查询，无事件时也持续占用CPU资源                 |
| 事件响应实时性 | 事件触发立即响应，延迟固定，实时性确定                       | 响应延迟取决于轮询周期，最差延迟等于一个完整周期            |
| 实现复杂度     | 需配套中断硬件与内核子系统支持，有上下文约束                 | 逻辑极简，仅需循环判断状态位，无额外机制依赖                |
| 事件丢失风险   | 只要中断不被屏蔽，事件触发必然被捕获，风险极低               | 单周期内事件发生后消失，下一次查询无法捕获，存在丢失风险    |
| 核心适用场景   | 按键输入、传感器就绪、CAN/以太网报文接收等低频次高实时性事件 | LED刷新、外设状态巡检、高频数据批量读取等高频次低实时性事件 |

嵌入式产品开发中，两种机制的选型可遵循明确的决策规则：
- 事件发生频率低于100Hz，且对响应延迟有明确要求时，优先选择中断驱动
- 事件发生频率高于1KHz，且事件处理逻辑极简时，优先选择轮询机制
- 电池供电的低功耗设备，默认首选中断驱动，最大程度降低空闲功耗

读者可通过Linux系统原生的`top`命令，直观验证两种机制的CPU占用差异。
同样的按键输入场景，中断驱动方案空闲时CPU占用率低于1%。
10ms周期的轮询方案，CPU固定占用率超过5%，设备休眠功耗提升3倍以上。<br>

### <strong>中断的核心分类（外部/内部）</strong>

延续轨道交通调度的场景，可清晰划分中断的两大核心类别。

外部中断对应来自线路区间、列车、站点的外部告警，事件来源在调度室之外。
内部中断对应调度室内部的设备故障、系统告警，事件来源在调度室内部。

按照中断信号的来源与触发方式，Linux系统将中断划分为两大类，核心边界清晰无交叉。
- 外部硬件中断，简称外部中断，信号来自CPU核心之外的硬件外设。
  它是嵌入式场景中最常用的中断类型，完全异步，可在CPU执行的任意时刻触发。
- 内部软件中断，简称内部中断或异常，信号来自CPU核心内部。
  它是CPU执行指令过程中触发的事件，与当前执行的指令流强相关，多为同步触发。

两类中断的典型应用场景如下：
```
┌──────────────────────────┐
│                Linux 中断核心分类                  │
├─────────────┬────────────┤
│      外部硬件中断        │      内部软件中断      │
│  （异步，CPU核外触发）   │ （同步/异步，核内触发）│
├─────────────┼────────────┤
│ ▶ GPIO按键/旋钮输入     │ ▶ CPU核内定时器溢出   │
│ ▶ 传感器数据就绪中断    │ ▶ 内存访问缺页异常    │
│ ▶ CAN/以太网报文接收    │ ▶ 除法零错误执行异常  │
│ ▶ UART串口数据接收      │ ▶ 系统调用软中断      │
│ ▶ SPI/I2C外设事件       │ ▶ 多核IPI核间中断     │
└─────────────┴────────────┘
```

读者可通过之前`/proc/interrupts`的输出，直观区分两类中断。
其中`arch_timer`（内核全局定时器）、`IPI`（多核核间中断），属于CPU核内触发的内部中断。
而`uart-pl011`（串口外设）、`gpio-mxc`（<span class="green">GPIO</span>外设）、`eth0`（以太网外设），属于典型的外部中断。

两类中断的核心边界与应用场景有明确区分。
外部中断是嵌入式外设事件响应的核心方案，是驱动开发中最常接触的中断类型。
内部中断主要用于处理CPU执行错误、系统调用等内核核心事件，普通驱动开发极少直接干预。
其底层实现逻辑详见3.6.2节内核<span class="red">中断处理</span>机制。<br>

### <strong>GICv2/v3 核心能力差异</strong>

我们可以将ARM架构的GIC，比作城市轨道交通的线网调度指挥中心。
CPU核心对应线路上的各个运营站台，外设中断信号对应列车的进站停靠请求。
GIC的核心职责，是接收全线路的进站请求，完成优先级仲裁后，分发到对应站台。

GIC（Generic Interrupt Controller，通用中断控制器），是ARM架构官方定义的标准中断控制器硬件。
它是CPU核心与外部外设中断之间的唯一硬件桥梁，是ARM中断系统的核心硬件基础。
它负责中断信号的采集、优先级仲裁、多核路由分发、虚拟化支持等核心功能。

```
┌────────────────────────┐
│                ARM GIC 硬件架构                │
├──────┬────────┬────────┤
│  外设中断源│  中断分发层    │  CPU核心集群   │
│  (SPI/PPI) │  (Distributor) │  (CPU0~CPUn)   │
└──────┴────────┴────────┘
```

GICv2是ARMv7-A、ARMv8-A早期架构的主流中断控制器，最多支持8个CPU核心。
GICv3是ARMv8.0及以上架构的官方标准控制器，原生支持大核数集群、虚拟化与安全中断隔离。

二者的核心能力差异，可通过下表清晰区分：
| 对比维度           | GICv2                                    | GICv3                                      |
| :------------------| :----------------------------------------| :------------------------------------------|
| 支持最大CPU核心数  | 最多8核                                  | 最多128核，支持多集群互联                  |
| 架构适配           | ARMv7-A、ARMv8-A早期架构                 | ARMv8.0及以上全系列架构                    |
| 虚拟化支持         | 需软件模拟，性能损耗大                   | 原生硬件虚拟化支持，虚拟中断直通           |
| 安全隔离           | 仅支持2个安全世界隔离                    | 支持TrustZone多安全域精细化隔离            |
| 中断号范围         | 最多支持1020个硬件中断                   | 支持LPI扩展，最多可支持百万级中断          |
| 嵌入式主流应用场景 | 工业控制、消费级嵌入式设备（如树莓派4B） | 车载、边缘计算、高端嵌入式设备（如RK3588） |

在嵌入式Linux系统中，可通过以下命令，直接查看当前开发板使用的GIC硬件版本：
```bash
dmesg | grep -i "gic version"
```
该命令的典型输出与解读如下：
```
[    0.000000] GICv2: Probed GIC version 2 on (null)
```
输出中的`GIC version 2`，即代表当前硬件使用的是GICv2控制器。

读者可在任意ARM架构开发板上执行该命令，验证硬件GIC版本。
比如树莓派4B默认使用GICv2，瑞芯微RK3588、NXP i.MX8系列默认使用GICv3。<br>

### <strong>中断触发方式的硬件定义</strong>

延续轨道交通调度的场景，中断触发方式，就是列车向调度中心发送进站请求的硬件规则。
电平触发，是列车一直停在进站口，持续发送请求信号，直到调度中心响应放行。
边沿触发，是列车到达进站口时，只发送一次单次脉冲请求信号，无论是否收到响应。

中断触发方式，是硬件层面定义的、中断信号有效的电气规则。
它决定了中断控制器，在什么电气条件下，会识别到一个合法的中断请求。
这是中断硬件架构的核心基础，直接决定了中断的可靠性与适用场景。
```
# 电平触发（Level Triggered）时序
信号电平  ─────┐                    ┌────────
                    │                    │
                    └──────────┘
                    ↓                    ↑
                中断有效              中断无效
                （持续有效电平）

# 边沿触发（Edge Triggered）时序
信号电平  ─────┐                    ┌────────
                    │                    │
                    └──────────┘
                    ↓
                跳变沿触发
                （单次脉冲有效）
```

电平触发的核心硬件规则：中断信号持续保持在预设的有效电平（高/低）时，中断请求持续有效。
只要中断控制器采样到有效电平，就会持续向CPU转发中断请求，直到信号恢复无效电平。
它的核心优势是不会丢失中断，适合UART、<span class="green">SPI</span>等低速外设，以及共享中断场景。

边沿触发的核心硬件规则：中断信号在预设的跳变沿（上升沿/下降沿）时，产生单次有效的中断请求。
中断控制器仅在跳变发生的瞬间，识别一次中断请求，后续电平保持不会触发新的中断。
它的核心优势是响应速度快，适合<span class="green">GPIO</span>按键、脉冲计数器等高频单次事件场景。

在嵌入式Linux系统中，可通过以下命令，查看所有中断的硬件触发方式：
```bash
cat /proc/interrupts | grep -E "Level|Edge"
```
该命令的典型输出与解读如下：
```
  1:      34021          0  GICv2  44 Level     uart-pl011
  2:       1892          0  GICv2  45 Edge      gpio-mxc
```
输出中第7列的`Level/Edge`，即为该中断的硬件触发方式。
示例中串口外设使用电平触发，<span class="green">GPIO</span>外设使用边沿触发，完全匹配场景选型规则。<br>

### <strong>虚拟中断ID与物理ID映射</strong>

延续轨道交通调度的场景，物理中断ID对应列车的全国统一真实车次号，是全路唯一的硬件标识。
虚拟中断ID对应调度中心给各个站台分配的内部运营编号，仅在站台内部生效。
ID映射，就是调度中心维护的、真实车次号与内部运营编号的一一对应关系表。

物理中断ID，是中断控制器硬件层面，为每一个中断源分配的唯一硬件编号。
它是外设中断信号的硬件身份标识，在整个中断控制器硬件中全局唯一，不可修改。

虚拟中断ID，是Linux内核或虚拟化层，为上层软件分配的、逻辑层面的中断编号。
它是软件层面可见的中断标识，也就是我们在`/proc/interrupts`中看到的系统中断号（IRQ号）。
```
┌────────────────────────┐
│         物理ID ↔ 虚拟ID 硬件映射关系          │
├──────────┬─────────────┤
│  外设硬件中断源    │  物理中断ID（硬件唯一）  │
├──────────┼─────────────┤
│  GIC中断控制器     │  硬件映射表（ID路由）    │
├──────────┼─────────────┤
│  Linux内核/CPU核心 │  虚拟中断ID（软件可见）  │
└──────────┴─────────────┘
```

ID映射机制的核心价值，体现在三个不可替代的作用：
- 硬件解耦：软件无需关注底层硬件的物理中断号，仅需操作虚拟ID，实现驱动的跨平台兼容
- 虚拟化支持：ARM虚拟化场景中，可为每个虚拟机分配独立的虚拟中断ID，实现中断硬件隔离
- 多核路由：通过ID映射，可将同一个物理中断，路由到不同的CPU核心，实现中断负载均衡

在嵌入式Linux系统中，可通过以下命令，直接查看虚拟ID与物理ID的映射关系：
```bash
cat /proc/interrupts | awk '{print "虚拟ID(IRQ): "$1" | 物理ID: "$6" | 外设: "$8}' | head -10
```
该命令的典型输出与解读如下：
```
虚拟ID(IRQ): 1 | 物理ID: 44 | 外设: uart-pl011
虚拟ID(IRQ): 2 | 物理ID: 45 | 外设: gpio-mxc
```
输出清晰展示了软件可见的虚拟IRQ号，与硬件层面物理中断号的一一对应关系。

读者可在自己的ARM开发板上执行该命令，验证不同外设的ID映射规则。
比如同一款串口外设，在不同硬件平台上的物理ID可能不同，但虚拟ID可通过内核适配保持一致。<br>
