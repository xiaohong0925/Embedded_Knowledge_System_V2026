### <strong>进程：内核资源分配的基本单元</strong>

分清进程和程序：知道 “led_blink.bin是程序，./led_blink &是进程，进程占 CPU/<span class="green">GPIO</span> 资源”；<br>
要理解进程，我们先从最熟悉的嵌入式场景切入
——你在开发板上编译运行一个“LED点亮程序”时，
这个存放在Flash里的`led_blink.bin`只是程序（Program） ，是一堆静态的指令和数据；

——当你执行`./led_blink &`后，内核会为它分配CPU时间、<span class="green">GPIO</span>引脚、内存空间等资源，
此时它就变成了进程（Process） ，是动态运行的指令集合。

简单说，进程是内核进行资源分配的最小单位，内核通过管理进程来管理硬件资源的使用。

--------------------------------------------------------------------------------------------------
在嵌入式系统中，进程的“资源分配”特性体现在每一个细节里：
- 温控进程（如`thermal-daemon`）：
    内核会为它分配固定的CPU时间片（确保周期性采样温度），并授权它访问温度传感器的IIO（工业I/O）接口资源，同时限制它的内存占用（避免影响核心业务）；
- LED控制进程：
    除了CPU和内存，内核会为它绑定特定的<span class="green">GPIO</span>引脚（通过<span class="red">设备树</span>或驱动分配），并通过文件系统抽象（`/sys/class/gpio`）让进程间接操作硬件，避免多个进程同时抢占同一GPIO导致冲突。
--------------------------------------------------------------------------------------------------
这里需要区分一个关键概念：进程和程序的核心差异是“动态性”与“资源占有权”

1. 程序是静态文件，不占用硬件资源；
2. 进程是程序的运行实例，拥有独立的资源集合（CPU、内存、I/O设备等）。
你可以在开发板上执行`ps -ef | grep led`看到这个LED进程的信息，这就是内核对进程的管理结果。<br>

### <strong>调度：内核资源调度的核心机制</strong>

理解调度的作用：知道 “多个进程抢 CPU 时，调度器按优先级分配，实时进程比普通进程优先”；<br>
嵌入式系统里很少只有一个进程在运行
——比如一个物联网网关，可能同时运行“网络数据转发进程”“本地数据解析进程”“日志记录进程”和“LED状态指示进程”。
这些进程都需要CPU资源，但CPU同一时间只能执行一个任务（单核场景），

这就需要一个“管理者”来决定：哪个进程先用CPU？用多久？遇到紧急任务时怎么切换？ 
- 这个管理者就是内核的调度器（Scheduler） ，调度的核心作用是“分配CPU时间资源，平衡系统目标”。

嵌入式系统的调度目标和服务器不同，更强调“实时响应、资源高效、低功耗”三个核心诉求，我们用具体场景解释：
- 实时响应（最关键）：
工业控制中的电机转速控制进程，要求每次接收到传感器数据后，必须在1ms内完成计算并输出控制信号
——调度器需要优先分配CPU给这个进程，避免被日志进程等非核心任务抢占；
- 资源高效：
在128M内存的低成本开发板上，同时运行3个进程时，调度器要避免某个进程长时间占用CPU导致其他进程“饿死”，比如让每个进程轮流使用CPU，确保资源利用率最大化；
- 低功耗：
手持设备（如智能手环）的心率采样进程，只需要每1秒运行一次（每次运行5ms），调度器可以在进程休眠时让CPU进入深度睡眠模式，减少功耗。

举个通俗的例子：
调度器就像食堂打饭窗口的收银员，进程是排队的学生。
收银员（调度器）要根据“紧急程度”（进程优先级）安排打饭顺序
——赶课的学生（实时进程）优先，普通学生（普通进程）按顺序排队；
同时控制每个学生的打饭时间（时间片），避免后面的人等太久。<br>

### <strong>进程、线程（LWP）与调度的关系</strong>

随着嵌入式系统功能越来越复杂，单一进程的“资源独占”特性会带来效率问题

——比如一个音视频处理进程，既要采集音频、编码视频，又要传输数据，若用一个进程串行执行，会导致延迟过高。
这时就需要线程（Thread） ，也叫轻量级进程（LWP，Lightweight Process）。

线程的核心特点是“共享进程资源，独立调度”：
- 共享资源：
同一进程的多个线程，会共享进程的内存空间（如全局变量、代码段）、打开的文件描述符（如网络socket）和I/O权限（如音频设备）；
- 独立调度：
每个线程有自己的执行路径和CPU上下文（寄存器、程序计数器），调度器可以单独为线程分配CPU时间，相当于“调度粒度更细”。

在嵌入式音视频处理场景中，线程的优势非常明显：
你可以创建3个线程：“音频采集线程”“视频编码线程”“数据传输线程”，们共享音视频缓存内存（避免数据拷贝），
调度器可以根据实时性需求分配优先级：让“音频采集线程”优先级最高（避免漏采数据），“视频编码线程”次之，“数据传输线程”最低（网络延迟可容忍）。

这里用命令验证线程与进程的关系：
在开发板上运行一个多线程程序（如`ffmpeg`转码视频），
执行`ps -L -p [进程PID]`，会看到同一PID下有多个LWP（线程ID），这就是调度器的调度对象

——内核调度的最小单位是线程，而不是进程。
比如一个进程包含3个线程，调度器会把它们当作3个独立的“调度实体”分配CPU时间，而它们共享的进程资源则由内核统一管理，避免冲突<br>

### <strong>进程树与标识体系</strong>

嵌入式系统的进程不是孤立的，而是以“树状结构”组织的
——就像家族树，有“根进程”“父进程”“子进程”，这种结构叫进程树（Process Tree） ，核心是通过PID和PPID关联。

（1）进程树的核心规则
- 根进程：系统启动后创建的第一个进程（PID=1，如systemd或init），是所有进程的“祖先”；  
- 父子关系：每个进程（除根进程）都由一个父进程创建，父进程可以创建多个子进程，子进程还可以创建“孙进程”；  
- 资源继承：子进程会继承父进程的部分资源（如打开的文件、环境变量、调度策略），比如父进程用`chrt`设置为实时策略后，创建的子进程默认也是实时策略。

（2）用命令可视化进程树
嵌入式系统中最直观的工具是`pstree`，直接展示进程间的父子关系：  
- 实操步骤1：在开发板执行`pstree`，会看到类似如下输出（简化版）：  
  ```
  systemd(1)───bash(200)───led_blink(123)
              ├── thermal-daemon(50)
              ├── networkd(60)
              └── sshd(80)───sshd(250)───bash(251)
  ```
- 输出解读：  
  - 根进程是`systemd(1)`，创建了`bash(200)`（终端进程）、`thermal-daemon(50)`（温控进程）等；  
  - `bash(200)`创建了我们运行的`led_blink(123)`（子进程）；  
  - `sshd(80)`（SSH服务进程）创建了`sshd(250)`（当前SSH连接进程），后者又创建了`bash(251)`（SSH终端进程）。

（3）进程树的实操查询
除了`pstree`，还可以用`ps`命令查看进程树的关联关系：  
- 命令1：`ps -efH`，`-H`选项表示“分层显示”，子进程会缩进显示，清晰看到父子关系；  
- 命令2：`ps -eo pid,ppid,comm --forest`，`--forest`选项用竖线和横线展示进程树结构，更直观。<br>

### <strong>进程状态机：核心五状态及调度触发</strong>

进程的状态会随着“自身行为”和“内核调度”动态变化，比如“等待IO”时从运行变阻塞，“IO完成”后从阻塞变就绪。
嵌入式系统中，进程的核心状态有5种，构成进程状态机（Process State Machine），状态转换是调度器触发的核心场景。

（1）核心五状态及含义
我们结合嵌入式场景，用“LED控制进程”和“网络转发进程”举例说明：

| 状态常量（内核）     | 状态标识（ps命令） | 核心含义                                                                 | 嵌入式场景举例                     |
|----------------------|--------------------|--------------------------------------------------------------------------|------------------------------------|
| TASK_RUNNING         | R                  | 就绪/运行：要么正在用CPU，要么在“就绪队列”等待CPU（参与调度）          | LED进程正在闪烁（运行）；等待CPU（就绪） |
| TASK_INTERRUPTIBLE   | S                  | 可中断等待：等待某个事件（如IO完成、信号），事件发生后被唤醒为就绪状态   | 网络进程等待接收数据（等待网卡IO） |
| TASK_UNINTERRUPTIBLE | D                  | 不可中断等待：等待关键IO（如磁盘读写），期间不响应信号（避免数据损坏）   | 日志进程向Flash写入日志（等待Flash IO） |
| TASK_STOPPED         | T                  | 已暂停：收到暂停信号（如SIGSTOP），需恢复信号（如SIGCONT）才能继续       | 用`kill -19 123`暂停LED进程        |
| EXIT_ZOMBIE          | Z                  | 僵尸状态：进程已退出，但父进程未回收资源（task_struct未释放）            | 子进程退出后，父进程未调用wait回收 |

（2）状态转换的核心场景（调度触发时机）
状态转换是调度器介入的关键，嵌入式系统中最常见的3种转换：  
1. 运行→可中断等待：进程需要等待IO（如网络进程等待数据），主动放弃CPU，进入等待状态
   ——调度器会选择就绪队列中的下一个进程运行；  
2. 可中断等待→就绪：IO事件完成（如网卡收到数据），内核唤醒进程，将其状态改为就绪
   ——调度器会判断该进程优先级是否高于当前运行进程，若高于则触发抢占；  
3. 运行→就绪：进程的时间片耗尽（CFS调度器分配的时间到），
   ——调度器将其转为就绪状态，选择下一个vruntime最小的进程运行。

（3）用命令查看进程状态
通过`ps`命令的`STAT`字段查看状态：  
- 实操步骤：执行`ps -eo pid,stat,comm`，输出类似：  
  ```
  PID  STAT  COMM
  1    Ss    systemd
  50   S     thermal-daemon
  123  R+    led_blink
  200  Ss+   bash
  ```
- 字段解读：  
  - `S`：TASK_INTERRUPTIBLE（可中断等待）；  
  - `Ss`：`S`表示状态，`s`表示该进程是“会话首进程”（如systemd）；  
  - `R+`：`R`表示TASK_RUNNING（运行），`+`表示该进程在前台运行；  
  - 若执行`kill -19 123`暂停LED进程，再执行命令会看到`STAT`变为`T`（TASK_STOPPED）。<br>

### <strong>僵尸进程：形成原因、风险与回收</strong>

（1）僵尸进程的形成原因
嵌入式系统中，进程退出后不会立刻消失
——进程执行`exit()`系统调用后，会释放内存、IO等资源，但`task_struct`（进程控制块）会保留（里面存着退出状态码等信息），直到父进程通过`wait()`或`waitpid()`系统调用“回收”这些信息，内核才会释放`task_struct`。  
如果父进程没有调用`wait()`，`task_struct`就会一直残留，进程变成僵尸进程（EXIT_ZOMBIE）。

（2）嵌入式场景的核心风险
僵尸进程本身不占用CPU和内存（资源已释放），但会占用PID编号
——嵌入式系统的PID是有限的（默认最大32768，低端设备可能更小），若大量僵尸进程堆积，会导致新进程无法创建（PID耗尽）；同时，`task_struct`占用少量内核内存，长期堆积会导致内核内存泄漏。

（3）僵尸进程的实操复现与回收
我们用简单代码和命令复现僵尸进程，并验证回收方法：  
1. 复现僵尸进程（嵌入式C代码，交叉编译后运行）：  
   ```c
   #include <stdio.h>
   #include <unistd.h>
   #include <sys/wait.h>

   int main() {
       pid_t pid = fork();  // 父进程创建子进程
       if (pid == 0) {
           // 子进程：执行后立刻退出
           printf("子进程PID：%d，父进程PID：%d\n", getpid(), getppid());
           return 0;  // 子进程退出，未被回收
       } else if (pid > 0) {
           // 父进程：不调用wait()，一直休眠
           printf("父进程PID：%d\n", getpid());
           while(1) sleep(1);  // 无限休眠，不回收子进程
       }
       return 0;
   }
   ```
2. 实操步骤：  
   - 编译运行：`arm-linux-gnueabihf-gcc zombie.c -o zombie && ./zombie`；  
   - 查看僵尸进程：执行`ps -eo pid,stat,comm`，会看到子进程的`STAT`为`Z`（僵尸状态），标注`[zombie] <defunct>`（defunct表示僵尸）；  
3. 回收方法：  
   - 方法1：修改父进程代码，在`while(1)`前添加`wait(NULL);`（主动回收子进程），重新运行后，子进程退出后不会变成僵尸；  
   - 方法2：杀死父进程（`kill -9 父进程PID`），僵尸进程会被根进程（PID=1）接管并回收。<br>

### <strong>调度器核心目标（嵌入式专属诉求）</strong>

内核调度器的本质是“CPU时间的分配者”，
但嵌入式系统的调度目标和服务器（追求高吞吐量）、PC（追求多任务流畅）有显著差异，核心围绕“实时响应、资源高效、低功耗” 三个诉求展开
——这三个诉求直接决定了嵌入式产品的“可用性”（比如工业设备的控制精度）、“成本”（比如低成本开发板的资源利用率）和“续航”（比如手持设备的待机时间）。

我们用嵌入式三大典型场景解释目标优先级：
- 实时响应（最高优先级）：
工业控制中的电机转速调节进程，需要每10ms采集一次编码器数据并输出PWM控制信号，若调度器让该进程等待超过1ms，就会导致电机转速波动（误差超过5%）。
此时调度器必须“优先保障实时进程的CPU使用权”，哪怕牺牲非核心进程（如日志打印）的运行机会；

- 资源高效（中优先级）：
128M内存的物联网网关，同时运行“MQTT数据转发进程”“本地温度采样进程”“LED状态指示进程”三个任务。
调度器要避免某一个进程“独占CPU”导致其他进程“饿死”——比如让每个进程轮流使用CPU（每次运行10ms切换），确保三个任务都能正常执行，资源利用率维持在80%以上；

- 低功耗（场景化优先级）：
智能手环的心率采样进程，仅需每2秒运行一次（每次运行10ms采集数据并计算），其余时间可让CPU进入“深度睡眠模式”（功耗从100mA降至5mA）。
调度器要在进程休眠时主动触发CPU休眠，在进程唤醒时快速分配CPU，平衡“响应速度”和“功耗”。

举个通俗的比喻：嵌入式调度器就像工厂的“生产线调度员”
——要优先安排“紧急订单”（实时进程）上线，确保交货时间；
要让每条生产线都有活干（资源高效），不浪费产能；
在没有订单时关掉部分生产线（低功耗），节省电费。<br>

### <strong>调度器基础分类（嵌入式常用3类）</strong>

Linux内核支持多种调度器，但嵌入式系统中99%的场景只用到3类核心调度器，
按“优先级从高到低”排序为：实时调度器→通用调度器→idle调度器。

调度器之间的优先级是“抢占式”的
——高优先级调度器管理的进程，能打断低优先级进程的运行。

（1）实时调度器（管理“紧急任务”）
- 核心定位：管理实时进程（对响应延迟有严格要求的进程，如控制、采样、音视频编码），优先级最高，能抢占其他所有进程的CPU使用权；
- 细分类型（嵌入式常用2种）：
  1. SCHED_FIFO（先进先出实时调度）：
“一旦运行，就独占CPU直到主动放弃”——适合“运行时间短、不可被打断”的任务，比如电机控制进程（每次运行5ms就主动休眠等待下一个周期）；
  2. SCHED_RR（时间片轮转实时调度）：
“同优先级进程轮流使用CPU，每个进程有固定时间片”——适合“需要周期性运行、可短暂打断”的任务，比如传感器采样进程（每个进程每次运行10ms，优先级相同的红外采样和温湿度采样轮流执行）；

- 嵌入式场景适配：
实时调度器的进程优先级范围是1-99（数值越大优先级越高），通常把“核心控制进程”设为90以上（如电机控制设99），“辅助采样进程”设为50-80（如温湿度采样设60）。

（2）通用调度器（管理“普通任务”）
- 核心定位：管理普通进程（无严格实时要求的进程，如日志打印、终端交互、文件拷贝），优先级低于实时调度器，不能抢占实时进程；
- 核心类型：CFS（完全公平调度器）——Linux 2.6.23后成为默认通用调度器，核心思想是“让每个普通进程获得‘公平的CPU时间’”；
- 通俗理解：CFS像“分蛋糕”，把CPU时间当作一块蛋糕，每个普通进程按“优先级权重”分蛋糕
——优先级高的进程分得多（比如nice值-20的进程比nice值19的进程多分配4倍时间），但不会让某个进程独占；

- 嵌入式场景适配：普通进程的优先级通过`nice`值调节（范围-20~19，值越小优先级越高），比如“日志进程”设为nice=10（少分CPU），“本地数据解析进程”设为nice=0（正常分配）。

（3）idle调度器（管理“CPU空闲时间”）
- 核心定位：当CPU没有实时进程和普通进程需要运行时，idle调度器会让CPU运行“idle进程”（每个CPU核对应一个idle进程，PID固定为0）；
- 嵌入式核心价值：idle进程不是“空转”，而是会触发CPU进入“低功耗模式”（如ARM的WFI指令，让CPU停止时钟等待中断）——这是嵌入式设备“待机功耗降低80%”的核心手段；
- 场景举例：智能手环在没有心率采样、屏幕显示时，所有核心都运行idle进程，CPU进入深度睡眠，功耗从100mA降至5mA。<br>

### <strong>用户态感知工具实操（3类核心工具）</strong>

嵌入式开发中，不需要修改内核就能“看到调度器工作状态”
——通过`ps`（进程状态查询）、`top`（实时监控）、`/proc`文件系统（调度参数读取）三类工具，就能直观感知进程的调度策略、优先级和运行状态。
以下操作均在ARM嵌入式开发板（如树莓派、IMX6ULL）上验证，Windows/Mac可通过SSH连接开发板操作。

（1）ps命令：查询<span class="red">进程调度</span>核心信息
`ps`是“进程状态查询”的基础工具，通过指定参数可精准提取“调度策略、优先级、状态”等核心字段，
嵌入式场景最常用的是`ps -eo pid,stat,niceness,prio,comm`（字段含义：进程ID、状态、nice值、优先级、进程名）。

- 实操步骤1：在开发板运行3个测试进程（模拟嵌入式典型场景）：
  1. 普通进程：`while true; do echo "log" >> /tmp/test.log; sleep 1; done &`（日志打印进程，默认SCHED_OTHER策略）；
  2. 实时进程：`chrt -f 99 while true; do :; done &`（SCHED_FIFO策略，优先级99的空循环进程）；
  3. 休眠进程：`while true; do sleep 5; done &`（周期性休眠进程，默认策略）；
- 实操步骤2：执行`ps -eo pid,stat,niceness,prio,comm`，输出类似如下（关键字段已标注）：
  ```
  PID   STAT   NI PRI COMM
  123   R      0  120 log_process  # 日志进程：R（运行）、NI=0（默认nice）、PRI=120（普通优先级）
  124   R      -   99 chrt         # 实时进程：R（运行）、NI=-（实时进程无nice值）、PRI=99（实时优先级）
  125   S      0  120 sleep        # 休眠进程：S（可中断等待）、NI=0、PRI=120
  ```
- 输出解读（调度核心信息）：
  1. `STAT`（进程状态）：`R`表示TASK_RUNNING（运行/就绪）、`S`表示TASK_INTERRUPTIBLE（休眠等待）
  ——实时进程124一直是`R`（因为空循环不释放CPU），日志进程123偶尔`R`（被实时进程抢占）；
  2. `NI`（nice值）：普通进程有值（-20~19），实时进程无值（显示`-`）
  ——日志进程123是默认nice=0；
  3. `PRI`（优先级）：实时进程优先级（1-99）低于普通进程（100-139）
  ——实时进程124的PRI=99，比普通进程123的120更高，所以能抢占CPU。

（2）top命令：实时监控调度动态
`ps`是“静态快照”，`top`是“动态监控”（默认每3秒刷新一次），能实时看到进程的CPU占用、优先级变化和调度抢占情况，嵌入式场景最常用的是“按CPU排序”“显示线程”两个操作。

- 实操步骤1：执行`top`进入实时监控界面，默认显示如下（简化版）：
  ```
  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
  124 root      99  -   4096    256    208 R  99.0   0.2   5:32.11 chrt
  123 root     120   0   4096    384    328 S   1.0   0.3   0:05.23 log_process
  125 root     120   0   4096    224    176 S   0.0   0.1   0:00.01 sleep
  ```
- 核心操作与解读（按键盘按键触发）：
  1. 按`P`（大写）：按`%CPU`字段排序——实时进程124的CPU占用99%（空循环独占），日志进程123仅1%（被抢占，只能在实时进程间隙运行）；
  2. 按`T`（大写）：按“累计运行时间”排序——实时进程124的`TIME+`（累计运行时间）增长最快，因为它抢占了大部分CPU；
  3. 按`1`（数字）：展开多核CPU占用——若开发板是双核，会看到CPU0占用99%（被实时进程占用），CPU1空闲（日志进程偶尔运行）；
  4. 按`q`（小写）：退出top界面。<br>

### <strong>实战感知：用信号触发调度状态变化</strong>

调度器的“状态切换触发”是抽象概念，我们通过“发送信号控制进程状态”的实验，直观感受“调度器如何响应进程状态变化”
——实验核心是：用`kill`命令发送`SIGSTOP`（暂停信号）和`SIGCONT`（恢复信号），观察进程状态变化及调度器的反应。

实验步骤（全程在开发板操作）：
1. 准备实验进程：运行一个“LED闪烁进程”（`./led_blink &`），执行`ps -eo pid,stat,comm`记录其PID（假设为150），此时状态为`R`或`S`（运行或短暂休眠）；
2. 发送暂停信号（SIGSTOP）：执行`kill -19 150`（19对应SIGSTOP信号），立即执行`ps -eo pid,stat,comm`——会发现LED进程的`STAT`变为`T`（TASK_STOPPED，暂停状态），LED停止闪烁；
   - 调度器反应：进程进入暂停状态后，调度器会将其从“就绪队列”中移除，不再分配CPU时间（所以LED停止闪烁）；
3. 发送恢复信号（SIGCONT）：执行`kill -18 150`（18对应SIGCONT信号），再执行`ps`命令——进程状态恢复为`R`或`S`，LED重新闪烁；
   - 调度器反应：进程恢复后，调度器会将其重新加入“就绪队列”，根据优先级分配CPU时间（若有实时进程，会等待实时进程释放后再运行）；
4. 验证抢占逻辑：先运行一个高优先级实时进程（`chrt -f 99 while true; do :; done &`），再恢复LED进程——会发现LED闪烁频率变低（CPU被实时进程抢占，LED进程只能偶尔获得CPU时间）。

实验结论（对应调度器核心逻辑）：
- 进程状态决定是否参与调度：只有`R`（运行/就绪）状态的进程会被调度器分配CPU，`T`（暂停）、`S`（休眠）状态的进程会被移出就绪队列；
- 优先级决定调度顺序：高优先级实时进程会持续抢占普通进程的CPU，导致普通进程（如LED）的运行机会减少。<br>
