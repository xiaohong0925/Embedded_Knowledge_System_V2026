### <strong>进程创建：fork时调度器初始化</strong>

进程的“从无到有”不仅是资源分配的过程，更是调度器“纳入管理”的起点
——当通过`fork()`系统调用创建子进程时，内核在分配内存、IO等资源的同时，调度器会完成“子<span class="red">进程调度</span>属性初始化”“加入就绪队列”“触发抢占判断”三个核心动作，确保子进程能被正常调度执行。

（1）调度器初始化的核心流程（fork系统调用联动逻辑）
`fork()`创建子进程的核心是内核的`copy_process()`函数，其中调度器的初始化由`schedule_tail()`和`sched_fork()`完成，具体流程如下（嵌入式单核场景为例）：
1.  复制并初始化task_struct调度字段：内核复制父进程的task_struct（写时复制机制），调度器会对核心调度字段做初始化：
   - 调度策略（policy）：默认继承父进程策略，若父进程未修改则为`SCHED_OTHER`（CFS调度）；
   - 优先级（prio）：继承父进程的nice值对应的优先级，默认父进程nice=0→prio=120（普通优先级范围100-139）；
   - 虚拟运行时间（vruntime）：初始化为父进程的vruntime（确保公平性，避免子进程“插队”）；
   - 进程状态（state）：设为`TASK_RUNNING`（就绪状态，等待CPU分配）。
2.  加入调度器就绪队列：调度器将子进程的task_struct插入对应调度类的就绪队列——若为普通进程则插入CFS的红黑树，若为实时进程则插入对应优先级的实时队列。
3.  触发抢占判断：若子进程的优先级高于当前运行的父进程（如父进程是普通进程，子进程被提前修改为实时进程），调度器会设置“抢占标志”，当`fork()`系统调用返回后触发进程切换，让子进程优先运行。

（2）嵌入式场景的调度初始化特征
嵌入式系统中，`fork()`的调度初始化逻辑直接影响实时性和资源利用率，典型场景：
- 普通场景（如日志子进程）：子进程继承父进程`SCHED_OTHER`策略，vruntime与父进程一致，调度器会让父子进程轮流运行，避免某一方独占CPU；
- 实时场景（如电机控制子进程）：父进程先通过`chrt -f 90 ./parent`设为实时策略，`fork()`后子进程自动继承`SCHED_FIFO`策略和90的优先级，无需重新配置，确保子进程快速响应。<br>

### <strong>进程执行：exec替换镜像后调度属性继承规则</strong>

当进程通过`exec()`系列函数替换自身镜像
（如`execl("./new_app", "new_app", NULL)`）时，进程的代码段、数据段会被新程序替换，但调度器管理的核心属性会“选择性继承”——这是嵌入式场景中“通过父进程配置调度策略，子进程执行新程序仍保确保实时性”的关键逻辑。

（1）调度属性的“继承-重置”规则
`exec()`替换镜像后，内核会保留父进程的“调度策略、优先级、CPU亲和性”等核心调度属性，但重置与“运行状态相关”的属性，具体规则如下表（嵌入式常用场景）：

| 调度属性                | 处理规则                  | 嵌入式场景意义                                                                 |
|-------------------------|---------------------------|------------------------------------------------------------------------------|
| 调度策略（policy）      | 完全继承                  | 父进程设为SCHED_FIFO实时策略，exec后子进程运行新程序仍为实时，无需重复配置       |
| 优先级（prio/nice）     | 完全继承                  | 父进程用`renice -5`提升优先级，exec后新程序仍保持高优先级，确保核心业务响应性   |
| CPU亲和性（cpuset）     | 完全继承                  | 父进程绑定CPU0，exec后新程序仍在CPU0运行，避免跨核切换的缓存开销               |
| 虚拟运行时间（vruntime）| 重置为当前就绪队列最小值  | 新程序启动视为“新任务”，vruntime重置确保公平调度，避免继承父进程的累积运行时间 |
| 进程状态（state）       | 重置为TASK_RUNNING（就绪）| 替换镜像后进程重新进入就绪队列，等待调度器分配CPU                              |

（2）关键例外：setuid/setgid场景的优先级重置
嵌入式系统中若`exec()`的新程序有`setuid`（切换用户）权限（如系统工具`ping`），为安全起见，内核会将优先级重置为默认nice=0对应的prio=120，调度策略重置为SCHED_OTHER——这是避免“低权限进程通过exec高权限程序获取高优先级”的安全机制。<br>

### <strong>进程终止：exit触发的调度清理</strong>

当进程通过`exit()`系统调用终止（或异常终止）时，
内核不仅会释放进程的内存、IO等资源，调度器还会完成“移除就绪队列→触发重新调度→处理僵尸进程”的清理动作，确保系统资源不泄漏且CPU不空闲。

（1）调度清理的核心流程（exit系统调用联动逻辑）
进程终止的调度清理流程由`do_exit()`和`schedule()`函数协同完成，具体步骤（嵌入式单核场景）：
1.  从调度队列中移除进程：内核将进程状态设为`EXIT_ZOMBIE`（僵尸状态）或`EXIT_DEAD`（即将释放），调度器调用`dequeue_task()`将其从对应调度类的就绪队列中移除（如CFS的红黑树、实时队列）——此时进程不再参与调度；
2.  触发调度器重新选程：若当前终止的进程是“正在运行的进程”（TASK_RUNNING状态），调度器会立即调用`schedule()`，从就绪队列中选择优先级最高的进程投入运行，避免CPU空闲；
3.  僵尸进程的调度处理：进程终止后，`task_struct`会保留（存储退出状态码），成为僵尸进程，直到父进程调用`wait()`/`waitpid()`回收——此阶段调度器不会将其纳入任何队列，仅当父进程回收后，内核才会释放`task_struct`，彻底完成调度清理；
4.  父进程未回收的兜底机制：若父进程先于子进程终止且未回收子进程，内核会将僵尸进程的父进程设为根进程（PID=1，如systemd），由根进程定期调用`wait()`回收，避免僵尸进程堆积。

（2）嵌入式场景的调度清理风险与规避
嵌入式系统中，调度清理的核心风险是“僵尸进程堆积导致PID耗尽”，尤其在频繁创建/终止子进程的场景（如数据采集进程每10秒创建子进程采样），需通过“主动回收”规避：
- 风险场景：父进程循环创建子进程，子进程采样后`exit()`，父进程未调用`wait()`——执行`ps`会看到大量`Z`状态的僵尸进程，PID逐渐耗尽后无法创建新进程；
- 规避方案：父进程通过`waitpid(-1, NULL, WNOHANG)`非阻塞回收子进程，或注册`SIGCHLD`信号处理函数（子进程终止时触发父进程回收）。<br>

### <strong>Linux调度器分层架构（sched_class层级关系）</strong>

Linux内核的调度器并非单一模块，而是通过分层调度类（sched_class） 实现“多策略共存”
——将不同类型的进程（实时、普通、空闲）交给不同的调度类管理，通过固定的优先级顺序实现“抢占式调度”。
这种架构既保证了实时进程的响应性，又兼顾普通进程的公平性，是嵌入式系统“多场景适配”的核心基础。

（1）核心分层：3类调度类的优先级排序
调度类的优先级从高到低为：rt_sched_class（实时调度类）→ fair_sched_class（公平调度类）→ idle_sched_class（空闲调度类）。
核心规则是“高优先级调度类的进程可抢占低优先级进程”，具体分工与优先级逻辑如下：

| 调度类                | 管理的进程类型          | 核心职责                                  | 优先级规则                                  
|-----------------------|-------------------------|-------------------------------------------|---------------------------------------------|
| rt_sched_class        | 实时进程（SCHED_FIFO/RR）| 保障毫秒级甚至微秒级响应，优先分配CPU      | 可抢占fair和idle类进程，自身按进程优先级排序 |
| fair_sched_class      | 普通进程（SCHED_OTHER） | 保证所有进程公平分配CPU时间，避免“饿死”    | 可抢占idle类，被rt类抢占，自身按CFS规则调度 | 
| idle_sched_class      | 空闲进程（PID=0）       | CPU空闲时运行，触发低功耗模式              | 仅当无rt和fair类进程运行时才执行            |

关键补充：每个调度类都实现了统一的接口（如`enqueue_task`加入队列、`pick_next_task`选下一个进程），内核通过“遍历调度类”选择下一个运行进程
——先查rt类，有就绪进程就选最高优先级的；没有再查fair类，选最公平的；最后查idle类，运行空闲进程。

（2）分层架构的嵌入式核心价值
嵌入式系统的“混合任务场景”依赖分层架构实现：
比如一个工业网关，同时运行“电机控制实时进程”（rt类）、“数据转发普通进程”（fair类）和“待机空闲进程”（idle类）
——当电机控制进程需要运行时，rt类直接抢占fair类的CPU，确保控制延迟≤1ms；当电机进程休眠时，fair类进程正常运行；当所有进程休眠时，idle类触发CPU深度睡眠，降低功耗。

（3）实操验证：分层架构的优先级抢占
在ARM嵌入式开发板上，通过“实时进程抢占普通进程”的实验，验证调度类的优先级关系：
1.  准备普通进程：运行一个CPU密集型普通进程（空循环，默认fair类）：
    ```bash
    while true; do :; done &  # 普通进程，PID记为P1
    ```
    执行`top`查看，P1的CPU占用≈99%，`PR`（优先级）=120，`NI`（nice值）=0。
2.  运行实时进程：运行一个rt类的实时进程（SCHED_FIFO策略，优先级99）：
    ```bash
    chrt -f 99 while true; do :; done &  # 实时进程，PID记为P2
    ```
3.  观察抢占效果：
    - 再次执行`top`，会发现P2的CPU占用≈99%，P1的CPU占用≈0%  ——rt类进程抢占了fair类进程的CPU；
    - 查看调度类关联：执行`cat /proc/P2/sched | grep "sched_class"`，输出`rt_sched_class`；执行`cat /proc/P1/sched | grep "sched_class"`，输出`fair_sched_class`；
4.  终止实时进程：执行`kill -9 P2`，P1的CPU占用立即恢复到99% ——rt类进程退出后，fair类进程重新获得CPU。

（4）核心架构点总结
分层架构的核心是“优先级隔离+职责分工”
——通过sched_class的层级关系，让不同需求的进程“各归其类”，既保证了实时进程的响应性（嵌入式核心诉求），又避免了普通进程的“饥饿问题”，同时通过idle类实现低功耗，完美适配嵌入式多场景需求。<br>

### <strong>CFS完全公平调度核心（公平调度的实现逻辑）</strong>

fair_sched_class的核心实现CFS（Completely Fair Scheduler，完全公平调度器） ，
其设计思想并非“给每个进程分配相同的CPU时间”，而是“给每个进程分配与其优先级权重匹配的CPU时间比例”
——优先级越高，获得的CPU时间越多，同时避免任何进程被长期剥夺运行权。
CFS是嵌入式系统中普通进程的默认调度器，支撑日志、交互、数据处理等非实时任务。

（1）核心设计：公平性的实现载体——红黑树运行队列
CFS没有传统意义的“时间片”，而是通过红黑树（Red-Black Tree） 管理所有就绪状态的普通进程，核心逻辑如下：
1.  红黑树的key值：vruntime（虚拟运行时间） 
    vruntime是进程“理论上应运行的时间”，计算规则为：`vruntime = 实际运行时间 × 1024 / 进程权重`。  
    - 权重与优先级挂钩：进程的nice值（-20~19）对应固定权重（nice=-20权重1024，nice=0权重1024/2=512，nice=19权重1024/8=128），权重越高，vruntime增长越慢；
    - 公平性体现：红黑树的左子树节点vruntime始终小于右子树，调度器每次选择红黑树最左节点（vruntime最小的进程）运行——确保“运行时间最短、权重占比不足”的进程优先获得CPU。
2.  红黑树的优势：进程的插入（新进程就绪）、删除（进程休眠/终止）、查询（选最左节点）均为O(log n)复杂度，即使嵌入式系统运行数十个普通进程，调度器也能高效决策。

（2）核心计算：vruntime与权重的映射关系
vruntime是CFS公平性的核心，我们通过“两个进程的调度案例”理解其计算逻辑：
- 场景：嵌入式开发板运行两个普通进程P3（nice=0，权重512）和P4（nice=19，权重128），CPU为单核1GHz（每秒10^9时钟周期）；
- 权重比例：P3:P4 = 512:128 = 4:1，理论上P3应获得80% CPU时间，P4获得20%；
- vruntime计算：若P3实际运行4ms，vruntime = 4ms × 1024 / 512 = 8ms；P4实际运行1ms，vruntime = 1ms × 1024 / 128 = 8ms——两者vruntime相等，达到“公平状态”，调度器会切换进程。

下表为嵌入式常用nice值与权重的映射关系（内核默认配置）：
| nice值 | 权重  | 相对CPU时间占比（与nice=0相比） | 嵌入式场景适配                  |
|--------|-------|--------------------------------|---------------------------------|
| -20    | 1024  | 200%（最高）                   | 核心普通进程（如数据解析）      |
| -10    | 690   | 135%                           | 较高优先级普通进程（如本地计算）|
| 0      | 512   | 100%（默认）                   | 常规普通进程（如终端交互）      |
| 10     | 340   | 66%                            | 低优先级普通进程（如日志备份）  |
| 19     | 128   | 25%（最低）                    | 最不重要进程（如调试日志）      |<br>

### <strong>实时调度核心模型（SCHED_FIFO/SCHED_RR）</strong>

rt_sched_class管理的实时进程采用“优先级抢占”模型，核心是“高优先级进程始终优先运行，同优先级进程按固定规则调度”。
嵌入式系统的实时场景（如工业控制、音视频采集）均依赖此模型，确保任务响应延迟可控。
rt_sched_class支持两种调度策略：SCHED_FIFO和SCHED_RR。

（1）SCHED_FIFO：先进先出实时调度
- 核心规则：“先就绪先运行，运行到主动放弃”
——> 一旦进程获得CPU，会一直运行直到主动调用`sleep()`（休眠）、`exit()`（终止），或被更高优先级的实时进程抢占；
若有多个同优先级的SCHED_FIFO进程，按“就绪顺序”排队，只有前一个进程放弃CPU，后一个才会运行。

- 嵌入式场景适配：适用于“运行时间短、不可被打断”的任务，如电机控制进程（每次运行5ms计算控制信号，然后`sleep(10ms)`休眠，等待下一个周期）
——> 避免被同优先级进程打断导致控制延迟。

（2）SCHED_RR：时间片轮转实时调度
- 核心规则：“同优先级进程按时间片轮转，不同优先级按抢占规则”
——> 与SCHED_FIFO的区别是“同优先级进程有固定时间片”（默认100ms，可通过`/proc/sys/kernel/sched_rr_timeslice_ms`调整），当进程运行满时间片后，会被放到同优先级队列末尾，让下一个同优先级进程运行；高优先级进程仍可随时抢占。

- 嵌入式场景适配：
适用于“需要周期性运行、可短暂打断”的任务，如传感器采样进程（每个进程运行50ms采样一次数据，时间片设为60ms，确保同优先级的多个传感器进程轮流采样，互不阻塞）。<br>

### <strong>调度触发时机（调度器何时介入）</strong>

调度器并非“随时都在工作”，而是在固定的触发时机介入，决定是否切换进程。

嵌入式系统的实时性优化，本质是“优化调度触发时机的响应速度”
——比如缩短中断返回后的调度延迟，确保高优先级进程快速被唤醒。 

核心触发时机有4类：
（1）时间片耗尽（仅SCHED_RR和CFS）
- 触发场景：SCHED_RR进程运行满时间片，或CFS进程的vruntime超过队列最小vruntime；
- 嵌入式案例：SCHED_RR的传感器采样进程（时间片50ms）运行满后，调度器触发切换，让同优先级的另一个采样进程运行。

（2）高优先级进程被唤醒
- 触发场景：低优先级进程运行时，高优先级进程从休眠状态被唤醒（如IO完成、信号到达）；
- 内核动作：唤醒后立即检查高优先级进程是否就绪，若就绪则设置“抢占标志”，触发调度；
- 嵌入式案例：电机控制进程（优先级99）休眠10ms后被定时器唤醒，此时正在运行的日志进程（优先级120）会被立即抢占，电机进程投入运行。

（3）进程状态切换
- 触发场景：运行中的进程主动调用`sleep()`（转为TASK_INTERRUPTIBLE）、`wait()`（等待子进程），或调用`exit()`（终止）；
- 内核动作：进程状态从运行转为非运行，调度器立即选择新进程运行，避免CPU空闲；
- 嵌入式案例：音视频编码进程调用`sleep(1ms)`等待数据，调度器立即切换到网络接收进程，利用CPU空闲时间接收数据。

（4）<span class="red">中断处理</span>完成后
- 触发场景：硬件中断（如网卡、定时器、<span class="green">GPIO</span>）处理完成后，内核返回<span class="red">用户态</span>前；
- 内核动作：检查是否有高优先级进程被中断唤醒，若有则触发调度；
- 嵌入式关键意义：工业控制中，传感器中断（如编码器脉冲）处理完成后，唤醒控制进程，中断返回后立即调度，确保控制延迟≤1ms。

（5）实操验证：高优先级进程唤醒触发调度
1.  运行低优先级普通进程：`nice -n 19 ./cpu_int &`（P11，CPU占用99%）；
2.  运行高优先级实时进程并休眠：`chrt -f 99 bash -c "sleep 5; ./cpu_int &"`（P12，休眠5秒后唤醒）；
3.  观察`top`输出：
    - 前5秒：P11占用99% CPU；
    - 5秒后：P12唤醒，P11立即降至0%，P12升至99%——验证“高优先级唤醒触发调度”。<br>

### <strong>上下文切换的核心内容（三大切换维度）</strong>

当调度器决定让一个进程（线程）替代当前运行进程时，必须完成上下文切换（Context Switch）
——即保存当前进程的“运行状态”，恢复目标进程的“运行状态”，确保目标进程能从上次暂停的位置继续执行。

上下文切换的核心是三大维度的状态迁移：
CPU通用寄存器、进程地址空间、内核栈，这三者的切换流程直接决定了切换耗时与系统实时性。

（1）CPU通用寄存器切换（最快但核心的切换）
CPU通用寄存器（如ARM的R0-R15、PC程序计数器、CPSR状态寄存器）是进程运行的“即时状态载体”
——存储着当前执行的指令地址（PC）、运算数据（R0-R3）、CPU状态（如中断屏蔽位）。
切换时需完成“保存→恢复”两步：
1.  保存当前进程寄存器：内核将当前进程的所有通用寄存器值写入其`task_struct`的`thread_struct`结构体（内核为每个进程分配的寄存器存储区）；
2.  恢复目标进程寄存器：从目标进程的`thread_struct`中读取寄存器值，加载到CPU的物理寄存器中，此时PC指向目标进程上次暂停的指令地址，进程可直接续跑。

嵌入式关键特征：
寄存器切换是“纯硬件指令操作”，耗时最短（单核ARM Cortex-A7架构下约1-2μs），但对实时性影响最直接
——比如1ms响应的电机控制任务，若每次切换耗时2μs，100次切换就会占用0.2ms的CPU时间，挤压核心任务的运行窗口。

（2）进程地址空间切换（耗时波动最大的切换）
进程地址空间是“内核为进程分配的虚拟内存范围”（如32位嵌入式系统的0-4GB虚拟地址），切换时核心是MMU（<span class="red">内存管理</span>单元）的页表切换
——确保CPU访问的虚拟地址能映射到目标进程的物理内存（避免地址冲突）。
具体流程：
1.  刷新TLB（Translation Lookaside Buffer）：
TLB是MMU的高速缓存，存储最近使用的“虚拟地址→物理地址”映射关系；由于不同进程的虚拟地址可能映射到同一物理地址，切换时必须清空TLB（或标记为无效），避免旧映射导致地址访问错误；
2.  加载目标进程页表基址：
每个进程的页表基址存储在`task_struct`的`mm_struct`结构体中，切换时将该基址写入MMU的页表基址寄存器（如ARM的TTBR0），后续CPU访问虚拟地址时，MMU会基于新页表基址解析物理地址。

嵌入式关键影响：
页表切换耗时波动大（ARM Cortex-A7架构下，空TLB时约5-10μs，TLB命中时可省略刷新耗时仅1-2μs），且与进程内存占用正相关
——运行大内存程序（如100MB的AI推理进程）的页表条目更多，TLB命中率更低，切换耗时更长。
这也是嵌入式实时场景中“建议核心任务内存占用≤10MB”的原因之一。<br>

### <strong>调度开销的嵌入式关注点（影响实时性的核心风险）</strong>

调度开销是“上下文切换耗时+调度器决策耗时”的总和，其中上下文切换耗时占比≥80%。
嵌入式系统的调度开销关注点与服务器（追求吞吐量）截然不同：不关注单次切换的绝对耗时，而关注“切换频率×单次耗时”的累积影响
——即使单次切换仅5μs，若1ms内发生200次切换，累积耗时就达1ms，直接导致实时任务无法按时执行。

（1）调度开销的核心影响因素（嵌入式场景量化）
我们以ARM Cortex-A7（单核800MHz，嵌入式主流架构）为例，量化不同因素对切换耗时的影响：
| 影响因素                | 单次耗时范围 | 占总切换耗时比例 | 嵌入式场景敏感点                          |
|-------------------------|--------------|------------------|-----------------------------------------|
| 通用寄存器切换          | 1-2μs        | 15%-25%          | 无敏感点，耗时固定                       |
| MMU页表切换（TLB空）    | 5-10μs       | 50%-70%          | 大内存进程、频繁跨进程切换场景            |
| MMU页表切换（TLB命中）  | 1-2μs        | 15%-25%          | 进程内存占用小、TLB命中率高（≥90%）场景  |
| 内核栈切换              | 0.5-1μs      | 5%-10%           | 无敏感点，耗时固定                       |
| 调度器决策（选进程）    | 0.5-1μs      | 5%-10%           | 就绪进程数多（≥50个）时决策耗时增加       |

嵌入式核心结论：
MMU页表切换是调度开销的“主要矛盾”，优化TLB命中率（如进程绑定CPU核心）可显著降低切换耗时。

（2）调度开销的累积影响（实时性风险案例）
以工业控制中“电机转速调节任务”为例，说明调度开销的致命影响：
- 任务要求：周期10ms，每次运行5ms（计算控制信号），响应延迟≤1ms；
- 风险场景：系统中同时运行10个普通进程，导致电机任务每次执行前发生2次上下文切换（单次切换耗时8μs，累积16μs）；运行1小时后，由于时钟中断触发的频繁切换，累积切换耗时达1.2ms，导致电机控制信号输出延迟1.5ms，转速波动超过10%（超出工业允许误差5%）；
- 根源分析：嵌入式实时任务的“周期”与“切换频率”正相关——周期越短的任务，被调度器选中的频率越高，累积切换开销越大。

（3）嵌入式场景的调度开销评估指标
评估调度开销需关注两个核心指标，可通过`perf`工具量化：
1.  上下文切换次数（CS）：单位时间内的切换次数，嵌入式实时场景建议≤100次/ms（即单次切换耗时≤10μs时，累积开销≤1ms）；
2.  切换耗时占比：切换总耗时占CPU总时间的比例，核心实时场景建议≤5%（如1ms周期任务，切换耗时≤50μs）。<br>

### <strong>内核优化上下文切换的关键手段（嵌入式适配落地）</strong>

Linux内核提供了多种优化上下文切换开销的手段，嵌入式系统需结合“实时性需求”“硬件资源”“任务特性”选择适配方案
——核心思路是“减少切换频率”“缩短单次切换耗时”“避免无效切换”。

（1）减少切换频率：核心任务CPU绑定与隔离
切换频率高的核心原因是“进程在多核间迁移”或“高频时钟中断触发调度”，通过CPU绑定与隔离可从根源减少切换：
1.  进程CPU亲和性绑定：通过`taskset`命令或`sched_setaffinity`系统调用，将核心实时进程绑定到固定CPU核心（如绑定到CPU0），避免进程在多核间迁移导致的“跨核切换”（跨核切换需刷新多核TLB，耗时是同核切换的2-3倍）；
    - 嵌入式实操：`taskset -c 0 chrt -f 99 ./motor_control`（将电机控制进程绑定到CPU0，实时优先级99）；
2.  内核启动参数隔离核心：通过`isolcpus`启动参数隔离1-2个CPU核心，仅运行核心实时任务，禁止内核将普通<span class="red">进程调度</span>到隔离核，避免普通进程与实时进程的切换；
    - 嵌入式适配：在U-Boot的`bootargs`中添加`isolcpus=1`（隔离CPU1），然后通过`taskset -c 1`将实时任务绑定到隔离核；
3.  关闭高频时钟中断：通过`nohz_full`启动参数，在隔离核上关闭周期性时钟中断（tick），仅当进程主动触发调度时才执行切换，减少无效切换；
    - 嵌入式适配：`bootargs`添加`nohz_full=1`（在隔离核1上关闭时钟中断），配合`isolcpus=1`使用，可使核心任务的切换频率降低90%。

（2）缩短单次切换耗时：优化MMU与栈配置
针对切换耗时的“主要矛盾”（MMU页表切换），通过内核配置与任务优化缩短单次耗时：
1.  优化TLB命中率：
    - 任务层面：控制核心实时任务的内存占用（建议≤10MB），减少页表条目数量，提升TLB命中率（从70%提升到90%可使页表切换耗时从8μs降至2μs）；
    - 内核层面：开启`CONFIG_HIGHPTE`配置（高端页表），增加TLB条目数量（ARM架构可从32个提升到64个）；
2.  内核栈复用与精简：
    - 配置核心实时任务的内核栈为8KB（避免栈溢出），普通任务为4KB（减少内存占用）；
    - 开启`CONFIG_THREAD_INFO_IN_TASK`配置（将线程信息嵌入task_struct），减少栈切换时的地址查找耗时；
3.  启用懒切换机制：
内核3.10+支持“懒TLB切换”（Lazy TLB），当切换到的目标进程与当前进程共享地址空间（如线程）时，不刷新TLB，直接复用旧映射
——线程切换耗时可从5μs降至2μs（嵌入式多线程场景核心优化）。

（3）避免无效切换：调度策略与参数调优
通过调整调度策略与内核参数，避免“不必要的切换”：
1.  实时任务用SCHED_FIFO策略：相比SCHED_RR，SCHED_FIFO无时间片限制，核心任务运行时不会因时间片耗尽被切换，减少同优先级任务的无效切换；
2.  调大调度粒度：通过`sysctl -w kernel.sched_min_granularity_ns=20000000`（设为20ms），增加普通进程的最小运行时间，减少普通进程间的切换频率（嵌入式非实时场景适用）；
3.  关闭不必要的进程唤醒：嵌入式低功耗场景中，通过`epoll`的`EPOLLET`边缘触发模式，替代`poll`的水平触发，减少IO事件导致的频繁进程唤醒与切换。

（4）嵌入式优化效果量化（Cortex-A7架构实测）
优化前（默认配置，电机控制进程未绑定核心）与优化后（绑定CPU1+isolcpus+nohz_full）的开销对比：
| 评估指标                | 优化前       | 优化后       | 优化效果                  |
|-------------------------|--------------|--------------|-------------------------|
| 上下文切换次数（次/ms） | 15           | 1            | 减少93.3%               |
| 单次切换耗时（μs）      | 8            | 2.5          | 减少68.75%              |
| 切换耗时占比            | 12%          | 0.25%        | 减少97.9%               |
| 电机控制延迟（ms）      | 1.8          | 0.3          | 降低83.3%（满足≤1ms要求）|<br>

### <strong>实操验证：调度开销的量化与优化效果验证</strong>

在ARM嵌入式开发板（Cortex-A7，800MHz）上，通过`vmstat`和`perf`工具量化调度开销，并验证CPU绑定的优化效果：

（1）基础工具：vmstat查看切换次数
`vmstat`是嵌入式系统查看上下文切换的基础工具，核心关注`cs`（上下文切换次数）列：
1.  查看全局切换次数：执行`vmstat 1`（每秒刷新一次），输出类似：
    ```
    procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
     r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
     2  0      0  51200  10240  81920   0    0     0     0  150 1500  5  3 92  0  0
    ```
    - 解读：`cs=1500`表示每秒发生1500次上下文切换，`sy=3`表示<span class="red">内核态</span>CPU占用3%（含切换开销）；
2.  运行核心任务后观察：执行`chrt -f 99 ./motor_control &`，再执行`vmstat 1`，会发现`cs`升至2500次/秒，`sy`升至8%——实时任务与普通任务的切换导致开销增加。

（2）精准工具：perf统计切换耗时与频率
`perf`支持追踪`context-switches`事件，精准量化切换开销：
1.  统计指定进程的切换情况：执行`perf stat -e context-switches -p [电机进程PID] 10`（统计10秒内的切换次数），输出类似：
    ```
     Performance counter stats for process id '[PID]' (10 seconds):

         2000      context-switches:u    #    0.200 K/sec

      10.001234567 seconds time elapsed
    ```
    - 解读：10秒内发生2000次切换，平均200次/秒，结合进程周期10ms，可算出每次周期发生2次切换；
2.  绑定CPU后对比：执行`taskset -c 1 chrt -f 99 ./motor_control &`，再用`perf`统计，输出：
    ```
     Performance counter stats for process id '[PID]' (10 seconds):

         200       context-switches:u    #    0.020 K/sec

      10.000987654 seconds time elapsed
    ```
    - 优化效果：切换次数减少90%，验证CPU绑定的有效性。

（3）实时延迟验证：cyclictest测试切换对延迟的影响
`cyclictest`是嵌入式实时性测试的核心工具，可量化调度开销导致的任务延迟：
1.  优化前测试：执行`cyclictest -t 1 -p 99 -n`（1个实时线程，优先级99，无负载），运行1分钟后输出：
    ```
    T: 0 ( 2801) P:99 I:1000 C:  6000 Min:      1 Max:    180 Avg:      5
    ```
    - 解读：`Max=180`表示最大延迟180μs，`Avg=5`表示平均延迟5μs；
2.  添加负载后测试：运行10个普通进程（`for i in {1..10}; do while true; do :; done &`），再执行`cyclictest`，输出：
    ```
    T: 0 ( 2801) P:99 I:1000 C:  6000 Min:      1 Max:   1200 Avg:     45
    ```
    - 解读：最大延迟升至1200μs（1.2ms），平均延迟45μs——普通进程与实时进程的切换导致延迟大幅增加；
3.  优化后测试：将实时线程绑定到隔离核（`taskset -c 1 cyclictest -t 1 -p 99 -n`），再添加负载，输出：
    ```
    T: 0 ( 2801) P:99 I:1000 C:  6000 Min:      1 Max:    300 Avg:      8
    ```
    - 优化效果：最大延迟降至300μs，恢复到接近无负载水平，验证隔离核+绑定的优化价值。<br>
