### <strong>基础优先级调整：nice/renice命令</strong>

`nice`和`renice`是调整普通进程优先级的核心工具（仅作用于SCHED_OTHER/CFS调度策略的进程），
通过修改`nice值`（范围-20~19，值越小优先级越高）间接调整进程的`prio`（优先级，范围100-139）。

两者的核心区别是：`nice`在进程启动时设置优先级，`renice`对已运行的进程动态调整。

（1）核心语法与参数解析
| 工具   | 核心功能                | 基础语法                                  | 关键参数说明                          |
|--------|-------------------------|-------------------------------------------|---------------------------------------|
| nice   | 启动进程时设置nice值    | `nice -n [nice值] 进程命令`               | `-n`：指定nice值（必填，可简写为`-[nice值]`） |
| renice | 调整运行中进程的nice值  | `renice [nice值] -p [进程PID]`            | `-p`：指定目标进程PID（必填，可批量指定多个PID） |

原理回顾（前序章节结论，不扩展）：
nice值与prio的映射关系为`prio = 120 + nice值`，如nice=-10→prio=110，nice=5→prio=125。

（2）实操步骤：从启动到动态调整
以ARM嵌入式开发板（如IMX6ULL）上的“数据解析进程”和“日志打印进程”为例，演示完整操作：
1.  用nice启动低优先级进程：  
    日志打印进程（非核心）启动时降低优先级（nice=10），避免占用核心资源：
    ```bash
    # 启动日志进程，设置nice=10（优先级130）
    nice -n 10 ./log_print &  # 进程PID记为1001
    ```
    验证结果：执行`ps -eo pid,comm,nice,prio | grep log_print`，输出如下（符合映射关系）：
    ```
    1001 log_print    10  130
    ```

2.  用renice动态提升核心进程优先级：  
    数据解析进程（核心）运行中发现响应较慢，提升其优先级（nice=-5）：
    ```bash
    # 1. 先启动数据解析进程（默认nice=0，prio=120）
    ./data_parse &  # PID记为1002
    # 2. 动态调整nice值为-5
    renice -5 -p 1002
    ```
    验证结果：执行`ps -eo pid,comm,nice,prio | grep data_parse`，输出如下（优先级提升）：
    ```
    1002 data_parse   -5  115
    ```<br>

### <strong>实时策略配置：chrt命令实战</strong>

`chrt`是配置实时<span class="red">进程调度</span>策略与优先级的核心工具（作用于SCHED_FIFO/SCHED_RR策略），
嵌入式实时场景（工业控制、音视频采集、传感器采样）必备。
其核心能力是“启动时配置”或“运行中修改”实时进程的策略与优先级（1-99，值越大优先级越高）。

（1）核心语法与策略说明
`chrt`的基础语法为：`chrt [选项] [优先级] [进程命令/PID]`，关键选项与策略对应关系如下：
| 选项 | 对应调度策略 | 核心特点                                  | 嵌入式场景适配                          |
|------|--------------|-------------------------------------------|---------------------------------------|
| -f   | SCHED_FIFO   | 先进先出，无时间片，高优先级独占CPU       | 电机控制、脉冲生成等不可打断的实时任务 |
| -r   | SCHED_RR     | 时间片轮转，同优先级进程轮流运行          | 多传感器采样、周期性数据处理等任务     |
| -p   | 查看/修改运行中进程 | 后跟PID，用于调整已启动进程的策略         | 实时任务运行中动态调优优先级           |

关键规则：
实时进程优先级（1-99）高于所有普通进程（100-139），高优先级实时进程可抢占低优先级实时进程。

（3）嵌入式实时配置的关键注意事项
- 优先级分配原则：核心控制任务（如电机、阀门）设90~99，辅助实时任务（如采样）设50~80，避免同优先级阻塞；
- SCHED_FIFO风险规避：若高优先级FIFO进程无主动休眠（如`sleep()`），会独占CPU导致其他进程“饿死”
——必须在代码中加入周期性休眠（如电机控制进程运行5ms后`sleep(10ms)`）；
- 时间片调整：SCHED_RR的默认时间片为100ms，嵌入式场景可通过`echo 50 > /proc/sys/kernel/sched_rr_timeslice_ms`调整为50ms（缩短同优先级任务等待时间）。<br>

### <strong>调度信息查询：schedtool与/proc文件</strong>

配置完成后，需通过查询工具验证效果并分析调度状态
——`schedtool`是专用调度信息查询工具，`/proc/[pid]/schedstat`是内核暴露的调度统计接口，
两者结合可实现“配置验证+性能分析”。

（1）schedtool命令：专用调度信息查询
`schedtool`可快速查看进程的调度策略、优先级、CPU亲和性等信息，部分嵌入式系统需手动安装（如Debian系`apt install schedtool`，CentOS系`yum install schedtool`）。

核心实操示例：
1.  查看<span class="red">进程调度</span>策略与优先级：
    ```bash
    # 查看电机控制进程（1003）的详细信息
    schedtool -p 1003
    ```
    预期输出（字段解读已标注）：
    ```
    PID 1003: ./motor_control
    Policy: SCHED_FIFO   # 调度策略
    Priority: 99         # 实时优先级
    Nice: 0              # nice值（实时进程无效）
    CPU affinity: 0-3    # CPU亲和性（绑定的核心范围）
    ```

2.  批量查询多个进程：
    ```bash
    # 同时查看数据解析（1002）、电机控制（1003）、日志（1001）进程
    schedtool -p 1001 1002 1003
    ```<br>

### <strong>核心系统调用解析（调度编程基础）</strong>

嵌入式调度编程的核心是通过Linux内核提供的调度相关系统调用，
在代码中动态配置进程的调度策略、优先级等属性，替代命令行工具（chrt/nice）的手动操作。
核心系统调用共4个，覆盖“设置-获取-校验”全场景，需重点掌握参数格式与返回值处理。

（1）核心系统调用清单（含语法与场景）
| 系统调用                 | 核心功能                                  | 函数原型                                                                 
| sched_setscheduler       | 设置进程调度策略与优先级                  | `int sched_setscheduler(pid_t pid, int policy, const struct sched_param *param);` |
| sched_getparam           | 获取进程当前优先级                        | `int sched_getparam(pid_t pid, struct sched_param *param);`              
| sched_get_priority_max   | 获取指定策略的最高优先级                  | `int sched_get_priority_max(int policy);`                                
| sched_get_priority_min   | 获取指定策略的最低优先级                  | `int sched_get_priority_min(int policy);`                                

（2）关键数据结构：struct sched_param
该结构体用于传递实时进程的优先级，仅`__sched_priority`字段有效（普通进程忽略），定义如下（内核头文件`<sched.h>`中）：
```c
struct sched_param {
    int __sched_priority;  // 实时进程优先级（1-99，仅SCHED_FIFO/SCHED_RR有效）
};
```
注意：
普通进程（SCHED_OTHER）的优先级通过`nice`值调整，需用`setpriority()`系统调用，本节重点覆盖实时进程编程（嵌入式核心场景）。

（3）返回值与错误处理（嵌入式编程关键）
所有调度系统调用的返回值规则一致：成功返回0，失败返回-1并设置`errno`。
嵌入式场景必须处理的常见错误如下：
| 错误码（errno） | 含义                                  | 嵌入式场景诱因                          | 解决方法                                  |
|-----------------|---------------------------------------|---------------------------------------|-----------------------------------------|
| EPERM           | 权限不足                              | 普通用户配置实时优先级，或优先级超过权限上限 | 用root权限运行，或配置limits.conf授权    |
| EINVAL          | 参数无效                              | 策略与优先级不匹配（如给SCHED_OTHER设1-99优先级） | 确保策略与优先级对应，实时策略用1-99      |
| ESRCH           | 进程不存在                            | PID错误或进程已终止                    | 校验PID有效性，或用0表示当前进程          |<br>

### <strong>实战代码：实时进程调度配置完整示例
以“嵌入式工业控制场景”的电机控制进程为例，编写完整代码：
实现“校验策略范围→设置SCHED_FIFO策略+最高优先级→运行中校验配置→周期性执行控制逻辑”，
含详细注释、错误处理及休眠防独占逻辑。</strong>

（1）完整代码（motor_control.c）
```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sched.h>
#include <errno.h>
#include <string.h>

// 错误处理宏定义（简化重复代码）
#define CHECK_RET(ret, msg) do { \
    if (ret == -1) { \
        fprintf(stderr, "[ERROR] %s: %s (errno=%d)\n", msg, strerror(errno), errno); \
        exit(EXIT_FAILURE); \
    } \
} while(0)

int main(void) {
    int ret;
    struct sched_param param;
    int policy = SCHED_FIFO;  // 目标调度策略：先进先出实时调度
    int max_prio, min_prio;

    /************************** 步骤1：校验调度策略的优先级范围 **************************/
    // 获取SCHED_FIFO策略的最高/最低优先级
    max_prio = sched_get_priority_max(policy);
    CHECK_RET(max_prio, "sched_get_priority_max");
    min_prio = sched_get_priority_min(policy);
    CHECK_RET(min_prio, "sched_get_priority_min");
    printf("SCHED_FIFO策略优先级范围：%d ~ %d\n", min_prio, max_prio);

    /************************** 步骤2：配置当前进程为实时进程 **************************/
    param.__sched_priority = max_prio;  // 配置为最高优先级（确保响应最快）
    // 调用sched_setscheduler：pid=0表示当前进程，策略=SCHED_FIFO，优先级=max_prio
    ret = sched_setscheduler(0, policy, &param);
    CHECK_RET(ret, "sched_setscheduler");
    printf("调度策略配置成功：SCHED_FIFO，优先级=%d\n", max_prio);

    /************************** 步骤3：运行中校验配置是否生效 **************************/
    struct sched_param get_param;
    // 获取当前进程的优先级
    ret = sched_getparam(0, &get_param);
    CHECK_RET(ret, "sched_getparam");
    // 获取当前进程的实际策略（防止配置被意外修改）
    int curr_policy = sched_getscheduler(0);
    CHECK_RET(curr_policy, "sched_getscheduler");
    printf("运行中校验：策略=%s，优先级=%d\n", 
           curr_policy == SCHED_FIFO ? "SCHED_FIFO" : 
           (curr_policy == SCHED_RR ? "SCHED_RR" : "SCHED_OTHER"), 
           get_param.__sched_priority);

    /************************** 步骤4：核心业务逻辑（电机控制） **************************/
    printf("电机控制进程启动，开始周期性执行控制逻辑...\n");
    while (1) {
        // 1. 核心控制逻辑（示例：读取编码器数据→计算PWM值→输出控制信号）
        printf("执行电机控制：读取数据→计算PWM→输出信号\n");
        
        // 2. 关键：主动休眠，释放CPU（避免SCHED_FIFO进程独占CPU导致系统卡死）
        usleep(1000);  // 休眠1ms，匹配1ms控制周期（嵌入式实时场景核心优化）
    }

    return EXIT_SUCCESS;
}
```<br>

### <strong>权限管控：嵌入式系统实时优先级授权</strong>

嵌入式系统中，普通用户默认无权限配置实时优先级（仅root可配置），
但工业场景中“业务进程需以普通用户运行”（安全要求，避免root权限滥用），需通过`/etc/security/limits.conf`配置权限，实现“普通用户授权实时优先级”。

（1）权限管控核心原理
Linux内核通过“资源限制（RLIMIT）”机制管控进程权限，实时优先级的授权本质是配置`RLIMIT_RTPRIO`（实时优先级上限）
——普通用户的该限制默认为0（无实时权限），root可通过配置文件修改上限。

（2）实操配置：/etc/security/limits.conf编辑
1.  编辑配置文件（需root权限）：
    ```bash
    vi /etc/security/limits.conf
    ```

2.  添加授权规则（末尾添加，格式：`用户 类型 资源 值`）：
    ```ini
    # 给普通用户"embed"授权：实时优先级上限99（可配置1-99的实时优先级）
    embed    hard    rtprio    99
    embed    soft    rtprio    99
    ```
    规则解读：
    - `embed`：目标普通用户名（根据实际场景修改，如“iot”“control”）；
    - `hard`：硬限制（用户无法突破的上限）；`soft`：软限制（用户可临时调整至硬限制）；
    - `rtprio`：资源类型（实时优先级限制）；
    - `99`：授权的最高实时优先级（嵌入式实时场景建议设99）。

3.  配置生效方法：
    - 方法1：普通用户重新登录（SSH断开重连，或执行`su - embed`切换用户）；
    - 方法2：重启开发板（嵌入式系统无复杂服务，重启最稳妥）。<br>

### <strong>扩展场景：动态修改调度策略（SCHED_FIFO→SCHED_RR）</strong>

实际场景中可能需要“运行中动态切换调度策略”
（如电机启动阶段用SCHED_FIFO保证快速响应，稳定运行后切换为SCHED_RR兼容多任务），基于核心系统调用实现动态修改，代码如下：

（1）动态切换核心代码（扩展自motor_control.c）
```c
// 在while循环前添加动态切换逻辑
printf("3秒后将策略从SCHED_FIFO切换为SCHED_RR...\n");
sleep(3);

// 切换为SCHED_RR策略，优先级保持99
int new_policy = SCHED_RR;
param.__sched_priority = 99;  // SCHED_RR与SCHED_FIFO优先级范围一致
ret = sched_setscheduler(0, new_policy, &param);
CHECK_RET(ret, "sched_setscheduler (switch to SCHED_RR)");

// 校验切换结果
curr_policy = sched_getscheduler(0);
CHECK_RET(curr_policy, "sched_getscheduler (after switch)");
printf("策略切换成功：%s，优先级=%d\n", 
       curr_policy == SCHED_RR ? "SCHED_RR" : "UNKNOWN", 
       param.__sched_priority);
```

（2）运行效果
程序启动后先以SCHED_FIFO运行3秒，然后切换为SCHED_RR，终端输出：
```
...
执行电机控制：读取数据→计算PWM→输出信号
3秒后将策略从SCHED_FIFO切换为SCHED_RR...
策略切换成功：SCHED_RR，优先级=99
执行电机控制：读取数据→计算PWM→输出信号
...
```
第三方验证：`cat /proc/[PID]/sched`显示`Policy: SCHED_RR`，确认切换生效。<br>

### <strong>常见问题排查（嵌入式编程踩坑指南）</strong>

（1）问题1：编译报错“undefined reference to `sched_getscheduler'”
- 原因：交叉编译器版本不匹配，或编译时未链接标准库；
- 解决：编译命令添加`-lc`（链接C标准库），确保交叉编译器架构与开发板一致（如ARM Cortex-A7对应`arm-linux-gnueabihf-gcc`）。

（2）问题2：运行时提示“Permission denied (errno=1)”
- 原因：普通用户未授权实时优先级，或权限配置未生效；
- 解决：
  1.  确认`limits.conf`配置正确；
  2.  普通用户重新登录或重启开发板；
  3.  临时测试：用`sudo ./motor_control`以root运行，排除权限问题。

（3）问题3：程序运行后开发板卡死
- 原因：SCHED_FIFO进程未添加休眠逻辑，独占CPU；
- 解决：
  1.  强制重启开发板（嵌入式开发板长按复位键）；
  2.  在核心逻辑后添加`usleep()`或`sleep()`，确保进程主动释放CPU。

（4）问题4：策略切换后优先级无效
- 原因：给SCHED_OTHER策略设置1-99优先级（普通进程不支持该范围）；
- 解决：普通进程用`setpriority(PRIO_PROCESS, 0, -10)`调整nice值，实时进程才用`sched_setscheduler()`设置1-99优先级。<br>
