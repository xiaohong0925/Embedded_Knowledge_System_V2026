### <strong>进程亲和性作用（嵌入式多核场景的核心价值）</strong>

在嵌入式多核系统（如双核Cortex-A7、四核Cortex-A53）中，
CPU亲和性（CPU Affinity） 是将进程“绑定到指定CPU核心”的机制，通过限制进程在多核间的迁移，解决“跨核上下文切换开销大”“缓存失效频繁”两大核心痛点
——这是嵌入式实时场景（如工业控制、自动驾驶）提升稳定性的关键优化手段。

（1）核心价值：为什么嵌入式多核必须做亲和性配置？
1.  减少跨核上下文切换开销：
跨核切换时，不仅要切换进程上下文，还要刷新两个核心的TLB（<span class="red">内存管理</span>单元缓存）和L2缓存，耗时是同核切换的2-3倍（ARM Cortex-A7中，同核切换约5μs，跨核切换约15μs）；绑定后进程仅在指定核运行，彻底避免跨核切换。
2.  提升缓存命中率：
进程运行时会将常用数据载入当前核的L1/L2缓存（命中率≥90%时数据访问延迟≤10ns），若进程迁移到其他核，原缓存失效需重新载入（命中率骤降为30%，延迟≥100ns）；绑定后缓存持续有效，尤其对数据密集型任务（如传感器数据解析）性能提升显著。
3.  实现核心负载隔离：
将实时进程绑定到专属核心，普通进程绑定到其他核心，避免普通进程抢占实时进程的CPU资源——嵌入式多核实时场景的“黄金法则”。

（2）嵌入式多核调度的默认问题（未配置亲和性时）
以“双核IMX6ULL开发板”运行“电机控制（实时）+ 日志打印（普通）”为例，未配置亲和性时的调度问题：
- 问题1：电机控制进程在CPU0和CPU1间频繁迁移，每次迁移导致控制延迟从1ms升至3ms（跨核切换+缓存失效）；
- 问题2：日志打印进程在CPU0运行时，抢占电机控制进程的CPU，导致控制信号输出延迟；
- 解决：绑定电机控制到CPU0，日志打印到CPU1，控制延迟稳定在1ms内，缓存命中率从60%升至95%。<br>

### <strong>工具实现：taskset与chcpu实操（快速配置亲和性）</strong>

嵌入式场景中，若无需代码级灵活配置，
可通过`taskset`（亲和性配置工具）和`chcpu`（CPU拓扑查看工具）快速实现亲和性绑定，适用于“调试阶段”或“简单场景部署”。

（1）核心工具：taskset语法与实操
`taskset`通过“CPU核心掩码”或“核心列表”指定进程绑定的核心，核心语法：
| 操作类型                | 命令格式                                  | 说明                                      |
|-------------------------|-------------------------------------------|-------------------------------------------|
| 启动时绑定              | `taskset -c [核心列表] 进程命令`          | `-c`：指定核心列表（如0、0-1，直观易懂）  |
| 运行中修改绑定          | `taskset -cp [核心列表] [进程PID]`        | `-p`：修改已运行进程的亲和性              |
| 查看进程当前亲和性      | `taskset -cp [进程PID]`                   | 输出进程绑定的核心列表                    |

关键概念：
CPU核心掩码是十六进制值，对应核心是否启用（如双核中，0x01=CPU0，0x02=CPU1，0x03=CPU0+CPU1），`-c`参数更直观，嵌入式场景优先使用。

（3）工具使用的关键注意事项
- 核心编号规则：嵌入式CPU核心从0开始编号（如四核为0-3），绑定前需通过`chcpu`确认核心状态（“online”表示可用）；
- 实时进程绑定原则：绑定到“物理核心”而非“逻辑核心”（嵌入式多核多为物理核心，无超线程），避免逻辑核心切换导致的缓存失效；
- 权限要求：修改运行中进程的亲和性需root权限，普通用户仅能绑定自身启动的进程。<br>

### <strong>编程实现：sched_setaffinity与cpu_set_t</strong>

工具配置适用于静态场景，
若需“根据运行状态动态调整亲和性”（如系统负载高时将非核心进程迁移到备用核），
需通过`sched_setaffinity`系统调用编程实现，核心是`cpu_set_t`结构体的操作。

（1）核心API与数据结构
1.  关键系统调用：
    | 系统调用                 | 功能                                  | 函数原型                                                                 |
    |--------------------------|---------------------------------------|--------------------------------------------------------------------------|
    | sched_setaffinity        | 设置进程CPU亲和性                    | `int sched_setaffinity(pid_t pid, size_t cpusetsize, const cpu_set_t *mask);` |
    | sched_getaffinity        | 获取进程当前亲和性                    | `int sched_getaffinity(pid_t pid, size_t cpusetsize, cpu_set_t *mask);` | 

2.  核心数据结构：cpu_set_t 
    用于表示“CPU核心集合”，通过专用宏操作（避免直接操作内存）：
    - `CPU_ZERO(&mask)`：初始化核心集合（清空所有核心）；
    - `CPU_SET(core, &mask)`：将指定核心加入集合；
    - `CPU_CLR(core, &mask)`：从集合中移除指定核心；
    - `CPU_ISSET(core, &mask)`：判断核心是否在集合中（返回非0表示是）。<br>

### <strong>实战技巧：嵌入式多核调度的深度优化
E级难度的核心是“突破基础配置，实现极致实时性”，结合嵌入式多核场景的痛点，以下两大技巧是工业级项目的必备优化手段。</strong>

（1）技巧1：绑定中断与对应处理进程到同一核（降低中断延迟）
嵌入式实时场景中，“<span class="red">中断处理</span>延迟”是关键瓶颈
——若中断在CPU1触发，而对应的<span class="red">中断处理</span>进程在CPU0运行，中断响应后需跨核调度进程，延迟增加50-100μs。
将“中断”与“处理进程”绑定到同一核，可将延迟降低至10μs以内。

实操步骤（以“串口中断+数据处理进程”为例）：
1.  步骤1：查看中断号与当前亲和性  
    串口1（ttymxc0）的中断号可通过`cat /proc/interrupts`查看：
    ```bash
    cat /proc/interrupts | grep ttymxc0
    ```
    预期输出（中断号为31）：
    ```
    31:        123   GIC-0  29 Level     ttymxc0
    ```

2.  步骤2：绑定中断到CPU0  
    通过`echo`命令修改`/proc/irq/[中断号]/smp_affinity`（十六进制掩码，0x01=CPU0）：
    ```bash
    echo 0x01 > /proc/irq/31/smp_affinity
    # 验证结果
    cat /proc/irq/31/smp_affinity  # 输出00000001
    ```

3.  步骤3：绑定<span class="red">中断处理</span>进程到CPU0 
    数据处理进程（PID1015）绑定到CPU0，与中断亲和性一致：
    ```bash
    taskset -cp 0 1015
    ```

4.  效果验证：
    用`cyclictest`测试中断延迟，未绑定前平均延迟50μs，绑定后降至8μs，提升显著。<br>

### <strong>常见问题排查（嵌入式多核实战踩坑指南）</strong>

（1）问题1：绑定核心后进程仍跨核运行？
- 原因：1. 进程创建了子进程，子进程未继承亲和性；2. 内核线程（如kworker）抢占核心；
- 解决：1. 子进程启动后重新绑定亲和性；2. 用`isolcpus`隔离核心，禁止内核线程运行。

（2）问题2：修改中断亲和性提示“Permission denied”？
- 原因：中断亲和性文件`/proc/irq/[中断号]/smp_affinity`仅root可修改，且部分中断（如系统时钟中断）不支持修改；
- 解决：用`sudo`执行修改命令，确认中断类型为“可配置亲和性”（非系统核心中断）。

（3）问题3：核心隔离后无法绑定进程到隔离核？
- 原因：`isolcpus`隔离的核心默认不对外暴露，需通过`taskset`强制绑定；
- 解决：确保绑定命令正确（`taskset -c 1 ./app`），若仍失败，检查`bootargs`中`isolcpus`参数是否正确。

（4）问题4：nohz_full配置后<span class="red">进程调度</span>异常？
- 原因：关闭时钟中断后，SCHED_RR的时间片机制失效（依赖时钟中断触发）；
- 解决：隔离核上的进程改用SCHED_FIFO策略，主动通过`sleep()`释放CPU。
