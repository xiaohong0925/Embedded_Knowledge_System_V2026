# 嵌入式专属实战场景

> 📊 **本节难度等级：** <span class="badge-ie">**IE级**</span>

---

### <strong>核心诉求与业务链路</strong>

- 业务背景：嵌入式音视频终端（如4K摄像头、实时对讲机）
需完成“音频/视频采集→编码压缩→网络传输”全链路处理，要求端到端延迟≤5ms（否则出现音画不同步、卡顿）；

- 核心痛点：
① 采集线程被普通进程抢占导致数据丢帧；
② 编码线程调度延迟超标；
③ 跨核切换导致缓存失效，编码效率下降；
④ 内核抢占机制不足，中断打断业务线程。

### <strong>分层优化方案（按“易落地→深优化”递进）</strong>

1. 基础优化：快速锁定核心线程优先级（I级，1天落地）
优化目标：避免采集/编码线程被低优先级进程抢占，快速将延迟控制在10ms内。
- 关键操作：
  1.  配置实时优先级：采集、编码、传输线程均设为SCHED_FIFO策略（无时间片，避免同优先级抢占），
优先级按“采集＞编码＞传输”排序（采集最高99，编码95，传输90）：
      ```bash
      # 启动采集线程（基础操作：命令行快速验证）
      chrt -f 99 ./audio_capture &  # 音频采集线程，PID=2001
      chrt -f 95 ./video_encode &   # 视频编码线程，PID=2002
      chrt -f 90 ./net_transfer &   # 网络传输线程，PID=2003
      ```
      （编程落地：用`sched_setscheduler`在线程初始化时配置，避免命令行手动操作，
代码参考3.1.3.2实时进程编程示例，核心是线程创建后立即绑定策略与优先级）

  2.  绑定CPU核心：将3个核心线程绑定到同一CPU核（如CPU0），避免跨核切换与缓存失效：
      ```bash
      # 命令行绑定（基础验证）
      taskset -c 0 -p 2001 2002 2003
      # 编程绑定（生产环境）：线程创建时调用`sched_setaffinity`绑定CPU0
      ```

- 效果验证：
  - 用`top -H -p 2001`查看线程优先级，`PR`列显示99（实时优先级）；
  - 用`cyclictest -t 3 -p 99 -a 0`测试CPU0上的线程延迟，基础优化后平均延迟≤8ms，最大延迟12ms（较默认配置降低50%）。<br>

### <strong>场景核心总结</strong>

音视频实时性优化的关键是“优先级隔离+核心绑定+中断控制+内核增强”，
按“基础优化快速落地→进阶追踪瓶颈→极致内核调优”的顺序，可避免盲目投入，根据项目需求灵活选择优化深度。<br>

### <strong>核心诉求与业务链路</strong>

- 业务背景：128M内存的ARM双核IoT网关（如工业Modbus网关），
需同时运行“网络数据转发（TCP/UDP）”“本地数据计算（协议解析、数据过滤）”“日志备份”三大类任务，
核心痛点是“网络转发吞吐量低（≤10Mbps）”“本地计算抢占网络资源”“内存溢出导致网关重启”。
- 核心目标：网络转发吞吐量≥30Mbps，本地计算响应延迟≤10ms，日志进程内存占用≤20M。

### <strong>分层优化方案（按“基础均衡→瓶颈突破→能效优化”递进）</strong>

1. 基础优化：核心任务隔离与优先级调整（I级，1天落地）
优化目标：避免任务间资源抢占，快速提升吞吐量至20Mbps。
- 关键操作：
  1.  CPU核心绑定：按“任务类型”分配核心，实现负载隔离：
      - 网络转发进程→CPU0（专注IO密集型任务）；
      - 本地计算进程→CPU1（专注计算密集型任务）；
      - 日志备份进程→CPU1（低优先级，不抢占计算进程）。
      ```bash
      # 绑定网络转发进程到CPU0
      taskset -c 0 ./net_forward &  # PID=2004
      # 绑定本地计算进程到CPU1
      taskset -c 1 ./local_calc &   # PID=2005
      # 降低日志进程优先级（nice=15），避免抢占核心任务
      nice -n 15 ./log_backup &     # PID=2006
      ```

  2.  内存限制：通过`ulimit`限制日志进程的最大内存，避免内存溢出：
      ```bash
      ulimit -m 20480  # 限制日志进程最大内存20M（20480KB）
      ./log_backup &
      ```

- 效果验证：
  - 用`sar -n DEV 1`监控网络吞吐量，从10Mbps提升至22Mbps；
  - 用`top`查看CPU负载，CPU0≈70%（网络），CPU1≈60%（计算+日志），无单核心过载。<br>

### <strong>场景核心总结</strong>

IoT网关负载均衡的关键是“资源隔离+精准限制+调度智能化”，
基础优化用绑定和优先级快速落地，进阶用cgroups和RPS突破瓶颈，极致优化用EAS实现“性能-功耗”平衡，避免过度依赖手动配置。<br>

### <strong>核心诉求与业务链路</strong>

- 业务背景：ARM big.LITTLE架构（如4核：2个Cortex-A73大核+2个Cortex-A53小核）的嵌入式设备（如AI边缘盒子），
需运行“高算力任务（AI推理、视频解码）”和“低功耗任务（传感器采集、蓝牙通信）”，核心目标是“高算力任务性能达标（AI推理延迟≤20ms）+ 低功耗任务功耗可控（待机功耗≤0.5W）”。
- 核心痛点：
① 高算力任务被调度到小核，性能不足；
② 低功耗任务占用大核，功耗飙升；
③ 核间迁移导致缓存失效，推理效率下降。

### <strong>分层优化方案（按“手动分配→智能调度→内核定制”递进）</strong>

1. 基础优化：核组绑定与任务分类（I级，1天落地）
优化目标：快速实现“大核跑算力任务，小核跑低功耗任务”，避免任务错配。
- 关键操作：
  1.  明确核组划分：big核（高性能）→CPU0-1，LITTLE核（低功耗）→CPU2-3；
  2.  任务绑定：
      ```bash
      # AI推理任务（高算力）绑定big核（CPU0-1）
      taskset -c 0-1 ./ai_infer &  # PID=2007
      # 视频解码任务（高算力）绑定big核
      taskset -c 0-1 ./video_decode &  # PID=2008
      # 传感器采集（低功耗）绑定LITTLE核（CPU2-3）
      taskset -c 2-3 ./sensor_collect &  # PID=2009
      # 蓝牙通信（低功耗）绑定LITTLE核
      taskset -c 2-3 ./bluetooth_comm &  # PID=2010
      ```

  3.  功耗监控与验证：用`powertop`查看功耗变化：
      - 未绑定前：大核运行传感器任务，功耗1.8W；
      - 绑定后：大核仅运行AI/解码，小核运行低功耗任务，功耗1.1W（降低39%）；
      - AI推理延迟从35ms降至22ms（接近目标）。<br>

### <strong>场景核心总结</strong>

异构多核优化的核心是“核组划分+任务-核心匹配+能效调度”，
基础优化用手动绑定快速落地，进阶用调度域和频率调整优化稳定性，极致优化用Energy Model和EAS实现“性能-功耗”的动态平衡，适合工业级边缘计算设备。<br>
