# 2 竞态产生的根源解析

> 📊 **本节难度等级：** <span class="badge-i">**I级**</span>

---

### <strong>竞态的本质是“执行流对共享资源的无序访问”，而嵌入式Linux中进程/线程、中断、多核这三类执行流的固有特性，是“无序访问”的核心诱因。
不同于[B]级内容对“执行流是什么”的场景化描述，[I]级内容需深入内核调度机制、硬件中断逻辑和多核缓存架构，解析这些特性“如何从根源上触发竞态”——只有理解根源，才能在后续同步控制中“精准施策”。</strong>


### <strong>进程/线程的调度机制：抢占式调度对共享资源的影响
进程/线程是<span class="red">用户态</span>与驱动交互的核心载体，而Linux的抢占式调度机制（Preemptive Scheduling）是进程级并发的“直接推手”。</strong>

1.1.1 抢占式调度的核心逻辑：“谁优先，谁执行”
Linux调度器（Scheduler）的核心目标是“最大化CPU利用率”和“保证响应性”，
其抢占式调度的本质是：高优先级执行流可随时打断低优先级执行流的执行，无需低优先级执行流主动释放CPU。
这种机制在提升系统响应性的同时，也为驱动共享资源埋下了“被无序访问”的隐患。

我们先拆解调度器的核心组成（以主流的CFS调度器为例，适用于普通进程）：
1.  调度实体与优先级：
每个进程/线程都是一个调度实体，Linux用“静态优先级”（nice值，范围-20~19，值越小优先级越高）和“动态优先级”（根据进程运行状态调整）决定调度权重——高权重的执行流获得更多CPU时间片。
2.  时间片分配与抢占触发：
CFS调度器为每个实体分配“虚拟运行时间”（vruntime），vruntime增长最慢的实体（即权重最高）被选为“下一个执行实体”。当高优先级实体的vruntime远小于当前运行实体时，调度器会触发“抢占”：
    - 保存当前执行实体的上下文（寄存器、程序计数器等）；
    - 切换到高优先级实体的上下文，执行其代码；
    - 当高优先级实体时间片耗尽或阻塞（如调用`sleep`），再恢复低优先级实体执行。

### <strong>中断的异步特性：中断上下文与进程上下文的资源竞争
中断是嵌入式系统“响应硬件事件”的核心机制，但其异步性（触发时间不可预测）和高优先级（可打断所有进程/线程）是“最难调试竞态”的根源。[I]级内容需深入中断的“硬件触发逻辑”和“内核处理流程”，解析其与进程上下文冲突的必然性。</strong>

1.2.1 中断的硬件-内核交互逻辑：“不可预测的高优先级插入”
中断的异步性源于“硬件事件的随机性”（如按键按下、传感器数据就绪），其触发与内核调度完全无关，且优先级由硬件设计决定（不可被进程/线程打断）。我们以“外部中断”为例，拆解硬件-内核的交互流程：
1.  硬件触发：外设（如按键）产生中断信号，发送到中断控制器（如ARM GIC）；
2.  优先级仲裁：中断控制器判断中断优先级，若高于当前运行的CPU任务，向CPU发送“中断请求”（IRQ）；
3.  内核接管：CPU暂停当前执行流（进程/线程），保存上下文，跳转到中断服务程序（ISR）入口；
4.  ISR执行：执行驱动注册的ISR，处理硬件事件（如修改共享变量、读取寄存器）；
5.  恢复执行：ISR执行完成后，内核恢复被打断的执行流上下文，继续执行。

这个流程的核心矛盾的：ISR的触发时间无法预测，且会“强制插入”进程/线程的任意执行步骤中
——若进程正执行驱动的“共享资源访问步骤”，ISR的插入会直接拆分该步骤，导致竞态。<br>

### <strong>多核架构的缓存一致性：不同核心缓存共享数据的同步问题
随着嵌入式多核CPU（如ARM Cortex-A53四核、RISC-V多核）的普及，多核并行执行成为竞态的新根源
——与单CPU的“交替执行”不同，多核架构中多个执行流在不同核心上“真正同时运行”，且核心间的缓存机制会导致“共享数据的一致性问题”，这是[I]级内容的重点和难点。</strong>

1.3.1 多核缓存的基本架构：“局部缓存与全局内存的矛盾”
多核CPU的每个核心都有独立的一级缓存（L1，分为指令缓存I-Cache和数据缓存D-Cache）和共享的二级缓存（L2）/三级缓存（L3），架构示意图如下：

缓存的核心作用是“减少CPU访问内存的延迟”——CPU读取数据时，会先从L1缓存获取，若未命中（Cache Miss）再逐级访问L2、L3、内存，并将数据缓存到各级缓存中；写入数据时，会先更新缓存，再通过“缓存回写策略”同步到内存。

这种架构的核心矛盾是：同一共享数据可能被多个核心的L1缓存持有，若某核心修改了缓存中的数据，未及时同步到内存或其他核心缓存，会导致不同核心读取到不同的“数据副本”
——这就是“缓存一致性问题”，是多核竞态的独特根源。<br>

### <strong>上一节解析了执行流的“动态特性”（抢占、异步、并行），而竞态的发生必须依托“静态的共享资源”——若多个执行流访问的是“私有资源”（如进程专属的栈数据），即便执行流频繁切换也不会产生竞态。
因此，[I]级内容的核心是：明确驱动中哪些资源是“共享的”、不同共享资源的竞态风险点是什么、如何判定资源访问是否安全。本节从“软件”“硬件”两个维度拆解共享资源，并给出可重入性判定的实操方法。</strong>


### <strong>软件层面：全局变量、链表、缓冲区等
软件共享资源是驱动代码中“人为定义的、可被多执行流访问的数据结构”，其竞态风险源于“数据结构的操作非原子性”和“访问范围未受限”。</strong>

2.1.1 全局变量与静态变量：最易忽视的共享风险
全局变量（定义在函数外的变量）和静态变量（用`static`修饰的变量，包括函数内静态变量）是驱动中最常见的软件共享资源，其核心风险是“默认无访问边界，所有执行流均可读写”。

我们按“风险等级”拆解典型场景：
1.  普通全局变量（如计数器、状态标志）
    典型案例：驱动中定义`static int dev_status = 0`（0=未初始化，1=就绪），`probe`函数（驱动初始化）将其设为1，`remove`函数（驱动卸载）将其设为0，同时`read`接口读取该状态。  
    竞态风险：`remove`函数执行“dev_status=0”时，若`read`接口正读取`dev_status`，会出现“状态读取错误”；更严重的是，若`remove`函数卸载硬件后，`read`接口因读取到旧值“1”而继续操作硬件，会导致“空指针访问”或“硬件操作异常”。  
    代码风险点示例（无同步控制）：
    ```c
    // 静态全局变量（共享资源）
    static int dev_status = 0;

    // 驱动初始化：写资源
    static int demo_probe(struct platform_device *pdev) {
        // 多步骤初始化硬件后设置状态
        dev_status = 1;  // 非原子写操作
        return 0;
    }

    // 驱动卸载：写资源
    static int demo_remove(struct platform_device *pdev) {
        dev_status = 0;  // 非原子写操作
        // 卸载硬件...
        return 0;
    }

    // 读接口：读资源
    static ssize_t demo_read(struct file *filp, char __user *buf, size_t count, loff_t *pos) {
        int status = dev_status;  // 非原子读操作
        copy_to_user(buf, &status, sizeof(status));
        return sizeof(status);
    }
    ```

2.  函数内静态变量：隐藏的共享陷阱
    函数内静态变量虽“作用域限于函数”，但“生命周期贯穿驱动运行全程”，多个执行流调用同一函数时会共享该变量。  
    典型案例：驱动`<span class="green">ioctl</span>`接口中用静态变量缓存硬件版本号（避免重复读取寄存器）：
    ```c
    static long demo_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
        if (cmd == GET_HW_VERSION) {
            static int hw_version = -1;  // 函数内静态变量（共享）
            if (hw_version == -1) {
                // 从硬件寄存器读取版本号（耗时操作）
                hw_version = readl(HW_VERSION_REG);
            }
            copy_to_user((void __user *)arg, &hw_version, sizeof(hw_version));
        }
        return 0;
    }
    ```
    竞态风险：进程A调用`<span class="green">ioctl</span>`时，`hw_version=-1`，进入“读取寄存器”步骤；此时进程B调用`ioctl`，发现`hw_version=-1`，也进入“读取寄存器”步骤——两个进程同时读取同一寄存器，若寄存器读取是“多步骤操作”（如先写地址再读数据），会导致读取到错误版本号。

[I]级核心认知：
所有全局变量和静态变量默认都是“共享资源”，除非能100%确认“仅单个执行流访问”（如仅`probe`函数初始化，后续无读写），否则必须做同步控制。<br>

### <strong>硬件层面：设备寄存器、IO端口、DMA缓冲区等</strong>

硬件共享资源是驱动操作的“物理对象”，其竞态风险源于“硬件资源的独占性”和“操作的时序敏感性”
——不同于软件资源可通过代码优化隔离，硬件资源的访问规则由硬件设计决定，违规访问会直接导致硬件异常。[I]级读者需掌握“硬件资源的访问特性”，避免因“<span class="red">并发</span>访问违反硬件规范”导致设备损坏。

2.2.1 设备寄存器：硬件控制的“核心入口”风险
寄存器是驱动与硬件交互的“最小单元”，分为“控制寄存器”“状态寄存器”“数据寄存器”等，其核心风险是“多执行流对寄存器的非原子配置”和“时序违规”。

按寄存器类型拆解风险场景：
1.  控制寄存器：多步骤配置的时序风险 
    控制寄存器（如PWM的频率寄存器、ADC的采样率寄存器）常需要“多步配置”（如先写配置值，再写使能位），步骤间被打断会导致“配置无效”或“硬件异常”。  
    典型案例：<span class="green">SPI</span>控制器的模式配置（CPOL=1，CPHA=1）：
    ```c
    // SPI控制寄存器地址定义
    #define SPI_MODE_REG 0x10000000
    #define SPI_EN_REG   0x10000004

    // 配置SPI模式（多步骤操作）
    static void spi_set_mode(void) {
        writel(0x03, SPI_MODE_REG);  // 步骤1：设置CPOL=1，CPHA=1
        // 若此处被中断或抢占打断，硬件未使能，配置未生效
        writel(0x01, SPI_EN_REG);    // 步骤2：使能SPI控制器
    }
    ```
    竞态风险：步骤1执行后，若被中断打断，中断服务程序调用`spi_set_mode`重新配置——步骤1被重复执行，步骤2执行时可能导致<span class="green">SPI</span>控制器“模式切换中使能”，触发硬件复位。

2.  数据寄存器：读写<span class="red">并发</span>的覆盖风险 
    数据寄存器（如<span class="green">UART</span>的发送/接收寄存器、GPIO的数据寄存器）用于传输数据，多个执行流同时读写会导致“数据覆盖”。  
    典型案例：<span class="green">GPIO</span>数据寄存器（`GPIO_DATA_REG`）控制LED亮灭：
    - 进程A调用`<span class="green">ioctl</span>`写`GPIO_DATA_REG=0x01`（点亮LED1）；
    - 进程B调用`<span class="green">ioctl</span>`写`GPIO_DATA_REG=0x02`（点亮LED2）；
    - 若两个进程的写操作被抢占拆分，可能出现“LED1和LED2均不亮”（如进程A写0x01后被打断，进程B写0x02后被打断，进程A恢复后写0x01覆盖）。

硬件寄存器的核心特性：
部分寄存器（如只读状态寄存器）支持多执行流同时读取，但“写寄存器”几乎都是独占操作，需严格保证“同一时间只有一个执行流写”。<br>

### <strong>共享资源的可重入性判定方法</strong>

可重入性（Reentrancy）是判断“函数/驱动是否存在竞态风险”的核心标准
——可重入函数指“多个执行流同时调用时，不会出现数据异常的函数”；反之则为“不可重入函数”。[I]级读者需掌握“可重入性的判定维度”和“实操方法”，从源头规避竞态。

2.3.1 可重入函数的核心判定维度
驱动函数的可重入性取决于“是否使用不可重入资源”，具体判定维度如下表：
| 判定维度                | 不可重入特征（存在则不可重入）                          | 可重入特征（全部满足则可重入）                          |
|-------------------------|---------------------------------------------------------|---------------------------------------------------------|
| 资源使用                | 使用全局变量、静态变量、全局链表、共享缓冲区            | 仅使用函数参数、局部变量（栈上）、进程私有数据（file->private_data） 
| 操作特性                | 执行非原子操作（如多步骤写寄存器、链表插入）            | 仅执行原子操作（如单指令读寄存器、局部变量运算）        |
| 外部依赖                | 调用不可重入函数（如`strtok`、未同步的驱动接口）、依赖硬件独占资源 | 调用可重入函数（如`memcpy`）、仅访问进程专属硬件资源 |
| 阻塞行为                | 在持有共享资源时阻塞（如持有锁时调用`msleep`）          | 无阻塞行为，或阻塞前释放所有共享资源                    |

典型可重入函数示例：仅使用局部变量和函数参数的加法函数：
```c
// 可重入函数：无共享资源，仅局部变量运算
static int add(int a, int b) {
    int sum = a + b;  // 局部变量（栈上，进程私有）
    return sum;
}
```

典型不可重入函数示例：使用静态变量的字符串分割函数（模拟`strtok`）：
```c
// 不可重入函数：使用静态变量保存分割位置
static char *my_strtok(char *str, const char *delim) {
    static char *save_ptr;  // 静态变量（共享）
    if (str != NULL) {
        save_ptr = str;
    }
    // 分割逻辑...（依赖save_ptr的状态）
    return token;
}
```<br>

### <strong>共享资源与竞态的核心关联总结</strong>

为帮助[I]级读者建立“资源-风险-控制”的关联思维，我们用表格梳理不同资源的核心风险及初步控制方向：
| 资源类型               | 典型实例                  | 核心竞态风险                  | 初步控制方向                  |
|------------------------|---------------------------|-------------------------------|-------------------------------|
| 软件-全局/静态变量     | 设备状态标志、计数器      | 非原子读写导致数据错乱        | 原子变量、互斥体              |
| 软件-链表/哈希表       | 设备节点链表              | 插入/删除步骤拆分导致结构损坏  | 读写锁、自旋锁                |
| 软件-缓冲区            | 环形接收缓冲区            | 读写位置错乱导致数据覆盖      | 原子指针、信号量              |
| 硬件-寄存器            | PWM控制寄存器             | 多步骤配置时序违规            | 自旋锁（中断上下文）、互斥体  |
| 硬件-MMIO/IO端口       | 中断屏蔽寄存器            | 缓存不一致、总线冲突          | ioremap_nocache、总线锁        |
| 硬件-DMA缓冲区         | ADC采集缓冲区             | CPU与DMA并行访问数据错乱      | DMA中断同步、内存屏障          |<br>

### <strong>前文解析了竞态的“动态诱因”（执行流特性）和“静态载体”（共享资源），而竞态控制的最高境界是“从根源避免竞态发生”——通过设计层面的规则约束，消除竞态产生的三个核心条件（多执行流、共享资源、非原子操作）中的任意一个。
本节的三个核心法则，分别从“共享资源范围”“访问原子性”“执行流交互”三个维度切断竞态链条，其优先级高于后续的“同步工具使用”（工具是“补救措施”，法则是“预防方案”）。</strong>


### <strong>最小权限原则：共享资源的访问范围控制
最小权限原则的核心逻辑：共享资源的访问者越少、访问权限越受限，竞态风险越低。
嵌入式驱动中，绝大多数竞态源于“无意识地扩大了共享资源的访问范围”（如将“进程专属数据”定义为全局变量），通过收缩访问范围和权限，可从根源消除共享，自然避免竞态。</strong>

3.1.1 核心原理：从“共享”到“私有”的转化
竞态的前提是“多执行流共享资源”，若能将“共享资源”转化为“执行流私有资源”，或严格限制访问者数量，就能直接消除竞态。该法则的本质是“减少参与<span class="red">并发</span>的执行流数量”，对应竞态产生条件的“第一点（多个执行流）”——无共享则无竞态。

举个直观的类比：一间办公室有一台公共打印机（共享资源），多人同时使用会排队（竞态）；若给每人配一台专属打印机（私有资源），则无排队问题。驱动开发中，“给每个进程配专属资源”就是最小权限原则的核心落地思路。<br>

### <strong>原子操作原则：不可分割的资源访问单元定义
原子操作原则的核心逻辑：若共享资源的访问必须存在，則将访问操作定义为“不可被打断的原子单元”，直接消除竞态产生的“第三点（非原子操作）”。
该法则是“无法避免共享时”的最优策略——操作不可被打断，即使多执行流<span class="red">并发</span>访问，也不会出现数据错乱。</strong>

3.1.1 核心原理：原子操作的“硬件+内核”双重保障
原子操作的“不可分割性”源于两个层面：
1.  硬件层面：CPU支持“单指令原子操作”，如ARM的`ldrex`（加载并锁定）和`strex`（存储并解锁）指令，可实现“读-改-写”的原子化；
2.  内核层面：基于硬件指令封装原子操作API（如`atomic_t`系列、位操作系列），屏蔽硬件差异，提供统一接口。

对[I]级读者，需明确：原子操作仅适用于“简单数据类型”（如整数、位），复杂操作（如链表插入、多寄存器配置）无法用原子操作实现，需结合其他法则或同步工具。<br>

### <strong>执行流隔离原则：进程/中断/多核间的资源访问边界
*行流隔离原则的核心逻辑：若共享资源和原子操作均无法避免，则将不同类型执行流的访问“物理隔离”或“时序隔离”*消除“多执行流同时访问”的条件。
该法则针对“执行流特性差异”（如中断的高优先级、多核的并行性），通过隔离策略切断执行流与共享资源的冲突链路</strong>

3.3.1 核心原理：基于执行流特性的“差异化隔离”
不同执行流的核心冲突在于“优先级差异”（中断>进程）和“运行载体差异”（多核并行），隔离策略需针对性设计：
- 进程-中断隔离：将中断的“共享资源访问”转移到进程上下文，避免中断直接访问共享资源；
- 多核隔离：为每个核心分配独立资源副本，避免跨核心共享；
- 时序隔离：通过“忙标志”或“调度控制”，让不同执行流在不同时间访问同一资源。<br>

### <strong>核心法则的优先级与适用场景总结</strong>

[I]级读者需建立“法则优先于工具”的思维，按以下优先级选择竞态避免策略，仅当法则无法覆盖时，才考虑自旋锁、互斥体等同步工具：
| 法则优先级 | 核心逻辑                | 适用场景                                  | 不适用场景                              | 嵌入式典型案例                          |
|------------|-------------------------|-------------------------------------------|-----------------------------------------|-----------------------------------------|
| 1          | 最小权限（消除共享）    | 进程专属数据、可拆分的资源                | 必须全局共享的资源（如总线、硬件寄存器）| 用file->private_data存储进程专属计数    |
| 2          | 原子操作（简化访问）    | 简单整数、位状态的共享访问                | 复杂操作（链表、多寄存器配置）          | 中断+进程的按键计数统计                |
| 3          | 执行流隔离（分开访问）  | 中断-进程冲突、多核并行、硬件总线共享      | 简单共享场景（无需隔离成本）            | 工作队列转移中断处理、I2C总线忙标志排队 |<br>
