### <strong>嵌入式系统的权限漏洞多源于“设备文件权限过松”或“内核层缺乏权限校验”——默认情况下，字符设备文件权限为`crw-rw----`（仅root和设备组可访问），但实际场景中需细化到“特定用户组可读写”“部分操作需管理员权限”。
权限控制需实现“<span class="red">用户态</span>文件权限隔离+内核层操作权限校验”的双重防护。</strong>


### <strong>文件权限配置：udev规则设置设备文件权限</strong>

udev是Linux<span class="red">用户态</span>设备管理工具，可通过规则文件定义“设备节点创建时的权限、所属用户组、符号链接”，替代传统的`mknod`命令手动配置，适配设备热插拔场景。核心解决“默认权限过松或过严”问题（如让“sensor组”用户可读写温湿度传感器，普通用户仅可读）。

###### 1.1.1 实战场景：SHT30传感器设备文件权限定制
以前面实战的<span class="green">I2C</span>接口SHT30传感器（设备节点`/dev/sht30_sensor`）为例，实现“仅`sensoradmin`用户组可读写，其他用户无权限”的权限配置：

###### 1.1.2 udev规则编写与部署
1.  **识别设备唯一属性**：udev通过“设备 subsystem、devpath、厂商ID、产品ID”等属性匹配设备，需先获取SHT30的udev属性：
   ```bash
   # 查看设备udev信息（替换sht30_sensor为实际设备名）
   udevadm info --query=property --name=/dev/sht30_sensor
   # 输出关键属性（不同设备值不同，需实际查询）：
   SUBSYSTEM=i2c
   DEVPATH=/devices/platform/soc/2100000.i2c/i2c-0/0-0044/sht30_sensor
   MAJOR=241
   MINOR=0
   ```

2.  **编写udev规则文件**：  
   在`/etc/udev/rules.d/`目录下创建`99-sht30-sensor.rules`，规则格式为“匹配条件+执行动作”：
   ```rules
   # 匹配条件：SUBSYSTEM为i2c且MAJOR=241（设备主号）
   SUBSYSTEM=="i2c", KERNEL=="sht30_sensor", ENV{MAJOR}=="241", \
   # 执行动作：设置权限660（组读写），所属用户root，所属组sensoradmin
   MODE="0660", OWNER="root", GROUP="sensoradmin", \
   # 可选：创建符号链接/dev/sensor/temp_humi（简化访问）
   SYMLINK+="sensor/temp_humi"
   ```
   - 权限说明：`0660`表示“所有者（root）和组（sensoradmin）可读写，其他用户无权限”，对应字符权限`crw-rw----`；
   - 组管理：需先创建`sensoradmin`组并添加用户（如将用户`operator`加入组）：
     ```bash
     groupadd sensoradmin
     usermod -aG sensoradmin operator  # 将operator用户加入组
     ```

3.  **生效规则并验证**：
   ```bash
   # 重新加载udev规则
   udevadm control --reload-rules && udevadm trigger
   # 验证设备节点权限（重点看权限位和所属组）
   ls -l /dev/sht30_sensor /dev/sensor/temp_humi
   # 输出示例（符合预期）：
   crw-rw---- 1 root sensoradmin 241, 0 12月 10 14:30 /dev/sht30_sensor
   lrwxrwxrwx 1 root root         12 12月 10 14:30 /dev/sensor/temp_humi -> ../sht30_sensor
   ```

###### 1.1.3 故障排查：udev规则不生效的解决方法
| 问题现象                | 排查步骤                                                                 | 解决方案                                                                 |
|-------------------------|--------------------------------------------------------------------------|--------------------------------------------------------------------------|
| 设备节点权限未变更      | 1. 用udevadm test验证规则匹配：<br>`udevadm test /devices/.../sht30_sensor` | 1. 检查规则中MAJOR/MINOR是否与实际一致；<br>2. 规则文件后缀必须为.rules |
| 符号链接未创建          | 1. 查看udev日志：<br>`journalctl -u systemd-udevd -f`                    | 1. 确保SYMLINK+="sensor/temp_humi"中路径无冲突；<br>2. 手动创建父目录`mkdir /dev/sensor` |
| 组权限生效但用户无权限  | 1. 检查用户是否在目标组：<br>`groups operator`                            | 1. 重新登录用户（usermod后需重新登录生效）；<br>2. 用`newgrp sensoradmin`临时切换组 |<br>

### <strong>内核层权限校验：capability机制的应用</strong>

传统的“root用户全权限，普通用户无权限”的权限模型过于粗放——例如“读取传感器数据”无需root权限，但“校准传感器零点”需管理员权限，若用root运行应用会扩大安全风险。Linux Capability机制将root权限拆分为39个细粒度权限（如`CAP_SYS_ADMIN`控制系统管理操作，`CAP_NET_ADMIN`控制网络配置），驱动可在特定操作前校验应用是否具备对应权限，实现“最小权限原则”。

###### 1.2.1 Capability核心原理与优势
- 核心原理：每个进程有三个Capability掩码（`Permitted`：进程可拥有的权限；`Effective`：当前生效的权限；`Inheritable`：子进程可继承的权限），驱动通过`capable()`函数校验进程的Effective掩码；
- 优势：相比SUID（给应用加root权限），Capability仅授予必要权限（如仅授予`CAP_SYS_RAWIO`而非全root），降低权限泄露风险。

###### 1.2.2 实战：驱动层Capability权限校验实现
以SHT30传感器驱动为例，实现“普通读取无需权限，校准操作需`CAP_SYS_ADMIN`权限”的校验逻辑：

1.  **驱动代码修改（新增权限校验）**：
   ```c
   #include <linux/capability.h>  // Capability头文件
   #include <linux/security.h>

   // 传感器校准命令（ioctl命令码）
   #define SHT30_IOCTL_CALIBRATE _IO(0x48, 0x01)

   // ioctl函数中新增权限校验逻辑
   static long sht30_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
       struct sensor_dev *dev = filp->private_data;

       switch (cmd) {
           // 普通操作：读取数据，无需权限校验
           case SHT30_IOCTL_GET_DATA:
               // ... 原有读取逻辑 ...
               return 0;

           // 敏感操作：校准传感器，需CAP_SYS_ADMIN权限
           case SHT30_IOCTL_CALIBRATE:
               // 核心：校验进程是否具备CAP_SYS_ADMIN权限
               if (!capable(CAP_SYS_ADMIN)) {
                   dev_err(&dev->client->dev, "Calibrate need CAP_SYS_ADMIN!\n");
                   return -EPERM;  // 返回权限不足错误
               }
               // ... 原有校准逻辑（如写入校准寄存器） ...
               dev_info(&dev->client->dev, "Sensor calibrated success!\n");
               return 0;

           default:
               return -EINVAL;
       }
   }
   ```

2.  **应用层权限配置与验证**：
   - 编译校准应用（`sensor_calibrate.c`），核心逻辑是调用`SHT30_IOCTL_CALIBRATE`命令；
   - 普通用户直接运行会失败（权限不足），给应用添加`CAP_SYS_ADMIN`权限后可运行：
     ```bash
     # 1. 交叉编译校准应用
     arm-linux-gnueabihf-gcc sensor_calibrate.c -o sensor_calibrate

     # 2. 普通用户（operator）直接运行（失败）
     su operator
     ./sensor_calibrate
     # 输出错误（驱动返回EPERM）：
     Ioctl calibrate failed! errno=1: Operation not permitted

     # 3. 给应用添加CAP_SYS_ADMIN权限（root执行）
     su root
     setcap cap_sys_admin+ep ./sensor_calibrate

     # 4. 验证权限已添加
     getcap ./sensor_calibrate
     # 输出示例（符合预期）：
     ./sensor_calibrate = cap_sys_admin+ep

     # 5. 普通用户再次运行（成功）
     su operator
     ./sensor_calibrate
     # 输出成功日志：
     Sensor calibrated success!

     # 6. 查看驱动日志（验证权限校验生效）
     dmesg | grep Calibrate
     # 输出示例：
     [1234.567890] Sensor calibrated success!
     ```

3.  **权限回收与管理**：
    ```bash
    # 移除应用的CAP_SYS_ADMIN权限
    setcap -r ./sensor_calibrate
    # 批量查看系统中带Capability的应用
    find / -xdev -type f -exec getcap {} \; 2>/dev/null
    ```<br>

### <strong>嵌入式接口的“数据安全”核心是“防注入、防篡改、防溢出”——工业场景中，恶意应用通过构造非法输入（如超长字符串、无效指针、越界数值）调用驱动接口，可能引发内核缓冲区溢出、数据错乱甚至系统崩溃。
需通过“输入参数校验”和“敏感操作身份认证”构建数据安全防线。</strong>


### <strong>输入参数校验：防止缓冲区溢出的校验逻辑</strong>

缓冲区溢出是嵌入式系统最常见的安全漏洞之一——若驱动未校验应用传入的参数长度，应用传入超长数据会覆盖内核内存，导致系统崩溃或权限提升。校验需覆盖“参数长度、数值范围、指针有效性”三大维度，重点关注`<span class="green">ioctl</span>`、`write`等接收用户数据的接口。

###### 2.1.1 漏洞复现：未校验参数导致的缓冲区溢出
以SHT30驱动的`write`接口为例，未校验输入长度时的漏洞代码：
```c
// 漏洞代码：未校验用户输入长度
static ssize_t sht30_write(struct file *filp, const char __user *buf, size_t count, loff_t *ppos) {
    struct sensor_dev *dev = filp->private_data;
    char kernel_buf[32];  // 内核缓冲区大小32字节

    // 直接拷贝用户数据（无长度校验，count>32时溢出）
    if (copy_from_user(kernel_buf, buf, count)) {
        return -EFAULT;
    }

    // ... 解析kernel_buf配置传感器 ...
    return count;
}
```
- 漏洞触发：应用向驱动写入40字节数据（超过内核缓冲区32字节）：
  ```c
  // 恶意应用代码（触发溢出）
  char bad_data[40] = {0x00};
  write(fd, bad_data, 40);  // 写入40字节
  ```
- 运行结果：内核崩溃，打印Oops日志（提示“stack smashing detected”）。

###### 2.1.2 加固方案：全维度输入校验实现
针对上述漏洞，从“长度、范围、指针”三个维度优化驱动代码，关键校验函数及逻辑如下：
```c
static ssize_t sht30_write(struct file *filp, const char __user *buf, size_t count, loff_t *ppos) {
    struct sensor_dev *dev = filp->private_data;
    char kernel_buf[32];
    int ret, config_val;

    // 校验1：输入长度校验（防止缓冲区溢出）
    if (count > sizeof(kernel_buf)) {
        dev_err(&dev->client->dev, "Input too long! max=%d, req=%zu\n",
                sizeof(kernel_buf), count);
        return -EINVAL;  // 输入过长，返回无效参数错误
    }

    // 校验2：用户指针有效性校验（防止非法指针）
    if (!access_ok(buf, count)) {
        dev_err(&dev->client->dev, "Invalid user buffer pointer!\n");
        return -EFAULT;
    }

    // 拷贝用户数据（此时count已安全）
    if (copy_from_user(kernel_buf, buf, count)) {
        return -EFAULT;
    }

    // 校验3：业务参数范围校验（防止无效配置）
    // 示例：解析“校准值”，范围需在0-100之间
    ret = kstrtoint(kernel_buf, 10, &config_val);
    if (ret < 0 || config_val < 0 || config_val > 100) {
        dev_err(&dev->client->dev, "Invalid config value! range 0-100\n");
        return -EINVAL;
    }

    // ... 安全的配置逻辑 ...
    dev_info(&dev->client->dev, "Config success! val=%d\n", config_val);
    return count;
}
```

###### 2.1.3 关键校验函数说明
| 校验类型         | 核心函数/宏                | 作用说明                                                                 |
|------------------|----------------------------|--------------------------------------------------------------------------|
| 长度校验         | 直接比较`count`与缓冲区大小 | 强制限制用户输入不超过内核缓冲区，是防溢出的第一道防线                   |
| 指针有效性校验   | `access_ok(type, addr, size)` | 校验用户空间指针是否可访问（避免应用传入无效指针如NULL、内核地址）       |
| 数值范围校验     | `kstrtoint`/`kstrtoul`     | 将用户输入的字符串转为整数并校验范围（避免如“校准值为-10”的非法配置）   |
| ioctl参数校验    | `_IOC_SIZE(cmd)`           | 校验ioctl命令对应的参数大小（如结构体大小不匹配时返回错误）              |<br>

### <strong>敏感操作限制：基于身份认证的接口访问控制</strong>

权限控制（Capability）解决“是否有权限操作”，身份认证解决“是否是合法用户操作”——工业场景中，部分敏感操作（如传感器校准、设备固件升级）不仅需要权限，还需验证操作发起者的身份（如仅允许“admin”用户执行）。可通过“用户ID（uid）+ 密钥验证”实现双重身份认证。

###### 2.2.1 实战：驱动层uid+密钥双重认证实现
以SHT30传感器的“校准操作”为例，实现“仅uid=1000（admin用户）且提供正确密钥，才能执行校准”的逻辑：

1.  **驱动代码修改（新增身份认证）**：
   ```c
   // 定义管理员uid和校准密钥（实际场景建议从设备树或配置文件读取）
   #define ADMIN_UID 1000
   #define CALIBRATE_KEY 0x12345678

   // 校准参数结构体（含密钥）
   struct calibrate_param {
       uint32_t key;        // 认证密钥
       int cal_val;         // 校准值
   };

   static long sht30_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
       struct sensor_dev *dev = filp->private_data;
       struct calibrate_param param;
       struct cred *cred = current_cred();  // 获取当前进程的凭证（含uid）

       switch (cmd) {
           case SHT30_IOCTL_CALIBRATE:
               // 校验1：Capability权限校验（基础权限）
               if (!capable(CAP_SYS_ADMIN)) {
                   return -EPERM;
               }

               // 校验2：用户身份uid校验（仅admin用户）
               if (cred->uid.val != ADMIN_UID) {
                   dev_err(&dev->client->dev, "Only admin(uid=%d) can calibrate!\n", ADMIN_UID);
                   return -EACCES;  // 权限被拒绝
               }

               // 校验3：密钥验证（防止权限泄露后滥用）
               if (copy_from_user(&param, (void __user *)arg, sizeof(param))) {
                   return -EFAULT;
               }
               if (param.key != CALIBRATE_KEY) {
                   dev_err(&dev->client->dev, "Invalid calibrate key!\n");
                   return -EKEYREJECTED;  // 密钥无效
               }

               // 校验4：校准值范围校验（业务安全）
               if (param.cal_val < 0 || param.cal_val > 100) {
                   return -EINVAL;
               }

               // ... 安全执行校准逻辑 ...
               dev_info(&dev->client->dev, "Admin calibrate success! val=%d\n", param.cal_val);
               return 0;

           // ... 其他命令逻辑 ...
       }
   }
   ```

2.  **应用层认证与验证**：
   ```c
   // 应用层校准代码（需传入正确uid和密钥）
   #include <stdio.h>
   #include <fcntl.h>
   #include <sys/ioctl.h>

   #define SHT30_IOCTL_CALIBRATE _IOW(0x48, 0x01, struct calibrate_param)
   struct calibrate_param {
       uint32_t key;
       int cal_val;
   };

   int main(void) {
       int fd;
       struct calibrate_param param = {
           .key = 0x12345678,  // 正确密钥
           .cal_val = 50       // 合法校准值
       };

       fd = open("/dev/sht30_sensor", O_RDWR);
       if (fd < 0) { perror("open"); return -1; }

       // 执行校准（需uid=1000且带CAP_SYS_ADMIN权限）
       if (ioctl(fd, SHT30_IOCTL_CALIBRATE, &param) < 0) {
           perror("ioctl calibrate");
           close(fd);
           return -1;
       }

       printf("Calibrate success!\n");
       close(fd);
       return 0;
   }
   ```

3.  **多场景验证结果**：
   | 测试场景                          | 预期结果                  | 实际输出                          |
   |-----------------------------------|---------------------------|-----------------------------------|
   | uid=1000+正确密钥+有CAP权限       | 校准成功                  | Calibrate success!                |
   | uid=1001（普通用户）+正确密钥     | 身份校验失败              | ioctl calibrate: Permission denied |
   | uid=1000+错误密钥（0x87654321）   | 密钥校验失败              | ioctl calibrate: Key rejected     |
   | uid=1000+正确密钥+校准值=150      | 数值范围校验失败          | ioctl calibrate: Invalid argument |

###### 2.2.2 工业级优化：密钥动态更新与存储安全
- 静态密钥风险：驱动中硬编码密钥易被逆向破解，优化为“从安全存储区（如eMMC加密分区）读取密钥”，或通过`<span class="green">ioctl</span>`命令动态更新密钥（需最高权限）；
- 密钥传输安全：若应用与驱动通过网络传输密钥，需加密传输（如AES加密），避免明文泄露。


### 接口安全加固全流程总结
嵌入式接口安全加固需遵循“**分层防护、最小权限、全链路校验**”三大原则，全流程如下：
```mermaid
flowchart TD
    A[安全需求分析] -->|明确敏感操作与风险| B[权限控制设计]
    B --> B1[udev规则配置设备文件权限]
    B --> B2[内核层Capability校验操作权限]
    A --> C[数据安全设计]
    C --> C1[输入参数校验（长度/范围/指针）]
    C --> C2[敏感操作身份认证（uid+密钥）]
    B1 --> D[驱动开发集成校验逻辑]
    B2 --> D
    C1 --> D
    C2 --> D
    D --> E[应用层权限配置（setcap/用户组）]
    E --> F[安全测试（漏洞复现+边界测试）]
    F --> G[上线运行+日志审计]
```<br>
