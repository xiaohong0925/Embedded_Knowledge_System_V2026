# 核心原理与数据结构

> 📊 **本节难度等级：** <span class="badge-ie">**IE级**</span>

---

### <strong>块设备驱动的核心是“内核通过标准化数据结构管理设备，通过调度机制优化IO性能”
——`request_queue`（IO请求队列）与`gendisk`（磁盘描述符）构成驱动的“骨架”，IO调度器是性能优化的“核心引擎”，而从`alloc_disk`到`add_disk`的注册流程则是驱动落地的“必经之路”。
本节从数据结构本质出发，结合代码与实操，打通“原理→实现→优化”的全链路认知。</strong>


### <strong>内核核心结构体：`request_queue`（IO请求队列）与`gendisk`（磁盘描述符）
块<span class="red">设备驱动</span>的所有交互均围绕这两个结构体展开：`request_queue`负责“接收、调度、分发”IO请求，`gendisk`负责“描述设备属性、暴露设备接口”，二者协同实现“设备管理+IO处理”的核心能力。</strong>

1. `request_queue`：IO请求的“调度与管理中心”
`request_queue`是块设备的“IO请求管家”——用户层的读写请求（如`read`/`write`）最终会转化为`request`结构体，存入`request_queue`的请求链表，经IO调度器优化后，由驱动的请求处理函数逐一处理。其核心作用可概括为“收请求、调顺序、发硬件”。

（1）关键成员解析（I级核心，掌握核心成员即可落地）
| 成员名 | 数据类型 | 核心作用 | 工业级配置示例（E级） |
|--------|----------|----------|----------------------|
| `queue_head` | `struct list_head` | 存储待处理的`request`请求链表 | 驱动无需直接操作，由内核调度器管理 |
| `elevator` | `struct elevator_queue *` | 指向当前使用的IO调度器（如Deadline） | 通过`blk_set_elevator`切换调度器，嵌入式常用`elevator_deadline` |
| `hardsect_size` | `unsigned int` | 设备物理扇区大小（如512字节） | 必须与硬件一致，否则导致读写错误：`queue->hardsect_size = 512;` |
| `max_sectors` | `unsigned int` | 单次IO最大扇区数（硬件限制） | eMMC通常设为256（128KB）：`blk_queue_max_sectors(queue, 256);` |
| `request_fn` | `request_fn_proc *` | 驱动的请求处理函数（核心回调） | 绑定自定义处理函数：`queue->request_fn = my_disk_request_fn;` |
| `private_data` | `void *` | 驱动私有数据指针（关联`gendisk`） | 存储设备上下文：`queue->private_data = gdisk;` |

（2）核心操作函数（I级实操，结合代码理解）
`request_queue`的生命周期由“初始化→配置→销毁”构成，核心函数如下：
```c
#include <linux/blkdev.h>

// 1. 初始化请求队列（绑定请求处理函数）
struct request_queue *blk_init_queue(request_fn_proc *rfn, struct spinlock *lock) {
    struct request_queue *queue = blk_alloc_queue(GFP_KERNEL);
    if (!queue) return NULL;
    queue->request_fn = rfn; // 绑定处理函数
    queue->queue_lock = lock; // 并发保护锁
    return queue;
}

// 2. 配置队列硬件限制（E级：必须匹配硬件能力）
void blk_queue_hardsect_size(struct request_queue *q, unsigned int size) {
    q->hardsect_size = size;
    q->logical_block_size = size;
    q->physical_block_size = size;
}

// 3. 销毁请求队列（资源释放，与初始化反向）
void blk_cleanup_queue(struct request_queue *q) {
    blk_destroy_queue(q);
}
```
> 关键提示（E级）：`blk_init_queue`的`lock`参数必须传入自旋锁（`spinlock`），因IO请求处理涉及中断上下文与进程上下文<span class="red">并发</span>，自旋锁可确保临界区原子性；若传入NULL，内核会自动创建锁，但建议自定义锁以提升灵活性。<br>

### <strong>IO调度机制：CFQ/Deadline/noop算法的特性与嵌入式选型建议
IO调度器（IO Scheduler，又称电梯调度器）是块设备性能优化的“核心引擎”，其作用是对`request_queue`中的请求进行“排序、合并、延迟控制”，以减少硬件寻道/切换延迟，提升IO效率。嵌入式场景中，调度器的选型直接决定<span class="green">eMMC</span>/SD卡的读写性能，需结合硬件特性精准适配。</strong>

1. 三大主流调度算法核心特性（I→E级对比）
| 算法名称 | 核心逻辑 | 优点 | 缺点 | 适用硬件 | 嵌入式场景建议 |
|----------|----------|------|------|----------|----------------|
| CFQ（Completely Fair Queueing，完全公平队列） | 为每个进程分配独立IO队列，按时间片轮询处理 | 进程IO公平性高，避免单个进程独占IO | 调度开销大，随机IO性能差 | 机械硬盘（PC场景） | 不推荐（嵌入式多为单进程或轻量多进程，公平性需求低，开销不可接受） |
| Deadline（截止时间调度） | 为每个请求设置截止时间（读请求500ms，写请求5s），优先处理接近截止时间的请求 | 随机IO性能优，读请求优先级高于写（符合嵌入式读多写少场景） | 公平性略低，可能出现单个进程短期占优 | eMMC、SD卡、机械硬盘 | 强烈推荐（嵌入式主流选择，平衡性能与开销） |
| noop（No Operation，无操作调度） | 仅合并连续请求，不排序（请求按接收顺序处理） | 调度开销极小，连续IO性能优 | 无排序优化，随机IO性能极差 | SSD、虚拟块设备（内存盘） | 推荐（SSD无寻道延迟，排序无意义；虚拟块设备速度快，无需调度） |

（1）关键算法深度解析（E级，理解选型本质）
- Deadline算法：嵌入式首选的核心原因  
  Deadline通过“双队列机制”实现性能与延迟的平衡：  
  1. 排序队列：按LBA地址排序，合并连续请求（减少硬件切换）；  
  2. 截止时间队列：按请求的截止时间排序，当请求接近截止时间时，从该队列取出优先处理，避免“饿死”（如长期被连续请求压制的随机读请求）。  
  嵌入式场景中，<span class="green">eMMC</span>的“页切换延迟”类似机械硬盘的寻道延迟，Deadline的排序可大幅减少页切换次数，同时读请求优先的策略适配“系统启动、固件读取”等读多写少场景。

- noop算法：SSD/UFS适配的核心逻辑  
  SSD（如UFS）采用闪存芯片存储，无机械寻道延迟，仅存在“块擦除延迟”，排序对性能提升无意义；反而CFQ/Deadline的调度开销会降低IO响应速度。noop仅合并连续请求，既减少无效操作，又保持极低开销，完美适配SSD特性。<br>

### <strong>块设备注册流程：`alloc_disk`到`add_disk`的关键步骤
块<span class="red">设备驱动</span>的注册流程是“数据结构初始化→资源配置→内核注册”的闭环，核心步骤围绕`gendisk`与`request_queue`的创建、绑定展开，每个步骤均有严格的顺序要求与坑点，工业级驱动需确保“资源申请与释放反向”，避免内存泄漏或内核崩溃。</strong>

1. 完整注册流程（I级步骤，结合代码框架）
以“基于内存的虚拟块设备”为例（简化版，可直接编译验证），注册流程分为7个关键步骤：
```c
#include <linux/module.h>
#include <linux/blkdev.h>
#include <linux/spinlock.h>

// 设备配置（I级：可修改为实际硬件参数）
#define DEV_NAME "virt_disk"  // 设备名称
#define DEV_MAJOR 250         // 主设备号（确保未被占用）
#define SECTOR_SIZE 512       // 扇区大小
#define TOTAL_SECTORS 102400  // 总扇区数（50MB：102400×512=52428800字节）

// 全局变量（E级：封装为结构体，便于多设备管理）
struct virt_disk_data {
    struct gendisk *gd;          // 磁盘描述符
    struct request_queue *queue; // IO请求队列
    spinlock_t queue_lock;       // 队列并发锁
    char *disk_buf;              // 虚拟磁盘内存缓冲区
};
static struct virt_disk_data virt_disk;

// 步骤1：请求处理函数（核心，驱动业务逻辑）
static void virt_disk_request_fn(struct request_queue *q) {
    struct request *req;
    struct virt_disk_data *data = q->private_data;
    sector_t lba = blk_rq_pos(req);  // 获取请求的LBA地址
    unsigned int len = blk_rq_sectors(req) * SECTOR_SIZE;  // 请求长度
    char *buf = data->disk_buf + lba * SECTOR_SIZE;  // 缓冲区地址

    // 处理读写请求（E级：需校验LBA+len不超过设备容量，避免越界）
    if (lba + blk_rq_sectors(req) > TOTAL_SECTORS) {
        blk_end_request_all(req, -EINVAL);
        return;
    }

    if (rq_data_dir(req) == READ) {
        // 读请求：从缓冲区拷贝到请求
        blk_rq_copy_from_buffer(req, buf, len);
    } else {
        // 写请求：从请求拷贝到缓冲区
        blk_rq_copy_to_buffer(req, buf, len);
    }

    blk_end_request_all(req, 0);  // 完成请求
}

// 步骤2：块设备接口集（绑定open/release等）
static struct block_device_operations virt_disk_fops = {
    .owner = THIS_MODULE,
    .open = NULL,  // 简化：无特殊初始化，可留空
    .release = NULL,
};

// 步骤3：模块初始化（注册核心流程）
static int __init virt_disk_init(void) {
    int ret = 0;

    // 1. 初始化私有数据（缓冲区、锁）
    virt_disk.disk_buf = kzalloc(TOTAL_SECTORS * SECTOR_SIZE, GFP_KERNEL);
    if (!virt_disk.disk_buf) return -ENOMEM;
    spin_lock_init(&virt_disk.queue_lock);

    // 2. 初始化请求队列（绑定处理函数与锁）
    virt_disk.queue = blk_init_queue(virt_disk_request_fn, &virt_disk.queue_lock);
    if (!virt_disk.queue) {
        ret = -ENOMEM;
        goto err_queue;
    }
    // 配置队列硬件限制（E级：必须与设备匹配）
    blk_queue_hardsect_size(virt_disk.queue, SECTOR_SIZE);
    blk_queue_max_sectors(virt_disk.queue, TOTAL_SECTORS);

    // 3. 分配gendisk结构体（次设备号数量1，无分区）
    virt_disk.gd = alloc_disk(1);
    if (!virt_disk.gd) {
        ret = -ENOMEM;
        goto err_gdisk;
    }

    // 4. 配置gendisk关键成员
    virt_disk.gd->major = DEV_MAJOR;
    virt_disk.gd->first_minor = 0;
    virt_disk.gd->fops = &virt_disk_fops;
    virt_disk.gd->queue = virt_disk.queue;
    virt_disk.gd->capacity = TOTAL_SECTORS;  // 设置总扇区数
    strcpy(virt_disk.gd->disk_name, DEV_NAME);  // 设置设备名称

    // 5. 注册主设备号
    ret = register_blkdev(DEV_MAJOR, DEV_NAME);
    if (ret < 0) {
        printk(KERN_ERR "register_blkdev failed: %d\n", ret);
        goto err_register;
    }

    // 6. 注册gendisk到内核（用户层可见）
    add_disk(virt_disk.gd);
    printk(KERN_INFO "virt_disk registered: /dev/%s (50MB)\n", DEV_NAME);
    return 0;

    // 错误处理（E级：反向释放资源，避免泄漏）
err_register:
    del_gendisk(virt_disk.gd);
    put_disk(virt_disk.gd);
err_gdisk:
    blk_cleanup_queue(virt_disk.queue);
err_queue:
    kfree(virt_disk.disk_buf);
    return ret;
}

// 步骤4：模块退出（释放核心流程）
static void __exit virt_disk_exit(void) {
    // 释放顺序：gendisk→主设备号→请求队列→缓冲区（与注册反向）
    del_gendisk(virt_disk.gd);
    put_disk(virt_disk.gd);
    unregister_blkdev(DEV_MAJOR, DEV_NAME);
    blk_cleanup_queue(virt_disk.queue);
    kfree(virt_disk.disk_buf);
    printk(KERN_INFO "virt_disk unregistered\n");
}

module_init(virt_disk_init);
module_exit(virt_disk_exit);
MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("Virtual Block Device Driver (Core Principle Demo)");
```<br>
