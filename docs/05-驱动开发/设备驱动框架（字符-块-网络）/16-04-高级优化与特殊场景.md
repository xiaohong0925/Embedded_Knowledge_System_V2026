### <strong>并发控制：`spinlock`/`mutex`在字符驱动中的应用（以按键中断为例）
嵌入式设备中，字符驱动的“<span class="red">并发</span>访问”是高频问题——如按键中断服务函数（ISR）修改按键状态（共享数据），用户层`read`函数同时读取该状态，多进程并发打开驱动设备等，若未做并发控制，会导致数据脏读、设备状态异常甚至内核崩溃。E级开发需掌握“并发场景识别、锁选型、死锁规避、性能调优”全链路能力。</strong>

1. <span class="red">并发</span>场景与锁选型（E级核心：基于场景选锁）
| 锁类型 | 适用场景 | 核心特性 | 性能（临界区1μs） | 禁忌场景 |
|--------|----------|----------|--------------------|----------|
| `spinlock`（自旋锁） | 短临界区（<10μs）、中断上下文 | 忙等（不切换进程）、关中断（中断上下文） | 约50ns/次加解锁 | 长临界区（导致CPU空转）、进程上下文睡眠 |
| `mutex`（互斥锁） | 长临界区（>10μs）、进程上下文 | 阻塞睡眠（释放CPU）、支持优先级继承 | 约200ns/次加解锁 | 中断上下文（无法睡眠） |
| `semaphore`（信号量） | 多进程资源共享（如允许N个进程访问） | 计数型互斥、支持睡眠 | 约300ns/次加解锁 | 中断上下文 |

> 关键结论[E]：按键中断场景中，中断服务函数（ISR）修改共享数据（临界区<1μs）用`spinlock`；用户层多进程访问设备（临界区>10μs，含<span class="green">GPIO</span>申请/释放）用`mutex`。<br>

### <strong>内存映射：`mmap`实现用户层直接访问设备寄存器
传统字符驱动通过`read`/`write`系统调用访问硬件寄存器，每次调用需陷入内核（约1μs开销），高频访问场景（如1MHz采样率）下性能瓶颈明显。`<span class="green">mmap</span>`可将设备物理寄存器映射到用户层虚拟地址，用户层直接访问（无系统调用开销），E级需掌握“地址映射原理、cache一致性、工业级坑点”。</strong>

1. 内存映射核心原理（E级，流程图+解析）
Linux内核地址映射机制
============================================================

         物理地址空间            内核虚拟地址空间          用户虚拟地址空间
        ┌─────────────┐         ┌─────────────┐          ┌─────────────┐
        │             │         │             │          │             │
        │  0x3F200000 ├─────────▶ 0xFFFF0000 ├──────────▶ 0x7F000000   │
        │   (物理地址) │         │ (内核虚拟地址)│          │ (用户虚拟地址) │
        │             │         │             │          │             │
        │  GPIO控制器  │         │<span class="green">ioremap</span>映射    │          │mmap映射      │
        │  寄存器      │         │              │          │              │
        └──────┬──────┘         └──────┬──────┘          └──────┬──────┘
               │                       │                        │
               │ 硬件访问               │ 内核空间访问            │ 用户空间访问
               ▼                       ▼                        ▼
        ┌─────────────┐         ┌─────────────┐          ┌─────────────┐
        │ 实际硬件操作  │         │ ioread32/   │          │ 直接内存读写  │
        │             │         │ iowrite32   │          │             │
        └─────────────┘         └─────────────┘          └─────────────┘
- 核心概念[E]：
  - `pfn`（Page Frame Number）：物理页号，计算方式`pfn = 物理地址 >> PAGE_SHIFT`（PAGE_SHIFT=12，对应4KB页）；
  - `<span class="green">ioremap</span>`：将物理地址映射到内核虚拟地址（非缓存），避免cache干扰寄存器读写；
  - `remap_pfn_range`：将内核虚拟地址对应的物理页映射到用户层虚拟地址空间。<br>

### <strong>扩展阅读：中断顶/底半部分离（高速传感器数据采集场景）</strong>

高速传感器（如1MHz采样率的ADC）若在中断服务函数中直接处理数据（如解析、存储、通知用户层），
会导致中断延迟超过传感器采样周期，引发数据丢失。中断顶/底半部分离是内核优化中断延迟的核心机制：
- 顶半部（Top Half）：ISR，快速处理（<1μs），仅清中断、拷贝数据到环形缓冲区，禁止睡眠；
- 底半部（Bottom Half）：tasklet/workqueue，延迟处理（可睡眠），解析数据、通知用户层。

1. 顶/底半部分离核心原理（E级）

- 选型建议[E]：
  - `tasklet`：适用于短延迟、小数据量（与ISR同CPU）；
  - `workqueue`：适用于长延迟、大数据量（可跨CPU、可睡眠）。<br>
