# I2C怎么调试——故障排查与逻辑分析仪

---

## 三大典型故障的波形特征

### <span class="orange"><strong>1. ACK 缺失：从机无响应</strong></span>

<span class="red">ACK 缺失</span>是 I2C 调试中最常见的故障。

<br>

主机发送从机地址后，SDA 线在第 9 个时钟周期仍为高电平，即 NACK。

<br>

**ACK 缺失的根因排查表：**

| 排查层级 | 检查项 | 验证方法 |
|---------|--------|---------|
| 硬件连接 | SDA/SCL 是否反接 | 万用表通断测试 |
| 硬件连接 | 上拉电阻是否缺失 | 示波器测量空闲电平 |
| 地址配置 | 从机地址是否匹配 A2/A1/A0 | 查阅数据手册核对 |
| 电气时序 | 信号边沿是否过缓 | 示波器测量上升沿 |
| 电源状态 | 从机供电是否正常 | 万用表测量 VCC |
| 总线占用 | 是否有其他主机锁定总线 | 逻辑分析仪全程抓取 |

<br>

**上拉电阻缺失的典型波形：**

无外部上拉时，SDA/SCL 只能被拉低，无法主动拉高。波形表现为信号"只有低电平没有高电平"，或高电平仅由弱漏电流维持。

<br>

**地址配置错误的排查：**

<span class="green">LM75</span>基地址为 0x48，若 A0 接 VCC 则地址变为 0x49。主机发送 0x48 时，A0=VCC 的器件不会 ACK。用 `i2cdetect -y N` 扫描可发现实际响应地址。

<br>

### <span class="orange"><strong>2. 时钟拉伸：从机主动降速</strong></span>

<span class="red">时钟拉伸（Clock Stretching）</span>是 I2C 从机通过拉低 SCL 来主动延长总线周期的机制。

<br>

**合法场景：**

- EEPROM 接收完一页数据后进入内部写入周期，拉低 SCL 拒绝新事务
- 从机 MCU 内部处理跟不上总线速率，临时降速

<br>

**异常场景：**

- 从机固件死锁，中断服务程序未释放 SCL
- 主机侧 I2C 控制器超时（通常 25ms~35ms）后报错

<br>

**Linux 内核的时钟拉伸处理：**

```dts
// 设备树：配置 I2C 超时容忍
&i2c1 {
    clock-frequency = <100000>;
    timeout-ms = <100>;     /* 最大容忍 100ms 时钟拉伸 */
};
```

<br>

### <span class="orange"><strong>3. 总线死锁：SDA 卡死</strong></span>

<span class="red">总线死锁（Bus Stuck）</span>指 SDA 线被某设备永久拉低，导致所有 I2C 事务无法发起 START 条件。

<br>

**死锁成因分析表：**

| 成因 | 触发条件 | 恢复方法 |
|------|---------|---------|
| 从机复位中断 | 从机正在发送数据时主机复位 | 主机发送 9 个 SCL 时钟脉冲 |
| 从机异常拉低 | 从机 GPIO 配置错误或软件 bug | 物理复位从机电源 |
| 主从竞争 | 多主机场景下仲裁失败 | 总线复位，重新初始化 |
| 热插拔 | 带电插拔导致信号冲突 | 避免热插拔，或加隔离器 |

<br>

**软件恢复：9 时钟脉冲法：**

```c
// 文件：i2c_bus_recovery.c
#include <linux/gpio.h>

#define SCL_GPIO    23
#define SDA_GPIO    24

void i2c_bus_recovery(void)
{
    int i;
    
    gpio_direction_output(SCL_GPIO, 1);
    gpio_direction_input(SDA_GPIO);   /* 观察 SDA 状态 */
    
    if (gpio_get_value(SDA_GPIO) == 0) {
        /* SDA 被卡住，发送 9 个 SCL 脉冲 */
        for (i = 0; i < 9; i++) {
            gpio_set_value(SCL_GPIO, 0);
            udelay(5);
            gpio_set_value(SCL_GPIO, 1);
            udelay(5);
            if (gpio_get_value(SDA_GPIO) == 1)
                break;   /* 从机已释放 SDA */
        }
        /* 发送 STOP 条件 */
        gpio_set_value(SCL_GPIO, 0);
        udelay(5);
        gpio_direction_output(SDA_GPIO, 0);
        udelay(5);
        gpio_set_value(SCL_GPIO, 1);
        udelay(5);
        gpio_set_value(SDA_GPIO, 1);   /* STOP：SCL 高时 SDA 上升沿 */
    }
}
```

<br>

**硬件防护：**

<span class="green">PCA9517</span>是带死锁恢复功能的 I2C 缓冲器。当检测到 SDA 被卡低超过 75ms 时，自动断开连接并发送恢复脉冲。

<br>

## 逻辑分析仪抓取与波形解读

### <span class="orange"><strong>1. Saleae Logic 配置</strong></span>

<span class="red">Saleae Logic</span>是最常用的 PC 端逻辑分析仪，支持实时 I2C 协议解码。

<br>

**硬件连接：**

| 逻辑分析仪通道 | 目标信号 | 备注 |
|--------------|---------|------|
| CH0 | SCL | 必须接入，作为时钟基准 |
| CH1 | SDA | 必须接入，作为数据通道 |
| GND | 目标板 GND | 共地，否则信号浮空 |

<br>

**采样率设置：**

采样率应 ≥ 4 倍 SCL 频率。标准模式 100kHz 需 ≥ 400kS/s，推荐 1MS/s 以上。快速模式 400kHz 需 ≥ 2MS/s，推荐 4MS/s 以上。

<br>

**触发条件：**

使用 "SDA 下降沿时 SCL 为高" 作为 START 条件触发。捕获模式选 "Normal"，预触发缓冲设为 25%。

<br>

**I2C 分析器添加：**

在 Saleae 软件中点击 "Add" → "I2C"，指定 SCL 和 SDA 对应通道，设置位序为 MSB First。

<br>

### <span class="orange"><strong>2. 正常波形 vs 故障波形对比</strong></span>

**Saleae Logic 解码输出示例（正常传输）：**

```text
Time [s],  Packet ID,  Type,   Data
0.001200,  0,          Start
0.001225,  0,          Address, 0x48 (Write)
0.001350,  0,          Data,    0x00
0.001475,  0,          Start
0.001500,  0,          Address, 0x48 (Read)
0.001625,  0,          Data,    0x19
0.001750,  0,          Data,    0x10
0.001875,  0,          NACK
0.001900,  0,          Stop
```

<br>

**ACK 缺失波形特征：**

第 9 个 SCL 周期 SDA 保持高电平（NACK）。Saleae 解码器会标注 "NACK"，并在后续无 STOP 情况下可能出现重复 START 重试。

<br>

**时钟拉伸波形特征：**

SCL 低电平持续时间远超正常 bit 周期。Saleae 时标显示 SCL 低电平宽度 > 100μs（标准模式应约 5μs）。

<br>

**总线死锁波形特征：**

SDA 线在整个观测窗口恒为低电平。发送 START 条件失败：SCL 高时 SDA 无法产生下降沿。

<br>

## Linux 内核 I2C 调试

### <span class="orange"><strong>1. 内核日志级别与错误码</strong></span>

<span class="red">Linux I2C 核心层</span>通过内核日志输出调试信息。

<br>

**常见 I2C 内核错误码表：**

| 错误码 | dmesg 关键字 | 含义 | 常见根因 |
|--------|-------------|------|---------|
| -ENXIO | "NACK" | 从机无 ACK | 地址错误/设备未上电 |
| -ETIMEDOUT | "timeout" | 总线超时 | 时钟拉伸过长/死锁 |
| -EBUSY | "bus busy" | 总线被占 | 多主机竞争或死锁 |
| -EIO | "I/O error" | 通用 IO 错误 | 信号完整性差 |
| -EAGAIN | "try again" | 暂时性失败 | 从机忙，需重试 |

<br>

**启用 I2C 调试日志：**

```bash
# 开启 i2c-core 的调试日志
$ echo 'file drivers/i2c/i2c-core-base.c +p' > /sys/kernel/debug/dynamic_debug/control

# 开启具体适配器驱动日志
$ echo 'file drivers/i2c/busses/i2c-bcm2835.c +p' > /sys/kernel/debug/dynamic_debug/control
```

<br>

**dmesg 典型输出解读：**

```bash
$ dmesg | grep i2c
[   12.345] i2c-bcm2835 3f804000.i2c: i2c transfer failed: -121
[   12.346] lm75 1-0048: Failed to read register: -121
[   12.347] i2c-bcm2835 3f804000.i2c: SDA may be stuck, attempting recovery
```

<br>

<span class="blue">解读：-121 对应 `-ENXIO`（即 NACK），`lm75` 驱动读取寄存器失败，`i2c-bcm2835` 适配器尝试总线恢复（9 时钟脉冲法）。</span>

<br>

### <span class="orange"><strong>2. strace 跟踪系统调用</strong></span>

```bash
$ strace -e ioctl i2cget -y 1 0x48 0x00 w
ioctl(3, I2C_SLAVE, 0x48)       = 0
ioctl(3, I2C_SMBUS, 0xbe91e3a4) = 0
```

<br>

<span class="blue">`I2C_SMBUS` ioctl 的第三个参数是 `struct i2c_smbus_ioctl_data` 指针，包含读写方向、寄存器地址、数据长度等信息。</span>

<br>

### <span class="orange"><strong>3. ftrace 抓取函数调用</strong></span>

```bash
# 启用 I2C 函数追踪
$ echo i2c_transfer > /sys/kernel/debug/tracing/set_ftrace_filter
$ echo function > /sys/kernel/debug/tracing/current_tracer
$ echo 1 > /sys/kernel/debug/tracing/tracing_on

# 执行 I2C 操作后查看追踪结果
$ cat /sys/kernel/debug/tracing/trace
# tracer: function
#  i2c-bcm2835-3f8   [000] ...1   12345.670000: i2c_transfer <- __i2c_transfer
#  i2c-bcm2835-3f8   [000] ...1   12345.670010: i2c_transfer: i2c-1: faddr=0x48, nr=2
```

## 本章小结

<br>

| 概念 | 一句话总结 |
|------|-----------|
| ACK 缺失 | 从机地址错误/未上电/上拉缺失，波形第 9 位高电平 |
| 时钟拉伸 | 从机拉低 SCL 降速，正常行为但过长则异常 |
| 总线死锁 | SDA 被永久拉低，需 9 个 SCL 脉冲或硬件复位 |
| 9 时钟脉冲 | I2C 标准恢复法，让从机完成未完成的发送 |
| PCA9517 | 带死锁恢复的 I2C 缓冲器，75ms 自动断开 |
| Saleae Logic | PC 逻辑分析仪，配置 I2C 解码器后自动解析 |
| dmesg -121 | `ENXIO` 错误码，通常表示 NACK/从机无响应 |
| dynamic_debug | 内核动态调试开关，无需重编译即可开启日志 |
| strace ioctl | 跟踪 i2cget 的系统调用参数 |
| ftrace | 内核函数级追踪，定位 I2C 传输失败点 |

<br>

## 练习

1. 使用 Saleae Logic 抓取一次失败的 I2C 读取，描述波形上 ACK 缺失位置的时标和电平状态。

2. 编写一段 Shell 命令，通过 `dynamic_debug` 开启 `i2c-bcm2835` 驱动的全部调试日志，并解释输出中 "SDA may be stuck" 的含义。

3. 某设备频繁出现时序错误，逻辑分析仪显示 SCL 低电平持续时间约 15ms。这是正常时钟拉伸还是异常死锁？请说明判断依据和恢复策略。

4. 对比示波器、专用协议分析仪、PC USB 逻辑分析仪三类工具在 I2C 调试中的适用场景和效率差异。
