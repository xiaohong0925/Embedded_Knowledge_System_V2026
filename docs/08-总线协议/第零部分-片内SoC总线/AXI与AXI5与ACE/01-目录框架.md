## 01-AXI协议基础与架构.md **[B→I]**

> <span class="badge-b">B</span> → <span class="badge-i">I</span>

### <strong>AXI 核心定义与价值</strong>

<span class="badge-b">B</span>

- 什么是 AXI（Advanced eXtensible Interface）
  - ARM AMBA3/4/5 协议家族中的高性能总线标准
  - 面向高带宽、低延迟的 IP 核互联：CPU ↔ DDR、GPU、DMA、视频编解码器
  - AXI4 是当前主流，AXI5 面向汽车/功能安全扩展
- 为什么嵌入式工程师必须懂 AXI
  - 所有 ARM Cortex-A 系列默认使用 AXI 接口（Cortex-A53/A72/A76）
  - DDR 控制器、PCIe 桥、Display 控制器都以 AXI 为对外接口
  - <span class="blue">不懂 AXI，连 DDR 控制器的数据手册都看不懂。</span>
- 典型芯片中的 AXI 身影
  - i.MX8：Cortex-A53 集群 ↔ CCI-550 ↔ Mali-G52 ↔ DDR4 控制器，全部 AXI4
  - RK3588：四核 A76 通过 AXI 连接 NPU、VPU、ISP 到 DDR

### <strong>AXI 与 AHB/APB 的本质区别</strong>

<span class="badge-b">B</span>

| 特性 | AXI4 | AHB | APB |
|------|------|-----|-----|
| 通道数 | 5 个独立通道 | 1 个共享通道 | 1 个简单通道 |
| 读写并发 | 读写可同时发 | 半双工，分时 | 单主从，分时 |
| 突发传输 | 支持（1-256 beat） | 支持（4/8/16 beat） | 不支持 |
| 乱序完成 | 支持（ID 路由） | 不支持 | 不支持 |
| 时钟域 | 独立时钟 | 共享时钟 | 共享时钟 |

- 一句话总结：AXI 用"通道分离 + 乱序完成"换带宽，用"握手简单"换易实现

### <strong>AXI 拓扑结构初览</strong>

<span class="badge-i">I</span>

- 点对点直连：Master（CPU）↔ Slave（DDR 控制器）
- 总线矩阵（NIC/NOC）：多主多从的交叉开关，ARM CoreLink NIC-400/CMN-600
- 层级互联：AXI 主通路 → AXI2AHB 桥 → AHB 中等外设 → AHB2APB 桥 → APB 低速寄存器
- 最低门槛认知：打开一份 SoC 框图，标出所有 AXI 接口，理解数据流向

## 02-AXI五通道与握手机制.md **[I→E]**

> <span class="badge-i">I</span> → <span class="badge-e">E</span>

### <strong>五通道架构详解</strong>

<span class="badge-i">I</span>

- 读地址通道（AR）：Master 发读请求（地址、突发长度、突发大小）
- 读数据通道（R）：Slave 返回读数据 + 响应
- 写地址通道（AW）：Master 发写请求（地址、突发参数）
- 写数据通道（W）：Master 发送写数据
- 写响应通道（B）：Slave 返回写完成确认
- 为什么分 5 个通道：读写地址分离 → 读写可并发；数据与响应分离 → 数据流单向、响应反向

### <strong>握手协议：VALID / READY</strong>

<span class="badge-i">I</span>

- 所有 5 个通道使用相同的握手：VALID（发送方有数据）+ READY（接收方准备好）
- 传输发生条件：VALID && READY，单周期完成
- 时序约束：VALID 一旦拉高必须保持到 READY 到来，不能中途取消
- 三种握手场景：
  - VALID 先等 READY（发送方慢）
  - READY 先等 VALID（接收方慢）
  - 同时有效（最理想，单周期握手）

### <strong>通道间依赖关系与死锁规避</strong>

<span class="badge-e">E</span>

- AXI 规范规定的依赖规则：
  - AW 必须在 W 之前或同时发出（不能先给数据再告诉地址）
  - W 可以在 AW 之前完成握手，但不能在 AW 之前开始传输
  - AR 与 AW 之间无依赖，读写完全独立
  - R 必须在 AR 之后，B 必须在 AW/W 之后
- 死锁场景：Master 等 B 响应才发下一个 AW，Slave 等收齐 W last beat 才发 B，双方互锁
- 解决：Master 不要等 B 再发下一个 AW，Slave 收到 WLAST 立即发 B

## 03-AXI突发传输与乱序完成.md **[I→E]**

> <span class="badge-i">I</span> → <span class="badge-e">E</span>

### <strong>突发传输（Burst）机制</strong>

<span class="badge-i">I</span>

- 突发类型：FIXED（固定地址）、INCR（递增地址）、WRAP（回卷地址）
- 突发参数：
  - AWLEN/ARLEN：突发长度（0-255，对应 1-256 beat）
  - AWSIZE/ARSIZE：每 beat 字节数（1/2/4/8/16/32/64/128）
  - AWBURST/ARBURST：突发类型
- 带宽计算：突发总字节 = (LEN+1) × 2^SIZE
- 典型配置：读取 1KB cache line = LEN=15, SIZE=6（64B beat）, 16 beat

### <strong>窄传输与非对齐传输</strong>

<span class="badge-i">I</span>

- 窄传输（Narrow Transfer）：Master 数据宽度 > 突发 SIZE，如 64bit 总线写 32bit 数据
  - 通过 WSTRB（写选通）标记有效字节位置
- 非对齐传输（Unaligned Transfer）：起始地址不是 beat 大小的整数倍
  - AXI4 支持：首 beat 地址任意，后续 beat 自动对齐递增
  - 硬件实现：Slave 侧地址调整 + 数据重排

### <strong>乱序完成（Out-of-Order）与 ID 路由</strong>

<span class="badge-e">E</span>

- AWID/ARID：Master 给每个事务打标签，同一 ID 的事务必须按序完成，不同 ID 可乱序
- 为什么需要乱序：DDR 控制器对不同 bank 的访问可以并行，先发的 A 事务可能在后发的 B 之后完成
- ID 位宽：通常 4-8 bit，支持 16-256 个独立事务流
- 总线矩阵中的 ID 扩展：NIC-400 自动扩展 ID 以区分不同 Master，返回时压缩还原

## 04-AXI性能优化与QoS.md **[E]**

> <span class="badge-e">E</span>

### <strong>AXI 性能指标与瓶颈分析</strong>

<span class="badge-e">E</span>

- 有效带宽 = 理论带宽 × 利用率（握手效率）× 协议开销比例
- 瓶颈常见位置：
  - DDR 控制器：bank conflict、refresh 周期、row miss 导致延迟突增
  - 总线矩阵：多主争用同一 Slave，仲裁策略影响延迟
  - Master 侧：突发长度太短，地址/数据通道握手间隔大
- 优化方向：增大突发长度、合并短事务、利用乱序隐藏延迟

### <strong>QoS（服务质量）机制</strong>

<span class="badge-e">E</span>

- AWQOS/ARQOS：4bit 优先级标记（0-15），数值越高优先级越高
- 典型分配：CPU 读 = 15，GPU = 12，DMA = 8，调试 = 0
- 总线矩阵仲裁：固定优先级 vs 轮询 vs 加权公平队列（WFQ）
- 注意：QoS 只在总线矩阵有效，点对点 AXI 无仲裁，QoS 信号被忽略

### <strong>AXI4 新增特性：Region、Cache、User 信号</strong>

<span class="badge-e">E</span>

- AWREGION/ARREGION：16 个区域划分，支持 NUMA 风格的内存分区
- AWCACHE/ARCACHE：4bit 缓存策略提示（bufferable/cacheable/allocate）
- AWUSER/ARUSER/WUSER/RUSER/BUSER：自定义侧带信号，厂商扩展

## 05-AXI嵌入式实战.md **[E]**

> <span class="badge-e">E</span>

### <strong>实战 1：DDR 控制器 AXI 接口配置与带宽测试</strong>

<span class="badge-e">E</span>

- 场景：验证 SoC DDR4 接口是否达到标称带宽
- 配置：Cortex-A53 通过 AXI4 连接 DDR 控制器，突发长度 16，SIZE=6（64B）
- 测试方法：
  - 用 DMA 发起连续突发读/写，统计总周期数
  - 计算：总字节 / (周期 × 时钟周期) = 实测带宽
  - 对比：标称 25.6GB/s（3200MT/s × 64bit），实测通常 60-70%
- 瓶颈定位：打开 DDR 控制器性能计数器，看 bank hit rate、refresh 占比

### <strong>实战 2：多主争用总线矩阵的延迟分析</strong>

<span class="badge-e">E</span>

- 场景：CPU 和 GPU 同时访问 DDR，GPU 延迟导致掉帧
- 工具：ARM CoreSight PMU 或自定义 AXI 监控逻辑
- 测量：记录每个 Master 的 AR/AW 发出到 R/B 返回的周期数
- 优化：调整 QoS 优先级、增加突发长度、分离读/写到不同 DDR rank

### <strong>实战 3：AXI 从机 RTL 设计与仿真验证 **[M]**</strong>

<span class="badge-m">M</span>

- 场景：自研加速器需要 AXI 接口接入 SoC
- RTL 设计要点：
  - 状态机：IDLE → 地址握手 → 数据握手 → 响应 → IDLE
  - FIFO 缓冲：接收地址先入 FIFO，数据通道独立处理
  - 支持突发：实现地址递增逻辑，处理 WLAST/RLAST
- 仿真验证：用 SystemVerilog AXI VIP（Verification IP）做主设备，验证从机时序
- 综合约束：AXI 接口必须寄存器化输出，避免组合逻辑延迟违反时序

## 06-AXI5与ACE缓存一致性.md **[E→M]**

> <span class="badge-e">E</span> → <span class="badge-m">M</span>

### <strong>AXI5 新增特性</strong>

<span class="badge-e">E</span>

- Trace 信号：ATB（AMBA Trace Bus）接口集成，支持 CoreSight 调试追踪
- PMU 信号：性能监控单元扩展，支持事务级 profiling
- 安全扩展：TrustZone 相关信号细化，支持动态安全状态切换
- 汽车功能安全：部分信号冗余、ECC 保护建议

### <strong>ACE：缓存一致性扩展</strong>

<span class="badge-m">M</span>

- 问题：双核 Cortex-A53，各带 L1 cache，同时读同一内存地址，如何保证数据一致？
- ACE 在 AXI4 基础上增加：
  - 缓存状态传递：Shareable/Inner/Outer 域划分
  - 窥探（Snoop）事务：一个核修改 cache line，通知其他核失效
  - 脏写回：cache eviction 时把修改过的数据写回内存
- ACE 事务类型：ReadNoSnoop、ReadOnce、ReadShared、ReadUnique、MakeInvalid
- 典型实现：CCI-550（Cache Coherent Interconnect）连接 4 个 ACE Master 到 DDR

### <strong>CHI 与 ACE 的未来替代 **[M]**</strong>

<span class="badge-m">M</span>

- CHI（Coherent Hub Interface）：ARM 新一致性协议，面向多芯/服务器
- 与 ACE 差异：CHI 是基于消息的包协议（类似 PCIe），ACE 是基于信号的 AXI 扩展
- 嵌入式选择：双核/四核用 ACE + CCI，八核以上考虑 CHI + CMN
