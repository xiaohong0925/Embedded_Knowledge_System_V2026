# 配置空间

<span class="badge-i">[Intermediate]</span>

<span class="red">配置空间（Configuration Space）</span> 是PCIe设备的一块标准化寄存器区域，系统通过它完成设备发现、能力枚举和资源分配。

---

## <strong>基础认知</strong>

### <strong>为什么需要配置空间</strong>

<span class="blue">配置空间是PCIe即插即用能力的基石。</span> 系统加电后，固件/操作系统通过配置空间读取设备身份、枚举能力、分配资源，无需人工干预。

<span class="green">配置空间大小</span>：传统256B + 扩展空间（通过ECAM映射可达4KB）。

---

## <strong>原理解析</strong>

### <strong>配置空间头部结构</strong>

```c
0x00: Vendor ID (16) | Device ID (16)
0x04: Command (16)   | Status (16)
0x08: Revision ID (8)| Class Code (24)
0x0C: Cache Line Sz  | Latency Timer | Header Type | BIST
0x10: BAR0 (32/64)
0x14: BAR1 (32)
0x18: BAR2 (32/64)
0x1C: BAR3 (32)
0x20: BAR4 (32/64)
0x24: BAR5 (32)
0x28: CardBus CIS    | Subsystem Vendor | Subsystem ID
0x2C: ...
0x30: ROM BAR
0x34: Capabilities Pointer
0x3C: Interrupt Line | Interrupt Pin | Min_Gnt | Max_Lat
```

<span class="red">Type 0 Header（Endpoint）与 Type 1 Header（Bridge/Switch）</span> 从0x10开始分化：

| 偏移 | Type 0 | Type 1 |
|------|--------|--------|
| 0x10 | BAR0 | 次级总线号 / 从属总线号 |
| 0x18 | BAR2 | 内存基址 / 内存上限 |
| 0x20 | BAR4 | IO 基址 / IO 上限 |

### <strong>Capability 链表结构</strong>

<span class="green">PCIe 采用链表机制扩展能力</span>， Capabilities Pointer（0x34）指向第一个 Capability 结构：

```
Capability Structure:
  +--------+----------+----------------+
  | Cap ID | Next Ptr | Cap-Specific   |
  |  8bit  |  8bit    |   varies       |
  +--------+----------+----------------+
```

<span class="blue">常见 Capability ID</span>：

| ID | 名称 | 说明 |
|----|------|------|
| 0x01 | Power Management | 电源管理 |
| 0x05 | MSI | 消息中断 |
| 0x11 | MSI-X | 扩展消息中断 |
| 0x10 | PCIe Capability | PCIe 专属能力 |
| 0x18 | SR-IOV | 虚拟化 |

### <strong>ECAM 映射公式</strong>

<span class="red">ECAM（Enhanced Configuration Access Mechanism）</span> 将配置空间映射到内存地址空间：

```
ECAM Address = Base + (Bus << 20) | (Device << 15) | (Function << 12) | Offset
```

即：<span class="blue">每个 Bus 占 1MB，每个 Device 占 32KB，每个 Function 占 4KB</span>。

### <strong>枚举时的 Bus 号回填</strong>

```
Root Complex (Bus 0)
  └── Bridge (Bus 0, Dev 1)
        └── 写入 Secondary Bus = 1
        └── 写入 Subordinate Bus = N
        └── 递归枚举 Bus 1 上的设备
```

<span class="blue">Subordinate Bus Number 必须覆盖桥下所有子总线</span>，否则位于深层 Bus 的设备无法被访问。

## <strong>技术教学</strong>

### <strong>lspci 读取配置空间</strong>

```bash
# 以十六进制 dump 配置空间前 256B
lspci -x -s 01:00.0
# 00: 86 80 01 11 06 04 10 00 ...

# 读取扩展配置空间（前4KB）
lspci -xxx -s 01:00.0
```

### <strong>读取特定 Capability</strong>

```bash
# 查看 PCIe Capability
lspci -vv -s 01:00.0 | grep -A20 "PCIExpress"
# LnkCap: Port #0, Speed 8GT/s, Width x4
# LnkCtl: ASPM L1 Enabled; RCB 64 bytes
```

### <strong>Linux 内核配置空间访问</strong>

```c
// 读取配置字
pci_read_config_word(dev, PCI_VENDOR_ID, &vendor);
pci_read_config_word(dev, PCI_DEVICE_ID, &device);

// 读取 Capability
int pos = pci_find_capability(dev, PCI_CAP_ID_EXP);
u16 link_cap;
pci_read_config_word(dev, pos + PCI_EXP_LNKCAP, &link_cap);
```

## <strong>软硬件实战</strong>

### <strong>场景一：识别设备类型</strong>

```bash
# 通过 Class Code 识别设备类型
lspci -nn | grep -i ethernet
# 01:00.0 Ethernet controller [0200]: Intel I210 [8086:1533]
# Class Code 0x020000 = Network Controller
```

<span class="blue">Class Code 高8位定义大类</span>：0x02=Network，0x03=Display，0x0C=Serial。

### <strong>场景二：枚举失败排查</strong>

```bash
# 检查 dmesg 中的 PCIe 枚举日志
dmesg | grep -i "pci\|ecam"
# [    0.123] pci 0000:00:00.0: [1234:5678] type 01 class 0x060400
# [    0.124] pci 0000:00:01.0: bridge configuration...
```

<span class="red">如果枚举卡住，检查 Secondary/Subordinate Bus Number 是否正确回填</span>。

## <strong>历史演进</strong>

- <span class="green">1992 年 PCI 1.0</span> — 定义 256B 配置空间、Type0/Type1 Header、IO/Memory 空间<br>
- <span class="green">2003 年 PCIe 1.0</span> — 引入 ECAM 映射，扩展配置空间至 4KB<br>
- <span class="green">2010 年 PCIe 3.0</span> — ARI（Alternative Routing-ID）扩展，支持 256 Function<br>
- <span class="green">2022 年 PCIe 6.0</span> — 配置空间机制不变，但 FLIT 模式改变访问时序

<span class="blue">配置空间是 PCIe 保持向后兼容的核心</span>，从 PCI 到 PCIe 6.0，枚举算法几乎不变。

## 小结与练习

| 要点 | 说明 |
|------|------|
| 核心概念 | 配置空间是 PCIe 设备的标准化寄存器区，用于枚举和资源分配 |
| 关键技能 | 掌握 Type0/Type1 Header 区别、Capability 链表遍历、ECAM 公式 |
| 常见误区 | 混淆 Secondary 与 Subordinate Bus Number；忽略 ARI 扩展 |

**练习**

1. 画出 Type 0 和 Type 1 Header 在 0x00-0x3F 的结构差异表。
2. 某 ECAM Base = 0xE0000000，求 Bus 2、Device 3、Function 0、Offset 0x10 的物理地址。
3. 分析为什么 Capability 链表比固定偏移布局更利于扩展。
