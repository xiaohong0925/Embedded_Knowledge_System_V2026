# 信号与多线程交互

> 📊 **本节难度等级：** <span class="badge-e">**E级**</span>

---

### <strong>信号（Signal）是Linux中进程间或进程与内核间的异步通信机制，用于处理“异常事件”（如段错误）或“主动通知”（如进程终止指令）。在嵌入式多线程场景中，信号的交互逻辑远比单线程复杂——若处理不当会导致线程意外终止、数据竞争、实时性破坏等致命问题（如工业控制线程被信号打断导致控制延迟）。本节聚焦嵌入式多线程的信号核心痛点：从“信号在多线程中的传递规则”切入，剖析嵌入式场景的典型陷阱，最终落地“安全处理+实时适配”的实战方案，配套调试工具与避坑指南。</strong>


### <strong>一、核心原理：多线程中信号的“传递与处理逻辑”</strong>

要解决信号与多线程的交互问题，必须先跳出“单线程信号认知”，明确Linux内核在多线程场景下的信号处理规则——这是嵌入式信号安全开发的底层依据，核心包括“信号传递目标”“线程信号掩码”“信号处理函数的执行上下文”三大关键点。

### 1.1 信号的传递：“进程级”触发与“线程级”处理
Linux信号本质是“进程级事件”，但处理过程由线程承担，内核遵循以下核心规则分配处理线程，直接决定嵌入式多线程的信号行为：
1.  **默认分配逻辑**：当进程收到信号时，内核会从“未屏蔽该信号的线程”中随机选择一个执行信号处理函数——若所有线程都屏蔽该信号，信号会被挂起（直到有线程解除屏蔽）。
    - 嵌入式风险点：若核心业务线程（如传感器采集）未屏蔽SIGINT（终端中断信号），远程调试时误发`kill -2 进程ID`会随机打断该线程，导致采集数据丢失。
2.  **定向传递逻辑**：通过`pthread_kill`函数可向指定线程发送信号（区别于`kill`命令向进程发信号），内核会强制由目标线程处理（即使该线程暂时屏蔽信号，信号也会挂起等待解除屏蔽）。
    - 嵌入式适用场景：多核网关中，主线程通过`pthread_kill`向绑定核心2的推理线程发送SIGUSR1（自定义信号），触发模型重新加载，避免影响其他线程。
3.  **信号的“挂起与递送”**：每个线程都有独立的“未决信号集”（Pending Signals），记录已收到但未处理的信号；若线程屏蔽某信号，该信号会存入未决集，直到屏蔽解除后才递送（执行处理函数）。

### 1.2 线程信号掩码：多线程信号隔离的“核心工具”
线程信号掩码（Signal Mask，用`sigset_t`结构体表示）是嵌入式多线程信号隔离的关键——通过屏蔽“非本线程需处理的信号”，可避免核心线程被无关信号打断。其核心特性如下：
-  **线程独立性**：每个线程有独立的信号掩码，通过`pthread_sigmask`函数修改（区别于单线程的`sigprocmask`），修改后仅影响当前线程；
-  **继承性**：子线程创建时会继承父线程的信号掩码——嵌入式开发中可在主线程初始化时配置“基础掩码”（如屏蔽SIGINT、SIGTERM），子线程再根据需求微调，减少重复配置；
-  **核心操作**：支持“屏蔽信号”（SIG_BLOCK）、“解除屏蔽”（SIG_UNBLOCK）、“替换掩码”（SIG_SETMASK）三类操作，嵌入式中最常用“屏蔽核心信号+定向递送”组合。

**嵌入式基础掩码配置示例**：主线程初始化时屏蔽所有非必要信号，子线程按需解除特定信号的屏蔽（如日志线程解除SIGUSR2的屏蔽，用于触发日志轮转）：
```c
#include <pthread.h>
#include <signal.h>
#include <stdio.h>

// 主线程初始化信号掩码：屏蔽SIGINT、SIGTERM、SIGUSR1（核心信号）
void init_signal_mask() {
    sigset_t mask;
    sigemptyset(&mask);
    // 加入需要屏蔽的信号（嵌入式核心线程需避免被这些信号打断）
    sigaddset(&mask, SIGINT);   // 终端中断
    sigaddset(&mask, SIGTERM);  // 进程终止
    sigaddset(&mask, SIGUSR1);  // 自定义信号1
    // 修改主线程信号掩码（SIG_BLOCK：在原有基础上增加屏蔽）
    if (pthread_sigmask(SIG_BLOCK, &mask, NULL) != 0) {
        perror("pthread_sigmask failed");
    }
}

// 日志线程：解除SIGUSR2的屏蔽，用于触发日志轮转
void *log_thread(void *arg) {
    sigset_t mask;
    sigemptyset(&mask);
    sigaddset(&mask, SIGUSR2);
    // 解除SIGUSR2的屏蔽（SIG_UNBLOCK：从掩码中移除）
    pthread_sigmask(SIG_UNBLOCK, &mask, NULL);

    // 信号处理函数注册（接收SIGUSR2触发日志轮转）
    struct sigaction sa;
    sa.sa_handler = log_rotate_handler;  // 自定义处理函数
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;
    sigaction(SIGUSR2, &sa, NULL);

    while (1) {
        write_log();  // 持续写日志
        usleep(100000);
    }
    return NULL;
}

int main() {
    init_signal_mask();  // 主线程配置基础掩码
    pthread_t tid;
    pthread_create(&tid, NULL, log_thread, NULL);  // 子线程继承掩码后微调
    pthread_join(tid, NULL);
    return 0;
}
```

### 1.3 信号处理函数的“特殊执行上下文”
信号处理函数的执行环境与普通线程函数完全不同，这是嵌入式多线程信号故障的高频诱因，核心特性如下：
1.  **独立栈空间**：信号处理函数运行在内核分配的独立栈中（而非线程的私有栈），栈大小有限（通常8KB），嵌入式中若处理函数内使用大缓冲区会导致栈溢出；
2.  **中断式执行**：信号处理函数会打断线程当前执行的代码（即使在临界区），处理完成后再回到断点处继续执行——若线程持有互斥锁时被信号打断，处理函数再尝试加锁会导致死锁；
3.  **内核态到用户态的切换**：信号由内核触发，处理函数运行在用户态，执行过程中若发生系统调用失败（如`write`返回EINTR），需手动处理“被信号中断”的场景。<br>

### <strong>二、嵌入式多线程信号的“三大致命陷阱”</strong>

嵌入式场景的资源约束（小内存、实时性要求）会放大信号与多线程的交互风险，以下三大陷阱是现场调试中最常见的故障根源，需重点规避。

### 2.1 陷阱1：信号处理函数调用“非安全函数”导致崩溃
**问题本质**：POSIX标准明确规定“信号处理函数只能调用信号安全函数（Signal-Safe Functions）”，嵌入式开发者常因调用`printf`、`malloc`、`pthread_mutex_lock`等非安全函数导致进程崩溃——这些函数的内部状态可能被信号打断破坏。
- 非安全函数的危害示例：`malloc`在分配内存时会修改全局内存管理链表，若线程执行`malloc`时被信号打断，信号处理函数再调用`malloc`会导致链表损坏，触发段错误。
- 嵌入式常见场景：传感器采集线程被SIGUSR1打断，处理函数中调用`printf("signal received\n")`，导致`printf`的缓冲区状态错乱，后续打印出现乱码或崩溃。

**信号安全函数清单（嵌入式常用）**：  
| 类别         | 安全函数示例                          | 非安全函数（禁止在信号处理中调用）       |
|--------------|---------------------------------------|------------------------------------------|
| IO操作       | `write`、`read`、`_exit`              | `printf`、`fprintf`（标准库缓冲）        |
| 信号操作     | `sigemptyset`、`sigaddset`、`sigaction` | -                                        |
| 内存操作     | -                                    | `malloc`、`free`、`calloc`（全局内存链表） |
| 同步操作     | -                                    | `pthread_mutex_lock`、`pthread_cond_wait` |

### 2.2 陷阱2：信号并发处理导致“数据竞争”
**问题本质**：多个信号同时递送时，内核会串行执行处理函数，但信号处理函数若操作共享资源（如全局变量），且未做同步保护，会导致数据竞争——嵌入式场景中该问题常表现为“配置参数偶尔错乱”“状态机异常跳转”。
- 故障示例：嵌入式网关中，SIGUSR1处理函数修改全局配置`g_config`，SIGUSR2处理函数读取`g_config`，两个信号并发触发时，读取到的`g_config`可能是“半修改”状态（如部分字段已更新，部分未更新）。

### 2.3 陷阱3：实时线程被信号打断导致“控制延迟”
**问题本质**：嵌入式实时线程（如电机控制、CAN总线数据处理）要求响应时间＜10ms，而信号处理函数的执行会强制打断线程，若处理函数执行时间过长（如超过5ms），会直接破坏实时性——这是车载、工业场景的致命缺陷。
- 典型场景：某电机控制线程（SCHED_FIFO策略，优先级80）每10ms执行一次控制逻辑，若SIGTERM信号的处理函数执行了8ms，会导致控制线程的响应延迟从10ms增至18ms，电机出现抖动。<br>

### <strong>三、嵌入式实战方案：信号安全处理的“四层防护”</strong>

针对上述陷阱，结合嵌入式场景的实时性与稳定性要求，落地“信号隔离→安全处理→实时适配→故障监控”的四层防护方案，覆盖从开发到调试的全流程。

### 3.1 第一层：信号隔离——用掩码+定向递送实现“线程信号专属化”
核心目标：让“非核心线程”承担信号处理职责，核心实时线程仅处理自身必要信号（如唤醒信号），避免无关信号打断。实战步骤如下：
1.  **主线程初始化基础掩码**：屏蔽所有非必要信号（如SIGINT、SIGTERM、SIGUSR1），子线程继承该掩码；
2.  **创建“信号处理专属线程”**：该线程解除所有需要处理的信号屏蔽，专门执行信号处理逻辑——核心实时线程保持信号屏蔽状态，不直接处理信号；
3.  **定向递送关键信号**：若需向核心线程发送信号（如唤醒信号），用`pthread_kill`定向发送，避免随机分配。

**嵌入式实战代码（信号专属线程）**：  
适用于工业网关场景，核心采集线程屏蔽所有信号，专属线程处理配置更新与日志轮转信号：
```c
#include <pthread.h>
#include <signal.h>
#include <unistd.h>

sigset_t g_signal_set;  // 需处理的信号集合
pthread_mutex_t g_config_mutex;  // 共享配置的互斥锁
Config g_system_config;  // 全局配置

// 信号处理专属线程（仅该线程解除信号屏蔽）
void *signal_handler_thread(void *arg) {
    int sig;
    // 解除需要处理的信号屏蔽（SIGUSR1：配置更新；SIGUSR2：日志轮转）
    pthread_sigmask(SIG_UNBLOCK, &g_signal_set, NULL);

    while (1) {
        // 等待信号（阻塞，直到有信号递送）
        if (sigwait(&g_signal_set, &sig) != 0) {
            perror("sigwait failed");
            continue;
        }

        // 处理不同信号（用互斥锁保护共享资源）
        switch (sig) {
            case SIGUSR1:
                pthread_mutex_lock(&g_config_mutex);
                update_config(&g_system_config);  // 安全修改配置（非信号处理函数内）
                pthread_mutex_unlock(&g_config_mutex);
                break;
            case SIGUSR2:
                rotate_log();  // 日志轮转（可调用非安全函数，因不在信号处理上下文）
                break;
            default:
                write(STDERR_FILENO, "Unknown signal\n", 14);
        }
    }
    return NULL;
}

// 核心采集线程（屏蔽所有信号，仅专注采集）
void *collect_thread(void *arg) {
    while (1) {
        collect_sensor_data();  // 传感器采集（实时性要求＜5ms）
        usleep(10000);  // 10ms采集一次
    }
    return NULL;
}

int main() {
    // 1. 初始化信号集合（需要处理的信号）
    sigemptyset(&g_signal_set);
    sigaddset(&g_signal_set, SIGUSR1);
    sigaddset(&g_signal_set, SIGUSR2);

    // 2. 主线程屏蔽信号，子线程继承
    pthread_sigmask(SIG_BLOCK, &g_signal_set, NULL);

    // 3. 初始化互斥锁
    pthread_mutex_init(&g_config_mutex, NULL);

    // 4. 创建信号专属线程和核心采集线程
    pthread_t sig_tid, collect_tid;
    pthread_create(&sig_tid, NULL, signal_handler_thread, NULL);
    pthread_create(&collect_tid, NULL, collect_thread, NULL);

    pthread_join(sig_tid, NULL);
    pthread_join(collect_tid, NULL);
    return 0;
}
```

### 3.2 第二层：安全处理——信号处理函数的“极简原则”
若必须在信号处理函数中执行逻辑（如紧急终止），需遵循“极简+安全”原则，核心要求如下：
1.  **仅调用信号安全函数**：如用`write`替代`printf`输出日志，禁止调用`malloc`、`mutex_lock`等；
2.  **不操作复杂共享资源**：若需修改共享变量，仅用原子操作（如`__sync_fetch_and_or`），避免数据竞争；
3.  **快速执行**：处理函数执行时间＜1ms，避免打断实时线程——复杂逻辑（如配置更新）通过“标志位唤醒”让普通线程执行。

**安全信号处理函数示例（嵌入式紧急终止场景）**：  
```c
#include <signal.h>
#include <unistd.h>

volatile atomic_t g_exit_flag = ATOMIC_INIT(0);  // 原子变量：退出标志

// 信号处理函数（极简，仅设置标志位）
void sigterm_handler(int sig) {
    __sync_fetch_and_or(&g_exit_flag, 1);  // 原子操作设置标志位
    write(STDERR_FILENO, "Exit signal received\n", 20);  // 安全IO
}

// 业务线程（循环检查标志位，安全退出）
void *business_thread(void *arg) {
    while (1) {
        // 检查退出标志位（原子读取）
        if (__sync_fetch_and_add(&g_exit_flag, 0) == 1) {
            release_resource();  // 释放资源（可调用非安全函数）
            pthread_exit(NULL);
        }
        do_business();  // 业务逻辑
        usleep(10000);
    }
    return NULL;
}

int main() {
    // 注册SIGTERM信号处理函数
    struct sigaction sa;
    sa.sa_handler = sigterm_handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;  // 禁止信号嵌套（避免并发处理）
    sigaction(SIGTERM, &sa, NULL);

    pthread_t tid;
    pthread_create(&tid, NULL, business_thread, NULL);
    pthread_join(tid, NULL);
    return 0;
}
```

### 3.3 第三层：实时适配——减少信号对实时线程的影响
针对实时线程（如电机控制、精密测量），需通过“信号屏蔽+唤醒优化”保障实时性，核心方案：
1.  **屏蔽非必要信号**：仅保留“唤醒信号”（如SIGUSR1）的解除屏蔽，其他信号全部屏蔽；
2.  **用信号替代轮询**：实时线程通过`sigwaitinfo`等待信号唤醒，而非轮询标志位——减少CPU占用（如休眠时CPU利用率从100%降至0%）；
3.  **限制信号处理时间**：唤醒信号的处理逻辑＜100us，避免影响控制周期。

**实时线程信号唤醒示例（嵌入式电机控制）**：  
```c
#include <pthread.h>
#include <signal.h>
#include <sched.h>

// 实时电机控制线程（SCHED_FIFO策略，优先级80）
void *motor_control_thread(void *arg) {
    sigset_t mask;
    sigemptyset(&mask);
    sigaddset(&mask, SIGUSR1);  // 仅解除唤醒信号的屏蔽
    pthread_sigmask(SIG_UNBLOCK, &mask, NULL);

    // 配置实时调度策略
    struct sched_param param;
    param.sched_priority = 80;
    pthread_setschedparam(pthread_self(), SCHED_FIFO, &param);

    struct siginfo si;
    while (1) {
        // 等待SIGUSR1信号唤醒（阻塞，CPU利用率0%）
        if (sigwaitinfo(&mask, &si) == -1) {
            perror("sigwaitinfo failed");
            continue;
        }

        // 电机控制逻辑（执行时间＜500us）
        adjust_motor_speed();
    }
    return NULL;
}

// 触发线程（定时发送SIGUSR1唤醒控制线程）
void *trigger_thread(void *arg) {
    pthread_t control_tid = *(pthread_t *)arg;
    while (1) {
        pthread_kill(control_tid, SIGUSR1);  // 定向发送唤醒信号
        usleep(10000);  // 10ms唤醒一次（控制周期10ms）
    }
    return NULL;
}

int main() {
    // 主线程屏蔽所有信号，子线程继承后微调
    sigset_t all_mask;
    sigfillset(&all_mask);
    pthread_sigmask(SIG_BLOCK, &all_mask, NULL);

    pthread_t control_tid, trigger_tid;
    pthread_create(&control_tid, NULL, motor_control_thread, NULL);
    pthread_create(&trigger_tid, NULL, trigger_thread, &control_tid);

    pthread_join(control_tid, NULL);
    pthread_join(trigger_tid, NULL);
    return 0;
}
```

### 3.4 第四层：故障监控——信号相关问题的“调试工具链”
嵌入式信号问题具有“偶发、异步”特性，需用专用工具定位，核心工具与操作如下：
1.  **信号发送与查看**：
    - 用`kill`命令向进程/线程发送信号（嵌入式常用）：
      ```bash
      kill -USR1 12345  # 向PID为12345的进程发送SIGUSR1
      kill -2 -12345    # 向进程组12345的所有进程发送SIGINT
      ```
    - 用`ps -L -o pid,tid,pcpu,stat,sig`查看线程的信号状态（`sig`列显示未决信号）。

2.  **信号处理函数调试**：
    - 用`gdb`设置信号断点，跟踪处理函数执行：
      ```gdb
      (gdb) handle SIGUSR1 stop print  # 收到SIGUSR1时暂停并打印
      (gdb) break sigterm_handler      # 在信号处理函数处设断点
      (gdb) run                        # 运行程序触发信号
      ```

3.  **信号导致的崩溃排查**：
    - 查看`dmesg`日志，定位信号导致的崩溃（如SIGSEGV导致的段错误）：
      ```bash
      dmesg | grep -i segfault  # 查找段错误日志，包含信号编号、崩溃地址
      ```
    - 用`addr2line`将崩溃地址转换为代码行（需带调试信息的可执行文件）：
      ```bash
      arm-linux-gnueabihf-addr2line -e motor_control -f 0x0000a123  # 转换地址到函数和行号
      ```<br>

### <strong>四、嵌入式多线程信号避坑清单与总结</strong>

信号与多线程的交互问题，本质是“异步信号”与“同步线程”的逻辑冲突——嵌入式场景的解决方案核心是“**隔离优先，极简处理**”：用专属线程承担信号处理职责，核心实时线程屏蔽无关信号，信号处理函数仅做“标志位设置”等极简操作。

### 嵌入式多线程信号避坑清单（上线前必查）
1.  [ ] 核心实时线程是否仅解除“必要唤醒信号”的屏蔽，其他信号全部屏蔽；

4.  [ ] 用`sigwait`替代“信号处理函数+全局变量”的组合，降低复杂度；
