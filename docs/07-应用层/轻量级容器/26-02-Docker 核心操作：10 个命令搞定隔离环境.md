### <strong>docker run -it --rm debian:bullseye` 交互式运行</strong>

嵌入式场景：在ARM64开发板上快速验证Debian环境是否包含`arm-linux-gnueabihf-gcc`交叉工具链，无需永久安装。

操作步骤与输出：
```bash
# 1. 首次术语解释：镜像（Image）是只读模板，容器（Container）是镜像的运行实例
# 拉取并启动一个临时Debian容器，-it分配交互式终端，--rm退出后自动删除容器
$ docker run -it --rm debian:bullseye
Unable to find image 'debian:bullseye' locally
bullseye: Pulling from library/debian
a0a1e1e11100: Pull complete 
Digest: sha256:1234567890abcdef...
Status: Downloaded newer image for debian:bullseye
root@a1b2c3d4e5f6:/#  # 已进入容器，hostname为容器短ID

# 2. 在容器内执行命令验证环境
root@a1b2c3d4e5f6:/# cat /etc/os-release
PRETTY_NAME="Debian GNU/Linux 11 (bullseye)"
NAME="Debian GNU/Linux"
VERSION_ID="11"

# 3. 检查架构（嵌入式需确认armhf/arm64匹配）
root@a1b2c3d4e5f6:/# uname -m
aarch64  # 与宿主机共享内核，故架构一致

# 4. 退出容器（输入exit或Ctrl+D）
root@a1b2c3d4e5f6:/# exit
exit
$ docker ps -a  # 退出后立即验证
CONTAINER ID   IMAGE     COMMAND   CREATED   STATUS    PORTS     NAMES
# 无记录，--rm已自动清理容器，不占用存储空间
```

嵌入式要点：`--rm`参数在资源受限设备上至关重要。普通容器退出后，其可写层（`/var/lib/docker/overlay2`）默认保留，占用Flash空间。嵌入式系统无冗余存储，必须养成`--rm`习惯，避免手动`docker system prune`的误删风险。<br>

### <strong>`docker ps -a` 查看所有容器（含已退出）</strong>

嵌入式场景：设备重启后发现磁盘空间莫名减少，需排查是否存在未清理的已退出容器占用overlay2层。

操作步骤与输出：
```bash
# 1. 启动两个测试容器（一个持续运行，一个立即退出）
$ docker run -d --name persistent nginx:alpine
a1b2c3d4e5f6...

$ docker run --name exited_test alpine echo "job done"
job done  # 容器执行完命令后立即退出

# 2. 查看运行中容器（默认不显示已退出）
$ docker ps
CONTAINER ID   IMAGE          COMMAND                  CREATED          STATUS          PORTS     NAMES
a1b2c3d4e5f6   nginx:alpine   "/docker-entrypoint.…"   10 seconds ago   Up 9 seconds    80/tcp    persistent

# 3. 查看所有容器（含已退出）
$ docker ps -a
CONTAINER ID   IMAGE          COMMAND                  CREATED              STATUS                      PORTS     NAMES
a1b2c3d4e5f6   nginx:alpine   "/docker-entrypoint.…"   20 seconds ago       Up 19 seconds               80/tcp    persistent
b7c8d9e0f1a2   alpine         "echo job done"          5 seconds ago        Exited (0) 4 seconds ago              exited_test

# 4. 关键信息解读（嵌入式关注点）
# STATUS列：
# - Up：运行中，需检查资源占用（docker stats）
# - Exited (0)：正常退出，code 0表示命令成功
# - Exited (137)：被kill，可能是OOM或docker stop

# 5. 排查磁盘占用（已退出容器仍占空间）
$ du -sh /var/lib/docker/overlay2/b7c8d9e0f1a2...
4.0K  # exited_test容器仅产生4KB元数据（无文件写入）

# 6. 清理策略（对比--rm的重要性）
$ docker rm exited_test  # 手动删除已退出容器
exited_test
$ docker ps -a
CONTAINER ID   IMAGE          COMMAND                  CREATED          STATUS          PORTS     NAMES
a1b2c3d4e5f6   nginx:alpine   "/docker-entrypoint.…"   1 minute ago     Up 1 minute     80/tcp    persistent
```

嵌入式要点：`docker ps -a`输出中的`NAMES`列若显示`random_name`，说明启动时未指定`--name`，Docker自动生成。在自动化脚本中务必命名容器，否则后续`docker exec/rm`需依赖变化的CONTAINER ID，增加维护成本。<br>

### <strong>`docker exec -it <容器名> bash` 进入运行中容器</strong>

嵌入式场景：网关设备上的Nginx容器突然返回502错误，需进入容器检查`/var/log/nginx/error.log`，但不想通过`docker logs`看混杂输出。

操作步骤与输出：
```bash
# 1. 运行后台容器（模拟生产服务）
$ docker run -d --name my_gateway nginx:alpine
c3d4e5f6a1b2...

# 2. 进入容器执行诊断（首次术语：Namespace（命名空间）是内核隔离机制）
$ docker exec -it my_gateway /bin/sh  # alpine默认无bash，用sh
/ #  # 提示符变化，已进入容器mount namespace

# 3. 验证进程隔离（容器内PID 1是nginx，与宿主机无关）
/ # ps aux
PID   USER     TIME  COMMAND
    1 root     0:00 nginx: master process nginx -g daemon off;
   32 nginx    0:00 nginx: worker process
   33 nginx    0:00 nginx: worker process
   34 root     0:00 /bin/sh
   40 root     0:00 ps aux

# 4. 验证文件系统隔离（容器内看不到宿主机进程）
/ # ls /proc/
1/     32/    33/    34/    40/    ...  # 仅容器内进程对应的PID目录
/ # cat /proc/1/cmdline 
nginx: master process nginx -g daemon off;

# 5. 宿主机对比（相同PID存在，但进程不同）
$ ps aux | grep nginx
root      1234  ...  nginx: master process nginx -g daemon off;
# 宿主机PID 1234 对应容器内PID 1，通过Namespace映射隔离

# 6. 查看Nginx错误日志（嵌入式排错核心操作）
/ # tail -f /var/log/nginx/error.log
2024/01/01 12:34:56 [error] 32#32: *1 connect() failed (111: Connection refused) 
while connecting to upstream, client: 192.168.1.100, server: localhost, 
upstream: "127.0.0.1:3000"  # 发现后端服务端口未启动

# 7. 退出exec会话（不影响容器主进程）
/ # exit
$ docker ps
CONTAINER ID   IMAGE          COMMAND                  CREATED          STATUS          PORTS     NAMES
c3d4e5f6a1b2   nginx:alpine   "/docker-entrypoint.…"   5 minutes ago    Up 5 minutes    80/tcp    my_gateway
# 容器仍Up，执行bash的进程已销毁
```

嵌入式要*：`docker exec`创建的进程与`docker run`主进程共享PID Namespace，但拥有独立的文件描述符表。在嵌入式调试中，可`exec`进入后`strace -p <主进程PID>`跟踪系统调用，无需在Dockerfile中预装strace，节省镜像体积。<br>

### <strong>`docker images` 和 `docker rmi` 清理无用镜像</strong>

嵌入式场景下，每次`docker build`都会留下`<none>:<none>`的中间层镜像。我经历过最惨的一次：瑞萨RZ/G2L的4GB eMMC，三个月没清理，最后连`apt-get update`的空间都没了，`dockerd`直接卡死。

先看家底：
```bash
$ docker images --format "table {{.Repository}}\\t{{.Tag}}\\t{{.Size}}\\t{{.ID}}"
REPOSITORY          TAG                 SIZE                IMAGE ID
gcc-arm-none-eabi   10-2020-q4-major    1.84GB              a3c5e8f2d9c1
alpine              latest              5.57MB              0548810523d7
<none>              <none>              2.1GB               7f3e9c1b8a2d  # 这是build失败的垃圾
```

`--format`参数在Buildroot自带的docker里可能不支持，那就用笨办法：
```bash
$ docker images | awk '$1=="<none>" && $2=="<none>" {print $3}' | xargs docker rmi -f
```
但强删`xargs docker rmi -f`是最后的倔强，先走正规流程：

安全删除三步走：
1. 找孤儿容器：`docker ps -a | grep Exited` 把退出的容器全删掉，否则镜像有依赖删不掉
2. 按标签删：`docker rmi gcc-arm-none-eabi:10-2020-q4-major` 这会删标签，如果还有镜像层被其他镜像引用，不会真删
3. 强制清垃圾：`docker image prune -a --filter "until=168h"` 清理7天前所有无用镜像

嵌入式专属坑点：
- ubi0: volatile status write failed**：当eMMC只剩几百MB时，Docker的元数据写操作会触发UBIFS的垃圾回收，导致整个系统IO卡死。解决方法是预留10%空间：`docker image prune -a --filter "until=24h"` 每天自动清理
- 镜像层碎片化：NAND flash上，overlay2的256层镜像会导致写放大3-5倍。我的习惯是用`--squash`合并层（需开启dockerd实验特性），但Yocto里这个选项默认没开，得改`docker.service`：
  ```ini
  # /lib/systemd/system/docker.service
  ExecStart=/usr/bin/dockerd --experimental=true
  ```<br>

### <strong>`-v /host:/container` bind mount 挂载源码/工具链</strong>

这是嵌入式开发最常用也最坑人的操作。假设你在Ubuntu x86上交叉编译ARM程序：
```bash
$ docker run -it --rm \
  -v /opt/gcc-arm-11.2/bin:/opt/toolchain \  # 挂载工具链
  -v /home/lin/work/project:/work \          # 挂载源码
  -w /work \
  -u $(id -u):$(id -g) \                     # 关键：同步UID
  gcc:11-arm \
  /opt/toolchain/arm-linux-gnueabihf-gcc main.c
```

权限崩掉的典型现场：
```bash
$ ls -l build/
-rw-r--r-- 1 root root 104K hello  # 容器里编译的，宿主机上root拥有，普通用户删不掉
$ rm build/hello
rm: cannot remove 'hello': Permission denied
```

嵌入式开发必须加 `-u $(id -u):$(id -g)`，否则容器内root创建的文件在宿主机上也是root，开发时每次都要`sudo chown`。
bind mount vs volume的抉择：
- bind mount（`-v /host/path:/container/path`）：
  - 优点：直接操作宿主机文件，工具链、源码实时同步，CI/CD友好
  - 缺点：容器内删文件会直接删宿主机，eMMC的写放大问题无法隔离
  - 适用场景：开发环境、需要持久化的配置文件
- volume（`-v my_vol:/data`）：
  - 优点：Docker管理，跨容器共享，ubifs上自动处理磨损均衡
  - 缺点：路径是`/var/lib/docker/volumes/...`不好找，Buildroot默认没开volume驱动
  - 嵌入式场景几乎不用，除非你做docker-compose多容器服务

性能陷阱：
在UBIFS上bind mount源码目录，每次`make`会触发大量小文件写操作，导致MTD驱动频繁gc。我的优化是：
```bash
$ docker run ... -v /home/lin/work:/work:ro \  # 源码只读挂载
  -v /tmp/build:/work/build \                  # 输出目录用tmpfs
  gcc:11-arm make -C /work OBJDIR=/work/build
```
把编译产物放tmpfs，最后`docker cp`出来，避免写NAND。<br>

### <strong>`docker cp` 在宿主机与容器间拷贝文件</strong>

`docker cp`比bind mount轻量，适合临时救场。比如现场设备上，容器里的应用崩溃了，但没挂载日志目录：
```bash
$ docker ps -a
CONTAINER ID   IMAGE     COMMAND   CREATED        STATUS                     PORTS   NAMES
a1b2c3d4e5f6   myapp     "./run"   2 hours ago    Exited (137) 5 minutes ago         sad_elion

# 把里面的日志拷出来
$ docker cp a1b2c3d4e5f6:/var/log/app.log /home/lin/debug/app_$(date +%Y%m%d_%H%M%S).log
```

容器停了也能拷，这是`docker cp`的最大优势。对比bind mount，如果容器起不来，你根本没办法挂载。

嵌入式排错典型流程：
1. 设备现场出问题，SSH连进去
2. `docker logs a1b2c3d4e5f6` 看应用日志（如果没输出到stdout）
3. `docker cp`把core dump文件弄出来（前提是容器里开了`ulimit -c unlimited`）
4. 在开发机上`gdb`分析

但别滥用`docker cp`传大文件。我曾经用`docker cp`传过800MB的固件包，结果dockerd把文件缓存在`/var/lib/docker/tmp`忘了删，把根分区撑爆了。所以：
- 小文件（<100MB）用`docker cp`
- 大文件用bind mount，或者干脆`scp`到宿主机<br>

### <strong>`-m 32m` 硬限制内存（触发OOM即kill）</strong>

在x86服务器上，`-m 32m`可能只是个软提醒。但在NXP i.MX6ULL（512MB RAM）上，这就是死刑立即执行。因为嵌入式系统通常没开swap（开了eMMC半年就磨穿），内存顶到上限直接触发内核的OOM killer，容器里所有进程一锅端。

实战场景：远程设备监控脚本
```bash
# 设备上跑个采集温湿度的Python容器，写代码时没注意内存泄漏
$ docker run -d --name monitor -m 64m myapp/python-monitor:latest

# 三天后，设备失联。SSH上去一看
$ dmesg | grep -A 5 "oom-killer"
[  259.347] Memory cgroup out of memory: Kill process 1234 (python3) score 1024 or sacrifice child
[  259.348] Killed process 1234 (python3) total-vm:132432kB, anon-rss:63456kB, file-rss:0kB, shmem-rss:0kB

# 用 docker inspect 确认内存限制已生效
$ docker inspect monitor --format '{{.HostConfig.Memory}}'
67108864  # 64MB = 64*1024*1024
```

这里面的门道：
- `-m 32m` 对应cgroup的`memory.limit_in_bytes = 33554432`，是**硬上限**，不是建议值。容器总内存（含page cache）超过这个数，内核立即挑一个oom_score最高的进程杀掉。
- 嵌入式必须加 `--memory-swap=32m`，否则默认swap是memory的两倍，你的eMMC要被写废了。这俩设成一样等于禁用swap。
- oom-kill-disable别用：`--oom-kill-disable`会让容器进程卡在内存申请上，系统直接僵死。不如让它快速死掉，systemd重启策略兜底。

调优技巧：
```bash
# 给关键容器留条活路，oom_score_adj调低（-1000表示绝不杀）
$ docker run -d -m 128m --oom-score-adj -500 my-critical-app

# 监控内存使用曲线（没装cadvisor也能看）
$ cat /sys/fs/cgroup/memory/docker/<container_long_id>/memory.usage_in_bytes
```<br>

### <strong>`--cpus 0.5` 限制CPU配额（防止抢占实时任务）</strong>

这个参数拯救过我的电机控制项目。CI编译容器在板子上跑`make -j$(nproc)`，直接把PREEMPT_RT核占满，电机PWM信号抖动超过5μs，设备当场狂震。

命令拆解：
```bash
# 限制容器最多用半个CPU核（50%时间片）
$ docker run -d --cpus 0.5 --cpuset-cpus 1,2 myapp/ci-builder

# 查看cgroup配额（核心原理）
$ cat /sys/fs/cgroup/cpu,cpuacct/docker/<container_id>/cpu.cfs_quota_us
50000
$ cat /sys/fs/cgroup/cpu,cpuacct/docker/<container_id>/cpu.cfs_period_us
100000
```
`50000/100000 = 0.5`，这就是`--cpus 0.5`的底层换算。**CFS调度器**（Completely Fair Scheduler）每100ms period里只给容器50ms运行时间。

嵌入式实战组合拳：
```bash
# 隔离CPU 0给实时任务，CPU 1-3给容器且限制总配额1.5核
$ docker run -d \
  --cpuset-cpus="1-3" \      # 能跑在哪几个核上
  --cpus="1.5" \             # 在这几个核里总共占多少配额
  --cpu-shares=512 \         # 多容器争抢时的权重（次要）
  myapp/non-realtime-worker
```
--cpus和--cpuset-cpus的区别：
- `--cpuset-cpus`：物理绑核，容器只能在这些核上跑。实时任务常用`isolcpus=0`内核参数隔离0号核，这里就不能用`--cpuset-cpus 0`。
- `--cpus`：时间片配额，即使绑了3个核，总运行时间也不能超标。适合"非关键但耗CPU"的后台任务。

实时性红线：
如果你的主控任务跑在PREEMPT_RT核上，容器千万别绑那个核。用`isolcpus=0`隔离后，容器再`--cpuset-cpus 1-3`，两者井水不犯河水。<br>

### <strong>`--pids-limit 100` 防止fork炸弹</strong>

这个参数在嵌入式里冷门但救命。某次现场设备日志进程疯狂fork子进程处理日志轮转，PID号飙到32768上限，systemd都fork不出新进程了，SSH直接拒绝连接。重启才恢复，但现场在青藏高原，开车过去要三天。

模拟灾难现场：
```bash
# 容器里跑个恶意脚本，每秒fork一次
$ docker run -d --name bomb myapp/fork-bomb:latest
$ docker exec bomb bash -c 'while true; do sleep 1 & done'

# 30秒后，宿主机fork失败
$ ssh root@device
-bash: fork: retry: Resource temporarily unavailable
```

修复方案：
```bash
# 限制容器最多100个进程（包括线程，在Linux里都是task_struct）
$ docker run -d --name safe-app --pids-limit 100 myapp/logger

# 查看pids cgroup
$ cat /sys/fs/cgroup/pids/docker/<container_id>/pids.max
100
$ cat /sys/fs/cgroup/pids/docker/<container_id>/pids.current
47  # 当前已用47个
```

嵌入式设置经验值：
- 轻量级监控容器：`--pids-limit 50`（主进程+几个线程）
- 编译容器：`--pids-limit 200`（`make -j4`会fork很多子进程）
- 留余量：systemd-journald、sshd、dockerd本身各占1个PID，总数别超过内核`kernel.pid_max`（默认32768）

排查技巧：
```bash
# 容器里看进程数限制
$ docker exec safe-app bash -c 'ulimit -u'
100

# 谁在疯狂fork？进容器看
$ docker exec safe-app ps aux | wc -l
98  # 快满了，赶紧排查
```

```bash
# 嵌入式容器黄金三角（内存+CPU+进程数）
$ docker run -d \

  --cpus 1.0 --cpuset-cpus 1-2 \
  --pids-limit 100 \

### <strong>`docker run -v $(pwd):/work -w /work gcc:11 gcc hello.c`</strong>

先准备一个最简单的C文件：
```bash
$ mkdir ~/docker-test && cd ~/docker-test
$ cat > hello.c <<EOF
#include <stdio.h>
int main() {
    printf("Hello from container PID: %d\n", getpid());
    return 0;
}
EOF
```

执行编译（关键命令）：
```bash
$ docker run -it --rm \
  -v $(pwd):/work \          # 把当前目录挂载到容器里的/work
  -w /work \                 # 容器启动后自动cd到/work
  gcc:11 \
  gcc hello.c -o hello       # 在容器里执行编译

$ ls -lh hello
-rwxr-xr-x 1 root root 16K Jan 10 14:23 hello
$ ./hello
Hello from container PID: 12345
```

发生了什么：
- `-v $(pwd):/work` 把**宿主机**的`~/docker-test`目录绑定到**容器**的`/work`目录。容器里gcc编译生成的`hello`二进制文件，会直接落在宿主机当前目录，不用`docker cp`来回倒腾。
- `-w /work` 让容器启动后自动把当前工作目录切到`/work`，省得在命令里写`cd /work && gcc ...`。
- `gcc:11` 镜像里自带arm交叉编译工具链？**不，`gcc:11`是x86镜像**。交叉编译得用`arm32v7/gcc`或`arm64v8/gcc`镜像，但原理一样。

嵌入式专属场景：
```bash
# 在x86宿主机上编译ARM程序（这才是日常）
$ docker run -it --rm \
  -v $(pwd):/work \
  -w /work \
  arm64v8/gcc:latest \       # 注意镜像架构
  aarch64-linux-gnu-gcc hello.c -o hello-arm

$ file hello-arm
hello-arm: ELF 64-bit LSB executable, ARM aarch64, version 1 (SYSV), dynamically linked...
```<br>

### <strong>观察容器内进程 `ps aux` 与宿主机隔离</strong>

编译完别急着删，我们进去看看容器的"小世界"：
```bash
# 启动一个后台容器，专门用来观察
$ docker run -d --name test-box -v $(pwd):/work -w /work gcc:11 sleep 3600
$ docker exec -it test-box bash

# 在容器里看进程
root@4f5a6b7c8d9e:/work# ps aux
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.0   2388   720 ?        Ss   14:23   0:00 sleep 3600
root         7  0.0  0.0   5896  2856 pts/0    Ss   14:25   0:00 bash
root        15  0.0  0.0   8892  1728 pts/0    R+   14:25   0:00 ps aux

# 在宿主机看同一时刻的进程
$ ps aux | grep sleep
root     12345  0.0  0.0   2388   720 ?        Ss   14:23   0:00 sleep 3600
```

PID隔离的真相：
- 容器里的PID 1是`sleep 3600`，但在宿主机看它是PID 12345（或其他随机数）。这就是PID namespace的魔术：容器里的进程号从1开始重新编排，看不到宿主机其他进程。
- 容器里执行`ps aux`只能看到自己namespace里的3个进程，宿主机上跑了上百个进程它一个都看不见。

验证隔离性：
```bash
# 在容器里尝试杀宿主机的init（PID 1）
root@4f5a6b7c8d9e:/work# kill -9 1
bash: kill: (1) - No such process  # 失败，因为容器里的PID 1是sleep，不是systemd

# 在容器里看系统load average
root@4f5a6b7c8d9e:/work# uptime
14:30:00 up 2 days, 3:10,  load average: 0.00, 0.00, 0.00
```
容器里的`uptime`显示load average为0，因为cgroup隔离了CPU统计信息，它看不到宿主机上其他进程的负载。<br>

### <strong>退出后容器消失（`--rm` 效果验证）</strong>

很多初学者记不住`--rm`，结果`docker ps -a`里堆满了Exited状态的僵尸容器，eMMC被`/var/lib/docker/containers`下的日志文件塞爆。

对比实验：
```bash
# 不带 --rm，容器退出后会残留
$ docker run -v $(pwd):/work -w /work gcc:11 gcc hello.c
$ docker ps -a | grep Exited
9a8b7c6d5e4f  gcc:11  "gcc hello.c"  Exited (0) 5 seconds ago  boring_keldysh

# 带 --rm，容器退出后自动删除
$ docker run --rm -v $(pwd):/work -w /work gcc:11 gcc hello.c
$ docker ps -a | grep boring_keldysh  # 找不到，已经被清理
```
--rm的原理：
docker run时`--rm`相当于在容器主进程退出后，自动执行`docker rm <container_id>`。省空间，但**调试时别用**。因为容器挂了你也`docker exec`不进去看现场。

嵌入式场景建议：
- 开发调试：别用`--rm`，方便`docker exec`进去排错
- CI/CD自动化构建：必须用`--rm`，否则Jenkins跑100次build，eMMC就满了
- 现场运行：配合`--restart unless-stopped`，容器崩溃自动重启，但Exited的不会残留

完整工作流（供复制粘贴）：
```bash
#!/bin/bash
# 编译阶段：用 --rm 自动清理
docker run --rm -v $(pwd):/work -w /work arm64v8/gcc:latest make

# 调试阶段：不用 --rm，留现场
docker run -d --name debug-box -v $(pwd):/work -w /work gcc:11 sleep 3600
docker exec -it debug-box bash  # 进去慢慢调

# 清理阶段：手动删除
docker stop debug-box && docker rm debug-box
```<br>

### <strong>`FROM debian:bullseye-slim` 镜像选型（越小越好）</strong>

`FROM` 不是"拿来用"，而是"基于它造个新的"。你选的基础镜像，直接决定了最终镜像的体积、安全漏洞数量、以及更新频率。

嵌入式场景选型矩阵：
| 基础镜像                | 体积| 包管理器 | 适用场景                     | 避坑指南 |
|------------------------|-----|----------|-----------------------------|----------|
| `debian:bullseye-slim` | 80MB| apt      | 需要编译环境，团队熟悉 Debian | 64位ARM用 `arm64v8/debian` |
| `alpine:latest`        | 5MB | apk      | 纯运行环境，C 程序静态编译    | musl libc 可能与 glibc 不兼容 |
| `busybox:glibc`        | 4MB | -        | 终极精简，只跑单个二进制      | 无包管理器，调试困难 |
| `scratch`              | 0KB | -        | Go/Rust 静态编译产物         |  连 `/bin/sh` 都没有，无法 exec 进入 |

指令示例：
```dockerfile
# 错误示范：没指定架构，Buildroot 构建时拉错镜像
FROM debian:bullseye-slim  # 在ARM板子上会报错：no matching manifest

# 正确示范：明确指定多架构标签
FROM arm64v8/debian:bullseye-slim

# 进阶示范：多阶段构建时，不同阶段用不同基础镜像
FROM gcc:11 as builder       # 编译阶段用重型镜像
FROM arm64v8/alpine as runner # 运行阶段用轻量镜像
```

镜像标签的坑：
- `latest` 标签会随时间变，今天构建的镜像和明天可能不一样。嵌入式建议用 精确版本号，如 `debian:bullseye-20240101-slim`
- `-slim` 去掉了文档、locale、manpage，体积减半。但 `python3-dev` 这类包可能装不上，因为依赖头文件被删了。需要编译时，先用 `bullseye` 装完再 `apt-get purge` 清理。<br>

### <strong>`RUN apt-get update && apt-get install -y ... && rm -rf /var/lib/apt/lists/*` 减少层大小</strong>

Dockerfile 每行指令都会生成一个镜像层（image layer），层是只读的，层层叠加像洋葱。`RUN apt-get update` 单独一行，会被缓存，但下次构建时可能 apt 源已更新，导致 `install` 失败。

缓存机制陷阱：
```dockerfile
# 错误示范：三层，每层都可能被缓存，但可能不一致
RUN apt-get update
RUN apt-get install -y python3
RUN rm -rf /var/lib/apt/lists/*

# 正确示范：一行，原子操作，缓存命中或全部重跑
RUN apt-get update && apt-get install -y python3 && rm -rf /var/lib/apt/lists/*

# 为什么必须删缓存？
# /var/lib/apt/lists/* 缓存了 apt 源索引，体积约 30MB，不删就打包进镜像了
```

嵌入式构建加速技巧：
```dockerfile
# 国内板子构建慢，换源并固化到镜像层
RUN sed -i 's/deb.debian.org/mirrors.aliyun.com/g' /etc/apt/sources.list \
    && apt-get update \
    && apt-get install -y --no-install-recommends \
        build-essential \
        crossbuild-essential-arm64 \
    && rm -rf /var/lib/apt/lists/*

# --no-install-recommends 不装推荐包，省 100MB+
```

层数过多的后果：
- 镜像层数默认最多 127 层（aufs 限制），超限构建失败
- 每层都是一个 tar 包，层数越多，docker pull 时 HTTP 请求越多，OTA 越慢
- Yocto 环境：`docker save` 导出镜像时，每层会生成一个文件，UBIFS 上小文件过多导致启动时 `mount` 超时<br>

### <strong>`WORKDIR /work` 设置工作目录（替代cd命令）</strong>

`WORKDIR` 不是简单的 `cd`，它会在镜像里创建目录（如果不存在），并且影响后续所有指令的默认路径。

对比实验：
```dockerfile
# 错误示范：cd 只在当前 RUN 有效，下一行又回根目录
RUN cd /usr/src && make
RUN cp app /opt/  # 这里的 cp 是在 / 目录执行，找不到 app

# 正确示范：WORKDIR 持久化
WORKDIR /usr/src
RUN make
RUN cp app /opt/  # 在 /usr/src 目录执行，能找到 app
```

嵌入式场景最佳实践：
```dockerfile
# 统一工作目录，避免硬编码绝对路径
WORKDIR /workspace

# 多用户开发时，WORKDIR 保证容器内路径一致
COPY . /workspace
RUN make -C /workspace/build

# 与 docker run -w 配合：
# Dockerfile 里 WORKDIR /workspace
# docker run -w /workspace/submodule 可以覆盖，灵活切换子项目
```

权限暗坑：
`WORKDIR` 创建的目录属主是 root，如果后续用 `-u $(id -u)` 运行容器，普通用户可能没权限写入。解决：
```dockerfile
# 在 Dockerfile 里提前修正属主（需要是root用户构建）
WORKDIR /workspace
RUN chown 1000:1000 /workspace  # 假设运行时UID=1000
```<br>

### <strong>`CMD ["/bin/bash"]` exec 格式 vs shell 格式差异</strong>

这是 Dockerfile 里最容易踩的坑，没有之一。`CMD` 指定容器启动时的默认命令，但exec 格式和shell 格式行为完全不同。

格式对比：
```dockerfile
# Shell 格式（默认用 /bin/sh -c 启动）
CMD /bin/bash -c "echo hello; sleep 10"
# 容器内 PID 1 是 /bin/sh，它拉起子进程执行命令
# 优点：支持管道、重定向、变量展开
# 缺点：收不到 SIGTERM 信号，docker stop 要等 10 秒超时

# Exec 格式（JSON 数组，直接 exec 执行）
CMD ["/bin/bash", "-c", "echo hello; sleep 10"]
# 容器内 PID 1 就是 /bin/bash，但 bash 还是会 fork 子进程
# 优点：能收到信号，docker stop 能快速退出
# 缺点：不支持裸写管道，必须显式调用 shell

# 正解：直接写入口程序
CMD ["/usr/bin/myapp", "-c", "/etc/config.ini"]
# PID 1 就是 myapp，信号直接送达，最干净
```

信号传递问题的血泪史：
```bash
# 用 Shell 格式写 Python 应用
CMD python3 /app/server.py
# docker stop 时，SIGTERM 发给 /bin/sh，sh 不转发给 python3
# 结果等 10 秒超时，被 SIGKILL 暴力杀掉，数据没来得及刷盘

# 正确姿势
CMD ["python3", "/app/server.py"]
# 或更稳的用 tini 做 init 进程
ENTRYPOINT ["/tini", "--"]
CMD ["python3", "/app/server.py"]
```

嵌入式场景建议：
- 前台程序必须用 Exec 格式：电机控制、数据采集这类不能优雅退出的，用 Exec 格式才能快速重启
- 调试阶段用 Shell 格式：临时容器需要进 bash 看看，`CMD /bin/bash` 方便，但生产环境记得改回来
- `tini` 或 `dumb-init`：在 Dockerfile 里安装 `tini`，解决僵尸进程回收问题（少数容器会产生僵尸进程，嵌入式系统没 systemd  inside 时很麻烦）

完整示例：
```dockerfile
FROM arm64v8/debian:bullseye-slim
RUN apt-get update && apt-get install -y tini && rm -rf /var/lib/apt/lists/*
WORKDIR /app
COPY . /app
# 编译
RUN make
# 运行
ENTRYPOINT ["/usr/bin/tini", "--"]
CMD ["/app/my-firmware-loader", "--daemon"]
```<br>

### <strong>多阶段构建 → 编译阶段含-dev包，运行阶段仅二进制</strong>

传统写法：在一个镜像里又装编译依赖，又跑应用程序。结果 `-dev` 包（头文件、静态库）占了 200MB，最终镜像 300MB，但运行时其实只需要 10MB 的二进制。

多阶段构建的魔法：
```dockerfile
# 阶段1：编译阶段（重型镜像，200MB）
FROM arm64v8/gcc:11 as builder
RUN apt-get update && apt-get install -y \
    libconfig-dev \          # -dev 包，编译时需要
    libiio-dev \             # 传感器库头文件
    && rm -rf /var/lib/apt/lists/*
WORKDIR /src
COPY . /src
RUN make -C /src/build

# 阶段2：运行阶段（轻量镜像，15MB）
FROM arm64v8/alpine:latest
RUN apk add --no-cache \
    libconfig \              # 运行时库，不带-dev
    libiio                 # 运行时库
COPY --from=builder /src/build/app /usr/local/bin/app
CMD ["/usr/local/bin/app"]
```

发生了什么：
- `FROM ... as builder` 给阶段起名字，后面能引用
- `COPY --from=builder` 把编译阶段的产物拷贝到运行阶段
- 最终镜像只包含阶段2，阶段1的 200MB 编译依赖不会被打包

嵌入式场景价值：
- OTA 省流量：300MB 变 15MB，4G 弱网环境升级成功率从 60% 提升到 95%
- 攻击面缩小：运行阶段没有 gcc、make、头文件，黑客进容器也编译不了恶意程序
- 镜像仓库省钱：云端私有仓库存储费直接砍 90%

实操排坑：
```bash
# 构建时看阶段日志
$ docker build --no-cache -t myapp:multi-stage .
[Stage 1/6] FROM arm64v8/gcc:11 as builder
[Stage 2/6] RUN apt-get update...
...
[Stage 6/6] CMD ["/usr/local/bin/app"]

# 对比镜像体积
$ docker images | grep myapp
myapp:multi-stage    15MB
myapp:single-stage   302MB  # 传统写法
```

Yocto 集成技巧：
在 Yocto 的 `docker build` 任务里，加 `--target builder` 可以只构建编译阶段，把产物导出给 SDK：
```bash
# 只构建 builder 阶段，导出工具链
$ docker build --target builder -t myapp-sdk .
$ docker run --rm myapp-sdk tar -czf - /opt/toolchain > toolchain.tar.gz
```<br>

### <strong>`.dockerignore` 排除宿主机 .git / build 目录</strong>

`docker build` 时，当前目录（context）的所有文件都会打包发给 dockerd。如果 `build/` 目录里有 2GB 的编译产物，`.git` 有 500MB 历史，`docker build` 卡在上传 step 1 半小时。

`.dockerignore` 内容（必须放在 Dockerfile 同级目录）：
```bash
# 排除所有编译产物
build/
*.o
*.a
*.so

# 排除版本控制
.git/
.gitignore

# 排除 IDE 配置
.vscode/
.idea/

# 排除宿主机临时文件
*.swp
*~

# 保留例外（以 ! 开头）
!build/essential-config.txt  # 这个文件需要
```

对比验证：
```bash
# 没 .dockerignore 时，context 大小
$ docker build -t test .
Sending build context to Docker daemon  2.345GB  # 传半天

# 加了 .dockerignore 后
$ docker build -t test .
Sending build context to Docker daemon  12.34MB   # 秒传
```

嵌入式场景的特殊排除项：
```bash
# 排除交叉编译工具链源码（在宿主机，不需要打进镜像）
toolchain-src/

# 排除设备树二进制（dtb 在宿主机测试用，镜像里用 board-specific 的）
*.dtb

# 排除调试符号文件（调试时在宿主机，生产镜像不需要）
*.debug
```

常见误解：
`.dockerignore` 和 `.gitignore` 语法类似，但作用完全不同：
- `.gitignore`：让 git 不追踪文件
- `.dockerignore`：让 `docker build` 不打包文件进 context
两者需要分别维护，别指望一个文件管两边。<br>

### <strong>`VOLUME` 声明挂载点（提示作用，非强制）</strong>

`VOLUME` 是 Dockerfile 里最鸡肋也最被误解的指令。它不挂载任何东西，只是在镜像元数据里打个标记，告诉使用者"这个目录建议挂载到宿主机"。

语法与效果：
```dockerfile
FROM arm64v8/debian:bullseye-slim
WORKDIR /app
VOLUME ["/app/data", "/app/logs"]  # 声明两个挂载点
COPY app /app/
CMD ["/app/app"]
```

运行时行为：
```bash
# 不指定 -v，Docker 会自动创建匿名卷
$ docker run -d myapp:latest
$ docker inspect <容器ID> | grep -A 5 "Mounts"
"Mounts": [
    {
        "Type": "volume",
        "Source": "/var/lib/docker/volumes/abc123/_data",
        "Destination": "/app/data"
    }
]

# 指定 -v，用宿主机的目录
$ docker run -d -v /mnt/sdcard/data:/app/data myapp:latest
```

嵌入式场景的实际意义：
1. 文档作用：告诉团队"这两个目录要挂载到持久化存储"，否则 OTA 升级时数据丢了别怪我
2. 权限预设：`VOLUME` 声明的目录，容器启动时会自动创建，属主是 root。如果应用要普通用户写，需在 Dockerfile 里提前 chown：
    ```dockerfile
    RUN groupadd -r app && useradd -r -g app app
    VOLUME ["/app/data"]
    RUN chown -R app:app /app/data
    ```

避坑指南：
- 别在 Dockerfile 里写 `VOLUME /app/config`，config 文件应该通过 `-v /host/config:/app/config:ro` 只读挂载，防止容器篡改配置
- 多个容器共享数据：用 `docker run --volumes-from <容器名>`，但在嵌入式里几乎不用，直接用 bind mount 更透明
- Yocto 环境：`VOLUME` 声明的目录会被打包进 rootfs 镜像，如果宿主机没这个目录，启动会失败。建议嵌入式里**别用 `VOLUME`**，全靠 `docker run -v` 显式挂载，出了问题一目了然

总结：
`VOLUME` 在嵌入式里的价值 ≈ 0，不如写个 `README.md` 告诉团队"启动时记得挂载 `/mnt/data:/app/data`"。但理解它的原理，能看懂别人的 Dockerfile 为什么不挂载也能跑。<br>

### <strong>`docker history --no-trunc` 查看每层具体修改</strong>

镜像体积失控时，第一步是定位"谁胖了"。`docker history` 像 CT 扫描，把洋葱一样的镜像层逐层剖开。

典型输出解读：
```bash
$ docker history myapp:build
IMAGE          CREATED        CREATED BY                                      SIZE      COMMENT
7a8b9c1d2e3f   5 minutes ago  /bin/sh -c make install                         12.5MB    
6b5a4c3d2e1f   6 minutes ago  /bin/sh -c ./configure --prefix=/usr            1.2MB     
5a4b3c2d1e0f   7 minutes ago  /bin/sh -c apt-get install -y libconfig-dev     45.3MB    
4b3a2c1d0e9f   8 minutes ago  /bin/sh -c apt-get update                       2.1MB     
3a2b1c0d9e8f   2 weeks ago    /bin/sh -c #(nop)  CMD ["/bin/bash"]            0B        
<missing>      2 weeks ago    /bin/sh -c #(nop) ADD file:abc123... in /       72MB      
```

嵌入式场景必看列：
- SIZE：每层增量。`apt-get install` 那层 45.3MB，是主要肥胖点
- CREATED BY：生成层的指令。`<missing>` 是基础镜像层，在本地被修剪过，看不到完整 ID
- COMMENT：可手动添加，记录优化意图

深潜：`--no-trunc` 看完整指令
```bash
$ docker history --no-trunc myapp:build | head -n 3
IMAGE                                                                     CREATED        CREATED BY                                                                                           SIZE
sha256:7a8b9c1d2e3f5e6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a   5 minutes ago  /bin/sh -c #(nop)  CMD ["/bin/bash"]                                                               0B
sha256:6b5a4c3d2e1f0a9b8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2a1b0c9d8e7f6a   6 minutes ago  /bin/sh -c apt-get update && apt-get install -y libconfig-dev && rm -rf /var/lib/apt/lists/*   45.3MB
```

镜像层存储原理（首次术语）：
- AUFS/Overlay2：每层是一个目录，`/var/lib/docker/overlay2/<layer-id>/diff`
- Cache ID：层内容与 ID 强相关，改一行命令就生成新 ID，无法复用旧层
- Parent 指针：每层记录父层 ID，形成链式结构，`docker pull` 就是下载这条链

嵌入式痛点：UBIFS 上的层灾难
在 UBIFS 上，每层是 64KB 对齐的，小文件填充导致写放大 5-10 倍。50 层的镜像，即使总数据量 100MB，实际占用 eMMC 可能 500MB+。

排坑指南：
```bash
# 找出层最多的镜像（层数 > 50 告警）
$ docker inspect myapp:build --format='{{.RootFS.Layers}}' | tr ' ' '\n' | wc -l
47

# 查看某层的真实存储位置
$ docker inspect myapp:build | jq '.[0].GraphDriver.Data'
{
  "LowerDir": "/var/lib/docker/overlay2/123456/diff:/var/lib/docker/overlay2/789abc/diff",
  "MergedDir": "/var/lib/docker/overlay2/def456/merged",
  "UpperDir": "/var/lib/docker/overlay2/def456/diff",
  "WorkDir": "/var/lib/docker/overlay2/def456/work"
}
```<br>

### <strong>`docker build --squash` 合并多层（需开启实验特性）</strong>

当层数太多且无法通过多阶段构建减少时，`--squash` 是终极手段：把所有层压成一层，镜像 ID 变色，历史记录清空。

启用实验特性（Buildroot 环境）：
```bash
# Buildroot 默认 dockerd 没开实验特性，需修改 service
# /etc/systemd/system/docker.service.d/override.conf
[Service]
ExecStart=
ExecStart=/usr/bin/dockerd --experimental=true

$ systemctl daemon-reload && systemctl restart docker
```

构建与对比：
```bash
# 正常构建：47 层
$ docker build -t myapp:layered .
$ docker images myapp:layered
REPOSITORY   TAG       IMAGE ID       CREATED          SIZE
myapp        layered   a1b2c3d4e5f6   10 seconds ago   215MB

# squash 构建：1 层
$ docker build --squash -t myapp:squashed .
$ docker images myapp:squashed
REPOSITORY   TAG        IMAGE ID       CREATED          SIZE
myapp        squashed   f6e5d4c3b2a1   15 seconds ago   198MB  # 体积也小了点
```

代价与权衡：
- 优点：层数=1，UBIFS 写放大消失，`docker pull` 只有一个 blob，OTA 快
- 缺点：构建缓存彻底失效，下次改一行代码也要全量构建，开发阶段别用
- 镜像 ID 变化：squash 后 ID 重新计算，无法通过 `docker diff` 看变更

嵌入式场景使用原则：
- 开发阶段：不用 squash，靠层缓存加速迭代
- 发布阶段：CI/CD 最后一个 job 加 `--squash`，生成生产镜像
- OTA 策略：squash 后镜像无法增量更新，每次 OTA 都是完整镜像，弱网环境慎用

排坑：
```bash
# squash 后 history 还在，但层 ID 全变 <missing>
$ docker history myapp:squashed
IMAGE          CREATED        CREATED BY                                      SIZE      COMMENT
f6e5d4c3b2a1   1 minute ago                                                   198MB     merge sha256:a1b2c3d4e5f6 to xxx  # 只有一层
<missing>      2 weeks ago    /bin/sh -c #(nop) ADD file:...                  72MB

# 验证 squash 是否生效：查看 overlay2 目录
$ ls /var/lib/docker/overlay2/ | wc -l
1  # 真的只有一层目录
```<br>

### <strong>`dive` 工具分析每层浪费空间</strong>

`dive` 是镜像分析的瑞士军刀，UI 直观显示每层文件树、变更类型、效率评分。

安装（离线环境）：
```bash
# 有网络时下载 static binary
$ wget https://github.com/wagoodman/dive/releases/download/v0.10.0/dive_0.10.0_linux_amd64.tar.gz
$ tar -xzf dive_0.10.0_linux_amd64.tar.gz -C /usr/local/bin

# Yocto 环境：在宿主机构建阶段安装，目标板只跑分析（dive 需要 glibc）
$ dive myapp:layered
```

UI 解读：
```
├── Layer 1: FROM debian:bullseye-slim [72MB]
│   └── Efficiency: 98%  # 基础镜像，没浪费
├── Layer 2: RUN apt-get update [2.1MB]
│   └── Efficiency: 45%  # 缓存文件没删，浪费 1.2MB
├── Layer 3: RUN apt-get install libconfig-dev [45.3MB]
│   └── Efficiency: 30%  # 安装后没删 /var/lib/apt/lists，浪费 30MB！
└── Layer 4: COPY . /app [5MB]
    └── Efficiency: 100% # 全用上了
```

嵌入式专属优化点：
1. 删除调试符号：`strip /usr/local/bin/app`，dive 会显示 size 下降
2. 合并 RUN 指令：dive 显示层数多，效率评分低，提示合并
3. 识别大文件：dive 文件树视图，一眼看到 `/usr/lib/libconfig.so.11.2` 占了 3MB，是否必需？

实战：用 dive 优化 50MB
```bash
# 第一次分析：效率 45%，浪费 30MB 主要是 apt 缓存
$ dive myapp:unoptimized
 inefficiency: 30MB (45%)

# 修改 Dockerfile，在 apt install 后加 && rm -rf /var/lib/apt/lists/*
# 重新构建
$ docker build -t myapp:optimized .

# 第二次分析：效率 92%，浪费 2MB
$ dive myapp:optimized
 inefficiency: 2MB (92%)
```

效率评分算法（核心原理）：
dive 计算（文件被修改次数 + 文件大小）/ 层数，如果某文件在多层里反复出现（如先删后加），评分就低。嵌入式镜像追求 >95%。

Yocto 环境限制：
- alpine 镜像里 dive 可能报错：`GLIBC_2.28 not found，因 dive 依赖 glibc，alpine 用 musl。解决：在 debian-based 容器里运行 dive 分析 alpine 镜像
- 无网络时：dive 第一次运行会下载 `~/.dive.yo` 配置，需提前在宿主机准备好<br>

### <strong>FROM 用 alpine 或 distroless 基础镜像</strong>

Alpine 方案（推荐，15MB）：
Alpine 用 musl libc，体积小但可能与某些工具链不兼容。测试方法：执行 `arm-linux-gcc --version`，若报错 `ld-musl-aarch64.so.1 not found`，则改用 debian-slim。

```dockerfile
# 阶段1：编译环境（仅在宿主机构建）
FROM arm64v8/gcc:11 as builder

# 阶段2：运行环境（部署到设备）
FROM arm64v8/alpine:latest

# 安装运行时库（musl 版本）
RUN apk add --no-cache libstdc++

# 从 builder 阶段拷贝工具链
COPY --from=builder /usr/local/arm-none-eabi /opt/toolchain

# 清理调试符号
RUN strip /opt/toolchain/bin/* || true

# 设置环境变量
ENV PATH="/opt/toolchain/bin:${PATH}"
CMD ["/bin/sh"]
```

Distroless 方案（激进，8MB）：
Google 的 distroless 镜像连 shell 都没有，纯运行时。适合静态编译的工具链，但无法 `docker exec` 进容器调试。

```dockerfile
FROM arm64v8/debian:bullseye-slim as builder
# ... 编译工具链 ...

FROM gcr.io/distroless/base-debian11:arm64  # 仅 8MB
COPY --from=builder /opt/toolchain /opt/toolchain
# 注意：没有 CMD，启动容器必须指定命令
```

选型决策树：
```
需要 exec 进去调试 → alpine
工具链必须 glibc → debian-slim（80MB），舍弃 alpine
极致体积 + 无需调试 → distroless
```

排坑记录：
- musl vs glibc：Alpine 上跑 `arm-none-eabi-gdb` 会崩溃，因为 GDB 依赖 glibc 的 `backtrace()`。解决：交叉编译工具链用 `debian-slim` 阶段编译，只拷贝 `.so` 到 alpine，gdb 留在 debian 阶段不拷贝。
- apk 找不到包：Alpine 的 arm64 仓库比 x86 少，某些 `libstdc++-dev` 没有。解决：编译阶段用 debian-slim，运行时 apk 只装 `-dev` 对应的运行时库。<br>

### <strong>静态编译工具链，COPY 后无需动态库</strong>

动态链接的灾难现场：
```bash
# 在 debian-slim 里编译工具链（动态链接）
$ arm-none-eabi-gcc --version
# ldd 显示依赖 10 个 .so
$ ldd /opt/toolchain/bin/arm-none-eabi-gcc
libz.so.1 => /lib/aarch64-linux-gnu/libz.so.1
libgmp.so.10 => /usr/lib/aarch64-linux-gnu/libgmp.so.10
...

# 拷贝到 alpine 容器
$ docker run -it my-gcc:dynamic arm-none-eabi-gcc
Error loading shared library libgmp.so.10: No such file or directory
```

静态编译改造：
```dockerfile
# builder 阶段：静态编译工具链
FROM arm64v8/debian:bullseye-slim as builder

# 下载官方 static 版本工具链
RUN wget https://developer.arm.com/-/media/Files/downloads/gnu/13.2.Rel1/binrel/arm-gnu-toolchain-13.2.Rel1-aarch64-arm-none-eabi.tar.xz \
    && tar -xf arm-gnu-toolchain-*.tar.xz \
    && mv arm-gnu-toolchain-*-arm-none-eabi /opt/toolchain

# 验证静态链接（关键检查）
RUN ldd /opt/toolchain/bin/arm-none-eabi-gcc | grep "not a dynamic executable" \
    || (echo "工具链不是静态链接！" && exit 1)

# runner 阶段：极简，无需任何 .so
FROM arm64v8/alpine:latest
COPY --from=builder /opt/toolchain /opt/toolchain
ENV PATH="/opt/toolchain/bin:${PATH}"
```

验证静态成功：
```bash
$ docker run --rm my-gcc:static ldd /opt/toolchain/bin/arm-none-eabi-gcc
not a dynamic executable  # 完美，不依赖任何 .so

$ docker images my-gcc:static
REPOSITORY   TAG      IMAGE ID       SIZE
my-gcc       static   a1b2c3d4e5f6   15.4MB
```

静态编译的代价：
- 体积增加：静态链接的 `arm-none-eabi-gcc` 15MB，动态的 5MB（但连带依赖 30MB）
- 法律风险：GPL 代码静态链接后，你的工具链镜像也必须开源。解决：用官方预编译 static 版本，或保留动态链接但手动拷贝 .so（不装 -dev 包）<br>

### <strong>测试镜像 `docker run --rm mytoolchain arm-linux-gcc --version`</strong>

测试是验证镜像可用性的最后一环，必须自动化，否则 OTA 后现场编译失败，设备变砖。

三步测试法：
1. 基础功能测试：
```bash
# 测试工具链版本号输出
$ docker run --rm my-toolchain:arm64 arm-none-eabi-gcc --version
arm-none-eabi-gcc (Arm GNU Toolchain 13.2.Rel1) 13.2.1 20231009

# 测试编译最小 C 程序
$ echo 'int main(){}' > test.c
$ docker run --rm -v $(pwd):/work -w /work my-toolchain:arm64 \
  arm-none-eabi-gcc test.c -o test
$ file test
test: ELF 32-bit LSB executable, ARM, EABI5 version 1 (SYSV), statically linked
```

2. 嵌入式场景测试：
```bash
# 测试生成目标设备的固件（以 Cortex-M4 为例）
$ docker run --rm -v $(pwd):/work -w /work my-toolchain:arm64 \
  arm-none-eabi-gcc -mcpu=cortex-m4 -mthumb -o firmware.elf main.c
$ docker run --rm -v $(pwd):/work -w /work my-toolchain:arm64 \
  arm-none-eabi-objcopy -O binary firmware.elf firmware.bin

# 验证固件大小
$ ls -lh firmware.bin
-rwxr-xr-x 1 1000 1000 12K Jan 10 15:30 firmware.bin
```

3. CI/CD 集成测试：
```yaml
# GitLab CI 示例
test-toolchain:
  image: docker:latest
  script:
    - docker build -t my-toolchain:${CI_COMMIT_SHA} .
    - docker run --rm my-toolchain:${CI_COMMIT_SHA} arm-none-eabi-gcc --version
    - docker run --rm -v $(pwd)/test:/work -w /work my-toolchain:${CI_COMMIT_SHA} \
        arm-none-eabi-gcc /work/test.c -o /work/test
    - ls test && file test | grep ARM  # 验证产物
  only:
    - master
```

OTA 模拟测试（关键环节）：
```bash
# 模拟弱网环境（限速 100KB/s）推送镜像
$ docker save my-toolchain:arm64 | pv -L 100k | ssh root@device "docker load"

# 在设备上验证
$ ssh root@device
device $ docker run --rm my-toolchain:arm64 arm-none-eabi-gcc main.c -o firmware
device $ ls firmware
firmware  # OTA 成功，现场编译可用
```

排坑清单：
- Q: 容器内编译报 `cannot execute binary file` 
  A: 工具链架构与容器架构不匹配。x86 宿主机必须用 `arm64v8/*` 镜像，并在容器内跑 ARM 工具链。
- Q: 编译极慢，比宿主机慢 10 倍
  A: 挂载目录在 UBIFS 上，I/O 太慢。改用 tmpfs 挂载：`docker run -v /tmp/build:/work`。
- Q: OTA 后工具链版本没变
  A: `docker build` 没加 `--no-cache`，用了旧层。CI 构建必须 `--no-cache --pull`。<br>

### <strong>工具链 2GB，团队重复安装浪费磁盘</strong>

典型场景：瑞萨 RZ/G2L 项目启动

项目经理在钉钉群发了个 `gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2`（1.8GB），说："大家都装一下，路径 `/opt/gcc-arm-none-eabi-10`"。结果：
```bash
# 新员工小A安装耗时记录
$ cd /tmp && tar -xjf gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2
# ... 等待5分钟，解压完成
$ sudo mv gcc-arm-none-eabi-10-2020-q4-major /opt/
$ du -sh /opt/gcc-arm-none-eabi-10-2020-q4-major
1.8G    /opt/gcc-arm-none-eabi-10-2020-q4-major

# 五人团队，每人一份
$ for host in dev01 dev02 dev03 ci-server nas; do ssh $host "du -sh /opt/gcc-arm-none-eabi*"; done
dev01: 1.8G
dev02: 1.8G
dev03: 1.8G
ci-server: 1.8G
nas: 1.8G  # 备份又一份
# 总计 9GB 空间，浪费 7.2GB（重复存储）
```

更隐蔽的灾难：版本漂移
两个月后，小B为了调试某个 bug，手动升级了 `/opt/gcc-arm-none-eabi-10/bin/arm-none-eabi-gdb`，从 10-2020 升级到 10.3-2021。Makefile 没改，但编译出的 binary 的调试信息格式变了：
```bash
# CI 服务器（旧版工具链）
$ arm-none-eabi-gcc -g main.c -o main.elf
$ arm-none-eabi-readelf -S main.elf | grep debug
  [28] .debug_info       PROGBITS        00000000 003a00 001234 00      0   0  1

# 小B的机器（新版工具链）
$ arm-none-eabi-gcc -g main.c -o main.elf
$ arm-none-eabi-readelf -S main.elf | grep debug
  [28] .debug_info       PROGBITS        00000000 003a00 001567 00      0   0  1  # 大小变了！

# 后果：使用新版 GDB 调试 CI 编译的旧版 binary，部分断点打不上
(gdb) break main
Cannot insert breakpoint 1: Error accessing memory address 0x8000: Input/output error.
```

传统解决方式的无效性：
- NFS 共享：`/opt/gcc-arm` 挂 NAS，但 CI 服务器在内网，网络抖动导致编译卡死
- 版本管理：`gcc-arm-none-eabi-10.tar.bz2` 重命名为 `gcc-arm-none-eabi-10.20201215.tar.bz2`，但没人维护版本对应表，半年后连文件名都看不懂

容器化价值量化：
- 存储：1.8GB 工具链 → 15MB 容器镜像（多阶段构建 + 静态编译），5 人团队总计 75MB，节省 98% 空间
- 时间：新员工入职 `docker pull my-toolchain:10-2020`，3 分钟完成
- 一致性：`docker run my-toolchain:10-2020 arm-none-eabi-gcc`，所有人、CI、现场设备，字节级一致<br>

### <strong>系统库版本差（gcc 7 vs gcc 11）导致链接错误</strong>

典型场景：Ubuntu 18.04 本地开发，Ubuntu 20.04 CI 构建

本地开发机是 Ubuntu 18.04，系统自带 `gcc-7`，但项目需要 C++17 特性，于是手动编译安装了 `gcc-11` 到 `/usr/local`。CI 服务器是 Ubuntu 20.04，系统自带 `gcc-9`，没人装 `gcc-11`。Makefile 里写的是 `CC=gcc-11`。

Jenkins 报错现场：
```bash
# Jenkins pipeline log
$ make ARCH=arm CROSS_COMPILE=arm-none-eabi-
arm-none-eabi-gcc -o main.o -c main.c
arm-none-eabi-gcc -o app.elf main.o -lnetwork -lssl
/usr/lib/gcc/arm-none-eabi/10.3.1/../../../arm-none-eabi/bin/ld: cannot find -lssl
collect2: error: ld returned 1 exit exit status
make: *** [Makefile:45: app.elf] Error 1

# CI 服务器上检查
$ ldd /usr/lib/arm-none-eabi/libssl.so
linux-vdso.so.1 (0x0000ffff)
libcrypto.so.1.0.0 => /usr/lib/aarch64-linux-gnu/libcrypto.so.1.0.0  # 宿主机的库！
libc.so.6 => /lib/aarch64-linux-gnu/libc.so.6
```

问题根源：交叉编译工具链依赖宿主机的库版本
工具链 `arm-none-eabi-gcc` 在链接时，会 **自动链接宿主机的 libgcc_s.so、libstdc++.so**，而这些库的版本随系统变：
- Ubuntu 18.04：`libstdc++.so.6.0.25`（GCC 7）
- Ubuntu 20.04：`libstdc++.so.6.0.28`（GCC 9）
- Ubuntu 22.04：`libstdc++.so.6.0.30`（GCC 11）

后果：ABI 不兼容
```bash
# 本地编译的可执行文件
$ readelf -d app.elf | grep NEEDED
0x00000001 (NEEDED)   共享库：[libstdc++.so.6]

# 在 CI 服务器上运行（虽然它是 ARM 设备，但系统库版本低）
$ ./app.elf
./app.elf: /usr/lib/libstdc++.so.6: version `CXXABI_1.3.11' not found (required by ./app.elf)
```

传统解决方式的泥潭：
- 统一开发机：强制所有人装 Ubuntu 20.04，但老员工机器上有历史项目依赖 18.04，重装成本巨大
- chroot 环境：`debootstrap` 创建 Ubuntu 20.04 chroot，但配置复杂，USB 调试器穿透困难
- LD_LIBRARY_PATH：手动指定库路径，但交叉编译工具链内部调用也会加载系统库，防不胜防

容器化根治方案：
```dockerfile
FROM arm64v8/ubuntu:20.04
RUN apt-get update && apt-get install -y gcc-11 g++-11
ENV CC=gcc-11 CXX=g++-11
```
所有人、CI、现场设备，都跑同一个 `ubuntu:20.04` 容器，宿主机系统版本彻底解耦。`ldd` 看到永远是容器内的 `libstdc++.so.6.0.28`，与宿主机无关。<br>

### <strong>CI/CD 构建环境不一致（本地能编，Jenkins 失败）</strong>

典型场景：依赖库漏提交

本地开发时 `apt-get install libconfig-dev`，编译通过。`.gitignore` 里写了 `*.o`，但忘了把 `libconfig` 的安装步骤写进 `README`。Jenkins 从零构建，没装 `libconfig-dev`，编译失败。

Jenkins 报错现场：
```bash
# Jenkins pipeline 片段
stage('Build') {
    steps {
        sh 'make CROSS_COMPILE=arm-none-eabi-'
    }
}

# Console Output
[Pipeline] sh
+ make CROSS_COMPILE=arm-none-eabi-
main.c:5:10: fatal error: libconfig.h: No such file or directory
    5 | #include <libconfig.h>
      |          ^~~~~~~~~~~~~
compilation terminated.
make: *** [Makefile:30: main.o] Error 1
[Pipeline] }
[Pipeline] // stage
```

排查过程：
```bash
# 本地能编译，因为装过 libconfig-dev
$ dpkg -l | grep libconfig-dev
ii  libconfig-dev:arm64  1.5-0.4build1  arm64  parsing and manipulation of structured configuration files

# Jenkins 容器内没装
$ docker exec -it jenkins-agent dpkg -l | grep libconfig-dev
# 无输出

# 查看 Makefile，没找到安装依赖的脚本
$ grep -r "apt-get" Makefile docs/
# 无输出
```

更隐蔽的灾难：版本号差一位
`libconfig-dev` 在 Ubuntu 20.04 是 1.5-0.4，在 Ubuntu 22.04 是 1.5-0.5。API 没变，但结构体大小调整了：
```c
// libconfig.h (Ubuntu 20.04)
struct config_t {
    unsigned int tab_width;  // 4 字节对齐
};

// libconfig.h (Ubuntu 22.04)
struct config_t {
    size_t tab_width;        // 8 字节对齐（size_t 在 64 位系统）
};
```
本地编译的二进制读写配置文件正常，Jenkins 编译的在目标设备上跑，结构体大小不匹配导致**段错误**。

传统解决方式的低效：
- Jenkins 插件装库：`sh 'apt-get install -y libconfig-dev'`，但 Jenkins 镜像每次启动都重装，浪费时间
- Jenkins Slave 快照：给 Jenkins Agent 打快照，但库更新后快照过时，回滚麻烦
- `requirements.txt`  ：只解决 Python 依赖，C/C++ 依赖没标准方案

容器化最终解：  Dockerfile 即文档
```dockerfile
FROM arm64v8/ubuntu:20.04
# 所有依赖写死版本，CI 和本地完全一致
RUN apt-get update && apt-get install -y \
    libconfig-dev=1.5-0.4build1 \
    libssl-dev=1.1.1f-1ubuntu2.19 \
    && rm -rf /var/lib/apt/lists/*
COPY . /src
WORKDIR /src
RUN make
```
Jenkins pipeline 简化到 3 行：
```groovy
pipeline {
    agent { dockerfile true }  # 自动用仓库里的 Dockerfile 构建环境

- 排查时间：从 2 小时（翻 Jenkins log、对比本地环境）缩短到 5 分钟（看 Dockerfile 第 3 行）

- 构建成功率：从 70%（环境差异导致随机失败）提升到 100%<br>

### <strong>Dockerfile 固化工具链，`FROM` 选长期支持镜像</strong>

工具链容器不是简单地把 `gcc-arm-none-eabi-10.tar.bz2` 拷贝进去，要考虑可维护性（版本升级）、可复现性（3年后还能编出同样 binary）、安全性（漏洞补丁）。

错误示范：一次性镜像
```dockerfile
FROM ubuntu:22.04
COPY gcc-arm-none-eabi-10.tar.bz2 /tmp/
RUN tar -xjf /tmp/gcc-arm-none-eabi-10.tar.bz2 -C /opt/
ENV PATH="/opt/gcc-arm-none-eabi-10/bin:${PATH}"
# 完事，但没人知道这工具链从哪下载的，哪天会过期
```

正确示范：可溯源、可升级
```dockerfile
FROM ubuntu:22.04

# 1. 元数据声明（谁、从哪、哪天）
LABEL maintainer="build-team@company.com"
LABEL toolchain.version="10-2020-q4-major"
LABEL toolchain.source="https://developer.arm.com/-/media/Files/downloads/gnu-rm/10-2020q4/gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2"
LABEL toolchain.checksum="sha256:fa8e6b7fb3bab344a8c3ff7cb341a5ce6b0c3b87441249a5c23a7b19d5d9b3a1"

# 2. 校验下载完整性（防篡改）
ADD ${toolchain.source} /tmp/toolchain.tar.bz2
RUN echo "${toolchain.checksum}  /tmp/toolchain.tar.bz2" | sha256sum -c -

# 3. 解压到标准路径
RUN tar -xjf /tmp/toolchain.tar.bz2 -C /opt/ && \
    rm /tmp/toolchain.tar.bz2

# 4. 安装系统依赖（工具链需要）
RUN apt-get update && apt-get install -y --no-install-recommends \
    lib32stdc++6 \  # 32位工具链在64位系统上需要
    libncurses5 \
    && rm -rf /var/lib/apt/lists/*

# 5. 环境变量与符号链接
ENV PATH="/opt/gcc-arm-none-eabi-10-2020-q4-major/bin:${PATH}"
ENV CROSS_COMPILE=arm-none-eabi-
WORKDIR /workspace

# 6. 健康检查（容器启动时验证）
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD arm-none-eabi-gcc --version || exit 1
```

FROM 选型的门道：
-   不要 `ubuntu:latest`  ：2025年 latest 是 24.04，2026年可能变 26.04，构建环境漂移。 锁定 LTS 版本 ：`ubuntu:22.04` 支持到 2032 年，可复现性有保障
-  架构必须显式 ：嵌入式开发在 x86 宿主机上跑 ARM 工具链，必须用 `--platform linux/amd64` 构建，或在 Dockerfile 头声明 `FROM --platform=linux/amd64 ubuntu:22.04`
-   最小化基础镜像 ：`ubuntu:22.04` 80MB，`debian:bullseye-slim` 60MB，`alpine` 5MB。但工具链通常依赖 glibc，alpine 可能不兼容， 权衡后选 debian-slim 

构建与版本管理：
```bash
# 构建时打标签，版本号与工具链一致
$ docker build -t my-toolchain:arm-none-eabi-10-2020q4-major-001 .
# 001 是镜像迭代号，工具链本身没变，但 Dockerfile 改了（比如补了个依赖）

# 推送到私有仓库，团队共享
$ docker push harbor.company.com/arm-toolchains/my-toolchain:arm-none-eabi-10-2020q4-major-001
```<br>

### <strong>Bind mount `-v /opt/toolchain:/opt/toolchain` 共享工具链</strong>

这是折中方案：工具链不打包进镜像，而是放在宿主机 `/opt/toolchain`，容器运行时 bind mount 进去。优点：镜像小（<100MB）、工具链升级只需替换宿主机目录；缺点：宿主机必须提前准备工具链，权限复杂。

适用场景：企业内部工具链 license 限制（不能公开分发），或工具链太大（5GB+），每次 docker pull 不现实。

实现步骤：

1. 宿主机准备（一次操作，永久共享）
```bash
# NFS/CI 自动分发工具链到所有开发机
$ sudo mkdir -p /opt/toolchain/arm-none-eabi-10-2020q4-major
$ sudo tar -xjf gcc-arm-none-eabi-10-2020-q4-major-x86_64-linux.tar.bz2 \
    -C /opt/toolchain/arm-none-eabi-10-2020q4-major --strip-components=1

# 修正权限，所有用户可读
$ sudo chown -R root:root /opt/toolchain
$ sudo chmod -R 755 /opt/toolchain
```

2. Dockerfile 设计（极简，只管环境）
```dockerfile
FROM debian:bullseye-slim

# 只装系统依赖，不拷工具链
RUN apt-get update && apt-get install -y --no-install-recommends \
    lib32stdc++6 libncurses5 && rm -rf /var/lib/apt/lists/*

# 创建挂载点
RUN mkdir -p /opt/toolchain

# 默认环境变量（可被 docker run -e 覆盖）
ENV CROSS_COMPILE=arm-none-eabi-
WORKDIR /workspace
```

3. 运行时挂载（关键）
```bash
$ docker run -it --rm \
  -v /opt/toolchain/arm-none-eabi-10-2020q4-major:/opt/toolchain \
  -e PATH="/opt/toolchain/bin:${PATH}" \
  my-toolchain-env:latest \
  arm-none-eabi-gcc --version
```

权限暗坑与解决方案：
坑 1：容器内 UID/GID 与宿主机不匹配**
```bash
# 容器内默认 root (0:0)，但工具链属主是宿主机 1000:1000
$ ls -l /opt/toolchain/bin/arm-none-eabi-gcc
-rwxr-xr-x 1 1000 1000 1234567 /opt/toolchain/bin/arm-none-eabi-gcc
# 容器内 root 运行报错：Permission denied
```
解 1：docker run 指定 UID
```bash
$ docker run -it --rm \
  -v /opt/toolchain:/opt/toolchain \
  -u $(stat -c "%u:%g" /opt/toolchain/bin/arm-none-eabi-gcc) \
  my-toolchain-env arm-none-eabi-gcc --version
```

坑 2：SELinux 拦截（RHEL/CentOS 环境）
```bash
# 挂载后容器内看不到文件
$ docker run -it --rm -v /opt/toolchain:/opt/toolchain my-toolchain-env ls /opt/toolchain
# 空

# 宿主机 dmesg 看 AVC 拒绝
$ dmesg | grep avc
type=AVC msg=audit(...): avc:  denied  { read } for  pid=1234 comm="arm-none-eabi-g" name="arm-none-eabi-gcc" dev="sda1" ino=12345 scontext=system_u:system_r:container_t:s0:c123,c456 tcontext=unconfined_u:object_r:usr_t:s0 tclass=file permissive=0
```
解 2：挂载时加 :z 或 :Z 标签
```bash
$ docker run -it --rm -v /opt/toolchain:/opt/toolchain:Z my-toolchain-env ...
# :Z 表示容器有读写权限，SELinux 重新打标签
```

性能陷阱：
```bash
# 工具链在 NFS 挂载目录上（/mnt/nfs/toolchain）
$ docker run -v /mnt/nfs/toolchain:/opt/toolchain my-toolchain-env make -j8
# 编译极慢，因为 NFS 延迟高，且 docker overlay2 在 NFS 上有 bug

# 解决：先用 rsync 同步到本地，再挂载本地目录
$ rsync -av /mnt/nfs/toolchain/ /opt/toolchain/
$ docker run -v /opt/toolchain:/opt/toolchain my-toolchain-env make -j8
```<br>

### <strong>源码目录挂载 `-v $(pwd):/work`，构建产物宿主机直接读取</strong>

这是开发阶段的神器：源码在宿主机用 VS Code 编辑，容器内编译，产物直接落回宿主机，无需 `docker cp`。但需处理 UID 同步 和 文件系统事件 两个问题。

基础用法：
```bash
$ docker run -it --rm \
  -v /opt/toolchain:/opt/toolchain \
  -v $(pwd):/work \          # 宿主机当前目录挂载到容器 /work
  -w /work \                 # 容器启动后自动 cd 到 /work
  my-toolchain-env \
  make CROSS_COMPILE=arm-none-eabi-
```

产物直接可用：
```bash
# 编译完成，产物在宿主机
$ ls -lh firmware.elf firmware.bin
-rwxr-xr-x 1 lin users 124K Jan 10 16:00 firmware.elf
-rwxr-xr-x 1 lin users  45K Jan 10 16:00 firmware.bin

# 直接烧录，无需 docker cp
$ st-flash write firmware.bin 0x08000000
```

UID 同步问题（核心痛点）：
```bash
# 容器内默认 root，编译产物属主是 root
$ docker run -v $(pwd):/work my-toolchain-env make
$ ls -l firmware.elf
-rwxr-xr-x 1 root root 124K Jan 10 16:00 firmware.elf

# 宿主机用户 lin 无法修改
$ rm firmware.elf
rm: remove write-protected regular file 'firmware.elf'? y
rm: cannot remove 'firmware.elf': Permission denied
```

解：docker run 指定当前用户 UID/GID
```bash
$ docker run -it --rm \
  -v $(pwd):/work \
  -u $(id -u):$(id -g) \     # 关键：容器内用宿主机用户身份运行
  my-toolchain-env make

# 产物属主正确
$ ls -l firmware.elf
-rwxr-xr-x 1 lin users 124K Jan 10 16:05 firmware.elf
```

副作用与修复：
```bash
# 指定 UID 后，容器内 /etc/passwd 没有该用户，whoami 报错
$ docker run -u $(id -u):$(id -g) my-toolchain-env whoami
whoami: cannot find name for user ID 1000

# 解决：挂载宿主机的 passwd 和 group（开发环境可用）
$ docker run -it --rm \
  -v $(pwd):/work \
  -u $(id -u):$(id -g) \
  -v /etc/passwd:/etc/passwd:ro \
  -v /etc/group:/etc/group:ro \
  my-toolchain-env whoami
lin
```

文件系统事件问题：
```bash
# 宿主机用 VS Code 修改 main.c 并保存
# 容器内 make 没触发重新编译（因为 make 依赖文件的 mtime）
$ docker run -v $(pwd):/work my-toolchain-env make
make: 'main.o' is up to date.  # 没重新编译！

# 原因：容器与宿主机的文件系统时钟不同步（NTP 延迟）
```
解：强制刷新或 touch
```bash
# 方案1：make 前 touch 所有源文件
$ docker run -v $(pwd):/work my-toolchain-env bash -c "touch src/*.c && make"

# 方案2：用 inotify 工具监控（复杂，不推荐）
# 方案3：用 ccache 缓存编译结果，不靠 mtime 判断
$ docker run -v $(pwd):/work -v ~/.ccache:/root/.ccache my-toolchain-env make
```

完整工作流脚本（可复制）：
```bash
#!/bin/bash
# build.sh
set -e

# 自动检测 UID/GID
USER_ID=$(id -u)
GROUP_ID=$(id -g)

# 启动容器编译
docker run -it --rm \
  -v /opt/toolchain:/opt/toolchain \
  -v $(pwd):/work \
  -w /work \
  -u ${USER_ID}:${GROUP_ID} \
  -e PATH="/opt/toolchain/bin:${PATH}" \
  my-toolchain-env:latest \
  make -j$(nproc) CROSS_COMPILE=arm-none-eabi-

# 编译完成，产物在宿主机
echo "构建完成，产物："
ls -lh firmware.*
```<br>

### <strong>GitLab CI `image: toolchain-arm:latest` 直接使用</strong>

GitLab Runner 的 `docker` executor 原生支持 `image` 指令，但 Runner 本身的配置决定一切。

极简版 `.gitlab-ci.yml`： 
```yaml
# 适用于 GitLab Runner 配置：privileged = false, volumes = ["/cache"]
stages:
  - build

build-firmware:
  stage: build
  image: harbor.company.com/arm-toolchains/arm-none-eabi-gcc:10-2020q4
  script:
    - arm-none-eabi-gcc --version
    - make CROSS_COMPILE=arm-none-eabi- all
  artifacts:
    paths:
      - firmware.bin
    expire_in: 1 week
```

隐藏的权限陷阱： 
```bash
# GitLab Runner 日志：cannot create '/builds/project/main.o': Permission denied
$ cat /var/log/gitlab-runner/current
ERROR: Job failed: exit status 1

# 原因：GitLab Runner 默认用 root 启动容器，但宿主机目录属主是 gitlab-runner (UID 999)
$ ls -l /home/gitlab-runner/builds/project/
-rw-r--r-- 1 gitlab-runner gitlab-runner 1234 main.c

# 容器内 root (0:0) 无法写入 gitlab-runner (999:999) 拥有的目录
```

解决方案：GitLab Runner `config.toml` 配置 
```toml
[[runners]]
  executor = "docker"
  [runners.docker]
    image = "debian:bullseye-slim"
    privileged = false
    volumes = ["/cache", "/home/gitlab-runner/toolchain:/opt/toolchain:ro"]
    # 关键：挂载宿主机的 passwd 和 group，让容器内能识别 gitlab-runner 用户
    volumes = ["/etc/passwd:/etc/passwd:ro", "/etc/group:/etc/group:ro"]
    # 关键：指定用户
    user = "gitlab-runner"
```

`.gitlab-ci.yml` 适配： 
```yaml
build-firmware:
  image: harbor.company.com/arm-toolchains/arm-none-eabi-gcc:10-2020q4
  # 必须覆盖 user，否则 docker executor 会强制用 gitlab-runner 用户
  user: root  # 或 "999:999"（直接写 UID）
  script:
    - id  # 确认容器内 UID=999
    - make
```

Docker-in-Docker 陷阱： 
如果 GitLab Runner 用 `docker:dind` 服务（需要构建 Docker 镜像），工具链容器内 **无法直接访问宿主机 Docker socket**：
```yaml
# 错误配置
build-firmware:
  services:
    - docker:dind
  image: my-toolchain:latest
  script:
    - docker ps  # 报错：Cannot connect to the Docker daemon

# 正确配置：工具链容器继承 Docker socket
build-firmware:
  image: my-toolchain:latest
  variables:
    DOCKER_HOST: tcp://docker:2375
  services:
    - docker:dind
  script:
    - docker info  # 现在可以访问
```

缓存策略（关键）： 
```yaml
# .gitlab-ci.yml，利用 GitLab Runner 的 /cache 目录
variables:
  CCACHE_DIR: "/cache/ccache"

build-firmware:
  image: my-toolchain:latest
  script:
    - ccache -z  # 清零统计
    - make CC="ccache arm-none-eabi-gcc"
    - ccache -s  # 打印命中率
  cache:  # GitLab 自动挂载 /cache
    paths:
      - ccache/
    key: "$CI_COMMIT_REF_NAME-ccache"
```

私有仓库认证：
```yaml
# 在 GitLab 项目 Settings -> CI/CD -> Variables 里加
# DOCKER_AUTH_CONFIG = {"auths":{"harbor.company.com":{"username":"ci","password":"..."}}}

build-firmware:
  image: harbor.company.com/arm-toolchains/gcc:latest
  # Runner 自动用这个 config 拉镜像
```<br>

Jenkins 的 `docker` 插件比 GitLab 更灵活，但 `inside()` 的 workspace 挂载行为诡异。

基础 Jenkinsfile：
```groovy
pipeline {
    agent {

```

`inside()` 的 UID 噩梦： 
```groovy
// 错误：产物属主 jenkins (1000)，但宿主机 jenkins 是 1001
pipeline {
    agent { docker 'my-toolchain:latest' }
    stages {
        stage('Build') {
            steps {
                sh 'make'
            }
            post {
                always {
                    archiveArtifacts artifacts: 'firmware.bin'  # 报错：Permission denied
                }
            }

}

// 正确：指定 user，与宿主机 jenkins UID 一致
pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            args '-u 1001:1001'  // 宿主机 jenkins 的 UID
        }
    }
    ...
}
```

获取宿主机 Jenkins UID 的 trick： 
```groovy
// 在 Jenkinsfile 开头动态获取
def HOST_UID = sh(script: 'id -u', returnStdout: true).trim()
def HOST_GID = sh(script: 'id -g', returnStdout: true).trim()

pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            args "-u ${HOST_UID}:${HOST_GID}"
        }
    }
    ...
}
```

Docker Pipeline 的 workspace 挂载机制： 
```groovy
// inside() 自动挂载：
// -v /var/jenkins_home/workspace/myproject:/var/jenkins_home/workspace/myproject:rw,z
// 但容器内 /workspace 软链接到 /var/jenkins_home/workspace/myproject
// 某些工具链在处理软链接路径时出错

// 解决：禁用默认 workspace，手动挂载
pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            args '-v /var/jenkins_home/workspace/myproject:/workspace:rw,z -w /workspace'
            reuseNode true  // 关键：在同一节点上跑，workspace 不变
        }
    }
    ...
}
```

Docker 镜像缓存（Jenkins 重启后丢失）：
```groovy
// Jenkins 默认每次构建拉最新镜像，浪费带宽
// 在 Jenkins 系统配置里加：Docker Pull Strategy = Never

// 或在 Jenkinsfile 里控制
pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            alwaysPull false  // 用本地缓存
        }
    }
}
```

并行构建的容器隔离：
```groovy
// 多分支并行构建，容器名冲突
pipeline {
    parallel {
        stage('Build ARMv7') {
            agent { docker 'my-toolchain:armv7' }
            steps { sh 'make' }
        }
        stage('Build ARMv8') {
            agent { docker 'my-toolchain:armv8' }
            steps { sh 'make' }
        }
    }
}
// 风险：两个容器同时写 workspace，文件污染

// 解：每个分支用独立 workspace
stage('Build ARMv7') {
    agent {
        docker {
            image 'my-toolchain:armv7'
            reuseNode false  // 强制分配新节点或新 workspace
        }
    }
}
```<br>

### <strong>构建缓存 `-v ~/.ccache:/root/.ccache` 加速重复编译</strong>

嵌入式项目动辄 5000+ 源文件，`ccache` 是生命线。容器化后，缓存必须跨构建持久化。

ccache 原理（首次术语）： 缓存编译时预处理后的代码 + 编译选项哈希，下次同样输入时直接返回缓存的 object 文件，跳过编译。

Dockerfile 集成： 
```dockerfile
FROM debian:bullseye-slim

# 安装 ccache
RUN apt-get update && apt-get install -y ccache && rm -rf /var/lib/apt/lists/*

# 配置 ccache（容器内）
ENV CCACHE_DIR=/root/.ccache
ENV CCACHE_MAXSIZE=5G
ENV CCACHE_SLOPPINESS=file_macro,time_macros  # 嵌入式需要，因为 __TIME__ 宏会变

# 包装编译器
RUN ln -s /usr/bin/ccache /usr/local/bin/arm-none-eabi-gcc && \
    ln -s /usr/bin/ccache /usr/local/bin/arm-none-eabi-g++
```

GitLab CI 缓存挂载：
```yaml
variables:
  CCACHE_DIR: /root/.ccache

build-firmware:
  image: my-toolchain:latest
  script:
    - ccache -z
    - make CC="ccache arm-none-eabi-gcc"
    - ccache -s  # 打印命中率
  cache:
    key: "ccache-${CI_COMMIT_REF_SLUG}"
    paths:
      - /root/.ccache
```

Jenkins 缓存挂载：
```groovy
pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            args '-v /var/jenkins_home/ccache:/root/.ccache:rw'
        }
    }
    stages {
        stage('Build') {
            steps {
                sh 'ccache -z'
                sh 'make CC="ccache arm-none-eabi-gcc"'
                sh 'ccache -s'
            }
        }
    }
}
```

命中率调优（嵌入式专属）： 
```bash
# 第一次构建：0%
$ ccache -s
cache hit rate: 0.00%

# 只改一个 .c 文件，第二次构建：99%
$ ccache -s
cache hit rate: 99.45%

# 修改头文件，导致所有文件重编：hit rate 下降
# 解决：用 ccache 的 "depend mode"
$ export CCACHE_DEPEND=1
$ make CC="ccache arm-none-eabi-gcc" CPPFLAGS="-MD -MP"
```

### <strong>容器内访问宿主机 USB JTAG 调试器（`--device=/dev/bus/usb/001/003`）</strong>

场景：编译完后直接在容器内烧录固件
```bash
# 期望工作流
$ docker run -it --rm \
  -v $(pwd):/work -w /work \
  -v /opt/toolchain:/opt/toolchain \
  my-toolchain:latest \
  bash -c "make && st-flash write firmware.bin 0x08000000"
```

第一步失败：设备不可见 
```bash
# 容器内看不到 USB 设备
$ docker run -it my-toolchain:latest ls /dev/bus/usb
ls: cannot access '/dev/bus/usb': No such file

# 宿主机上设备存在
$ lsusb
Bus 001 Device 003: ID 0483:3748 STMicroelectronics ST-LINK/V2
```

正确挂载：
```bash
# 方式1：挂载整个 USB 总线（推荐，热插拔可用）
$ docker run -it --rm --privileged my-toolchain:latest ls /dev/bus/usb/001/003
# 但 privileged 权限太大，违反安全规范

# 方式2：精确挂载单个设备（生产环境）
$ docker run -it --rm \
  --device=/dev/bus/usb/001/003 \
  my-toolchain:latest st-flash write firmware.bin 0x08000000

# 方式3：通过 cgroup devices 白名单（最安全）
$ docker run -it --rm \
  --device-cgroup-rule='c 189:* rwm' \
  my-toolchain:latest st-flash write firmware.bin 0x08000000
# c 189:* 表示 character device, major=189 (USB), minor=任意
```

udev 规则自动化（宿主机配置）： 
```bash
# /etc/udev/rules.d/99-stlink.rules
SUBSYSTEM=="usb", ATTR{idVendor}=="0483", ATTR{idProduct}=="3748", MODE="0666", GROUP="plugdev"

# 重载 udev
$ sudo udevadm control --reload-rules && sudo udevadm trigger

# 验证设备权限
$ ls -l /dev/bus/usb/001/003
crw-rw-rw- 1 root plugdev 189, 2 Jan 10 17:00 /dev/bus/usb/001/003
```

SELinux/AppArmor 拦截（Ubuntu/Debian 默认启用）：
```bash
# 容器内访问设备被 SELinux 拒绝
$ docker run --device=/dev/bus/usb/001/003 my-toolchain:latest st-flash
st-flash: error opening device "/dev/bus/usb/001/003": Operation not permitted

# 宿主机 dmesg 看拒绝日志
$ dmesg | grep avc | tail -n 1
type=AVC msg=audit(...): avc:  denied  { read write } for  pid=12345 comm="st-flash" \
  name="003" dev="devtmpfs" ino=12345 scontext=system_u:system_r:container_t:s0:c123,c456 \
  tcontext=system_u:object_r:usb_device_t:s0 tclass=chr_file permissive=0

# 临时解决：加 --security-opt 标签
$ docker run --device=/dev/bus/usb/001/003 \
  --security-opt label=type:container_runtime_t \
  my-toolchain:latest st-flash
```

热插拔困境与解决方案： 
```bash
# 容器运行中拔掉 ST-LINK，再插上，设备号变了（003→004）
$ lsusb
Bus 001 Device 004: ID 0483:3748 STMicroelectronics ST-LINK/V2

# 容器内无法自动感知新设备号
# 解：用 --privileged + mount propagation
$ docker run -it --rm --privileged \
  --mount type=bind,source=/dev/bus/usb,target=/dev/bus/usb,bind-propagation=shared \
  my-toolchain:latest bash

# 在容器内执行 udevadm monitor，实时监听设备插拔
root@container:/work# udevadm monitor --kernel --property
KERNEL[12345.678] add      /devices/pci0000:00/0000:00:14.0/usb1/1-2 (usb)
KERNEL[12345.679] add      /devices/pci0000:00/0000:00:14.0/usb1/1-2/1-2:1.0 (usb)
```<br>

### <strong>权限问题：`docker run -u $(id -u):$(id -g)` 保持宿主机 UID/GID</strong>

场景：容器内编译后，产物在宿主机无法删除
```bash
# 不加 -u，产物属主 root
$ docker run -v $(pwd):/work my-toolchain make
$ ls -l firmware.bin
-rwxr-xr-x 1 root root 45K Jan 10 18:00 firmware.bin
$ rm firmware.bin
rm: cannot remove 'firmware.bin': Permission denied
```

初级解决：docker run -u
```bash
$ docker run -v $(pwd):/work -u $(id -u):$(id -g) my-toolchain make
$ ls -l firmware.bin
-rwxr-xr-x 1 lin users 45K Jan 10 18:05 firmware.bin  # 完美
```

中级陷阱：容器内用户不在 /etc/passwd 
```bash
$ docker run -u $(id -u):$(id -g) my-toolchain whoami
whoami: cannot find name for user ID 1000

# 导致某些构建脚本失败（比如 Python setup.py 查询用户名）
$ docker run -u $(id -u):$(id -g) my-toolchain python3 setup.py build
...
File "/usr/lib/python3.9/getpass.py", line 169, in getuser
    return pwd.getpwuid(os.getuid())[0]
KeyError: 'getpwuid(): uid not found: 1000'
```

中级解决：挂载 passwd 和 group 
```bash
$ docker run -it --rm \
  -v $(pwd):/work \
  -u $(id -u):$(id -g) \
  -v /etc/passwd:/etc/passwd:ro \
  -v /etc/group:/etc/group:ro \
  my-toolchain whoami
lin
```

高级陷阱：sudo 权限 
```bash
# 某些构建脚本需要 sudo 安装临时依赖
$ docker run -u $(id -u):$(id -g) my-toolchain sudo apt-get install xyz
sudo: you do not exist in the passwd database
```

高级解决：容器内配置 sudoers（Dockerfile 层面）
```dockerfile
FROM my-toolchain:latest

# 安装 sudo
RUN apt-get update && apt-get install -y sudo && rm -rf /var/lib/apt/lists/*

# 允许任意用户无密码 sudo（仅构建容器）
RUN echo "ALL ALL=(ALL) NOPASSWD:ALL" >> /etc/sudoers

# 创建构建用户（UID 1000，与宿主机一致）
RUN groupadd -g 1000 builder && useradd -m -u 1000 -g 1000 builder

# Jenkins/GitLab 运行时用 -u 1000:1000
```

构建产物权限的终极问题：setuid/setgid 位丢失
```bash
# 容器内编译 busybox，需要 setuid 位
$ make CC="arm-none-eabi-gcc" CONFIG_BUSYBOX_SUID=y

# 宿主机上检查，setuid 位没了
$ ls -l _install/bin/busybox
-rwxr-xr-x 1 lin users 1234K Jan 10 18:10 busybox  # 没有 s 位！

# 原因：Docker 默认挂载是 nosuid
# 解：用 --privileged 或 --security-opt（不推荐）
# 或改为编译后手工 chmod（推荐）
$ docker run ... my-toolchain bash -c "make && chmod +s _install/bin/busybox"
```

Jenkins 专属陷阱：
```groovy
// Jenkins 默认用 jenkins 用户启动容器，但 jenkins 用户在容器内是 root 组
pipeline {
    agent { docker { image 'my-toolchain:latest' } }
    stages {
        stage('Build') {
            steps {
                sh 'touch test'  // 创建文件属主 root:root
            }
        }
    }
}
// 后果：Jenkins 清理 workspace 时 Permission denied

// 解：在 Jenkins 系统配置里加 Docker 模板
// Manage Jenkins -> Nodes -> Configure Clouds -> Docker Agent templates
// Add Environment variable: "JENKINS_USER=jenkins" 和 "JENKINS_GROUP=jenkins"
```<br>

### <strong>时区同步：`-v /etc/localtime:/etc/localtime:ro`</strong>

场景：编译日志带时间戳，跨时区团队协作排查
```bash
# 容器内默认 UTC 时间，但宿主机是 Asia/Shanghai
$ docker run my-toolchain date
Thu Jan 10 10:00:00 UTC 2025

$ date
Thu Jan 10 18:00:00 CST 2025
# 日志时间差 8 小时，排错时搞不清先后关系
```

基础解决：挂载 localtime 
```bash
$ docker run -it --rm \
  -v /etc/localtime:/etc/localtime:ro \
  my-toolchain date
Thu Jan 10 18:00:00 CST 2025  # 与宿主机一致
```

中级问题：时区数据库缺失 
```bash
# 某些极简镜像（alpine/distroless）没有 /usr/share/zoneinfo
$ docker run -it -v /etc/localtime:/etc/localtime:ro alpine date
Thu Jan 10 10:00:00 UTC 2025  # 没生效

# 因为 alpine 用 musl，读取时区方式不同
$ docker run -it alpine sh -c "apk add tzdata && TZ=Asia/Shanghai date"
Thu Jan 10 18:00:00 CST 2025
```

Dockerfile 层面解决： 
```dockerfile
FROM alpine:latest

# 安装时区数据
RUN apk add --no-cache tzdata

# 设置默认时区为上海（可运行时覆盖）
ENV TZ=Asia/Shanghai

# 挂载宿主机的 localtime（推荐）
# 或在运行时：docker run -e TZ=Asia/Shanghai
```

高级场景：构建时间戳 reproducibility（可复现性）
```bash
# Dockerfile 里用 SOURCE_DATE_EPOCH 固化时间
FROM debian:bullseye-slim
ENV SOURCE_DATE_EPOCH=1640000000  # 固定为 2021-12-20 20:33:20 UTC

# 编译时用 --date 选项
RUN gcc -o app main.c -Wno-date-time && touch -t $(date -d @$SOURCE_DATE_EPOCH +%Y%m%d%H%M.%S) app

# 结果：无论何时构建，app 的文件时间戳一致，便于校验
```

Jenkins/GitLab 日志时间错乱： 
```groovy
// Jenkins 控制台日志时间是 Jenkins 服务器时间
// 容器内编译日志是 UTC 时间，混合后难以分析

// 解：统一用 UTC，或强制容器同步 Jenkins 时区
pipeline {
    agent {
        docker {
            image 'my-toolchain:latest'
            args '-v /etc/localtime:/etc/localtime:ro'
        }
    }
    stages {
        stage('Build') {
            steps {
                sh 'date'  // 确认与 Jenkins 时间一致
            }
        }
    }
}
```

法律合规：审计日志时区
某些行业（汽车、医疗）要求所有日志必须带时区且可追溯。容器内若用 UTC，需明确记录：
```bash
# 编译脚本头部打印时间戳
echo "Build started at: $(date -Iseconds --utc) UTC" > build.log
# 2025-01-10T18:00:00+08:00  # ISO 8601 格式，带时区偏移
```<br>
