### <strong>单文件完整流程：.c→.o→可执行文件的命令链（gcc -c/-o参数用法）</strong>

交叉编译的单文件流程看似简单（直接用交叉编译器编译生成可执行文件），但深入拆解“源码（.c）→目标文件（.o）→可执行文件”的三步链路，能帮我们理解“编译”与“链接”的核心分工——这是后续多文件编译、增量编译的基础。

很多新手习惯“一步编译”（直接生成可执行文件），但拆分步骤的核心价值是：当文件被修改时，仅重新编译该文件的目标文件，无需全量重编，提升大型项目效率。

我们以“交叉编译ARM架构的串口打印程序”为例（比Hello World更贴近嵌入式场景），完整演示拆分流程，使用Linaro的`arm-linux-gnueabihf-gcc`工具链。

1.  步骤1：编写单文件源码（uart_demo.c）
    源码实现“初始化虚拟串口+打印调试信息”的简单逻辑（嵌入式开发中串口打印是基础调试手段），代码如下：
    ```c
    #include <stdio.h>
    #include <string.h>

    // 模拟串口初始化（嵌入式实际开发中会操作硬件寄存器）
    void uart_init(void) {
        printf("[UART] Init success (baud rate: 115200)\n");
    }

    // 模拟串口发送字符串
    void uart_send(const char *data) {
        printf("[UART] Send: %s\n", data);
    }

    int main(void) {
        uart_init();                  // 初始化串口
        uart_send("Cross Compile Demo");  // 发送数据
        return 0;
    }
    ```
    保存源码到宿主机器的`~/cross_demo/single_file/`目录下。

2.  步骤2：编译源码为目标文件（.o，编译阶段）
    编译阶段的核心是“将.c源码翻译成目标架构的机器指令，生成目标文件”，不进行链接（不关联库文件）。关键参数`-c`（仅编译不链接）和`-o`（指定输出文件名），命令如下：
    ```bash
    # 进入工作目录
    cd ~/cross_demo/single_file/
    # 交叉编译生成目标文件uart_demo.o（ARM架构）
    arm-linux-gnueabihf-gcc -c uart_demo.c -o uart_demo.o -march=armv7-a -Wall
    ```
    参数说明：
    - `-c`：告诉编译器“仅执行编译步骤，生成目标文件，不链接”；
    - `-march=armv7-a`：适配ARMv7-A架构（如树莓派2），避免生成不兼容指令；
    - `-Wall`：开启所有编译警告（新手必加，提前发现逻辑隐患）。

    验证结果：执行`ls`命令会看到`uart_demo.o`文件，用`file`命令确认架构：
    ```bash
    file uart_demo.o
    # 预期输出：uart_demo.o: ELF 32-bit LSB relocatable, ARM, EABI5 version 1 (SYSV), not stripped
    ```
    关键信息“relocatable”（可重定位）说明这是目标文件，还未关联库和入口地址，无法直接运行。<br>

### <strong>多文件编译：Makefile的交叉编译适配（指定CC=arm-linux-gnueabihf-gcc）</strong>

当项目包含多个源码文件（如`main.c`调用`func.c`的函数）时，手动分步编译会变得繁琐（修改一个文件需重新执行多条编译命令）

Makefile是解决多文件自动化编译的核心工具——它通过“目标-依赖-命令”的规则，自动识别修改的文件并仅重编译该文件，同时支持“一键编译”“一键清理”，是嵌入式开发的必备技能。

交叉编译场景的核心适配点是“指定交叉编译器（CC变量）”，避免默认使用本地gcc。

1.  多文件项目场景构建
    先创建一个典型的多文件项目（`main.c`为主函数入口，`func.c`提供工具函数，`func.h`为头文件），目录结构如下：
    ```
    ~/cross_demo/multi_file/
    ├── main.c    # 主函数，调用func.c的函数
    ├── func.c    # 工具函数实现（如字符串拼接）
    └── func.h    # 工具函数声明
    ```
    各文件源码：
    - `main.c`：
      ```c
      #include <stdio.h>
      #include "func.h"  // 引用自定义头文件

      int main(void) {
          char buf[32] = {0};
          // 调用func.c的字符串拼接函数
          str_cat(buf, "Cross ", "Compile");
          printf("Result: %s\n", buf);
          return 0;
      }
      ```
    - `func.c`：
      ```c
      #include "func.h"
      #include <string.h>

      // 自定义字符串拼接函数（将s2和s3拼接到s1）
      void str_cat(char *s1, const char *s2, const char *s3) {
          strcpy(s1, s2);
          strcat(s1, s3);
      }
      ```
    - `func.h`：
      ```c
      #ifndef FUNC_H
      #define FUNC_H
      // 函数声明
      void str_cat(char *s1, const char *s2, const char *s3);
      #endif
      ```

2.  手动编译多文件（理解繁琐性）
    先演示手动编译流程，凸显Makefile的必要性：
    ```bash
    # 1. 编译func.c为目标文件（修改后仅需重编这一步）
    arm-linux-gnueabihf-gcc -c func.c -o func.o -march=armv7-a -Wall
    # 2. 编译main.c为目标文件（修改后仅需重编这一步）
    arm-linux-gnueabihf-gcc -c main.c -o main.o -march=armv7-a -Wall
    # 3. 链接所有目标文件为可执行程序
    arm-linux-gnueabihf-gcc main.o func.o -o multi_demo -static -march=armv7-a
    # 4. 部署运行（验证结果）
    scp multi_demo root@192.168.1.100:/root/
    ssh root@192.168.1.100 "./multi_demo"
    # 预期输出：Result: CrossCompile
    ```
    问题凸显：若项目有10个文件，每次修改一个文件需手动执行“编译该文件→链接所有文件”，效率极低且易出错——Makefile可解决此问题。<br>

### <strong>新手避坑：编译警告的潜在风险（如“unused variable”与调试信息保留）</strong>

嵌入式开发中“编译成功≠程序正确”，新手常忽略编译警告、误删调试信息，导致后续调试或量产时出现难以定位的问题。
本部分聚焦两个核心坑点，搭配实操案例说明规避方法。

1.  避坑点1：编译警告是“潜在错误”，绝非“可忽略提示”
    编译器警告（如“unused variable”“implicit declaration of function”）不影响编译通过，但大概率是逻辑错误或兼容性问题。例如“implicit declaration of function”（函数未声明）可能导致链接时找不到函数实现，“unused variable”可能是变量名拼写错误（如把`buf`写成`buf1`）。

    核心规避方法：编译时必加`-Wall`参数开启所有警告，强制修复所有警告后再进行后续测试。
    实操案例（警告触发与修复）：
    ```bash
    # 1. 故意写有警告的代码（main.c中定义变量但未使用）
    echo '#include <stdio.h> int main(){int unused_var; printf("Warning Demo\n"); return 0;}' > warn_demo.c

    # 2. 不加-Wall编译（无警告提示，隐藏风险）
    arm-linux-gnueabihf-gcc warn_demo.c -o warn_demo -static
    # 输出：无任何警告，编译“成功”

    # 3. 加-Wall编译（触发警告，暴露风险）
    arm-linux-gnueabihf-gcc warn_demo.c -o warn_demo -static -Wall
    # 预期警告输出：
    # warn_demo.c: In function ‘main’:
    # warn_demo.c:1:29: warning: unused variable ‘unused_var’ [-Wunused-variable]
    #  int main(){int unused_var; printf("Warning Demo\n"); return 0;}

    # 4. 修复警告（删除未使用变量或使用它）
    echo '#include <stdio.h> int main(){int used_var=1; printf("Warning Fixed! %d\n", used_var); return 0;}' > warn_demo.c
    # 重新编译（无警告）
    arm-linux-gnueabihf-gcc warn_demo.c -o warn_demo -static -Wall
    # 部署运行（正常输出）
    ssh root@192.168.1.100 "./warn_demo"
    # 预期输出：Warning Fixed! 1
    ```
    关键结论：嵌入式开发中“零警告编译”是基本要求，`-Wall`参数必须加入编译命令或Makefile的`CFLAGS`中。<br>

### <strong>sysroot配置两种方式：--sysroot参数 vs 环境变量SYSROOT</strong>

`sysroot`是交叉编译中“隔离宿主库与目标机库”的核心配置，前序章节已讲透其“目标机文件系统镜像”的核心原理，
本小节聚焦实操配置——两种主流配置方式（临时参数、全局环境变量）的操作步骤、适用场景及差异对比，解决“不同项目如何灵活适配sysroot”的问题。

1.  核心前提：明确sysroot的来源与路径
    配置前需先确认sysroot的路径（新手常因路径错误导致配置失效），主流sysroot来源有两种：
    - 工具链内置sysroot：Linaro、Buildroot等工具链默认自带，路径通常在工具链安装目录下的`sysroot`文件夹（如`/opt/arm-linux-gnueabihf/sysroot`）；
    - 自定义sysroot：从目标机拷贝的文件系统镜像（如`/home/user/arm_sysroot`，前序章节有拷贝方法）。
    可通过工具链自带命令快速查看内置sysroot路径：
    ```bash
    # 查看交叉编译器的默认sysroot路径（以Linaro工具链为例）
    arm-linux-gnueabihf-gcc --print-sysroot
    # 预期输出：/opt/arm-linux-gnueabihf/sysroot（工具链内置路径）
    ```

2.  方式1：--sysroot参数（临时配置，项目级生效）
    通过编译命令后加`--sysroot=路径`指定sysroot，仅对当前编译命令生效，优点是“灵活适配不同项目的sysroot需求”（如项目A用内置sysroot，项目B用自定义sysroot），是开发中最常用的方式。
    - 实操步骤（以编译uart_demo.c为例）：
      ```bash
      # 1. 进入项目目录（单文件项目，复用前序源码）
      cd ~/cross_demo/single_file/
      # 2. 用--sysroot指定工具链内置sysroot，编译程序
      arm-linux-gnueabihf-gcc uart_demo.c -o uart_demo_sysroot1 \
        --sysroot=/opt/arm-linux-gnueabihf/sysroot \  # 临时指定sysroot
        -march=armv7-a -static -Wall
      # 3. 验证：用readelf查看库依赖是否来自sysroot（静态链接无动态依赖）
      arm-linux-gnueabihf-readelf -d uart_demo_sysroot1 | grep NEEDED
      # 预期输出：无任何输出（静态链接已嵌入库，依赖来自sysroot）
      ```
    - 适用场景：多项目开发（不同项目用不同sysroot）、临时验证自定义sysroot兼容性。
    - 避坑点：路径必须写绝对路径（相对路径易因当前目录变化导致失效），如误写`--sysroot=sysroot`会报错“cannot find sysroot directory”。<br>

### <strong>头文件（-I）与库文件（-L）的强制指定（解决“找不到头文件/库”问题）</strong>

交叉编译中“找不到头文件（fatal error: xxx.h: No such file or directory）”“库依赖缺失（undefined reference to xxx）”是新手最高频报错，核心原因是“编译器未搜索到自定义/第三方依赖的路径”。

通过`-I`（指定头文件路径）和`-L`（指定库文件路径）可强制指定搜索范围，精准定位依赖。

1.  核心逻辑：搜索路径的“强制追加”与优先级
    交叉编译器默认仅搜索“工具链内置路径”和“sysroot系统路径”，第三方库（如自定义驱动库、开源库libcurl）的路径不在默认范围，需通过`-I`和`-L`强制追加。
    追加后路径优先级为：-I指定路径 > 内置路径 > sysroot系统路径（头文件）；-L指定路径 > 内置路径 > sysroot系统路径（库文件）——确保优先使用项目专属依赖。

### <strong>静态链接（-static）与动态链接的选型场景（嵌入式量产优先静态链接原因）</strong>

交叉编译的“链接方式选型”直接影响程序的部署难度、运行稳定性和体积
——静态链接（-static）将库嵌入程序，动态链接（默认）仅记录库依赖。嵌入式场景因“存储有限、部署环境复杂、追求稳定性”，量产时优先选静态链接，但需明确两者的核心差异及选型依据。

1.  静态链接与动态链接的核心差异（实操对比）
    以`uart_demo.c`为例，分别用两种链接方式编译，通过“体积、依赖、部署难度”三个维度对比差异：
    | 对比维度       | 静态链接（-static）                          | 动态链接（默认，无参数）                    |
    |----------------|----------------------------------------------|---------------------------------------------|
    | 编译命令       | arm-linux-gnueabihf-gcc ... -static          | arm-linux-gnueabihf-gcc ...                 |
    | 程序体积       | 较大（嵌入库文件）                           | 较小（仅记录库依赖）                        |
    | 动态依赖       | 无（用readelf查看无NEEDED项）                | 依赖目标机库（如libc.so.6）                 |
    | 部署难度       | 简单（仅拷贝程序即可）                       | 复杂（需同步拷贝依赖的.so库到目标机）       |
    | 运行稳定性     | 高（不依赖目标机库版本）                     | 低（目标机库版本不匹配会报错）              |

    实操验证：
    ```bash
    # 1. 静态链接编译
    arm-linux-gnueabihf-gcc uart_demo.c -o uart_static -static -march=armv7-a
    # 2. 动态链接编译
    arm-linux-gnueabihf-gcc uart_demo.c -o uart_dynamic -march=armv7-a --sysroot=xxx
    # 3. 对比体积（静态体积大很多）
    ls -lh uart_static uart_dynamic
    # 预期输出：
    # -rwxr-xr-x 1 user user 880K 10月  6 15:30 uart_static  # 较大
    # -rwxr-xr-x 1 user user  16K 10月  6 15:31 uart_dynamic # 较小
    # 4. 对比动态依赖（静态无依赖，动态依赖libc）
    arm-linux-gnueabihf-readelf -d uart_static | grep NEEDED  # 无输出
    arm-linux-gnueabihf-readelf -d uart_dynamic | grep NEEDED  # 输出：libc.so.6
    ```

2.  嵌入式场景的选型依据（量产优先静态链接的3个核心原因）
    嵌入式设备（如工业控制器、智能家居终端）与PC的最大差异是“部署环境不可控”（目标机可能缺库、库版本老旧），因此静态链接成为量产首选，核心原因如下：
    - 原因1：避免“目标机缺库”导致运行失败
      动态链接的程序部署到目标机时，若目标机缺少`libc.so.6`或版本过低（如程序依赖2.27版本，目标机是2.23），会报错“version `GLIBC_2.27' not found”。静态链接将libc等库嵌入程序，彻底摆脱对目标机库的依赖。
    - 原因2：简化量产部署流程
      量产时需给成百上千台设备烧写程序，静态链接仅需拷贝一个可执行文件，动态链接需同步拷贝多个.so库（还需注意路径），极易因漏拷库导致批量部署失败。
    - 原因3：提升运行稳定性（嵌入式核心诉求）
      嵌入式设备通常7x24小时运行，动态链接若因库文件被误删、升级导致版本不兼容，会直接引发程序崩溃；静态链接无此风险，程序运行仅依赖自身，稳定性更高。

3.  动态链接的适用场景（何时不选静态链接）
    静态链接并非万能，以下场景优先选动态链接：
    - 场景1：多个程序共享同一库（节省存储）
      若目标机运行10个程序都依赖`libcurl.so`，动态链接仅需存储一份库文件（约500KB），静态链接每个程序都嵌入一份（总5MB），对存储极小的设备（如128KB Flash）不友好。
    - 场景2：库文件需要独立升级
      若库文件需频繁修复漏洞（如加密库），动态链接仅需替换.so库即可，静态链接需重新编译所有依赖该库的程序并重新部署。
    - 场景3：程序体积严格受限（如物联网传感器）
      部分极简设备（如蓝牙传感器）存储仅几KB，动态链接的小体积优势凸显，此时需确保目标机预烧录所需库文件。

4.  新手避坑：静态链接的2个关键注意点
    - 避坑1：静态链接需工具链支持静态库（部分精简工具链仅含动态库），若编译报错“cannot find -lc”，需安装工具链的静态库包（如Linaro的`libc6-dev-armhf-cross`）；
    - 避坑2：静态链接程序体积较大，可结合`strip`命令精简（如前序章节所述），体积可减小50%以上，兼顾稳定性和体积。<br>

### <strong>交叉编译的“编译→部署→运行”全流程中，新手最易卡壳在“编译成功但运行失败”“链接阶段报错”等场景。

### <strong>架构不匹配：“cannot execute binary file”的3步排查法</strong>

1.  报错现象与核心根因
    - 典型报错日志（ARM开发板运行x86程序）：
      ```bash
      root@arm-board:~# ./hello_x86
      -bash: ./hello_x86: cannot execute binary file: Exec format error
      ```
    - 3种常见根因（按出现频率排序）：
      1.  程序架构与目标机架构大类不匹配（如x86程序部署到ARM、32位ARM程序部署到64位ARM）；
      2.  指令集版本不兼容（如ARMv8程序部署到ARMv7开发板，启用了目标机不支持的NEON指令）；
      3.  ABI不匹配（如HF硬浮点程序部署到无FPU的SF软浮点目标机）。

2.  3步标准排查法（流程图+命令）

3.  实操案例1：程序架构与目标机大类不匹配（x86→ARM）
    - 复现错误：用本地gcc编译x86程序，部署到ARM开发板：
      ```bash
      # 宿主端：用本地gcc编译x86架构程序
      gcc hello.c -o hello_x86
      # 部署到ARM开发板
      scp hello_x86 root@192.168.1.100:/root/
      # 目标机运行报错：cannot execute binary file
      ```
    - 排查过程：
      ```bash
      # Step1：宿主端查程序架构（确认是x86）
      file hello_x86
      # 输出：hello_x86: ELF 64-bit LSB executable, x86-64, version 1 (SYSV)...
      # Step2：目标机查架构（确认是ARM）
      ssh root@192.168.1.100 "cat /proc/cpuinfo | grep 'CPU architecture'"
      # 输出：CPU architecture: 7（ARMv7架构）
      # Step3：对比差异：x86≠ARM，根因明确
      ```
    - 解决方法：用ARM交叉编译器重新编译：
      ```bash
      arm-linux-gnueabihf-gcc hello.c -o hello_arm -march=armv7-a -static
      scp hello_arm root@192.168.1.100:/root/
      ssh root@192.168.1.100 "./hello_arm"  # 运行成功
      ```

4.  实操案例2：指令集版本不匹配（ARMv8→ARMv7）
    - 复现错误：用`-march=armv8-a`编译ARMv8程序，部署到ARMv7开发板：
      ```bash
      # 宿主端：编译ARMv8指令集程序
      arm-linux-gnueabihf-gcc hello.c -o hello_armv8 -march=armv8-a -static
      # 目标机运行报错：cannot execute binary file
      ```
    - 排查过程：
      ```bash
      # Step1：宿主端查程序指令集（ARMv8）
      arm-linux-gnueabihf-readelf -A hello_armv8 | grep "Tag_CPU_arch"
      # 输出：Tag_CPU_arch: v8（ARMv8架构）
      # Step2：目标机查支持的指令集（ARMv7）
      ssh root@192.168.1.100 "cat /proc/cpuinfo | grep 'Features'"
      # 输出：Features : swp half thumb fastmult vfp edsp thumbee neon vfpv3...（无ARMv8指令）
      ```
    - 解决方法：修改`-march`参数为`armv7-a`，适配目标机指令集：
      ```bash
      arm-linux-gnueabihf-gcc hello.c -o hello_armv7 -march=armv7-a -static
      ssh root@192.168.1.100 "./hello_armv7"  # 运行成功
      ```

5.  避坑点：
用`file`命令查看程序架构时，需注意“32位/64位”差异（如ARMv8分32位AArch32和64位AArch64），64位程序无法在32位目标机运行<br>

### <strong>库缺失：“undefined reference to”的库依赖追溯（ldd交叉版本用法）</strong>

“undefined reference to `xxx'”是链接阶段的高频报错，核心是“编译器找不到函数`xxx`的实现”
——要么是没链接对应的库（漏加`-l`参数），要么是库路径不对（漏加`-L`参数），要么是库的架构/版本不兼容。这类错误的排查关键是“先定位函数属于哪个库，再确认库是否存在、路径是否正确、架构是否匹配”。

1.  报错现象与核心根因
    - 典型报错日志（链接时漏加libjson-c库）：
      ```bash
      /tmp/cc123xyz.o: In function `main':
      json_demo.c:(.text+0x20): undefined reference to `json_object_new_object'
      json_demo.c:(.text+0x34): undefined reference to `json_object_object_add'
      collect2: error: ld returned 1 exit status
      ```
    - 3种常见根因（按出现频率排序）：
      1.  漏加库链接参数（如依赖libjson-c却没加`-ljson-c`）；
      2.  库文件路径不在编译器搜索范围（漏加`-L`参数，或路径错误）；
      3.  库文件架构不兼容（如链接了x86的libjson-c，而非ARM版本）。

2.  3步标准排查法（流程图+命令）

3.  实操案例：漏加库链接参数+库路径未指定
    - 复现错误：编译依赖libjson-c的程序，未加`-ljson-c`和`-L`参数：
      ```bash
      # 宿主端：编译json_demo.c，漏加库参数
      arm-linux-gnueabihf-gcc json_demo.c -o json_demo -march=armv7-a -Wall
      # 报错：undefined reference to `json_object_new_object'
      ```
    - 排查过程：
      ```bash
      # Step1：定位函数所属库（搜索引擎查得：json_object_new_object属于libjson-c）
      # Step2：确认ARM版本的libjson-c是否存在（假设路径：/home/user/arm_libs/libjson-c/lib）
      ls /home/user/arm_libs/libjson-c/lib/libjson-c.so
      # 输出：/home/user/arm_libs/libjson-c/lib/libjson-c.so（库存在）
      # Step3：验证库架构是否匹配（必须是ARM架构）
      file /home/user/arm_libs/libjson-c/lib/libjson-c.so
      # 输出：ELF 32-bit LSB shared object, ARM, EABI5 version 1 (SYSV)...（架构匹配）
      ```
    - 解决方法：编译时加`-L`（库路径）和`-l`（库名）参数：
      ```bash
      arm-linux-gnueabihf-gcc json_demo.c -o json_demo \
        -L/home/user/arm_libs/libjson-c/lib  # 指定库路径
        -ljson-c                             # 链接libjson-c库
        -march=armv7-a -Wall
      # 编译成功，部署到目标机运行正常
      ```

4.  关键工具：交叉版本ldd的用法（追溯动态库依赖）
    若程序是动态链接，可通过交叉编译器配套的ldd工具（如`arm-linux-gnueabihf-ldd`）查看程序依赖的库是否缺失，避免目标机运行时因缺库报错：
    ```bash
    # 宿主端：用交叉ldd查看动态库依赖
    arm-linux-gnueabihf-ldd json_demo
    # 预期输出（无“not found”说明依赖完整）：
    #  libjson-c.so.5 => /home/user/arm_libs/libjson-c/lib/libjson-c.so.5 (0x400e0000)
    #  libc.so.6 => /opt/arm-linux-gnueabihf/sysroot/usr/lib/libc.so.6 (0x40120000)
    # 若输出“libjson-c.so.5 => not found”，说明库路径未正确配置
    ```
    避坑点：不可用本地`ldd`（x86版本）查看ARM程序，会报错“not a dynamic executable”，必须用交叉工具链的ldd。<br>

### <strong>权限问题：目标机运行“Permission denied”的权限配置与chmod用法</strong>

“Permission denied”是部署后运行时的常见错误，核心是“目标机对程序文件/目录无对应权限”
——与交叉编译本身无关，但新手常误以为是编译问题，浪费大量时间。
这类错误的排查关键是“先查文件权限，再查目录权限，最后查挂载状态”。

1.  报错现象与核心根因
    - 典型报错日志：
      ```bash
      root@arm-board:~# ./hello_arm
      -bash: ./hello_arm: Permission denied
      ```
    - 3种常见根因（按出现频率排序）：
      1.  程序文件无“执行权限”（最常见，上传后未添加x权限）；
      2.  程序所在目录无“读/执行权限”（如部署到`/tmp`目录，目录权限为`r--`）；
      3.  目标机文件系统挂载为“只读”（如SD卡挂载时加了`ro`参数，无法执行文件）。

2.  3步标准排查法（流程图+命令）

3.  实操案例1：程序文件无执行权限（最常见）
    - 复现错误：上传程序后未加执行权限，直接运行：
      ```bash
      # 宿主端：编译ARM程序并上传
      arm-linux-gnueabihf-gcc hello.c -o hello_arm -march=armv7-a -static
      scp hello_arm root@192.168.1.100:/root/
      # 目标机运行报错：Permission denied
      ```
    - 排查与解决：
      ```bash
      # Step1：目标机查程序权限（无x权限，权限位为-rw-r--r--）
      root@arm-board:~# ls -l hello_arm
      # 输出：-rw-r--r-- 1 root root 880000 Oct  7 10:30 hello_arm
      # Step2：添加执行权限（chmod +x）
      root@arm-board:~# chmod +x hello_arm
      # Step3：验证权限（已添加x权限，权限位为-rwxr-xr-x）
      root@arm-board:~# ls -l hello_arm
      # 输出：-rwxr-xr-x 1 root root 880000 Oct  7 10:30 hello_arm
      # 运行成功
      root@arm-board:~# ./hello_arm
      # 输出：Hello Cross Compile!
      ```

4.  实操案例2：文件系统挂载为只读
    - 复现错误：程序部署到SD卡挂载目录（挂载为只读）：
      ```bash
      # 目标机：查看挂载状态（/dev/mmcblk0p2挂载为ro）
      root@arm-board:~# mount | grep /mnt/sdcard
      # 输出：/dev/mmcblk0p2 on /mnt/sdcard type ext4 (ro,relatime)
      # 部署程序到该目录并运行
      scp hello_arm root@192.168.1.100:/mnt/sdcard/
      ssh root@192.168.1.100 "chmod +x /mnt/sdcard/hello_arm && /mnt/sdcard/hello_arm"
      # 报错：Permission denied
      ```
    - 排查与解决：
      ```bash
      # Step1：确认挂载为只读（ro），根因明确
      # Step2：重新挂载为可读写（rw）
      root@arm-board:~# umount /mnt/sdcard
      root@arm-board:~# mount -o rw /dev/mmcblk0p2 /mnt/sdcard
      # Step3：再次运行程序（成功）
      root@arm-board:~# /mnt/sdcard/hello_arm
      # 输出：Hello Cross Compile!
      # （可选）永久设置为rw：修改/etc/fstab，将ro改为rw
      ```

5.  chmod权限补充说明（嵌入式常用）
    嵌入式场景中，程序权限常用`chmod 755`（而非仅`+x`），确保“所有者可读写执行，其他用户可读执行”，避免因权限不足导致的其他问题：
    ```bash
    # 嵌入式常用：添加全用户可执行权限（更稳妥）
    chmod 755 hello_arm
    # 权限位解析：7=4(r)+2(w)+1(x)，5=4(r)+1(x)，即-rwxr-xr-x
    ```<br>
