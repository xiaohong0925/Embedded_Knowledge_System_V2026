## <span class="blue">本节导读

MMU是怎么找到物理地址的？想象你在图书馆找一本书——页表就是索引卡片，TLB就是放在你口袋里的最近借阅记录。<br>
本节带你深入Linux地址转换的完整路径：从虚拟地址出发，逐级穿越4级页表的迷雾，最终抵达物理页帧；同时揭示TLB这个"高速缓存"如何拯救每一次地址转换的性能，以及多核时代TLB shootdown带来的代价。学完后，你将理解为什么TLB miss是性能杀手，为什么嵌入式系统频繁调用vmalloc会变慢。

## <span class="blue">VA→PA的地址转换与TLB [I][M]

### 图书馆的索引系统

你走进图书馆找《Linux设备驱动程序》第三版。最原始的方法是从头到尾扫描书架——这相当于没有MMU的CPU直接访问物理地址，慢得不可接受。于是图书馆引入了索引卡片系统（页表），按书名、作者、分类号层层索引。但你每次借书都要走一遍完整索引流程？太浪费时间。聪明的读者会把最近借的书的**精确位置**记在小纸条上塞进口袋——下次再借，直接掏口袋就行。这张小纸条，就是TLB。

### 4级页表Walk：逐级查表之旅

当一个虚拟地址（VA）需要转换为物理地址（PA）时，MMU启动一次**页表遍历（Page Table Walk）**，流程如下：

**第一步：确定页表基址**

MMU从TTBR寄存器获取PGD（Page Global Directory）的物理基地址。用户空间地址使用TTBR0_EL1，内核空间地址使用TTBR1_EL1。这是整个Walk的起点——就像拿到图书馆主索引柜的钥匙。

**第二步：逐级索引**

MMU从虚拟地址中提取各级索引，像剥洋葱一样层层深入：

| Walk阶段 | 操作 | 从VA提取的索引位 | 输出 |
|----------|------|-----------------|------|
| PGD级 | TTBR + pgd_index → 读PGD条目 | [47:39]，9位 | PUD基地址 |
| PUD级 | PUD基 + pud_index → 读PUD条目 | [38:30]，9位 | PMD基地址 |
| PMD级 | PMD基 + pmd_index → 读PMD条目 | [29:21]，9位 | PTE基地址 |
| PTE级 | PTE基 + pte_index → 读PTE条目 | [20:12]，9位 | 物理页帧号PFN |
| 最终拼接 | PFN << 12 + VA[11:0] | [11:0]，12位 | 物理地址PA |

每一级读取都是一次**物理内存访问**。走完4级，MMU至少进行了4次内存读取。在DDR4-3200上，每次访问约100ns，累计400ns以上。对一条指令来说，这慢得无法忍受——而现代CPU每纳秒能执行数条指令。

**第三步：得到物理地址**

PTE中存放着物理页帧号（PFN）。MMU将其左移12位（乘以4KB），再与虚拟地址的低12位（页内偏移）拼接，得到最终的物理地址。CPU拿着这个地址去访问DDR，取回指令或数据。

### TLB：地址转换的加速器

Translation Lookaside Buffer（TLB）是集成在MMU内部的**全相联高速缓存**，专门缓存"虚拟页号 → 物理页框号"的映射关系。它的核心使命很单纯：避免每次地址转换都走一遍4级页表Walk。

TLB的工作原理朴素而高效：

- **TLB查找**：MMU拿到虚拟地址后，先查TLB。用虚拟页号（VA[47:12]）作为Key去匹配。
- **TLB Hit**：找到了对应条目，直接取出PFN，与页内偏移拼接成PA。整个过程在CPU流水线内完成，**延迟仅数个时钟周期**（纳秒级）。
- **TLB Miss**：没找到对应条目，MMU无奈启动4级页表Walk，从内存中逐层读取。Walk完成后，将新映射**填入TLB**（替换某个旧条目），供下次使用。

> 💡 **关键洞察**：TLB的容量极小——ARM64的典型实现有32~64个条目。但它利用了程序访问的**时间局部性**：最近访问过的页面很可能再次被访问。这32个条目覆盖的地址空间（32×4KB = 128KB）虽小，却能拦截绝大多数地址转换请求。

### TLB Hit vs TLB Miss：数十倍的性能鸿沟

两者的性能差距是数量级的。以一个3GHz的ARM64 CPU为例：

| 场景 | 耗时 | 换算 | 对程序的影响 |
|------|------|------|-------------|
| TLB Hit | ~3-5 时钟周期 | ~1-2 ns | 几乎无感，流水线顺畅 |
| TLB Miss + Walk | 400+ ns | ~1200+ 周期 | 流水线 stall，性能暴跌 |
| TLB Miss + Page Fault | 5-10 μs | 数百万周期 | 进入内核态，可能swap |

一次TLB Miss的代价是一次TLB Hit的**数百倍**。更糟的是，TLB Miss会触发流水线停顿（Pipeline Stall），后续指令无法发射，CPU执行单元空转。若Miss频繁发生，CPU的有效性能可能暴跌至峰值的几分之一。

> ⚠️ **陷阱**：TLB Miss容易被误测为"CPU性能不足"。`perf`工具中，`dTLB-load-misses`和`iTLB-load-misses`事件是识别TLB瓶颈的关键指标。

## <span class="blue">TLB Shootdown：多核时代的隐形代价 [E]

### 场景引入：一个CPU改了页表，其他CPU怎么办

你运行着一个8核ARM服务器，进程A在用户空间执行。CPU 0执行了`munmap()`，将某段虚拟地址解除映射。CPU 0更新了进程的页表——PTE被清零。但CPU 1~7的TLB里，可能还缓存着这段虚拟地址的旧映射！如果CPU 1继续用旧TLB条目访问该地址，就会访问到**已经释放或重新分配的物理页**，引发数据损坏或安全漏洞。

这个问题叫**TLB不一致**。解决它的机制叫**TLB Shootdown**。

### Shootdown的完整流程

当一个CPU（发起者）修改了页表后，需要确保所有CPU的TLB与最新页表保持一致。流程如下：

**第一步：发起IPI（Inter-Processor Interrupt）**

发起者向持有该进程上下文的所有CPU广播IPI。在Linux内核中，`smp_call_function_many()`或`on_each_cpu_mask()`负责投递这个消息。

**第二步：目标CPU收到中断**

收到IPI的CPU从当前工作上下文中被中断。它们需要停下手中的活，处理TLB刷新请求。

**第三步：执行本地TLB Flush**

每个目标CPU调用`local_flush_tlb_mm_range()`或类似的本地刷新函数，使指定地址范围的TLB条目失效。ARM64上对应`tlbi vmalle1is`或更细粒度的`tlbi vae1is, x0`指令。

**第四步：同步确认**

目标CPU完成Flush后，通过内存屏障（`dsb ish`）和同步原语通知发起者。发起者等待所有CPU确认后，才认为Shootdown完成。

整个Shootdown的时序：

```
CPU 0 (发起者)          CPU 1~N (目标者)
    |                         |
更新页表(PTE=0)              |
    |                         |
发送IPI -------------------->|
    |                   收到IPI,中断
    |                   执行tlbi指令
    |                   写完成标志
等待所有确认 <----------------|
    |
继续执行
```

### Shootdown的性能代价：为什么vmalloc变慢了

`vmalloc()`在内核空间分配虚拟连续但物理不连续的内存。每次`vmalloc()`都需要分配PTE并建立映射，`vfree()`则需要解除映射。这两个操作都会修改页表，触发TLB Shootdown。

问题出在**频率**和**范围**：

- `kmalloc()`分配的内存通常在直接映射区（线性映射），页表建立后基本不变，极少触发Shootdown。
- `vmalloc()`每分配/释放一次，页表就被修改，Shootdown就跟着来一次。8核服务器上，一次Shootdown涉及7次IPI投递、7次中断响应、7次TLB指令执行，总耗时可达**数十到数百微秒**。

频繁调用`vmalloc()`/`vfree()`的系统，CPU们不是在干活，而是在互相发中断、刷TLB。实测数据显示，某些驱动模块把大块缓冲区用`vmalloc()`管理，在高并发的I/O路径上，TLB Shootdown开销可占总CPU时间的**5%~15%**。

> 🔴 **危险**：在嵌入式系统上，这个问题更尖锐。嵌入式SoC通常核数不多（2~4核），但CPU频率低、中断响应慢。一次Shootdown占用的相对时间比例反而更高。更要命的是，嵌入式系统的驱动（camera、codec、DMA控制器）常需要频繁分配/释放大块缓冲区——开发者顺手就用`vmalloc()`，却不知道性能正在悄悄流失。

> ⚠️ **陷阱**：TLB Shootdown在内核态异步发生，用户很难直接感知。通过`perf stat -e tlb:tlb_flush`或查看`/proc/interrupts`中IPI计数，可以间接发现Shootdown是否过于频繁。

### 怎么避免Shootdown带来的性能损耗

最根本的策略：**减少不必要的页表修改**。

- 用`kmalloc()`+ scatterlist替代`vmalloc()`，只要物理地址不连续是可以接受的（DMA有IOMMU支持时）。
- 预分配内存池，避免频繁`vmalloc()`/`vfree()`。内核的`genalloc`子系统就是为这个场景设计的。
- 对于必须`vmalloc()`的场景，尽量减少分配粒度，延长缓冲区生命周期。

> 💡 **内核视角**：Linux内核在`mm/tlb.c`中实现了Shootdown的优化。`tlb_gather_mmu()`接口允许批量累积多个页表修改，一次性发起Shootdown，减少IPI次数。但根本问题仍在于页表修改的频率——这是设计层面需要控制的。

## <span class="blue">本节总结

| 概念 | 要点 | 操作/关联 |
|------|------|----------|
| 页表Walk | TTBR→PGD→PUD→PMD→PTE，逐级读取 | 4次内存访问，每次~100ns |
| TLB | 缓存VA→PA映射的高速缓冲 | Hit时纳秒级，Miss时走Walk |
| TLB Miss | TLB未命中，触发页表Walk | 性能损失可达数百倍 |
| TLB Shootdown | 多核间同步TLB更新的机制 | 修改页表后广播IPI，其他核flush |
| vmalloc代价 | 频繁修改页表→频繁Shootdown | 嵌入式系统尤为敏感 |

TLB是现代内存管理的第一道防线，它把地址转换从"每次4次访存"压缩到"几乎零开销"。但当页表变动（尤其是多核环境下）时，Shootdown的隐性代价可能吃掉这份红利。理解这个权衡，是做出正确内存分配决策的基础。
