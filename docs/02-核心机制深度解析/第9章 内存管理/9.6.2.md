OOM killer的选择逻辑可能跟你想象的不一样——它不是杀"最不重要的"进程，而是杀"释放最多内存"的进程。这意味着你的关键服务可能最先被杀！我见过太多这样的惨案：一个MySQL实例跑了几十GB内存，业务完全依赖它，结果OOM一来，MySQL第一个倒——因为它"贡献"的内存最多。反过来说，一个无关紧要的守护进程只占了几MB，OOM killer连看都懒得看它。

这个看似反直觉的逻辑，源自内核里一个叫做`badness()`的评分函数。早年间（2.6时代），这个函数就叫`badness()`，后来代码重构换了个名字，但核心思路一直没变：给每个进程打分，分最高的被选中。分数怎么算？有一套相对固定的加权公式。

**知识点55 [I][M]**

`oom_score`的计算内核源码位于`mm/oom_kill.c`，最终通过`proc`文件系统暴露给用户空间。影响分数的因素有好几个，我把主要的列出来：

| 因素 | 影响方向 | 说明 |
|------|----------|------|
| RSS（常驻内存） | 分数↑ | 物理页框占用，**权重最大**。RSS越大，分数越高 |
| 进程拥有的子进程数 | 分数↑ | 杀一个父进程连带子进程都消失，"性价比"高 |
| nice值 | 分数↑ | nice>0（低优先级）的进程分数会稍增 |
| 进程运行时间 | 分数↓ | 运行越久分数越低，老进程受保护 |
| oom_score_adj | 可正可负 | 用户空间的调节杠杆，范围-1000~+1000 |

用大白话说，公式核心就是：**RSS总量 × 进程数量因子 × 优先级因子 ÷ 运行时间因子 + oom_score_adj**。

`oom_score_adj`这个东西非常关键，它是用户空间唯一能直接干预OOM killer选择的手段。取值范围是-1000到+1000：

- 默认值0——不干预，全凭内核判断
- -1000——**永远不被选中**。你的关键进程设成这个，OOM killer直接跳过它
- +1000——**最先被杀**。即便它RSS不大，也优先选它当替罪羊

> **陷阱**：`oom_score_adj`需要root权限才能写到负值，普通用户只能写正值。这是安全设计——否则普通用户把自己的进程设为-1000，就能合法地"挤死"其他用户的进程。

```bash
# 查看当前进程的oom_score和oom_score_adj
cat /proc/self/oom_score
cat /proc/self/oom_score_adj

# 保护关键进程（需要root）
echo -1000 > /proc/$(pidof mysqld)/oom_score_adj

# 主动标记可牺牲的进程
echo 1000 > /proc/$(pidof backup_job)/oom_score_adj
```

这里有个认知上的陷阱要特别提醒。很多人以为OOM killer是在做"重要性排序"——先杀不重要的，保留重要的。错。它的设计目标是**在最短时间内释放最多内存，同时尽可能少地杀死进程**。这是一个工程优化问题，不是业务重要性判断。你的MySQL很重要？抱歉，它占的内存太多了，杀了它系统就能活，那就杀它。

历史上曾经有过`oom_kill_allocating_task`这个参数。开启它之后，谁触发OOM就杀谁，简单粗暴。这个策略在某些场景下反而更"公平"——至少不会因为RSS大就被株连。但它在通用场景下表现不好，因为触发OOM的进程未必是罪魁祸首，可能只是一个无辜的小进程碰巧申请了内存。所以现在默认策略仍然是`badness()`评分机制。

```
+-----------------------------------+
|        badness() 评分流程          |
+-----------------------------------+
| 1. 遍历所有 eligible 进程          |
| 2. 计算每个进程的 oom_score        |
|    ├── RSS × 基础权重              |
|    ├── × 进程数量因子（含子进程）   |
|    ├── × 优先级因子（nice值）       |
|    ├── ÷ 运行时间衰减              |
|    └── + oom_score_adj 用户调整    |
| 3. 选出 oom_score 最高的进程       |
| 4. 执行 kill，如无效果依次往下选    |
+-----------------------------------+
```

所以真正靠谱的线上策略是什么？两点：第一，给关键业务进程设置`oom_score_adj = -1000`，这是硬保护；第二，更重要的事情在前面——监控和告警要在OOM发生前把问题暴露出来，OOM killer永远是最后一道防线，不是常规手段。

**知识点56 [I]**

理解`oom_score_adj`最好的方式就是亲手试。三个命令，三种命运：

```bash
# 终端1：启动一个内存消耗者
$ stress --vm 1 --vm-bytes 1G --vm-hang 0 &
$ PID=$!

# 查看默认值下的评分
cat /proc/$PID/oom_score        # 一个中等偏高的数字
cat /proc/$PID/oom_score_adj    # 0

# 让它变成"金刚不坏之身"
echo -1000 > /proc/$PID/oom_score_adj
cat /proc/$PID/oom_score        # 变成 0，永远垫底

# 或者让它成为"头号替罪羊"
echo 1000 > /proc/$PID/oom_score_adj
cat /proc/$PID/oom_score        # 数字飙升，OOM时第一个被杀
```

你可以开两个stress进程，一个设-1000，一个设+1000，然后制造OOM场景（比如用`stress --vm 4 --vm-bytes 2G`把内存压爆），观察哪个先死。结果 predictable 得很：+1000的那个稳死，-1000的那个大概率活到最后。这个小实验值得做一次，亲眼看过结果，你对OOM killer的理解会上一个台阶。
