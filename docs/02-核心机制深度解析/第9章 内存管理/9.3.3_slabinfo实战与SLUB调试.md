# 9.3.3 slabinfo 实战与 SLUB 调试

> slabinfo 是内存泄漏的"照妖镜"——某个 cache 的 `num_objs` 持续增长不回落，就像水缸里的水位不断上涨，迟早会溢出来。

在 Linux 内核中，SLUB allocator 负责管理小块内存的分配与回收。当驱动或内核模块存在内存泄漏时，泄漏的内存往往以 slab 对象的形式积存在对应的 kmem_cache 中。`/proc/slabinfo` 提供了观察这些 cache 状态的窗口，而 `CONFIG_SLUB_DEBUG` 则为深入诊断提供了利器。本节从实战监控脚本出发，结合真实案例，讲解如何利用 slabinfo 定位 SLUB 泄漏，并介绍 SLUB 调试机制的原理与开销。

---

## 知识点 119 [I][M] /proc/slabinfo 关键列解读与泄漏监控实战

### 119.1 /proc/slabinfo 关键列解析

`/proc/slabinfo` 是内核导出的 SLUB/SLAB allocator 统计信息接口。在 SLUB 模式下，每一行代表一个 kmem_cache，关键列的含义如下（以 `slabinfo -l` 或 `cat /proc/slabinfo` 输出为例）：

```text
# name            <active_objs> <num_objs> <objsize> <objperslab> <pagesperslab> ...
kmalloc-256       12345         12800      256       16           1               ...
kmalloc-512       4096          4096       512       8            1               ...
dentry            524288        524288     192       21           1               ...
```

| 列名 | 含义 | 泄漏诊断价值 |
|------|------|-------------|
| `active_objs` | 当前被分配出去、正在使用的对象数量 | 反映实际业务负载 |
| `num_objs` | cache 中总对象数（包括已分配和空闲的） | **核心指标**，持续增长 = 泄漏信号 |
| `objsize` | 单个对象的大小（字节） | 用于估算泄漏总量 |

**关键洞察**：当 `num_objs` 在一段时间内单调递增且 `active_objs ≈ num_objs`（即空闲对象极少），说明分配出去的对象几乎没有被回收，这是典型的内存泄漏特征。相反，如果 `num_objs` 增长但 `active_objs` 保持稳定或回落，则可能只是 cache 的正常预热或波动。

### 119.2 泄漏特征识别

在生产环境中，识别 SLUB 泄漏可遵循以下判别准则：

1. **持续增长性**：连续采样（如每 10 分钟一次）发现某个 cache 的 `num_objs` 单调递增，且增长斜率不随业务低谷期而收敛。
2. **高活跃度**：`active_objs / num_objs > 0.9`，说明 cache 中几乎无空闲对象，分配器不断向 buddy system 申请新的 page 来扩充该 cache。
3. **objsize 匹配**：泄漏 cache 的 `objsize` 应与代码中 `kmalloc(size, GFP_*)` 的 size 对应。例如 `kmalloc-256` 对应 `kmalloc(128~256)` 的分配请求。
4. **总量估算**：泄漏速率 ≈ Δ`num_objs` × `objsize` / 采样间隔。若每天泄漏几十 MB，一周即可达到数 GB，最终触发 OOM。

### 119.3 实战：slabinfo 监控脚本

下面是一个可用于生产环境的轻量级监控脚本，定期记录 slabinfo 并检测异常增长：

```bash
#!/bin/bash
# slab_monitor.sh - 监控 /proc/slabinfo 检测 SLUB 泄漏
# 用法: ./slab_monitor.sh [采样间隔秒数] [增长阈值百分比]

INTERVAL=${1:-600}          # 默认 10 分钟采样一次
THRESHOLD=${2:-20}          # num_objs 增长超过 20% 触发告警
LOG_DIR="/var/log/slabmon"
mkdir -p "$LOG_DIR"

while true; do
    TIMESTAMP=$(date '+%Y%m%d_%H%M%S')
    SNAPSHOT="$LOG_DIR/slabinfo_$TIMESTAMP"
    cp /proc/slabinfo "$SNAPSHOT"

    if [[ -f "$LOG_DIR/slabinfo_prev" ]]; then
        awk -v thresh="$THRESHOLD" '
        # 读取上一轮数据
        NR==FNR && !/^#/ { prev[$1]=$3; next }
        # 对比当前数据
        !/^#/ {
            if ($1 in prev && prev[$1] > 0) {
                growth = ($3 - prev[$1]) * 100 / prev[$1]
                if (growth > thresh) {
                    printf "[WARN] %s: num_objs %.0f%% growth: %d -> %d "\
                           "(objsize=%d, active=%d/%d)\n",
                           $1, growth, prev[$1], $3, $4, $2, $3
                }
            }
        }
        ' "$LOG_DIR/slabinfo_prev" "$SNAPSHOT"
    fi

    ln -sf "$SNAPSHOT" "$LOG_DIR/slabinfo_prev"
    sleep "$INTERVAL"
done
```

**脚本要点**：

- 采用双快照对比机制，通过 `num_objs` 的百分比增长触发告警，避免对正常波动的误报。
- 告警信息中同时输出 `active_objs` 与 `num_objs` 的比值，帮助快速判断是否为真实泄漏。
- 所有快照保存在 `/var/log/slabmon`，事后可用 `diff` 或 `awk` 做离线趋势分析。

**运维建议**：对于内存敏感型业务，建议将脚本接入 Prometheus 或 Zabbix，把 `num_objs` 和 `active_objs` 作为时序指标定期采集。设置两条告警线——**警告线**（1 小时内 `num_objs` 增长 50%）和**紧急线**（`active_objs / num_objs > 0.95` 且内存余量低于 10%）。紧急线触发时应自动保存 `slabinfo` 快照并准备热重启预案，防止 OOM 导致雪崩。

### 119.4 案例：网关设备 kmalloc-256 cache 泄漏

**背景**：某企业网关设备（基于 ARM64、内核 5.10）运行一周后触发 OOM Killer，杀死关键业务进程，导致设备失联。

**排查过程**：

1. **slabinfo 初筛**：登录设备后执行 `cat /proc/slabinfo | sort -k3 -n -r | head`，发现 `kmalloc-256` 的 `num_objs` 高达 1200 万，占用约 3 GB 内存，且 `active_objs` 与 `num_objs` 几乎相等。
2. **趋势确认**：通过上述监控脚本的历史日志回溯，`kmalloc-256` 的 `num_objs` 从上线时的 2 万持续线性增长，每日增量约 40 万对象（≈ 100 MB/天）。
3. **关联代码**：使用 `CONFIG_SLUB_DEBUG` 开启 tracking（见知识点 120），捕获分配栈，发现泄漏对象都来自一个网络驱动的错误处理路径。
4. **根因定位**：该驱动在 RX 路径的异常分支中，将包含指针的结构体直接传入 `kfree`，但该指针实际指向的是栈上临时变量，而非 `kmalloc` 返回的堆对象。由于 `kfree` 对非法指针未做有效校验（fastpath 直接返回），导致对应的 `kmalloc-256` 对象"失踪"——分配计数存在，却无人释放。

**修复方案**：修正错误处理路径，确保只释放由 `kmalloc()` 返回的合法指针；对暂存数据结构使用嵌入式对象而非指针传递。修复后 `kmalloc-256` 的 `num_objs` 稳定在正常水位，设备连续运行 30 天无 OOM。

**经验总结**：slabinfo 监控是发现 SLUB 泄漏的第一道防线。当某个 cache 的 `num_objs` 像水缸里的水位一样持续上涨时，必须立刻关联业务代码，结合 SLUB 调试手段追踪分配栈，否则等待系统的只有 OOM。

## 知识点 120 [E] CONFIG_SLUB_DEBUG 调试机制与性能开销

当 slabinfo 监控发现异常增长的 cache 后，下一步需要定位"是谁在泄漏"。`CONFIG_SLUB_DEBUG` 提供了三种核心调试能力，可在不修改业务代码的情况下捕获分配/释放行为和内存越界访问。

### 120.1 SLUB_POISON（对象毒化）

**原理**：在对象被释放时，SLUB 用特定字节模式（`0x6b` 即 `POISON` 和 `0x5a` 即 `RED_INACTIVE`）填充整个对象区域。如果后续代码错误地访问了已释放的"悬垂指针"，读到的数据将是可识别的毒化模式，便于快速确认 use-after-free。

```text
已释放对象内存布局:
+----------------------------------+
| 0x6b 0x6b 0x6b ...  (POISON)    |  <- 整个对象填充 0x6b
+----------------------------------+
```

**检测场景**：访问已释放对象时数据异常；`kmem_cache_free` 时发现对象内容与毒化模式不符（说明该对象从未被正确分配或在释放后被改写）。

### 120.2 Red Zone（红色保护区域）

**原理**：在每个 slab 对象的前后各插入一段被称为 "red zone" 的保护区域（通常填充 `0xbb` 模式）。如果代码存在越界写（buffer overflow/underflow），会破坏 red zone 的填充模式，SLUB 在下次分配或释放时检测到模式损坏即触发 `BUG()` 或告警。

```text
带 Red Zone 的对象布局:
+----------------+----------+----------------+
|  Red Zone      |  Object  |   Red Zone     |
|  (before)      |          |   (after)      |
|  0xbb...       |  data    |   0xbb...      |
+----------------+----------+----------------+
```

**启用方式**：启动参数添加 `slub_debug=RZ`，或特定 cache 级别开启：

```bash
# 为 kmalloc-256 开启 red zone 和 poison
echo 'slub_debug=RPZ' > /sys/kernel/slab/kmalloc-256/validate
```

### 120.3 Tracking（分配/释放栈追踪）

**原理**：为每个对象额外存储其最近一次分配和释放的调用栈（call stack）。当发现泄漏或内存损坏时，可通过 `slabinfo -v` 或 `/sys/kernel/slab/<name>/alloc_calls` 读取栈信息，直接定位到问题代码行。

```bash
# 查看 kmalloc-256 中尚未释放对象的分配栈
cat /sys/kernel/slab/kmalloc-256/alloc_calls

# 示例输出片段:
  4096 __kmalloc_track_caller+0x234
        my_driver_rx_handler+0x1a8
        netif_receive_skb+0x58
        ...
```

**使用建议**：tracking 对内存的额外消耗最大（每个对象需存储两个栈指针），建议仅在定位特定 cache 时动态开启，而非全局启用。

### 120.4 性能开销与使用策略

| 调试选项 | 内存开销 | 分配速度影响 | 典型场景 |
|----------|----------|-------------|----------|
| `F` (SLUB_POISON) | 低（仅释放时填充） | -10%~-15% | use-after-free 检测 |
| `Z` (Red Zone) | 中（每对象增加 2×redzone） | -15%~-25% | 越界写检测 |
| `T` (Tracking) | 高（每对象存储栈信息） | -30%~-50% | 泄漏源定位 |
| `PZT` (全开) | 高 | -40%~-60% | 深度调试 |

**启用 CONFIG_SLUB_DEBUG 的方法**：内核编译阶段需选中 `Kernel hacking > Memory Debugging > SLUB debugging on by default`。若内核已支持但未默认开启，可通过启动参数动态激活，无需重新编译：

```bash
# GRUB 启动参数示例：对 kmalloc-256 开启全部调试
slub_debug=PZT,kmalloc-256
```

**核心原则**：`CONFIG_SLUB_DEBUG` 及其子选项**只用于调试环境**，绝不应在性能敏感的生产系统中全局开启。实际工作中推荐的做法是：

1. 生产环境仅运行轻量级 slabinfo 监控脚本（知识点 119）。
2. 一旦发现异常，在测试环境复现问题，按需开启特定 debug 选项。
3. 使用启动参数粒度控制，例如 `slub_debug=P,dentry` 仅对 `dentry` cache 开启 poison。
4. 调试完成后务必关闭 debug，恢复正常的分配性能。

**调试技巧**：当泄漏对象的分配栈指向内核通用路径（如 `__kmalloc`）而非具体业务函数时，可结合 `kmemleak`（`CONFIG_DEBUG_KMEMLEAK`）做交叉验证。kmemleak 会扫描内存中未被引用的已分配对象，适合检测完全丢失引用的"孤儿"对象；而 SLUB tracking 更适合分析对象仍在引用但释放逻辑有缺陷的场景。两者配合使用，可覆盖绝大多数内核内存泄漏模式。

通过 slabinfo 的宏观监控与 SLUB_DEBUG 的微观追踪相结合，可以形成一套完整的"发现—定位—修复"内存泄漏工作流。
