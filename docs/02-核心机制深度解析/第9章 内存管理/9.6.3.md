既然OOM killer可能杀掉你的关键进程，那怎么保护它们？

这是个很现实的问题。你花了大力气调好的数据库进程、核心服务进程，万一在内存紧张时被OOM killer选中，整个系统的可用性就崩塌了。好消息是，Linux提供了不止一种保护手段，从简单粗暴到精细灵活，各有适用场景。

**知识点57 [I][M] 三种保护关键进程的方法**

第一种方法最简单粗暴，也是我用得最多的——**把`oom_score_adj`设为-1000**。

每个进程在`/proc/<pid>/oom_score_adj`文件里都有一个整数分值，范围从-1000到+1000。OOM killer计算最终的badness score时，会把`oom_score_adj`加到基础分上。基础分是内核根据进程的内存占用、运行时间等因素算出来的。如果你把`oom_score_adj`写成-1000，这个进程的最终得分就被锁定为0，OOM killer永远不会选中它。

```bash
# 让当前shell进程免疫OOM
$ echo -1000 > /proc/$$/oom_score_adj

# 让某个关键服务（pid为1234）免疫
$ echo -1000 > /proc/1234/oom_score_adj

# 在systemd服务文件中直接设置
[Service]
OOMScoreAdjust=-1000
```

这招很管用，但它有两个局限。一是只对单个进程有效，如果你有十几个关键进程要保护，得逐个设。二是它**完全禁止**了该进程被选为牺牲品，意味着如果系统真的走到OOM绝境，这个进程即使占着大量内存也不会被杀——系统可能因此直接卡死。

第二种方法走cgroup v2的路子——**用`memory.oom_control`禁用整个cgroup的OOM killer**。

```bash
# 创建一个cgroup并把关键进程加进去
$ mkdir /sys/fs/cgroup/my_critical_group
$ echo 1 > /sys/fs/cgroup/my_critical_group/cgroup.procs   # 把进程移进去

# 把OOM killer改成通知模式（不杀进程，只发事件）
$ echo 1 > /sys/fs/cgroup/my_critical_group/memory.oom_control
```

`memory.oom_control`设为1后，这个cgroup里的进程不会被OOM killer杀死，取而代之的是内核会产生一个"oom_kill"事件通知到user space。上层可以通过cgroup的eventfd机制监听这个事件，自己决定怎么处理——比如杀一个非关键的子进程，或者触发告警让运维介入。

这个方法的粒度是整个cgroup，适合保护一组相关进程。比如你的Web服务由主进程+多个工作进程组成，把它们放在同一个cgroup里统一保护，管理起来比逐个设`oom_score_adj`清爽多了。

第三种方法更高级——**用户空间的OOM handler**，代表作是systemd-oomd。

内核的OOM killer有个固有缺陷：它触发时系统已经处于极端内存压力下了，此时做任何事情都可能失败（包括杀死进程——因为`do_exit()`本身也要分配内存）。与其在内核里做这种"最后一搏"，不如在用户空间提前感知、提前行动。

```bash
# 查看systemd-oomd服务状态
$ systemctl status systemd-oomd

# systemd-oomd通过监控cgroup的内存压力来提前触发回收
# 在service文件中配置OOM过滤策略
[Service]
ManagedOOMSwap=kill          # 交换分区压力大时允许杀
ManagedOOMMemoryPressure=kill
ManagedOOMMemoryPressureLimitPercent=80%
```

systemd-oomd监听cgroup的 PSI（Pressure Stall Information）指标，在内存压力达到阈值但还没到OOM的程度时，就主动选择并杀死低优先级的进程。这种"提前动手"的策略比内核OOM killer优雅得多——系统还有足够资源来干净地结束进程，而不是在绝境中拼死一搏。

**知识点58 [I] 三种方法对比**

| 方法 | 粒度 | 灵活性 | 实现复杂度 | 适用场景 |
|------|------|--------|-----------|----------|
| `oom_score_adj=-1000` | 单个进程 | 低，非黑即白 | 极低，一行命令 | 单个关键进程的快速保护 |
| cgroup v2 `memory.oom_control` | 整个cgroup | 中，可切通知模式 | 中，需cgroup配置 | 一组关联进程的整体保护 |
| userspace OOM handler (如systemd-oomd) | 可配置多维度 | 高，基于PSI提前决策 | 高，需完整策略配置 | 复杂系统的精细化内存管理 |

说实话，这三种方法并不是互斥的。我自己的做法通常是：**核心单进程**用`oom_score_adj=-1000`，**服务组**用cgroup统一管理，**系统层面**再跑systemd-oomd做兜底。三层防护叠加，OOM killer就很难误伤到真正重要的东西了。

不过要提醒一句：过度保护也是有代价的。你把关键进程都设为免疫，等于把OOM killer的刀钝掉了。真到了内存耗尽的时候，系统可能既不杀进程也不响应——直接hang住。所以保护策略要配合内存监控一起用，不能一设了之。
