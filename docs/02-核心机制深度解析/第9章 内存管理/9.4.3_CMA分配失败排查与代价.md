# 9.4.3 CMA分配失败排查与代价

## 知识点124 CMA分配失败的"钉子户"困境与调优策略 [E]

> **引入**：CMA区域明明有空间，但 `dma_alloc_coherent` 还是失败？最可能的原因是：有"钉子户"不肯搬走——某些不可迁移页混入了CMA区域。

---

### 一、失败场景：CMA 区域为何"有空间却分配不出连续大块"

CMA（Contiguous Memory Allocator，连续内存分配器）的核心机制是在系统启动时从内存中**预留一片可迁移区域**，供后续 DMA 等需要物理连续大页的场合使用。理想情况下，当 CMA 分配请求到来时，系统可以通过**页面迁移（page migration）**将这片区域内的可迁移页搬到别处，腾出连续的物理大块。

但实战中，这片"预留的净土"往往会被**不可迁移的"钉子户"**悄悄占据：

| "钉子户"类型 | 混入机制 | 迁移特性 |
|:---:|:---|:---|
| **`mlock()` 锁定的用户页** | 用户进程通过 `mlock`/`mlockall` 将页锁定在物理内存中，防止换出。若该页恰好落在 CMA 区域，则成为不可迁移节点 | `PG_mlocked` 标记的页通常不可迁移 |
| **驱动通过 `kmalloc`/`__get_free_pages` 申请的 DMA buffer** | 未使用 CMA 专用接口，而在系统初始化中后期通过通用分配器申请，分配器可能从 CMA 区域返页 | 这些页若被驱动长期持有且未标记可迁移，则"钉死" |
| **内核数据结构、模块静态分配** | 早期内核代码或模块加载时申请的内存，恰好落入 CMA 物理地址范围 | 内核页通常不可迁移 |
| **initrd / initramfs 残留** | 早期启动阶段加载的 initrd 镜像，释放不彻底时残留在 CMA 区域 | 释放时机晚于 CMA 首次使用 |

当这些不可迁移页散落在 CMA 区域时，即使 `cat /proc/cma` 显示**使用率不高、空闲总空间充足**，也可能因为空闲页被不可迁移页**碎片化阻隔**，无法凑出一块**足够大的连续物理内存**，导致 `cma_alloc()` 或 `dma_alloc_coherent()` 返回 `-ENOMEM`。

---

### 二、排查路径：从现象到根因的三步法

#### 步骤 1：确认 CMA 区域整体状态

```bash
cat /proc/cma
```

输出示例及关键字段解读：

```
Number of CMA areas: 2
...
cma0: size=256 MiB, used=128 MiB, free=128 MiB
  alloc fail count: 15
  largest free block: 4 MiB
```

| 字段 | 含义 | 排查要点 |
|:---|:---|:---|
| `used` / `free` | 已用 / 剩余空间 | `free` 很大但仍失败 → 高度怀疑碎片化 |
| `alloc fail count` | 累计分配失败次数 | 若持续增长，确认问题正在发生 |
| `largest free block` | 最大连续空闲块 | 若此值小于申请大小，则失败原因明确 |

**关键判断**：当 `free` 总量 > 申请大小，但 `largest free block` < 申请大小时，即为**典型的 CMA 碎片化分配失败**。

#### 步骤 2：查看内核日志定位具体失败点

```bash
dmesg | grep -i "cma_alloc\|dma_alloc"
```

典型失败日志：

```
[  123.456] cma: cma_alloc: alloc failed, req-size: 1024 pages, ret: -12
[  123.458] dma_alloc_coherent: size=4194304, fail from cma
[  123.460] my_driver: failed to allocate DMA buffer (4 MiB)
```

结合 `largest free block` 的数据，可交叉验证失败原因是否为连续空间不足。

#### 步骤 3：追踪不可迁移页的 occupant

```bash
# 查看 CMA 区域的页状态分布（需开启 CONFIG_CMA_DEBUGFS）
cat /sys/kernel/debug/cma/cma0/bitmap

# 查看哪些进程占用了 CMA 区域的页（配合 /proc/pid/pagemap 分析）
# 或使用内核 ftrace 跟踪 cma_alloc / cma_release 调用栈
echo 'func cma_alloc' > /sys/kernel/debug/tracing/set_ftrace_filter
cat /sys/kernel/debug/tracing/trace
```

若怀疑 `mlock` 导致，可用 `ps -eo pid,comm,vsz,rss,pmem | grep -i mlock` 定位可疑进程；若怀疑驱动未释放 DMA buffer，检查驱动代码中 `dma_free_coherent` 的调用对称性。

---

### 三、预防与修复：四个维度的策略

| 策略 | 具体操作 | 适用场景 |
|:---|:---|:---|
| **增大 CMA 区域** | 修改设备树（DTS）`linux,cma-default` 大小，或 bootargs 中 `cma=512M` | 系统物理内存充裕，且 DMA 需求大的场景 |
| **提前 CMA 初始化顺序** | 在内核配置中确保 `cma_init` 早于可能分配大块内存的子系统；驱动加载顺序调整 | 存在驱动竞争 CMA 区域的情况，越早预留越"纯净" |
| **驱动规范释放 DMA buffer** | 确保 `dma_alloc_coherent` 与 `dma_free_coherent` 严格配对；模块卸载时释放全部 buffer | 驱动存在内存泄漏或异常退出路径未清理的情况 |
| **限制 mlock 作用域** | 对关键业务进程使用 `ulimit -l` 限制 mlock 上限；或将其绑定到非 CMA 的 NUMA 节点 | 用户态进程大量 mlock 导致 CMA 碎片化 |

### 四、代价分析：CMA 预留的"机会成本"

CMA 并非"白嫖"的连续内存池——它的本质是用**时间换空间**（迁移开销换连续大块），同时伴随明确的**内存代价**：

| 代价维度 | 说明 | 量化参考 |
|:---|:---|:---|
| **通用可用内存减少** | CMA 预留区域在未被 DMA 使用时，虽可被可迁移页利用，但**不可用于不可迁移页分配**（如内核栈、`kmalloc` 高阶分配等） | 预留 256 MiB CMA → 系统 `MemTotal` 等效减少 256 MiB 的"通用弹性" |
| **页面迁移开销** | CMA 分配触发迁移时，需复制页内容、更新页表、刷新 TLB，延迟可达**毫秒级** | 大块分配（如 4 MiB/1024 pages）迁移延迟约 2–10 ms，取决于 CPU 和内存带宽 |
| **分配延迟不可预测** | 若 CMA 区域碎片化严重，`dma_alloc_coherent` 可能从**快速路径**跌入**慢速迁移路径**，甚至失败 | RT 场景下可能导致确定性延迟破坏 |
| **内存浪费（内部碎片）** | CMA 按页块（通常 4 MiB 对齐）管理，申请非整数倍大小时产生内部碎片 | 申请 3 MiB 于 4 MiB 块中，浪费约 25% |

---

### 五、平衡策略：预留多大的 CMA 才够？

| 系统类型 | CMA 推荐占比 | 理由 |
|:---|:---|:---|
| **嵌入式 / IoT（小内存）** | 总内存的 5%–10% | 内存宝贵，DMA 需求通常较小（如音频、传感器 buffer） |
| **移动设备（手机/平板）** | 总内存的 10%–15% | GPU、相机、视频编解码需要较多连续 buffer |
| **桌面 / 工作站** | 总内存的 10%–20% | 显卡显存映射、视频采集、高速外设 DMA |
| **服务器 / 高性能计算** | 总内存的 20%–30%，或固定 1–4 GiB | RDMA NIC、FPGA、NVMe 控制器等大容量 DMA 需求 |
| **实时系统（RT）** | 根据最坏情况 DMA 需求**精确预留**，避免运行时迁移 | 确定性优先，宁可多预留也不触发迁移延迟 |

---

### 六、总结：CMA 问题的核心口诀

> **"有空间 ≠ 有连续空间；看 `/proc/cma` 总空闲，更要看最大块；查 `dmesg` 定失败点；早初始化、防泄漏、控 mlock 是预防三板斧；预留是双刃剑，按需分配才是正道。"**
