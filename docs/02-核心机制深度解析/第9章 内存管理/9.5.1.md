服务器上swap是标配。内存不够了？往磁盘上一扔，虽然慢点，但服务能接着跑。可嵌入式设备呢？你给它配个swap试试——Flash的擦写次数撑得住吗？就算你用eMMC甚至SD卡当swap，那写入速度能让你怀疑人生。于是乎，大多数嵌入式产品干脆就不配swap，内存多大就是多大，天花板明摆着。

可问题来了：没有swap，内存吃紧的时候怎么办？答案是——直接OOM。内核的OOM killer拎着刀出来，挑个进程一刀砍死。没有缓冲，没有余地，用户看着设备突然重启或者应用莫名其妙消失，一脸懵。

**知识点45 [I]**

嵌入式不给swap，根子出在Flash的物理特性上。NAND Flash的每个块擦写次数有限，SLC撑个十万次，TLC可能几千次就挂了。swap这种频繁读写的场景，简直就是Flash寿命的催命符。就算你有再大的Flash，也经不起swap来回折腾。

更麻烦的是速度。往Flash写一页（4KB）， latency 动辄几十毫秒，跟DDR内存的纳秒级差了五个数量级。这要是真swap起来，系统卡成PPT，用户体验直接崩盘。所以很多嵌入式产品从设计之初就把swap这条路封死了。

但没有swap的代价也很真实。来看个场景：你的设备跑着跑着，某个模块突然申请了一大块内存，free内存瞬间见底。有swap的系统还能把冷页面换出去撑一会儿；没swap的系统呢？`pagefault`一路走到`do_page_fault`，发现没有可用的物理页，OOM killer就被触发了。以下是OOM killer决策的核心路径：

```c
// mm/oom_kill.c  ——  select_bad_process() 的简化逻辑
static struct task_struct *select_bad_process(unsigned int *ppoints,
				       unsigned long totalpages,
				       const nodemask_t *nodemask,
				       bool force_kill)
{
	struct task_struct *g, *p;
	struct task_struct *chosen = NULL;
	unsigned long chosen_points = 0;

	// 遍历所有进程，打分挑"最差"的那个
	for_each_process_thread(g, p) {
		unsigned int points = oom_badness(p, totalpages);
		if (!chosen || points > chosen_points) {
			chosen = p;
			chosen_points = points;
		}
	}
	return chosen;  // 返回被选中的"倒霉蛋"
}
```

| 场景 | 有swap | 无swap |
|------|--------|--------|
| 内存不足时 | 冷页面换出到磁盘，系统继续运行 | 直接触发OOM killer |
| 性能表现 | swap期间卡顿，但有缓冲 | 无缓冲，进程直接被杀死 |
| Flash寿命 | 不受影响（磁盘swap） | 不涉及swap，但可能频繁重启 |
| 用户体验 | 慢但能忍 | 应用闪退或设备重启 |

说白了，嵌入式在内存储管理上走了一条极端的路：要么够用，要么死。这显然不是好办法。那有没有第三条路？

有，就是zswap。

**知识点46 [I]**

zswap的思路很巧妙——既然不能往磁盘swap，那能不能在内存里做"内部swap"？具体来说，就是在内存和真正的swap设备之间插入一个压缩缓存层。

写路径是这样的：当内核准备把一个匿名页换出去时，zswap先截胡，把这个页压缩一下，然后存到一块预先分配的内存池里。压缩比通常能做到2:1到3:1，也就是说，原来4KB的页，压缩完可能只占1~2KB。这样一来，用一部分RAM当作"压缩后的swap区"，相当于凭空多出了一块内存。

读路径更关键：如果后续访问正好命中了zswap里的压缩页，直接原地解压返回，速度比读Flash快好几个数量级；如果没命中，那没办法，该走正常swap流程就走。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   匿名页     │ ──→ │   zswap     │ ──→ │  真实swap    │
│  (4KB)      │     │ 压缩缓存层   │     │  (Flash/磁盘)│
└─────────────┘     └─────────────┘     └─────────────┘
                           │
                           ↓ 压缩存储
                    ┌─────────────┐
                    │  内存压缩池  │
                    │ (1~2KB/页) │
                    └─────────────┘
```

> ⚠️ **注意**：zswap虽然叫"swap"，但它并不真正消耗swap分区空间——页面被压缩后存在内存里。只有当zswap自身内存池满了，或者配置了`zswap_enabled=0`时，才会回退到真正的swap设备。

zswap的出现，本质上是在"内存不够就OOM"和"swap到Flash伤寿命"之间找了个平衡点。它用CPU的算力（压缩/解压）换取了内存空间和I/O寿命，对于CPU相对富余、但内存和Flash紧张的嵌入式设备来说，这笔买卖通常是划算的。
