`/proc/slabinfo` 是很多工程师知道但真拿来定位泄漏时却想不起用的文件。MemFree 掉多少只能告诉你"内存少了"，slabinfo 能精确到"哪个 cache、多大的对象、具体漏了多少"。Meminfo 看到的是水面上的冰山，slabinfo 能帮你找到水底下到底是哪一块在化。

**知识点31 [I][M]**

先看 slabinfo 的输出：

```
# cat /proc/slabinfo
slabinfo - version: 2.1
# name            <active_objs> <num_objs> <objsize> ...
kmalloc-256        12288        12288       256     ...
kmalloc-512         4096         8192       512     ...
sock_inode_cache    2341         2341       640     ...
dentry             87654        90120       192     ...
```

盯紧前三列：`active_objs`（已分配未释放的对象数）、`num_objs`（cache 中对象总数）、`objsize`（单个对象大小）。

泄漏的典型特征就两条：一是 `num_objs` 持续增长，只升不降或者升多降少；二是 `active_objs` 几乎等于 `num_objs`，说明绝大部分对象都"活着"，没人释放。正常负载下这两个数字之间是有差距的——对象用完会归还到 freelist，供下次分配复用，所以 `num_objs` 会略高于 `active_objs`。如果你看到某一行长期 `active_objs ≈ num_objs` 且数字越爬越高，基本就是漏了。kmalloc 系列的 cache 最值得盯，因为驱动里 kmalloc/kfree 用得最多。

下面这个脚本够朴素也够实用，定期采样、超阈值报警：

```bash
#!/bin/bash
# slab_monitor.sh — 简易slab泄漏监控

CACHE_NAME="${1:-kmalloc-256}"
THRESHOLD="${2:-100}"           # 单次增长超过100个对象即报警
INTERVAL=5
LOGFILE="/var/log/slab_${CACHE_NAME}.log"

prev_num=0
echo "=== $(date): 开始监控 [$CACHE_NAME] ===" > "$LOGFILE"

while true; do
    line=$(grep "^${CACHE_NAME} " /proc/slabinfo 2>/dev/null)
    [ -z "$line" ] && sleep "$INTERVAL" && continue

    active=$(echo "$line" | awk '{print $2}')
    num=$(echo "$line" | awk '{print $3}')

    # 判断增长
    if [ "$prev_num" -ne 0 ]; then
        delta=$((num - prev_num))
        if [ "$delta" -gt "$THRESHOLD" ]; then
            echo "$(date): ALERT $CACHE_NAME +$delta ($prev_num->$num)" \
                 "active=$active" >> "$LOGFILE"
        fi
    fi

    # active/num 利用率长期>95%也报
    if [ "$num" -gt 1000 ]; then
        ratio=$((active * 100 / num))
        [ "$ratio" -gt 95 ] && echo "$(date): WARN $CACHEName ${ratio}% ($active/$num)" >> "$LOGFILE"
    fi

    prev_num="$num"
    sleep "$INTERVAL"
done
```

部署时挂成 systemd timer 或 cron，输出接 syslog 再配告警通知就能干活。实际运维中，这种监控最好跑在业务低峰期做基线采样——先把正常状态下的 num_objs 波动范围摸清楚，再设阈值，否则容易误报。更完善的做法是同时扫 kmalloc-64/128/256/512，awk 批量处理，有异常的单独高亮。

这三列数据组合起来，漏了哪、漏多少、漏的是多大的对象，一目了然。实际排查时先把 slabinfo 按 num_objs 排个序，`sort -k3 -n`，尾巴上那几个增长异常的 cache 就是重点怀疑对象。

| 列名 | 含义 | 泄漏诊断中的用途 |
|:---|:---|:---|
| `active_objs` | 已分配且未释放的对象数 | 接近 num_objs 说明对象没被归还 |
| `num_objs` | cache 中对象总数 | **持续增长不回落**是泄漏的硬指标 |
| `objsize` | 单个对象大小 | 估算泄漏内存：num_objs × objsize |

**知识点32 [I]**

前年有个工业网关的案例。ARM 设备跑一周必 OOM，重启又活，再跑一周再挂。`cat /proc/slabinfo | sort -k3 -n` 末尾永远是 `kmalloc-256` 在领跑，num_objs 飙到七位数，开机时才几千。其他 cache 都正常，就这一行疯涨。当时先用 `slabtop -o` 确认 kmalloc-256 的活跃对象比例接近 100%，然后打开 `slab_account` 做详细追踪缩小范围，最后 kmemleak 一把揪出了问题路径。

排查到最后，是某网络驱动的错误分支里写错了释放函数：正确路径走 `dev_kfree_skb(skb)`，错误路径作者手快写成了 `kfree(skb)`，而那个 skb 指针在错误分支指向的是**栈上的局部变量**，根本不是 kmalloc 出来的。kfree 收到野指针，slub 内部的 freelist 被破坏，后续同一 cache 的对象释放时回不到正确的 slab，整批对象"失联"——alloc 计数在涨，free 却找不对地方，漏得悄无声息。修复这一行后 kmalloc-256 回到正常水平，再也没 OOM。

这类"错误路径释放错对象"在 out-of-tree 驱动里不少见。遇到慢增长 OOM，先把 slabinfo 拉出来按 num_objs 排序看一眼，哪个 cache 异常突出再针对性追代码，比漫无目的地审计所有 kmalloc 高效得多。
