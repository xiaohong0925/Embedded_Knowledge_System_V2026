如果你去翻Linux内核邮件列表里2007年那段的存档，能看到一场挺有意思的争论。Christoph Lameter——当时还在SGI干活——贴出了一段新代码，说要把SLAB分配器替掉。他的理由直截了当："SLAB做了太多本不需要做的事。"这句话后来成了内核社区里的经典引用。争论归争论，SLUB最终在2.6.22版本里合入了主线，并且在2.6.23之后慢慢成为默认选项。到今天，你随手编译一个内核，`CONFIG_SLUB` 已经是默认勾上的。

**知识点23 [I]**

那SLAB到底哪不好？要说清楚这个，得先看看SLAB当初的设计意图。2.4内核引入SLAB的时候，Jeff Bonwick（Solaris的VM大佬）带来的理念是：对象缓存不仅要管理内存，还要优化CPU缓存的命中——也就是所谓的"着色"（coloring）。简单说，不同slab的基地址要按L1 cache line大小错开，这样不同slab里同偏移位置的对象不会映射到同一个cache set上。想法很精巧，但实际带来了不少麻烦。

首先，着色要求系统严格跟踪每个slab里有多少空闲对象、多少在用对象，于是SLAB维护了三条链：满链（full）、半满链（partial）、空链（empty）。这三条链的增删改查逻辑非常复杂，尤其是NUMA架构下，每个node还要有自己的kmem_list3结构，代码量噌噌往上长。早年间我读过一段SLAB的核心代码，光是一个 `__cache_free` 路径就嵌了七八层判断，看半天才捋清楚对象到底该放回哪条链。

Christoph的SLUB干干脆脆地砍掉了着色——实测表明在现代CPU的复杂缓存体系下，SLAB着色带来的性能收益微乎其微，维护成本却高得离谱。与此同时，SLUB把三条链简化成了一条partial链：只有还包含空闲对象的slab才挂在链上，完全满的slab不需要链管理（直接由使用者持有），完全空的slab直接还给伙伴系统。数据结构也从 `kmem_cache` + `kmem_list3` + `slab` 三层套娃，简化成了一个 `kmem_cache` 直接管理page。代码量从SLAB的几千行降到了SLUB的一千多行。

那为什么内核默认选SLUB，而不是更老的SLAB，或者更极端的SLOB？说白了，SLUB找到了一个甜点：它在大多数情况下跟SLAB性能持平甚至更优（partial链的查找更快），同时代码简单得多， bug自然也少。内核维护者都喜欢这种"代码少、活还好"的方案。

**知识点24 [I]**

内核编译时，这三种分配器是三选一的互斥选项，看 `mm/Kconfig` 里的定义就能明白：

| 配置选项 | 适用场景 | 特点 | 代码规模 |
|---------|---------|------|---------|
| `CONFIG_SLUB` | 通用桌面/服务器（默认） | 无着色、单partial链、per-cpu本地缓存 | 约1500行 |
| `CONFIG_SLAB` | 兼容性需求（已基本废弃） | 完整着色、三链管理、NUMA感知的复杂结构 | 约4000+行 |
| `CONFIG_SLOB` | 极小内存嵌入式系统 | 极简链表分配器，不做对象缓存，直接malloc/free式管理 | 约600行 |

SLOB值得单独提一句。Matt Mackall当年写SLOB是给嵌入式用的——比如在只有4MB内存的古老设备上跑Linux。SLOB不做任何缓存优化，就是一条全局链表按best-fit策略分配。你把它开在现代机器上跑，性能能差出一个数量级，但它的存在意义很明确：把内核的内存管理代码压到最小。如果你的板子内存小于16MB，SLOB可能还是值得考虑的；否则老老实实选SLUB。

```c
// mm/Kconfig 中的三选一配置示意
choice
    prompt "Slab allocator"
    default SLUB_CPU_PARTIAL

config SLUB
    bool "SLUB (Unqueued Allocator)"
    help
        SLUB是Linux内核默认的slab分配器，
        相比SLAB大幅简化了代码，同时保持了良好的性能。

config SLAB
    bool "SLAB"
    help
        传统SLAB分配器，保留以兼容旧系统，
        新系统不建议使用。

config SLOB
    bool "SLOB (Simple Allocator)"
    depends on EXPERT
    help
        极简分配器，仅用于极小内存的嵌入式系统。
endchoice
```

> **注意**：SLOB依赖 `CONFIG_EXPERT` ，也就是说默认菜单里你是看不到它的。只有在手动开启expert模式后，SLOB选项才会暴露出来。这个设计本身就在告诉你——除非你真的知道自己在干什么，否则别碰它。

三种分配器的演化路线其实很清晰：SLAB是"学院派"的精致设计，SLOB是"生存派"的极简方案，SLUB则是"工程派"的务实选择——砍掉那些纸上谈兵的理论，保留真正有用的优化。内核社区里这类演化故事很多，SLAB到SLUB的切换算是其中教科书级别的一例。
