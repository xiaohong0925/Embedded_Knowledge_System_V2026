**知识点17 [E][M]** — 5种迁移类型与pageblock

分页之后还有个让人头疼的老大难问题——内存碎片。你系统跑着跑着，明明空闲内存加起来还有好几个G，可就是申请不到连续的大块物理页。这事儿在内核里存在了几十年，社区试过各种办法，最后MIGRATE_TYPES这套分类机制算是比较成熟的方案。

话说早年间，内核把物理内存一股脑扔在一起管理，内核栈、slab缓存、用户进程的匿名页全混在一个zone里。某天kcompactd想规整内存，发现一块2MB的区域里夹着几个内核数据结构用的不可迁移页，整片区域就没法搬了——好比搬家的时候有个柜子焊死在地板上，搬家公司只能绕道走。Linus当初拍板：得把不同类型的页分开住。

于是内核定义了5种迁移类型，每种类型在自己的"朋友圈"里分配页面，互不搅和：

| 迁移类型 | 典型用途 | 能否迁移 | 备注 |
|:---|:---|:---:|:---|
| **UNMOVABLE** | 内核栈、内核数据段、部分slab | 否 | 内核启动早期就固定的内存，搬不了 |
| **RECLAIMABLE** | 文件缓存（page cache）、buffer cache | 可回收 | 不迁移，但回收后重新读盘即可 |
| **MOVABLE** | 用户进程匿名页、文件映射页 | 是 | 页表一改就能搬走，规整的主力军 |
| **CMA** | 连续内存分配器预留区域 | 是 | 给DMA/CMA预留的"弹性空间" |
| **RESERVE** | 紧急情况预留（如atomic分配） | — | 不到万不得已不动用 |

这5类的核心思想简单粗暴：**别让不可迁移的页混进可迁移页的地盘**。一个MOVABLE类型的pageblock里全是用户页，规整时拎起来就能走；但要是混进几个内核栈页，这块地就"钉死"了，再规整也腾不出连续空间。

说到这，得提一下**pageblock_order**这个概念。内核以pageblock为单位来划分迁移类型，默认值通常是一个大页的大小——x86上就是`9`（2MB，即512个4KB页）。这个粒度很有意思：太小的话管理开销大，太大的话浪费严重。`pageblock_order`的定义在`mm/page_alloc.c`里：

```c
#ifdef CONFIG_HUGETLB_PAGE
    #define pageblock_order     HUGETLB_PAGE_ORDER
#else
    #define pageblock_order     (MAX_ORDER-1)
#endif
```

有hugetlbfs时就用大页order，否则退而取`MAX_ORDER-1`。这个设计很聪明——保证了pageblock天然对齐大页边界，规整出来的连续区域直接就能拿来用。

每个pageblock的迁移类型记录在`struct zone`的`pageblock_flags`位图里。分配器会根据请求GFP标志选类型：普通用户页走MOVABLE，内核栈这种走UNMOVABLE，文件缓存走RECLAIMABLE。类型之间也不是完全老死不相往来——当某类内存紧张时，内核允许从其他类型"偷"pageblock，但这是个带有降级策略的操作，不是随便拿的。

**知识点18 [I]** — 读`/proc/pagetypeinfo`看页面分布

想知道当前系统里这5类页面各有多少、分布在哪？`/proc/pagetypeinfo`是个宝藏文件。来看个实际例子：

```bash
$ cat /proc/pagetypeinfo
Page block order: 9
Pages per block:  512

Free pages count per migrate type at order       0      1      2      3      4 ...
Node    0, zone      DMA, type    Unmovable      2      1      1      0      0 ...
Node    0, zone      DMA, type  Reclaimable      0      0      1      0      0 ...
Node    0, zone      DMA, type      Movable      5      3      2      1      1 ...
Node    0, zone      DMA, type          CMA      0      0      0      0      0 ...
Node    0, zone      DMA, type      Reserve      0      0      0      1      0 ...
...（Node 0, zone DMA32 同理）...
Node    0, zone   Normal, type    Unmovable     12      5      3      2      1 ...
Node    0, zone   Normal, type  Reclaimable    256    128     64     32      8 ...
Node    0, zone   Normal, type      Movable   1024    512    256    128     64 ...
```

头两行告诉你pageblock_order和每block多少页——这里order 9就是512页，也就是2MB。下面的数据按Node、Zone、迁移类型三维度展开，每行是一个(order=0到order=10)的空闲页分布直方图。

实际排错的时候怎么看呢？如果你发现**MOVABLE行在order≥4的位置全是0**，说明可迁移页已经碎得不成样子了，用户申请大页（比如透明大页）大概率会失败。反过来，如果**UNMOVABLE行在高位order有数字**，那说明内核数据"霸占"了本不该它待的大块连续区域——这往往是某个驱动一次性申请了太多不可迁移内存。

还有一个经验：CMA那行如果一直是0，要么你的系统没开CMA，要么CMA区域已经被占满。调试DMA相关问题时，这一行值得盯紧。

说实话，这个文件的数据量不小，真出问题的时候建议抓两次快照对比一下——碎片化问题往往是动态演变的，单看一个时间点容易错过线索。
