你的 camera 驱动在设备运行两天后突然返回 NULL——`dma_alloc_coherent` 失败了。但你敲下 `cat /proc/meminfo`，看到 `MemFree` 还有两百多兆。两百多兆的内存，连一块 4MB 的连续物理页都要不到？

Buddy 分配器当然有内存，问题是这些内存全碎了。

**知识点35 [I][M]**

嵌入式设备里对大块连续物理内存的需求无处不在。相机 ISP 需要连续 buffer 做图像采集，视频编解码器要把一帧帧数据喂进硬件，DMA 控制器更是直接——它读的是物理地址， scatter-gather 虽然能缓解，但不少老硬件或性能敏感的场景下，一块连续的 DMA buffer 仍是刚需。

这些需求落到底层，最终都会走到 Buddy 分配器面前。Buddy 算法的核心是把物理内存按 2 的幂次切分成不同阶（order）的块。4MB 对应 order-10（假设页大小 4KB），这意味着它需要找到一块连续 1024 页的物理空间。

设备刚启动时没问题，内存空空如也，随便分配。但运行时间一长，各种进程来了又走，kmalloc、vmalloc、页面缓存进进出出，内存被切割得七零八落。Buddy 分配器维护着从 order-0 到 order-MAX 的空闲链表，高阶的空闲块在系统运行过程中被逐步拆分成低阶块来满足小粒度分配，而这些低阶块散落在各处，很难再合并回高阶。

```
# 设备刚启动时的 buddyinfo —— 高阶空闲块充裕
cat /proc/buddyinfo
Node 0, zone   Normal  1024   512   256   128    64    32    16     8     2     1
                                         ^ order-5 以上还有空闲

# 运行两天后的 buddyinfo —— 高阶块归零
cat /proc/buddyinfo
Node 0, zone   Normal  8192  4096  2048  1024   512   256   128    64     0     0
                                                             ^^^^^ ^^^^^
                                                             order-5 以上空闲为 0
```

上面的输出对比很能说明问题。order-5 对应 128KB，往上走 order-8 是 1MB、order-10 是 4MB。当这些高阶空闲计数变成 0 时，不管你总的 free 内存还有多少，`alloc_pages(GFP_ATOMIC, 10)` 都会直接失败。Buddy 不是不够聪明——它的合并策略在页回收和释放时已经尽力了，但"外碎片"（external fragmentation）是连续分配的天然敌人。

```c
/* 典型的相机驱动代码，dma_alloc_coherent 底层最终调用 alloc_pages */
void *buf = dma_alloc_coherent(dev, 4 * 1024 * 1024, &dma_handle, GFP_KERNEL);
if (!buf) {
    /* 运行两天后几乎必定会走到这里 */
    dev_err(dev, "Failed to allocate 4MB DMA buffer\n");
    return -ENOMEM;
}
```

这就是连续内存分配问题的本质：**总量充足，但连续不足**。Buddy 分配器解决不了外碎片，因为内存碎片化的根因不是分配算法，而是工作负载本身。

> **陷阱**：`/proc/meminfo` 中的 `MemFree` 和 `Buddy` 高阶空闲数是两个完全不同的指标。看到 "还有几百兆内存" 就断定不是内存问题，是排查连续内存分配失败时最常见的误区。始终用 `/proc/buddyinfo` 看高阶块的分布。

**知识点36 [I]**

我见过一个安防设备的案例。那是一台搭载 512MB DDR 的嵌入式 NVR，跑 Linux 4.9 内核。camera 子系统每路通道需要 4MB DMA buffer 做 H.264 编码输入。开发阶段一切正常，产线跑了两天老化测试后，四路 camera 同时打开时最后一路必定失败。

工程师查了一下午，从驱动到 V4L2 框架再到 meminfo，死活想不通——明明还剩 200 多兆。直到有人顺手敲了 `cat /proc/buddyinfo`，才发现 order-5 以上的空闲计数全是 0。内存碎得像打翻的拼图，拼不出一块 4MB 的连续区域。

这个场景的解决思路，就是 CMA（Contiguous Memory Allocator）。CMA 的设计目标很直接：在内核启动时从系统中**预留**出一块连续的物理内存区域，但这块内存不是死的——它被标记为"可迁移"（movable），平时可以被用户空间的匿名页、缓存页等 movable 类型分配器借用。只有当设备驱动真正需要连续内存时，CMA 才把这些借出去的页迁走，腾出连续空间。

```
┌─────────────────────────────────────────────────────────────┐
│                     系统物理内存                              │
│  ┌─────────────┬─────────────────────────┬────────────────┐ │
│  │  固定使用    │      CMA 区域 (可迁移)    │   普通内存区域   │ │
│  │ (kernel/    │  预留，可被 movable 页    │  正常分配回收    │ │
│  │  reserved)  │  借用，需要时强制迁移      │                │ │
│  └─────────────┴─────────────────────────┴────────────────┘ │
└─────────────────────────────────────────────────────────────┘
                        ↑
              CMA = "兜底"的连续内存池
```

说白了，CMA 是给设备驱动留的一块"私房钱"。平时你可以借去花，但关键时候必须还回来。这种"预留但不独占"的策略，在内存紧张的嵌入式设备上尤其重要——直接硬预留几十兆连续内存太奢侈，完全靠 Buddy 又靠不住。CMA 取了个中间路线，这也是它能成为当前嵌入式内核标配机制的原因。

| 机制 | 是否预留连续内存 | 平时能否他用 | 分配可靠性 |
|------|---------------|-----------|----------|
| 直接 Buddy 分配 | 否 | 是 | 低（碎片化后高阶分配必败） |
| 硬预留（mem=） | 是 | 否 | 高（但内存浪费严重） |
| **CMA** | **是** | **可迁移页可借用** | **高（延迟迁移开销）** |

下一节我们来看看 CMA 的初始化流程——它到底怎么从 DTS 里把这块"私房钱"划出来的。
