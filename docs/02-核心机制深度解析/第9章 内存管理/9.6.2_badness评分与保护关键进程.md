# 9.6.2 badness评分与保护关键进程

> **引入**：OOM killer的选择逻辑可能跟你想象的不一样——它不是杀"最不重要的"进程，而是杀"释放最多内存"的进程。这意味着你的关键服务可能最先被杀！理解badness评分机制，是构建高可靠系统的必修课。

---

## 知识点 131：oom_score计算与关键认知 [I][M] ⏱️15min

### 一、oom_score的内核计算逻辑

Linux内核中的OOM killer通过 `oom_badness()` 函数为每个进程计算一个0~1000之间的评分——`oom_score`。评分越高，越容易被选中杀死。这个评分不是随意给的，而是由多个维度加权计算得出：

**1. RSS（Resident Set Size）——权重最大**

这是oom_score中占比最高的因素。进程的RSS越大，oom_score越高。内核代码中大致逻辑为：

```
omm_score = total_vm_pages * 1000 / total_pages
```

进程占用的物理内存页数占总可用内存页数的比例，乘以1000进行归一化。一个占用50%物理内存的进程，仅此一项就能拿到500分以上的基础分。

**2. 进程运行时间与nice值——小幅调节**

内核会考虑进程的运行时间和nice值：
- **运行时间长的进程**：oom_score略微降低。运行久的进程可能承载重要业务，且已投入大量CPU资源。
- **nice值高的进程（低优先级）**：oom_score略微升高。nice值高的进程本就是被调度器"嫌弃"的，OOM killer倾向于优先牺牲它们。

这两项调整幅度通常不超过几十分，在RSS主导的计算中影响有限。

**3. 子进程与cgroups（现代内核）**

在启用了cgroups的现代内核中，OOM killer还会考虑进程所属cgroup的总内存占用。若一个进程管理大量子进程，其oom_score会反映整个cgroup的内存压力。

### 二、oom_score_adj：用户空间的调整手段

内核暴露了一个用户空间可写的接口 `/proc/<pid>/oom_score_adj`，允许管理员主动调整进程的oom_score，范围从 **-1000 到 +1000**：

| oom_score_adj值 | 效果 |
|---------------|------|
| +1000 | oom_score直接拉满到1000，最先被杀 |
| 0 | 完全依赖内核计算的oom_score（默认） |
| -1000 | oom_score强制为0，**永远不被OOM killer选中** |
| 其他值 | 线性加减到内核计算的oom_score上 |

**关键认知**：设置 `oom_score_adj = -1000` 是保护关键进程最直接的手段。例如保护sshd服务：

```bash
# 方法1：systemd服务中设置
[Service]
OOMScoreAdjust=-1000

# 方法2：运行时手动设置
echo -1000 > /proc/$(pgrep sshd)/oom_score_adj
```

`oom_score_adj` 替代了早期内核的 `oom_adj`（范围-17到+15，已废弃）。旧接口在新内核中虽仍兼容，但应使用新标准接口。

### 三、观测oom_score：理解系统的选择倾向

每个进程都有两个只读接口可供观察：
- `/proc/<pid>/oom_score`：最终的综合评分（0~1000）
- `/proc/<pid>/oom_score_adj`：用户设置的偏移值

```bash
# 查看系统中oom_score最高的进程（最危险的进程）
for pid in $(ls /proc | grep -E '^[0-9]+$'); do
    score=$(cat /proc/$pid/oom_score 2>/dev/null)
    comm=$(cat /proc/$pid/comm 2>/dev/null)
    echo "$score $comm"
done 2>/dev/null | sort -rn | head -10
```

这个简单脚本输出的结果往往会让你吃惊——排名第一的很可能是你的MySQL、Redis或Java应用，而不是那些你认为"不重要"的小工具。

### 四、核心认知误区：OOM killer不杀"最不重要的"

这是运维工程师最常见的误解，必须纠正：

**OOM killer的目标是"用最少的选择释放最多的内存"**，而不是"找到系统中最不重要的进程"。

这意味着：
- 你的**数据库主进程**（占用8GB内存）比**日志转发代理**（占用10MB）更容易被杀
- 你的**Java应用服务器**（Heap 16GB）比**监控探针**（几十MB）更容易被杀
- 一个**刚启动的、业务关键的内存密集型服务**，可能比**运行数月的低优先级工具**先死

这种设计有其合理性——系统即将崩溃时，杀死一个大内存进程可能立即挽救整个系统，而杀死多个小进程可能需要多次迭代、耗时更久。但对运维人员而言，如果不加保护，关键业务反而成为"靶子"。更危险的是，OOM killer的选择是"静默"的：除非查看内核日志（`dmesg` 中的 `Killed process` 行），否则你很难第一时间知道是谁被杀、为什么被杀。

### 五、三类保护策略概览

针对OOM killer的选择逻辑，业界发展出三种主流保护方法：

| 方法 | 作用粒度 | 核心机制 | 适用场景 |
|-----|---------|---------|---------|
| `oom_score_adj` | 单个进程 | 调整评分，-1000免杀 | 保护关键守护进程 |
| cgroup `memory.oom_control` | 整个cgroup | 禁用OOM killer，改为通知 | 容器化部署，批量保护 |
| userspace OOM handler | 系统级 | 用户空间提前介入决策 | 复杂策略、精细化管控 |

这三种方法各有优劣，将在下一个知识点中详细对比。

### 本节要点回顾

- `oom_score` 主要由 **RSS物理内存占用** 决定，运行时间和nice值只做微调
- `oom_score_adj` 范围 **-1000~+1000**，-1000意味着完全免杀
- **OOM killer选"释放最多内存的"，不是"最不重要的"**——大内存关键服务最危险
- 保护策略分三个层次：单进程调节、cgroup级禁用、用户空间接管

## 知识点 132：三种OOM保护方法对比 [I] ⏱️10min

三种保护方法怎么选？下表从关键维度做横向对比：

| 对比维度 | oom_score_adj=-1000 | cgroup v2 memory.oom_control | userspace OOM handler |
|---------|---------------------|------------------------------|----------------------|
| **作用粒度** | 单个进程 | 整个cgroup | 整个系统 |
| **实现复杂度** | 极简（一行配置） | 中等 | 较高（需额外组件） |
| **额外开销** | 无 | 极低 | 中等（监控+决策） |
| **灵活性** | 低（只能调分数） | 中 | 高（任意自定义策略） |
| **典型代表** | systemd `OOMScoreAdjust` | Kubernetes容器 | systemd-oomd、earlyoom |

**oom_score_adj=-1000**：最直接的手段。进程的oom_score被强制归零，OOM killer会直接跳过它。零依赖、任何Linux系统都支持、无额外开销。在systemd服务中只需一行 `OOMScoreAdjust=-1000` 即可。缺点也很明显：只能逐个进程保护，规模化困难；若关键进程恰好是内存泄漏元凶，系统可能因无法杀它而整体卡死。

**cgroup v2 memory.oom_control**：当cgroup达到内存上限时，将OOM killer行为从"直接杀死"切换为"通知用户空间"。内存压力事件通过eventfd机制通知用户空间程序，由它自行决定如何释放内存（如清理缓存、触发GC、优雅重启等）。优点是一次配置保护整个服务组，与Docker、Kubernetes天然集成；缺点是需要cgroup v2环境，禁用后若用户空间处理不及时，cgroup内进程可能被内核冻结。

**userspace OOM handler**：现代Linux发行版推荐的方向。在用户空间运行守护进程，持续监控系统内存压力和cgroup状态，**在OOM killer触发之前就采取行动**。代表项目包括systemd-oomd（基于PSI预测，systemd 248+内置）、earlyoom（轻量级）和Facebook oomd。核心优势是提前数秒甚至数十秒介入，可按业务策略决策（如先杀非核心副本、先清理page cache）；缺点是需要额外组件，策略配置需理解PSI指标，且极端场景下内存瞬间耗尽时可能来不及反应。

| 你的场景 | 推荐方案 |
|---------|---------|
| 1~2个核心进程必须保活（如sshd） | `oom_score_adj=-1000` |
| 容器化部署，批量服务分级保护 | cgroup v2 `memory.oom_control` |
| 大规模集群，智能渐进OOM处理 | systemd-oomd |
| 嵌入式/资源受限设备 | earlyoom |

实践中**三种方法往往组合使用**：oom_score_adj保护绝对不可杀的进程，cgroup为容器划分安全边界，systemd-oomd做系统级智能预防。

### 本节要点回顾

- **oom_score_adj=-1000** 最简单，零开销但灵活性差
- **cgroup v2 memory.oom_control** 将杀戮改为通知，适合容器批量保护
- **userspace OOM handler** 最灵活，能在OOM触发前提前干预
- 生产环境推荐**组合使用**：核心进程免杀 + cgroup分级 + 用户空间预测处理
