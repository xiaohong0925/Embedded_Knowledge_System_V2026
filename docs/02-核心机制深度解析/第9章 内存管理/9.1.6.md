你`cat`过`/proc/self/maps`吗？满屏的十六进制数字看得眼晕？其实每一列都有明确的含义，搞懂了回头看，这块"内存地图"能告诉你一个进程所有的地址空间秘密。不过在解读之前，咱们得先搞清楚内核空间的布局是怎么分的——用户空间的布局你已经在前面看过了，内核那边又是另一套逻辑。

**知识点11 [I][M] 内核空间的三大区域**

内核空间不像用户空间那样自由散漫，它的布局是严格分区的，核心就三大块。

第一块是**线性映射区（linear mapping）**，也有人叫"直接映射区"。这部分就是物理内存一对一映射到内核空间，内核启动早期就建好了页表。你 familiar 的`__pa()`和`__va()`宏——物理地址和虚拟地址互转——全靠这个区域撑着。早期的内核代码、数据段、Buddy分配器分配的页，大多落在这里。特点是简单、高效，不用走复杂的页表遍历。缺点是必须连续，物理上不连续的内存没法往里塞。

这就引出了第二块——**vmalloc区**。非连续映射，专门解决"大段连续虚拟地址但物理内存分散"的需求。vmalloc内部也是通过页表一级级映射的，所以访问速度比线性映射区慢一些（多几次TLB walk），但胜在灵活。ioremap映射设备寄存器、大模块的动态内存申请，都走这里。

第三块是**modules区**，加载内核模块（`.ko`文件）用的代码段和数据段区域。模块代码需要在内核空间执行，但又不能跟静态编译进内核的代码混在一起，所以单独划了一块。模块加载时，内核在这里分配虚拟地址，然后把模块的代码段、数据段、BSS段映射进去。

这么划分的根本原因是**不同的内存需求要用不同的映射策略**。线性映射区追求的是快和简单，vmalloc追求的是灵活，modules区追求的是隔离和动态。三者各司其职，内核用了几十年，这套分区逻辑一直没怎么变。

```
内核空间布局示意（32位经典模型）

0xFFFFFFFF +------------------+
           |   保留/ fixmap    |  （临时固定映射，如early ioremap）
0xFF800000 +------------------+
           |    modules区      |  （模块代码/数据，约128MB）
0xF8000000 +------------------+
           |    vmalloc区      |  （非连续映射，大小可调）
           |                   |
high_mem   +------------------+  （vmalloc与linear mapping分界线）
           |                  |
           |  线性映射区       |  （linear mapping，物理内存直接映射）
           |  （PKMAP区域在   |
           |   高端内存附近）   |
           |                  |

           |     用户空间       |

```

**知识点12 [I] /proc/pid/maps逐列解读**

好了，回到`cat /proc/self/maps`。它的输出长这样：

```
$ cat /proc/self/maps
555555554000-555555555000 r--p 00000000 08:01 1310734                    /bin/cat
555555555000-555555557000 r-xp 00001000 08:01 1310734                    /bin/cat
555555557000-555555558000 r--p 00003000 08:01 1310734                    /bin/cat
555555558000-555555559000 r--p 00004000 08:01 1310734                    /bin/cat
555555559000-55555555a000 rw-p 00005000 08:01 1310734                    /bin/cat
7ffff7a00000-7ffff7a28000 r--p 00000000 08:01 1180003                    /usr/lib/x86_64-linux-gnu/libc.so.6
7ffff7a28000-7ffff7bbd000 r-xp 00028000 08:01 1180003                    /usr/lib/x86_64-linux-gnu/libc.so.6
7ffffffde000-7ffffffff000 rw-p 00000000 00:00 0                          [stack]
...（还有heap、vdso等，省略）
```

一共六列，对照表一看就明白：

| 列 | 含义 | 示例说明 |
|---|---|---|
| address | 虚拟地址范围（起始-结束） | `555555554000-555555555000` 表示这段区域占4KB |
| perms | 权限 | `rwxp` 或 `rwxs`，见下文 |
| offset | 文件内偏移 | 对映射文件而言，从文件哪个位置开始映射 |
| dev | 设备号 | `08:01` 即主设备号8、次设备号1，对应`sda1` |
| inode | 文件inode号 | 0表示不是文件映射（如匿名映射、栈） |
| pathname | 路径或标记 | 如`/bin/cat`、`[stack]`、`[heap]`、`[vdso]` |

重点说权限列。四个字符分别是**r**ead（读）、**w**rite（写）、e**x**ecute（执行）、**p/s**。前三个好理解，第四个`p`代表**private**（私有/写时复制，COW），`s`代表**shared**（共享映射）。

`p`是最常见的。比如加载一个可执行文件，代码段和只读数据段通常是`r--p`，数据段是`rw-p`。这个`p`意味着：多个进程共享同一套物理页，但一旦某个进程试图写入，内核会给它复制一份私有的副本——这就是COW机制。`s`则出现在`mmap(MAP_SHARED)`的场景，比如共享内存、映射同一个文件的多个进程要看到彼此的修改，这时候物理页是真正的共享。

有个小窍门：如果看到权限里有`w`和`x`同时出现（`rwxp`），那多半是JIT编译器或者某些解释器的内存区域——可写又可执行，安全上属于高风险区。普通可执行文件的代码段都是`r-xp`，数据段是`rw-p`，不会把写和执行放在同一个区域。

现在你再`cat /proc/self/maps`，是不是觉得那些十六进制数字不再面目可憎了？每一行都是一个VMA（虚拟内存区域），合起来就是这张完整的进程内存地图。
