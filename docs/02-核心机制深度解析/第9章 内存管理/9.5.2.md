zswap的核心定位很清晰：当页面要被swap out时，先尝试压缩它，把压缩后的数据塞进一块预留的RAM池子里。如果压缩存不下，或者池子满了，再老老实实往磁盘swap写。这个"拦截-压缩-缓存"的逻辑，全在`zswap_frontswap_store()`这一个函数里展开。

**知识点47 [E][M] zswap_frontswap_store()的完整流程**

整个store流程可以拆成四步，但每一步里都藏着工程上的权衡。

第一步，找zpool。zswap内部维护了一个zpool实例，这是实际存放压缩后页面的地方。当前主流内核支持三种zpool后端：zsmalloc、zbud和z3fold。zsmalloc的压缩比最高，它会把不同大小的对象塞进同一个物理页框，几乎没有内部碎片，但分配路径复杂；zbud最简单，一个页框固定存两个压缩后的页面，逻辑干净但空间利用率低；z3fold折中，一个页框最多存三个页面，内部碎片比zbud好，实现又比zsmalloc简单。选哪种？看你对CPU还是内存更敏感——zsmalloc省内存但耗CPU，zbud反过来。

```c
/* mm/zswap.c - store入口 */
int zswap_frontswap_store(unsigned type, pgoff_t offset,
                         struct page *page)
{
    struct zswap_entry *entry;
    struct crypto_comp *tfm;
    void *dst;          /* 压缩后的目标缓冲区 */
    unsigned int dlen;  /* 压缩后长度 */
    int ret;

    /* 1. 从zpool申请缓冲区 */
    dst = zpool_malloc(zswap_pool, PAGE_SIZE, GFP_KERNEL, &handle);
    if (!dst)
        goto zpool_fail;   /* 池子满了，回退到磁盘swap */

    /* 2. 获取当前CPU的压缩变换器，执行压缩 */
    tfm = *get_cpu_ptr(zswap_comp_tfms);
    ret = crypto_comp_compress(tfm, src, PAGE_SIZE, dst, &dlen);
    put_cpu_ptr(zswap_comp_tfms);

    if (ret || dlen > zpool_get_total_size(zswap_pool)) {
        /* 压缩失败或者压缩完比原始还大，直接写swap */
        zpool_free(zswap_pool, handle);
        goto compress_fail;
    }

    /* 3. 创建entry并记录 (type, offset) -> handle 的映射 */
    entry = zswap_entry_find_get(type, offset);  /* 老的entry？ */
    if (!entry) {
        entry = kzalloc(sizeof(*entry), GFP_KERNEL);
        entry->offset = offset;
        entry->handle = handle;
        entry->length = dlen;
        zswap_rb_insert(&tree->rbroot, entry);   /* 插入红黑树 */
    }
    /* ... */
}
```

第二步，压缩页面。zswap用的是内核的crypto压缩API，默认算法是lzo或者zstd，可以通过`zswap.compressor`启动参数改。这里有个细节：压缩是per-CPU进行的，每个CPU独占一个`crypto_comp`实例，避免多核竞争锁。如果压缩失败，或者压缩后大小反而增加了（这种情况虽然少见，但随机数据确实可能），zswap不会死磕，直接把原始页面写回磁盘swap。

第三步，存入zpool。压缩成功后，数据被写入zpool，`zpool_malloc()`返回一个opaque的handle，这个handle不是物理地址，而是zpool内部的定位标识。不同zpool的handle含义不同——zsmalloc里它可能是一个分块编号，zbud里则是页框加slot的组合。

第四步，记录映射。zswap为每个swap分区维护一棵红黑树，键是`(type, offset)`，值就是那个handle。下次frontswap来读这个页面时，直接用type和offset查树，找到handle再读zpool。红黑树选得挺讲究：swap操作的offset天然不保证顺序，哈希表虽然O(1)但扩展开销大，红黑树的O(logN)在这种场景下反而稳当。

整个读写路径如下：

```mermaid
graph TD
    A[页面被swap out] --> B{zswap池子有空间?}
    B -->|有| C[申请zpool缓冲区]
    B -->|满| D[回退到磁盘swap]
    C --> E[per-CPU压缩页面]
    E --> F{压缩成功且缩小?}
    F -->|是| G[写入zpool]
    F -->|否| D
    G --> H[红黑树记录 (type,offset)->handle]
    H --> I[swap out完成]
    J[页面被swap in] --> K[frontswap_load]
    K --> L[红黑树查 (type,offset)]
    L --> M[从zpool读取压缩数据]
    M --> N[解压还原页面]
    N --> O[返回正确页面]

    style D fill:#f99,stroke:#333
    style I fill:#9f9,stroke:#333
    style O fill:#9f9,stroke:#333
```

> ⚠️ **陷阱**：zpool满了之后的回退路径。很多人以为zswap开了就一定会压缩，实际上当池子耗尽，页面会直接走磁盘swap——这跟没开zswap一样慢。所以`zswap.max_pool_percent`这个参数要合理设置，默认20%有时候太小。另外，压缩本身也需要时间，如果CPU已经很忙，zswap的收益会打折扣。

| zpool后端 | 压缩比 | CPU开销 | 实现复杂度 | 适用场景 |
|-----------|--------|---------|-----------|----------|
| zsmalloc | 最高 | 较高 | 高 | 内存极度紧张 |
| zbud | 较低 | 低 | 低 | CPU敏感型负载 |
| z3fold | 中等 | 中等 | 中等 | 通用折中方案 |

**知识点48 [I] zswap vs zram**

这俩经常被搞混，因为它们做的事看起来差不多——都是拿RAM来暂存swap数据。但核心差别在于：zswap有后备，zram没有。

zswap的背后始终站着一个真正的swap设备（磁盘分区或swap文件）。zpool里的压缩数据只是"缓存"，即使zswap本身出问题、或者机器掉电，数据在磁盘swap上还有一份。而zram是一个纯粹的内存块设备，它本身就没有后备存储，掉电就丢，本质上是用压缩来"扩容"RAM。

这个区别决定了它们的适用场景。如果你的系统本来就有swap分区，zswap是更好的选择——它在RAM里做压缩缓存，热点页面访问飞快，冷页面自然回落到磁盘。而zram适合那些不想碰磁盘的场景，比如嵌入式设备根本没有可靠的存储介质做swap，或者云服务器为了节省I/O费用干脆不用磁盘swap。

换个说法：zswap是swap的"加速器"，zram是swap的"替代品"。一个锦上添花，一个另辟蹊径。
