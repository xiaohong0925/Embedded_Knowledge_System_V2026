kmalloc的分配路径表面上看就一行调用，内核里却走了一条精心设计的"高速公路"。这条路由三层加速通道组成，每一层都是前一层"兜不住"时的退路。设计这套机制的人想得很明白：绝大部分分配请求都很小、很频繁，必须让热路径快到极致，至于那些 corner case，慢慢处理也无妨。

**知识点27 [E][M]**

当你在内核里写下 `kmalloc(64, GFP_KERNEL)`，真正干活的是 `slab_alloc`，而它内部又调到了 `___slab_alloc`。这个名字里三个下划线就是在告诉你——这是底层实现，别随便碰。`___slab_alloc` 内部其实是一条 if-else 链，判断条件非常简单：我本地的 freelist 还有没有？partial 链表还有没有？Buddy 系统还有没有？三层判断，对应三条路径。

第一层叫 fast path。每个 CPU 有一个 `kmem_cache_cpu` 结构，里面挂了一个 freelist 指针，指向一堆已经拆好、随时能用的对象。注意这个词——"per-CPU"，意思是这个 freelist 归当前 CPU 独占，不需要跟别的 CPU 抢，也不需要加任何锁。你要一个 64 字节的对象？直接从 freelist 头部摘一个，更新指针，返回。全程大概十来个指令，比一次函数调用还快。这个设计的聪明之处在于：内核里绝大多数 kmalloc 都是这种"来了就走"的短请求，把它们伺候好了，整体性能就不会差。

```mermaid
graph TD
    A[kmalloc(size, flags)] --> B[slab_alloc<br/>匹配kmem_cache]
    B --> C{per-CPU freelist<br/>是否非空?}
    C -->|是: fast path| D[从freelist头部摘一个对象<br/>~10条指令 无锁]
    D --> E[返回指针]
    C -->|否| F{per-CPU partial<br/>是否非空?}
    F -->|是: slow path| G[从partial链表取一个slab<br/>赋给freelist]
    G --> H[freelist更新为slab->freelist]
    H --> D
    F -->|否: fallback| I[向Buddy申请新页面]
    I --> J[初始化页面为slab<br/>建立freelist链表]
    J --> H
```

那什么时候 fast path 会失败呢？比如你连续分配了很多同尺寸的对象，把 freelist 掏空了。这时候进入第二层，slow path。内核会去翻 `kmem_cache_cpu` 里的 partial 链表——partial 链表存的是"已经用了一些、还剩一些"的 slab。从 partial 里挑一个，把它整个交给 freelist，然后回到 fast path 的逻辑再取一次。 partial 操作需要用 `kmem_cache_cpu` 的 lock，但这个锁竞争并不激烈，因为不同 CPU 各自操作各自的 partial，基本不会撞车。

如果 partial 链表也空了，那就真的没货了，只能走到第三层 fallback：向 Buddy 系统要新的物理页面。Buddy 的分配本身就有开销（页表遍历、可能的碎片整理），拿到页之后还要把它切成一个一个对象、串成 freelist 链表——这一套初始化下来，跟前面那十来个指令完全不是一个量级。所以这一层叫 fallback，意思是"兜底"，平时尽量别走到这里。

这三层路径的内核源码大体长这样（已简化，略去了调试和统计代码）：

```c
static __always_inline void *___slab_alloc(struct kmem_cache *s,
                gfp_t gfpflags, int node, unsigned long addr,
                struct kmem_cache_cpu *c)
{
    void *freelist;
    struct page *page;

    // ===== 第一层: fast path =====
    // 直接读freelist，无锁，单个CPU独享
    freelist = c->freelist;
    if (likely(freelist)) {
        c->freelist = get_freepointer(s, freelist);  // 指向下一个
        return freelist;                             // 直接返回
    }

    page = c->page;
    if (!page)
        goto new_slab;

    // ===== 第二层: slow path =====
    // freelist为空，看看partial链表
    if (slub_percpu_partial(c)) {
        page = c->page = slub_percpu_partial(c);
        slub_set_percpu_partial(c, page);   // 更新partial链表
        c->freelist = page->freelist;       // 拿到新的freelist
        goto redo;
    }

new_slab:
    // ===== 第三层: fallback =====
    // 真的没货了，找Buddy要新页
    page = new_slab_objects(s, gfpflags, node, &freelist);
    if (unlikely(!page)) {
        // 连Buddy都给不了，返回NULL
        return NULL;
    }

    c->page = page;
    c->freelist = freelist;
redo:
    // 拿到新freelist后，再走一次fast path
    freelist = c->freelist;
    c->freelist = get_freepointer(s, freelist);
    return freelist;
}
```

三个关键点记一下：

| 层级 | 触发条件 | 核心操作 | 延迟量级 | 是否加锁 |
|:---:|:---|:---|:---:|:---:|
| fast path | per-CPU freelist 非空 | 链表头摘取 | ~10条指令 | 无锁 |
| slow path | freelist空，partial非空 | 从partial取slab替换freelist | 百条指令级 | 本CPU lock |
| fallback | partial也空 | 向Buddy申请新页+初始化freelist | 千条指令级 | zone lock等 |

有趣的是，这三层路径的名字 fast/slow/fallback 并不是内核源码里的官方命名，而是社区老工程师们口头上的说法。你看源码里是看不到一个叫 `fast_path()` 的函数的，它就是前面那一段 `if (likely(freelist))` 的自然分支。命名不统一，但概念很统一——因为每个人读这段代码，脑子里都会浮现出同样的三层画面。

**知识点28 [I]**

如果你是第一次看到这段代码，可能会被那些宏定义和 type trick 绕晕。抽掉细节之后，分配逻辑的本质非常简洁——就是一层一层往下 fallback，直到拿到东西为止：

```c
// 极度简化的伪代码，展示核心 fallback 逻辑
void *my_kmalloc(struct kmem_cache *s)
{
    struct kmem_cache_cpu *c = this_cpu_ptr(s->cpu_slab);

    // Layer 1: 直接取（无锁，热路径）
    if (c->freelist) {
        void *p = c->freelist;
        c->freelist = c->freelist->next;
        return p;
    }

    // Layer 2: 从partial链表搬一个slab过来（轻量锁）
    if (c->partial) {
        struct slab *slab = pop_from_partial(c);
        c->freelist = slab->freelist;
        c->page     = slab;
        void *p = c->freelist;
        c->freelist = c->freelist->next;
        return p;
    }

    // Layer 3: 彻底没货，找Buddy系统要（重操作）
    struct page *newpage = alloc_pages(gfpflags, s->order);
    if (!newpage)
        return NULL;           // 内存不足，kmalloc返回NULL

    c->freelist = init_slab_freelist(newpage, s->size);
    c->page = newpage;

    void *p = c->freelist;
    c->freelist = c->freelist->next;
    return p;
}
```

三个 `if` 分支，对应三个世界。你平时写驱动调 kmalloc，99% 的情况都在第一个 if 里走完了。但压测或者开机高峰期， buddy 的分配压力一上来，第三层 fallback 的频率会明显升高——这时候你可能就要考虑是不是该调调 slab 的 batch count，或者干脆预分配一批对象了。

最后提一个坑：`___slab_alloc` 前面那三个下划线不只是装饰。内核里命名惯例是，一个下划线表示"内部接口，谨慎调用"，三个下划线基本就是在说"别碰，这是实现细节"。你要是写模块的时候手滑调了这个函数，编译也许能过，但后果自负。
