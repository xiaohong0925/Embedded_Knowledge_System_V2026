# 9.2.2 外碎片的形成与检测

> **版本**：v2.0
> **标签**：#memory-fragmentation #buddy-system #embedded #dma

---

## 引入

你 `cat /proc/buddyinfo` 看到一堆数字，order 0 有 500 个，order 1 有 200 个……但 order 4 以上是 0。这意味着什么？意味着你有空闲内存，但没有一块超过 64KB 的连续内存。这就是外碎片——内存管理中最隐蔽的杀手之一，尤其在 7×24 小时运行的嵌入式系统中，它会在你毫无防备的时候让 DMA 分配失败、驱动崩溃、camera 丢帧。本节将深入剖析外碎片的形成机理，并教你用 buddyinfo 这把"听诊器"提前发现危险信号。

---

## 知识点 111 [E][M]：外碎片——明明有内存，却分配不到

### 外碎片的定义

**外碎片（External Fragmentation）** 指的是：系统中**空闲内存的总量很充足**，但这些空闲页分散在物理地址空间的各个角落，**没有足够大的连续物理块**来满足分配请求。

想象一下，你有 10MB 空闲内存，但它们由成百上千个 4KB 的小块组成，中间夹杂着已分配的页面。当你需要申请一块 128KB 连续的 DMA 缓冲区时，buddy 系统遍历所有空闲页块，找不到满足 order=5（32 个连续页）的块，分配失败——尽管空闲内存总和远超 128KB。

外碎片与内碎片不同：内碎片是分配粒度造成的已分配内存内部浪费（如 slab 分配 96B 对象却占用 128B），而外碎片是**未分配内存之间的空隙浪费**。更棘手的是，buddy 系统的合并条件极为严格——只有两个相邻的、大小相同的、且物理地址连续的空闲块才能合并成更高 order 的块。一旦中间插入一个已分配的页，这对"伙伴"就永远无法合并，哪怕它们各自空了几百天。

### 嵌入式系统：外碎片的"重灾区"

相比服务器和桌面系统，**嵌入式系统是外碎片问题的高发区**，原因有三：

**1. 超长期运行，无重启缓冲**

服务器可以按月维护重启，但嵌入式设备（安防 NVR、工业网关、车载 T-Box）往往要求 7×24 小时甚至数月连续运行。内存一旦被分配、释放，留下的空洞没有机会通过重启来"清零"。随着时间推移，小空洞逐渐累积，大连续块越来越稀缺。

**2. 反复加载卸载模块**

嵌入式系统常动态加载内核模块（KO）：插入 USB 摄像头时加载驱动，拔掉时卸载；开启 AI 推理时加载 NPU 固件，结束后释放。这些模块的加载/卸载大小不一、顺序随机，像"打地鼠"一样在物理内存中制造不规则的空洞。一个 2MB 的模块卸载后，可能很快被三个 512KB 的模块割裂成无法再合并的碎片。

**3. DMA 对连续性的苛刻要求**

嵌入式外设（ISP、VPU、NPU、SDIO 控制器）大量使用 DMA，而很多旧版 DMA 控制器不支持 IOMMU/SMMU  scatter-gather，**必须依赖物理连续的内存块**。当 order≥4 的连续块耗尽时，即使还有大量零散内存，DMA 分配也会失败——这是服务器几乎不会遇到的困境。

### /proc/buddyinfo：外碎片的"体检报告"

Linux 通过 `/proc/buddyinfo` 暴露 buddy 分配器的实时状态。每一行对应一个内存 zone，各列表示不同 order 的空闲块数量：

```
$ cat /proc/buddyinfo
Node 0, zone    DMA     3     2     1     0     1     1     0     0     1     0     0 
Node 0, zone  DMA32   485   210    98    45    12     5     2     1     0     0     0 
Node 0, zone  Normal 1203   856   432   198    67    23     8     3     1     0     0 
```

从左到右，第 1~11 列分别对应 **order 0 到 order 10**。上面的 `DMA32` 行解读如下：order 0 有 485 个空闲块（每个 1 页 = 4KB），order 1 有 210 个（每块 2 页 = 8KB）……order 7 有 1 个（每块 256 页 = 1MB），**order 8 及以上为 0**（即没有 ≥2MB 的连续空闲块）。

#### 危险信号：order ≥ 4 接近 0

对于嵌入式系统，**order ≥ 4（即 64KB 以上连续块）的数量接近 0 是一个明确的危险信号**。因为：

- 常见的 DMA ring buffer、fb 缓冲区、video buffer 通常需要 64KB~512KB 连续内存
- order 4 = 16 页 = 64KB，order 5 = 32 页 = 128KB，order 6 = 64 页 = 256KB
- 一旦这些 order 的空闲块耗尽，`dma_alloc_coherent()`、`alloc_pages(GFP_DMA, order=5)` 等调用将直接失败，触发 `ENOMEM`

### Buddy Order 与页数/大小换算表

| Order | 页数（Pages） | 大小（KB，4KB/页） | 大小（MB） | 典型用途 |
|:---:|:---:|:---:|:---:|:---|
| 0 | 1 | 4 KB | — | 小对象、kmalloc-16/32 |
| 1 | 2 | 8 KB | — | 内核栈、小 DMA buf |
| 2 | 4 | 16 KB | — | 网络 sk_buff 环 |
| 3 | 8 | 32 KB | — | 音频 buffer |
| **4** | **16** | **64 KB** | — | **DMA 描述符环、小 fb** |
| **5** | **32** | **128 KB** | — | **Camera ISP 输入 buf** |
| **6** | **64** | **256 KB** | — | **VPU 中间帧、大 DMA** |
| 7 | 128 | 512 KB | 0.5 MB | NPU 权重缓存 |
| 8 | 256 | 1024 KB | 1.0 MB | 4K video 单帧（YUV） |
| 9 | 512 | 2048 KB | 2.0 MB | 大图像缓冲 |
| 10 | 1024 | 4096 KB | 4.0 MB | CMA 预留前的后备 |

> **记忆口诀**：order 4 是 64KB 分水岭，嵌入式系统必须确保 order 4~6 有充足库存。

#### 解读技巧：数字分布的形态学

健康系统的 buddyinfo 呈"**倒金字塔**"：低 order 数量多，高 order 数量少，逐级递减。如果看到：

- **断崖式下跌**：order 3 有 200 个，order 4 突然跳到 2 个——说明系统正在经历严重的碎片化，大块刚被割裂
- **顶部归零但底部庞大**：order 0~2 上千个，order 4 以上全 0——经典的外碎片场景，总量充足但无连续块
- **均匀扁平分布**：各 order 数量差不多——可能是刚做过 compaction，或系统负载极低

> **监控建议**：在嵌入式设备中，将 `cat /proc/buddyinfo | awk '{print $6,$7,$8}'`（即 order 4/5/6）加入周期性健康检查脚本，任一数值持续低于 2 即触发告警。更精细的做法是编写一个轻量的监控脚本，每隔 10 分钟采样 buddyinfo 的 order 4~6 列，计算三个 order 的总和。如果连续 3 次采样都低于阈值（如总和 < 5），则通过 syslog 上报 `FRAGMENTATION_WARNING` 事件，供云端运维平台聚合分析。这种"趋势告警"比单点阈值更能避免噪声误报。

## 知识点 112 [E]：实战案例——安防 Camera DMA 分配失败

### 故障现象

某安防 IPC（嵌入式 Linux，512MB DDR）运行 camera 抓图 + RTSP 推流，**连续运行约 48 小时后**，用户态抓图程序报错：

```
[  86420.345678] video0: dma_alloc_coherent of size 131072 failed
[  86420.345712] isp-driver: failed to allocate DMA buffer for raw frame
```

同时应用层 `ioctl(VIDIOC_DQBUF)` 返回 `ENOMEM`，camera 画面卡死，重启应用无效，**只有重启系统才能恢复**。

### 排查过程

**第一步：确认内存总量是否耗尽**

```
$ cat /proc/meminfo | grep MemFree
MemFree:          89024 kB
```

89MB 空闲，总量充足，排除 OOM。

**第二步：确认不是内存泄漏**

```
$ cat /proc/slabinfo | grep -E "^(kmalloc|dentry|inode)" | head -5
kmalloc-256      5120  5120  256   16 ...
kmalloc-1024     1280  1280 1024   4 ...
```

slab 缓存未见异常增长；`vmstat 1 5` 显示 `so`（swap out）为 0，`si`（swap in）为 0。排除内核对象泄漏和 swap 压力。

**第四步：检查 buddyinfo，定位碎片化**

```
$ cat /proc/buddyinfo
Node 0, zone  Normal  2156   987   456   123    45    12     3     0     0     0     0 
```

关键发现：
- order 0~3 数量健康（2156 / 987 / 456 / 123），说明小碎页充足
- **order 5（128KB）仅剩 3 个**，order 6（256KB）及以上**全部为 0**
- camera 驱动需要 order=5（32 页 × 4KB = 128KB）的连续 DMA 缓冲来存放一帧 RAW 图像

结论：**外碎片导致 DMA 分配失败**，不是内存泄漏。

**第五步：追溯碎片化根因**

通过 `slabinfo` 和 `vmstat` 交叉验证，发现系统在过去 48 小时内：

1.  频繁加载/卸载 `motion_detect.ko`（运动检测模块，约 2MB），在物理内存中留下不规则空洞
2.  AI 推理进程周期性启动/退出，每次分配不同大小的 NPU 输入/输出 buffer
3.  这些分配/释放不断割裂原本连续的 128KB/256KB 块，导致 order ≥ 5 的块无法维持

值得注意的是，这个设备的 motion_detect 模块原本设计为按需加载——无运动时卸载以节省内存。这种"良好意图"的设计反而成了碎片化的根源。后续将该模块改为常驻内存后，碎片化速率显著降低。

### 解决方案

| 方案 | 实施方式 | 适用场景 | 备注 |
|:---|:---|:---|:---|
| **紧急恢复** | 重启系统 | 线上故障止血 | 立竿见影，但治标不治本 |
| **手动 Compaction** | `echo 1 > /proc/sys/vm/compact_memory` | 有 console 权限的调试场景 | 触发内存压缩，可能耗时数百 ms，期间阻塞分配 |
| **启用自动 Compaction** | 调整 `vm.compaction_proactiveness` | 内核 ≥5.10 的系统 | 平衡性能与碎片化 |
| **预留 CMA** | dtb/bootargs 预留 8~16MB CMA | 有规划的产品阶段 | 治本方案，专用于大块连续 DMA 分配 |
| **改用 SG DMA** | 升级驱动使用 `dma_map_sg()` | 硬件支持 IOMMU/SMMU | 从根本上消除对连续物理内存的依赖 |

在该案例中，最终采用 **CMA 预留 + 驱动升级双管齐下**：

- bootargs 追加 `cma=16M` 为 camera/VPU 预留连续池
- ISP 驱动后续版本改用 scatter-gather DMA，彻底摆脱对物理连续内存的依赖

> **排查口诀**：DMA 失败先看 buddyinfo，总量够但大块归零 = 外碎片；重启能好、越跑越差 = 碎片累积；CMA 预留是救星，SG DMA 是根本解。

## 本节要点回顾

1. **外碎片 ≠ 内存不足**：空闲总量充足，但缺乏连续大块
2. **嵌入式是高危场景**：长期运行 + 模块反复加载 + DMA 连续性要求
3. **buddyinfo 是核心诊断工具**：重点关注 order 4~6，接近 0 即为危险信号
4. **DMA 失败排查路径**：`meminfo` 看总量 → `buddyinfo` 看连续性 → `slabinfo/vmstat` 追溯根因
5. **解决路径分级**：重启（止血）→ compaction（缓解）→ CMA 预留（预防）→ SG DMA（根治）
