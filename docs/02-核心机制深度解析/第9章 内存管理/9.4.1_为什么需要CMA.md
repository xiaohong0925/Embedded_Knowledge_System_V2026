# 9.4.1 为什么需要CMA

> **知识点121** [I][M] 嵌入式设备的大块连续内存需求与CMA的设计目标

你的camera驱动在设备运行两天后突然返回NULL，`dma_alloc_coherent`失败了。但`free`显示还有内存！为什么？这就是Buddy外碎片的威力——小碎片很多，大碎片没有。

---

## 嵌入式设备的大块连续内存困境

在嵌入式Linux系统中，有一类硬件子系统对内存有着极为苛刻的要求：**它们必须使用物理上连续的内存块**，而且通常要求的尺寸不小。典型的"大户"包括：

- **Camera ISP（图像信号处理器）**：需要为每一帧图像分配连续的物理缓冲区。1080p@30fps的RAW10格式帧，单帧就可能需要8MB以上的连续物理内存；而ISP通常需要同时维护3~5个缓冲队列，总需求量轻松突破30MB。

- **视频编解码器（VPU/Codec）**：H.264/H.265解码时，参考帧缓冲区必须是物理连续的。4K视频的参考帧可能高达16MB，多路并发时连续内存需求成倍增长。

- **DMA控制器**：尤其是没有IOMMU或Scatter-Gather能力的传统DMA引擎，要求源/目的缓冲区在物理地址空间绝对连续。很多嵌入式SoC的DMA只支持32位地址甚至更低，进一步压缩了可用空间。

这些需求都有一个共同特征：**在设备初始化或运行过程中，必须一次性获得大块连续物理内存，否则硬件根本无法工作**。

---

## Buddy系统的"阿喀琉斯之踵"：外碎片

Linux内核默认的物理内存管理器是Buddy（伙伴）系统。Buddy系统的设计目标是高效管理不同大小的内存块（以2的幂次方的order来组织），但它有一个众所周知的弱点：**外碎片（External Fragmentation）**。

外碎片的含义是：系统中空闲的物理内存总量并不少，但这些空闲页散落在各处，无法组合成一块足够大的连续区域。

让我们看一个真实的生产案例。某安防IP Camera设备基于ARM Cortex-A7 + 256MB DDR，运行嵌入式Linux，Camera ISP需要分配order=5（128KB）的DMA缓冲区用于图像帧缓冲。

**设备刚启动时**，`/proc/buddyinfo`显示：

```
Node 0, zone   Normal
  520  210   85   40   18    8    4    2    1    0    0
```

此时order=5有8个空闲块（每个128KB），DMA分配请求轻而易举就能满足。

**设备连续运行48小时后**，同样的命令输出变为：

```
Node 0, zone   Normal
  1840  920  410  180   65   22    6    1    0    0    0
```

注意到什么了吗？order=5的空闲块从8个暴跌到只剩1个，而**order=5及以上的大内存块全部归零**。与此同时，系统空闲内存总量仍然有约50MB——但这些内存被切割成了成百上千的小块（order=0有1840个4KB页，约7.2MB；order=1有920个8KB块……），没有任何一块能满足128KB的连续分配请求。

最终，Camera驱动的`dma_alloc_coherent(GFP_DMA, 128KB)`返回NULL，视频流中断，系统日志中出现：

```
[isp] dma_alloc_coherent failed for frame buffer (size=131072, flags=0x8000)
[isp] Failed to initialize capture buffer queue
```

这就是Buddy外碎片的典型症状。在内存分配/释放的长期交替中，大块内存被不断拆分、分配、释放后，释放回来的小页被填充到不同位置，再也无法合并回原始的大块。**时间越长，外碎片越严重，大块连续内存的分配成功率越低**。

传统的解决方案（如`compact_memory`手动触发内存规整、调整`min_free_kbytes`等）都有明显的局限性：规整操作开销大且无法保证100%成功；预留静态DMA缓冲池则严重浪费内存——那块预留的大区域在Camera不工作时完全闲置。

---

## CMA的解题思路：可迁移的连续内存预留

Contiguous Memory Allocator（CMA，连续内存分配器）正是为解决这一矛盾而生。CMA的核心设计思想非常巧妙：**在内核启动时预留一片连续的物理内存区域，但这段内存在平时可以被普通的 movable（可迁移）页分配使用；只有当需要大块连续内存时，才通过页迁移技术将上面的可迁移页搬到别处，从而"腾出"一片干净的连续区域**。

换句话说，CMA同时实现了两个目标：

1. **有连续内存可用**：预留区域保证了在任何时候都有一段足够大的连续物理地址空间，不会被不可迁移的分配打散。

2. **不浪费内存**：预留区域在空闲时可以被文件页缓存、匿名页等可迁移内容使用，而不是空转。

在ARM等嵌入式架构中，CMA通常通过设备树（Device Tree）的`linux,cma`节点来配置，可以精确指定预留区域的大小和位置。当驱动调用`dma_alloc_from_contiguous()`时，CMA子系统负责页迁移和分配，整个过程对驱动透明——驱动仍然调用熟悉的`dma_alloc_coherent()`接口，只是背后多了一层"紧急备用"的连续内存保障。

理解了CMA的"为什么"，下一节我们就深入到内核源码，看看这片"可迁移的连续区域"究竟是如何被管理和回收的。

---

**关键要点总结：**

| 要点 | 说明 |
|------|------|
| 需求来源 | Camera ISP、VPU编解码、传统DMA都需要大块连续物理内存 |
| 核心痛点 | Buddy外碎片导致长期运行后大块连续分配必然失败，与系统总空闲量无关 |
| 典型症状 | `/proc/buddyinfo`高order空闲块为0，`dma_alloc_coherent`返回NULL |
| CMA设计目标 | 预留可迁移连续区域，平时不浪费、需要时可腾出，兼顾连续性与利用率 |
