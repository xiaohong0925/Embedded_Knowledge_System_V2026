**知识点33 [E]**

SLUB作为默认分配器，内核给它配了一套相当实用的调试组合拳——`CONFIG_SLUB_DEBUG`。你碰到内存损坏、UAF或者想抓个泄漏现场的时候，这套东西比从头打kprobe舒服多了。三板斧：`SLUB_POISON`、`red zone`和`tracking`。

第一板斧，**SLUB_POISON**。给释放的对象填充固定魔数（`0x6b6b6b6b`，也就是ASCII的`kkkk`），分配的未初始化对象填`0x5a5a5a5a`。这个设计思路很朴素：如果你看到代码里读出了一个满屏`kkkk`的值，八成就是有人拿着 dangling pointer 在读已经 free 的内存。来看看释放时的填充逻辑：

```c
// mm/slub.c — slab_free_hook()
static __always_inline void slab_free_hook(struct kmem_cache *s, void *x)
{
    // ...
    if (!(s->flags & SLAB_TYPESAFE_BY_RCU))
        memset(kasan_reset_tag(object), POISON_FREE, s->object_size);
}
```

POISON_FREE 在源码里就是 `0x6b`，内核叫它 `SLUB_RED_ACTIVE` 的反向逻辑——其实就是让你一眼认出"这块地已经归还了"。

第二板斧，**red zone**（红区）。在每个对象的屁股后面加一段保护区，填充固定模式值。如果有人踩过了对象边界去写，这段红区就会变脏，SLUB在下次分配时一检查就能抓个现行。布局很简单：

```
┌──────────────┬──────────────┬──────────────┐
│   对象数据    │   red zone   │   对齐填充    │
│  object_size │  sizeof(void*)│   padding    │
└──────────────┴──────────────┴──────────────┘
```

第三板斧，**tracking**。记录每次分配和释放的调用栈。这个是通过在 `kmem_cache` 里挂 `struct track` 实现的，出事以后用 `slabinfo -v` 或者直接看 `/sys/kernel/slab/xxx` 下面的 trace 文件就能知道这块内存是谁申请的、谁放的。

那怎么开？内核配置里勾选 `CONFIG_SLUB_DEBUG=y`，启动参数根据你需要加：

```bash
# 全开
slub_debug=PZT

# 含义：
# P = SLUB_POISON   — 填充已释放对象
# Z = SLUB_RED_ZONE — 启用red zone检查
# T = SLUB_TRACK    — 记录分配/释放栈

# 也可以只针对某个slab开，比如只给 kmalloc-256 开
slub_debug=PZT,kmalloc-256
```

> **陷阱**：`slub_debug` 参数里如果写了不认识的字母，内核不会报错，而是直接静默忽略。我就见过有人写了 `slub_debug=POISON` 然后纳闷为啥没效果——得写 `P`，不是单词全称。

**知识点34 [I]**

调试是好，但这东西不是白给的。开了 `SLUB_POISON` 和 `red zone` 之后，每个对象至少要多出一两个 cache line 的填充；`tracking` 更狠，每个对象背后挂一个 `struct track`，64位上占几十个字节。综合下来，分配速度大概掉两到五成，内存占用也会往上蹿一截。数字不用记那么死，核心就一条：**这是调试手段，不是运行态配置**。我见过有人在压力测试环境里常年挂着 `slub_debug=PU`，性能指标一直上不去，排查了半天才发现是这玩意儿在拖后腿。

> **陷阱**：有些发行版（比如某些调试版内核）默认开了 `CONFIG_SLUB_DEBUG_ON=y`，也就是一启动就自动启用全套调试。如果你发现你的虚拟机/slab性能莫名其妙比物理机差很多，先看看 `slub_debug` 生效了没有——`cat /sys/kernel/slab/options` 一眼就能确认。
