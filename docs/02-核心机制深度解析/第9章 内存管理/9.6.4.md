OOM发生了，dmesg跳出来一堆信息。满屏的数字和进程名，看得人头皮发麻。那堆输出到底在说什么？别急，我来逐行翻译。

**知识点59 [I][M]**

oom_kill事件的入口在`mm/oom_kill.c`的`dump_header()`。它会往内核日志里输出三段关键信息。先看一段真实的例子：

```
[  123.456] Out of memory: Kill process 2847 (myapp) score 912 or sacrifice child
[  123.457] Killed process 2847 (myapp) total-vm:1048576kB, anon-rss:823456kB,
           file-rss:2048kB, shmem-rss:0kB, UID:1000 pgtables:2048kB oom_score_adj:0
[  123.458] oom_reaper: reaped process 2847 (myapp), now anon-rss:0kB
```

逐行拆解——

**第一行：判决宣告。**
`"Kill process 2847 (myapp) score 912"`——进程2847（myapp）的oom_score是912，满分1000，这差不多是板上钉钉要死。注意到那个略带惊悚的`"or sacrifice child"`，其实是内核在告诉你：如果不杀这个进程，就得杀掉它的子进程来凑内存。Linux的OOM killer遵循"杀父留子"还是"杀子留父"有一套内部逻辑，但通常你看到的是父进程直接被毙掉。

**第二行：验尸报告。**
这是信息密度最大的一行，各个字段的含义对照下表：

| 字段 | 含义 | 排查关注点 |
|------|------|-----------|
| total-vm | 进程总虚拟地址空间 | 一般很大，不用紧张 |
| anon-rss | 匿名映射驻留内存（堆、栈、malloc）| **重点关注** — 持续增长的anon-rss是内存泄漏的典型特征 |
| file-rss | 文件映射缓存（代码段、so库、mmap文件）| 通常稳定，异常高说明大量文件映射 |
| shmem-rss | 共享内存（tmpfs、shmem段）| 高说明tmpfs或IPC用得狠 |
| pgtables | 页表本身占用的内存 | 在大量小页映射时可能显著增长 |
| oom_score_adj | 用户可调的调整值（-1000到1000）| 如果值被人为调低，可能是开发在"掩耳盗铃" |

**第三行：oom_reaper扫尾。**
这是较新版本内核引入的优化。oom_reaper是一个内核线程，负责异步回收被杀死进程的匿名内存页，避免在进程退出路径上长时间阻塞。`"now anon-rss:0kB"`说明清理完毕。

说到这儿，有个陷阱很多人踩过——盯着total-vm发呆。你看上面例子中total-vm是1GB，但实际吃内存的是anon-rss的800多MB。total-vm包含了未映射的虚拟地址、mmap预留区等虚胖成分，看那个数字只会自己吓自己。真正决定进程会不会被OOM killer盯上的，是rss相关字段的加权和。

```
+------------------+     +------------------+     +------------------+
|   OOM触发        | --> |  dump_header()   | --> |  oom_reaper回收   |
|  (alloc_pages    |     |  输出判决+内存   |     |  异步清理匿名页   |
|   失败)          |     |  状态到dmesg     |     |  避免exit阻塞     |
+------------------+     +------------------+     +------------------+
```

**知识点60 [I]**

日志到手了，怎么反推根因？分三条路走——

**路一：看有没有"肥仔"。** 如果某个进程的anon-rss占到系统内存的很大一部分，而且它的total-vm和anon-rss差值很小（说明虚拟内存基本都落实了），这就是典型的内存泄漏现场。结合`/proc/<pid>/smaps`看哪个vma在膨胀，再对比代码找malloc/realloc的漏洞。

**路二：看buddyinfo判断碎片。** OOM日志本身不直接输出buddy信息，但你可以对照同时刻的`/proc/buddyinfo`。如果高order（比如order-3及以上）全部为零，只有低order有页，说明系统没缺内存，只是缺*连续*的内存。这种情况在大页或DMA申请时尤其致命——明明还有几百MB空闲，却触发OOM。

**路三：看是不是CMA背锅。** 如果OOM发生在驱动模块、相机或者GPU相关的场景，查一下`/proc/cma_info`。CMA区域虽然会计入系统总内存，但对普通分配不可见。如果CMA被耗尽而普通zone还有余量，内核也可能走OOM路径。我见过一个案例：平板上相机驱动反复打开失败，最后OOM killer出来背锅，实际是CMA被之前崩溃的相机实例占着没释放。

> 陷阱：不要把"Killed process"和进程自身崩溃混为一谈。被OOM killer杀掉的进程退出码是9（SIGKILL），而且dmesg里会有明确的oom_kill记录。如果应用日志里只看到"Killed"但dmesg没有OOM记录，那可能是systemd的OOM killer（systemd-oomd）或者外部监控脚本干的，别查错方向。
