# 7.6.2 根文件系统的挂载过程

> 所属：第7章 启动流程与初始化 > 7.6 根文件系统挂载
> 难度：[I→E] | 预计阅读时间：25分钟

## 本节导读

当内核完成子系统初始化后，面临一个根本性的悖论：**挂载根文件系统需要读取文件，但读取文件又需要先挂载根文件系统**。本节深入分析内核如何破解这一"鸡生蛋"问题，从 `prepare_namespace()` 的调用链到 `initramfs` 的解压机制，再到 `pivot_root` 完成向真实根文件系统的切换——覆盖内核源码 `init/do_mounts.c` 中的完整挂载路径。

## 知识点1：挂载根文件系统 [E] ~1200字

### 问题场景

嵌入式设备上电后，内核已经解压并运行，`start_kernel()` 完成了内存管理、进程调度、VFS等核心子系统的初始化。但此时系统中还没有任何已挂载的文件系统——`/dev/console` 只是设备号 `(5,1)` 的抽象，`init` 程序还躺在某个存储介质上无法访问。

**核心问题**：内核如何从一块裸的存储介质（eMMC/SD/NAND/NOR/网络）上找到并挂载根文件系统，然后执行 `/sbin/init`？

### 机制深入：prepare_namespace() 调用链

挂载根文件系统的入口在 `init/main.c` 中。`kernel_init()` 在释放 `initmem` 之前调用 `prepare_namespace()`：

```c
/* init/main.c - kernel_init() 代码片段 */
if (execute_command)
    ret = run_init_process(execute_command);
...
/* 在尝试执行init之前，必须先有根文件系统 */
if (!ramdisk_execute_command)
    ramdisk_execute_command = "/init";
prepare_namespace();    /* <-- 挂载根文件系统的入口 */
```

`prepare_namespace()` 的执行流程遵循严格的优先级策略：

```mermaid
graph TD
    A[kernel_init] --> B[prepare_namespace]
    B --> C{root_delay 参数?}
    C -->|设置了| D[msleep 等待]
    C -->|未设置| E[mount_root]
    D --> E
    E --> F{root=?}
    F -->|/dev/ram| G[mount_block_root<br/>从initrd挂载]
    F -->|/dev/mmcblk0p2| H[mount_block_root<br/>从块设备挂载]
    F -->|nfs| I[mount_nfs_root]
    F -->|空或/dev/nfs| J[try_nfs_root]
    F -->|无root参数| K[mount_block_root<br/>尝试默认设备]
    G --> L[do_mount_root<br/>实际执行mount]
    H --> L
    I --> L
    J -->|失败| M[try_md<br/>尝试其他介质]
    K --> M
    M -->|都失败| N[panic]
    L --> O[initrd_load]
    O --> P[mount_root
          最终挂载根FS]
    P --> Q[/sbin/init]
```

**关键代码路径 `prepare_namespace()` → `mount_block_root()` → `do_mount_root()`：**

```c
/* init/do_mounts.c - prepare_namespace() */
void __init prepare_namespace(void)
{
    int is_floppy;

    /* 1. 如果配置了rootdelay，等待设备就绪（关键！） */
    if (root_delay) {
        printk(KERN_INFO "Waiting %d sec before mounting root device...\n",
               root_delay);
        ssleep(root_delay);
    }

    /* 2. 创建 /dev/root 符号链接（fs_name_block() 或 fs_name_nfs()） */
    /* root_device_name 由 root= 参数解析得到 */
    is_floppy = MAJOR(ROOT_DEV) == FLOPPY_MAJOR;

    /* 3. 如果有initrd，先处理（见知识点2） */
    if (initrd_load())
        goto out;

    /* 4. 真正的根设备挂载 */
    mount_root();       /* --> mount_block_root() 或 mount_nfs_root() */
out:
    /* 5. 挂载devtmpfs到/dev（如果启用） */
    devtmpfs_mount("dev");
    sys_mount(".", "/", NULL, MS_MOVE, NULL);  /* 将当前目录移到/ */
    sys_chroot(".");    /* 切换根目录到当前目录 */
}
```

`mount_root()` 根据 `ROOT_DEV` 的值分派到不同路径：

```c
/* init/do_mounts.c - mount_root() */
void __init mount_root(void)
{
    /* ... */
#ifdef CONFIG_BLOCK
    if (ROOT_DEV == 0) {
        /* 未指定root=参数，尝试从默认块设备挂载 */
        if (mount_nfs_root())
            return;
        /* ... 尝试识别根设备 ... */
    }
    mount_block_root(root_device_name, root_mountflags);
#endif
}
```

**`do_mount_root()` 是最终的执行者**，它直接调用 `sys_mount()`：

```c
/* init/do_mounts.c - do_mount_root() */
static int __init do_mount_root(char *name, char *fs, int flags, void *data)
{
    int err = sys_mount(name, "/root", fs, flags, data);
    if (err)
        return err;
    
    /* 将 /root 挂载点移到当前进程的根目录 */
    sys_chdir("/root");
    ROOT_DEV = current->fs->pwd.dentry->d_sb->s_dev;
    printk(KERN_INFO "VFS: Mounted root (%s filesystem) on device %u:%u.\n",
           current->fs->pwd.dentry->d_sb->s_type->name,
           MAJOR(ROOT_DEV), MINOR(ROOT_DEV));
    return 0;
}
```

⚠️ **常见陷阱**：`sys_chdir("/root")` 只是改变了 `init` 进程的当前工作目录到 `/root`，**还没有改变全局的根目录**。真正的根目录切换发生在 `prepare_namespace()` 末尾的 `sys_mount(".", "/", NULL, MS_MOVE, NULL)` 和 `sys_chroot(".")` 组合调用中。

### 根文件系统来源对比表

| 来源类型 | `root=` 参数示例 | 设备识别 | 适用场景 | 启动速度 | 内核配置依赖 |
|---------|-----------------|---------|---------|---------|------------|
| **initramfs** | 无需（或 `root=/dev/ram0`） | 内存地址 `__initramfs_start` | 早期用户态、磁盘解密、启动修复 | ⚡ 最快（内存直接访问） | `CONFIG_BLK_DEV_INITRD=y`, `CONFIG_INITRAMFS_SOURCE` |
| **eMMC/SD块设备** | `root=/dev/mmcblk0p2` | 主设备号179，动态发现 | 绝大多数嵌入式产品 | 🐢 依赖存储速度 | 对应块设备驱动（如 `CONFIG_MMC_SDHCI`） |
| **MTD/NOR Flash** | `root=/dev/mtdblock3` | MTD子系统映射 | 低成本IoT设备、工控 | 🐢 受Flash读取速度限制 | `CONFIG_MTD_BLOCK=y` |
| **NFS** | `root=/dev/nfs nfsroot=192.168.1.100:/exports` | 网络协议栈 | 开发调试、无盘工作站 | 🐢🐢 依赖网络 | `CONFIG_ROOT_NFS=y`, `CONFIG_IP_PNP` |
| **UBIFS** | `root=ubi0:rootfs ubi.mtd=3` | UBI卷管理 | 大容量NAND Flash | 🐢 受NAND读取速度限制 | `CONFIG_UBIFS_FS=y` |
| **USB大容量存储** | `root=/dev/sda1` | SCSI/USB子系统 | 外接存储启动 | 🐢 受USB枚举速度限制 | `CONFIG_USB_STORAGE=y` |

🔴 **安全提醒**：在生产环境中使用 NFS 根文件系统时，务必配合 NFSv4 + Kerberos 或至少使用 `root_squash` 选项，否则 init 进程将以 root 权限通过不受保护的网络运行。

### 实践案例：eMMC 分区块设备挂载

某工业网关设备（NXP i.MX6ULL）使用 8GB eMMC，分区布局如下：

```
eMMC (mmcblk0)
├── p1: 32MB  vfat  /boot    (kernel dtb)
├── p2: 512MB ext4  /        (rootfs)
└── p3: 剩余   ext4  /data    (用户数据)
```

U-Boot 传递的启动参数：

```bash
# U-Boot bootargs 设置
setenv bootargs 'console=ttymxc0,115200 root=/dev/mmcblk0p2 rootfstype=ext4 \
    rootwait rw rootflags=data=ordered'
```

**`rootwait` 参数的本质**：`prepare_namespace()` 中并没有显式处理 `rootwait`。实际上，当块设备驱动采用异步探测（如 MMC 子系统的 `mmc_rescan` 工作队列）时，**`mount_block_root()` → `do_mount_root()` → `sys_mount()` 链可能在设备尚未就绪时就被调用**。`rootwait` 的作用是在用户态 init 脚本中轮询等待设备节点出现。如果未加 `rootwait`，可能遇到：

```
VFS: Cannot open root device "mmcblk0p2" or unknown-block(179,2): error -6
Please append a correct "root=" boot option; here are the available partitions:
```

💡 **技巧**：在调试时，添加 `initcall_debug` 和 `rootdelay=5` 可以清晰看到块设备驱动 probe 和 mount 的时序关系。

## 知识点2：initramfs [E] ~1200字

### 问题场景

设想一个加密磁盘场景：根文件系统位于 LUKS 加密的 eMMC 分区上。内核启动时需要加载 cryptsetup 来解密——但 cryptsetup 本身就在加密的根文件系统里。如何打破这个死锁？

**initramfs（初始 RAM 文件系统）就是答案**：它是一个嵌入在内核镜像中的小型文件系统，在真实根文件系统可用之前提供早期用户态环境。

### 机制深入：为什么 initramfs 总是先被挂载？

initramfs 的优先级高于任何块设备或网络根文件系统。内核保证 initramfs 在 `prepare_namespace()` 的早期就被挂载，因为它解决了三个根本性难题：

1. **驱动加载悖论**：真实根文件系统所在设备（如 NVMe、SAS RAID）的驱动可能未被编译进内核，而是以模块形式存在于根文件系统上——根文件系统又需要这个驱动才能访问。
2. **存储栈复杂性**：现代存储可能涉及 LVM → LUKS → MD RAID → NVMe 的多层抽象，每层都需要用户态工具初始化。
3. **内核体积约束**：将所有可能的根设备驱动都静态编译进内核会使镜像体积膨胀数倍。

**initramfs 的数据嵌入**：在链接阶段，链接器脚本 `arch/arm/kernel/vmlinux.lds.S`（以 ARM 为例）定义了两个符号：

```c
/* include/asm-generic/vmlinux.lds.h - initramfs 段定义 */
#define INIT_RAM_FS                            \
    .init.ramfs : AT(ADDR(.init.ramfs) - LOAD_OFFSET) {  \
        __initramfs_start = .;                  \
        *(.init.ramfs)                          \
        __initramfs_end = .;                    \
    }
```

构建时，`usr/Makefile` 调用 `gen_init_cpio` 将 `CONFIG_INITRAMFS_SOURCE` 指定的文件打包成 cpio 归档，然后经 `gzip`/`xz`/`lz4` 压缩后链接到 `.init.ramfs` 段。内核启动时，这段数据已经位于内存中，**不需要任何块设备驱动**。

**`populate_rootfs()` 解压到 tmpfs**：

```c
/* init/initramfs.c - populate_rootfs() */
static int __init populate_rootfs(void)
{
    char *err;

    /* 1. 初始化 decompress 缓冲区 */
    err = initrd_load();
    if (err)
        return 0;

    /* 2. 如果内核配置了内置 initramfs */
    if (__initramfs_size >= 8) {
        /* 将压缩数据解压到根目录 */
        if (crc32(0, __initramfs_start, __initramfs_size) != INITRAMFS_CRC)
            printk(KERN_ERR "initramfs: CRC check failed\n");
        else
            err = unpack_to_rootfs(__initramfs_start, __initramfs_size);
    }
    /* 3. 如果还有外部 initrd（遗留的 ramdisk 机制），也处理 */
    /* ... */
    return 0;
}
rootfs_initcall(populate_rootfs);   /* 在 core_initcall 之后执行 */
```

`unpack_to_rootfs()` 将 cpio 归档逐条解压到 `rootfs`（一个特殊的 tmpfs 实例），创建目录、设备节点、符号链接和普通文件。解压完成后，VFS 的根目录 `/` 已经是一个可用的文件系统——尽管它只在内存中。

⚠️ **常见陷阱**：`populate_rootfs()` 使用 `rootfs_initcall` 修饰符，这意味着它在 initcall 序列中优先级很高。但如果某个 `core_initcall`（如 tmpfs 初始化 `shmem_init()`）失败，`unpack_to_rootfs()` 会因为 `rootfs` 不可用而静默失败，留下一个空的 `/`。

### initramfs 工作机制速查表

| 属性 | 说明 |
|------|------|
| **存储位置** | 内核镜像的 `.init.ramfs` 段，物理内存中 |
| **文件系统类型** | `rootfs`（tmpfs 的特殊实例），挂载在 `/` |
| **压缩格式** | gzip（默认）、bzip2、lzma、xz、lzo、lz4、zstd |
| **归档格式** | newc/sv4cpio cpio 格式 |
| **大小限制** | 受限于内存，典型 2-20MB |
| **生命周期** | 默认在 `pivot_root` 后被丢弃（或保留在内存） |
| **配置路径** | `CONFIG_INITRAMFS_SOURCE`（文件路径或 cpio 归档） |
| **替代机制** | 遗留 initrd（已废弃，使用独立的内存块设备） |

### 实践案例：磁盘加密启动流程

某金融终端设备要求全盘加密，启动流程如下：

```bash
# initramfs 中的 /init 脚本（简化版）
#!/bin/busybox sh

# 1. 挂载基础虚拟文件系统
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev

# 2. 等待加密设备出现
while [ ! -e /dev/mmcblk0p2 ]; do
    sleep 0.1
done

# 3. 交互式获取密钥（或通过 TPM）
cryptsetup luksOpen /dev/mmcblk0p2 rootfs_crypt

# 4. 激活 LVM（如有）
vgchange -ay vg0

# 5. 挂载真实根文件系统
mount /dev/mapper/vg0-root /mnt/root

# 6. 切换到真实根文件系统（见知识点3）
exec switch_root /mnt/root /sbin/init
```

这个流程中，`/init` 脚本执行时所处的 `/` 就是 initramfs 提供的 tmpfs。所有必要的工具（`busybox`、`cryptsetup`、`lvm`）都静态链接后打包进 initramfs。

## 知识点3：pivot_root [E] ~1200字

### 问题场景

initramfs 成功解压，早期 `/init` 完成了它的使命（加载驱动、解密磁盘、挂载真实根到 `/mnt/root`）。现在需要把系统从临时的内存文件系统切换到持久化的真实根文件系统上。

**最直接的想法**：在真实根上执行 `chroot("/mnt/root")`？不行——`chroot` 只改变当前进程的根目录视图，**不会重新挂载 `/` 本身**。initramfs 中的进程仍然运行在内核初始的 `rootfs` 上。

### 机制深入：为什么用 pivot_root 而不是直接 mount？

`pivot_root` 的设计目标是将 **整个系统** 的挂载根从 A 切换到 B，而不是单个进程的根目录视图。它的核心语义是：

1. 将 **当前的根目录** 移动到某个 **新位置**（如 `/initrd`）
2. 将 **指定的新目录** 提升为 **新的根目录**
3. 所有进程（至少是 init 及其后代）自然过渡到新根上

**`pivot_root` 系统调用实现**：

```c
/* fs/namespace.c - sys_pivot_root() */
SYSCALL_DEFINE2(pivot_root, const char __user *, new_root,
        const char __user *, put_old)
{
    /* 1. 安全检查：new_root 和 put_old 必须是目录 */
    /* 2. new_root 不能是当前根或当前工作目录的祖先 */
    /* 3. put_old 必须在 new_root 之下 */
    /* 4. 两个目录不能在同一挂载点上 */
    /* 5. new_root 必须是挂载点（如果不是，隐式 bind mount） */
    /* 6. 执行挂载点交换 */
    ...
    /* 关键：交换 vfsmount 的 mnt_root 和 mnt_mountpoint */
    attach_mnt(new_mnt, real_mount(old_parent), mountpoint);
    ...
}
```

`pivot_root` 不是简单的目录切换——它 **操作的是挂载树结构**。`attach_mnt()` 将 `new_mnt`（真实根的 vfsmount）连接到挂载树的根部，同时将旧的根挂载移到 `put_old` 位置。

### pivot_root vs chroot 对比表

| 维度 | `pivot_root` | `chroot` |
|------|-------------|----------|
| **操作对象** | 挂载树（mount tree），全局影响 | 单个进程的根目录（`fs->root`） |
| **对 `/` 的影响** | 重新挂载 `/`，所有进程可见新根 | 仅当前进程看到新根 |
| **旧根处理** | 必须指定放置位置（`put_old`），可后续 umount | 旧根仍在挂载树中，进程可能 `chdir..` 逃逸 |
| **使用场景** | initramfs → 真实根切换、容器运行时 | 进程沙箱、FTP chroot jail、简单隔离 |
| **安全强度** | 强（旧根可完全卸载） | 弱（root 用户可 `chdir("/")` 回到真实根） |
| **调用权限** | CAP_SYS_ADMIN | CAP_SYS_CHROOT |
| **参数要求** | new_root 必须是挂载点，put_old 是其子目录 | 任意目录即可 |
| **经典用户** | `switch_root`（util-linux）、systemd、Docker | `sshd`、`vsftpd`、`proftpd` |

### 关键代码路径：switch_root 的实现

嵌入式系统中最常用的是 `busybox` 的 `switch_root` 命令，其实现逻辑等价于：

```c
/* 伪代码：switch_root 等价逻辑 */
int switch_root(const char *new_root, const char *init)
{
    /* 1. 确保 new_root 是绝对路径的挂载点 */
    /* 2. 将当前根（initramfs）移到 new_root 下的 /initrd */
    mount(".", "/", NULL, MS_MOVE, NULL);        /* 移动当前目录 */
    pivot_root(".", ".");                         /* 交换 . 和 . 自己 */
    /* 更标准的实现： */
    pivot_root(new_root, put_old_path);           /* 核心：交换根 */
    
    /* 3. 切换到新的根目录 */
    chdir("/");
    
    /* 4. 卸载旧的 initramfs（释放内存） */
    umount2("/initrd", MNT_DETACH);               /* 或 umount2(".", MNT_DETACH) */
    
    /* 5. 执行真实根上的 init */
    execl("/sbin/init", "init", NULL);
}
```

💡 **技巧**：在内核源码的 `prepare_namespace()` 中，你也可以看到类似的模式——只是发生在内核态：

```c
/* init/do_mounts.c 末尾 */
sys_mount(".", "/", NULL, MS_MOVE, NULL);   /* 将 /root 移到 / */
sys_chroot(".");                             /* 确认新的根 */
```

注意内核使用的是 `MS_MOVE` + `chroot` 组合，而不是 `pivot_root`。**原因**：`pivot_root` 要求调用者是新根下的进程，而内核此时还在初始的 `rootfs` 上下文中。

⚠️ **常见陷阱**：

1. **忘记卸载旧根**：`switch_root` 后如果不 `umount2("/initrd", MNT_DETACH)`，initramfs 的内存将永远被占用（因为还有引用计数）。在内存紧张的嵌入式设备上，这会浪费数 MB RAM。

2. **文件描述符泄漏**：如果 `/init` 脚本打开了 `/dev/console` 或其他文件，在 `exec switch_root` 之前应该关闭它们。否则这些文件描述符指向旧的 initramfs 中的文件，阻止旧根卸载。

3. **`pivot_root` 后进程还在旧根上**：只有执行了 `chdir("/")` 的进程才在新根上。如果某些内核线程或后台进程仍持有旧目录引用，`umount` 会返回 `EBUSY`。

### 实践案例：initramfs 到 ext4 根的完整切换时序

以下是通过 `initcall_debug` 和 printk 追踪到的 ARM64 嵌入式板完整时序：

```
[    0.000000] Kernel: arch/arm64/boot/Image
[    0.456789] calling  populate_rootfs+0x0/0x120 @ 1      <-- rootfs_initcall
[    0.458012] initramfs: unpacking to rootfs...
[    0.512345] initramfs: done (4.2MB unpacked to 12.8MB)
[    0.600000] Freeing initrd memory: 4204K
[    1.234567] /init started (pid=1, from initramfs)
[    1.300000] mmcblk0: mmc0:0001 DG4016 14.7 GiB          <-- eMMC 就绪
[    1.450000] EXT4-fs (mmcblk0p2): mounted filesystem with ordered data mode
[    1.451000] VFS: Mounted root (ext4 filesystem) on device 179:2.

[    1.500000] systemd[1]: systemd 247 running in system mode  <-- 真实 init
```

这个时序清晰展示了：**initramfs 的生命周期仅约 1 秒**，它在这段时间内完成了 eMMC 驱动加载、设备枚举、根文件系统挂载，然后通过 `switch_root` 优雅退出。

---

## 本节总结

根文件系统的挂载是内核到用户态的"最后一公里"，涉及三个精密协作的机制：

1. **`prepare_namespace()` → `do_mount_root()` 调用链** 负责最终执行 mount 系统调用，支持块设备、NFS 等多种来源。`rootwait` 和 `rootdelay` 是解决异步设备探测时序问题的关键参数。

2. **initramfs 通过 `populate_rootfs()` 解压到 tmpfs**，提供早期用户态环境。它嵌入在内核镜像的 `.init.ramfs` 段中，**不需要任何块设备驱动**即可访问，从而破解了"驱动在根文件系统上 / 根文件系统需要驱动"的死锁。

3. **`pivot_root` 通过操作挂载树结构**（而非进程级 `chroot`），完成从 initramfs 到真实根的系统性切换。`switch_root` 工具封装了这一过程，但需注意及时卸载旧根以释放内存。

对于嵌入式工程师，理解这些机制的意义在于：能够根据产品需求（启动速度、安全性、存储介质）选择正确的根文件系统策略，并在启动失败时从串口日志中精确定位问题所在层级。
