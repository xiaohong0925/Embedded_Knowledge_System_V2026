## 三、查漏补缺 — 5个未被核心问题覆盖但重要的速查知识点

### 速查1：eFuse烧录与Secure Boot链的完整性验证

- **一句话解释**：eFuse是一次性可编程熔丝阵列，烧录后不可逆，用于存储公钥哈希或启动配置位，构成Secure Boot信任根。
- **为什么重要**：Secure Boot的安全性完全依赖eFuse的不可篡改性；烧录错误的公钥哈希会导致设备永远无法验证合法镜像，成"砖头"。
- **快速参考**：
  - eFuse位宽：通常256-bit(哈希) + 32-bit(配置)
  - 关键配置位：SECURE_BOOT_EN、BOOT_MODE_SEL
  - 烧录命令示例：`fuse prog 0 0x0 <hash_value>` (i.MX系列)
  - 验证是否启用：`fuse sense 0 0` 读取状态位
- **全书螺旋关联**：与第5章「ARM64体系结构」特权级/安全态关联 → 第7章BootROM签名验证 → 第12章「安全机制」TrustZone/Secure Boot深度展开

---

### 速查2：U-Boot环境变量存储位置与持久化机制

- **一句话解释**：U-Boot环境变量可以存储在MMC/SPI-NOR/NAND/EEPROM等多种介质中，通过`saveenv`命令持久化，启动时自动加载。
- **为什么重要**：环境变量控制bootcmd、bootargs、bootdelay等关键行为；存储位置配置错误会导致环境变量丢失或磨损存储介质。
- **快速参考**：
  - 配置宏：`CONFIG_ENV_IS_IN_MMC` / `CONFIG_ENV_IS_IN_SPI_FLASH` / `CONFIG_ENV_IS_IN_NAND`
  - 默认偏移：`CONFIG_ENV_OFFSET` (如0x800000)
  - 大小：`CONFIG_ENV_SIZE` (通常8KB/16KB)
  - 冗余：`CONFIG_SYS_REDUNDAND_ENVIRONMENT` 支持双备份
  - 命令：`printenv` / `setenv` / `saveenv`
- **全书螺旋关联**：第6章「存储技术」→ 第7章启动流程控制 → 第10章「可靠性与升级」OTA升级的环境变量管理

### 速查3：设备树overlays(dtbo)与运行时动态合并

- **一句话解释**：设备树overlays是可以在主设备树之上动态叠加的片段，用于在不变更基础DTB的前提下适配不同的扩展板或硬件变体。
- **为什么重要**：同一基础板型有多种硬件配置变体时，无需为每种变体编译完整DTB，只需叠加不同的overlay即可。
- **快速参考**：
  - 编译：`dtc -@ -I dts -O dtb -o overlay.dtbo overlay.dts`
  - U-Boot加载：`fdt apply <addr>` 将overlay合并到内存中的DTB
  - 内核配置：`CONFIG_OF_OVERLAY=y`
  - 典型场景：Raspberry Pi HAT扩展板、BeagleBone Cape
  - 注意：overlay中的`__symbols__`节点用于引用主DTB中的节点
- **全书螺旋关联**：第7章设备树传递 → 第8章「驱动模型」设备树匹配 → 第10章硬件兼容性与BSP管理

### 速查4：printk日志级别与早期控制台(earlycon)

- **一句话解释**：printk通过8个日志级别(0-7)控制输出优先级；earlycon允许在内核console_init之前通过轮询方式输出调试信息。
- **为什么重要**：启动阶段调试依赖日志输出；earlycon是追踪内核启动问题的关键工具，尤其在console_init之前的"黑窗期"。
- **快速参考**：
  - 日志级别：`KERN_EMERG(0)` → `KERN_DEBUG(7)`
  - 默认级别：通常`CONFIG_CONSOLE_LOGLEVEL_DEFAULT=7`
  - bootargs启用earlycon：`earlycon=uart8250,mmio32,0xff1a0000`
  - 内核配置：`CONFIG_DEBUG_LL=y` + `CONFIG_EARLY_PRINTK=y`
  - 动态调整：`dmesg -n 8` 或 `/proc/sys/kernel/printk`
- **全书螺旋关联**：第7章内核启动(console_init时间点) → 第9章「调试技术」printk/动态调试 → 第13章「性能优化」启动时间分析

### 速查5：看门狗(Watchdog)在启动链中的喂狗策略

- **一句话解释**：启动链的每个阶段(BootROM→SPL→U-Boot→内核→init)都需要在Watchdog超时前喂狗或关闭，否则系统会反复复位。
- **为什么重要**：启动时间优化和安全启动验证可能耗时较长，必须确保Watchdog超时阈值覆盖整个启动流程，或在各阶段合理处理。
- **快速参考**：
  - 典型超时：1-60秒(可配置)
  - BootROM：通常自动关闭或设极长超时
  - SPL：`CONFIG_HW_WATCHDOG` 开启，需在board_init_f中初始化
  - U-Boot：`wdt`命令族(start/stop/reset)，`CONFIG_WATCHDOG`
  - 内核：`CONFIG_WATCHDOG_CORE`，用户空间`/dev/watchdog`
  - 关键策略：SPL/U-Boot阶段若启动时间不确定，应在跳转前关闭WDT，由内核重新管理
- **全书螺旋关联**：第4章「外设与总线」WDT硬件原理 → 第7章启动链各阶段处理 → 第11章「可靠性设计」看门狗与异常恢复机制
