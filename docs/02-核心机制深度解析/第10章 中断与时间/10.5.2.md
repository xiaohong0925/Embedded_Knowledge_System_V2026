时间子系统的整体脉络已经清楚了，但真要动手读代码或者调试定时器问题，你还得理解它的**分层架构**。Linux内核把时间相关的功能拆成了四层，每层只做一件事，彼此通过明确的接口对接。这个设计花了好几轮重构才定型，理解了它，很多看似诡异的行为就能说得通了。

**最底层：clocksource——"现在几点了"**

clocksource是系统的"时钟源"，它只干一件事：提供一个单调递增的计数器，让你能读取当前的时间戳。ARM平台上最常见的clocksource就是Generic Timer的CNTVCT（Virtual Counter），你读它的时候，本质上就是在问CPU："现在计数器跑到多少了？"

clocksource的核心结构体是`struct clocksource`，里面最关键的两个字段：`read`函数指针和`mask`、`mult`、`shift`这几个用于ns转换的参数。内核会把所有注册的clocksource挂到一个全局链表`clocksource_list`上，然后选一个rating最高的作为当前使用的源。这个选择是动态的——如果系统发现了更好的时钟源，随时可以切过去，你上层业务无感知。

```c
struct clocksource {
    u64 (*read)(struct clocksource *cs);
    u64 mask;
    u32 mult;
    u32 shift;
    int rating;
    /* ... */
};
```

**第二层：clockevent——"到点了叫醒我"**

如果说clocksource回答的是"现在几点"，那clockevent回答的就是"能不能在某个未来时刻叫醒我"。它提供的是**一次性**的定时中断编程能力。

clockevent的核心接口是`set_next_event(unsigned long delta, struct clock_event_device *evt)`。你告诉它："过delta个cycles之后给我来一发中断。"到点了，它触发中断，然后**就完了**——它不会自动reload，也不会周期性重复，这就是clockevent和旧时代tick-only定时器最本质的区别。

`struct clock_event_device`里有个重要的`features`字段，标注了这个设备支持的能力。比如`CLOCK_EVT_FEAT_ONESHOT`表示支持单次触发模式，`CLOCK_EVT_FEAT_PERIODIC`表示还能模拟传统的周期性tick。现在的主流硬件基本都支持oneshot。

```c
struct clock_event_device {
    void (*set_next_event)(unsigned long evt, struct clock_event_device *);
    void (*set_state_oneshot)(struct clock_event_device *);
    void (*event_handler)(struct clock_event_device *);
    int  features;
    /* ... */
};
```

**第三层：tick device——"搭桥的"**

clocksource和clockevent各自独立工作，谁来把它们串起来？tick device就是干这个的。

`struct tick_device`这个结构体非常简单，本质上就是一个clockevent设备的包装，外加一个模式标记。它有两种工作模式：

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| TICKDEV_MODE_PERIODIC | 周期性模式，clockevent被配置为自动reload | 传统模式，现在已基本不用 |
| TICKDEV_MODE_ONESHOT | 单次触发模式，每次tick到期后重新编程 | 当前主流，支持tickless |

在oneshot模式下，tick device每次中断处理完都会重新计算下一个tick的到期时间，然后调用底层的`set_next_event()`去编程。这个"到期后重新编程"的循环，就是整个时间子系统跳动的心脏。tick device的`event_handler`默认挂的是`tick_handle_periodic()`或者`hrtimer_interrupt()`，取决于你是否启用了高精度定时器。

**最上层：hrtimer——"给用户用的精密定时器"**

hrtimer框架建立在clockevent的oneshot能力之上，它提供了一套红黑树管理的定时器队列，支持纳秒级精度。你用的`timerfd`、`alarm`、`POSIX timer`，底层走的都是hrtimer。

hrtimer的核心思路是：每次把红黑树里最近到期的那个定时器时间，通过tick device编程到clockevent里。到期中断来了，处理完这个定时器，再找下一个最近的，重新编程。这样一来，系统不需要再维护一个固定的tick频率，空闲时甚至可以完全停掉tick，这就是NO_HZ/tickless的根基。

**知识点111 [E] 分层的意义**

你可能会想：搞这么多层，累不累？说实话，早期内核确实没分这么清，那时候就一个固定的PIT（Programmable Interval Timer），每隔1ms硬中断一次，简单粗暴。问题是后来硬件花样多了——有的CPU自带高精度计数器但中断能力弱，有的中断控制器定时精度一般但触发稳定，你怎么办？

分层的精妙之处在于**clocksource和clockevent的解耦**。这两个本质上关注的是不同维度：clocksource解决"度量时间"的问题，clockevent解决"触发事件"的问题。一个系统完全可以clocksource用ARM Generic Timer的CNTVCT（因为它读取快、精度高），clockevent用SoC自带的GPT（General Purpose Timer，因为它中断编程灵活）。

这种组合式的架构让板级支持变得极其灵活。你写一个BSP的时候，clocksource和clockevent各自注册，tick device会自动把它们配对起来。我见过一个案例：某款车规芯片的Generic Timer在中断方面有个小bug，但计数器读取没问题，于是clocksource继续用CNTVCT，clockevent切到GPT上，问题就绕过去了。如果没这层抽象，这种workaround根本做不了。

另外，分层也让NO_HZ和高精度定时器的实现成为可能。hrtimer只管往红黑树里插定时器，tick device负责到底层编程，clocksource提供精确的时间戳——各干各的，互不干涉。这个架构从2.6.16时代开始逐步成型，经历了Thomas Gleixner等人好几轮重构，现在看起来是理所当然的，但在当时确实是很大的进步。
