**知识点126 [I][M] tickless与C-state的深度协同**

tickless和cpuidle这两个子系统，看似各干各的——一个管时钟中断，一个管CPU空闲状态——但实际上它们是绑在一根绳上的蚂蚱。tickless给了CPU「入睡」的机会，而cpuidle决定了它能睡多沉。

传统周期性tick模式下，哪怕CPU无事可做，每隔1毫秒（HZ=1000时）就来一个tick中断把它踹醒。这就好比每秒钟都有人敲门问你"睡了没"，你能睡踏实才怪。tickless把这个烦人的敲门声关掉之后，CPU才有机会真正闭眼睁眼地休息，顺着C-state的阶梯一步步往下走。

C-state的阶梯长这样：C0是运行态，C1让CPU暂停时钟、C2关掉更多东西、C3（也叫Sleep）连L1/L2缓存都flush掉，更深的C6/C7连电压都降了。每往下一级，功耗就断崖式下跌，但代价是醒来时需要更多时间。这个trade-off没有免费午餐——省下的每一毫瓦电力，都用微秒的延迟换了。

来看看具体数字（以典型的x86桌面处理器为例）：

| C-state | 别名 | 典型功耗（相对C0） | 退出延迟（典型值） | 主要行为 |
|---------|------|-------------------|-------------------|----------|
| C0 | Running | 基准（100%） | 0 | 正常执行指令 |
| C1 | Halt | ~30-40% | ~1 μs | 停止时钟，缓存保持 |
| C1E | — | ~20-25% | ~2 μs | C1 + 降频降压 |
| C2 | Stop-Clock | ~15-20% | ~5-10 μs | 部分总线时钟停止 |
| C3 | Sleep | ~5-10% | ~20-50 μs | L1/L2缓存flush，PLL保持 |
| C6 | Deep Sleep | ~1-3% | ~100-300 μs | 保存架构状态，核心电压降零 |
| C7 | — | ~0.1-1% | ~300-1000 μs | 连最后一组共享缓存也flush |

这组数字你得记个大概。实际产品里不同代系的CPU差异很大，但趋势是一致的：C1到C6，功耗能差两个数量级，延迟也能差两个数量级。

tickless在这里扮演的角色，就是把这个阶梯的「入口」打开了。没有tickless时，CPU每次idle最多在C1打个盹——反正1毫秒后还得醒。有了tickless，idle时间如果能撑到几十微秒，governor就会评估：要不要进C3？如果能撑到几百微秒，C6也不是不行。这个决策逻辑在`menu_governor`的`menu_select()`里，核心就是看**下一次已知的唤醒事件有多远**——而这正是tickless通过`next_timer_expiry`告诉cpuidle的信息。

下面这段代码展示了cpuidle governor如何基于tick信息做决策：

```c
/* 简化自 kernel/sched/cpuidle.c 的核心逻辑 */
static int menu_select(struct cpuidle_driver *drv, struct cpuidle_device *dev)
{
    s64 latency_req = pm_qos_request(PM_QOS_CPU_DMA_LATENCY);
    ktime_t delta_next;
    int i;

    /* 拿到下一次定时器到期时间 —— tickless的核心贡献 */
    delta_next = tick_nohz_get_sleep_length(&dev->next_timer);

    /* 遍历所有C-state，找一个最深的、满足延迟约束的 */
    for (i = drv->state_count - 1; i > 0; i--) {
        struct cpuidle_state *s = &drv->states[i];

        if (s->disabled)
            continue;
        if (s->exit_latency > latency_req)
            continue;
        if (s->exit_latency * 3 > delta_next)
            /* 退出延迟超过idle时长的1/3就不划算 */
            continue;

        return i; /* 选中这个C-state */
    }
    return 0; /*  fallback到C0/C1 */
}
```

注意那行`s->exit_latency * 3 > delta_next`——这是内核的一个经验 heuristic，意思是如果进出的开销占了idle时长的一大块，那进去睡一觉就不值了。这个系数3不是拍脑袋定的，而是多年来实测调出来的。

现在说一个我见过的真实案例。这是第10章第二个问题场景的延续：一款嵌入式音频设备，启用tickless后反而出现了爆音。排查过程很曲折——CPU进了C3之后，一个原本期望在几十微秒内响应的hrtimer（音频buffer填充定时器），因为C3的50μs退出延迟被拖了后腿。buffer underrun，DAC没东西可播，喇叭就"啪"地爆一声。

这个问题在`dmesg`里通常看不出端倪，因为hrtimer并没有"丢失"，只是精度严重退化。诊断的关键是看`/sys/devices/system/cpu/cpuidle/current_driver`和`current_governor_ro`，然后用`trace-cmd`抓一下`cpuidle`事件的trace：

```bash
# 查看当前cpuidle governor和可用的C-state
$ cat /sys/devices/system/cpu/cpu0/cpuidle/current_driver
intel_idle
$ cat /sys/devices/system/cpu/cpu0/cpuidle/current_governor_ro
menu
$ ls /sys/devices/system/cpu/cpu0/cpuidle/state*/name
state0/name  state1/name  state2/name  state3/name  state4/name
$ cat /sys/devices/system/cpu/cpu0/cpuidle/state*/name
C0
C1E
C3
C6
C7s

# 实际命中各个C-state的时间（纳秒级）
$ cat /sys/devices/system/cpu/cpu0/cpuidle/state*/time
$ cat /sys/devices/system/cpu/cpu0/cpuidle/state*/usage
```

音频那个案子，最终的解法不是禁用tickless（那太粗暴了），而是调高`PM_QOS_CPU_DMA_LATENCY`的约束到100μs左右，把C3及以下挡在外面。cpuidle governor看到延迟需求收紧了，就会自动收敛到C1/C1E这个层级，音频hrtimer就能按时醒了。

**知识点127 [I] 平衡策略：保留"热CPU"的艺术**

tickless和cpuidle的配置，本质是一个「省多少电」与「冒多大险」的权衡。延迟敏感的场景——音频、工业控制、高频交易、实时通信——你得学会给CPU「保温」，别让它睡太沉。

具体怎么做？有这么几条路子：

第一，用`PM_QOS_CPU_DMA_LATENCY`。这是应用层能直接喊话cpuidle的通道。延迟敏感的任务（比如PulseAudio、JACK）可以通过`/dev/cpu_dma_latency`注册一个QoS约束，内核会保证CPU的C-state深度不超过这个延迟允许的范围。值设成0就是「不许进idle」，设成一个合理的微秒数就是「可以浅睡，不许深睡」。

第二，如果整机都是延迟敏感型，干脆在boot param里关掉deep C-state：`intel_idle.max_cstate=2`或者`processor.max_cstate=2`，把C3以下一刀切掉。这招虽然糙，但很管用。

第三，更精细的做法是绑核。把延迟敏感的线程绑到一个或几个CPU上，对这些CPU收紧QoS约束，其他CPU正常进deep idle。这样省电和性能两头都顾得上。

说白了，tickless把门关上了（不再周期性敲门），cpuidle决定门关上之后你睡客厅还是睡卧室。客厅省电少、醒来快；卧室省电多、但起床气大。作为系统设计者，你得根据业务特点告诉内核：这房间，最多睡到哪一级。
