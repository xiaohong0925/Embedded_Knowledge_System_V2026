irqsoff tracer是ftrace里我最喜欢的一个工具。不是说别的tracer不好，而是这个工具解决的问题太典型了——你的系统莫名其妙丢中断、定时器漂移、网卡收包有毛刺，大概率是某个地方把中断关了太久。irqsoff tracer就是专门抓这种"惯犯"的。

它的原理很简单：内核里每次调用`local_irq_disable()`和`local_irq_enable()`，tracer都在旁边记一笔。它持续追踪当前最大的那段中断关闭窗口，一旦发现更长的，就更新记录。最终trace文件里留下的，就是整个系统运行期间**最坏情况下的中断关闭时间**，以及完整的调用栈。

**知识点84 [I][M]  irqsoff tracer实战：找到那个关中断太久的家伙**

好，上实操。假设你的设备偶发性地出现几十微秒的时序抖动，你怀疑有人在中断上下文里做了不该做的事。

先挂载tracefs，切换到irqsoff tracer：

```bash
# mount -t tracefs tracefs /sys/kernel/debug/tracing
# cd /sys/kernel/debug/tracing
# echo irqsoff > current_tracer
# echo 0 > tracing_on          # 先停掉，准备好再开
# echo > trace                 # 清空之前的记录
# echo 1 > tracing_on
```

让系统跑一阵你的业务负载，然后关掉tracing，看结果：

```bash
# echo 0 > tracing_on
# cat trace
```

你可能会看到类似这样的输出：

```
# tracer: irqsoff
#
# irqsoff latency trace v1.1.5 on 5.15.71-rt51
# --------------------------------------------------------------------
# latency: 823 us, #4/4, CPU#2 | (M:preempt_rt VP:0, KP:0, SP:0 HP:0 #P:4)
#    -----------------
#    | task: swapper/2-0 (uid:0 nice:0 policy:0 rt_prio:0)
#    -----------------
#
#                  _------=> CPU#
#                 / _-----=> irqs-off
#                | / _----=> need-resched
#                || / _---=> hardirq/softirq
#                ||| / _--=> preempt-depth
#                |||| /
#                |||||     delay
#  cmd     pid   ||||| time  |   caller
#     \   /      |||||  \    |   /
  <idle>-0     2d...    0us!: __irq_svc -> my_driver_irq_handler
  <idle>-0     2d..h    2us!: my_driver_irq_handler -> spi_read_data
  <idle>-0     2d..h  823us : trace_hardirqs_on
```

看到这一行关键信息了吗？`latency: 823 us`。这就是你要的——最坏情况下中断被关了823微秒。再看看调用链：`my_driver_irq_handler`里调了`spi_read_data`，也就是说这个驱动的顶半部（hardirq handler）里直接读了SPI总线，忙等了800多微秒。

这就是典型的"顶半部干太多活"。正确做法是把SPI读取放到底半部（tasklet、workqueue或者threaded irq）里去，顶半部只做最紧急的事——比如读个状态寄存器、清个中断标志，然后就赶紧开中断走人。

irqsoff tracer的列头里面，`| d...`那个`d`表示中断是关着的，`.`表示开着。这串标记配合delay时间戳，能让你一眼看出中断是从哪个函数开始关、哪个函数结束开的。

> **陷阱：** irqsoff tracer记录的是**当前观测到的最大值**，不是平均值。如果你系统跑了很长时间，trace里留下的可能是很久以前的一次异常峰值。排查时建议先`echo > trace`清空，再在有负载的情况下跑一个固定时长，这样抓到的数据更有针对性。

> **另一个陷阱：** 如果内核编译时没有开`CONFIG_IRQSOFF_TRACER`，`/sys/kernel/debug/tracing`下面压根看不到irqsoff这个选项。确认你的内核配置，很多发行版默认只开了function和function_graph。

```bash
# grep IRQSOFF /boot/config-$(uname -r)
CONFIG_IRQSOFF_TRACER=y
```

**知识点85 [I]  function tracer和function_graph tracer的配合**

irqsoff tracer告诉了你"谁"和"多久"，但它没告诉你"具体每一步干了什么"。这时候就需要另外两个tracer打配合。

function tracer能记录指定函数的每次进入和退出。比如你已经知道嫌疑人是`my_driver_irq_handler`，可以：

```bash
# echo function > current_tracer
# echo my_driver_irq_handler > set_ftrace_filter
# echo 1 > tracing_on
```

这样trace里就只有这个函数的调用记录，不会被其他信息淹没。

function_graph tracer则更直观——它画出一个调用树，每个函数的耗时都标在旁边：

```bash
# echo function_graph > current_tracer
# echo my_driver_irq_handler > set_graph_function
```

它的输出长这样：

```
 0)               |  my_driver_irq_handler() {
 0)               |    spi_read_data() {
 0)   823.000 us  |      /* 这里就是你关中断的罪魁祸首 */
 0)   825.000 us  |    }
 0)   826.000 us  |  }
```

irqsoff负责定位问题，function_graph负责解剖细节，这两个搭配着用，基本上什么中断延迟问题都能揪出来。我的习惯是先用irqsoff抓最坏情况，确定了嫌疑函数，再用function_graph看它的内部耗时分布，效率比单用一个tracer高得多。
