写代码这件事，有书上的正确用法，就有实际上各种匪夷所思的搞法。这一节咱们不聊原理了，聊点实在的——那些年我在底半部机制上踩过的坑，以及怎么捞出来。

**知识点96 [E] 踩坑清单：底半部三大经典翻车现场**

下面这三条，是我自己或同事实打实踩过、导致线上问题的案例。建议打印出来贴在显示器边上。

| 编号 | 陷阱 | 现象 | 根因 |
|:---:|:---|:---|:---|
| 1 | tasklet里调用可能睡眠的函数 | Oops / 硬锁死 | tasklet运行于中断上下文，禁止睡眠 |
| 2 | softirq处理函数耗时过长 | 用户态进程饿死，系统卡顿 | softirq持续抢占CPU，schedule()不被调用 |
| 3 | workqueue的max_active设太大 | 并发过多，资源耗尽或死锁 | 超出设备/锁的承载能力 |

**陷阱1：tasklet里睡了——必炸**

这是最经典的入门翻车。tasklet跑在`softirq`上下文，本质上还是中断的延伸，当前进程是`ksoftirqd`或者干脆没有进程上下文（中断直调路径）。你在这个里面调用`kmalloc(GFP_KERNEL)`、`mutex_lock()`、`schedule_timeout()`，任何一个可能引发调度器切换的操作，内核都会毫不客气地给你一个`BUG: scheduling while atomic`的Oops。

```c
/* 千万别这么干 */
void my_tasklet_handler(unsigned long data)
{
    struct my_dev *dev = (struct my_dev *)data;

    /* 爆炸点：GFP_KERNEL可能睡眠 */
    dev->buf = kmalloc(4096, GFP_KERNEL);

    /* 爆炸点：mutex可能睡眠 */
    mutex_lock(&dev->lock);
    ...
}
```

那怎么办呢？在tasklet里只配用`GFP_ATOMIC`，锁只配用`spin_lock`。需要睡眠的操作？挪到workqueue里去。这没有商量余地。

**陷阱2：softirq吃太多CPU，进程饿死了**

这个问题比陷阱1更阴险，因为它不会Oops，系统就是"莫名其妙地卡"。如果你的网卡驱动在NAPI的poll函数里一次性收了太多包，或者在block softirq里刷写缓存太久，`ksoftirqd`就会霸占CPU。最坑的是——softirq的优先级比普通过程高，它不停地跑，调度器没机会把CPU让给用户进程。

你可能发现SSH连上去敲个命令都要好几秒，但`top`里CPU使用率并不高——因为时间都被softirq偷走了，还不显式计入用户态。

解决思路很简单：poll函数里严格遵守`budget`限制，该退出就退出，别贪心。内核设计`budget`和`poll_list`不是白设计的。

**陷阱3：workqueue并发数设太猛，锁撑不住了**

`alloc_workqueue()`的时候把`max_active`设成`WQ_UNBOUND`或者一个很大的数，看起来能提高并发，但如果你的work函数里要竞争同一个锁，或者要访问同一个硬件寄存器，并发大了反而互相堵死。更糟的是，如果你的work里还会申请大量内存，几十个worker同时跑，OOM也不是不可能。

```c
/* 危险的写法：无限制并发 */
struct workqueue_struct *wq = alloc_workqueue("my_wq", 
                                               WQ_UNBOUND, 0); /* max_active=0=无限制 */

/* 相对安全的写法 */
struct workqueue_struct *wq = alloc_workqueue("my_wq",
                                               WQ_UNBOUND, 4); /* 最多4个并发 */
```

`max_active`设成多少？看你的硬件通道数和锁粒度。一般先从1或者CPU数开始，实测后再调。

**知识点97 [I] 出了问题怎么查：底半部调试三板斧**

踩坑不可怕，可怕的是踩了不知道怎么查。下面三个调试手段，是我排障时的标准流程。

**第一板斧：`/proc/softirqs`看分布**

这个文件是排查softirq CPU倾斜的第一现场。打开它，你会看到每种softirq类型在每个CPU上的触发次数：

```bash
$ cat /proc/softirqs
                    CPU0       CPU1       CPU2       CPU3
          HI:          2          0          1          0
       TIMER:    1234567     987654    1122334     876543
      NET_TX:        123        456        789        012
      NET_RX:    8901234    7654321    8901234    7654321
       BLOCK:       5678       9012       3456       7890
     IRQ_POLL:          0          0          0          0
      TASKLET:         45         67         89         12
       SCHED:     234567     345678     456789     567890
     HRTIMER:      12345      23456      34567      45678
         RCU:     345678     456789     567890     678901
```

看什么？看两点。第一，`NET_RX`是不是集中在某几个CPU上——如果是，说明你的网卡RSS配置有问题或者IRQ亲和性没设好。第二，数字是不是在暴涨但你的业务没流量——可能是中断风暴，检查硬件或者驱动。

**第二板斧：`ps`盯紧worker线程**

workqueue背后是一组内核线程。找到它们，看状态：

```bash
# 查看所有workqueue对应的worker线程
$ ps -eo pid,comm,stat | grep kworker
  187 kworker/0:0H    S
  234 kworker/0:1     S
  345 kworker/u2:0    S
  456 kworker/u2:1    D    # 注意这个D状态！
```

线程名里的`u2`表示unbound wq上的worker，`0`表示绑定CPU0的worker。状态列尤其要盯：大量`D`（uninterruptible sleep）意味着worker卡在IO或者锁上；`R`（running）占比过高说明workqueue在拼命干活，可能任务堆积了。

如果是自己创建的workqueue，worker名字会带你指定的前缀，更容易识别。

**第三板斧：`trace`抓workqueue执行**

ftrace的`workqueue`事件是最直接的手段，能看到每个work的入队、执行、耗时全过程：

```bash
# 开启workqueue trace
$ echo 1 > /sys/kernel/debug/tracing/events/workqueue/enable
$ cat /sys/kernel/debug/tracing/trace_pipe

# 典型输出：
  kworker/u2:1  workqueue_execute_start: work struct 00000000deadbeef: function my_work_handler
  kworker/u2:1  workqueue_execute_end:   work struct 00000000deadbeef
```

如果`execute_start`之后很久才看到`execute_end`，说明你的work函数耗时太长，考虑拆分或者优化。如果连`execute_start`都等不到，说明worker不够用了，检查`max_active`和系统负载。

三个陷阱 + 三板斧，底半部的问题基本都能兜住了。还是那句话：代码写对了只是及格，知道怎么查问题才是真本事。
