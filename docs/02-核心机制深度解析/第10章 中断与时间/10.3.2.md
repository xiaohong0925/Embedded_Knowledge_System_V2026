softirq这个名字起得挺有意思——"软中断"，听起来好像比硬中断"软"一些，优先级更低似的。但恰恰相反，**它是Linux底半部机制里优先级最高的那个**。如果你的网卡驱动里既用了softirq又用了tasklet，softirq会先执行。

为什么这样设计？你把时间线拨回到上世纪90年代末，网络流量开始爆发，每秒钟进中断的次数从几百飙升到几万。Linus和那时候的工程师们发现，中断处理函数里哪怕只做一点点事，CPU也扛不住了。必须把" Ack硬件"和"搬运数据"拆开——前者留在顶半部，越快越好；后者丢到底半部慢慢消化。

但底半部也要分个三六九等。有些活儿，比如网络收包，延迟哪怕多一毫秒都是真金白银的吞吐量损失。softirq就是为这种"我虽然不急到微秒级，但我也不想排队"的场景量身定制的。

---

**知识点88 [E][M]**

softirq的核心数据结构非常简单，就藏在内核源码`kernel/softirq.c`里：

```c
/* kernel/softirq.c */
static struct softirq_action softirq_vec[NR_SOFTIRQS] __cacheline_aligned;
```

`NR_SOFTIRQS`定义成32，理论上最多32种softirq。但你真去翻内核代码，实际注册的也就十种上下——这不是内核偷懒，而是Linus有意为之。softirq是静态编译期就定好的，不像tasklet那样可以在模块里随手创建。每一种都要在`include/linux/interrupt.h`的枚举里占一个坑位：

| 枚举值 | 名称 | 用途 |
|--------|------|------|
| 0 | HI_SOFTIRQ | 高优先级tasklet的底座 |
| 1 | TIMER_SOFTIRQ | 定时器（hrtimer底层用到） |
| 2 | NET_TX_SOFTIRQ | 网络发包 |
| 3 | NET_RX_SOFTIRQ | 网络收包 |
| 4 | BLOCK_SOFTIRQ | 块设备IO完成 |
| 5 | IRQ_POLL_SOFTIRQ | 块设备irq poll模式 |
| 6 | TASKLET_SOFTIRQ | 普通tasklet的底座 |
| 7 | SCHED_SOFTIRQ | 负载均衡 |
| 8 | HRTIMER_SOFTIRQ | 高精度定时器 |
| 9 | RCU_SOFTIRQ | RCU回调处理 |

看到没？**HI_SOFTIRQ和TASKLET_SOFTIRQ本质上不是独立的softirq用户**，它们只是tasklet的两条"跑道"——高优先级tasklet跑在HI_SOFTIRQ上，普通tasklet跑在TASKLET_SOFTIRQ上。真正干活的是NET_RX_SOFTIRQ、BLOCK_SOFTIRQ这些。

softirq的执行流程其实比大多数人想象的更直接：

```mermaid
flowchart TD
    A[网卡收到数据包] --> B[驱动顶半部执行]
    B --> C[raise_softirq(NET_RX_SOFTIRQ)]
    C --> D[irq_exit调用do_softirq]
    D --> E[__do_softirq遍历pending位图]
    E --> F[执行net_rx_action]
    F --> G[处理NAPI轮询队列]
    style C fill:#e1f5e1
    style D fill:#fff3cd
```

`raise_softirq()`只做一件事：把当前CPU的`__softirq_pending`位图里对应位set上。然后等顶半部的中断处理完，`irq_exit()`会顺手调用`do_softirq()`，真正干活的是`__do_softirq()`——它会循环检查pending位图，把置位的softirq一个个执行掉。

这里有个细节值得留意：**同一种softirq可以在多个CPU上同时执行**。`net_rx_action`在CPU0跑的时候，CPU1完全可以也在跑`net_rx_action`。这对网络性能是好事——多核并行收包。但这也意味着，如果你在softirq处理函数里操作全局数据结构，必须自己加锁。内核不会帮你串行化。

**知识点89 [E]**

softirq很快，但快是有代价的。这三个限制你务必刻在脑子里：

**第一，不能睡眠。** softirq跑在原子上下文，没有进程 backing它。你想在里面调用`kmalloc(GFP_KERNEL)`或者`mutex_lock()`？直接崩。只能用`GFP_ATOMIC`，只能上自旋锁。

**第二，同一个softirq可能在多个CPU同时执行。** 刚说过，这是特性也是陷阱。除非你的softirq处理函数天生就是per-CPU的（比如NAPI的poll_list），否则跨CPU的数据访问必须自己同步。

**第三，也是最隐蔽的——softirq能饿死普通进程。** `__do_softirq()`里有一段经典逻辑：它会最多连续处理10个softirq才肯退出。如果这段时间内一直有新的softirq被raise起来（比如网卡拼命发包），它会一直循环。10次循环结束后如果还有pending，它会识趣地唤醒ksoftirqd，把自己降级到进程上下文去执行。这个设计很精妙：**既保证了突发流量下的低延迟，又避免了无限抢占用户态进程**。

```c
/* kernel/softirq.c - __do_softirq 核心循环 */
for (;;) {
    struct softirq_action *h = softirq_vec;
    int max_restart = MAX_SOFTIRQ_RESTART;  /* 默认10次 */

    do {
        if (pending & 1) {
            h->action(h);   /* 执行注册的action */
        }
        h++;
        pending >>= 1;
    } while (pending);

    if (likely(!pending))
        break;

    if (++restart >= max_restart) {
        /* 10次到了还没完，交给ksoftirqd */
        wakeup_softirqd();
        break;
    }
}
```

实际排障的时候，如果你发现某个CPU的`si`（softirq）时间飙高，而用户进程像是被冻住了，十有八九就是这个循环在作祟。用`perf top`或者`watch -n1 cat /proc/softirqs`看一眼，哪个类型的计数在疯涨，基本就能定位方向。

> **陷阱**：在模块代码里你不能直接创建新的softirq类型，因为`NR_SOFTIRQS`是编译期常量，枚举也是固定的。如果你写了一个自定义驱动想走底半部，**老老实实选tasklet或者workqueue**，别想着自己加一种softirq——这套机制只给内核子系统核心层用。
