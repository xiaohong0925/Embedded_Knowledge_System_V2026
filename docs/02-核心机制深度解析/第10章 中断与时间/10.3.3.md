softirq虽然高效，但用起来还是有点繁琐——你得自己管理32个vec的优先级，还得操心同一softirq在不同CPU上并行的问题。对大多数驱动开发者来说，这未免太重了。于是内核搞了个"轻量化封装"，叫tasklet。

**知识点90 [E][M]**

tasklet说白了就是把softirq包了一层皮。它只占用两个softirq向量：`HI_SOFTIRQ`和`TASKLET_SOFTIRQ`，前者优先级更高一些。你不需要碰底层的softirq注册逻辑，直接声明一个`struct tasklet_struct`，填好处理函数，就能用。

核心数据结构很简单：

```c
struct tasklet_struct {
    struct tasklet_struct *next;
    unsigned long state;
    atomic_t count;
    void (*func)(unsigned long);
    unsigned long data;
};
```

`state`只有两个位有意义：`TASKLET_STATE_SCHED`表示已经被调度但还没执行，`TASKLET_STATE_RUN`表示正在某个CPU上运行。`count`是一个引用计数，非零时tasklet会被禁能。这套设计很紧凑，没有多余的字段。

tasklet最关键的特性——**同一种tasklet串行执行**。同一个tasklet实例，同一时间只能在一个CPU上运行。这是通过`TASKLET_STATE_RUN`标志配合原子操作来保证的。看`tasklet_action()`的核心逻辑就能明白：

```c
static void tasklet_action(struct softirq_action *a)
{
    struct tasklet_struct *list;

    local_irq_disable();
    list = __this_cpu_read(tasklet_vec.head);
    __this_cpu_write(tasklet_vec.head, NULL);
    __this_cpu_write(tasklet_vec.tail, &__get_cpu_var(tasklet_vec).head);
    local_irq_enable();

    while (list) {
        struct tasklet_struct *t = list;
        list = list->next;

        if (tasklet_trylock(t)) {   /* 测试并设置 STATE_RUN */
            if (!atomic_read(&t->count)) {
                if (!test_and_clear_bit(TASKLET_STATE_SCHED, &t->state))
                    BUG();
                t->func(t->data);
                tasklet_unlock(t);
                continue;
            }
            tasklet_unlock(t);
        }
        /* 如果没拿到锁或count非零，放回队列尾部 */
        local_irq_disable();
        t->next = NULL;
        *__this_cpu_read(tasklet_vec.tail) = t;
        __this_cpu_write(tasklet_vec.tail, &t->next);
        __raise_softirq_irqoff(TASKLET_SOFTIRQ);
        local_irq_enable();
    }
}
```

`tasklet_trylock()`就是保证串行的关键——它尝试设置`STATE_RUN`位，如果该tasklet已经在别的CPU上跑了，返回false，当前CPU就把这个tasklet塞回队列尾部，等下一轮softirq再试。这套机制确保了同一个tasklet函数不会两个CPU同时进去，但**不同**的tasklet之间仍然是并行的。

再看看`tasklet_schedule()`的调用流程，它的路径很短：

```c
static inline void tasklet_schedule(struct tasklet_struct *t)
{
    if (!test_and_set_bit(TASKLET_STATE_SCHED, &t->state)) {
        __tasklet_schedule(t);
    }
}
```

先检查`STATE_SCHED`位，如果已经被调度过了，直接返回——避免重复入队。然后把tasklet挂到当前CPU的`tasklet_vec`链表上，最后触发`TASKLET_SOFTIRQ`。等当前硬中断返回，或者`local_bh_enable()`的时候，softirq就会把它捞出来执行。

```
          tasklet_schedule()
                |
                v
      test_and_set_bit(STATE_SCHED)
                |
        +-------+-------+
        | 已设置        | 未设置
        v               v
      直接返回    __tasklet_schedule()
                          |
                          v
                    挂入CPU tasklet_vec
                          |
                          v
                     raise_softirq()
                          |
                          v
                    中断返回时执行
```

tasklet特别适合的场景是那种"中断上半部太长了，想拆出去但又不想搞太复杂"的活儿。比如GPIO中断里读个寄存器状态、更新个软件标志位、唤醒个等待队列——简单直接，几行代码搞定。早年间几乎每个字符设备驱动都会用到tasklet，它比workqueue快（不用切上下文），又比裸softirq好写。

**知识点91 [I]**

但tasklet的坑也不少，而且有些是无法修补的设计缺陷。

最致命的规矩和softirq一样——**不能睡眠**。tasklet跑在softirq上下文，没有进程 backing，调度器不认它。你要是在tasklet里调了`kmalloc(GFP_KERNEL)`或者搞了锁竞争导致休眠，内核直接oops给你看。我见过有人在中断处理里用`mutex_lock()`，然后在tasklet里再锁同一个mutex，死锁倒没有，直接触发调度器断言——因为tasklet试图睡眠了。

更要命的是，tasklet的串行保证只在**同一个实例**上有效。如果你定义了全局tasklet，那它确实不会并行；但如果你每个设备实例动态创建一个tasklet，那不同的tasklet实例之间是并行的，如果你的处理函数访问了共享资源，还是得自己加锁。很多人以为"用了tasklet就不用管同步了"，这是错觉。

这些年来，tasklet的这些问题逐渐暴露。它既不能睡眠，又没法像workqueue那样灵活控制优先级和执行CPU，而且串行模型在很多现代场景下反而成了瓶颈。所以内核社区在较新版本里已经把tasklet标记为deprecated，文档里明确推荐使用threaded IRQ或者workqueue来替代。threaded IRQ本质上是让中断下半部跑在一个内核线程里，能睡眠、能调优先级、能被调度器公平对待，除了延迟稍大一点，其他方面全面碾压tasklet。

如果你的代码还在用tasklet，不用急着全部重写——它短期内不会被删掉，内核保证向后兼容。但新写的驱动，建议直接上threaded IRQ或workqueue，少走冤枉路。
