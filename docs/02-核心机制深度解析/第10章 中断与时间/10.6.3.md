NO_HZ_IDLE解决了idle CPU的tick问题，但你可能马上想到一个更激进的问题——如果CPU一直在忙，tick不也一直在打扰吗？

确实是。NO_HZ_IDLE只停idle CPU的tick，但生产环境里很多场景是CPU满载运行的。比如高性能计算（HPC）跑一个单任务占满整个物理核，或者实时系统里一个关键任务独占CPU。这时候tick每毫秒（或每毫秒级别）来一次，纯粹是添乱。频繁采样`update_process_times()`、检查是否需要调度、更新各种统计量——这些对单个长任务来说毫无意义，因为它根本没有竞争者，调度个啥？

这就引出了NO_HZ_FULL。

**知识点124 [E]**

NO_HZ_FULL的核心思路很直接：如果一个CPU上只有一个可运行任务，那这个CPU根本不需要tick来触发调度。没有竞争者，没有优先级变化，没有超时需要处理—— tick存在的理由几乎为零。

内核怎么判断这个条件？关键代码在`tick_nohz_full_update_allowed()`和相关路径里。当调度器发现某CPU的runqueue上仅剩当前任务时（`rq->nr_running == 1`），就可以进入full dynticks模式，直接关掉tick。此后直到有新任务被migrate到这个CPU、或者当前任务阻塞/退出，tick才会恢复。

```c
/* kernel/time/tick-sched.c */
static bool tick_nohz_full_can_stop(void)
{
    struct rq *rq = this_rq();

    /* 只有单个任务时才允许停tick */
    if (rq->nr_running > 1)
        return false;

    /* 还有timer pending？不能停 */
    if (timer_pending_on_cpu(smp_processor_id()))
        return false;

    /* 检查是否需要更新jiffies等全局状态 */
    ...
    return true;
}
```

注意上面代码里的细节——不只是看`nr_running`，timer、hrtimer、RCU的grace period推进都可能阻止tick停下。RCU尤其麻烦：tick traditionally承担RCU的quiescent state标记，如果tick完全停了，RCU子系统需要另一条路来知道"这个CPU还活着"。这就是为什么NO_HZ_FULL需要CONFIG_RCU_NOCB_CPU把RCU callback offload到其他CPU上处理。

从实现角度看，NO_HZ_FULL比NO_HZ_IDLE复杂得多。idle CPU停tick时，恢复tick的触发点很清晰——有任务要wake up。但busy CPU上只有一个任务在跑，这个任务可能跑很久，内核必须确保：hrtimer能工作、RCU能推进、负载均衡不被破坏。每一个都是硬骨头。

来看下两种模式的区别：

| 对比项 | NO_HZ_IDLE | NO_HZ_FULL |
|--------|-----------|-----------|
| 停tick的CPU状态 | 仅idle CPU | idle + busy CPU |
| 前提条件 | CPU无任务运行 | CPU上**恰好1个**任务运行 |
| 配置选项 | `CONFIG_NO_HZ_IDLE` | `CONFIG_NO_HZ_FULL` |
| RCU要求 | 无特殊要求 | 通常需`RCU_NOCB_CPU` |
| 调度器依赖 | 无 | 需要isolcpus配合 |
| 适用场景 | 笔记本/手机节能 | HPC、实时系统、低延迟交易 |
| 实现复杂度 | 中等 | 高 |
| 内核版本支持 | 较早即支持 | 3.10+ 逐步成熟 |

这个表格你一眼就能看出，NO_HZ_FULL不是NO_HZ_IDLE的简单扩展，而是走了一条完全不同的路。

**知识点125 [I]**

配置NO_HZ_FULL有一些硬门槛。首先必须在编译时开启`CONFIG_NO_HZ_FULL=y`，然后在boot参数里用`nohz_full=`指定哪些CPU进入full dynticks模式。但你很快会发现，只写`nohz_full=2-7`远远不够。

关键在于——怎么保证这些CPU上真的只有一个任务？

答案是`isolcpus`。你需要把目标CPU从调度域里隔离出来，让普通任务默认不会调度上去，然后手动把目标task绑定过去。典型配置长这样：

```
isolcpus=2-7 nohz_full=2-7 rcu_nocbs=2-7
```

`isolcpus`保证只有affinity明确指定CPU 2-7的任务才会上去跑；`rcu_nocbs`把RCU callback推到别的CPU处理，避免RCU反过来唤醒tick；两者加在一起，才能让NO_HZ_FULL真正生效。

那适用场景呢？说实话，NO_HZ_FULL不是银弹。它对普通服务器应用收益很小——你的Java服务、Nginx、数据库哪个不是满runqueue的？只有下面几类负载值得折腾：

- **HPC单线程仿真**：一个进程绑一个核，跑几小时，减少tick能省掉数万次上下文切换的开销
- **高频交易**：延迟每少1微秒都是真金白银，tick interrupt是确定的干扰源
- **DPDK/网络数据平面**：用户态轮询已经占满CPU，内核tick纯属添乱

⛔ **陷阱**：不少人配了`nohz_full`却忘了`isolcpus`，结果系统任务、内核线程时不时跑上去，nr_running一会儿变2、一会儿变3，tick停了又启、启了又停，性能反而更差。check生效与否的最简单方法：`cat /proc/sys/kernel/nohz_full_loaded`，以及用`trace-cmd`抓`nohz_full`相关tracepoint看实际停tick的时长。

另一个常见误区是以为NO_HZ_FULL能提升所有场景的性能。如果CPU上本来就堆了多个任务，调度器需要tick来做时间片轮转，NO_HZ_FULL根本不会激活。它解决的是一个非常特定的痛点，不是万能药。
