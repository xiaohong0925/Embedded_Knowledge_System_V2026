**知识点128 [I]**

tickless省电归省电，副作用也是实实在在的，有些甚至能直接误导你的性能分析。

第一个副作用是CPU使用率统计不准。`top`、`vmstat`、`/proc/stat`这些工具骨子里都是数tick的——它们靠统计"这段时间里CPU有多少个tick落在用户态、多少个落在核心态"来算负载。no_hz模式下CPU可能好几秒都没tick，那这段时间的负载怎么算？内核只能在进出idle时靠`account_idle_ticks()`做估算，或者等时钟中断到来时采样。结果就是你经常看到`top`的CPU使用率锯齿状跳变，idle忽高忽低，严重时甚至看到"100% idle"但其实CPU刚跑完一波密集计算。这个坑我踩过不止一次——客户说"CPU看起来很闲"，实际他的定时器批量处理全躲过了tick采样窗口。

第二个副作用是`jiffies`精度降低。这个全局变量在没有tick的CPU上没人更新。你用`jiffies`做短延迟或超时判断，粒度就从"1/HZ"退化成了"两次tick之间的任意值"。比如`schedule_timeout(1)`期望等一个tick，在tickless CPU上可能一等就是几十毫秒。内核的`tick_nohz_stop_tick()`在停tick前会检查有无待处理定时器，但也只能解决部分问题。驱动里依赖`jiffies`做精确计时的，最好换成`ktime_get()`或`hrtimer`。

```
# tickless下top可能看到不稳定的CPU使用率
$ top -b -n 3 | grep "Cpu(s)"
%Cpu(s):  0.0 us,  0.0 sy,  0.0 ni,100.0 id,  0.0 wa  <-- 看起来全闲？
%Cpu(s): 15.2 us, 28.4 sy,  0.0 ni, 56.4 id,  0.0 wa   <-- 突然跳变
```

| 副作用 | 影响范围 | 应对策略 |
|--------|---------|---------|
| CPU使用率统计不准 | `top`、`/proc/stat`、`vmstat` | 改用`perf stat`基于事件采样 |
| jiffies精度降低 | `schedule_timeout()`、驱动超时 | 使用`hrtimer`或`ktime_get()` |

**知识点129 [I]**

tickless生效了吗？直接看`/proc/timer_list`最靠谱。找到对应CPU的Tick Device段，如果`nohz_mode`显示1或2，且`last_jiffies`与`next_jiffies`间隔远大于1，说明tick已成功关掉。

```
# 查看tick设备状态
$ cat /proc/timer_list | grep -A 8 "Tick Device: mode.*dynticks"
Tick Device: mode:     1    # 1 = 动态tick生效
Clock Event Device: lapic-deadline
next_event:         1234567890123 nsecs
```

若启用了`nohz_full`，用`dmesg | grep -i "NOHZ"`看启动日志，内核会打印每个full tickless CPU的初始化信息。动态调试还有个技巧：内核命令行配`nohz_full=`时加上`printk.time=1`，对比tick中断与实际事件的时间戳，一眼就能看出no_hz带来的延迟变化。想动态调整隔离CPU，可检查`/sys/devices/system/cpu/cpuX/nohz_full`节点，前提是你的内核开启了`CONFIG_NO_HZ_FULL=y`。
