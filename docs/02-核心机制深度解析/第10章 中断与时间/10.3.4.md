workqueue是Linux内核里最常用的底半部机制，也是三者之中最"重"、最灵活的一个。说它"重"，是因为它运行在进程上下文；说它灵活，恰恰也是因为这一点——你的handler可以睡眠，可以调用任何内核API，不用再像tasklet那样束手束脚。

这个设计不是凭空来的。早年间的内核只有BH（bottom half），后来演变成tasklet，但开发者很快发现：很多驱动程序在中断底半部里需要访问文件系统、需要分配大量内存、需要持有互斥锁——这些操作都可能睡眠。tasklet和softirq都不行，怎么办？干脆让底半部跑在内核线程里好了。于是workqueue应运而生。

## workqueue的线程池架构

内核维护了一组worker线程池（`worker_pool`），每个CPU有对应的bound pool，也有跨CPU共享的unbound pool。当你调用`queue_work()`时，内核并不是立刻创建一个新线程，而是把`work_struct`挂到`pool_workqueue`（简称pwq）的链表上，再由空闲的worker线程来认领执行。

整个数据结构的层级关系是这样的：

```
workqueue_struct（逻辑workqueue，如system_wq）
    ├── pool_workqueue（pwq，绑定到具体的worker_pool）
    │       └── worker_pool（线程池，管理一组worker）
    │               ├── worker（工作线程，kworker/*）
    │               ├── worker
    │               └── worker
    └── pool_workqueue（如果有多个pool）
```

光看文字可能有点晕，画个图就明白了。下面这张图描述了一个bound workqueue的完整调度链路：

```mermaid

    A[驱动代码 queue_work] -->|"将work_struct入队"| B[pool_workqueue<br/>pwq]
    B -->|"链表头pwq->pool->worklist"| C[worker_pool]
    C -->|"唤醒或复用"| D[worker线程<br/>kworker/0:0]
    D -->|"执行"| E[process_one_work]

    style A fill:#e1f5e1
    style F fill:#ffe1e1
```

`process_one_work()`是worker线程的核心循环。它会从`pool->worklist`上摘下一个`work_struct`，执行你注册的回调函数，然后回去继续 poll。如果没有工作可做，worker会进入idle状态等待被唤醒，而不是 busy-waiting——这点很重要，说明worker线程在没有任务时是不消耗CPU的。

内核还会根据负载动态调整worker的数量。任务多了就增派worker（上限由`wq_max_active`控制），任务少了就让多余的worker退出。这套机制在`kernel/workqueue.c`里实现得相当精巧，不过日常使用你基本不用操心这些细节。

## 三种内置的workqueue

内核预定义了几个常用的workqueue，绝大部分驱动直接用它们就行：

| workqueue名称 | 类型 | 特点 | 典型用途 |
|---|---|---|---|
| `system_wq` | bound | 默认workqueue，普通优先级 | 一般性延迟工作 |
| `system_highpri_wq` | bound | 高优先级worker线程 | 需要快速响应的底半部 |
| `system_unbound_wq` | unbound | 不绑定特定CPU，可在任意CPU上执行 | 不依赖CPU局部性的任务 |

`system_wq`就是`keventd`的现代替代品。你调用`schedule_work()`时，本质上就是往这个队列里扔任务。默认情况下，每个CPU上最多同时运行两个worker（`WQ_MAX_ACTIVE = 512`是上限，但system_wq只用2个），这个并发度对大多数场景足够了。

`system_highpri_wq`的worker线程有`MAX_RT_PRIO/2`的实时优先级，适合对延迟敏感的场景，比如网络包处理的某些环节。但注意，高优先级worker越多，对系统的实时干扰就越大，别滥用。

`system_unbound_wq`比较特殊。它的worker不绑定到特定CPU，调度器可以把它迁移到任意CPU上执行。好处是：如果你的work回调里要做大量CPU密集操作，不会因为某个CPU负载过高而卡住；坏处也很明显——失去了CPU缓存亲和性，cache miss会增加。要不要用unbound wq，得看你的工作负载特征。

## ordered workqueue：保证执行顺序

有些场景下，你提交的几个work必须严格按照先后顺序执行，不能并行。内核提供了ordered workqueue来解决这个问题——本质上它只用一个worker线程，天然就串行了。

```
alloc_ordered_workqueue("my_ordered_wq", 0);
```

创建出来的ordered wq每次只激活一个worker。你依次`queue_work(wq, &work_a)`、`queue_work(wq, &work_b)`、`queue_work(wq, &work_c)`，它们一定按a→b→c的顺序执行，不会乱序，也不会并行。这在做一些需要严格顺序控制的后台操作时非常方便，比如块设备的flush链路。

当然，代价就是完全没有并行度。如果你的work都很耗时，ordered wq可能会成为瓶颈。此时你需要在"顺序保证"和"并发性能"之间做权衡——老问题，没有银弹。

## 调用链走读

来串一下从`queue_work()`到回调函数被执行的完整路径：

1. `queue_work(struct workqueue_struct *wq, struct work_struct *work)`
2. 内部调用`queue_work_node()` → `__queue_work()`
3. `__queue_work()`根据wq类型找到对应的`pool_workqueue`
4. 将work挂入`pool->worklist`链表尾部
5. 如果pool里有idle的worker，唤醒它；否则可能创建新worker
6. `worker_thread()`被唤醒后调用`process_one_work()`
7. `process_one_work()`取出work，执行`work->func(work)`
8. 回调函数返回后，worker继续处理下一个work，或者睡眠等待

```
queue_work()
  └── __queue_work()
        └── pwq = *wq->cpu_pwq[]
        └── list_add_tail(&work->entry, &pool->worklist)
        └── wake_up_worker(pool)
              └── worker_thread()
                    └── process_one_work()
                          └── work->func(work)
```

源码在`kernel/workqueue.c`里，有兴趣的话建议你顺着这个调用链翻一翻。特别留意`process_one_work()`里对`work->func`的调用点，你会发现它是在释放了pool锁之后调用的——这意味着你的回调在执行期间，pool里的其他worker可以并行处理别的work。

## 代码示例：work_struct的初始化和调度

实际使用workqueue非常简单，跟tasklet差不多，区别只是把`tasklet_schedule`换成`queue_work`。下面是一段完整的示例，展示了如何初始化work、如何调度到系统默认workqueue，以及如何创建并使用自定义workqueue：

```c
#include <linux/workqueue.h>

/* 定义work结构和数据 */
static struct work_struct my_work;
static struct workqueue_struct *my_wq;

/* work回调函数，运行在进程上下文 */
static void my_work_handler(struct work_struct *work)
{
    pr_info("work handler running in process context, pid=%d\n",
            current->pid);

    /* 这里可以睡眠！可以调用kmalloc(GFP_KERNEL)，可以获取mutex */
    msleep(10);  /* 完全没问题 */
}

/* 模块初始化 */
static int __init my_init(void)
{
    /* 方式一：使用系统默认的system_wq */
    INIT_WORK(&my_work, my_work_handler);
    schedule_work(&my_work);  /* 等价于 queue_work(system_wq, &my_work) */

    /* 方式二：创建自己的workqueue */
    my_wq = alloc_workqueue("my_wq", WQ_UNBOUND, 1);
    if (!my_wq)
        return -ENOMEM;

    /* 重新初始化并调度到自定义wq */
    INIT_WORK(&my_work, my_work_handler);
    queue_work(my_wq, &my_work);

    return 0;
}

/* 模块退出 */
static void __exit my_exit(void)
{
    /* 必须先销毁workqueue，确保所有work都已完成 */
    destroy_workqueue(my_wq);
}

module_init(my_init);
module_exit(my_exit);
```

几个关键点：

- `INIT_WORK()`宏负责初始化`work_struct`，绑定回调函数。同一个work在被调度前只能初始化一次，重复初始化会导致链表指针混乱。
- `schedule_work()`是最简接口，直接往`system_wq`里扔。它等价于`queue_work(system_wq, work)`，省了你传入wq参数的麻烦。
- 如果创建的是bound wq（默认），worker线程名会带CPU编号，比如`kworker/0:0`、`kworker/1:0`，在`ps`命令里很容易辨认。
- 模块卸载前**务必**调用`destroy_workqueue()`或者`cancel_work_sync()`，否则还在运行的work回调会导致oops。`cancel_work_sync()`会等待当前work执行完毕才返回，这个阻塞行为在卸载路径里通常是可接受的——总比你直接`rmmod`出一个use-after-free要好。

> 陷阱：`flush_work()`和`cancel_work_sync()`的区别容易搞混。`flush_work()`是等待work执行完成，但不取消已经提交的work；`cancel_work_sync()`则会尝试取消pending的work，如果已经在运行了就等待它结束。模块卸载时用后者更保险。
