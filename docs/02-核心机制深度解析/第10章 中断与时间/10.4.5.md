前面聊了线程化中断的机制，但你心里可能有个疑问：`request_threaded_irq()`主线早就有了，PREEMPT_RT还折腾什么？

好问题。这里有个根本区别。

**知识点106 [E]**

主线内核的`request_threaded_irq()`是"自愿"的——驱动作者主动选。但PREEMPT_RT的哲学是"强制"：一旦开了RT模式，**所有能线程化的中断，全部线程化**，没得商量。

普通内核也能体验一把"全民irq_thread"，靠的是`threadirqs`启动参数：

```bash
/* bootloader命令行加上 */
linux /boot/vmlinuz root=/dev/sda1 threadirqs
```

内核初始化时会为每个中断自动创建`irq_thread`，不需要驱动显式调用`request_threaded_irq()`。但这只是个"预览版"——它不改动调度器和锁的实现，真正的实时保障还得靠完整的RT补丁。

PREEMPT_RT的目标远比中断线程化宏大：**让内核中几乎所有路径都可被抢占。** 传统内核里关中断、拿spinlock、搞原子操作的地方，在RT眼里全是"不可抢占的毒瘤"。RT补丁把spinlock改成可睡眠的rt_mutex，缩小关中断区域，软中断也全面交给`ksoftirqd`线程。配合强制中断线程化，最终效果是这样的：

```mermaid
graph LR
    A[硬件中断] --> B[顶半部 几微秒]
    B --> C[irq_thread 被唤醒]
    C --> D[调度器裁决]
    D -->|有更高优先级RT任务| E[RT任务先跑]
    D -->|RT任务未就绪| F[irq_thread执行]
    E --> G[RT任务完成]
    G --> F
```

链路的关键在D节点——中断变成线程后就要过调度器这一关。你的SCHED_FIFO电机控制任务优先级90，irq_thread默认50？那对不起，网卡中断再多，调度器也会先让电机控制跑。

| 机制 | 主线内核 | PREEMPT_RT |
|:---|:---|:---|
| 中断线程化 | 驱动自愿，`request_threaded_irq()` | 强制，所有中断 |
| spinlock | 原子操作，不可抢占 | 映射为rt_mutex，可抢占 |
| 软中断 | 中断上下文直调/ksoftirqd | 全部由ksoftirqd线程处理 |
| 调度延迟 | 几十到几百微秒 | 几微秒到几十微秒 |

> ⚠️ **陷阱**：开了`threadirqs`不等于系统"实时了"。spinlock仍不可抢占，软中断仍可能直调——延迟瓶颈不止中断。我见过有人开这个参数就拍胸脯保证实时性，cyclictest一跑峰值还是几百微秒，根子在这里。

**知识点107 [I]**

PREEMPT_RT这个项目从2004年开始，Ingo Molnar和Thomas Gleixner等人一路往主线推。二十年里，RT补丁从独立的外部补丁集，被一点点拆解、打磨、合入主线。

早期是基础设施：hrtimer、generic IRQ层重构、线程化中断都是那时候进主线的。中期是锁的改造，把raw_spinlock和spinlock分开。到了近几年，`CONFIG_PREEMPT_RT`这个核心配置选项本身也终于进了主线：

```
Kernel Features  --->
    Preemption Model  --->
        (X) Fully Preemptible Kernel (Real-Time)
```

选中后，强制中断线程化、spinlock变rt_mutex、优先级继承PI全开，全部自动启用。你不再需要打外部补丁——至少在支持的架构上是这样。

```c
/* 代码里判断RT是否启用 */
#ifdef CONFIG_PREEMPT_RT
    /* RT环境下的特殊处理 */
#else
    /* 传统路径 */
#endif
```

但PREEMPT_RT的故事远未完结。核心补丁虽已合入，不同架构适配、外设驱动的RT兼容性验证仍是持续过程。ARM64上相对成熟，边缘架构和古老驱动可能还有坑。产品落地前，cyclictest跑几天几夜仍是必不可少的仪式：

```bash
# RT延迟测试
cyclictest -m -S -p 90 -i 200 -n -l 1000000
```

看Max列，峰值控制在几十微秒以内才算达标。偶尔飙高？ftrace抓一下——大概率是某个驱动在顶半部里干了不该干的事，故事又回到10.4.1的开头去了。
