想象一家医院的急诊室。深夜，120救护车呼啸而至，送来一位急救病人。急诊医生有几个铁律：第一，他不能接到一半说"我困了，去休息室睡会儿再回来处理"；第二，他只有急诊室这一小块操作空间，不能把病人推到走廊占满整个医院；第三，他必须尽快完成初步处置，因为下一位急救病人可能已经在路上了，而急诊室同一时间只能处理一个最紧急的病例。

Linux中断处理程序的**顶半部**（top half），就是这个急诊医生。

**知识点76 [I][M]**

顶半部有三条硬约束，每一条都是从血泪教训里总结出来的。

**不能睡眠。** 顶半部运行在中断上下文，不是进程上下文。这意味着它没有一个 backing 的 `task_struct`，调度器不认识它——它就像一个"黑户"。如果你在里面调用 `schedule()` 或者任何可能睡眠的函数（比如 `kmalloc(GFP_KERNEL)`、`mutex_lock()`），内核会直接在 `__might_sleep` 里给你甩一个 `BUG` 出来。我见过新手在定时器中断里想等一个信号量，结果整个系统直接 panic。记住：**中断上下文没有进程可以换出去，所以它不能睡眠，这是物理上不可能的。**

**栈空间极小。** 在32位系统上，顶半部的栈通常只有1个page（4KB）；64位上可能是2个page（8KB）。而且这还不是给你一个人用的——中断栈和内核栈的关系在不同架构上实现不同，x86-64从2.6.32开始引入了单独的 per-CPU 中断栈，但空间依然紧张。如果你在里面定义一个 `char buf[4096]` 的局部数组，恭喜你，栈溢出可能在下一个函数调用时就会以一种极其诡异的方式表现出来。我的建议是：**顶半部里不要有大局部变量，所有数据结构的准备都在底半部或进程上下文中做。**

**尽快完成。** 顶半部执行期间，根据中断控制器的配置和 `local_irq_disable` 的状态，当前CPU上同优先级（甚至全部）中断是被屏蔽的。你拖得越久，丢失中断的风险越大。在嵌入式系统里，我见过一个网络驱动的顶半部里做了太多校验工作，结果导致串口中断丢失，调试信息残缺不全，问题查了三天。

这三条约束决定了顶半部的"注册-执行"模型。内核提供 `request_irq`（较新代码用 `request_threaded_irq`）来注册顶半部处理函数：

```c
/* 经典的中断注册接口 */
int request_irq(unsigned int irq, 
                irq_handler_t handler,
                unsigned long flags,
                const char *name, 
                void *dev_id);
```

| 参数 | 含义 |
|------|------|
| `irq` | 硬件中断线号 |
| `handler` | 顶半部处理函数指针，原型 `irqreturn_t handler(int irq, void *dev_id)` |
| `flags` | 中断触发方式与属性（`IRQF_SHARED`、`IRQF_TRIGGER_RISING` 等） |
| `name` | 设备名，显示在 `/proc/interrupts` 中 |
| `dev_id` | 设备私有指针，共享中断时用于区分不同设备 |

`handler` 的返回值有三种：`IRQ_HANDLED`（已处理）、`IRQ_NONE`（不是我设备的，别滥发）、`IRQ_WAKE_THREAD`（配合 threaded irq 唤醒内核线程）。

!> **陷阱**：`dev_id` 必须是唯一指向你设备结构的指针。共享中断时，内核靠它来判断该调用哪个设备的 `free_irq`。我见过有人传个 `NULL` 进去，卸载驱动时 `free_irq(irq, NULL)` 把别人的中断也释放了，结果两个驱动同归于尽。

```mermaid
graph LR
    A[硬件中断信号] --> B[中断控制器]
    B --> C[CPU响应，关中断]
    C --> D[查irq_desc表]
    D --> E[执行handler: 顶半部]
    E --> F[标记底半部需要执行]
    F --> G[开中断，返回]
    G --> H[底半部异步执行]
```

**知识点77 [I]**

那为什么顶半部会有这些约束？根子在于**上下文**。

进程上下文里，CPU在执行某个进程的内核态代码，调度器随时可以通过 `need_resched` 把它换出去——因为有 `task_struct`，有 `mm_struct`，有完整的现场可以保存。但中断来临时，CPU是"借"当前运行进程的场地来执行中断代码的。这个借用关系非常脆弱：没有独立的地址空间可以换出，栈也是临时借用的或者极小的专用栈。

换句话说，中断上下文是调度器视线之外的"灰色地带"。调度器不知道你的存在，也就无法为你做上下文切换。不能睡眠，本质上是因为**没有谁能接管CPU**——当前进程的内核栈被你占了，你一睡，那个进程也跟着陪葬。栈空间小，是因为每个CPU都分配一个大的中断栈太浪费，而共享内核栈又太危险。尽快完成，是因为你握着中断关闭的锁，整个系统的响应延迟都捏在你手里。

早年间（2.4内核时代），顶半部和底半部还没有分得这么清，`cli()`/`sti()` 满天飞，一个慢速设备的顶半部能把整个系统的中断响应拖垮。后来内核逐渐演化出如今的顶半部/底半部分离架构——目的只有一个：**让急诊医生只做最关键的判断和分流，真正的治疗交给后续的专科医生。**
