**知识点104 [I] — threaded IRQ 的甜头和苦头**

Threaded IRQ 最吸引人的地方，说到底就三个字：**能睡觉**。hardirq 里不能调用 `msleep()`、不能拿 mutex、不能访问可能阻塞的接口——这条铁律被 threaded IRQ 打破了。中断处理一旦跑在进程上下文里，调度器就拿它当一个普通线程看待，该睡就睡，该等就等。

更妙的是优先级可控。你可以给中断线程绑一个具体的 `SCHED_FIFO` 优先级，比如：

```c
struct irqaction *action = kzalloc(sizeof(*action), GFP_KERNEL);
action->thread = kthread_create(irq_thread_fn, dev, "irq-%d", irq);
/* 把中断线程优先级设得比普通任务高，但低于硬中断 */
sched_setscheduler_nocheck(action->thread, SCHED_FIFO,
                           &(struct sched_param){ .sched_priority = 40 });
```

这样，这个中断线程就会和系统中的实时任务一起参与调度，谁优先级高谁先跑，再也不用像传统中断那样"一竿子插到底"。多个中断线程之间也能公平竞争，低优先级中断不会把高优先级线程饿死——调度器说了算。

但甘蔗没有两头甜。代价是**额外的上下文切换**。传统中断的处理路径是：硬件触发 → CPU 跳向量表 → 直接执行 `handler()` → 返回。Threaded IRQ 呢？硬件触发后，先跑一遍简短的硬中断 `handler()`（通常只关中断源、唤醒线程），然后切换到这个中断线程的上下文，再由调度器决定是否立即执行。一来二去，至少多了一次上下文切换。

问题就出在这里。如果你的中断来得特别频繁——比如网络万兆网卡收包，每秒百万级中断——这多出来的切换开销就会变成沉重的负担。每次切换要保存/恢复寄存器、刷新 TLB、调度器做决策，积少成多。说白了，高频中断场景下，你省的那点开发麻烦，抵不上性能的损失。

| 维度 | 传统中断 (hardirq) | Threaded IRQ |
|:---|:---|:---|
| 执行上下文 | 中断上下文 | 进程上下文（内核线程） |
| 能否睡眠 | 否，禁止阻塞 | 是，可调用 `msleep()`、mutex 等 |
| 优先级控制 | 无，由硬件/向量决定 | 可调，通过 `sched_setscheduler()` |
| 调度行为 | 抢占一切线程，不可被抢占 | 与普通线程公平竞争 |
| 上下文切换开销 | 最小（直接跳转返回） | 大（hardirq → 线程 → 调度） |
| 延迟 | 低且确定 | 受调度影响，有抖动 |
| 适用频率 | 高频（微秒级周期） | 中低频（毫秒级周期） |
| 代码复杂度 | 需自行管理底半部 | `request_threaded_irq()` 一站式搞定 |

**陷阱：别在 GPIO 翻转或微秒级定时中断上用 threaded IRQ。** 有个做工业控制的兄弟，把编码器中断（周期几十微秒）改成了 threaded，结果脉冲计数丢得一塌糊涂，排查了一周才发现是上下文切换把 timing 吃掉了。

**知识点105 [I] — 什么时候该上 threaded IRQ**

说了半天优缺点，关键问题还是：我这段中断处理，到底该不该用 threaded IRQ？

说实话，最典型的情况就两种。

**第一种：中断处理里必须访问会睡眠的资源。** 比如你的中断 handler 要读一个 I2C 传感器——I2C 总线操作底层要拿 `i2c_adapter` 的锁，传输过程中可能 `msleep()` 等总线空闲。这在 hardirq 里是万万做不得的，传统做法你得拆成顶半部和 tasklet/workqueue，代码七零八落。有了 threaded IRQ，一个函数写到底，逻辑清爽多了。

```c
static irqreturn_t sensor_irq_thread(int irq, void *data)
{
    struct sensor_dev *dev = data;
    /* 直接在 interrupt thread 里读 I2C，可以睡眠 */
    i2c_smbus_read_byte_data(dev->client, REG_STATUS);
    ...
    return IRQ_HANDLED;
}
```

**第二种：你需要给不同中断精细地排优先级。** 比如一个系统里既有触摸屏中断（延迟敏感但不频繁），又有 SD 卡插拔检测（不敏感）。用 threaded IRQ 你可以把触摸屏中断线程设成 `SCHED_FIFO` 优先级 50，SD 卡设成 10，调度器自然会按你的意图来分配 CPU。传统中断做不到这一点，来了就处理，没得商量。

反过来，如果你处理的是一个周期固定的千兆网卡收包中断、或者 GPIO 按键去抖（几十毫秒的抖动时间里你根本不在乎那点切换开销），那就老老实实走老路。高频硬实时走 hardirq + tasklet，低频才上 threaded IRQ——选工具之前，先量量你的中断周期。
