**知识点74 [I]** —— 看中断去哪了？/proc/interrupts是最好用的"望远镜"。

```
$ cat /proc/interrupts
           CPU0       CPU1       CPU2       CPU3
  0:         23          0          0          0   IO-APIC    2-edge      timer
  1:          0        180          0          0   IO-APIC    1-edge      i8042
  8:          1          0          0          0   IO-APIC    8-edge      rtc0
  9:          0          0          0          0   IO-APIC    9-fasteoi   acpi
 42:       1024       2048        512        256   PCI-MSI 65536-edge  eth0-rx-0
```

注意eth0-rx-0那行——CPU1上2048次，CPU0才1024次，分布不均对吧？左边第一列是全局中断号，后面每一列对应一颗CPU的中断触发次数。长期盯着看，能一眼发现谁把某颗核"打爆"了。

那怎么把中断赶到别处去？打开`/proc/irq/<n>/smp_affinity`就行。

```bash
# 查看irq 42当前亲和性
cat /proc/irq/42/smp_affinity
# 输出: f  （二进制1111，四颗CPU都能处理）

# 只让CPU1处理（bitmask：0b0010 = 2）
echo 2 > /proc/irq/42/smp_affinity
```

这个bitmask跟CPU编号对应：`0x01`(1)→CPU0，`0x02`(2)→CPU1，`0x04`(4)→CPU2，以此类推。内核收到中断后，直接把42号硬中断派发给指定的CPU。

**知识点75 [I]** —— 为什么要手动绑？一句话：别让一颗核干所有活。所有网卡中断砸在CPU0上，这颗核就会又当"收包员"又当"调度员"，cache miss还高。把RX队列分散到多核，缓存局部性和吞吐量都能好不少。早年间我见过一台4核服务器，满载时CPU0的软中断飙到90%，另外三颗核闲着——一查irqbalance压根没跑，所有网卡中断全堆在CPU0上。手动绑完，吞吐量直接提了四成。

不过提醒一句：smp_affinity只对**硬中断**有效，软中断（ksoftirqd）的调度是另一回事。网卡多队列场景下，通常`smp_affinity`跟队列数一一对应，配合上RSS（Receive Side Scaling）才最省事。你要是不确定怎么分，先装个`irqbalance`让它自动跑，再微调。

> **陷阱：** 用`taskset`绑ksoftirqd进程没用！软中断处理的是已经分发完的数据，真正的"拦截"发生在硬中断层，必须用`smp_affinity`改。
