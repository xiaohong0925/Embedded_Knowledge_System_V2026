## 12.5.3 断电安全实践与选型

> **核心观点**：掉电安全是有代价的——最高安全级别写性能下降 15-20%。关键数据用最高安全，临时数据用最低安全，中间数据用中等安全。安全不是越多越好，而是够用就好。

---

### 知识点 197 [E][M] 断电场景六大常见陷阱

嵌入式设备运行环境复杂，供电不稳定、人为误操作、电池耗尽等因素均可能触发意外掉电。以下六种陷阱是工程实践中最高频的故障根因，覆盖应用层、文件系统层与存储介质层。

**陷阱 1：不写 `fsync` 导致掉电数据丢失**

应用程序写完文件后仅调用 `fclose()` 或依赖进程退出时的隐式刷盘，数据仍滞留在页缓存（Page Cache）中。掉电后，已"成功写入"的数据完全丢失。这是最普遍且最难察觉的陷阱——测试环境通常有正常关机流程，问题只在真实掉电时暴露。

**陷阱 2：启用 `atime` 导致 Flash 寿命衰减**

每次读取文件都会更新 inode 的访问时间戳，产生额外的写回流量。对于读取频繁但内容不变的配置文件、共享库或日志轮转后的旧日志，`atime` 更新会持续磨损 Flash 存储块，加速达到擦写极限（P/E Cycles）。

**陷阱 3：ext4 直接运行在裸 NAND 上无法管理坏块**

ext4 依赖底层提供线性的块设备接口，本身不具备坏块管理（BBM）、磨损均衡（Wear Leveling）或 ECC 纠错能力。裸 NAND 无 FTL（Flash Translation Layer）时，必须使用 UBI/UBIFS、YAFFS2 或 JFFS2 等原生管理坏块的文件系统。

**陷阱 4：误用 `data=ordered` 以为数据受 journal 保护**

`data=ordered` 仅保证元数据（metadata）原子性提交到 journal，并在提交前将关联的用户数据块刷盘。它**不保证**用户数据本身被记录到 journal 中。掉电时，如果数据块尚未刷盘，文件内容仍可能损坏或表现为"旧数据+新元数据"的不一致状态。

**陷阱 5：忽视 `discard`/`trim` 导致 eMMC 性能持续下降**

eMMC 内部 FTL 需要知道哪些逻辑块已被删除，才能回收空闲页并执行垃圾回收。长期未开启 `discard` 时，FTL 将大量已删除块视为有效数据，写入放大（WAF）急剧上升，随机写性能随使用时间线性衰减。

**陷阱 6：`tmpfs` 存储持久数据导致重启丢失**

`/tmp`、`/var/run` 等目录使用 `tmpfs` 时，数据完全驻留内存。将需要持久化的配置、状态或日志写入这些路径，重启后数据必然丢失。此问题在系统升级脚本或容器化部署中尤为常见。

#### 排查命令示例

```bash
# 检查当前挂载选项与屏障状态
findmnt -no OPTIONS /data

# 查看文件系统日志模式（ordered/journal/writeback）
tune2fs -l /dev/mmcblk0p2 | grep "Default mount options"

# 统计 atime 引发的写流量（观察 bi/bo 增量对比）
vmstat 1 10

# 检测 discard 是否启用
lsblk -D /dev/mmcblk0

# 确认 tmpfs 挂载点与大小
mount | grep tmpfs
df -h | grep tmpfs

# 模拟掉电测试（物理断电 / 热插拔 / 电源开关循环）
# 配合 dd + sync / fsync 对比数据完整性
```

```text

│               ├── 否 → 【陷阱4】评估升级为 data=journal

│                       ├── 是 → 【陷阱3】迁移至 UBIFS/YAFFS2
│                       └── 否 → 是否启用 barrier？

└── 否 → Flash 寿命是否异常衰减？

        │       ├── 否 → 【陷阱2】挂载时添加 noatime
        │       └── 是 → 检查写入频率与日志策略
        └── 否 → 写性能是否持续下降？

                        │       └── 否 → 检查固件擦除流程

```

#### 六大陷阱症状、根因与修正对照表

| 陷阱 | 典型症状 | 技术根因 | 修正措施 |
|:---|:---|:---|:---|
| 1. 不写 `fsync` | 掉电后文件为空或旧版本 | 数据滞留页缓存，未落盘 | 关键写操作后调用 `fsync()`/`fdatasync()`；或打开文件时使用 `O_DSYNC`/`O_SYNC` |
| 2. `atime` 磨损 | Flash P/E 周期消耗过快，`vmstat` 写流量与读流量成正比 | 每次读文件触发 inode atime 更新回写 | 挂载参数添加 `noatime` 或 `relatime` |
| 3. ext4 跑裸 NAND | 坏块出现后出现不可修复的 I/O 错误，文件系统只读 | ext4 无坏块管理能力，依赖底层 FTL | 裸 NAND 改用 UBIFS + UBI；或外置 FTL 控制器 + ext4 |
| 4. `data=ordered` 误解 | 掉电后文件大小正确但内容全 0 或垃圾数据 | 用户数据不在 journal 中，仅元数据受保护 | 关键分区改用 `data=journal`；或应用层做双缓冲写 |
| 5. 忽视 `discard` | eMMC 使用数月后随机写 IOPS 下降 50% 以上 | FTL 无法回收无效页，垃圾回收开销增大 | 挂载添加 `discard`；或每周执行 `fstrim -v /data` |
| 6. `tmpfs` 存持久数据 | 每次重启配置归零、日志缺失 | `tmpfs` 基于内存，不刷任何存储介质 | 持久数据移至 `/data` 等物理分区；`tmpfs` 仅存放运行时临时文件 |

### 知识点 198 [E] 安全级别与性能 Trade-off

断电保护的安全强度直接影响 I/O 吞吐性能。通过组合 data 模式、barrier 开关与 fsync 策略，可构建三级安全体系，按数据关键程度差异化配置。

**最高安全级：`data=journal` + `barrier=1` + 每次写后 `fsync()`**

此配置下，用户数据与元数据均写入 journal，barrier 强制按序刷盘，`fsync` 确保应用级持久化。掉电后最多丢失最后一次 `fsync` 之后的数据，journal 重放可恢复文件系统一致性。**代价是写性能下降 15-20%**，因为同一份数据经历"journal 写入 → 数据区写入"双路径，且 barrier 强制串行化刷盘操作，无法合并批量 I/O。

适用场景：金融交易记录、设备校准参数、OTA 升级包等一旦丢失将导致系统不可恢复的关键数据。

**中等安全级：`data=ordered` + `barrier=1`**

元数据受 journal 保护，数据块在元数据提交前强制刷盘，barrier 保证顺序。不写 `fsync` 时，最多丢失最后 5 秒（默认 commit 周期）内的数据；写了 `fsync` 则数据级安全。**写性能较最高安全级提升约 10-15%**，因用户数据不重复写入 journal。

适用场景：普通业务日志、用户配置文件、中等重要度的缓存数据。

**最低安全级：`data=writeback` + `barrier=0`**

数据块与元数据提交无强制顺序，禁用 barrier 后 I/O 调度器可自由重排序写操作，合并度最高。掉电风险最大：可能出现元数据已更新但数据块仍是旧内容，或文件指向未初始化块（暴露旧数据）。**写性能最优**，接近裸设备吞吐。

适用场景：临时下载目录、编译中间文件、可重建的缓存、重启后自动刷新的监控数据。

#### mount 选项示例

```bash
# 最高安全级 — 关键数据分区
mount -t ext4 -o data=journal,barrier=1,noatime /dev/mmcblk0p3 /critical

# 中等安全级 — 通用数据分区（嵌入式默认推荐）
mount -t ext4 -o data=ordered,barrier=1,noatime /dev/mmcblk0p2 /data

# 最低安全级 — 临时数据分区
mount -t ext4 -o data=writeback,barrier=0,noatime /dev/mmcblk0p4 /tmpdata
```

#### 安全级别与性能 Trade-off

```text
安全强度
    ▲
  高│  ┌────────────────── 最高安全 (data=journal+barrier+fsync)
    │  │                         写性能: -15~20%
    │  │
  中│  ├────────────────── 中等安全 (data=ordered+barrier)
    │  │                         写性能: -5~8%
    │  │
  低│  └────────────────── 最低安全 (data=writeback,barrier=0)
    │                            写性能: 基准
    └──────────────────────────────► 写性能
```

#### 三级安全配置与适用场景

| 安全级别 | mount 选项组合 | fsync 要求 | 写性能影响 | 掉电丢失范围 | 适用场景 |
|:---|:---|:---|:---|:---|:---|
| 最高安全 | `data=journal,barrier=1` | 每次写后显式 `fsync` | 下降 15-20% | 仅最后一次 `fsync` 后数据 | 密钥库、校准参数、OTA 包、交易记录 |
| 中等安全 | `data=ordered,barrier=1` | 关键写调用 `fsync` | 下降 5-8% | 最近 5 秒内数据（默认 commit=5） | 应用日志、用户配置、运行时状态 |
| 最低安全 | `data=writeback,barrier=0` | 无需 `fsync` | 基准性能 | 不可预估，元数据可能不一致 | 临时文件、可重建缓存、编译中间产物 |

在实际项目中，建议将存储划分为多个分区，按数据关键程度分别挂载为不同安全级别，而非整盘统一配置。这种"分区差异化安全策略"是平衡可靠性与性能的最佳实践。
