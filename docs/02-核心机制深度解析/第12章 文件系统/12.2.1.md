### 12.2.1 页缓存机制与 address_space

你`write()`了一个文件，函数立刻返回了——数据真的写到磁盘了吗？没有！它在 page cache 里躺着，等待 flusher 线程来收拾。这就是 Linux 文件系统的核心设计：写操作先快后慢。这种基于页缓存（Page Cache）的 I/O 架构是理解 Linux 文件系统性能、数据一致性和故障排查的核心基础。

---

#### 知识点 172 [E][M] 页缓存工作机制

**页缓存（Page Cache）的本质与定位**

页缓存是 Linux 内核在内存中维护的磁盘数据缓存层，位于 VFS 与具体文件系统之间。其核心思想非常简单：读文件时，数据先从磁盘加载到内存的 page cache 中；后续对该文件同一区域的读取操作，直接从内存返回，无需再次访问磁盘。对于写操作，数据同样先写入 page cache，由内核后台线程异步刷回磁盘。这种设计将慢速的块设备 I/O 转换为快速的内存访问，极大提升了文件系统性能。

页缓存以页（page，通常 4KB）为基本单位管理数据。每个被缓存的磁盘页面对应一个`struct page`实例。当进程发起`read()`系统调用时，内核首先检查目标数据是否已在 page cache 中——若命中则直接复制到用户空间；若未命中则触发缺页或读取流程，从磁盘加载数据。

**`address_space`：inode 到 page cache 的桥梁**

`struct address_space`是连接 inode 与其页缓存的核心数据结构，定义在`<linux/fs.h>`中。每个 inode 实例通过`i_mapping`字段指向一个`address_space`对象，而`address_space`的`host`指针回指到所属 inode。这一双向关联使得内核可以从文件路径快速定位到其缓存页集合。

`address_space`中的关键字段包括：`i_pages`（存储所有缓存页索引的根节点）、`nrpages`（当前缓存页总数）、`a_ops`（指向`address_space_operations`结构，定义了读取、写入、预读等回调函数）。`a_ops`中的`readpage`和`writepage`方法由具体文件系统实现，负责底层的数据传输。

这种设计的精妙之处在于解耦：VFS 层只关心"某个文件的第 N 页是否在缓存中"，而无需关心底层是 ext4、XFS 还是 Btrfs。页缓存机制通过`address_space`为所有文件系统提供了统一的缓存管理层。

**Radix Tree / XArray：页帧索引结构**

为了高效地从文件偏移量查找对应的`struct page`，内核早期使用 Radix Tree（基数树），从 Linux 4.20 开始逐步替换为 XArray。两者都是基于索引（index，即文件内的页编号）的快速查找结构。

XArray 相比 Radix Tree 的优势在于接口更简洁、并发控制更高效（利用 RCU 实现读端无锁）、且天然支持多索引条目（multi-index entries，用于表示大页等连续区域）。`address_space->i_pages`作为 XArray 的根，通过`xa_load()`、`xa_store()`等操作管理页帧。给定一个页索引`pgoff_t index`，查找对应 page 的时间复杂度为 O(log₆₄N)，在树高度通常不超过 3~4 的情况下，实际性能接近 O(1)。

**核心代码：`pagecache_get_page()`调用链**

```c
// 页缓存查找核心函数
struct page *pagecache_get_page(struct address_space *mapping, pgoff_t index,
                                 int fgp_flags, gfp_t gfp)
{
    struct page *page;

repeat:
    // 1. 在 XArray 中查找目标页
    page = find_get_entry(mapping, index);
    if (xa_is_value(page))        // 可能是 swap entry
        return page;
    if (page && !fgp_flags)       // 找到且无需额外操作
        return page;

    // 2. 未找到，需分配新页
    if (!page && (fgp_flags & FGP_CREAT)) {
        page = __page_cache_alloc(gfp);
        if (!page)
            return NULL;

        // 3. 将新页插入 XArray
        err = add_to_page_cache_lru(page, mapping, index, gfp);
        if (unlikely(err)) {
            put_page(page);
            if (err == -EEXIST)
                goto repeat;    // 竞争条件，重试查找
            return NULL;
        }
    }
    return page;
}
```

上述流程展示了页缓存查找的核心逻辑：首先在 XArray 中查找（`find_get_entry`），若未命中且调用方允许创建（`FGP_CREAT`），则分配新页并插入缓存（`add_to_page_cache_lru`）同时加入 LRU 链表供回收参考。

**`read()`路径：缓存命中与未命中的分支流程**

```mermaid
graph TD
    A[read(fd, buf, count)] --> B[VFS: vfs_read]
    B --> C[generic_file_read_iter]
    C --> D{pagecache_get_page<br/>查找目标页}

    D --"缓存命中"--> E[拷贝 page 数据到用户 buf]
    E --> F[mark_page_accessed<br/>标记访问]
    F --> G[返回读取字节数]

    K --> L[拷贝数据到用户 buf]
    L --> G
```

**Page Cache I/O vs 直接 I/O（Direct I/O）对比**

| 对比维度 | Page Cache I/O（默认） | Direct I/O（O_DIRECT） |
|---------|----------------------|----------------------|
| 数据路径 | 用户空间 ↔ Page Cache ↔ 磁盘 | 用户空间 ↔ 磁盘（绕过缓存） |
| 读延迟 | 命中时微秒级，未命中时毫秒级 | 始终毫秒级（磁盘延迟） |
| 写语义 | 异步写：数据先存缓存，后台刷盘 | 同步写：直接提交磁盘 |
| 数据一致性 | 存在丢失风险（断电丢数据） | 更高一致性（但非绝对） |
| 内存占用 | 消耗 RAM 缓存数据 | 几乎不消耗（仅缓冲结构） |
| 适用场景 | 通用文件访问、重复读取 | 数据库、大文件顺序访问、自缓存应用 |
| 对齐要求 | 无特殊要求 | 文件偏移、长度、buffer 地址均需对齐块大小 |
| 预读支持 | 内核自动预读 | 无预读，完全由应用控制 |

#### 知识点 173 [E] `write()`如何写入 Page Cache

**写操作的本质：用户空间到内核 page 的复制**

当进程调用`write(fd, buf, count)`时，内核并不会直接将数据刷到磁盘。相反，整个写路径的核心是找到（或创建）目标文件区域对应的缓存页，然后将用户空间数据复制进去。关键函数`copy_from_user()`将用户缓冲区数据安全地拷贝到内核 page 中，同时完成用户态地址的合法性校验。

**关键步骤与状态标记**

数据成功拷贝到 page cache 后，内核执行两个关键标记操作：

1. **`mark_page_accessed(page)`**：将该页标记为"最近被访问"，更新其 LRU 状态。这确保该页在内存回收（kswapd）时被赋予较高的留存优先级，避免"刚写入就被回收"的尴尬局面。

2. **`set_page_dirty(page)`**：将该页标记为"脏页"（设置`PG_dirty`标志位）。脏页表示其内存中的数据与磁盘上的数据不一致，必须在后续某个时间点被写回磁盘。脏页由内核`flusher`线程（即`bdflush`/`kflushd`的后继者）定期扫描并刷盘。

**为什么`write()`不直接写磁盘？**

这种"延迟写回"设计背后有深层考量：

- **性能聚合**：多个小写操作可以合并为一次大的磁盘 I/O，减少磁头寻道和旋转延迟，充分利用磁盘顺序写带宽。
- **避免阻塞用户态**：`write()`立刻返回，进程无需等待慢速磁盘操作，系统吞吐量大幅提升。
- **覆盖写消除**：同一页上的多次写操作可能仅保留最终结果，避免中间状态的无效刷盘。
- **调度灵活性**：内核可以根据系统负载、I/O 压力、磁盘空闲时段等因素智能选择最佳刷盘时机。

当然，代价也很明显：系统崩溃或断电时，尚未刷盘的脏数据会丢失。这就是为什么关键数据写入后必须调用`fsync()`或`fdatasync()`强制刷盘。

**核心代码：`generic_file_write_iter()`调用链**

```c
// 通用文件写入口
ssize_t generic_file_write_iter(struct kiocb *iocb, struct iov_iter *from)
{
    struct file *file = iocb->ki_filp;
    struct address_space *mapping = file->f_mapping;
    struct inode *inode = mapping->host;
    loff_t pos = iocb->ki_pos;
    ssize_t written;

    // 1. 检查文件是否以写模式打开
    if (!(file->f_mode & FMODE_WRITE))
        return -EBADF;

    // 2. 获取 inode 写锁，防止并发写冲突
    inode_lock(inode);

    // 3. 处理 O_APPEND：将位置移到文件末尾
    if (file->f_flags & O_APPEND)
        pos = iocb->ki_pos = i_size_read(inode);

    // 4. 核心写操作：逐页处理
    written = generic_perform_write(file, from, pos);

    // 5. 更新文件大小（如果发生扩展）
    if (pos + written > i_size_read(inode))
        i_size_write(inode, pos + written);

    inode_unlock(inode);

    // 6. 更新文件偏移和修改时间
    iocb->ki_pos = pos + written;
    file->f_mtime = current_time(inode);
    return written;
}
```

`generic_perform_write()`进一步调用`iov_iter_copy_from_user_atomic()`完成实际数据拷贝，并在每页处理完毕后调用`flush_dcache_page()`和`set_page_dirty()`。

    C --> D{检查权限<br/>O_APPEND?}

**`write()`完整流程步骤表**

| 步骤 | 函数/操作 | 说明 |
|-----|----------|------|
| 1 | `vfs_write()` | VFS 入口，参数校验，调用文件操作表的`write_iter` |
| 2 | `generic_file_write_iter()` | 通用写处理，加 inode 锁，处理`O_APPEND` |
| 3 | `generic_perform_write()` | 逐页处理用户数据 |
| 4 | `pagecache_get_page(FGP_CREAT)` | 查找或创建目标缓存页 |
| 5 | `copy_from_user_atomic()` | 将用户数据拷贝到内核 page |
| 6 | `set_page_dirty()` | 标记 page 为脏，加入脏页链表 |
| 7 | `mark_page_accessed()` | 更新 LRU 状态，提高回收优先级 |
| 8 | 更新`i_size`与时间戳 | 若写入超出原大小则扩展文件 |
| 9 | 返回写入字节数 | `write()`系统调用返回（数据尚未落盘） |
| 10 | `flusher`线程异步刷盘 | 满足条件后`writeback_single_inode()`刷脏页 |
| 11 | `fsync()`强制刷盘（可选） | 应用层主动要求同步，确保数据持久化 |

**实践案例：设备断电导致配置丢失**

某嵌入式设备在运行过程中通过配置文件保存运行参数。开发团队发现设备偶发断电重启后，最近一次保存的配置全部丢失，回退到更早版本。问题根因定位：保存配置的代码路径仅调用了`write()`，从未调用`fsync()`。由于写操作仅将数据存入 page cache，脏页尚未被 flusher 线程刷回磁盘时设备断电，数据自然丢失。

解决方案：在所有关键配置写入路径后增加`fsync(fd)`调用，确保数据在函数返回前已落盘。对于性能敏感场景，可采用`fdatasync()`（仅刷数据不刷元数据）或批量写入后统一`fsync`的策略，在持久性与性能之间取得平衡。此案例深刻揭示了"`write()`返回不等于数据落盘"这一 Linux I/O 基本事实，也是嵌入式与服务器系统开发中必须内化的设计准则。
