# 12.4.1 MTD子系统基础

> 裸Flash不是块设备——你不能像操作硬盘一样直接读写它。它有三大怪癖：操作单位是擦除块（不是扇区）、写之前必须先擦除、没有内置坏块管理。MTD子系统就是来驯服这头野兽的。

## 知识点187 MTD本质：为裸Flash提供统一接口 [E][M]

### 一、MTD的设计定位

MTD（Memory Technology Device，存储技术设备）是Linux内核中为**裸Flash芯片**（Raw Flash）提供的抽象层。与SATA/NVMe硬盘、eMMC、SD卡等块设备不同，NOR Flash和NAND Flash这类存储介质在硬件行为上有着根本性的差异——它们没有内置的控制器来屏蔽底层物理特性，需要软件直接面对Flash的所有怪癖。MTD子系统的使命正是：**为上层提供一个统一的设备视图，同时向下隐藏不同厂商、不同类型Flash芯片的差异**。

在内核架构中，MTD位于VFS和块设备层之下、硬件驱动之上。用户空间可以通过字符设备接口（`/dev/mtdN`）直接访问Flash，也可以通过MTD之上的UBI/UBIFS、JFFS2、YAFFS2等文件系统间接使用。这种分层设计让Flash管理既具备裸设备访问的灵活性，又支持高级文件系统的便利性。

### 二、MTD与块设备的三大本质区别

理解MTD的关键，在于牢牢把握它与标准块设备的三条分界线：

| 维度 | 块设备（硬盘/eMMC） | MTD设备（裸Flash） |
|:---|:---|:---|
| **最小操作单位** | 扇区（Sector），通常512B或4KB | 擦除块（Erase Block），NOR为4~128KB，NAND为128KB或256KB |
| **写入约束** | 直接覆盖写（in-place update） | **必须先擦后写**（Read-Erase-Write），比特只能从1变0，不能从0变1 |
| **坏块管理** | 控制器内置坏块管理（BBM），对系统透明 | **无内置坏块管理**，必须由软件（驱动/文件系统）检测并跳过 |

**第一，操作单位是擦除块而非扇区。** 向Flash写入一个新字节前，必须先把整个擦除块清零。这意味着哪怕只修改1字节数据，也可能触发128KB擦除+重写操作。这种"粗粒度"的写入特性，决定了Flash不适合频繁的随机小写场景。

**第二，必须先擦后写。** Flash的物理特性决定了每个存储单元（cell）的电荷只能单向变化：擦除操作将所有位设为1，写入操作只能将1改为0。如果要将0变回1，必须执行整块的擦除。这个约束是Flash区别于RAM和硬盘的根本特征，也是Wear Leveling和Log-structured File System等技术的存在前提。

**第三，无坏块管理。** 硬盘和eMMC的控制器会维护坏块表（BBT），自动将坏块重映射到备用区域，操作系统对此完全无感知。而裸Flash出厂时即可能带有坏块（NAND Flash数据手册通常允许1%~2%的初始坏块），且使用过程中会持续产生新的坏块。MTD层和上层文件系统必须显式地检测坏块标记（通常位于OOB区域的第一或第二字节），并维护坏块表来规避这些区域。

```
┌─────────────────────────────────────────────────────┐
│                    用户空间 (VFS)                      │
├──────────────────────┬──────────────────────────────┤
│  UBIFS / JFFS2 /     │   块设备模拟层 (mtdblock)      │
│  YAFFS2 / cramfs     │   （不推荐用于可写分区）          │
├──────────────────────┴──────────────────────────────┤
│                  MTD 子系统核心层                       │
│         （mtd_info 统一管理，隐藏芯片差异）               │
├─────────────────────────────────────────────────────┤
│   NOR Flash驱动    │    NAND Flash驱动               │
│   (cfi_probe等)    │    (onfi/samsung等)            │
├─────────────────────────────────────────────────────┤
│   SPI/Memory总线    │    NAND控制器/EBI总线           │
├─────────────────────────────────────────────────────┤
│         NOR芯片          │        NAND芯片              │
└─────────────────────────────────────────────────────┘
```

### 三、MTD设备类型：NOR vs NAND

MTD子系统管理的裸Flash主要分为两大阵营：

| 特性 | NOR Flash | NAND Flash |
|:---|:---|:---|
| **读取粒度** | 字节级随机读（Byte-addressable） | 页级顺序读（Page-based，通常2KB/4KB） |
| **写入粒度** | 按字/字节写入 | 必须按页写入 |
| **擦除块大小** | 4KB ~ 128KB | 128KB ~ 256KB |
| **容量密度** | 较小（MB级，通常≤128MB） | 大（GB级，可达数TB） |
| **读取速度** | 快（类似内存随机访问） | 较慢（需命令序列+等待） |
| **写入/擦除速度** | 慢（擦除可达秒级） | 快（擦除毫秒级） |
| **位翻转（Bit-Flip）** | 极少 | 常见，必须依赖ECC纠错 |
| **接口** | Memory-mapped（可直接执行代码） | I/O命令接口（地址/数据复用） |
| **成本/位** | 高 | 低 |
| **典型应用** | 启动代码（U-Boot、固件）、代码执行（XIP） | 大数据存储（根文件系统、用户数据） |

**NOR Flash**的核心优势在于**内存映射（eXecute-In-Place, XIP）**能力——CPU可以直接从NOR Flash取指执行，无需先加载到RAM。这使得NOR成为嵌入式系统启动介质的首选。但NOR的写入和擦除速度很慢，且容量扩展受限。

**NAND Flash**以极高的存储密度和低廉的成本著称，是大容量数据存储的绝对主力。但其接口是命令式的（发送命令→发送地址→读写数据），且存在位翻转问题——随着制程缩小，相邻单元间的电荷干扰会导致读出的数据偶尔出现比特错误，因此**ECC（Error Correcting Code）校验是NAND使用的必要条件**，而非可选功能。

### 四、struct mtd_info：MTD设备的核心描述

内核中每个MTD设备都由一个`struct mtd_info`实例描述。以下是其关键字段：

```c
struct mtd_info {
    u_char type;                    /* MTD类型：MTD_NORFLASH / MTD_NANDFLASH / MTD_MLCNAND */
    uint64_t size;                  /* 总容量（字节） */
    u_int erasesize;                /* 擦除块大小 */
    u_int writesize;                /* 写入页大小（NAND）/ 1（NOR） */
    u_int oobsize;                  /* 每页OOB区域大小（NAND） */
    
    /* 核心操作函数集 */
    struct mtd_info *next;
    int (*_read)(struct mtd_info *mtd, loff_t from, size_t len,
                 size_t *retlen, u_char *buf);
    int (*_write)(struct mtd_info *mtd, loff_t to, size_t len,
                  size_t *retlen, const u_char *buf);
    int (*_erase)(struct mtd_info *mtd, struct erase_info *instr);
    int (*_read_oob)(struct mtd_info *mtd, loff_t from,
                     struct mtd_oob_ops *ops);
    int (*_write_oob)(struct mtd_info *mtd, loff_t to,
                      struct mtd_oob_ops *ops);
    int (*_block_isbad)(struct mtd_info *mtd, loff_t ofs);
    int (*_block_markbad)(struct mtd_info *mtd, loff_t ofs);
    
    struct device dev;
    int usecount;                   /* 打开引用计数 */
    void *priv;                     /* 驱动私有数据 */
};
```

`type`字段标识MTD设备的具体类型，`size`、`erasesize`、`writesize`、`oobsize`共同定义了Flash的物理几何参数。`erasesize`是MTD与块设备分野的核心参数——它决定了分配和管理数据的最小单元。所有以`_`前缀命名的函数指针构成了MTD设备的标准操作向量，具体驱动（如`s3c2410_nand`、`spi_nor`）负责实现这些回调，向MTD核心层注册设备。

## 知识点188 MTD核心操作 [E]

### 一、三大核心API

MTD导出标准化的读、写、擦除三件套，三者构成Flash操作的完整闭环：

**`mtd_read()`** —— 从指定偏移读取数据，不涉及擦除。但NAND读取可能触发位翻转，返回正数表示ECC已纠正的错误，需调用者关注。

**`mtd_write()`** —— 向指定偏移写入数据。**写入前必须确保目标区域已被擦除**（全1状态），否则新数据与旧数据按位与会导致数据损坏。写入地址须与`writesize`对齐（NAND按页写）。此外，Flash的写入只能从1变0，一旦某位置0后想恢复为1，整块擦除是唯一途径。

**`mtd_erase()`** —— 通过`struct erase_info`指定擦除范围，`addr`和`len`必须对齐到`erasesize`。擦除是Flash最耗时操作（NAND 1~3ms/块，NOR数百ms到1s），且擦除次数有限（SLC约10万次，MLC约3000~1万次），**任何擦除调度都必须考虑磨损均衡**。

```c
struct erase_info {
    struct mtd_info *mtd;
    uint64_t addr;          /* erasesize对齐 */
    uint64_t len;           /* erasesize整数倍 */
    uint64_t fail_addr;
    u_char state;           /* PENDING / DONE / FAILED */
};
```

### 二、OOB区域：Flash的"幕后管家"

NAND Flash每页除主数据区（Main Area）外，还附带**OOB（Out-Of-Band）**备用区域：

```
┌──────────────────────────────────┬─────────────────────┐
│      Main Area (2048 bytes)      │  OOB Area (64 B)    │
│                                  │  ┌──────────────┐   │
│  用户数据（文件内容等）           │  │ Bad Block Mk │1B │
│                                  │  ├──────────────┤   │
│  按页读写，存储的"正面舞台"      │  │ ECC Syndrome │6B │
│                                  │  ├──────────────┤   │
│                                  │  │ FS Metadata  │N B│
│                                  │  └──────────────┘   │
└──────────────────────────────────┴─────────────────────┘
                    NAND Flash 页结构（2048+64B）
```

OOB承载三项关键职责：**坏块标记**（首字节非`0xFF`即表示坏块，MTD驱动初始化时扫描全片建立坏块表）、**ECC校验码**（NAND的物理特性决定了读出的数据几乎必然存在位翻转，ECC引擎读取时重新计算校验子并与OOB存储值比对，Hamming码可纠1位/检2位，BCH可纠数十位，制程越先进所需ECC强度越高）、**文件系统元数据**（YAFFS2的Object Header、JFFS2节点标记等，帮助建立逻辑到物理的映射）。

### 三、MTD分区表

MTD支持将物理Flash划分为多个逻辑分区。定义方式有两种：

**1. 内核命令行** —— 通过`mtdparts`参数静态定义：

```
mtdparts=nand0:512k(bootloader),128k(env),4M(kernel),-(rootfs)
```

**2. 设备树（DTB）** —— ARM架构主流方式：

```dts
partitions {
    compatible = "fixed-partitions";
    bootloader@0 { label = "bootloader"; reg = <0x0 0x80000>; read-only; };
    kernel@80000   { label = "kernel";   reg = <0x80000 0x400000>; };
    rootfs@480000  { label = "rootfs";   reg = <0x480000 0>; };
};
```

### 四、MTD操作示例

```c
static int my_mtd_operate(struct mtd_info *mtd)
{
    struct erase_info erase = {0};
    u_char *buf = kmalloc(mtd->writesize, GFP_KERNEL);
    size_t retlen;

    /* 1. 检查坏块 */
    if (mtd->_block_isbad(mtd, 0x480000))
        return -EIO;

    /* 2. 擦除（必须先擦后写！） */
    erase.mtd = mtd; erase.addr = 0x480000; erase.len = mtd->erasesize;
    if (mtd_erase(mtd, &erase) || erase.state == MTD_ERASE_FAILED)
        goto err;

    /* 3. 写入 */
    memset(buf, 0x5A, mtd->writesize);
    if (mtd_write(mtd, 0x480000, mtd->writesize, &retlen, buf))
        goto err;

    /* 4. 读回验证（ret>0仅表示ECC纠正了位翻转） */
    ret = mtd_read(mtd, 0x480000, mtd->writesize, &retlen, buf);
    kfree(buf);
    return (ret < 0) ? ret : 0;
}
```

### 五、MTD分区 vs 块设备分区

| 对比维度 | MTD分区 | 块设备分区（硬盘/eMMC） |
|:---|:---|:---|
| **分区信息存储** | `mtdparts`内核参数或DTB | MBR/GPT存储在设备首扇区 |
| **分区工具** | 编译/启动时静态定义 | `fdisk`/`parted`运行时动态修改 |
| **坏块感知** | 分区边界**自动跳过坏块**，逻辑→物理可能不连续 | 逻辑与物理严格线性映射 |
| **对齐要求** | 必须与`erasesize`对齐 | 通常与扇区（512B/4KB）对齐 |
| **可见设备** | `/dev/mtdN`, `/dev/mtdblockN` | `/dev/sdaN`, `/dev/mmcblk0pN` |
| **底层暴露** | 直接暴露擦除、OOB、坏块特性 | 完全屏蔽，表现为线性块数组 |

MTD分区的"静态"特质根植于Flash的物理本质：坏块导致逻辑分区在物理上可能碎片化，擦除块的原子性要求分区必须严格对齐——这不是对完美介质的逻辑切割，而是对有缺陷、有磨损的物理器件的谨慎管理。
