### 11.4.3 udev规则与实战

**知识点147 [I]**

本节导读：前两节你已经理解了udev的事件驱动机制和uevent的传递路径。本节我们进入真正的"动手环节"——学习如何编写udev规则文件，让udev按照你的意愿自动管理设备节点。学完本节，你将能够为自己的硬件编写定制规则，并掌握一套完整的调试方法。

---

#### 规则文件放在哪

udev的规则文件存放在 `/etc/udev/rules.d/` 目录下，文件名以 `.rules` 结尾。关键在于，udev **按文件名的字典序（ASCII排序）加载规则**，所以你会看到文件名通常带数字前缀：

```
/etc/udev/rules.d/
├── 50-udev-default.rules      # 系统默认规则
├── 60-persistent-storage.rules # 磁盘相关
├── 70-persistent-net.rules     # 网卡命名
└── 99-custom.rules             # 你的自定义规则放这里
```

💡 **提示**：数字越小越早加载，后加载的规则可以覆盖前面的。建议自定义规则用 `99-` 开头，确保它在系统默认规则之后执行，避免被意外覆盖。

⚠️ **陷阱**：不要直接修改 `/lib/udev/rules.d/` 下的系统规则文件——升级时会被覆盖。你的自定义规则一律放在 `/etc/udev/rules.d/`。

---

#### 规则语法：匹配 + 动作

一条udev规则的本质很简单：**如果设备满足条件A，就执行动作B**。语法结构长这样：

```
<匹配键>=="<值>", <匹配键>=="<值>", <动作键>="<值>"
```

注意匹配条件用 `==`（精确匹配），动作赋值用 `=` 或 `+=`（追加）。

**常用匹配键一览：**

| 匹配键 | 说明 | 示例 |
|---|---|---|
| `SUBSYSTEM` | 子系统名称 | `"leds"`, `"usb"`, `"block"` |
| `KERNEL` | 内核中的设备名 | `"sda1"`, `"ttyUSB0"` |
| `ATTR{属性}` | sysfs中的设备属性 | `ATTR{idVendor}=="1234"` |
| `ENV{环境变量}` | udev内部环境变量 | `ENV{DEVTYPE}=="disk"` |

**常用动作键一览：**

| 动作键 | 说明 | 示例 |
|---|---|---|
| `SYMLINK` | 创建符号链接（`+=`追加） | `SYMLINK+="myled"` |
| `MODE` | 设置设备节点权限 | `MODE="0666"` |
| `GROUP` | 设置设备节点属组 | `GROUP="dialout"` |
| `RUN` | 触发时执行脚本 | `RUN+="/usr/bin/myscript.sh"` |
| `OWNER` | 设置设备节点属主 | `OWNER="root"` |

---

#### 实战示例

**示例1：给LED设备创建人性化的符号链接**

假设你有一个LED设备在 `/sys/class/leds/status_led`，内核名叫 `status_led`。你想给它创建一个更好记的链接：

```udev
SUBSYSTEM=="leds", KERNEL=="status_led", SYMLINK+="leds/board_status"
```

规则生效后，你会看到 `/dev/leds/board_status` 指向实际的设备节点。

**示例2：给特定USB设备开放权限**

嵌入式开发中经常遇到这个问题：普通用户无法访问USB串口。查一下设备的vendor/product ID，一条规则搞定：

```udev
SUBSYSTEM=="usb", ATTR{idVendor}=="1234", ATTR{idProduct}=="5678", MODE="0666"
```

🔴 **危险**：`MODE="0666"` 会让所有用户读写该设备。在正式发布的产品中，建议用 `GROUP="xxx"` 配合用户组来控制权限，而不是全局开放。

**示例3：设备插入时执行自定义脚本**

```udev
SUBSYSTEM=="usb", ATTR{idVendor}=="abcd", RUN+="/usr/local/bin/usb-plug.sh"
```

这个脚本在设备插入时由udev触发执行。注意脚本要尽量轻量——udev是事件驱动的，阻塞太久会影响后续事件处理。

#### 调试udev规则的三大法宝

写规则最怕"改了不生效"，这三个命令帮你定位问题：

**1. udevadm monitor —— 实时监控uevent**

```bash
$ udevadm monitor --kernel --udev
monitor will print the received events for:
KERNEL - the kernel uevent
UDEV  - the udev event (after rules processed)

KERNEL[1234.567] add    /devices/platform/leds/leds/status_led (leds)
UDEV  [1234.589] add    /devices/platform/leds/leds/status_led (leds)
```

`--kernel` 看原始内核事件，`--udev` 看规则处理后的结果。两边对照，能快速判断规则是否命中。

**2. udevadm info —— 查看设备的"底细"**

```bash
$ udevadm info /sys/class/leds/myled
P: /devices/platform/leds/leds/myled
L: 0
E: SUBSYSTEM=leds
E: DEVPATH=/devices/platform/leds/leds/myled
E: PRODUCT=1/1/1/0
```

重点是 `P:`（sysfs路径）和 `E:`（环境变量）行，这些是你写规则的"素材库"。

**3. udevadm test —— 模拟规则执行**

这是最实用的调试手段——不需要真的拔插设备，直接模拟udev对某个设备执行规则：

```bash
$ udevadm test /sys/class/leds/myled 2>&1 | tail -20
```

输出会逐条显示哪些规则匹配、哪些不匹配、最终应用了哪些动作。如果规则没生效，从这里找原因是最高效的。

💡 **提示**：修改规则文件后，执行 `udevadm control --reload-rules` 重新加载，然后配合 `udevadm test` 验证。如果是USB设备，可以物理拔插一次，或者用 `echo 0 > /sys/bus/usb/devices/.../authorized` 来软复位触发事件。

#### 本节总结

| 知识点 | 内容速查 |
|---|---|
| 规则存放位置 | `/etc/udev/rules.d/*.rules`，按文件名排序 |
| 匹配语法 | `<键>=="<值>"`，如 `SUBSYSTEM=="leds"` |
| 动作语法 | `<键>="<值>"` 或 `<键>+="<值>"` |

| 调试三板斧 | `udevadm monitor`, `udevadm info`, `udevadm test` |
| 重载规则 | `udevadm control --reload-rules` |
