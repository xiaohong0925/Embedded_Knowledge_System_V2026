### 11.5.2 dtbo编译与加载

**本节导读**

上一节我们了解了设备树Overlay的设计思想，这一节就来动手——把`.dts`文件变成可加载的`.dtbo`，再通过ConfigFS接口动态应用到正在运行的系统中。学完后，你将掌握从源码编译到运行时加载的完整链路，并能在不重启的情况下为系统添加新设备节点。

---

#### 编译dtbo：给dtc一个特殊的信号

Overlay源文件（`.dts`）编译成二进制（`.dtbo`）的过程与普通设备树几乎一样，但有一个关键区别：**Overlay中引用了目标基树里已经存在的节点**，这些引用在编译时是"悬空的"。

这就用到了`-@`（或`--symbols`）参数：

```bash
# 编译设备树Overlay
dtc -@ -O dtb -o myoverlay.dtbo myoverlay.dts
```

`-@`告诉dtc：**允许存在未解析的符号引用**，同时生成`__symbols__`节点，记录所有带标签的节点地址。内核加载时靠这个符号表来定位目标节点。

一个典型的Overlay源文件`myoverlay.dts`长这样：

```dts
/dts-v1/;
/plugin/;

/ {
    compatible = "brcm,bcm2711";

    fragment@0 {
        target-path = "/soc/gpio";
        __overlay__ {
            my_led {
                compatible = "gpio-leds";
                label = "status";
                gpios = <&gpio 25 0>;
            };
        };
    };
};
```

💡 **提示**：如果没有`-@`，dtc遇到`&gpio`这类外部引用会直接报错。`plugin;`关键字也是Overlay特有的，它告知dtc这是一个可插拔片段。

---

#### 内核必须开启的两扇大门

要让Overlay机制跑起来，内核编译选项里必须打开这两个开关：

| 配置项 | 作用 |
|--------|------|
| `CONFIG_OF_OVERLAY=y` | 开启设备树Overlay运行时合并能力 |
| `CONFIG_CONFIGFS=y` | 启用ConfigFS，暴露`/sys/kernel/config/device-tree/overlays/`接口 |

说白了，缺了`CONFIG_OF_OVERLAY`，内核根本不认识dtbo；缺了`CONFIG_CONFIGFS`，你连门把手都摸不着。

⚠️ **陷阱**：这两个配置通常是嵌入式发行版（Buildroot、Yocto）默认开启的，但如果你在裁剪内核时手一抖关掉了，加载时会出现`write error: No such file or directory`——不是文件不存在，而是ConfigFS目录根本没挂载出来。

---

#### ConfigFS接口：动态加载的入口

ConfigFS是一种基于内存的文件系统，设备树Overlay把它变成了一个**活接口**：

```
/sys/kernel/config/device-tree/overlays/
```

加载Overlay只需三步：

```bash
# 1. 挂载ConfigFS（如果尚未挂载）
mount -t configfs none /sys/kernel/config

# 2. 创建一个新的overlay目录（相当于"申请插槽"）
mkdir /sys/kernel/config/device-tree/overlays/myoverlay

# 3. 把编译好的dtbo写入dtbo文件
cat myoverlay.dtbo > /sys/kernel/config/device-tree/overlays/myoverlay/dtbo
```

写入`dtbo`文件的那一瞬间，内核的`of_overlay_apply()`被触发，流程如下：

```
用户写入dtbo
    |
    v
内核调用 of_fdt_unflatten_tree() 解析dtbo
    |
    v
of_overlay_apply() 将fragment中的__overlay__节点
    合并到target-path/target指定的目标节点
    |
    v
合并后的设备树触发 of_platform_populate()
    |
    v
匹配的device_driver被唤醒 → 设备创建 → uevent发送
```

你看，整个过程一气呵成。内核帮我们做了最脏最累的活：节点合并、地址解析、设备实例化。

💡 **提示**：`target-path`也可以用`target = <&label>`的形式引用带标签的节点，更不容易写错路径。

---

#### 验证与卸载

加载成功后，最直接的验证方式是看新设备是否出现：

```bash
# 检查设备是否被创建
ls /sys/bus/platform/devices/ | grep my_led

# 查看内核日志确认合并过程
dmesg | tail -20
```

卸载同样简单——删除目录即可：

```bash
rmdir /sys/kernel/config/device-tree/overlays/myoverlay
```

内核会自动撤销设备树的修改，并销毁对应的平台设备。但要注意：**如果设备当前正在被驱动占用（比如LED已打开、GPIO正在被操作），卸载可能会失败或触发资源冲突。**

🔴 **危险**：在设备处于活跃状态时强行卸载Overlay，可能导致内核oops或资源泄漏。确保驱动已释放所有资源（如`rmmod`驱动）后再执行卸载。

---

#### 一个完整的实战流程

把上面的步骤串起来，一个典型的开发调试流程如下：

```bash
# 1. 编辑Overlay源文件
vim myoverlay.dts

# 2. 编译
dtc -@ -O dtb -o myoverlay.dtbo myoverlay.dts

# 3. 拷贝到目标板（如果是交叉编译环境）
scp myoverlay.dtbo root@target:/tmp/

# 4. 目标板上加载
mkdir /sys/kernel/config/device-tree/overlays/test
cat /tmp/myoverlay.dtbo \
    > /sys/kernel/config/device-tree/overlays/test/dtbo

# 5. 验证
dmesg | grep "my_led"
ls /sys/class/leds/

# 6. 测试完毕，卸载
rmdir /sys/kernel/config/device-tree/overlays/test
```

这个流程在调试外设时特别高效——改DTS、编译、加载、验证，全程不用重启板子。

⚠️ **陷阱**：Overlay加载是**有顺序依赖的**。如果你的Overlay依赖某个已加载的Overlay创建的节点，必须先加载前者。这在模块化设计中需要特别注意。

**本节总结**

| 知识点 | 要点 |
|--------|------|
| 编译命令 | `dtc -@ -O dtb -o xxx.dtbo xxx.dts`，`-@`允许未解析引用 |
| 内核配置 | `CONFIG_OF_OVERLAY=y` + `CONFIG_CONFIGFS=y` |
| ConfigFS挂载点 | `/sys/kernel/config/device-tree/overlays/` |
| 加载三步 | `mkdir`创建目录 → `cat`写入dtbo → 内核自动合并 |
| 卸载方法 | `rmdir`对应overlay目录 |
| 加载触发链 | dtbo写入 → of_overlay_apply() → 设备树合并 → 设备创建 → uevent |
| 安全注意 | 设备活跃时卸载可能引发崩溃，确保驱动已释放资源 |
