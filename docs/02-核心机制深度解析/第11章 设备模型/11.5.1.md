## 11.5.1 为什么需要overlay

### 本节导读

到目前为止，你学的设备树都是"全量编译"模式——把整块板子的硬件描述写进一个.dts，编译成.dtb，然后在内核启动时一次性加载。这种模型对于大多数产品来说是够用的，毕竟硬件定了，设备树也就定了。

但有些场景下，硬件不是一成不变的。FPGA的比特流可以动态切换，扩展板可以热插拔，某些外设需要运行时按需启用。全量dtb在这些场景下就显得过于笨重了。**本节我们一起理解设备树overlay（DT Overlay）出现的动机——它本质上是在运行时"打补丁"修改设备树，而不是推倒重来。** 学完这一节，你会清楚overlay解决的三个核心场景，以及为什么它比重新编译dtb更优雅。

---

### 场景1：FPGA动态重配置

假设你在做一个Zynq平台的项目，FPGA fabric中有一块可重配置的RP（Reconfigurable Partition）。你加载bitstream A时需要一个SPI控制器；换成bitstream B时又需要一个I2C控制器。两个bitstream对应的硬件截然不同。

用全量dtb怎么搞？你需要维护两份完整的设备树，每次切换bitstream都要重启、加载新的dtb。这不仅流程长，而且两份dtb中大部分内容是重复的——CPU、DDR、固定外设的节点完全一样。

💡 **提示**：overlay的精髓就在这儿——把"变化的部分"抽出来，以.dts片段的形式单独描述。切换FPGA配置时，只需要加载对应的overlay，内核就会自动把这些节点合并到现有设备树中。公共部分不动，只动差异部分。

说白了，overlay让设备树也能玩"模块化"。

### 场景2：扩展板热插拔

想象你用的是BeagleBone Black或者树莓派这样的平台，主板上有一个 cape / HAT 接口。用户插入一块扩展板（比如一个带ADC的cape），内核需要知道这块板上有一个新的ADC设备。

问题是：内核启动时这块板子还不一定插着。你把ADC节点写进主dtb里，那不插cape的时候内核也会去probe这个设备，结果自然是probe失败，log里多一行error。你不写吧，插了cape又不识别。

overlay把这个难题拆解得很漂亮：**主dtb只描述底板固定的硬件；扩展板的设备信息以overlay形式存放在文件系统中**（通常放在/lib/firmware，或者由扩展板自带的EEPROM提供标识）。当检测到cape插入时，再通过用户空间工具把对应的overlay加载进来，ADC设备就这样"凭空出现"了。

⚠️ **陷阱**：overlay不是魔术——它需要平台硬件层面的支持来触发加载。比如BeagleBone的cape接口有EEPROM和ID引脚，U-Boot或内核可以通过这些信号识别cape型号，再决定加载哪个overlay。没有检测机制的话，你手动echo overlay名称到configfs也是可行的，只是不够"自动"。

### 场景3：运行时启用/禁用外设

你可能在不少dts里见过这样的写法：

```dts
&uart2 {
    status = "disabled";
};
```

然后在某个产品 variants 的dts里再改成 `"okay"`。这种做法的问题在于它是**静态的**——编译时就定了，运行时改不了。

overlay让你可以在运行时把一个status="okay"的节点patch进设备树，不需要重新编译任何东西。配合configfs接口，甚至可以做到shell里一条命令启用外设：

```bash
echo my_overlay > /sys/kernel/config/device-tree/overlays/my_overlay/path
```

这对调试和灵活性来说意义重大。想想看，你不用重编dtb、不用重启，现场就能开关一个外设验证功能，调试效率差了一个数量级。

### 对比：overlay vs 重新编译dtb

| 维度 | 重新编译dtb | 使用overlay |
|------|-----------|------------|
| 是否需要重启 | ✅ 必须重启生效 | ❌ 运行时动态加载 |
| 维护成本 | 多份完整dtb，重复内容多 | 公共部分一份，差异部分独立维护 |
| 灵活性 | 编译时决定，运行时不可变 | 按需加载，可逆（可卸载） |
| 适用场景 | 硬件固定的产品 | FPGA重配置、扩展板、动态调试用 |
| 版本管理复杂度 | 高（多个dtb文件） | 中（dtb + 多个dtbo） |

表格一目了然：overlay不是来取代全量dtb的——**底板固定的硬件仍然应该用主dtb描述**。overlay的战场在"变化"和"动态"。

🔴 **危险**：不要滥用overlay把所有东西都做成动态的。主dtb仍然应该是你硬件描述的"single source of truth"。overlay太多、层叠关系太复杂，会让系统行为变得难以追踪。建议把overlay的使用范围严格限制在真正会变化的硬件上。
