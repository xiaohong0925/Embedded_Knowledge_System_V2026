# 11.5.3 overlay的应用与限制

> 所属章节：第11章 设备树深度实践 > 11.5 动态加载：Device Tree Overlay
> 难度：[E] | 预计阅读时间：12分钟

## 本节导读
本节把overlay从"纸面原理"拉到"真实战场"——你会看到内核里一条完整的应用链条：overlay怎么和设备树合并、怎么变成真正的设备节点、又怎么触发驱动probe。同时我们也会直面overlay的硬伤：它并不能随心所欲地改设备树，有些操作就是做不到。学完后，你能独立完成overlay的加载流程，也能在遇到"怎么删不掉节点"时心里有数。

## 知识点164：overlay的应用全流程——从DTOB到设备上线 [E] ~800字

### 把overlay应用比作"给房子加装阳光房"

想象你已经住进一套精装房（基板设备树），现在想在阳台上搭一间阳光房（overlay）。整个流程是：先画好设计图（DTOB二进制），交给物业审批（内核dtc解析合并），然后施工队进场搭建（创建新设备节点），最后通水通电（触发uevent、驱动probe）。少了任何一步，阳光房都住不了人。

overlay的应用过程在内核里就是这么一条**四步链**：

```
DTOB加载 → 设备树合并 → 新设备节点创建 → uevent上报 → 驱动probe
```

### 第一步：DTOB进内核

通过`configfs`把DTOB写进去（上节讲的`dtbo`文件写入），或者U-Boot在启动阶段通过`fdt apply`合并。无论哪种方式，最终都是让内核的Device Tree子系统拿到overlay的数据。

### 第二步：设备树合并——"补丁打进去"

内核里的`of_overlay_apply()`负责把overlay里的节点和属性**叠加**到基板设备树上。这个过程有点像`git apply`：

- overlay里的新节点 → 在基板树对应位置**新增**
- overlay里的属性 → **覆盖**基板树同名属性的值
- `__overlay__`标签下的内容 → 展开后挂到基板树的`target`节点下

### 第三步：创建新设备节点——"把纸面节点变活"

合并后的设备树只是内存里的数据结构，还没变成内核里真正的`struct device`。这一步由`of_platform_populate()`或`of_platform_device_create()`出手，把新出现的节点转换成platform_device（或其他总线设备）。

### 第四步：uevent + probe——"通知驱动来认领"

新设备创建后，内核通过`kobject_uevent()`发送`add`事件，用户空间的udev/mdev收到后做节点权限处理。同时，匹配该设备的驱动收到probe通知，开始初始化硬件。如果一切顺利，`dmesg`里会看到类似这样的流水账：

```
[  12.345678] OF: overlay: applying overlay '__overlay__'
[  12.346012] my_driver: probe of my_device@0 returned 1
```

### overlay的限制——它能做和不能做的事

很多新手第一次用overlay都会栽跟头："我怎么删不掉基板自带的节点？""`#address-cells`改不了？"这不是你操作错了，是overlay**设计上就不支持**。

| 操作类型 | 能做到？ | 说明 |
|---------|---------|------|
| 添加新节点 | ✅ 可以 | overlay的核心功能，在任意位置挂新节点 |
| 修改已有属性值 | ✅ 可以 | 用新值覆盖旧值，比如改`status = "okay"` |
| 新增属性 | ✅ 可以 | 给已有节点增加原来没有的属性 |
| 删除已有节点 | ❌ 不行 | 只能"盖"不能"拆"，overlay没有remove节点机制 |
| 删除已有属性 | ❌ 不行 | 没有`__delete__`之外的通用手段 |
| 修改`#address-cells` | ❌ 不行 | 属于核心结构属性，内核解析时已经固化 |
| 修改`#size-cells` | ❌ 不行 | 同上，改变会破坏子节点reg的解析逻辑 |
| 修改`interrupt-parent` | ❌ 不行 | 中断树已经在早期建立，动态改会出乱子 |
| 改根节点的`compatible` | ❌ 不行 | 机器类型一旦确定不能变 |

⚠️ **陷阱**：当你发现overlay没生效时，先别急着怀疑语法。用`dtc -I fs`导出合并后的设备树（下节知识点165会讲），确认你的属性到底写进去没有。很多时候是`target`路径写错了一个字母，导致overlay挂到了错误的分支上。

🔴 **危险**：overlay合并失败**不会**导致内核panic，但可能会留下一个半残的设备树状态——部分节点加了、部分没加。这时候如果驱动probe到了半成品节点，可能会oops。建议每次只加一个overlay，确认没问题再叠加下一个。

## 知识点165：overlay调试——肉眼确认合并结果 [E] ~600字

### 调试overlay的两种"实锤"手段

overlay加载后看不见摸不着，怎么确认它真的生效了？两个最直接的方法：**看状态**和**导树**。

### 方法一：通过configfs看overlay状态

如果你是通过`/sys/kernel/config/device-tree/overlays/`加载的DTOB，内核会在里面给每个overlay创建一个目录，里面有个`status`文件告诉你当前状态：

```bash
# 代码1：查看overlay加载状态
$ ls /sys/kernel/config/device-tree/overlays/
my_overlay

$ cat /sys/kernel/config/device-tree/overlays/my_overlay/status
applied
```

`status`常见的几种返回值：

| 状态值 | 含义 |
|-------|------|
| `applied` | 已成功合并到设备树 |
| `deferred` | 依赖条件未满足，等待后续重试 |
| （目录不存在） | DTOB写入失败，根本没挂上去 |

如果状态不是`applied`，先检查`dmesg`里有没有`OF: overlay:`开头的报错。常见原因包括：`target`节点找不到、phandle引用越界、overlay内部语法错误。

### 方法二：用dtc导出合并后的完整设备树

这是最"实锤"的调试方法——直接把内核内存里的设备树dump出来，肉眼对比overlay是不是真的写上去了：

```bash
# 代码2：导出当前运行中的完整设备树
$ dtc -I fs /sys/firmware/devicetree/base -o merged.dts
```

这个命令把`/sys/firmware/devicetree/base`（内核暴露的设备树sysfs接口）反向转成DTS源文件。打开`merged.dts`，你可以直接搜你overlay里加的节点名或属性名，确认：

1. 节点是不是挂在了正确的父节点下面
2. 属性值是不是你想要的那个
3. 有没有被其他overlay或基板配置意外覆盖

💡 **提示**：如果系统没有`dtc`命令，可以用`apt install device-tree-compiler`安装。嵌入式板子上空间有限，建议在宿主机上通过NFS或scp把导出的文件传出来分析。

有时候overlay明明状态是`applied`，但设备就是没反应。这时候要分两步排查：

1. **先看设备节点出没出现**：`ls /sys/bus/platform/devices/`（或对应总线目录）看看目标设备名在不在列表里。如果节点都没有，说明`of_platform_populate()`那步走岔了，检查`compatible`属性拼写和`status = "okay"`有没有漏。
2. **再看驱动匹没匹配上**：`ls /sys/bus/platform/drivers/<你的驱动名>/`看看有没有bind到这个设备。如果设备节点有了但驱动没probe，问题在驱动端——要么是`of_match_table`没对上，要么是probe函数里某个资源申请失败返回了`-EPROBE_DEFER`。

⚠️ **陷阱**：`dtc -I fs`导出的是**合并后**的设备树，你已经分不清哪个节点来自基板、哪个来自overlay。如果需要做这种区分，建议在overlay里给新增节点加一个自定义属性做标记，比如`overlay-source = "my_overlay"`，导出后一搜就知道哪些是你加的。

## 本节总结

| 概念 | 核心要点 | 自查操作 |
|------|---------|---------|
| 应用四步链 | DTOB加载 → 设备树合并 → 创建设备 → uevent/probe | `dmesg \| grep OF` 查看流水日志 |
| 能做的 | 添加节点、修改/新增属性 | 用`dtc -I fs`导出确认 |
| 不能做的 | 删除节点/属性、改`#address-cells`等核心属性 | 不要浪费时间尝试，换方案 |
| 状态查看 | `/sys/kernel/config/device-tree/overlays/*/status` | `cat status`看`applied`还是`deferred` |
| 实锤调试 | `dtc -I fs /sys/firmware/devicetree/base`导出完整设备树 | 反向转成DTS后grep目标节点 |
