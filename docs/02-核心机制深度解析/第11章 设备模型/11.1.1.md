# 11.1.1 为什么需要设备模型

**知识点130 [I]**

## 本节导读

设备模型（Device Model）是Linux内核中一套统一的设备管理机制。本节带你回顾早期内核在没有统一设备模型时的困境，理解为什么2.6版本要花大力气引入这套框架。学完后，你会明白kobject、device、bus、driver这些核心抽象到底是为了解决什么问题而诞生的——**知其所以然，后续学起来才顺**。

---

想象一下，你写一个LED驱动。在2.6以前的内核里，你可能直接这样干：

```c
/* 早期驱动的典型写法 */
#define LED_PORT 0x378

static int __init led_init(void)
{
    if (!request_region(LED_PORT, 1, "led"))
        return -EBUSY;
    outb(0xff, LED_PORT);
    return 0;
}
```

硬编码I/O地址、自己申请资源、自己释放。驱动跑起来没问题——**直到系统里有了第二个驱动**。

## 各自为政的混乱年代

早期内核的驱动开发，说白了就是一种"荒野求生"模式。每个驱动作者都按照自己的理解来管理硬件资源：

- 这个驱动直接写死I/O端口 `0x378`
- 那个驱动硬编码中断号 `IRQ 7`
- 内存映射地址各自分配，互不通报

没有统一的注册机制，内核根本不知道系统里有哪些设备、哪些驱动。我们来看看这种状况会带来什么实际麻烦。

**资源冲突是最直接的噩梦。** 两个驱动同时申请同一个I/O端口，谁会失败取决于加载顺序——后加载的那个可能崩溃，也可能静默地覆盖前一个驱动的数据。调试这种问题，说实话，能让人掉层皮。

**热插拔？想都别想。** USB设备插上去，内核无从得知"有新设备来了"。没有统一的设备树，没有通知机制，插拔事件根本无处可报到。早期的USB子系统为了绕开这个限制，搞了不少临时补丁，但治标不治本。

**用户空间更是一抹黑。** 你想看看系统里有哪些设备？抱歉，没有 sysfs。`/proc` 目录下的信息零零散散，每个子系统自己决定暴露什么、怎么命名。一个PCI设备的寄存器信息可能藏在一堆二进制数据里，另一个设备的参数又是在另一个完全不同的文件格式里。写个枚举工具？你得给每个子系统单独写解析逻辑。

```c
/* 这是2.4时代常见的"资源申请+设备注册"混合写法 */
static int __init my_driver_init(void)
{
    request_region(io_base, io_len, "mydrv");  /* 自己管I/O */
    request_irq(irq, handler, 0, "mydrv", NULL); /* 自己管中断 */
    register_chrdev(major, "mydrv", &fops);     /* 自己管字符设备 */
    /* 设备信息？散在各处，没有统一视图 */
}
```

说白了，**驱动和驱动之间没有"共同语言"**，驱动和设备之间也没有标准的关系描述。

## 2.6的统一答案

2003年发布的Linux 2.6内核做了一件事：**把所有设备、驱动、总线都纳入一个统一的层次结构中**。

```mermaid
graph TD
    subgraph 早期内核 "各自为政"
        A[LED驱动] -->|硬编码| A1[I/O 0x378]
        B[串口驱动] -->|硬编码| B1[IRQ 4]
        C[网卡驱动] -->|硬编码| C1[MEM 0xD0000]
        A -.->|互不知晓| B
        B -.->|互不知晓| C
    end

    subgraph 2.6+ 统一设备模型
        D[总线 Bus] --> E[设备 Device]
        D --> F[驱动 Driver]
        E -->|绑定| F
        D --> G[kobject<br/>统一抽象]
        E --> G
        F --> G
        H[sysfs] -->|暴露| D
        H -->|暴露| E
        H -->|暴露| F
    end
```

这套模型的核心思路不复杂：**用kobject提供统一的基类，用device/driver/bus描述关系，用sysfs向用户空间输出标准化的设备视图。**

| 问题 | 2.4及之前 | 2.6+ 统一设备模型 |
|------|----------|----------------|
| 资源管理 | 各驱动自行申请，无协调 | 统一注册，自动检测冲突 |
| 设备发现 | 内核无全局设备列表 | 设备树层次结构，枚举一目了然 |
| 驱动匹配 | 手动指定，无标准流程 | bus总线自动匹配device与driver |
| 热插拔 | 不支持，或靠临时补丁 | 事件通知链，统一处理 |
| 用户空间可见性 | /proc信息零散 | sysfs标准化目录结构 |
| 电源管理 | 各驱动各搞各的 | 统一PM回调框架 |

💡 **提示**：如果你有机会翻看2.4内核的源码，你会发现 `request_region()` 和 `request_irq()` 虽然很早就存在，但它们之间没有任何关联机制。直到设备模型引入后，这些资源申请才被整合进统一的设备生命周期管理中。

⚠️ **陷阱**：不要误以为设备模型只是"为了好看"的抽象。很多后来的功能——比如udev的设备节点自动创建、`/sys` 文件系统、`firmware` 加载机制——都建立在设备模型的基础上。跳过这一章去学驱动框架，等于跳过了地基。

## 本节总结

| 概念 | 含义 |
|------|------|
| 设备模型的核心目标 | 统一管理设备、驱动、总线，解决资源冲突和可见性问题 |
| 早期内核的痛点 | 硬编码资源、驱动互不通信、无热插拔、用户空间不可见 |
| 统一设备模型的关键组件 | kobject（基类）、device（设备）、driver（驱动）、bus（总线） |
| sysfs的作用 | 将设备层次结构以标准化目录暴露给用户空间 |
| 设备模型带来的衍生能力 | 热插拔、udev、统一电源管理、设备枚举 |
