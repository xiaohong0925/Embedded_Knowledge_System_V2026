## 11.1.2 总线-设备-驱动三角关系

**知识点131 [I][M]**

### 本节导读

上一节我们把设备模型比作"家族"，这一节聚焦家族里最重要的铁三角——总线、设备、驱动。搞懂这三者的关系，你就理解了Linux内核中"硬件怎么找到软件、软件怎么接管硬件"的核心机制。学完本节，你能画出bus_type、device、device_driver的协作关系图，明白kobject的层次结构如何在`/sys`目录下展开，也能看懂内核中`driver_register()`和`device_register()`的调用链。

---

### 婚介所的比喻

假设你有一块I2C接口的温度传感器芯片。内核里已经写好了I2C总线的驱动框架（`i2c_bus_type`），也注册了温度传感器的驱动程序（`lm75_driver`），但硬件和软件谁也不认识谁——谁该来牵这根线？

答案是：**总线（bus_type）就是婚介所**。

```
婚介所（bus_type）：手里有适龄男青年的名单，也有单身女青年的档案
男方（device）：硬件芯片——我有这些资源（寄存器地址、中断号）
女方（device_driver）：软件实现——我能处理这类硬件
```

男方来登记 → 婚介所翻看女方档案 → match成功 → 通知女方去男方家probe → 配对完成。

说白了，这个流程就是Linux设备模型的核心运作方式。

### 三个核心结构体

来看看源码里的定义（kernel 5.x）：

```c
/* 婚介所：定义匹配规则和操作流程 */
struct bus_type {
    const char      *name;
    int (*match)(struct device *dev, struct device_driver *drv);
    int (*probe)(struct device *dev);
    int (*remove)(struct device *dev);
    struct subsys_private *p;  /* 内含 kset，用于 sysfs */
    /* ... */
};

/* 男方：描述一个具体的硬件实例 */
struct device {
    struct kobject          kobj;
    struct device           *parent;
    struct bus_type         *bus;
    struct device_driver    *driver;
    const char              *init_name;
    struct resource         *resource;  /* 寄存器、中断等资源 */
    void                    *platform_data;
    /* ... */
};

/* 女方：描述一类设备的软件支持 */
struct device_driver {
    const char              *name;
    struct bus_type         *bus;
    struct module           *owner;
    const struct of_device_id   *of_match_table;
    int (*probe)(struct device *dev);
    int (*remove)(struct device *dev);
    struct kobject          kobj;
    /* ... */
};
```

注意三者的共同特征：`bus_type`字段像一根红线，把三方串在一起。

### 匹配过程：match → probe

当设备或驱动注册到总线时，婚介所就开始工作了：

```
device_register(dev)          driver_register(drv)
       |                             |
       v                             v
   bus_add_device()            bus_add_driver()
       |                             |
       +----------+   +--------------+
                  |   |
                  v   v
           bus_probe_device()
                  |
                  v
       bus_for_each_drv()  ← 遍历该总线上所有驱动
                  |
                  v
          __device_attach()
                  |
                  v
           drv->bus->match(dev, drv)  ← 匹配！
                  |
                  v (match返回1)
           driver_probe_device()
                  |
                  v
           really_probe()
                  |
                  v
           drv->probe(dev)  ← 驱动接管设备
```

💡 **提示**：`match()`函数是总线类型决定的。platform总线比名字，`i2c`总线比设备地址，`pci`总线比vendor/device ID。不同婚介所，匹配标准不一样——很合理。

⚠️ **陷阱**：如果`match()`和`probe()`都返回0，看起来一切正常，但实际上match返回0表示"不匹配"，probe返回0表示"成功"。语义相反，查bug时容易搞混。

### kobject 到 sysfs 的映射

每个device、device_driver、bus_type内部都嵌入了kobject。kobject就像每个对象的"身份证"，通过`parent`指针串成一棵层次树，直接映射到`/sys`的目录结构：

```
/sys/bus/           ← bus_register() 创建
├── platform/       ← bus_type->name
│   ├── devices/    ← 该总线上所有device的软链接
│   │   └── my_led@0 → /sys/devices/platform/my_led@0
│   └── drivers/    ← 该总线上所有driver的目录
│       └── my_driver/
│           └── bind / unbind  ← 手动绑定/解绑接口
│
├── i2c/
│   ├── devices/
│   └── drivers/
│
└── pci/
    ├── devices/
    └── drivers/

/sys/devices/       ← 硬件拓扑的真实树
├── platform/
│   └── my_led@0/   ← device_register() 创建
│       ├── driver -> /sys/bus/platform/drivers/my_led_drv
│       └── ...
```

🔴 **危险**：`/sys/bus/XXX/drivers/YYY/bind`文件允许手动把一个设备和驱动强制绑定。如果在运行中echo了错误的设备名，可能导致正在工作的驱动被拆散——生产环境别随便玩这个。

💡 **提示**：`udev`守护进程监控`/sys`的变化，当新的device目录出现时，udev根据规则创建`/dev`下的节点。所以kobject → sysfs → udev → `/dev`这条链路，就是设备从内核暴露到用户空间的完整路径。

### 一个真实的视角

三角关系看似抽象，实际在开发中无处不在。你写dts设备树节点（描述device），编译进内核后`of_platform_populate()`会自动创建platform device；你写的驱动里填充`platform_driver`结构（描述device_driver），`module_platform_driver()`宏帮你注册；platform bus_type早就由内核初始化好了。三方就位，`match`和`probe`就会自动发生——这背后是 hundreds 行的内核代码在协作，但你只需要理解这个三角关系，就能理清调用链。

---

### 本节总结

| 概念 | 比喻 | 核心职责 | 对应sysfs路径 |
|------|------|----------|---------------|
| `bus_type` | 婚介所 | 定义`match()`规则，协调probe/remove | `/sys/bus/<name>/` |
| `device` | 男方（硬件） | 描述具体硬件实例的资源、名称、拓扑关系 | `/sys/devices/.../` |
| `device_driver` | 女方（驱动） | 提供`probe()/remove()`，填充`of_match_table` | `/sys/bus/.../drivers/<name>/` |
| `kobject` | 身份证 | 提供引用计数和层次结构，映射为sysfs目录 | 每个对象对应一个目录 |
| `match()` | 相亲条件 | 总线特定的匹配逻辑（名字/地址/ID等） | 无，内核函数 |
| `probe()` | 上门安装 | 驱动初始化设备，申请资源 | 通过`bind`触发 |
