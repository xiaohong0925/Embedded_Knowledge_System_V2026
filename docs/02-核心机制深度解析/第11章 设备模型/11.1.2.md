# 11.1.2 总线-设备-驱动三角关系

> 所属章节：第11章 Linux设备模型深度解析 > 11.1 设备模型核心架构
> 难度：[I→M] | 预计阅读时间：15分钟

## 本节导读

你之前写过字符设备驱动，也知道驱动最终是要操作硬件的。但你有没有好奇过——内核怎么知道哪个驱动对应哪个设备？驱动又是怎么"找到"硬件资源的？答案就在本节要说的"三角关系"里。学完这节，你能画清楚bus、device、driver三者的关系图，明白match和probe到底谁调谁，再也不怕看内核启动log里那一堆"probe""bind"的打印了。

## 知识点131：bus、device、driver——内核里的"婚恋三角" [I→M] ~1200字

### 先把三角关系比作"婚介所"

 Linux的设备模型里，总线（bus_type）、设备（device）、驱动（device_driver）这三者构成了一个铁三角。别被那些struct定义吓到，本质上这就跟现实生活中的婚恋市场一个样：

- **bus_type（总线）** = **婚介所**。它手里掌握两套名单：一份是"待嫁女"（挂在总线上的驱动），一份是"待娶男"（挂在总线上的设备）。婚介所还定了一套规则叫match——什么样的男方和女方可以配对。
- **device（设备）** = **男方**。它手里有实打实的"资产"——硬件资源：寄存器地址、中断号、DMA通道等。这些资产存在`struct resource`数组里，或者存在设备树（DT）解析出来的属性中。
- **device_driver（驱动）** = **女方**。它手里没有硬件，有的是"手艺"——probe、remove、suspend、resume这些回调函数。驱动知道自己会伺候什么样的设备（靠`of_match_table`或`id_table`来表达择偶标准）。

当设备注册到总线上时，婚介所（bus）就会翻出驱动名单，一个个比对："这个驱动支持你的型号吗？"——这就是`match()`。对上了眼，婚介所就安排双方见面——调用驱动的`probe()`。probe里，驱动会去申请内存、映射IO、注册中断，正式"嫁入"设备。反过来，驱动先注册到总线上也一样——婚介所会去扫设备名单，看有没有合适的"男方"。

### 三个关键结构体的核心字段

这三个结构体定义在`<linux/device.h>`里，日常开发你不需要记住所有字段，但下面这些必须眼熟：

```c
// 总线类型——婚介所的规则手册
struct bus_type {
    const char      *name;           // 总线名称，如 "platform"
    int (*match)(struct device *dev, struct device_driver *drv);
    int (*probe)(struct device *dev);
    int (*remove)(struct device *dev);
    struct subsys_private *p;        // 内部链表：设备和驱动都挂在这里
};

// 设备——男方，手里有硬件资源
struct device {
    struct device       *parent;     // 父设备，构成层次结构
    struct bus_type     *bus;        // 我挂在哪条总线上
    struct device_driver *driver;    // 匹配成功后指向我的驱动
    void                *platform_data;  // 板级私有数据
    struct resource     *resource;   // IO内存、中断等资源（老方式）
    struct device_node  *of_node;    // 设备树节点（新方式）
    struct kobject      kobj;        // 嵌入的kobject，用于sysfs
};

// 驱动——女方，手里有操作手法
struct device_driver {
    const char          *name;
    struct bus_type     *bus;        // 我要挂到哪条总线上找对象
    int (*probe)(struct device *dev);
    int (*remove)(struct device *dev);
    const struct of_device_id *of_match_table;  // 设备树匹配表（择偶标准）
    struct driver_private *p;        // 内部管理用
};
```

注意一个关键点：`struct device`里**内嵌**了一个`struct kobject kobj`，不是指针，是直接包含。这意味着每个设备天然就是一个内核对象，自带引用计数和sysfs目录。

### 三者的职责对比

| 角色 | 核心数据 | 关键回调 | 职责比喻 | 你写代码时主要碰谁 |
|------|---------|---------|---------|-----------------|
| bus_type | 设备链表、驱动链表 | match()、probe()、remove() | 婚介所，撮合配对 | 很少直接写，platform_bus_type已写好 |
| device | resource/of_node | 无（被动等驱动来操作） | 男方，出硬件资源 | 设备树里描述，或手动填充platform_device |
| device_driver | of_match_table、id_table | probe()、remove() | 女方，出软件实现 | **经常写**，你的驱动主要就是实现这个 |

### match 到底怎么比的？

不同总线的match逻辑不一样：

- **platform总线**：先比`id_table`（名字匹配），再比`of_match_table`（设备树的compatible属性匹配）。
- **USB总线**：比vendor ID和product ID，跟设备树没关系。
- **PCI总线**：比vendor ID、device ID、class码。
- **I2C/SPI总线**：也比`of_match_table`里的compatible。

所以你会发现，现代ARM嵌入式开发里，不管是platform、i2c还是spi驱动，你都在写`of_match_table`，靠设备树里的compatible字符串来配对。这就是"婚介所"的统一规则在不同总线上的具体实现。

### kobject/kset：看不见的"户籍系统"

这三个结构体底层都依赖kobject来组织。你可以把kobject理解为"户籍档案卡"：每张卡有名字、引用计数、指向父母的指针。一堆相关的卡归到一个kset里，kset再挂到更大的kset下，最终在sysfs里呈现出`/sys/bus/`、`/sys/devices/`、`/sys/drivers/`这样的目录树。

```
/sys/bus/platform/          <- bus_type对应的kset
├── devices/                <- 挂在这总线上的设备
│   └── soc/serial@ff000000 <- 一个device的kobject
└── drivers/                <- 挂在这总线上的驱动
    └── serial              <- 一个device_driver的kobj
```

这不是巧合——sysfs里的目录层级就是kobject父子关系直接映射出来的。你`ls /sys/devices/`看到的每一个目录，背后都对应一个`struct device`里的kobject。

## 知识点132：kobject和kset——设备模型的"钢筋混凝土" [I] ~600字

### kobject：万物之源

如果说bus、device、driver是"看得见"的大构件，那kobject就是浇筑它们的"钢筋混凝土"。`struct kobject`定义在`<linux/kobject.h>`，非常精简：

```c
struct kobject {
    const char          *name;       // 在sysfs中显示的目录名
    struct list_head    entry;       // 用于挂到kset的链表中
    struct kobject      *parent;     // 父对象，决定sysfs中的层级
    struct kset         *kset;       // 所属集合
    struct kobj_type    *ktype;      // 类型描述（含release回调和默认属性）
    struct sysfs_dirent *sd;         // sysfs内部表示
    struct kref         kref;        // 引用计数
};
```

kobject干三件事，你记住就行：**命名**（我叫什么）、**找爹**（我在哪一层）、**计数**（我什么时候能释放）。内核通过`kobject_init()`初始化、`kobject_add()`注册到sysfs、`kobject_put()`减少引用计数。计数归零时，调用ktype里的`release()`做清理。

### kset：kobject的"户口本"

kset是同一类kobject的集合，它自己内部也内嵌了一个kobject——所以kset本身在sysfs里也是一个目录。比如`devices_kset`对应`/sys/devices/`，`bus_kset`对应`/sys/bus/`。

```c
struct kset {
    struct list_head    list;        // 该集合下的所有kobject链表
    spinlock_t          list_lock;
    struct kobject      kobj;        // 内嵌kobject，代表这个集合自身
    const struct kset_uevent_ops *uevent_ops;  // 热插拔事件通知
};
```

当你把一个设备的kobject的`kset`字段指向`devices_kset`，这个设备就自动出现在`/sys/devices/`下了。父kobject设成谁，它就在sysfs里出现在谁的下级目录。这套机制让内核在添加设备时，自动建立起整齐的目录层次，不用你手动去创sysfs节点。

### sysfs映射：kobject的"倒影"

sysfs本质上是一个虚拟文件系统，挂载在`/sys`下。它的每个目录、每个文件，都直接对应一个kobject的层次结构或一个kobject的属性（attribute）。当你`cat /sys/devices/soc/serial/ff000000/uevent`时，你读的不是磁盘文件，而是内核里那个device的kobject关联的属性回调函数返回的字符串。

这个映射是**自动的**：你注册一个kobject，sysfs里就多一个目录；你给kobject加一组`struct attribute`，sysfs里就多几个可读写的文件。这就是为什么设备模型代码和`/sys`下的内容总是同步的——它们本就是一体两面。

## 本节总结

| 概念 | 核心要点 | 自查方法 |
|------|---------|---------|
| bus_type | 总线是撮合者，定义match规则和probe入口 | `ls /sys/bus/` 看有哪些总线 |
| device | 代表硬件，手里有resource/of_node资源 | `ls /sys/devices/` 看设备树 |
| device_driver | 代表软件实现，提供probe/remove回调 | `ls /sys/bus/platform/drivers/` |
| 三角配对 | 设备或驱动注册时，bus调用match→probe完成绑定 | 看内核log里的"probe"信息 |
| kobject | 基类对象，提供引用计数和层次结构 | 理解所有设备模型结构都内嵌kobj |
| sysfs映射 | kobject的层次直接反映为/sys下的目录树 | 对比`/sys/devices/`和你的设备树节点 |
