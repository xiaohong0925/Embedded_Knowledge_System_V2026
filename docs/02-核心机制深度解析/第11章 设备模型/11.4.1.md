# 11.4.1 sysfs三视图

> 所属章节：第11章 设备驱动模型 > 11.4 sysfs文件系统
> 难度：[I→M] | 预计阅读时间：12分钟

## 本节导读
本节带你理解sysfs里三个最核心的目录——`/sys/bus/`、`/sys/class/`、`/sys/devices/`——它们本质上是**同一批设备的不同排列方式**。学完后，你能用三种视角描述同一个LED设备，并且知道查驱动该去哪儿、看功能该去哪儿。

---

## 知识点154：`/sys/bus/`、`/sys/class/`、`/sys/devices/`——同一个设备的"三张名片" [I→M] ~1000字

### 三个目录到底啥关系？

想象你有一个同事叫老王，他在公司里有三种身份标签：

| 标签维度 | 按什么组织 | 老王的例子 |
|---------|----------|-----------|
| **bus** | 按所属部门 | "研发部·嵌入式组" |
| **class** | 按技能特长 | "会写驱动的C程序员" |
| **devices** | 按工位位置 | "3号楼·5层·27号工位" |

sysfs里的三个目录就是类似的关系——**同一批设备，按三种不同的逻辑重新排队**。设备本身只有一个物理实体（在`/sys/devices/`里），但内核同时给它建立了两条"软链接"，分别挂到对应的bus和class下面。

### 按总线组织：/sys/bus/

`/sys/bus/`是按"设备挂在哪条总线上"来归类。常见子目录：

```
/sys/bus/
├── platform/          # 平台总线（没有实体总线的片上设备）
├── i2c/               # I2C总线
├── spi/               # SPI总线
├── pci/               # PCI/PCIe总线
├── usb/               # USB总线
└── ...
```

每个bus目录下又分`devices/`和`drivers/`两个子目录。比如你要看I2C-0总线上挂了哪些设备：

```bash
# 代码1：查看I2C总线上注册的设备
$ ls /sys/bus/i2c/devices/
i2c-0  0-0050  0-0068
```

这里的`0-0050`就表示I2C-0总线上、从机地址为0x50的芯片。驱动开发者匹配设备时，内核就是按这个bus路径来做`match()`的。

### 按功能组织：/sys/class/

`/sys/class/`是按"设备提供什么功能"来归类，完全不管它接在哪条总线上。比如：

```
/sys/class/
├── leds/              # LED设备（可以是GPIO驱动、I2C扩展芯片、PWM等）
├── gpio/              # GPIO字符设备
├── net/               # 网卡（eth0、wlan0等）
├── input/             # 输入设备（键盘、鼠标、触摸屏）
├── rtc/               # 实时时钟
├── backlight/         # 背光
└── ...
```

`/sys/class/`是给**用户空间应用程序**看的。你想控制LED的亮度，直接找到`/sys/class/leds/`下面的`brightness`文件就行，根本不用关心这个LED是接到GPIO上的、I2C扩展芯片上的、还是SOC内置的PWM模块。

### 按物理拓扑组织：/sys/devices/

`/sys/devices/`是唯一反映**真实硬件连接关系**的目录。它是整个sysfs的"物理骨架"，其他两个视图都是它的"软链接快捷方式"。

```
/sys/devices/
├── platform/          # 平台设备
│   └── soc/
│       └── 2000000.pinctrl/    # 实际的GPIO控制器
├── pci0000:00/        # PCIe根复合体
└── virtual/           # 虚拟设备（如loopback网卡）
```

### 同一个设备的三条路径

下面以开发板上一个接到GPIO上的状态LED为例，展示它在三个视图中的路径：

| 视图维度 | sysfs路径 | 视角说明 |
|---------|----------|---------|
| **bus** | `/sys/bus/platform/devices/gpio-leds` | 注册到platform总线上的设备 |
| **class** | `/sys/class/leds/status_led/` | 作为一个LED功能设备 |
| **devices** | `/sys/devices/platform/gpio-leds/leds/status_led/` | 物理拓扑中的真实位置 |

你可以自己验证这个软链接关系：

```bash
# 代码2：验证class目录下的LED实际是devices的软链接
$ ls -l /sys/class/leds/status_led
lrwxrwxrwx 1 root root 0 Jan 10 09:12 /sys/class/leds/status_led -> ../../devices/platform/gpio-leds/leds/status_led/
```

看到那个箭头了吧？`class/leds/`下面的每个条目其实都指向`devices/`里的某个真实节点。bus目录下同理。

### 一张图搞清三者的本质

```
                    ┌─────────────────────┐
                    │   物理实体（唯一）    │
                    │  /sys/devices/...   │
                    │   (真实硬件拓扑)      │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
        ┌─────────┐     ┌──────────┐     ┌──────────┐
        │  /bus/  │     │ /class/  │     │ /dev/    │
        │按总线组织│     │ 按功能组织│     │ 设备节点  │
        │驱动匹配用│     │用户操作  │     │应用程序  │
        └─────────┘     └──────────┘     │ open()用  │
                                         └──────────┘
```

[图1：sysfs三视图关系示意图]

🔴 **危险**：`class`和`bus`下面的目录都是软链接，不是真正的文件系统节点。如果你写脚本时用了`cp -r`或者`du`去递归处理`/sys/class/`，可能会陷入循环链接或者报错。操作sysfs一定要记住：**读属性文件OK，直接操作真实路径更稳妥**。

## 知识点155：实战——ls查看与三个视图的使用场景 [I] ~600字

### 动手验证

打开你的开发板或者虚拟机，按下面的步骤走一遍，比看十遍文档都管用。

```bash
# 代码3：三个视图的入口目录长什么样
$ ls /sys/bus/ | head -5
ac97       container  edac       i2c        nd
$ ls /sys/class/ | head -5
ata_device  bdi         block       bluetooth   devcoredump
$ ls /sys/devices/ | head -5
breakpoint  cpu         i2c-0       i2c-1       i2c-2

# 代码4：找一个具体的网卡，看它在三个视图中的位置
$ ls /sys/class/net/
eth0  lo

# 这个eth0是从哪儿链接过来的？
$ readlink -f /sys/class/net/eth0
/sys/devices/platform/soc/1c30000.ethernet/net/eth0
```

用`readlink -f`追查一下，你会发现所有`class/`和`bus/`下的节点都指向`devices/`里的某个角落。这就是sysfs的设计核心：**devices是树干，bus和class是两棵不同的枝叶**。

### 三个视图，三种排查思路

实际工作中，不同的问题要去不同的目录查。

| 排查场景 | 该看的视图 | 为什么 | 常用命令 |
|---------|----------|--------|---------|
| 驱动有没有加载成功 | `/sys/bus/.../drivers/` | 看驱动名是否出现在对应总线的drivers目录 | `ls /sys/bus/platform/drivers/` |
| 设备有没有被识别 | `/sys/bus/.../devices/` | 看设备名是否挂在对应总线上 | `ls /sys/bus/i2c/devices/` |
| 应用程序怎么操作设备 | `/sys/class/.../` | 功能接口都在这里，不需要知道底层总线 | `echo 1 > /sys/class/leds/xxx/brightness` |
| 硬件连接对不对 | `/sys/devices/` | 反映真实的SOC → 总线 → 设备的树状拓扑 | `tree /sys/devices/platform` |
| 排查中断号冲突 | `/sys/devices/` → `irq` | 从物理设备节点查分配的IRQ | `cat /sys/devices/.../irq` |

### 一个真实的小故事

有一次我调一块板子，I2C触摸屏没反应。我的排查顺序是这样的：

1. **先看bus**：`ls /sys/bus/i2c/devices/` → 发现没有`1-0038`（触摸屏的I2C地址），说明内核根本没识别到这个设备，大概率是设备树没配或者硬件没接上；
2. **修复设备树后**，重新查看bus目录，设备出现了；
3. **再看class**：`ls /sys/class/input/` → 出现了`event1`，说明input子系统已经把它注册成输入设备；
4. **最后进devices**：`cat /sys/devices/platform/soc/1c2b000.i2c/i2c-1/1-0038/name` → 确认芯片型号匹配正确。

整个过程用到了三个视图，每个视图回答了一个不同层面的问题。**bus告诉你" kernel 认没认"，class告诉你"能不能用"，devices告诉你"物理上长啥样"**。

💡 **提示**：写udev规则或者systemd的`.device`单元时，通常用`/sys/class/`下的路径最方便，因为这个视角跟应用程序看到的`/dev/`节点是一一对应的。

⚠️ **陷阱**：`/sys/class/`下的目录名不一定是固定的。比如有的内核版本网卡叫`eth0`，有的叫`enp2s0`。写脚本时别硬编码设备名，最好用`udev`的`SYMLINK`或者按class目录遍历。

## 本节总结

| 概念 | 核心要点 | 自查操作 |
|------|---------|---------|
| `/sys/bus/` | 按总线组织，用于驱动匹配和设备枚举 | `ls /sys/bus/i2c/devices/` |
| `/sys/class/` | 按功能组织，给用户空间提供统一操作接口 | `ls /sys/class/leds/` |
| `/sys/devices/` | 按物理拓扑组织，是sysfs的真实"骨架" | `tree /sys/devices/platform` |
| 三者关系 | class和bus都是devices的软链接视图 | `readlink -f /sys/class/xxx/yyy` |
| 排查思路 | 认设备→查bus、用功能→查class、看拓扑→查devices | 按上表场景选择视图 |
