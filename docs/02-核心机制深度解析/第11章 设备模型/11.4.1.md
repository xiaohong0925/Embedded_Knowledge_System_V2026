### 11.4.1 sysfs三视图

**本节导读**：sysfs 是设备模型在用户空间的"投影"。同一个设备，内核会从不同角度向用户空间呈现。本节我们一起搞清楚 `/sys/bus/`、`/sys/class/`、`/sys/devices/` 这三个目录的划分逻辑，理解它们之间的关联。学完你将能准确找到任何设备的 sysfs 入口，不再在三者之间迷路。

---

当你用 `ls /sys/` 查看时，会看到一堆目录。对于刚接触设备模型的开发者来说，最 confusing 的就是：**同一个设备，为什么在好几个地方都能看见？**

打个比方。假设你有一台公司配的笔记本电脑：
- 在**组织架构**视角，它属于"研发部→嵌入式组"的资产
- 在**功能分类**视角，它是一台"开发工作站"
- 在**物理存放**视角，它在"3号楼→5层→工位A7"

三种视角描述的是**同一台电脑**，但组织方式完全不同。sysfs 的三视图正是这个道理。

---

#### 视图一：/sys/bus/ —— 按总线类型组织

`bus` 目录按**总线类型**对设备进行归类。系统中的每一种总线都有一个子目录：

```
/sys/bus/
├── platform/          ← 平台总线，片上资源都在这里
├── i2c/               ← I2C 总线设备
├── spi/               ← SPI 总线设备
├── pci/               ← PCI/PCIe 设备
├── usb/               ← USB 设备
├── sdio/              ← SDIO 设备
└── ...
```

每个总线目录下又有两个子目录：
- `devices/` —— 挂在这条总线上的所有设备
- `drivers/` —— 绑定到这条总线的所有驱动

💡 **提示**：`/sys/bus/platform/devices/` 是嵌入式开发最常打交道的地方，SoC 内部的外设（UART、GPIO 控制器、定时器等）大多挂在这里。

---

#### 视图二：/sys/class/ —— 按功能类别组织

`class` 目录不关心设备接在什么总线上，它按**功能类别**分组。你想找"所有的 LED 设备"或"所有的网卡"，来这就对了：

```
/sys/class/
├── leds/              ← 所有 LED 设备
├── net/               ← 所有网络接口
├── input/             ← 所有输入设备（键盘、鼠标、触摸屏）
├── gpio/              ← GPIO 相关
├── rtc/               ← 实时时钟
├── tty/               ← 所有串口终端
├── block/             ← 块设备
└── ...
```

💡 **提示**：写应用层代码时，`/sys/class/` 往往是最顺手的入口。比如控制 LED，直接去 `/sys/class/leds/` 下找对应的 `brightness` 文件即可，不用管 LED 芯片接在 I2C 还是 GPIO 上。

---

#### 视图三：/sys/devices/ —— 按物理拓扑组织

`devices` 目录反映的是**真实的硬件层次结构**。它描述的是物理上的父子关系：

```
/sys/devices/
├── platform/          ← platform 设备根
│   └── 1f00000000.bus/
│       └── gpio-controller@.../
├── pci0000:00/        ← PCIe 根复合体
│   └── 0000:00:1f.3/  ← 具体 PCIe 设备
├── system/            ← CPU、内存等系统级设备
└── virtual/           ← 虚拟设备（如 loopback 网卡）
```

内核设备模型中的父子关系（`struct device` 里的 `parent` 指针）在这里一目了然。

---

#### 三者的关系：同一个设备，三张名片

关键问题来了：这三个视图怎么关联起来？

答案是——**软链接（symbolic link）**。同一个设备对象在内核中只有一份 `struct device`，但在 sysfs 中通过符号链接出现在多个位置。

来看看一张简化的对应关系图：

```
                    内核中的同一个 device 对象
                        (struct device)
                              |
        +---------------------+---------------------+
        |                     |                     |
        v                     v                     v
   /sys/bus/            /sys/class/          /sys/devices/
   platform/            leds/                platform/
   devices/             sys-led/             gpio-controller@0/
   |
   +-- sys-led -> ../../../devices/platform/gpio-controller@0/sys-led

   /sys/class/leds/sys-led
   |
   +-- sys-led -> ../../devices/platform/gpio-controller@0/sys-led
```

说白了，`/sys/bus/` 和 `/sys/class/` 目录下的设备条目本质上都是**指向 `/sys/devices/` 中真实节点的软链接**。`/sys/devices/` 才是设备的"大本营"。

用一个更直观的示意图表示：

```
┌─────────────────────────────────────────────────────────────┐
│                     /sys/devices/ (物理拓扑)                  │
│                      设备的"大本营"                            │
│                                                             │
│  platform/                                                  │
│   └── gpio-controller@0/  ←────── 真实 device 目录           │
│        └── sys-led/                                         │
│                                                             │
│         ▲                          ▲                        │
│         │  软链接                    │  软链接                 │
│         │                          │                        │
├─────────┼──────────────────────────┼────────────────────────┤
│         │  /sys/bus/               │  /sys/class/           │
│         │  (按总线组织)              │  (按功能组织)           │
│         │                          │                        │
│  platform/devices/           leds/                          │
│   └── sys-led ───────────────┘                              │
│        (链接到 devices/platform/gpio-controller@0/sys-led)  │
└─────────────────────────────────────────────────────────────┘
```

---

#### 实战：查看 platform 设备

在任意嵌入式 Linux 开发板上执行：

```bash
$ ls /sys/bus/platform/devices/
```

你会看到一串熟悉的名字：

```
40100000.serial
40102000.spi
40300000.i2c
42000000.timer
gpio-keys
leds-gpio
```

这些就是当前系统注册的所有 platform 设备。点进去看：

```bash
$ ls -l /sys/bus/platform/devices/40100000.serial
lrwxrwxrwx ... /sys/bus/platform/devices/40100000.serial -> ../../../devices/platform/40100000.serial/
```

看到那个 `->` 了吗？这就是软链接，它最终指向 `/sys/devices/platform/` 下的真实目录。

⚠️ **陷阱**：不要误以为 `/sys/bus/platform/devices/` 下的文件是设备本身。如果你在脚本里用 `readlink -f` 解析这些路径，会发现它们的"真身"都在 `/sys/devices/` 下面。

---

#### 三视图对比速查

| 维度 | `/sys/bus/` | `/sys/class/` | `/sys/devices/` |
|:---|:---|:---|:---|
| **组织依据** | 总线类型 | 功能类别 | 物理拓扑层次 |
| **典型子目录** | `platform/`, `i2c/`, `spi/` | `leds/`, `net/`, `input/` | `platform/`, `pci0000:00/` |
| **是否真实节点** | 软链接 | 软链接 | 真实目录 |
| **常用场景** | 查看某总线上有哪些设备 | 按功能查找设备 | 看硬件拓扑、找父设备 |
| **对应内核结构** | `struct bus_type` | `struct class` | `struct device` 的 parent 链 |

💡 **提示**：记住一个口诀——"devices 是家，bus 和 class 是分门别类的电话簿"。
