# 11.2.1 设备树到platform_device

> 所属章节：第11章 设备驱动进阶 > 11.2 平台设备与设备树
> 难度：[E→M] | 预计阅读时间：15分钟

## 本节导读
本节解决一个很多驱动工程师纠结的问题：设备树里写了节点，为什么驱动probe还是没被执行？答案往往藏在status属性里。另外，我们会拆开of_platform_populate()的运作过程，看它是怎么把设备树节点翻译成内核里的platform_device和resource数组的。搞懂这个，后面写平台驱动时心里才有底。

---

## 知识点136：of_platform_populate()——设备树节点的"造物主" [E][M] ~1200字

### 从设备树节点到platform_device：谁干的活？

设备树（DTB）被内核解成一棵device_node树之后，并不能直接对接驱动模型。驱动认的是platform_device，不认device_node。这中间需要一个"翻译官"把device_node翻译成platform_device——这个翻译官就是`of_platform_populate()`。

内核在板级初始化阶段会调用它，通常藏在`arch/arm/mach-xxx/board.c`或者`arch/arm64/kernel/setup.c`的某个角落里。它的逻辑说起来不复杂：遍历设备树根节点下的子节点，对每个status="okay"的节点，创建一个对应的platform_device。

```
// 伪代码示意，源码在 drivers/of/platform.c
void of_platform_populate(struct device_node *root, 
                          const struct of_device_id *matches,
                          struct device_node *parent)
{
    for_each_child_of_node(parent, child) {
        if (!of_device_is_available(child))  // 检查status="okay"
            continue;                        // disabled？跳过！
        
        // 创建platform_device
        pdev = of_platform_device_create_pdata(child, ...);
        
        // 递归处理子节点
        of_platform_populate(NULL, NULL, child);
    }
}
```

注意第5行那个`of_device_is_available()`——它只认status="okay"。如果你的设备树节点没写status，默认就是"okay"；但如果写了status="disabled"，这里直接continue，后面的platform_device根本不会被创建。

🔴 **血泪教训**：调试驱动时probe没执行，第一反应查compatible对不对，第二反应就应该查设备树里的status是不是"disabled"！我见过太多人在compatible上纠结半天，最后发现是设备树从别的板子抄过来时status="disabled"忘了改。这是probe不执行的**头号原因**。

### resource数组：platform_device的"硬件户口本"

platform_device创建出来后，光有个名字和compatible还不够，驱动需要知道硬件资源在哪——内存地址是多少？中断号是多少？这些信息从设备树的属性来，填充到platform_device的resource数组里。

具体的映射关系由`of_address_to_resource()`和`of_irq_to_resource()`完成：

- 设备树里的`reg = <0xFF750000 0x1000>` → 变成`IORESOURCE_MEM`类型的resource，start=0xFF750000，end=0xFF750FFF
- 设备树里的`interrupts = <0 36 4>` → 变成`IORESOURCE_IRQ`类型的resource，start=end=36（SPI中断号）

一个节点可以有多个reg条目，也可以有多个interrupts条目，内核会把它们按顺序塞进resource数组。驱动probe时通过`platform_get_resource()`或者`platform_get_irq()`按索引去取。

### 整个转化流程一览

```mermaid
flowchart TD
    A[设备树节点 .dts] -->|dtb展开为| B[device_node树]
    B -->|of_platform_populate 遍历| C{status = ?}
    C -->|"okay或未指定"| D[创建 platform_device]
    C -->|"disabled"| E[直接跳过<br/>probe不会执行]
    D -->|reg属性| F[resource[0]:<br/>IORESOURCE_MEM]
    D -->|interrupts属性| G[resource[1]:<br/>IORESOURCE_IRQ]
    D -->|其他属性| H[存入 dev.platform_data<br/>或设备节点指针]
    F --> I[platform_device 完整结构]
    G --> I
    H --> I
    I -->|驱动匹配| J[执行 platform_driver.probe]
```

### 设备树属性与platform_resource映射关系

| 设备树属性 | 转化函数 | resource类型 | 说明 |
|-----------|---------|-------------|------|
| `reg = <addr size>` | `of_address_to_resource()` | `IORESOURCE_MEM` | 寄存器基地址和长度，可多个条目 |
| `interrupts = <...>` | `of_irq_to_resource()` | `IORESOURCE_IRQ` | 中断号，可多个条目 |
| `reg-names = "ctl", "fifo"` | 辅助命名 | — | 配合platform_get_resource_byname()使用 |
| `interrupt-names = "rx", "tx"` | 辅助命名 | — | 配合platform_get_irq_byname()使用 |
| `dma-names = "rx", "tx"` | `of_dma_request_slave_channel()` | 不放入resource | DMA通道单独申请，不走resource数组 |

⚠️ **陷阱**：很多新手以为设备树里写了interrupts，驱动里直接用`request_irq()`硬编码中断号。这是错误的！正确做法是用`platform_get_irq(pdev, 0)`从platform_device的resource数组里取。这样即使硬件换了中断号（不同板子设备树不同），驱动代码也不用改。

💡 **技巧**：如果你的设备有多个reg区间，比如`reg = <0xFF750000 0x1000>, <0xFF751000 0x1000>`，那么`platform_get_resource(pdev, IORESOURCE_MEM, 0)`取第一个，`platform_get_resource(pdev, IORESOURCE_MEM, 1)`取第二个。索引顺序就是设备树里写的顺序。

### 为什么disabled节点一定要跳过？

这里有个设计哲学问题：为什么不能给disabled节点也创建设备，只是不probe？

答案很简单：platform_device一旦创建就会进入设备模型，会被总线匹配、uevent上报、 sysfs导出。如果disabled节点也走这套流程，内核里会多出一堆"幽灵设备"——它们存在于sysfs，有目录有属性，但永远用不了。这不仅浪费内存，还会让调试时产生困惑（ls /sys/bus/platform/devices/ 能看到它，但读写不了）。

所以status="disabled"的设计意图是：这个硬件在**当前板级配置**中物理上不存在或者不被使用。内核应该当它不存在，连device都不要创建。

## 知识点137：platform_device.resource结构与调用链 [E] ~600字

### struct resource到底长什么样？

resource数组虽然名字叫"资源"，但本质是几个整数加标志位，定义在`<linux/ioport.h>`：

```c
struct resource {
    resource_size_t start;      // 资源起始地址/中断号
    resource_size_t end;        // 资源结束地址（中断时start=end）
    const char *name;           // 资源名称，通常取自reg-names/interrupt-names
    unsigned long flags;        // 资源类型：IORESOURCE_MEM / IORESOURCE_IRQ / IORESOURCE_DMA
    // ... 链表指针等内部字段
};
```

平台驱动里最常见的用法：

```c
static int my_driver_probe(struct platform_device *pdev)
{
    struct resource *res;
    int irq;

    // 取第一个内存资源
    res = platform_get_resource(pdev, IORESOURCE_MEM, 0);
    if (!res)
        return -ENODEV;
    
    // 物理地址转虚拟地址
    void __iomem *base = devm_ioremap_resource(&pdev->dev, res);
    
    // 取第一个中断
    irq = platform_get_irq(pdev, 0);
    if (irq < 0)
        return irq;
    
    // 申请中断
    devm_request_irq(&pdev->dev, irq, my_isr, 0, "my_dev", dev);
    
    // ...
}
```

`devm_ioremap_resource()`这一行实际上同时做了几件事：把res里的start/end传给`request_mem_region()`标记占用，再调用`ioremap()`建立页表映射。如果start/end对应的区域已经被别的驱动占了，这里会直接返回-EBUSY——这就是resource结构的保护作用，防止两个驱动踩同一块硬件寄存器。

### of_platform_populate()从哪被调起来？

很多驱动工程师只写驱动不写板级代码，对调用链不太熟悉。实际在ARM64上，调用路径大致是这样的：

```
start_kernel()
  └── setup_arch()
        └── unflatten_device_tree()      // 把dtb展开成device_node树
        └── ...
  └── rest_init()
        └── kernel_init()
              └── kernel_init_freeable()
                    └── do_basic_setup()
                          └── do_initcalls()  // 或者板级特定代码
                                └── of_platform_populate()  // 这里！
```

具体的，很多SoC厂商会在自己的板级initcall里调用：

```c
// 以某ARM64 SoC为例
static int __init my_soc_init(void)
{
    struct device_node *np;
    
    np = of_find_node_by_path("/soc");
    if (!np)
        return -ENODEV;
    
    // 遍历/soc下的所有节点，创建platform_device
    of_platform_populate(np, of_default_bus_match_table, NULL);
    
    return 0;
}
arch_initcall_sync(my_soc_init);  // 在arch_initcall阶段执行
```

`of_default_bus_match_table`是内核提供的默认匹配表，会把常见的总线节点（simple-bus、amba等）也递归展开。如果你的设备树soc节点compatible里带了"simple-bus"，那of_platform_populate会把soc下面的所有子节点都递归处理一遍。

## 本节总结

| 概念 | 核心要点 | 自查方法 |
|------|---------|---------|
| of_platform_populate() | 遍历设备树，为status="okay"的节点创建platform_device | 搜内核源码，看板级代码里在哪调用 |
| status="disabled" | 节点直接被跳过，platform_device不会创建，probe不会执行 | 驱动probe不执行时，先查设备树这个属性 |
| resource数组 | platform_device携带的硬件资源列表，含MEM/IRQ/DMA | `platform_get_resource()` / `platform_get_irq()` |
| struct resource | start/end/flags/name四个核心字段，描述一段硬件资源 | 读`<linux/ioport.h>` |
| 调用链 | setup_arch()展开device_node → initcall里调of_platform_populate() |  tracing或者加printk跟踪 |
