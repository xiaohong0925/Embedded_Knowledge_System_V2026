## 11.1.4 kobject/kset与sysfs映射

### 本节导读

你已经在/sys/下见过各种设备和驱动文件，但它们是怎么来的？本节揭示sysfs目录背后的机制——**kobject**。学完本节，你能理解kobject如何将设备模型的层次结构自动映射为/sys/目录树，掌握kobject和kset的核心用法，并能在/sys/下"读懂"内核设备模型的组织结构。

---

### kobject：设备模型的原子单位

**知识点137 [I]**

kobject（kernel object）是Linux设备模型中最基础的构建块。它本身是内嵌结构体——其他结构体（如device、bus_type）将kobject嵌入其中，从而获得设备模型的全部能力。

kobject的核心定义在 `include/linux/kobject.h` 中，关键字段如下：

```c
struct kobject {
    const char          *name;       /* 目录名 */
    struct list_head    entry;       /* 链表节点 */
    struct kobject      *parent;     /* 父对象，构成层次树 */
    struct kset         *kset;       /* 所属集合 */
    struct kobj_type    *ktype;      /* 属性操作回调 */
    struct sysfs_dirent *sd;         /* sysfs目录项 */
    struct kref         kref;        /* 引用计数 */
    ...
};
```

四个字段最重要：**name** 决定 `/sys/` 下的目录名；**parent** 指针决定目录的层级位置；**ktype** 提供该对象的属性和释放回调；**kref** 负责生命周期管理。

```c
/* 注册一个kobject → 在sysfs中创建目录 */
int kobject_add(struct kobject *kobj, struct kobject *parent,
                const char *fmt, ...);
void kobject_del(struct kobject *kobj);
```

当调用 `kobject_add()` 时，内核沿 parent 指针找到根节点，在 sysfs 中逐级创建目录。这就是 `/sys/` 目录树和设备模型层次树一一对应的根本原因。

### kset：kobject的容器

kset 是 "kobject 的集合"。它包含一个链表（list_head）将同类型的 kobject 串联起来，同时自身也嵌入一个 kobject——这意味着 kset 自己就是 sysfs 中的一个目录。

```c
struct kset {
    struct list_head    list;        /* kobject链表 */
    struct kobject      kobj;        /* kset自身也是kobject */
    const struct kset_uevent_ops *uevent_ops;
};
```

说白了，kset 承担两个角色：一是**容器**——管理一组有相同 ktype 的 kobject；二是**目录**——在 sysfs 中表现为一个父目录。

来看一个典型的映射关系：

```
        kobject层次树                    sysfs目录层次
    ┌──────────────────┐            ┌────────────────────┐
    │ kset: devices    │            │ /sys/devices/      │
    │ (kobj.name)      │───────────>│                    │
    │  ├── kobj: pci0  │            │  ├── pci0000:00/   │
    │  ├── kobj: usb1  │            │  ├── usb1/         │
    │  └── kobj: usb2  │            │  └── usb2/         │
    └──────────────────┘            └────────────────────┘
```

parent 指针建立起目录层次，`kset` 字段决定kobject挂在哪个集合里。

### ktype与属性文件

目录有了，里面的文件从哪来？答案在 `kobj_type`：

```c
struct kobj_type {
    void (*release)(struct kobject *kobj);
    const struct sysfs_ops *sysfs_ops;
    struct attribute **default_attrs;   /* 默认属性列表 */
    const struct kobj_ns_type_operations *(*child_ns_type)(...);
};

struct attribute {
    const char  *name;      /* 文件名，如 "uevent" */
    struct module *owner;
    umode_t     mode;       /* 文件权限 */
};
```

`default_attrs` 数组中的每个 attribute 都会变成 sysfs 目录下的一个文件。读写这些文件时，内核通过 `sysfs_ops->show()` 和 `store()` 回调将操作转发给驱动。

```
kobject注册
   │
   ├── 根据 parent 创建目录路径
   │      └── /sys/devices/pci0000:00/
   │
   └── 遍历 ktype->default_attrs[]
          ├── 创建文件 "vendor"   → /sys/devices/.../vendor
          ├── 创建文件 "device"   → /sys/devices/.../device
          └── 创建文件 "uevent"   → /sys/devices/.../uevent
```

💡 **提示**：attribute 只定义了文件名和权限，真正的读写逻辑在 sysfs_ops 中。这是一种典型的"数据+操作分离"设计。

### uevent：kobject状态变化的广播

kobject 状态变化时会发送 uevent，用户空间的 udevd 监听后做出响应。核心函数是 `kobject_uevent()`：

```c
/* 发送ADD事件 → udevd收到 → 创建设备节点 */
int kobject_uevent(struct kobject *kobj, enum kobject_action action);
```

当设备驱动调用 `device_add()` 时，内部会触发 `KOBJ_ADD` 事件。udevd 收到后查询 devtmpfs，在 `/dev/` 下创建对应的设备节点。

```
驱动调用 device_add()
        │
        ▼
    kobject_uevent(KOBJ_ADD)
        │
        ▼
    netlink广播给用户空间
        │
        ▼
    udevd 收到 → 创建 /dev/xxx → 加载模块/权限设置
```

⚠️ **陷阱**：如果你的设备在 `/sys/` 下出现了目录但没有 `/dev/` 节点，大概率是 uevent 没发出去，或者 udevd 没有正常运行。检查 `dmesg | grep uevent` 或手动触发：`echo add > /sys/.../uevent`。

### 实战：在/sys/下验证映射

假设你的系统有一个PCI网卡，来看看kobject的映射效果：

```bash
# 查看设备目录层次（kobject parent链路的体现）
$ ls /sys/devices/pci0000:00/0000:00:1f.6/
broken_parity_status  device      enable      modalias
power                 subsystem   uevent      vendor

# 向上追溯parent链路
$ readlink /sys/devices/pci0000:00/0000:00:1f.6/subsystem
/sys/bus/pci

# 手动触发uevent（发送KOBJ_CHANGE事件）
$ echo change > /sys/devices/pci0000:00/0000:00:1f.6/uevent
```

`uevent` 文件本身就是一个 attribute——写入 `add` / `remove` / `change` 会触发对应的事件。这招很管用，调试时常用来手动刷新 udev 规则。

💡 **提示**：`/sys/` 下的软链接（如 `subsystem` → `/sys/bus/pci`）实际上反映了kobject在多个集合中的注册关系——一个设备既属于总线层次，又属于设备层次。这种"多视图"是kset和kobject灵活组合的结果。

### 本节总结

| 概念 | 作用 | sysfs映射 |
|------|------|-----------|
| kobject | 设备模型原子单位 | 对应 `/sys/` 下的一个目录 |
| kobj->name | 对象名称 | 目录名 |
| kobj->parent | 父对象指针 | 目录层级关系 |
| ktype | 属性操作和释放回调 | 目录下的文件及其读写行为 |
| kset | kobject集合 | 包含子目录的父目录 |
| attribute | 属性定义 | 目录下的文件 |
| uevent | 状态变化通知 | `/sys/.../uevent` 文件 + netlink广播 |
