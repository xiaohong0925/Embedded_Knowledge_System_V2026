## 11.3.4 匹配竞争与优先级

### 本节导读

想象这样一个场景：你的板子上有两个驱动模块，都声称自己能支持同一个传感器——内核到底会选谁？本节讨论 **匹配竞争** 现象：当多个驱动的 `of_match_table` 包含相同的 `compatible` 字符串时，设备究竟绑定到哪个驱动。学完后，你能分析竞争成因，掌握三种标准解决策略，并在实际项目中通过 `dmesg` 和 `/sys` 文件系统快速定位"抢设备"问题。

---

### 匹配竞争场景

假设你的板级设备树里有一个温湿度传感器节点：

```dts
&i2c1 {
    sht30@44 {
        compatible = "sensirion,sht30";
        reg = <0x44>;
    };
};
```

现在，内核里有两个不同的驱动模块都想接管这个设备：

| 驱动模块 | of_match_table 中的 compatible |
|---------|-------------------------------|
| `sht3x-generic.ko` | `"sensirion,sht30"` |
| `sht3x-advanced.ko` | `"sensirion,sht30"` |

两个驱动的匹配表一模一样。问题来了——**内核会选哪个？**

来看 `driver_probe_device()` 的核心逻辑（简化）：

```c
// drivers/base/dd.c
int driver_probe_device(struct device_driver *drv, struct device *dev)
{
    // 1. 检查 dev->driver 是否已绑定
    if (dev->driver)
        return -EBUSY;        // ← 已经被绑定了！

    // 2. 调用 match 函数（bus.match 或 drv->of_match_table）
    ret = drv->bus->match ? drv->bus->match(dev, drv) : 1;
    if (ret == 0)
        return -ENODEV;       // 不匹配

    // 3. 真正执行 probe
    return really_probe(dev, drv);
}
```

`really_probe()` 的关键检查在这里：

```c
static int really_probe(struct device *dev, struct device_driver *drv)
{
    if (dev->driver) {
        pr_err("Device %s is already bound to driver %s\n",
               dev_name(dev), dev->driver->name);
        return -EBUSY;        // ← 🔴 第二次匹配直接失败
    }
    dev->driver = drv;        // 抢占绑定
    ...
}
```

说白了，**谁先执行 probe 成功，设备就是谁的**。第二个驱动再过来，只能看着 `dev->driver` 已被占用的现实，打印一条错误日志，然后默默退出。

---

### 竞争结果与隐患

先注册、先匹配的驱动胜出——但这个"先"字值得警惕。

⚠️ **陷阱**：加载顺序依赖 `Makefile` 中的编译顺序、`modules.builtin` 的链接顺序，以及 `initcall` 优先级。这些在不同内核版本中可能变化，导致**今天能用，明天崩了**。

假设你构建了两个 `.ko` 文件，在启动脚本中依次 `insmod`：

```bash
insmod /lib/modules/sht3x-generic.ko    # 先加载 → 成功绑定
insmod /lib/modules/sht3x-advanced.ko   # 后加载 → 报 EBUSY
```

`dmesg` 里会看到什么？

```
sht3x-advanced: probe of i2c-1-44 failed with error -16   # -16 = EBUSY
Device i2c-1-44 is already bound to driver sht3x-generic
```

这条日志就是排查竞争问题的关键线索。

---

### 三种解决策略

搞竞争不如避免竞争。来看业界常用的三种解法：

**策略一：细化 compatible（推荐）**

把通用的和特定的 `compatible` 分开，让设备树"精确点名"：

```dts
compatible = "sensirion,sht30-adv", "sensirion,sht30";
```

`"sensirion,sht30-adv"` 专属于 advanced 驱动，`"sensirion,sht30"` 作为 fallback。💡 **提示**：设备树匹配是从左到右优先的，越具体的 `compatible` 放越前面。

**策略二：合并驱动（一个驱动覆盖多个变体）**

与其让两个驱动抢一个设备，不如把逻辑合并到一个驱动里：

```c
static const struct of_device_id sht3x_dt_ids[] = {
    { .compatible = "sensirion,sht30", .data = &sht30_info },
    { .compatible = "sensirion,sht31", .data = &sht31_info },
    { .compatible = "sensirion,sht35", .data = &sht35_info },
    { }
};
```

`of_device_id.data` 字段存每个变体的差异化参数，probe 时通过 `of_device_get_match_data()` 取出。这样一套代码管所有变体，干净利落。

**策略三：用 .data 字段做运行时分支**

如果两个驱动的差异点很小（比如寄存器地址偏移），甚至可以保留同一个 `compatible`，在 `probe` 里通过 `.data` 指针来区分行为：

```c
static int sht3x_probe(struct i2c_client *client)
{
    const struct sht3x_chip_data *chip;
    chip = of_device_get_match_data(&client->dev);
    if (!chip)
        return -EINVAL;

    // 根据 chip->flags 决定采样精度、校准逻辑等
    if (chip->flags & SHT3X_FLAG_HIGH_PRECISION)
        setup_high_precision_mode(dev);
    ...
}
```

### 实战排查

怀疑有竞争？两步确认：

**第一步**：`dmesg | grep -E "already bound|failed with error"`

**第二步**：看 `/sys` 里设备到底绑给了谁：

```bash
# 查看设备当前绑定的驱动
cat /sys/bus/i2c/devices/1-0044/driver/module/drivers/i2c:sht3x-generic
# 输出：/sys/module/sht3x_generic/drivers/i2c:sht3x-generic

# 或者直接看 symbolic link
ls -l /sys/bus/i2c/devices/1-0044/driver
# lrwxrwxrwx ... driver -> ../../../bus/i2c/drivers/sht3x-generic
```

如果发现设备绑错了驱动，可以手动解绑重试：

```bash
echo "1-0044" > /sys/bus/i2c/drivers/sht3x-generic/unbind
echo "1-0044" > /sys/bus/i2c/drivers/sht3x-advanced/bind
```

🔴 **危险**：生产环境中不要随意 `unbind` 正在工作的驱动，可能导致资源泄漏或硬件状态不一致。这个技巧**只用于调试阶段**。

---

### 本节总结

**知识点144 [E]** —— 当多个驱动匹配同一设备时，先注册成功的驱动抢占绑定，后续驱动收到 `EBUSY` 错误。

| 要点 | 说明 |
|------|------|
| 竞争成因 | 多个驱动的 `of_match_table` 包含相同的 `compatible` 字符串 |
| 匹配结果 | 先 `probe` 成功的驱动绑定设备，后续驱动返回 `-EBUSY` |
| 排查日志 | `dmesg` 中搜索 `"already bound"` 或 `error -16` |
| 查看绑定 | `/sys/bus/<bus>/devices/<dev>/driver` 指向当前绑定的驱动 |
| 策略一 | 细化 `compatible`，用 `compatible = "specific", "generic"` 顺序区分 |
| 策略二 | 合并驱动，一个驱动覆盖多个硬件变体 |
| 策略三 | 使用 `of_device_id.data` 字段区分不同硬件的行为差异 |
| 手动切换 | 通过 `unbind`/`bind` 调试，但**生产环境慎用** |

### 下一步

匹配竞争的本质是驱动间的"资源抢占"。但在实际工程中，**单个驱动内部的资源管理**同样容易踩坑——I2C 地址冲突、中断号重复申请、GPIO 被重复导出，这些都是日常调试的常客。下一节我们将把目光从驱动间转向驱动内部，看看 `probe()` 函数里的资源申请应该如何设计才能健壮可靠。
