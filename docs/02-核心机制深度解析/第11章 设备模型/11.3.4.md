# 11.3.4 匹配竞争与优先级

> 所属章节：第11章 设备驱动模型深度探究 > 11.3 驱动与设备的匹配过程
> 难度：[E] | 预计阅读时间：12分钟

## 本节导读
本节聊一个实际做驱动时很容易踩到的坑——多个驱动都能匹配同一个设备，到底谁胜出？这不是理论问题，是真的会在线上环境搞出"绑错了driver，硬件工作异常"的诡异bug。学完后，你能搞清楚平台总线上的匹配竞争规则，并掌握三种常用的规避和解决策略。

## 知识点152：当两个驱动"看上"同一个设备时，谁赢？ [E] ~800字

### 先来一个真实的"翻车"现场

假设你的DTS里有个设备节点长这样：

```dts
my_device: my_device@43c00000 {
    compatible = "myvendor,my-device";
    reg = <0x43c00000 0x1000>;
};
```

内核里有两个驱动都声称自己能伺候这个设备：

- **驱动A**（老版本，通用型）：`compatible = "myvendor,my-device"`
- **驱动B**（新版本，针对V2芯片）：`compatible = "myvendor,my-device", "myvendor,my-device-v2"`

系统启动时，驱动A先注册，顺利绑上了。驱动B后注册，内核在dmesg里吐一句：

```
[    2.345678] my_device my_device.0: Device my_device.0 is already bound to driver my_driver_a
```

然后驱动B就默默走开了。**问题**：你的板子其实是V2芯片，驱动B才能发挥完整功能，结果却被驱动A"截胡"了。硬件只跑在兼容模式，性能打折，甚至有的寄存器操作不正确，时不时来个超时错误。

这就是典型的**驱动匹配竞争**——多个驱动的`of_device_id`表都能hit到同一个`compatible`字符串，但内核只让一个驱动绑定设备。

### 平台总线的匹配优先级规则

Platform bus的匹配逻辑说复杂也复杂，说简单就两条：

```
1. 匹配精度优先：exact match > wildcard match
2. 同精度下：先注册者胜出（first-come-first-served）
```

第一条"匹配精度"啥意思？看这张表就明白了：

| 匹配类型 | 说明 | 优先级 |
|---------|------|--------|
| 特定设备名 | `platform_device`名字和`platform_driver`的`id_table`精确匹配 | 最高 |
| OF完整匹配 | `compatible`字符串一对一精确命中 | 高 |
| OF通配匹配 | 驱动用了`"vendor,dev-*"`这种通配（实际of_match_table不支持真通配，是列表覆盖） | 中 |
| ACPI匹配 | 通过`_HID`/`_CID`匹配 | 中 |
| 兜底name匹配 | 老的基于设备名的匹配 | 低 |

**关键点**：Platform驱动的`of_match_table`里如果多个驱动的`compatible`条目有重叠，比如驱动A写了`"myvendor,my-device"`，驱动B写了`"myvendor,my-device-v2"`但同时为了兼容也写了`"myvendor,my-device"`，那它们对DTS里的`"myvendor,my-device"`就是**同精度竞争**——谁`platform_driver_register()`先执行，谁拿到设备。

### 排查诊断：到底是谁绑了我的设备？

遇到怀疑竞争绑定的情况，按下面步骤查：

```bash
# 代码1：查看设备当前绑定的驱动
$ cat /sys/bus/platform/devices/my_device.0/driver/driver.name
my_driver_a

# 或者直接看符号链接
$ ls -l /sys/bus/platform/devices/my_device.0/driver
... /sys/bus/platform/devices/my_device.0/driver -> ../../../../bus/platform/drivers/my_driver_a
```

再看dmesg确认竞争发生：

```bash
# 代码2：搜索设备绑定相关的内核日志
$ dmesg | grep -i "my_device\|already bound"
[    2.345678] my_device my_device.0: Device my_device.0 is already bound to driver my_driver_a
```

如果想看完整的probe过程，可以开动态debug：

```bash
# 代码3：开platform驱动动态调试
$ echo 'file drivers/base/platform.c +p' > /sys/kernel/debug/dynamic_debug/control
$ dmesg -C && reboot
```

⚠️ **陷阱**：很多工程师看到硬件工作"基本正常"就不追究了，没发现绑的是通用驱动而非优化驱动。性能测试时数据不好看，排查半天才发现驱动绑定错了，浪费时间。

🔴 **危险**：`compatible`字符串不要写得太宽泛。比如不要写一个驱动匹配`"arm,pl011"`又想覆盖所有UART，结果把人家专门优化过的串口驱动挤掉了。

## 知识点153：解决匹配竞争的三板斧 [E] ~600字

### 策略一：从DTS侧细化compatible

最直接的办法——DTS里把设备描述得更精确：

```dts
// 修改前：太笼统
compatible = "myvendor,my-device";

// 修改后：精确到芯片版本
compatible = "myvendor,my-device-v2", "myvendor,my-device";
```

内核匹配会从左到右遍历`compatible`数组，优先找能匹配`"myvendor,my-device-v2"`的驱动。驱动B注册了这个精确条目，就能优先绑定，不再和驱动A抢。

DTS的设计哲学就是"从特定到通用"——第一个字符串尽可能具体，后面的放兼容值做fallback。这样即使内核里没有专用驱动，也能回退到通用驱动把系统跑起来。

### 策略二：合并驱动，用.data字段区分变体

如果两个驱动80%代码雷同，只是寄存器地址差几个byte、时序参数不同，那不如合并成一个驱动，靠`of_device_id`的`.data`指针区分硬件变体：

```c
static const struct of_device_id my_driver_of_match[] = {
    { .compatible = "myvendor,my-device",    .data = &my_device_v1_info },
    { .compatible = "myvendor,my-device-v2", .data = &my_device_v2_info },
    { /* sentinel */ }
};
MODULE_DEVICE_TABLE(of, my_driver_of_match);
```

`probe()`里这样取：

```c
static int my_driver_probe(struct platform_device *pdev)
{
    const struct of_device_id *match = of_match_device(my_driver_of_match, &pdev->dev);
    const struct my_device_variant *variant = match->data;

    dev_info(&pdev->dev, "Detected %s\n", variant->name);
    // variant->fifo_depth, variant->irq_handler 等按版本配置
}
```

`.data`指向的结构体可以塞寄存器偏移、时序参数、回调函数指针。一个驱动cover多个硬件版本，匹配竞争自然消失，维护起来也更省心。

| 策略 | 适用场景 | 改动范围 | 维护成本 |
|------|---------|---------|---------|
| 细化compatible | 驱动分离，硬件版本明确 | DTS + 驱动of_match_table | 低 |
| 合并驱动+.data | 变体差异小，代码重复高 | 驱动代码重构 | 中（省一个驱动文件） |

💡 **提示**：实际项目中两种策略经常组合用——DTS里写精确的`compatible`，驱动里用`.data`做差异化处理。这是内核社区的推荐做法，clean又robust。选策略的核心判断标准很简单：如果两个驱动代码差异超过30%，就保持分离，靠细化compatible避免竞争；如果差异很小，果断合并，用`.data`做运行时区分。

## 本节总结

| 概念 | 核心要点 | 自查操作 |
|------|---------|---------|
| 匹配竞争 | 多个驱动hit同一compatible时，先注册者胜出 | `dmesg \| grep "already bound"` |
| 匹配精度 | exact match优先级高于通配/兜底匹配 | 检查驱动`of_match_table`的条目设计 |
| 排查绑定关系 | 确认设备实际绑了哪个driver | `cat /sys/bus/platform/devices/<dev>/driver/driver.name` |
| 解决策略 | 细化compatible、合并驱动、.data字段区分 | 检查DTS的compatible精度和驱动的variant处理 |
