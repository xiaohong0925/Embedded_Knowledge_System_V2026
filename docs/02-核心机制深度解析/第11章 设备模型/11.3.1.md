# 11.3.1 compatible匹配原理

> 所属章节：第11章 设备树深入 > 11.3 compatible与OF匹配机制
> 难度：[E→M] | 预计阅读时间：12分钟

## 本节导读
这节我们把"compatible怎么干活"这件事彻底拆开来看。很多人写设备树的时候照葫芦画瓢填了个`compatible = "xxx"`，驱动里也写了个`of_match_table`，但它俩到底是怎么对上的？内核是逐个字符比对的还是有啥技巧？匹配失败了又会怎样？这两个知识点讲完后，你能拿着代码跟人说清楚从设备树节点到驱动probe的完整匹配链路。

---

## 知识点146：compatible属性与of_match_table的"相亲"过程 [E][M] ~1000字

### compatible本质上是个"候选名单"

先看一个真实的设备树节点：

```dts
// 代码1：某I2C温度传感器节点
temp_sensor@48 {
    compatible = "ti,tmp102", "ti,tmp101";
    reg = <0x48>;
};
```

注意`compatible`是个**字符串列表**，从左到右是"从具体到通用"的优先级。`"ti,tmp102"`最具体——就是这个型号；`"ti,tmp101"`更通用——tmp102可能兼容tmp101的寄存器接口。

这就像你去相亲，先约最心仪的那个，不行再约备胎，顺序很重要。

### 驱动里的of_match_table是另一份名单

驱动代码长这样：

```c
// 代码2：tmp102驱动的of_match_table
static const struct of_device_id tmp102_of_match[] = {
    { .compatible = "ti,tmp102" },
    { .compatible = "ti,tmp101" },
    { .compatible = "ti,tmp100" },
    { }, /* 最后一个必须是空条目，表示结束 */
};
MODULE_DEVICE_TABLE(of, tmp102_of_match);

static struct i2c_driver tmp102_driver = {
    .driver = {
        .name = "tmp102",
        .of_match_table = tmp102_of_match,
    },
    .probe = tmp102_probe,
};
```

驱动把自己的"我能伺候哪些设备"登记在`of_match_table`里。注意这里也是以兼容程度排列的，最具体的放前面。

### __of_match_node()：内核的"红娘"

真正的匹配发生在`__of_match_node()`函数里（`drivers/of/base.c`）。别被下划线开头的名字吓到，它的逻辑非常直白：

```c
// 代码3：__of_match_node()核心逻辑（简化版）
const struct of_device_id *__of_match_node(
    const struct of_device_id *matches,
    const struct device_node *node)
{
    // 遍历驱动of_match_table中的每一个条目
    for (; matches->name[0] || matches->type[0] ||
           matches->compatible[0]; matches++) {
        // 检查compatible字段是否匹配
        if (matches->compatible[0]) {
            // of_device_is_compatible()内部：
            // 遍历设备节点compatible属性的每一个字符串
            // 逐个与matches->compatible进行strcmp比较
            if (of_device_is_compatible(node, matches->compatible))
                return matches;  // 匹配成功，返回这个条目
        }
    }
    return NULL;  // 全都试过了，没一个对上
}
```

看明白了吗？**双重循环**：外层遍历驱动的`of_match_table`，内层遍历设备节点的`compatible`字符串列表。驱动拿出一个名字，设备节点把名单上所有人叫出来挨个比对，对上了就返回成功。

这里有个微妙的细节：`of_device_is_compatible()`在比较时，不只是简单的`strcmp`——它还支持通配符（`*`）和空字符串匹配，后面知识点147会细说。

### 完整匹配流程：从设备树到驱动probe


[图1：compatible匹配完整流程]

流程中的黄色步骤就是`__of_match_node()`的核心——它像电话本查询一样朴素：你报一个名字，我翻一页页查。没有哈希、没有缓存，就是字符串比对。为什么这样设计？因为设备树节点数量通常不多（几十到几百个），这种O(n×m)的复杂度完全可以接受。

⚠️ **陷阱**：很多人以为只要compatible字符串"看起来一样"就能匹配上，但经常踩这些坑：
- 大小写不对：`"TI,tmp102"`和`"ti,tmp102"`在内核眼里是两回事，strcmp严格区分大小写
- 多了空格：`"ti,tmp102 "`（尾部空格）也会导致匹配失败
- 用了短横线和下划线混用：`"ti,tmp-102"` vs `"ti,tmp_102"`，这是不同的厂商前缀规范问题

💡 **提示**：调试匹配问题时，打开内核配置`CONFIG_OF_DEBUG=y`，然后在启动参数里加`of_debug=1`，内核会打印每个设备节点的匹配过程，帮你定位是驱动没注册还是compatible写错了。

## 知识点147：精确匹配与通配符——顺序就是一切 [E] ~600字

### 通配符匹配：内核留的"模糊查找"后门

`of_device_is_compatible()`除了精确字符串比较，还支持一种通配符写法：

```c
// 代码4：通配符匹配示例
static const struct of_device_id generic_i2c_sensor_match[] = {
    { .compatible = "vendor,specific-model" },   // 先精确匹配
    { .compatible = "generic-i2c-sensor" },      // 再通用匹配
    { .compatible = "*" },                       // 通配符：匹配任何compatible！
    { },
};
```

`.compatible = "*"`就像正则表达式里的`.*`，啥都能匹配上。但这个**极其危险**——如果你把它放在`of_match_table`的第一位，那不管什么设备树节点都会匹配到这个驱动，后面的具体条目永远轮不到。

### 为什么顺序如此重要？

先看一个反面教材：

```dts
// 代码5：顺序写错的设备树（错误示范）
my_device {
    // 反了！应该是 "mychip,mydev-v2" 在前
    compatible = "generic-device", "mychip,mydev-v2";
};
```

这样的后果是：内核匹配时先找到`"generic-device"`对应的通用驱动，即使系统里明明有`"mychip,mydev-v2"`的专用驱动，它永远用不上——因为通用驱动先"抢"到了设备。

正确的写法：

```dts
// 代码6：正确的compatible顺序
my_device {
    compatible = "mychip,mydev-v2", "mychip,mydev", "generic-device";
};
```

这个规则叫**"most specific first"**（最具体优先）。设备树规范和OF匹配机制都要求遵循这个约定，不是随便排的。

### 一个真实的血泪教训

某次项目中，硬件从v1升级到v2，寄存器布局变了，驱动需要区分。工程师在设备树里改成：

```dts
compatible = "mychip,mydev";
```

但忘了v2板子应该用`"mychip,mydev-v2"`。结果旧驱动匹配上了，按v1的寄存器 offsets 去读写，导致读出来的温度数据完全错误，调试了两天才发现是compatible字符串少了`-v2`后缀。

内核的匹配机制不会帮你"智能选择"最合适的驱动——它就是按顺序比对，第一个对上的就用。这个设计简单、可预测，但也意味着**你在设备树和驱动里写的字符串必须严格对应**。

🔴 **危险**：修改设备树的compatible字符串后，一定要同步检查驱动的`of_match_table`，并确保二者的顺序逻辑一致。单侧修改是嵌入式开发里最常见的self-inflicted wound（自找的伤）。

## 本节总结

| 概念 | 核心要点 | 自查方法 |
|------|---------|---------|
| compatible列表 | 设备树中是从具体到通用的字符串数组 | `cat /proc/device-tree/.../compatible` |
| of_match_table | 驱动声明支持的设备列表，顺序同样重要 | 查看驱动源码中的`static const struct of_device_id` |
| __of_match_node | 双重循环比对：驱动条目 × 设备节点compatible字符串 | 开`CONFIG_OF_DEBUG`看匹配日志 |
| 通配符 | `"*"`可匹配任意字符串，但切勿滥用 | 检查of_match_table是否放末尾 |
| 匹配失败 | 设备无法绑定驱动，不会自动panic，但功能不可用 | `ls /sys/bus/platform/devices/`看有无driver符号链接 |
