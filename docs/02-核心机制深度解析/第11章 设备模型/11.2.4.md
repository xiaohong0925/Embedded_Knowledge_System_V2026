# 11.2.4 devm_*资源管理

> 所属章节：第11章 驱动开发进阶 > 11.2 资源与中断管理
> 难度：[E] | 预计阅读时间：12分钟

## 本节导读
写驱动最怕什么？不是逻辑复杂，而是资源泄漏——`ioremap`了忘记`iounmap`，申请了IRQ忘记`free_irq`，`kzalloc`了忘记`kfree`。设备热插拔几次，系统内存就被啃光了。本节介绍内核提供的"自动回收"机制 devm_*，让你写驱动时少操一半的心，同时也告诉你它不能帮你兜哪些底。

---

## 知识点142：devm_*的本质——给资源绑个"自动销毁器" [E] ~800字

### devm 到底是什么来头

`devm_kzalloc()`、`devm_ioremap_resource()`、`devm_clk_get()`……这些以 `devm_` 开头的函数，背后是一套叫 **device resource management（devres）** 的子系统。它的核心思想很简单：**把资源和 device 结构体绑在一起，设备走了，资源自动释放**。

内核里每个 `struct device` 内部都藏着一个链表头 `devres_head`。当你调用 `devm_xxx()` 申请资源时，内核不仅把资源给你分配好，还会往这个链表上挂一个节点。这个节点里存着两样东西：一是资源的指针，二是一个释放回调函数。等到设备被卸载（`probe` 的反过程，通常是 `remove` 被调用或者 `probe` 失败后的清理），内核会遍历这条 devres 链表，逐个调用释放回调，就像退房时酒店自动帮你清空房间一样。

```
struct device {
    ...
    struct list_head devres_head;  // devres 链表头
    ...
};
```

### 常用 devm 函数对照表

下面这张表列了驱动开发中最常用的 devm 函数，以及它们对应的"传统手动释放"版本。看完你会发现规律：`devm_` 前缀一加，后半截名字基本不变。

| devm 版本（自动释放） | 传统版本（手动释放） | 用途说明 |
|----------------------|---------------------|---------|
| `devm_kzalloc()` | `kzalloc()` + `kfree()` | 设备绑定的内存分配 |
| `devm_ioremap_resource()` | `request_mem_region()` + `ioremap()` + `iounmap()` | IO 内存映射 |
| `devm_request_irq()` | `request_irq()` + `free_irq()` | 申请中断 |
| `devm_clk_get()` | `clk_get()` + `clk_put()` | 获取时钟 |
| `devm_gpio_request()` | `gpio_request()` + `gpio_free()` | 申请 GPIO |
| `devm_regulator_get()` | `regulator_get()` + `regulator_put()` | 获取电源稳压器 |
| `devm_dma_alloc_coherent()` | `dma_alloc_coherent()` + `dma_free_coherent()` | 一致性 DMA 内存 |

```c
// 代码1：典型的 devm 风格 probe（简洁很多）
static int my_probe(struct platform_device *pdev)
{
    void __iomem *base;
    struct clk *clk;
    int irq;

    base = devm_ioremap_resource(&pdev->dev,
            platform_get_resource(pdev, IORESOURCE_MEM, 0));
    if (IS_ERR(base))
        return PTR_ERR(base);

    irq = platform_get_irq(pdev, 0);
    if (irq < 0)
        return irq;

    clk = devm_clk_get(&pdev->dev, NULL);
    if (IS_ERR(clk))
        return PTR_ERR(clk);

    return devm_request_irq(&pdev->dev, irq, my_handler,
                            0, "my_drv", dev);
}
```

注意看上面的代码——整个 `probe` 函数里 **没有一行释放代码**。如果是传统写法，每一步出错都得 `goto` 到对应的清理标签，代码膨胀一倍不止。

### 优势和局限要心里有数

devm 最大的好处是**代码简洁、不怕漏**。驱动作者不用再维护冗长的错误处理路径，也不用担心 `remove` 函数里漏了哪一步。内核维护者做过统计，用 devm 重构后的驱动平均能减少 30%~50% 的资源管理代码。

但 devm 不是银弹，它有两个硬限制：

🔴 **局限一：只绑定 struct device**。如果你的资源生命周期和 device 不一致——比如一个 DMA buffer 要在设备 A 和设备 B 之间共享，或者你需要在 `remove` 之后还保留一段内存——那就不能用 devm。devm 的申请第一个参数必须是 `struct device *dev`，这本身就是强约束。

🔴 **局限二：不是零成本抽象**。每个 devm 资源都要在 devres 链表上占一个节点，申请和释放时多一次链表操作。对于极端性能敏感的场景（比如每秒创建销毁上千次设备的测试代码），手动管理反而更快。当然，对 99% 的硬件驱动来说，这点开销完全可以忽略。

⚠️ **陷阱**：devm 分配的内存**不能在中断上下文里释放**。因为 devres 的清理是在进程上下文遍历链表时完成的，如果你的中断处理函数依赖某种"立即释放"的语义，devm 帮不了你。

## 知识点143：传统写法 vs devm 写法——同一个驱动的两面 [E] ~600字

### 先看传统写法——手动挡的苦涩

下面是一个简化版 GPIO 按键驱动的传统 `probe`，感受一下手动管理资源的酸爽：

```c
// 代码2：传统手动管理方式（麻烦但可控）
static int gpio_key_probe(struct platform_device *pdev)
{
    struct gpio_key *key;
    int irq, ret;

    key = kzalloc(sizeof(*key), GFP_KERNEL);
    if (!key)
        return -ENOMEM;

    key->gpio = devm_gpio_request(&pdev->dev, 15, "key");  // 哦豁，混用了
    // 其实这里已经矛盾了——如果后面出错，gpio 释放了吗？
    irq = gpio_to_irq(15);
    ret = request_irq(irq, key_handler, IRQF_TRIGGER_FALLING,
                      "gpio_key", key);
    if (ret) {
        kfree(key);          // 注意：irq 申请成功前出错，不用 free_irq
        return ret;
    }

    ret = clk_prepare_enable(key->clk);  // 又忘检查返回值了？
    if (ret) {
        free_irq(irq, key);
        kfree(key);
        return ret;
    }
    return 0;
}
```

发现问题了吗？传统写法里，**每增加一个资源，错误处理路径就要翻倍**。四个资源就有四个 `goto` 标签或者四层嵌套 `if`。更恶心的是，如果你中途想加一个 `regulator_get`，得把后面的所有错误处理都改一遍——漏一个就是泄漏。

### devm 写法——自动挡的爽感

```c
// 代码3：devm 方式（错误路径几乎消失）
static int gpio_key_probe(struct platform_device *pdev)
{
    struct gpio_key *key;
    int irq;

    key = devm_kzalloc(&pdev->dev, sizeof(*key), GFP_KERNEL);
    if (!key)
        return -ENOMEM;

    irq = devm_request_irq(&pdev->dev, gpio_to_irq(15), key_handler,
                           IRQF_TRIGGER_FALLING, "gpio_key", key);
    if (irq < 0)
        return irq;

    key->clk = devm_clk_get(&pdev->dev, NULL);
    if (IS_ERR(key->clk))
        return PTR_ERR(key->clk);

    return clk_prepare_enable(key->clk);  // clk 仍须手动 enable/disable
}
```

对比很明显：devm 版本没有 `goto`，没有层层嵌套的清理代码。但注意一个细节——`clk_prepare_enable()` **没有 devm 版本**。因为时钟的 enable/disable 是运行时操作，不是一次性的资源申请，devm 管不到这一步。你依然要在 `remove` 里手动 `clk_disable_unprepare()`。

### 关键认知：probe 失败前的资源谁来管？

这里有个很多新手踩过的坑：**`devm_xxx()` 在 `probe` 函数返回错误之前分配的资源，是不会自动释放的**。

什么意思？假设你的 `probe` 里连续调了三个 devm 函数，前两个成功了，第三个失败了，你直接 `return -EINVAL`。此时前两个 devm 资源**已经挂在 devres 链表上了**，但由于 `probe` 失败，设备并没有真正"注册成功"，内核会把整个 device 结构体清理掉——连带 devres 链表一起释放。所以实际上它们是会被释放的。

但！如果你**在 `probe` 成功之后又临时申请了一些非 devm 资源**，然后发现有问题想回滚——这时候 devm 不会帮你。因为 device 已经存活了，devres 的自动清理要等到 `remove` 才会触发。

💡 **提示**：最安全的策略是——`probe` 里只用 devm 函数，且所有 devm 调用在 `probe` 返回 0 之前完成。如果 `probe` 返回错误，内核会自动清理已挂接的 devres；如果 `probe` 成功，这些资源会陪伴设备直到 `remove`。

⚠️ **陷阱**：`devm_kzalloc()` 分配的内存虽然会在设备卸载时自动释放，但如果你在中途需要**提前释放**（比如某个缓冲区用完就想归还），devm 不支持这种操作。它只能"全有或全无"地跟着设备走。

## 本节总结

| 概念 | 核心要点 | 自查方法 |
|------|---------|---------|
| devres 机制 | 每个 device 维护一个资源链表，卸载时自动调用释放回调 | 查看 `struct device` 的 `devres_head` 字段 |
| devm_ 命名规律 | 传统函数名前加 `devm_`，第一个参数多一个 `struct device *` | 对比 `kzalloc()` 和 `devm_kzalloc()` 的原型 |
| 核心优势 | 消灭冗长的错误处理路径，防止资源泄漏 | 数一下你的 probe 里还有没有 `goto err_xxx` |
| 硬限制 | 资源生命周期必须与 device 一致；不支持提前释放 | 问自己：这个资源要不要在 remove 之后继续存活？ |
| probe 失败处理 | probe 返回错误前，已分配的 devm 资源会被内核自动清理 | 模拟一次中途返回 `-ENODEV`，检查有无泄漏 |
