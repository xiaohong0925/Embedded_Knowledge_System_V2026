### 11.2.4 devm_* 资源管理

**知识点 140 [E]**

**本节导读：** 前面讲到的 `ioremap()`、`request_irq()`、`clk_get()`……这些 API 都有一个共同的痛点——**你申请了资源，就必须在某个地方释放它**。一旦遗漏，就是内存泄漏、中断无法重新注册、时钟引用计数不准。Linux 内核从 2.6 后期引入了一套以 `devm_` 为前缀的"设备托管资源"API，让驱动代码在资源管理这件事上彻底告别"手动记账"。本节我们来理解 devm 的本质，看它如何让驱动变得更简洁、更不容易出错。

---

#### devm 的本质：把释放逻辑"挂"到设备上

`devm_*` 的全称是 **Device Resource Management**。它的设计思路非常直观：

> 资源是你为某个设备申请的，那么当这个设备消失时，这些资源理应被自动回收。

内核在 `struct device` 内部维护了一条 **devres 链表**。当你调用 `devm_kzalloc()`、`devm_ioremap_resource()` 等函数时，内核不仅完成资源申请，还会往这条链表上挂一个**释放回调**。等设备被卸载（probe 失败、`driver unbind`、模块移除）时，内核会遍历这条链表，逐个调用释放函数。

用图来理解它的结构：

```
┌─────────────────┐
│ struct device   │
│  ···            │
│  devres_head ───┼──► ┌──────────┐   ┌──────────┐   ┌──────────┐
│  ···            │    │ devres A │──►│ devres B │──►│ devres C │──► NULL
└─────────────────┘    │(kfree)   │   │(iounmap) │   │(free_irq)│
                       └──────────┘   └──────────┘   └──────────┘
```

说白了，devm 帮你做了一件每个驱动工程师都想做的事：**把"申请-释放"配对起来，且让释放时机由框架把控。**

#### 常见的 devm 函数

以下是驱动开发中最常用的 devm 函数及其对应关系：

| devm 函数 | 传统对应函数 | 自动释放动作 |
|---|---|---|
| `devm_kzalloc()` | `kzalloc()` | `kfree()` |
| `devm_ioremap_resource()` | `ioremap()` / `request_mem_region()` | `iounmap()` / `release_mem_region()` |
| `devm_clk_get()` | `clk_get()` | `clk_put()` |
| `devm_request_irq()` | `request_irq()` | `free_irq()` |
| `devm_platform_ioremap_resource()` | `platform_get_resource()` + `ioremap()` | `iounmap()` |
| `devm_gpio_request()` | `gpio_request()` | `gpio_free()` |
| `devm_regulator_get()` | `regulator_get()` | `regulator_put()` |

💡 **提示**：`devm_ioremap_resource()` 是推荐的 IO 内存映射方式，它在内部同时完成了 `request_mem_region()` 和 `ioremap()` 两个步骤，资源释放时也会对应做两步清理。如果你还在用裸的 `ioremap()`，建议逐步迁移过来。

#### 代码对比：同一驱动的两种写法

来看看传统方式和 devm 方式的真实差异。假设我们要写一个最简化的 platform 驱动，需要分配私有结构体、映射寄存器、申请中断：

**传统方式——每个错误路径都要手动"倒着"释放：**

```c
static int my_drv_probe(struct platform_device *pdev)
{
    struct my_dev *priv;
    void __iomem *base;
    int irq, ret;

    priv = kzalloc(sizeof(*priv), GFP_KERNEL);
    if (!priv)
        return -ENOMEM;

    base = ioremap(MY_BASE, MY_SIZE);
    if (!base) {
        ret = -ENOMEM;
        goto err_free_priv;
    }

    irq = platform_get_irq(pdev, 0);
    ret = request_irq(irq, my_isr, 0, "my_drv", priv);
    if (ret)
        goto err_iounmap;

    priv->base = base;
    priv->irq  = irq;
    platform_set_drvdata(pdev, priv);
    return 0;

err_iounmap:
    iounmap(base);
err_free_priv:
    kfree(priv);
    return ret;
}

static int my_drv_remove(struct platform_device *pdev)
{
    struct my_dev *priv = platform_get_drvdata(pdev);

    free_irq(priv->irq, priv);
    iounmap(priv->base);
    kfree(priv);
    return 0;
}
```

⚠️ **陷阱**：注意 `err_*` 标签的维护成本。如果后续又增加了一个 `devm_clk_get` 对应的 `clk_get`，你需要在中间插入一个新的错误跳转标签，并且前面的所有路径都要考虑——这个"倒着释放"的逻辑一旦多了，漏掉一个就是泄漏。

**devm 方式——错误路径不需要任何手动释放：**

```c
static int my_drv_probe(struct platform_device *pdev)
{
    struct my_dev *priv;
    int irq;

    priv = devm_kzalloc(&pdev->dev, sizeof(*priv), GFP_KERNEL);
    if (!priv)
        return -ENOMEM;

    priv->base = devm_platform_ioremap_resource(pdev, 0);
    if (IS_ERR(priv->base))
        return PTR_ERR(priv->base);

    irq = platform_get_irq(pdev, 0);
    if (irq < 0)
        return irq;

    if (devm_request_irq(&pdev->dev, irq, my_isr, 0, "my_drv", priv))
        return -EINVAL;

    priv->irq = irq;
    platform_set_drvdata(pdev, priv);
    return 0;
}

static int my_drv_remove(struct platform_device *pdev)
{
    /* devm 资源自动释放，remove 里常常什么都不用做 */
    return 0;
}
```

对比一下：devm 版本的 `remove` 函数里啥也不用干。probe 出错时，之前已经申请成功的 devm 资源会被内核自动清理。**代码行数少了，心智负担也轻了。**

#### devm 的局限：不是所有场景都适用

说实话，devm 虽好，但它并非银弹。有两个重要的限制你需要清楚：

**第一，只在 probe 成功"之前"有效。** 内核的 devres 释放时机是 `device_release()`，也就是设备与驱动解绑的时候。如果你的驱动在 probe 成功后的某个时刻（比如运行时 `open()` 里）用 `devm_kzalloc()` 分配了内存——那这块内存要到设备卸载时才释放，而不是 `close()` 时。这可能导致资源被不必要地长期占用。

**第二，不适合需要精细控制释放时机的场景。** 比如某些驱动在 `remove()` 里需要先关闭中断、再刷新队列、最后释放内存——有严格的先后顺序。devm 的释放顺序是按链表遍历的，你无法精细控制。

🔴 **危险**：如果你需要在 `remove()` 中做特定的关闭序列（比如先通知固件、再释放 DMA、最后断电），不要把这些步骤交给 devm 自动处理。手动控制顺序才是安全的做法。

#### 本节总结

| 要点 | 内容 |
|---|---|
| devm 的本质 | 将资源申请与设备生命周期绑定，设备卸载时自动释放 |
| 核心机制 | devres 链表 + 释放回调，挂在 `struct device` 上 |
| 常用 API | `devm_kzalloc()`、`devm_ioremap_resource()`、`devm_clk_get()`、`devm_request_irq()` 等 |
| 最大优势 | 省去错误路径的手动释放逻辑，`remove()` 可大幅简化 |
| 关键局限 | ① 只适用于 probe 阶段的资源；② 无法精细控制释放顺序 |
| 使用原则 | probe 里的"一次性"资源优先用 devm；需要时序控制的仍手动管理 |
