# 11.3.2 of_match_table的两种组织

> 所属章节：第11章 设备驱动开发实战 > 11.3 platform设备驱动
> 难度：[E] | 预计阅读时间：12分钟

## 本节导读
上一节你知道了probe函数的触发机制，但还有一个关键环节没讲——驱动怎么告诉内核"我支持哪些设备"？本节介绍`of_match_table`的两种写法：最传统的`of_device_id`数组，以及需要兼容ACPI时的双表并存方案。看完你会明白为什么很多驱动里明明有匹配表，modprobe却死活不自动加载——八成是漏了MODULE_DEVICE_TABLE。

---

## 知识点148：of_device_id数组——最传统、最常见的写法 [E] ~800字

### 最朴素的匹配表

先直接看代码，这是你在Linux内核源码里见得最多的一种写法：

```c
static const struct of_device_id mydrv_match[] = {
    { .compatible = "myvendor,mydevice" },
    { .compatible = "myvendor,mydevice-v2" },
    {}, /* 最后一行必须是空条目，表示结束 */
};
MODULE_DEVICE_TABLE(of, mydrv_match);

static struct platform_driver my_driver = {
    .driver = {
        .name = "my_driver",
        .of_match_table = mydrv_match,
    },
    .probe = my_probe,
    .remove = my_remove,
};
```

就这么简单。`of_device_id`数组里每一行填一个`.compatible`字符串，跟你设备树节点里的`compatible = "myvendor,mydevice"`对上号，匹配就算成立。最后一个空`{}`不能省，内核遍历时靠它来终止循环，忘了会踩内存越界。

### MODULE_DEVICE_TABLE(of, mydrv_match) 到底起什么作用？

这里很多人写了几十次都没真正理解。这行宏的作用分两个层面：

**编译层面**：它把`mydrv_match`这个数组标记出来，打包进驱动模块的一个特殊段（`__mod_of_device_table`）。编译后可以用`modinfo`或`objdump`看到这段数据。

**运行时层面**：当你把驱动编译成模块（`.ko`）并放到`/lib/modules/$(uname -r)/`下面，`depmod`工具会扫描所有模块的`__mod_of_device_table`段，生成`modules.alias`文件。这个文件里会有一行：

```
alias of:N*T*Cmyvendor,mydevice* my_driver
```

这就是`modprobe`能自动加载驱动的关键。udev在检测到新设备、读出设备树的compatible字符串后，会去`modules.alias`里查匹配项，找到对应的模块名，再调用`modprobe`加载。没有`MODULE_DEVICE_TABLE`，你的驱动就算`.of_match_table`写得再完美，也只是一个"隐身"的驱动——手动`insmod`能工作，但系统永远不知道怎么自动找到它。

### 多个compatible怎么处理？

实际项目中经常遇到一个硬件有多代版本，设备树节点长这样：

```dts
mydevice: mydevice@1000000 {
    compatible = "myvendor,mydevice-v2", "myvendor,mydevice";
};
```

注意顺序：从前到后是从具体到通用。驱动里的匹配表通常会把所有历史版本都列上，保证新老设备树都能匹配上：

```c
static const struct of_device_id mydrv_match[] = {
    { .compatible = "myvendor,mydevice-v2" },  // 新版型
    { .compatible = "myvendor,mydevice" },      // 旧版型，兜底
    {},
};
```

匹配时内核会按数组顺序逐个比对，所以如果你想优先用新版本的初始化逻辑，把v2放前面更稳妥。

### 一个容易踩的坑：of_match_ptr() 宏

有些驱动代码里你会看到 `.of_match_table = of_match_ptr(mydrv_match)`，这个宏的定义长这样：

```c
#ifdef CONFIG_OF
#define of_match_ptr(_ptr)  (_ptr)
#else
#define of_match_ptr(_ptr)  NULL
#endif
```

它的作用很简单：如果内核编译时没开CONFIG_OF（设备树支持），整个匹配表就填NULL。这在x86等不依赖设备树的平台上能避免编译错误，但写代码的时候别被它搞糊涂了——本质上还是填的`mydrv_match`。

## 知识点149：ACPI双表并存——跑在x86上的驱动怎么写 [E] ~600字

### 一份驱动，两套匹配表

如果你的驱动既要跑在ARM（设备树）上，又要跑在x86（ACPI）上，`of_match_table`就不够用了。这时候需要同时提供`acpi_match_table`：

```c
static const struct of_device_id mydrv_of_match[] = {
    { .compatible = "myvendor,mydevice" },
    {},
};
MODULE_DEVICE_TABLE(of, mydrv_of_match);

static const struct acpi_device_id mydrv_acpi_match[] = {
    { "MYV0001", 0 },   /* ACPI _HID值 */
    { },
};
MODULE_DEVICE_TABLE(acpi, mydrv_acpi_match);

static struct platform_driver my_driver = {
    .driver = {
        .name = "my_driver",
        .of_match_table = mydrv_of_match,
        .acpi_match_table = ACPI_PTR(mydrv_acpi_match),
    },
    .probe = my_probe,
};
```

对比一下两种表的区别：

| 对比项 | of_match_table（设备树） | acpi_match_table（ACPI） |
|-------|------------------------|-------------------------|
| 匹配依据 | compatible字符串 | ACPI _HID/_CID |
| 数据结构 | `struct of_device_id` | `struct acpi_device_id` |
| MODULE_DEVICE_TABLE | `MODULE_DEVICE_TABLE(of, ...)` | `MODULE_DEVICE_TABLE(acpi, ...)` |
| 平台 | ARM/RISC-V等嵌入式 | x86 PC/服务器 |

`ACPI_PTR`宏和`of_match_ptr`是一个思路：没开ACPI编译选项时就填NULL。

### 驱动注册时的实际行为

`platform_driver_register()`内部会依次检查：

1. 如果当前平台启用了OF（设备树）且设备节点有compatible字段 → 走`of_match_table`
2. 否则如果平台启用了ACPI且设备有_HID/_CID → 走`acpi_match_table`
3. 否则fallback到`.name`字符串匹配（传统的platform_device_register方式）

所以对于跨平台驱动来说，这两张表不是"二选一"的关系，而是各管各的井水不犯河水。内核在运行时只会匹配当前平台生效的那张表。

### 血泪教训：漏写MODULE_DEVICE_TABLE的后果

最后再说一个调试时经常遇到的抓狂问题。你编译了驱动，推到板子上，设备树也改好了，reboot之后驱动就是没probe。查日志没有报错，手动`insmod`驱动模块却瞬间工作——八成是MODULE_DEVICE_TABLE漏写或者宏名写错了。

典型错误：
- 漏写`MODULE_DEVICE_TABLE(of, mydrv_match);` —— modprobe不知道这个驱动对应什么设备
- 写成`MODULE_DEVICE_TABLE(of, mydrv_of_match)`但数组名实际是`mydrv_match` —— 编译不过还好，就怕数组名改了宏没改
- acpi表写了`MODULE_DEVICE_TABLE(acpi, ...)`但udev规则依赖的是of别名 —— 在ARM板上acpi别名没用

快速自查方法：

```bash
# 查看模块是否注册了of别名
modinfo my_driver.ko | grep alias
# 正常应该输出：alias: of:N*T*Cmyvendor,mydevice*

# 查看系统生成的别名数据库
grep myvendor /lib/modules/$(uname -r)/modules.alias
```

如果这两条都没输出，说明MODULE_DEVICE_TABLE没生效，modprobe当然找不到你的驱动。手动insmod只是绕过了整个自动匹配机制，治标不治本。

## 本节总结

| 概念 | 核心要点 | 自查方法 |
|------|---------|---------|
| of_device_id数组 | 设备树匹配的传统写法，最后一项留空`{}` | 检查compatible字符串与设备树完全一致 |
| MODULE_DEVICE_TABLE(of, ...) | 把匹配信息写入模块元数据，支撑modprobe自动加载 | `modinfo xxx.ko \| grep alias` |
| acpi_match_table | x86/ACPI平台的匹配表，与of_match_table并存 | 检查ACPI _HID与硬件ACPI表一致 |
| ACPI_PTR/of_match_ptr | 条件编译宏，没开对应配置时填NULL | 看内核config里CONFIG_OF/CONFIG_ACPI是否启用 |
