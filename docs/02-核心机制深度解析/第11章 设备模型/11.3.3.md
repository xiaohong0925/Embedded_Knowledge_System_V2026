### 11.3.3 设备树绑定规范

**知识点143 [E]**

---

**本节导读**

设备树写好了，驱动也编译进内核了，可设备就是不起来——这种问题你遇到过吗？很多时候根因不是驱动bug，而是设备树节点里的某个属性拼写错了、类型写错了，或者漏掉了必需字段。设备树绑定规范（DT binding）就是解决这类问题的"契约文件"：它用机器可解析的格式定义了每种设备节点"应该长什么样"。本节我们一起理解YAML绑定规范的结构、学习如何用dtc在编译阶段做自动验证。读完这一节，你能读懂内核源码树中的.yaml绑定文件，并把它变成你项目里的"防火墙"。

---

#### 绑定规范是什么：一份"节点体检表"

说白了，绑定规范就是一份**对每个compatible对应设备节点的结构化描述**。它规定了：

- 这个节点**必须有哪些**属性（required）
- 这个节点**可以有哪些**可选属性（optional）
- 每个属性的**数据类型**是什么（string、uint32、phandle、数组…）
- 每个属性的**取值范围**或**枚举值**

在没有绑定规范的时代，这些信息散落在`Documentation/devicetree/bindings/`目录下的`.txt`文件中。人确实能读，但机器看不懂——拼写错误、类型错误只能等到运行驱动probe失败时才发现。

💡 **提示**：绑定规范本质上是**设备树与驱动之间的"接口契约"**。设备树编写者按照这个契约填节点，驱动按照同样的契约读节点，dtc则负责在中间"验货"。

---

#### 从txt到YAML：为什么换格式？

老式的`.txt`绑定规范长这样（节选）：

```
* I2C controller for Generic I2C

Required properties:
- compatible: Must be "generic-i2c-controller"
- reg: address and length of the controller registers
- interrupts: interrupt number

Optional properties:
- clock-frequency: desired I2C bus frequency in Hz
```

人读起来没问题，但dtc工具没法自动解析。YAML格式的核心优势在于**结构化+可机器解析**。新的`.yaml`绑定规范让dtc可以在编译设备树的同时，自动校验每个节点是否符合"契约"。

来看看一个真实的YAML绑定规范示例：

```yaml
# Documentation/devicetree/bindings/gpio/gpio-pl061.yaml
$schema: http://devicetree.org/meta-schemas/core.yaml#

title: ARM PrimeCell PL061 GPIO controller

maintainers:
  - Linus Walleij <linus.walleij@linaro.org>

properties:
  compatible:
    const: arm,pl061

  reg:
    maxItems: 1

  interrupts:
    maxItems: 1

  gpio-controller: true

  '#gpio-cells':
    const: 2

  gpio-line-names:
    minItems: 1
    maxItems: 8

required:
  - compatible
  - reg
  - gpio-controller
  - '#gpio-cells'

additionalProperties: false
```

这个文件的信息密度很高：

| 字段 | 含义 |
|------|------|
| `$schema` | 引用核心schema，定义YAML本身的语法规则 |
| `properties` | 列出该节点所有合法属性及其约束条件 |
| `compatible.const` | compatible必须是精确的`"arm,pl061"` |
| `required` | 缺少其中任何一个属性，dtc都会报错 |
| `additionalProperties: false` | 不允许出现未定义的属性， typo会被当场抓住 |

---

#### 关键schema语法：allOf、anyOf与条件约束

实际项目中，很多绑定规范比上面的示例复杂得多。比如一个控制器IP核可能有多个版本，不同版本的寄存器布局或属性要求不同。这时候`allOf`和`anyOf`就派上用场了：

```yaml
allOf:
  - $ref: gpio-controller.yaml#
  - if:
      properties:
        compatible:
          contains:
            const: vendor,model-v2
    then:
      properties:
        clock-names:
          minItems: 2
    else:
      properties:
        clock-names:
          minItems: 1
```

这段schema的意思是：

- `allOf` —— **同时满足**多个子约束
- `if / then / else` —— 条件分支：如果compatible包含`vendor,model-v2`，则`clock-names`至少要有2个条目；否则至少1个

⚠️ **陷阱**：`allOf`和`anyOf`的区别容易搞混。`allOf`是"**全部都要满足**"（AND关系），`anyOf`是"**至少满足一个**"（OR关系）。写绑定规范时弄反了，会导致dtc验证逻辑完全错误。

---

#### dtc验证：编译时抓bug

YAML绑定规范的价值，最终要通过**dtc的验证**来兑现。dtc在较新版本中已经集成了对YAML binding的支持。编译时加上`--bindings`参数即可：

```bash
# 指定绑定规范目录进行验证编译
dtc --bindings Documentation/devicetree/bindings/ \
    -I dts -O dtb \
    -o output.dtb \
    my-board.dts
```

如果设备树中有任何节点不符合对应的YAML绑定规范，dtc会在编译阶段输出明确的错误信息，而不是让你在运行时抓瞎：

```
Error: /soc/gpio@e000a000: 'interrupts' is a required property
       missing from node
Error: /soc/gpio@e000a000: 'gpio-line-names' has 9 entries,
       but maximum is 8
```

🔴 **危险**：不要在没有开启绑定验证的情况下就把设备树提交到项目中。一个拼错的属性名（比如`interupts`少了一个`r`）可能导致驱动probe时返回`-EINVAL`，而你却要花几个小时才能定位到问题。

```mermaid
flowchart LR
    A[编写 .dts 设备树] --> B[dtc编译 + --bindings验证]
    B -->|验证通过| C[生成.dtb]
    B -->|属性缺失/类型错误| D[编译报错，立即修复]
    C --> E[驱动probe成功]
    D --> A
```

#### 绑定规范在哪里找？

内核源码树中的绑定规范按照**子系统**组织：

```
Documentation/devicetree/bindings/
├── i2c/           # I2C控制器和外设
├── spi/           # SPI控制器和外设
├── gpio/          # GPIO控制器
├── net/           # 以太网控制器
├── timer/         # 定时器
├── usb/           # USB控制器
├── display/       # 显示相关
├── interrupt-controller/  # 中断控制器
└── ...
```

💡 **提示**：当你要为一个新硬件编写设备树节点时，**先去这个目录找对应的.yaml文件**。它是最权威的参考——比芯片手册更直接告诉你"内核期望看到什么属性"。
