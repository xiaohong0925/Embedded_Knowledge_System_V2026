# 11.3.3 设备树绑定规范

> 所属章节：第11章 设备树 > 11.3 设备树规范与工具
> 难度：[E] | 预计阅读时间：12分钟

## 本节导读
设备树节点不是"想怎么写就怎么写"的。内核源码里有一份YAML格式的绑定规范（Binding），相当于设备树的"填写说明"。本节带你理解这份规范的结构，学会用`dtc`命令验证设备树是否符合规范，并搞清楚它和驱动里的`of_match_table`是怎么配合的——对不上暗号，驱动就probe失败。

---

## 知识点150：YAML绑定规范——设备树的"填写说明" [E] ~1000字

### 绑定规范是干什么用的？

想象你去银行办业务，柜员递给你一张**申请表**，上面用红字标注了：姓名（必填）、身份证号（必填）、手机号（选填）。如果你漏填了必填项，或者把身份证号写成了18个字母，柜员会直接打回来让你重写。

设备树绑定规范（Device Tree Binding）就是这张**申请表的标准模板**。它用YAML格式定义了某个硬件节点"必须有哪些属性"、"可选有哪些属性"、"每个属性的类型是什么"。所有规范文件都存放在内核源码的：

```
Documentation/devicetree/bindings/
```

比如你想知道`gpio-leds`节点该怎么写，就去查`Documentation/devicetree/bindings/leds/leds-gpio.yaml`。这份文件不是给人类看着玩的——它是给`dtc`（设备树编译器）用来做**自动化校验**的。

### YAML绑定规范的关键字段

一份典型的绑定规范长这样（以简化版`gpio-leds`为例）：

```yaml
# 代码1：YAML绑定规范的结构示例
$schema: "http://devicetree.org/meta-schemas/core.yaml#"
title: GPIO LEDs

maintainers:
  - Linux LED Subsystem <linux-leds@vger.kernel.org>

description: |
  每个LED用一个GPIO控制，可以设置默认状态、触发器等。

properties:
  compatible:
    const: gpio-leds

  led:
    type: object
    properties:
      gpios:
        description: GPIO引脚引用
        maxItems: 1

      label:
        description: LED的用户可见名称
        type: string

      linux,default-trigger:
        description: 默认触发器
        type: string

    required:
      - gpios

required:
  - compatible

additionalProperties: false
```

这几个字段是核心：

| 字段 | 作用 | 类比 |
|------|------|------|
| `description` | 描述这个绑定是干嘛的，人类的可读说明 | 申请表顶部的"填写须知" |
| `properties` | 定义每个属性的名称、类型、取值范围 | 表格里的各个字段 |
| `required` | 必填属性列表，漏写就报错 | 标了红色星号的必填项 |
| `allOf` | 必须同时满足多个子模式的约束 | "既要A又要B" |
| `anyOf` | 满足任意一个子模式即可 | "A或者B至少满足一个" |

`allOf`和`anyOf`是JSON Schema的标准语法，用来处理复杂的继承和组合场景。比如一个eMMC控制器可能同时要满足"MMC主机控制器的基本规范"和"本公司SoC的DMA扩展规范"，这时候就用`allOf`把两个子规范拼起来。

### 用dtc验证绑定规范

写完设备树后，别急着编译内核，先用`dtc`的`--bindings`选项检查一下：

```bash
# 代码2：用dtc验证设备树是否符合YAML绑定规范
$ dtc -I dts -O dtb \
    --bindings-dir Documentation/devicetree/bindings/ \
    -o myboard.dtb myboard.dts

# 如果有属性缺失或类型错误，会输出类似下面的警告：
myboard.dts:45.3-20: Warning (standard_unit_address): /leds/led@0: \
    missing property 'gpios' required by schema for compatible 'gpio-leds'
```

看到这种警告就要立刻修，别抱有侥幸心理。很多驱动probe失败的根本原因，不是驱动代码有bug，而是设备树节点里某个required属性拼写错了、类型不对、或者根本不存在。

🔴 **危险**：内核社区从5.x版本开始大力推进YAML绑定规范，很多旧的`.txt`格式绑定文档正在被迁移到`.yaml`。如果你参考的是网上某篇2017年的博客，里面的属性名可能已经过时了。一切以内核源码`Documentation/devicetree/bindings/`目录下的YAML文件为准。

💡 **提示**：绑定规范文件名通常和驱动的compatible字符串对应。比如compatible是`gpio-leds`，就去搜`leds-gpio.yaml`。找不到的时候可以`grep -r "gpio-leds" Documentation/devicetree/bindings/`。

⚠️ **陷阱**：`dtc`的版本很重要。旧版dtc可能不支持YAML schema校验。建议用你目标内核版本自带的`scripts/dtc/dtc`，或者确保系统dtc版本>=1.6.0。

## 知识点151：绑定规范与of_match_table——对不上暗号就握手失败 [E] ~600字

### 绑定规范和驱动是怎么配对的？

驱动的`of_match_table`里填的是驱动能识别的compatible字符串列表：

```c
// 代码3：驱动中的of_match_table示例
static const struct of_device_id gpio_led_of_match[] = {
    { .compatible = "gpio-leds" },
    {},
};
MODULE_DEVICE_TABLE(of, gpio_led_of_match);
```

设备树里的`compatible`必须和这个表里的某一项**精确匹配**，驱动子系统才会调用该驱动的probe函数。绑定规范YAML文件则在中间扮演"质检员"——它确保设备树节点里除了compatible之外，其他属性也写得对。

### 凭感觉写 vs 按规范写

我见过太多人在设备树里这样写LED节点：

```dts
// 代码4：凭感觉写的错误示例（别学！）
myleds {
    compatible = "gpio-leds";
    led0 {
        gpio = <&gpio1 3 GPIO_ACTIVE_HIGH>;  // 错误！应该是gpios（复数s）
        name = "status";                       // 错误！应该是label
    };
};
```

看起来"差不多"对吧？但驱动代码里明确用`of_get_named_gpio()`去读`gpios`属性，用`of_property_read_string()`去读`label`。你写成`gpio`和`name`，驱动probe时找不到这些属性，要么 silently fail（静默失败，LED不亮但内核没报错），要么返回`-EINVAL`，具体表现取决于驱动作者有没有认真检查返回值。

按规范写应该是这样的：

```dts
// 代码5：符合绑定规范的gpio-leds节点
/ {
    leds {
        compatible = "gpio-leds";

        led0: led-0 {
            gpios = <&gpio1 3 GPIO_ACTIVE_HIGH>;
            label = "status";
            linux,default-trigger = "heartbeat";
        };
    };
};
```

对比一下，关键差异：

| 项目 | 凭感觉写 | 按规范写 |
|------|---------|---------|
| GPIO属性 | `gpio`（单数） | `gpios`（复数，带s） |
| 名称属性 | `name` | `label` |
| 触发器 | 没写 | `linux,default-trigger` |
| dtc校验 | 不报具体错（可能只报warning） | clean通过 |
| 驱动probe | 大概率失败或行为异常 | 正常识别 |

💡 **提示**：一个省事的技巧是——写设备树节点之前，先去`drivers/`目录下找对应驱动的源码，看看它`of_match_table`里填了什么compatible，然后用`grep -r`在`Documentation/devicetree/bindings/`里找到对应的YAML文件，照着required列表一项一项抄。

⚠️ **陷阱**：绑定规范是"最低要求"，不是"全部要求"。有些驱动在probe时还会去读YAML里标记为optional的属性，如果optional属性的名字你写错了，驱动可能只是行为异常而不是彻底失败。这种bug最难排查。

## 本节总结

| 概念 | 核心要点 | 自查操作 |
|------|---------|---------|
| YAML绑定规范 | 定义设备树节点必填/可选属性的"填写说明" | 查看`Documentation/devicetree/bindings/`下的`.yaml`文件 |
| 关键字段 | `description`、`properties`、`required`、`allOf`/`anyOf` | 阅读目标驱动的绑定规范原文 |
| dtc校验 | `--bindings-dir`参数让dtc按YAML schema检查设备树 | `dtc -I dts -O dtb --bindings-dir ...` |
| 与驱动的关系 | `compatible`必须匹配驱动的`of_match_table`，属性必须匹配绑定规范 | 对照驱动源码和YAML规范一起检查 |
| 常见翻车点 | 属性名拼错单复数、漏写required属性、用旧版txt规范 | 以目标内核版本的YAML为准 |
