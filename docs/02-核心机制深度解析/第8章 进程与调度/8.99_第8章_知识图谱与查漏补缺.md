## 二、查漏补缺（Gap-Fill Index）

> 以下知识点虽未在核心问题中深入展开，但在实际内核开发/调试/面试中频繁出现，建议速查掌握。

---

### 查漏 1：task_struct 的 PID 命名空间链与 pid_nr()

**一句话解释**  
task_struct不直接存储全局PID，而是通过struct pid结构体关联多个命名空间中的PID值，pid_nr()返回指定ns中的PID号。

**为什么重要**  
容器场景中同一进程在不同PID namespace中拥有不同PID（如容器内PID=1，宿主机PID=2345），理解pid结构体是实现容器监控、进程追踪和跨namespace通信的基础。

**快速参考**  
```
task_struct → struct pid *thread_pid → pid_namespace层级链表
upid.nr = 局部PID, upid.ns = 指向namespace
pid_nr(struct pid *pid) → 当前ns中的PID号
pid_vnr() → 虚拟PID（当前ns）
pid_tgid() → 线程组ID
```

**螺旋关联**  
task_struct(8.1.1) → PID namespace(8.1.1) → unshare/setns系统调用 → 容器进程管理 → /proc挂载点隔离 → Docker/Kubernetes PID隔离

---

### 查漏 2：RCU 与进程退出的异步回收

**一句话解释**  
进程exit时通过call_rcu()异步释放task_struct，利用RCU宽限期（grace period）确保所有读端临界区退出后再回收内存，避免ABA问题和use-after-free。

**为什么重要**  
/proc文件系统、调度器负载统计、ptrace等多个子系统可能持有task_struct引用但不持有锁，RCU是保证这些并发读场景安全的关键机制，也是内核面试高频考点。

**快速参考**  
```
do_exit() → exit_notify() → release_task() → __put_task_struct() → call_rcu(delayed_put_task_struct)
delayed_put_task_struct() → RCU回调中执行kfree_rcu(tsk, rcu)
/proc下文件操作通过get_task_struct()/put_task_struct()管理引用计数
注意：exit_code/exit_signal在ZOMBIE状态通过wait()读取，不能提前释放
```

**螺旋关联**  
task_struct生命周期(8.1.2) → RCU同步原语 → SLAB回收器 → /proc实现 → wait_queue与exit通知 → 孤儿进程处理 → init进程接管

### 查漏 3：CFS 组调度（Group Scheduling）与 cpu.cgroup 递归计费

**一句话解释**  
CFS支持以cgroup为调度实体（task_group），通过sched_entity嵌套结构实现层级化的CPU时间分配，父cgroup的配额按比例分配给子cgroup和叶子任务。

**为什么重要**  
容器场景（Docker/LXC/K8S）中多个cgroup共享CPU时，组调度确保"容器间公平"而非"进程间公平"，是云原生资源隔离的调度基础。缺乏组调度理解会导致CPU quota配置失效的调试困境。

**快速参考**  
```
task_group → sched_entity[NR_CPU]（cgroup的调度实体）
cfs_rq → tg_runnable_avg（任务组运行负载）
CPU时间分配路径：root tg → child tg A (cpu.weight=300) → child tg B (cpu.weight=100)
  → tg A获得3/4时间，tg B获得1/4时间
cgroup v1: cpu.shares；cgroup v2: cpu.weight
开启CONFIG_CGROUP_SCHED / CONFIG_FAIR_GROUP_SCHED
注意：组调度层级越深，调度开销越大
```

**螺旋关联**  
CFS vruntime(8.3.2) → task_group结构 → cpu.shares/weight(8.5.2) → cgroup v2统一层次(8.5.1) → systemd slice-unit(8.5.3) → 容器CPU限制 → 云厂商ECS CPU超售

### 查漏 4：no_hz / tickless 与调度器时间记账

**一句话解释**  
CONFIG_NO_HZ_IDLE/CONFIG_NO_HZ_FULL使空闲CPU（或全核）停止周期时钟中断，减少功耗；调度器通过vtime (virtual time) 记账机制在tick缺失时精确补偿vruntime。

**为什么重要**  
tickless是移动设备和数据中心节能的关键内核特性，但会导致传统基于tick的调度统计失效。理解vtime_guest/vtime_accounting对于分析虚拟机/容器场景中CPU steal time和调度延迟至关重要。

**快速参考**  
```
NO_HZ_IDLE: CPU空闲时停止tick，醒来时一次性补偿
NO_HZ_FULL: 用户态单任务时停止tick，内核态保留tick
vtime_accounting: 通过vtime_switch()在上下文切换时精确计算delta_exec
vtime_delta()补偿函数用于tickless期间的vruntime修正
/proc/timer_list 查看tick状态
常见问题：dynticks编程模型下的race condition（vtime与rq clock不同步）
```

### 查漏 5：内核抢占 (Kernel Preemption) 与调度时机

**一句话解释**  
CONFIG_PREEMPT系列配置允许内核代码在执行过程中被更高优先级任务抢占，调度检查点分散在spin_unlock、中断返回、系统调用返回等位置，调度器通过need_resched标志触发延迟调度。

**为什么重要**  
内核延迟敏感场景（实时、音频、交互）必须开启抢占，但过度抢占增加上下文切换开销和锁竞争。理解抢占模型是配置实时内核(CONFIG_PREEMPT_RT)和调试"意外高延迟"问题的核心。

**快速参考**  
```
抢占模型层级：
  CONFIG_PREEMPT_NONE — 服务器，无内核抢占
  CONFIG_PREEMPT_VOLUNTARY — 桌面，显式抢占点(cond_resched)
  CONFIG_PREEMPT — 低延迟，任意内核位置可抢占（除临界区）
  CONFIG_PREEMPT_RT — 全实时，spinlock→mutex，硬中断→线程

关键机制：
  preempt_count() — 跟踪抢占禁用嵌套深度
  spin_lock() → preempt_disable()；spin_unlock() → preempt_enable() → 检查need_resched
  need_resched标志 — TIF_NEED_RESCHED设置后下次抢占点触发调度
  cond_resched() — 自愿让出，用于长时间运行的内核循环

常见问题：
  关中断+关抢占的临界区内发生长时间操作 → 调度延迟
  PREEMPT_RT下所有中断处理线程化 → irq_handler注册变化
```

**螺旋关联**  
调度时机(8.3) → need_resched → 抢占计数(preempt_count) → 自旋锁与关中断 → 实时抢占(CONFIG_PREEMPT_RT) → SCHED_FIFO调度(8.4.1) → 火星探路者bug(8.4.2) → 优先级继承(8.4.2) → 硬实时系统延迟保证
