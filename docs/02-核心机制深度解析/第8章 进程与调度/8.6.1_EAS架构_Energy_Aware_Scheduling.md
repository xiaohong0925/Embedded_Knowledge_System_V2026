# 8.6.1 EAS架构：Energy Aware Scheduling

> 所属：第8章 调度与实时性 > 8.6 异构调度专题
> 难度：[I→E] | 预计阅读时间：35分钟

## 本节导读

ARM big.LITTLE架构将高性能大核与高能效小核集成在同一SoC中，传统Linux CFS调度器以吞吐量为唯一优化目标，无法在这种异构拓扑上做出能效最优的调度决策。本节深入解析Energy Aware Scheduling（EAS）的内核实现：从Energy Model的数据来源到`find_energy_efficient_cpu()`的唤醒路径决策逻辑，帮助你理解Linux调度器如何在性能与能效之间取得平衡。

---

## 知识点1：为什么需要EAS [I] ~800字

### 问题场景

假设你正在调试一款搭载ARM big.LITTLE架构的移动设备（4×Cortex-A53 + 4×Cortex-A72），用户抱怨以下现象：

- **发热严重**：轻量级任务（如后台同步）频繁被放到big核上执行，导致不必要的功耗
- **卡顿明显**：重负载场景下，任务在小核上"堆积"，无法及时迁移到大核
- **续航波动大**：同样的使用模式，电池消耗时间差异显著

这些问题的根源在于：**传统CFS调度器对异构CPU拓扑"视而不见"**。

### big.LITTLE架构的核心挑战

| 特性 | big核心（高性能） | LITTLE核心（高能效） |
|------|------------------|---------------------|
| 微架构 | Out-of-Order, 大缓存 | In-Order, 小缓存 |
| 最高频率 | 2.0+ GHz | 1.5 GHz |
| 典型功耗 | 1000~2000 mW | 50~300 mW |
| 峰值性能（capacity） | 1024（归一化） | 256~512（归一化） |
| 能效曲线斜率 | 陡峭（高频功耗非线性增长） | 平缓（功耗随频率线性增长） |
| 典型微架构 | Cortex-A72/A76/A78 | Cortex-A53/A55 |

⚠️ **常见陷阱**：很多工程师误以为"小核总是更省电"。实际上，当小核跑到最高频时，其能效可能比big核的最低频还要差。EAS正是为了解决这种非直觉的能效权衡而设计的。

### 传统调度器的局限

在EAS出现之前，Linux调度器面对big.LITTLE时存在以下问题：

1. **负载均衡盲目性**：`load_balance()`以CPU负载（load_avg）为唯一指标，不知道big核和LITTLE核的能效差异。一个负载为100的任务迁移到big核或LITTLE核，在调度器看来"没有区别"。

2. **唤醒路径缺乏能效意识**：`select_task_rq_fair()`选择CPU时只考虑亲和性和负载，不比较不同CPU上的能效代价。

3. **DVFS与调度脱节**：cpufreq governor独立决策频率，调度器无法预测任务放置对频率选择的影响。

💡 **技巧**：可通过`/sys/devices/system/cpu/cpu*/cpufreq/scaling_available_frequencies`查看各CPU的频率范围，通过`/proc/sys/kernel/sched_energy_aware`确认EAS是否启用。

### EAS的解决思路

EAS的核心洞察是：**调度器作为唯一知晓所有任务需求的子系统，应该主导能效决策**。具体包括：

- 在**任务唤醒路径**中引入能效估计，选择能耗最低的CPU
- 建立统一的**Energy Model**，量化每个CPU在不同频率下的功耗
- 与`schedutil` governor协作，确保频率请求与能耗预测的一致性

## 知识点2：EAS的核心思想与架构 [E] ~1200字

### Energy Model（EM）框架

EM是EAS的基础，它为每个**Performance Domain**（PD）维护一张功耗-性能表。一个PD是一组共享相同微架构且频率一起调整的CPU。

```
+---------------------+     +---------------------+
|  Performance Domain 0  |     |  Performance Domain 1  |
|  (LITTLE cluster)      |     |  (big cluster)         |
|  CPU0, CPU1, CPU2, CPU3 |     |  CPU4, CPU5, CPU6, CPU7 |
+----------+----------+     +----------+----------+
           |                           |
    +------v------+             +------v------+
    | EM Table    |             | EM Table    |
    | Cap | Power |             | Cap | Power |
    | 128 |  50mW |             | 341 | 400mW |
    | 256 | 150mW |             | 682 | 800mW |
    | 512 | 300mW |             |1024 |1700mW |
    +-------------+             +-------------+
```

EM的核心数据结构定义在`include/linux/energy_model.h`：

```c
struct em_perf_state {
    unsigned long performance;  /* 该频率下的capacity值 */
    unsigned long frequency;    /* 频率（KHz） */
    unsigned long power;        /* 该状态的功耗（微瓦） */
    unsigned long cost;         /* 能效代价系数，用于EAS决策 */
    unsigned long flags;
};

struct em_perf_domain {
    struct em_perf_table __rcu *em_table;
    struct list_head node;
    int nr_perf_states;         /* OPP数量 */
    unsigned long cpus[];       /* 该PD包含的CPU掩码 */
};
```

💡 **技巧**：`cost`字段的计算公式为 `cost = power × max_frequency / frequency`。该值越大表示该OPP的能效越差，EAS用它快速比较不同CPU的能效优劣。

### EAS唤醒路径的关键代码

当CFS任务被唤醒时，`select_task_rq_fair()`会调用`find_energy_efficient_cpu()`进行能效最优的CPU选择。以下是该函数的简化逻辑：

```c
/* kernel/sched/fair.c */
static int find_energy_efficient_cpu(struct task_struct *p, int prev_cpu)
{
    unsigned long prev_delta = ULONG_MAX, best_delta = ULONG_MAX;
    unsigned long p_util = task_util_est(p);  /* 任务估计utilization */
    int best_cpu = prev_cpu;
    struct perf_domain *pd;

    /* 遍历root domain下的所有performance domain */
    rcu_read_lock();
    for_each_pd(rd, pd) {
        unsigned long cpu_util, cpu_cap;
        int cpu, max_spare_cap_cpu = -1;
        unsigned long max_spare_cap = 0;

        /* 找到该PD中spare capacity最大的CPU */
        for_each_cpu_and(cpu, perf_domain_span(pd), rd->span) {
            cpu_util = cpu_util_next(cpu, p, cpu);
            cpu_cap = capacity_of(cpu);
            if (cpu_cap - cpu_util > max_spare_cap) {
                max_spare_cap = cpu_cap - cpu_util;
                max_spare_cap_cpu = cpu;
            }
        }

        /* 计算将任务放到该CPU后的系统总能耗 */
        if (max_spare_cap_cpu >= 0) {
            unsigned long energy = compute_energy(p, max_spare_cap_cpu, pd, rd);
            unsigned long delta = energy - base_energy;

            if (delta < best_delta) {
                best_delta = delta;
                best_cpu = max_spare_cap_cpu;
            }
        }
    }
    rcu_read_unlock();
    return best_cpu;
}
```

`compute_energy()`函数的核心逻辑是模拟任务迁移后的能耗：

```c
static unsigned long compute_energy(struct task_struct *p, int dst_cpu,
                                    struct perf_domain *pd,
                                    struct root_domain *rd)
{
    unsigned long max_util, sum_util, energy = 0;
    struct cpumask *pd_mask = perf_domain_span(pd);
    int cpu;

    for_each_cpu_and(cpu, pd_mask, rd->span) {
        unsigned long util = cpu_util_next(cpu, p, dst_cpu);
        /* ... 累加utilization ... */
        sum_util += schedutil_cpu_util(cpu, util, CPUFREQ_ENERGY);
        max_util = max(max_util, util);
    }

    /* 调用EM框架API计算该PD的能耗 */
    energy = em_cpu_energy(pd->em_pd, max_util, sum_util, 
                           capacity_orig_of(cpu));
    return energy;
}
```

🔴 **安全提醒**：`find_energy_efficient_cpu()`运行在任务唤醒路径上，对延迟极为敏感。EM复杂度检查 `C = Nd × (Nc + Ns)` 必须小于`EM_MAX_COMPLEXITY`（2048），否则EAS将拒绝启用。

### Over-utilization机制

EAS并非在所有场景下都优于传统调度器。当系统重载时，能效优化的边际收益趋近于零，而优先级公平性变得更重要。EAS通过**over-utilized**标记实现自动切换：

- 当某个CPU的利用率超过其capacity的80%时，整个root domain被标记为`over-utilized`
- EAS被禁用，调度器回退到传统的基于nice值的负载均衡
- 当所有CPU都低于80%阈值时，EAS重新启用

| 系统状态 | EAS行为 | Load Balance行为 |
|---------|--------|----------------|
| 轻/中载（<80% capacity） | 启用，能效最优任务放置 | 禁用，避免破坏EAS的放置决策 |
| 重载（≥80% capacity） | 禁用，回退CFS策略 | 启用，确保公平性 |

⚠️ **常见陷阱**：如果在高负载场景下发现任务优先级没有被正确尊重（如交互任务卡顿），检查`/proc/sys/kernel/sched_energy_aware`是否被强制开启且over-utilized检测是否正常工作。

## 知识点3：EM的数据来源与设备树配置 [E] ~800字

### EM数据的三种来源

Linux内核支持多种方式向EM框架提供功耗数据，不同平台可根据自身条件选择：

| 数据来源 | 精度 | 适用场景 | 内核版本 |
|---------|------|---------|---------|
| Device Tree `opp-microwatt` | 中（静态数据） | 有OPP表的ARM/ARM64平台 | 5.1+ |
| SCMI Firmware | 高（固件动态提供） | 支持SCMI的现代SoC | 5.2+ |
| Driver Callback (`est_power`) | 高（自定义测量） | 特殊硬件/需动态调整 | 5.0+ |
| `cpufreq_register_em_with_opp()` | 低（公式估算） | 快速原型/缺少实测数据 | 5.1+ |

### 设备树配置示例

最常见的方式是通过设备树中的`operating-points-v2`结合`opp-microwatt`属性提供功耗数据。以下是一个big.LITTLE平台的典型配置：

```dts
/* arch/arm64/boot/dts/rockchip/rk3399.dtsi 简化示例 */

/* LITTLE cluster (Cortex-A53) OPP表 */
cluster0_opp: opp-table0 {
    compatible = "operating-points-v2";
    opp-shared;

    opp-408000000 {
        opp-hz = /bits/ 64 <408000000>;
        opp-microvolt = <800000>;
        opp-microwatt = <50000>;     /* 功耗数据，单位微瓦 */
    };
    opp-816000000 {
        opp-hz = /bits/ 64 <816000000>;
        opp-microvolt = <825000>;
        opp-microwatt = <120000>;
    };
    opp-1416000000 {
        opp-hz = /bits/ 64 <1416000000>;
        opp-microvolt = <950000>;
        opp-microwatt = <350000>;
    };
};

/* big cluster (Cortex-A72) OPP表 */
cluster1_opp: opp-table1 {
    compatible = "operating-points-v2";
    opp-shared;

    opp-408000000 {
        opp-hz = /bits/ 64 <408000000>;
        opp-microvolt = <800000>;
        opp-microwatt = <150000>;
    };
    opp-1200000000 {
        opp-hz = /bits/ 64 <1200000000>;
        opp-microvolt = <900000>;
        opp-microwatt = <800000>;
    };
    opp-1800000000 {
        opp-hz = /bits/ 64 <1800000000>;
        opp-microvolt = <1100000>;
        opp-microwatt = <2000000>;
    };
};

/* CPU节点引用OPP表 */
cpus {
    cpu0: cpu@0 {
        compatible = "arm,cortex-a53";
        operating-points-v2 = <&cluster0_opp>;
        capacity-dmips-mhz = <(512 / 2)>;   /* 相对capacity */
    };
    
    cpu4: cpu@4 {
        compatible = "arm,cortex-a72";
        operating-points-v2 = <&cluster1_opp>;
        capacity-dmips-mhz = <(1024 / 2)>;
    };
};
```

### 内核解析流程

设备树中的`opp-microwatt`数据由以下路径解析并注册到EM框架：

```
Device Tree
    |
    v
OPP framework (drivers/opp/of_opp.c)
    |-- of_init_opp_table() 解析OPP表
    |-- _opp_add() 添加每个OPP点
    |
    v
cpufreq-dt driver (drivers/cpufreq/cpufreq-dt.c)
    |-- cpufreq_driver.register_em = dt_cpufreq_register_em()
    |   --> cpufreq_register_em_with_opp() 或 em_dev_register_perf_domain()
    |
    v
EM Framework (kernel/power/energy_model.c)
    |-- em_dev_register_perf_domain()
    |-- 构建em_perf_domain + em_perf_state表
    |-- 计算cost值
    |
    v
Scheduler (kernel/sched/topology.c)
    |-- build_sched_domains()
    |-- 关联em_perf_domain到root domain
    |-- 启用sched_energy_present
```

在启动日志中可以看到EM注册的关键信息：

```bash
# dmesg | grep -i "energy"
[    0.123456] energy_model: EM: pd0 CPUs 0-3 <128/256/512> mW
[    0.123789] energy_model: EM: pd1 CPUs 4-7 <400/800/1700> mW
[    0.234567] sched: Energy Aware Scheduling enabled
```

🔴 **安全提醒**：`opp-microwatt`的值必须来自实际测量或可信的固件数据。错误的功耗值会导致EAS做出相反的调度决策——将任务放到高功耗CPU上，严重劣化系统能效。

### 运行时EM更新

从Linux 5.16+开始，EM支持运行时更新，这对thermal场景尤为重要。当温度升高触发降频时， thermal驱动可以更新EM中的power值，EAS会根据新的功耗数据调整调度决策：

```c
/* thermal驱动中更新EM的伪代码 */
struct em_perf_table *new_table = em_table_alloc(pd);
/* 根据当前温度重新计算各OPP的功耗 */
for (i = 0; i < pd->nr_perf_states; i++) {
    new_table->state[i].power = 
        thermal_adjusted_power(orig_power, current_temp);
}
em_dev_compute_costs(dev, new_table->state, pd->nr_perf_states);
em_dev_update_perf_domain(dev, new_table);
```

## 实践案例：某平板设备的EAS调优

**场景**：一款基于RK3399（2×A72 + 4×A53）的教育平板，用户反馈在线视频播放时发热严重。

**分析过程**：

1. **确认EAS状态**：`cat /proc/sys/kernel/sched_energy_aware` 返回1，EAS已启用。

2. **检查EM数据**：
   ```bash
   $ cat /sys/devices/system/cpu/energy_model/pd0/ps*
   freq: 1416000 power: 350000 cost: 252
   $ cat /sys/devices/system/cpu/energy_model/pd1/ps*
   freq: 1800000 power: 2000000 cost: 1137
   ```

3. **发现问题**：视频解码任务（util_avg≈150）在播放开始时频繁被放到A72上。原因是EM中A53高频点的cost值设置不合理，导致EAS误判A72更"划算"。

4. **修复**：修正DTS中A53 cluster的`opp-microwatt`值（原值偏低约20%），重新编译后EAS正确将视频任务留在A53上运行。实测播放温度降低4.2°C，续航提升约12%。

💡 **技巧**：可通过`tracefs`的`sched:sched_wakeu` tracepoint捕获实际的任务放置决策，结合`em_cpu_energy()`的返回值验证EAS行为是否符合预期。

## EAS vs 传统调度器对比

| 维度 | 传统CFS调度器 | EAS（Energy Aware Scheduling） |
|------|-------------|------------------------------|
| **优化目标** | 吞吐量最大化 / 负载均衡 | 能效最大化，同时保证性能 |
| **CPU拓扑认知** | 无（所有CPU视为对等） | 完整感知capacity和功耗差异 |
| **任务放置依据** | load_avg + wake_affine | EM能耗估计 + spare capacity |
| **DVFS协作** | 独立（cpufreq自行决策） | 与schedutil紧耦合 |
| **适用场景** | 同构SMP系统 | 异构big.LITTLE/DynamIQ系统 |
| **负载均衡** | 周期性执行 | 轻载时禁用，避免破坏能效布局 |
| **复杂度** | O(1) wake-up | O(Nd×(Nc+Ns))，受EM复杂度限制 |
| **内核启用条件** | 默认开启 | 需CONFIG_ENERGY_MODEL + 异构拓扑 + schedutil |

## 本节总结

EAS是Linux调度器在异构CPU时代的重要演进。核心要点回顾：

1. **问题本质**：big.LITTLE的非对称功耗-性能特性需要调度器具备"能效意识"，传统CFS无法满足
2. **核心机制**：通过Energy Model量化每个调度决策的能耗代价，在唤醒路径选择最优CPU
3. **数据来源**：设备树`opp-microwatt`、SCMI固件或驱动回调，最终统一为`em_perf_state`表
4. **关键权衡**：EAS在轻/中载时优化能效，在重载时通过over-utilized机制自动回退，确保不损失性能公平性
5. **调试要点**：确认`sched_energy_present`启用、验证EM数据合理性、trace唤醒路径决策
