## 方向1：中断的工程背景与核心概念 [B→I]

在深入内核数据结构之前，必须先建立对中断机制的直觉认知——它不是什么高深莫测的魔法，而是工程实践中对"CPU如何高效响应外设"这一古老问题的回答。

### 小点1：为什么需要中断——轮询的困境与事件驱动的必然

在嵌入式系统中，CPU与外设交换数据只有两种基本方式：主动查询或被动通知。<br>最原始的做法是**轮询**（Polling，主动查询模型）——CPU以固定周期反复读取外设的状态寄存器，直到发现数据就绪。

<>轮询的第一道工程天花板是CPU空转与功耗失控。

假设一个工业传感器每秒仅产生一次有效数据，CPU为了不错过它，必须以毫秒级间隔查询状态寄存器。这意味着即使外设99%的时间处于空闲，CPU也无法进入低功耗状态，空转功耗直接吃掉电池续航。如果不引入中断，手持设备或物联网节点的待机时间将从数周暴跌到数小时。

第二道天花板是响应延迟与事件漏检的结构性矛盾。轮询间隔设得太长，两次查询之间发生的紧急事件（如电机过流、**CAN**（Controller Area Network，控制器局域网）总线错误帧）会被完全漏掉；间隔设得太短，CPU又陷入前述的忙等待陷阱。这种"要么漏信号，要么空转"的困境，在实时控制系统中尤为致命——机械臂碰到限位开关后，不能等到CPU下一次查询才紧急制动。

**中断**（Interrupt，异步通知机制）正是为了打破这个僵局而设计的。它不是CPU主动去查，而是外设在需要服务时，通过硬件信号线向CPU发送一条异步请求。CPU收到后，保存当前执行现场，跳转到专门的处理程序，处理完再恢复现场继续原任务。空闲期间CPU可以执行**WFI**（Wait For Interrupt，等待中断指令）进入休眠，外设来信号时才被唤醒。

> 💡 **直观理解**：中断把"CPU主动查"变成了"外设主动叫"。就像餐厅给每桌安装服务铃——客人需要时按铃，服务员只在听到铃声时才行动，其余时间可以处理其他事务。

两种模式的工程差异可以用一张表格直观对比。

| 维度 | 轮询（Polling） | 中断（Interrupt） |
|:---|:---|:---|
| CPU角色 | 主动查询，忙等待 | 被动响应，异步触发 |
| 响应延迟 | 取决于查询间隔，最坏情况为一个完整周期 | 硬件信号传播+上下文切换，通常为微秒级 |
| CPU占用 | 接近100%（空转） | 仅**ISR**（Interrupt Service Routine，中断服务程序）执行期间占用 |
| 功耗表现 | 高，CPU无法进入低功耗状态 | 空闲时可执行WFI休眠 |
| 可靠性风险 | 间隔过长导致事件漏检 | 中断风暴可能导致系统过载 |

表格对比了原理，但开发板上到底谁在吃CPU？Linux把这一切暴露得很直白——一行命令就能看到轮询的代价。

```bash
# 终端1：模拟轮询GPIO状态（每秒查询1000次）
$ while true; do cat /sys/class/gpio/gpio0/value > /dev/null; done &

# 终端2：观察CPU空闲率
$ top -bn1 | grep "Cpu(s)"
%Cpu(s): 97.5 us,  2.3 sy,  0.0 ni,  0.0 id,  0.0 wa...
# idle(id)接近0%，CPU被轮询吃满
```

由于上述缺陷，中断机制应运而生。它把"CPU主动查"变成了"外设主动叫"，不仅解放了CPU，也让系统具备了真正的实时响应能力。但外设发出的这条"异步请求"，在硬件和软件层面到底意味着什么？CPU被打断后，又是怎样保证原任务不受影响的？

### 小点2：中断的本质定义——异步信号、执行现场、ISR

外设的数据就绪时刻与CPU的指令执行节奏完全独立。CPU正在计算浮点运算时，网卡数据包可能突然到达——两者在时序上没有任何约定。这种**异步性**（Asynchrony，时序无关性）是中断设计的核心矛盾：CPU不能预知外设何时发信号，但必须保证处理完信号后，原程序像什么都没发生一样继续执行。

如果不保存被打断时的精确状态，**程序计数器**（`PC`，Program Counter）和通用寄存器的值会在中断处理过程中被覆盖。

等处理结束，CPU根本不知道回到哪条指令、用哪个寄存器值继续，整个程序直接跑飞。因此，中断机制的第一要务是**现场保存**（Context Saving）。

CPU收到中断请求后，硬件自动将当前**执行现场**（Execution Context，包括程序计数器、程序状态寄存器**CPSR**（Current Program Status Register，当前程序状态寄存器）/**SPSR**（Saved Program Status Register，保存的程序状态寄存器）以及通用寄存器组）压入当前模式下的专用栈或银行化寄存器。随后CPU跳转到**中断服务程序**（Interrupt Service Routine，`ISR`）。这是内核或驱动工程师编写的专用函数，负责读取外设状态、搬运数据、清除中断标志。`ISR`执行结束后，CPU通过中断返回指令（如ARM的**ERET**（Exception Return，异常返回））从栈中恢复现场，主程序精确回到被打断的指令继续执行。

整个流程可以用ASCII时序图表示。

```
主程序执行流
  |
  |  正常指令流
  v
  +-----------------+
  |  外设发起中断   |<---- 异步信号到达（与指令流无关）
  +-----------------+
  |
  v
  +-----------------+
  |  保存执行现场   |<---- PC/寄存器 → 栈/SPSR（硬件自动完成）
  +-----------------+
  |
  v
  +-----------------+
  |  执行 ISR       |<---- 中断服务程序（软件处理外设事件）
  +-----------------+
  |
  v
  +-----------------+
  |  恢复执行现场   |<---- 栈/SPSR → PC/寄存器（硬件自动完成）
  +-----------------+
  |
  v
  回到主程序断点继续执行
```

图示展示了硬件层面的自动切换，但Linux内核在软件层面是如何把外设信号路由到正确的`ISR`的？

内核为每个硬件中断线维护一个**中断描述符**（`irq_desc`，interrupt descriptor），其中挂接着一个**中断动作描述符**（`irqaction`，interrupt action）链表，每个`irqaction`记录驱动注册的`ISR`函数指针。以ARM64为例，中断入口由`arch/arm64/kernel/entry.S`中的汇编代码接管，最终汇聚到C层的`handle_arch_irq`（架构相关中断处理入口），进而调用`kernel/irq/handle.c`中的`__handle_irq_event()`遍历该链表：

```c
// kernel/irq/handle.c:__handle_irq_event() 第 157-175 行
irqreturn_t __handle_irq_event(struct irq_desc *desc)
{
    irqreturn_t retval = IRQ_NONE;           // 初始标记：尚未处理
    unsigned int irq = desc->irq_data.irq;   // 获取虚拟中断号
    struct irqaction *action = desc->action; // 取出ISR链表头

    do {
        irqreturn_t res;
        res = action->handler(irq, action->dev_id); // 执行驱动注册的ISR
        if (res == IRQ_HANDLED)              // 若ISR认领了该中断
            retval = IRQ_HANDLED;              // 更新总状态
        action = action->next;                 // 遍历共享中断的下一个action
    } while (action);

    return retval;                             // 返回处理结果给上层流控
}
```

这段代码展示了中断处理的核心契约：内核通过`irq_desc`找到挂接的`irqaction`链表，逐个调用驱动注册的`handler()`。返回`IRQ_HANDLED`表示事件已被消费；返回`IRQ_NONE`则继续尝试下一个action。这个链式结构正是"共享中断"的实现基础，也是Linux将硬件信号映射到软件回调的关键枢纽。`irq_desc`和`irqaction`的详细数据结构解析，详见3.6.2节。

理解了中断的异步本质和`ISR`的执行模型，还有一个关键区分不能忽略：这些"打断"信号并不都来自外部硬件。如果不分清两者，调试时会在错误的方向上浪费数小时。

### 小点3：外部中断与内部异常——来源与触发方式差异

内核日志里经常同时出现两种"打断"记录：一种是`/proc/interrupts`里计数器持续增长的外部事件，另一种是`dmesg`里带着精确地址的`do_page_fault`（缺页异常处理函数）报错。

新手很容易把它们统称为"中断"，但排错时混用这两个概念会导致方向性错误——你在追查硬件信号毛刺，实际上问题出在内存越界。

**外部中断**（External Interrupt）来自CPU核外的硬件设备。**GPIO**（General Purpose Input Output，通用输入输出）电平翻转、定时器到期、网卡收到数据包、**UART**（Universal Asynchronous Receiver/Transmitter，通用异步收发传输器）接收到字节，这些事件通过中断控制器（如**GIC**（Generic Interrupt Controller，通用中断控制器））路由到CPU。它们的共同特征是：与当前执行的指令无关，随时可能发生，具有真正的异步性。

**内部异常**（Internal Exception）则源于CPU执行指令流时检测到的特殊情况。除以零、访问未映射内存（缺页）、执行非法操作码、断点调试指令，这些都不是外设发起的，而是指令执行本身的副产品。它们总是与某条特定指令绑定，因此被称为**同步事件**（Synchronous Event）。

ARM架构用一张向量表统一处理所有"打断"，但入口逻辑和修复策略截然不同。

| 类型 | 来源 | 触发时机 | 与指令关系 | 内核入口 | 典型处理 |
|:---|:---|:---|:---|:---|:---|
| 外部中断（`IRQ`（Interrupt Request，中断请求）/`FIQ`（Fast Interrupt Request，快速中断请求）） | 外设 via GIC | 异步，任意时刻 | 与当前指令无关 | `handle_arch_irq` | 读取外设状态，清中断标志 |
| 同步异常 | CPU自身 | 执行特定指令时 | 与触发指令精确绑定 | `do_mem_abort` / `do_el0_sync` | 尝试修复（如缺页填充）或终止进程 |
| 异步异常 | 外部错误信号 | 异步，但通常与总线事务相关 | 与指令流松散相关 | `do_serror` | 记录错误地址，终止或恢复 |

外部中断的`ISR`处理完外设事件后，无条件回到断点继续执行——它从不"修复"主程序。同步异常则完全不同：它必须尝试修复触发条件，成功后重新执行那条指令。

以ARM64缺页异常为例，内核通过**异常综合寄存器**（`ESR`（Exception Syndrome Register，异常综合寄存器））保存异常原因，通过**寄存器保存结构**（`struct pt_regs`）记录触发时的CPU状态：

```c
// arch/arm64/mm/fault.c:do_mem_abort() 第 300-320 行（近似）
static int __kprobes do_mem_abort(unsigned long addr, unsigned int esr,
                                  struct pt_regs *regs)
{
    struct siginfo info;
    int fault; // 错误处理结果

    // ESR保存了异常原因和触发指令特征
    // 内核根据ESR判断是缺页、权限错误还是外部中止
    fault = do_page_fault(addr, esr, regs); // 尝试修复（如分配页表）

    if (fault == 0)                         // 修复成功
        return 0;                           // 返回后重新执行触发指令

    // 修复失败，准备向用户态发送信号
    info.si_signo = SIGSEGV;                // SIGSEGV（Segmentation Fault，段错误信号）
    info.si_addr = (void __user *)addr;     // 记录故障地址
    arm64_notify_die("Oops", regs, &info, esr); // 终止或发送信号
    return 0;
}
```

这段代码揭示了同步异常的核心特征：它必须尝试**修复现场**（如缺页时分配物理页），修复成功后返回并重新执行触发指令。而外部中断的`ISR`从不"修复"主程序，只是处理外设事件，然后无条件回到断点。如果在这里混用概念，试图在`ISR`里修复指令流，逻辑会直接崩溃。

从调试视角看，区分两者也很简单。外部中断的触发不可预测，通常用`/proc/interrupts`观察计数增长；同步异常则可以通过`dmesg`看到精确的错误地址和触发指令。

```bash
# 查看外部中断统计（计数器持续增长）
$ cat /proc/interrupts | grep "timer"
  19:     23456789     23456790     ...  IR-IO-APIC    18-fasteoi   acpi

# 查看最近发生的同步异常（精确到地址和PID）
$ dmesg | tail -n 5
[  123.456] do_page_fault(): 0x00000000 pid=1234 (test) ...
```

> ⚠️ **注意**：`IRQ`和`FIQ`是ARM架构的术语，在x86中对应`INTR`（Interrupt，中断）和`NMI`（Non-Maskable Interrupt，不可屏蔽中断）。不同架构的中断控制器实现差异很大，本章聚焦通用机制，具体SoC实现详见第1章。

外部中断让系统具备响应外部世界的能力，内部异常则是CPU自我保护的最后防线。两者共享"打断当前执行流"的机制，但来源、触发方式和处理哲学完全不同。

## 方向2：ARM中断控制器架构 [I]

搞清了中断的本质和分类，下一个问题自然浮现：这些来自外设的异步信号，在芯片内部是如何被接收、识别和分发的？

### 小点1：GICv2与v3核心能力差异——多核时代的扩展瓶颈

在ARM多核系统中，中断控制器是连接外设与CPU核的中央枢纽。最早广泛部署的**通用中断控制器**（GIC，Generic Interrupt Controller）v2版本，在单核和早期双核时代尚能胜任，但进入多核**SMP**（Symmetric Multi-Processing，对称多处理）和虚拟化时代后，其架构出现了明显的扩展瓶颈。

GICv2的第一道天花板是中断分发能力的硬限制。

其**分发器**（Distributor，负责全局中断路由的中央组件）和**CPU接口**（CPU Interface，连接CPU核的本地中断接口）采用全局共享寄存器设计。在8核以上的服务器级SoC中，GICv2无法将中断精确绑定到任意目标核或核组，导致负载均衡困难。如果不升级中断控制器架构，多核系统的**中断导引**（Interrupt Steering，中断定向分发）将完全依赖软件轮询补救，实时性大幅下降。

第二道天花板是虚拟化支持的缺失。

GICv2没有硬件级别的虚拟中断注入能力，**虚拟机监控器**（Hypervisor，管理虚拟机的软件层）必须通过软件模拟向**客户操作系统**（Guest OS，虚拟机内运行的操作系统）发送中断。每次虚拟中断的注入都需要**虚拟机退出**（VM-Exit，从虚拟态切换到宿主态），上下文切换开销高达数千个时钟周期。如果不引入硬件虚拟化支持，ARM服务器在云计算场景中的性能将严重落后于x86的**APICv**（Advanced Programmable Interrupt Controller Virtualization，高级可编程中断控制器虚拟化）方案。

**GICv3**正是为了解决这些扩展瓶颈而设计的。它在GICv2的Distributor和CPU Interface基础上，引入了**重分发器**（Redistributor，每个CPU核独立的中断配置单元）和**中断转换服务**（ITS，Interrupt Translation Service，基于中断ID的动态路由组件）。Redistributor为每个CPU核提供独立的中断配置和状态管理，解决了GICv2全局锁竞争问题；ITS则支持基于中断ID的动态路由，为**PCIe**（Peripheral Component Interconnect Express，高速外设互联标准）**MSI**（Message Signaled Interrupt，消息信号中断）等场景提供硬件级中断翻译。

> 💡 **直观理解**：GICv2像老式总机电话——所有线路接到一个总机台，接线员忙不过来；GICv3像现代呼叫中心——每个客服有独立工位，还能自动转接。

两种架构的核心能力差异可以通过下表对比。

| 能力维度 | GICv2 | GICv3 |
|:---|:---|:---|
| 最大中断数 | 1020（SPI 0-987） | 大于1000（无硬性上限，依赖实现） |
| CPU核数支持 | 最多8核 | 最多256核（或更多） |
| 中断亲和性 | 有限，基于8位目标掩码 | 灵活，基于Affinity Routing |
| 虚拟化支持 | 无硬件虚拟中断注入 | 支持vPE（虚拟处理单元）和List Register |
| **LPI**（Locality-specific Peripheral Interrupt，局部特定外设中断） | 不支持 | 支持（基于ITS动态分配） |
| 典型应用场景 | 嵌入式MCU、早期手机SoC | 服务器、高端移动平台、虚拟化场景 |

表格对比了架构差异，但你的开发板上到底跑的是哪个版本？Linux内核在启动时会通过设备树（Device Tree）解析中断控制器版本，并注册对应的**中断芯片操作集**（`irq_chip`，内核中抽象中断控制器硬件操作的函数表）。

```bash
# 查看设备树中GIC节点信息
$ dmesg | grep -i gic
[    0.000000] GICv3: 224 SPIs implemented
[    0.000000] GICv3: 0 Extended SPIs implemented
[    0.000000] GICv3: CPU0: found redistributor 0 region 0:0x00000000ff841000

# 查看当前中断控制器的irq_chip注册
$ cat /proc/interrupts | head -n 5
           CPU0       CPU1       CPU2       CPU3       
  3:          0          0          0          0     GICv3  3 Level     kbd
  7:          0          0          0          0     GICv3  7 Level     arch_timer
```

由于上述扩展瓶颈，GICv3已成为ARM64服务器的标准配置。但中断控制器只是信号路由的硬件基础，外设发出的中断请求在电气层面还有两种截然不同的触发方式，理解它们的硬件行为是编写可靠中断处理逻辑的前提。

### 小点2：中断触发方式——电平触发与边沿触发的硬件陷阱

外设向中断控制器发送请求时，电气信号的维持方式决定了CPU侧的处理策略。中断控制器内部有一个**挂起状态位**（Pending Bit，记录中断是否等待CPU处理的标志），其置位和清除逻辑因触发方式而异。如果搞不清这两种触发方式的硬件行为差异，驱动代码会出现间歇性中断丢失或中断风暴等难以复现的故障。

**电平触发**（Level-Triggered）的本质是：外设将中断请求线持续拉高（或拉低），直到CPU响应并清除中断源。中断控制器检测到有效电平后，将Pending位置位，CPU开始处理。如果CPU处理完毕后外设仍未撤除电平信号，中断控制器会再次将其置为Pending，导致重复进入**中断服务程序**（ISR，Interrupt Service Routine）。

这种"不撤线就重发"的特性，在ISR处理逻辑不完整时会引发灾难。

假设一个网卡驱动在ISR中只清除了中断控制器侧的Pending位，却忘记读取网卡**先进先出缓冲区**（FIFO，First In First Out，先进先出缓冲区）中的数据包，网卡的中断线仍然保持高电平。CPU退出ISR的瞬间，中断控制器再次检测到有效电平，立即重新触发中断。整个CPU将陷入无限循环，系统表现为**中断风暴**（Interrupt Storm，ISR被无限重入导致的系统假死）。

**边沿触发**（Edge-Triggered）则完全不同：外设仅在电平跳变瞬间（上升沿或下降沿）产生一个脉冲信号，中断控制器捕获这个跳变沿后将其锁存为Pending状态。即使外设随后立即撤除信号，Pending位仍然保持，直到CPU清除。这种"一次跳变只发一次"的特性，避免了电平触发的重入风险。

但边沿触发引入了另一个陷阱：如果跳变沿发生在中断被**屏蔽**（Mask，禁止中断控制器响应该线路）期间，控制器无法捕获这个瞬间脉冲，中断直接丢失。假设CPU正在执行临界区代码，临时关闭了某个**通用输入输出**（GPIO，General Purpose Input Output，通用输入输出）中断；恰在此时，传感器产生了一个极窄的脉冲。由于中断控制器当时处于屏蔽状态，跳变沿没有被锁存，CPU退出临界区后永远收不到这个事件。

两种触发方式的硬件行为差异可以用ASCII时序图直观展示。

```
电平触发（Level-Triggered）时序
  高电平 ─┐        ┌─────────────────────────┐        ┌─
          │        │  中断线持续拉高          │        │
  低电平  └────────┘                         └────────┘
                  ↑                         ↑
              第一次触发                 ISR未清源→再次触发
              
边沿触发（Edge-Triggered）时序
  高电平 ─┐   ┌──┐              ┌──┐
          │   │  │              │  │
  低电平  └───┘  └──────────────┘  └──────────────
              ↑                  ↑
           上升沿触发         屏蔽期间跳变→丢失
```

图示展示了电气行为差异，但Linux内核是如何根据触发方式选择处理策略的？内核为电平触发注册了`handle_level_irq`（电平中断处理函数），为边沿触发注册了`handle_edge_irq`（边沿中断处理函数），两者在**确认**（ack，Acknowledge，通知控制器已收到中断）和**中断结束**（eoi，End of Interrupt，通知控制器可接收新中断）时机上截然不同。

```c
// kernel/irq/chip.c:handle_level_irq() 第 530-560 行（节选）
void handle_level_irq(struct irq_desc *desc)
{
    irq_mask_and_ack_irq(desc);  // 先mask再ack：防止重入
    handle_irq_event(desc);      // 执行ISR
    // 不立即eoi，等待ISR处理完外设，电平自然撤除
}

// kernel/irq/chip.c:handle_edge_irq() 第 500-525 行（节选）
void handle_edge_irq(struct irq_desc *desc)
{
    // 先ack锁存Pending位，再执行ISR
    // 即使ISR执行期间新跳变到来，也会被锁存为新的Pending
    desc->irq_data.chip->irq_ack(&desc->irq_data);
    handle_irq_event(desc);
}
```

电平触发要求ISR必须先处理外设源头再解除屏蔽，边沿触发则允许更早ack以接收同线路的新事件。如果在这两种模式之间混用处理逻辑，系统会出现不可预期的中断行为。

理解了触发方式的硬件行为，下一个关键问题是：外设侧的硬件中断号（如GIC的SPI 123）是如何映射到Linux内核中的软件中断号的？

### 小点3：hwirq、virq与IRQ number——三层映射的解耦设计

在Linux内核中，中断号不是单一整数，而是经过多层抽象的三级映射体系。如果不理解这个映射层次，阅读设备树或调试`/proc/interrupts`时会产生严重混淆——你会看到同一中断在不同上下文中被称为不同的数字。

第一层是**硬件中断号**（`hwirq`，Hardware IRQ Number）。这是中断控制器视角下的物理标识，由芯片手册定义。例如GICv3中，**共享外设中断**（SPI，Shared Peripheral Interrupt，共享外设中断）的`hwirq`范围是32~1019，**私有外设中断**（PPI，Private Peripheral Interrupt，私有外设中断）的范围是16~31。这个号码直接对应中断控制器的寄存器位域，是硬件层面的绝对编号。

第二层是**Linux虚拟中断号**（`virq`，Virtual IRQ Number）。这是内核为每个注册中断分配的全局唯一软件标识，通过`irq_alloc_descs()`（中断描述符分配函数）动态分配。`virq`与`hwirq`之间没有固定对应关系——同一个`hwirq`在不同**SoC**（System on Chip，片上系统）上可能映射到不同的`virq`，内核通过**中断域**（`irq_domain`，管理hwirq到virq映射的内核对象）维护这种映射。

第三层是**设备树中断号**（DT IRQ，Device Tree IRQ Number）。这是设备树节点中`interrupts = <...>`属性使用的号码，通常是`hwirq`经过偏移或类型编码后的值。例如某设备树节点写`interrupts = <0 123 4>`，表示使用SPI类型、`hwirq`为123、触发方式为上升沿的中断。

三层映射的关系可以通过下表厘清。

| 层级 | 名称 | 范围/特性 | 由谁管理 | 典型值示例 |
|:---|:---|:---|:---|:---|
| 第一层 | `hwirq` | 芯片手册定义，固定 | 中断控制器硬件 | SPI 123 |
| 第二层 | `virq` | 内核动态分配，全局唯一 | `irq_domain` + `irq_alloc_descs` | 456（无固定规律） |
| 第三层 | DT IRQ | 设备树属性中的编码值 | 设备树解析器 | `<0 123 4>` |

表格展示了概念分层，但内核是如何在启动时建立这种映射的？当内核解析设备树时，`irq_domain`的**翻译**（`xlate`，Translate，将设备树属性转换为hwirq的回调函数）函数将设备树中的`<type hwirq flags>`三元组转换为`hwirq`，再通过**映射**（`map`，Map，建立hwirq到virq对应关系的回调函数）函数分配对应的`virq`。

```bash
# 查看hwirq与virq的映射关系
$ cat /proc/interrupts | head -n 10
           CPU0       CPU1       
  3:          0          0     GICv3  3 Level     kbd
  7:          0          0     GICv3  7 Level     arch_timer
 19:     234567     234568     GICv3 19 Level     timer
# 第一列是virq，GICv3列后的是hwirq

# 查看特定irq的详细映射
$ cat /proc/irq/19/spurious
count: 0
unhandled: 0
# hwirq 19 映射到 virq 19（此处恰好相同，但非必然）
```

由于设备树抽象和`irq_domain`的引入，驱动开发者无需关心`hwirq`的具体数值，只需使用内核分配的全局`virq`。这种解耦机制使得同一驱动源码可以在不同SoC上复用，而不必硬编码硬件中断号。

> ⚠️ **注意**：`hwirq`与`virq`的数值关系并非固定。在x86上`virq`通常从0开始连续分配，在ARM上可能因设备树解析顺序而出现跳跃。调试时应以`/proc/interrupts`第一列的`virq`为准，而非直接引用芯片手册的`hwirq`。
