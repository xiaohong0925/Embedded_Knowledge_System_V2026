    6.2.2 sysfs接口：用户空间控制GPIO¶

    所属章节：第6章 GPIO与设备树 > 6.2 GPIO子系统基础 难度：[B→I] | 预计阅读时间：25分钟

    本节导读¶

    本节将带你通过Linux最经典的sysfs接口，在用户空间直接控制GPIO引脚——从导出引脚、设置方向到读写电平值，完成一次"点亮LED+读取按键"的完整实战。即使你没有任何驱动开发经验，也能在5分钟内让板子上的灯亮起来。

    知识点1：sysfs GPIO接口原理与基本操作 [B][M] ~1,000字¶

    在Linux内核中，GPIO子系统为用户空间提供了两条访问路径：sysfs接口（传统方式）和字符设备接口（libgpiod，现代推荐方式）。本节先聚焦于sysfs接口，因为它是理解GPIO控制原理的最佳起点——底层做了什么，sysfs一览无余。

    sysfs接口是什么¶

    sysfs是Linux内核提供的一种虚拟文件系统，挂载在 /sys 目录下。它把内核中的设备、驱动、总线等内部结构，以文件和目录的形式"暴露"给用户空间。GPIO子系统也在其中占据了一席之地：/sys/class/gpio/。

    你可以把这个目录想象成GPIO的"前台接待处"。所有关于GPIO的操作，都要先到这里"登记"。

    核心目录与文件一览¶

    进入 /sys/class/gpio/ 后，你会看到以下内容（以树莓派为例）：

    $ ls /sys/class/gpio/
export    gpiochip0    gpiochip100    unexport
    • export：向内核申请某个GPIO引脚的使用权。写入引脚编号，内核就会为你创建对应的控制目录。
    • unexport：向内核归还GPIO引脚，释放资源。
    • gpiochipX：代表一个GPIO控制器（chip），每个控制器管理一组GPIO引脚。

    当你成功 export 一个引脚（比如GPIO22）后，内核会动态创建 /sys/class/gpio/gpio22/ 目录，里面包含：

    $ ls /sys/class/gpio/gpio22/
active_low  direction  edge  power  subsystem  uevent  value
    文件名 权限 功能说明
    direction 读写 设置/获取引脚方向：in（输入）或 out（输出）
    value 读写 读写引脚电平值：0（低电平）或 1（高电平）
    active_low 读写 极性反转：0（正常，1=高电平）、1（反转，1=低电平）
    edge 读写 中断触发方式：none、rising、falling、both

    [表1：sysfs GPIO核心文件速查表]

    GPIO编号的换算：从原理图到sysfs编号¶

    ⚠️ 陷阱：很多初学者第一次操作GPIO就栽在编号不对上。

    开发板上的丝印可能是 GPIO22、P1_15、CON1_22 等各种名字，但sysfs只认一个数字：内核GPIO编号。这个编号与芯片datasheet中的GPIO编号通常一致（如BCM2835的GPIO22就是编号22），但不同的开发板可能有不同的映射关系。

    💡 提示：在原理图或开发板手册中查找 "GPIO Number" 或 "Linux GPIO" 对应的数字。以树莓派为例，BCM编号和sysfs编号是直接对应的——BCM GPIO22就是sysfs的22。

    导出与释放GPIO¶

    导出（export）——向内核申请使用某个GPIO：

    # 将GPIO22导出到用户空间
echo 22 > /sys/class/gpio/export

    执行后，如果成功，会自动出现 /sys/class/gpio/gpio22/ 目录。如果失败（比如编号超出范围），你会在终端或 dmesg 中看到错误信息。

    释放（unexport）——用完后归还内核：

    # 释放GPIO22
echo 22 > /sys/class/gpio/unexport

    执行后，/sys/class/gpio/gpio22/ 目录会消失。这是一个良好的习惯——就像借了图书馆的书要记得归还。

    设置方向与读写值¶

    导出成功后，必须先设置方向，才能读写值。

    设置为输出（点亮LED用）：

    echo out > /sys/class/gpio/gpio22/direction
echo 1 > /sys/class/gpio/gpio22/value   # 输出高电平
echo 0 > /sys/class/gpio/gpio22/value   # 输出低电平

    设置为输入（读取按键用）：

    echo in > /sys/class/gpio/gpio22/direction
cat /sys/class/gpio/gpio22/value        # 读取当前电平，输出0或1

    💡 提示：cat 读取 value 时，如果是输入模式，会返回当前引脚的真实电平；如果是输出模式，则返回你最后一次写入的值（相当于"回读"）。

    sysfs GPIO操作流程¶

    下面这张图展示了从用户空间到内核，一次完整GPIO操作的数据流：

    flowchart TD
    A[用户空间程序] -->|"echo 22 > export"| B[/sys/class/gpio/export\n虚拟文件]
    B -->|"内核GPIO子系统处理"| C{编号合法?}
    C -->|否| D[返回错误\ndmesg可见原因]
    C -->|是| E[创建 /sys/class/gpio/gpio22/\n及相关属性文件]

    H -->|"echo 22 > unexport"| I[内核回收资源\n目录自动删除]
    I --> J[GPIO回到内核管理]

    知识点2：LED控制实战——从命令行到Shell脚本 [B][M] ~900字¶

    理论够了，现在动手。本节以树莓派（或其他任何支持sysfs的开发板）为例，目标是用GPIO22控制一个LED，完成"亮→灭→闪烁"的完整过程。

    硬件准备¶

    你需要： - 一颗LED（任意颜色） - 一个220Ω限流电阻（千万别省！） - 两根杜邦线

    🔴 危险：GPIO引脚通常只能提供约16mA电流，LED直接连接可能烧毁引脚或芯片。限流电阻是必须的。

    连接方式：GPIO22 → 电阻 → LED正极（长脚） → LED负极（短脚） → GND。

    [图2：LED与GPIO连接示意图——GPIO22接220Ω电阻到LED阳极，LED阴极接地]

    命令行逐步操作¶

    打开开发板的终端（SSH或直接连接），按顺序执行：

    第1步：导出GPIO22

    echo 22 > /sys/class/gpio/export

    如果提示 "Permission denied"，说明当前用户没有权限——见知识点3的解决方案。

    第2步：确认目录已创建

    ls /sys/class/gpio/gpio22/

    应该能看到 direction、value 等文件。

    第3步：设置为输出方向

    echo out > /sys/class/gpio/gpio22/direction

    💡 提示：如果LED连接后默认就亮了，可能是引脚默认电平为高。你可以通过 active_low 文件反转逻辑，或者调整硬件连接。

    第4步：点亮LED

    echo 1 > /sys/class/gpio/gpio22/value

    此时LED应该亮起。

    第5步：熄灭LED

    echo 0 > /sys/class/gpio/gpio22/value

    第6步：让LED闪烁

    用shell循环实现一个简单的闪烁效果：

    #!/bin/bash

GPIO=22

# 导出并设置方向
echo $GPIO > /sys/class/gpio/export
echo out > /sys/class/gpio/gpio${GPIO}/direction

# 闪烁10次
for i in $(seq 1 10); do
    echo 1 > /sys/class/gpio/gpio${GPIO}/value
    sleep 0.5
    echo 0 > /sys/class/gpio/gpio${GPIO}/value
    sleep 0.5
done

# 清理
echo $GPIO > /sys/class/gpio/unexport
echo "LED演示结束"

    [代码1：LED闪烁Shell脚本]

    将上述内容保存为 led_blink.sh，然后执行：

    chmod +x led_blink.sh
sudo ./led_blink.sh

    读取按键状态（输入模式实战）¶

    GPIO作为输入的用法同样简单。假设按键一端接GPIO24，另一端接3.3V（按下为高电平）：

    # 导出GPIO24
echo 24 > /sys/class/gpio/export
echo in > /sys/class/gpio/gpio24/direction

# 读取当前状态
cat /sys/class/gpio/gpio24/value
# 按下按键后再次读取
cat /sys/class/gpio/gpio24/value

# 用完释放
echo 24 > /sys/class/gpio/unexport

    ⚠️ 陷阱：如果按键另一端接的是3.3V而非GND，且没有设置内部下拉电阻，那么按键松开时引脚会浮空（电平不确定），导致读取值随机跳变。解决方案是在硬件上加下拉电阻（10kΩ接GND），或在支持 bias-pull-down 的内核版本中使用设备树配置内部上下拉。

    知识点3：sysfs接口的安全限制与权限问题 [B] ~400字¶

    sysfs接口虽然简单易用，但它在设计上有几个安全限制，了解它们能帮你少走很多弯路。

    权限壁垒：为什么总是"Permission denied"¶

    sysfs的GPIO文件默认只有 root用户 或 gpio用户组 成员才能写入。普通用户执行 echo 22 > export 时，几乎都会遇到：

    bash: /sys/class/gpio/export: Permission denied

    解决方案有三种：

    1. 临时方案：命令前加 sudo 

      sudo sh -c 'echo 22 > /sys/class/gpio/export'
      💡 提示：sudo 不能直接重定向，需要用 sh -c 把整个命令包起来，或者 sudo tee。

    2. 推荐方案：将当前用户加入 gpio 组 

      sudo usermod -aG gpio $USER
      然后注销并重新登录，使权限生效。

    3. udev自动化（高级）：编写udev规则，在GPIO导出时自动修改权限。部分发行版（如Raspberry Pi OS）已经预装了这类规则。

    gpiochip的保护机制¶

    不是所有GPIO都能被用户空间随意占用。某些GPIO在内核启动时已被设备树分配给了特定驱动（如I2C、SPI、UART的复用引脚），或者已经被内核内部使用。

    当你尝试 export 这些引脚时，内核会拒绝请求：

    echo 2 > /sys/class/gpio/export
bash: echo: write error: Device or resource busy

    这说明GPIO2已经被其他内核模块占用了。dmesg 可以看到更详细的占用信息。

    ⚠️ 陷阱：不要试图通过sysfs强行导出已被内核驱动使用的GPIO，这可能导致设备冲突甚至系统不稳定。

    sysfs接口的未来¶

    💡 提示：从Linux内核4.8版本开始，社区引入了 gpiochip 字符设备接口（/dev/gpiochipN），配合 libgpiod 库，提供了更现代、更安全的GPIO访问方式。sysfs接口虽然仍被大量设备使用，但官方已标记为弃用（deprecated）——它不是明天就消失，但新开发的项目应该优先考虑 libgpiod。

    sysfs入口 /sys/class/gpio/ 是用户空间控制GPIO的大门 cd /sys/class/gpio/

    下一步¶

    sysfs接口虽然直观，但它有一个先天不足：无法同时原子地操作多个GPIO，也无法方便地利用中断事件。下一节（6.2.3），我们将认识更现代、更强大的 libgpiod 库与字符设备接口——它提供更安全的并发访问、更简洁的API，以及更丰富的中断与事件检测能力。sysfs是历史，libgpiod 是未来。
