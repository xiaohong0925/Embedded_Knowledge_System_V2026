    5.5.3 启动脚本 /etc/init.d/rcS¶

    所属章节：第5章 文件系统与根文件系统 > 5.5 初始化系统与启动脚本 难度：[B→I] | 预计阅读时间：25分钟

    本节导读¶

    本节带你理解嵌入式 Linux 的第一个用户空间启动脚本 rcS 的作用，亲手编写一个能挂载核心文件系统、配置主机名、启动网络的实用版本，并让它真正执行起来。学完后，你将拥有一个可以"开箱即用"的根文件系统初始化流程。

    知识点1：rcS 是什么？为什么要写它？[I] ~1,000字¶

    1.1 rcS 的定位¶

    在上一节（5.5.2）中，我们提到 BusyBox 的 init 程序会读取 /etc/inittab。其中最关键的一行通常是：

    ::sysinit:/etc/init.d/rcS

    这行配置的意思是：在进入任何交互式登录之前，先执行 /etc/init.d/rcS 脚本。换句话说，rcS 是内核完成启动后，用户空间中第一个被执行的脚本，它承担着"把系统从裸机状态带入可用状态"的全部职责。

    可以把 rcS 理解为系统的"早间闹钟"——它负责一项项唤醒系统的各个功能模块：先亮灯（挂载文件系统）、再报名字（设置 hostname）、再打开通信（启动网络）。没有它，系统虽然内核已启动，但用户几乎无法与之交互。

    [图1：rcS 在启动流程中的位置]

    flowchart LR
    A[内核启动完成] --> B[/sbin/init]
    B --> C[读取 /etc/inittab]
    C --> D{找到 sysinit 行}
    D --> E[/etc/init.d/rcS]
    E --> F[挂载 proc/sys/dev]
    F --> G[设置 hostname]
    G --> H[配置网络]
    H --> I[启动 shell / 登录]
    style E fill:#ffcc66,stroke:#333,stroke-width:2px

    1.2 rcS 的三项核心任务¶

    一个最小可用的 rcS 脚本通常需要完成以下工作：

    任务一：挂载伪文件系统（/proc、/sys、/dev）¶

    Linux 内核运行后，会通过三个特殊的"虚拟"文件系统向用户空间暴露内部信息：

    挂载点 文件系统类型 作用 不挂载的后果
    /proc proc 暴露进程信息、系统参数（ps、top 依赖它） ps 命令显示为空
    /sys sysfs 暴露设备树、总线、驱动信息 /sys/class/net 不存在，无法查看网卡
    /dev devtmpfs / tmpfs 提供设备节点（/dev/null、/dev/mtdblock0 等） 打不开任何设备文件

    💡 提示：在较新的内核（3.0+）中，devtmpfs 可以在内核启动时自动挂载到 /dev，但传统做法仍是在 rcS 中显式挂载，以确保兼容性。

    任务二：设置主机名（hostname）¶

    主机名是设备在网络中的"名字标签。默认情况下，BusyBox 启动后主机名为(none)`，命令提示符看起来像这样：

    (none) login:

    通过 hostname 命令可以修改：

    hostname myembedded

    执行后提示符会变成：

    myembedded login:

    💡 提示：主机名也可以通过写入 /proc/sys/kernel/hostname 来设置，效果相同。

    任务三：启动网络接口¶

    没有网络，嵌入式设备就成了"信息孤岛"。rcS 中至少需要：

    1. 启用回环接口 lo：这是本机内部通信的基础，127.0.0.1 依赖它
    2. 配置以太网接口 eth0：分配 IP 地址（静态或 DHCP）
    # 启用回环
ifconfig lo up
ifconfig lo 127.0.0.1

# 配置以太网（静态IP示例）
ifconfig eth0 192.168.1.100 netmask 255.255.255.0

    ⚠️ 陷阱：ifconfig 属于 BusyBox 的精简版本，不支持现代 Linux 中 ip 命令的所有高级特性，但在嵌入式场景中完全够用。如果你的根文件系统中找不到 ifconfig，需要确认 BusyBox 编译时开启了 CONFIG_IFCONFIG。

    知识点2：一步一步编写 rcS 脚本 [I] ~800字¶

    2.1 脚本的基本结构¶

    rcS 本质上是一个 Shell 脚本，通常使用 /bin/sh（在 BusyBox 系统中就是 ash 或 hush）解释执行。一个良好的 rcS 脚本应当：

    • 以 #!/bin/sh 开头（shebang）
    • 打印启动信息，方便串口调试时观察进度
    • 每步完成后做简单的成功/失败检查
    • 末尾给出"启动完成"提示

    2.2 完整 rcS 脚本示例¶

    以下是经过多个实际项目验证的、适合初学者的 rcS 脚本模板。建议逐行阅读注释，理解每行命令的意图。

    #!/bin/sh
# /etc/init.d/rcS - 系统初始化脚本
# 这是BusyBox init执行的第一个用户空间脚本

# 打印启动信息
# ==============
echo "================================"
echo "    Starting system init..."
echo "================================"

# 1. 挂载 /proc 文件系统
# ----------------------
# proc 文件系统是内核向用户空间展示进程和系统信息的窗口
# ps、top、free 等命令都依赖它
echo "Mounting /proc..."
mount -t proc proc /proc
if [ $? -eq 0 ]; then
    echo "  [OK] /proc mounted"
else
    echo "  [FAIL] /proc mount failed"
fi

# 2. 挂载 /sys 文件系统
# ---------------------
# sysfs 暴露设备、总线、驱动等内核对象信息
echo "Mounting /sys..."
mount -t sysfs sysfs /sys
if [ $? -eq 0 ]; then
    echo "  [OK] /sys mounted"
else
    echo "  [FAIL] /sys mount failed"
fi

# 3. 挂载 /dev 文件系统
# ---------------------
# devtmpfs 让内核自动创建设备节点，省去手动 mknod 的麻烦
echo "Mounting /dev..."
mount -t devtmpfs devtmpfs /dev
if [ $? -eq 0 ]; then
    echo "  [OK] /dev mounted"
else
    # 如果 devtmpfs 不支持（老内核），回退到 tmpfs + mdev
    echo "  devtmpfs not available, trying tmpfs..."
    mount -t tmpfs tmpfs /dev
    # 创建基本设备节点（ BusyBox mdev 方式）
    echo /sbin/mdev > /proc/sys/kernel/hotplug
    mdev -s
fi

# 4. 挂载 /tmp 为 tmpfs（可选但强烈推荐）
# ---------------------------------------
# tmpfs 将 /tmp 放在内存中，避免写 Flash，加速临时文件访问
echo "Mounting /tmp..."
mount -t tmpfs tmpfs /tmp

# 5. 设置主机名
# -------------
echo "Setting hostname..."
hostname "myembedded"
echo "  [OK] Hostname set to: $(hostname)"

# 6. 配置网络接口
# ---------------
echo "Configuring network..."

# 6.1 启用回环接口（lo）——必须！很多程序依赖 127.0.0.1
ifconfig lo up
ifconfig lo 127.0.0.1

# 6.2 启用以太网接口并配置静态IP
# 请根据你的实际网络环境修改 IP、掩码、网关
ifconfig eth0 up
ifconfig eth0 192.168.1.100 netmask 255.255.255.0
route add default gw 192.168.1.1

echo "  [OK] Network configured:"
ifconfig | grep -E "lo|eth0" -A 1

# 7. 创建必要的符号链接
# ---------------------
# 某些程序期望 /var/run 和 /var/log 存在
mkdir -p /var/run /var/log /var/lock

# 8. 启动完成
# -----------
echo "================================"
echo "    System init complete!"
echo "    Login with: root (no pwd)"
echo "================================"

    2.3 逐段解读¶

    行号段 内容 为什么要做
    1 #!/bin/sh 声明解释器，让内核知道用哪个程序执行此脚本
    4-7 echo 启动横幅 串口调试时一眼看出 rcS 开始执行了
    10-18 挂载 /proc ps、free、cat /proc/cpuinfo 等命令的前提
    21-29 挂载 /sys 查看网卡名、设备树属性都依赖它
    32-46 挂载 /dev 先有 /dev，才能打开串口、Flash、网卡等设备
    49-52 挂载 /tmp 临时文件放内存，减少 Flash 磨损
    55-58 hostname 设置设备在网络中的名字
    61-75 ifconfig、route 启用网卡、配 IP、设网关，让设备能联网
    78-80 mkdir -p 创建运行时目录，避免程序因找不到路径而报错

    💡 提示：上面的脚本使用了 if [ $? -eq 0 ] 来检查命令是否成功。$? 是 Shell 的特殊变量，保存了上一个命令的退出码。0 表示成功，非 0 表示失败。这是调试 rcS 时最有用的技巧之一。

    🔴 危险：如果你使用 NFS 作为根文件系统，不要在 rcS 中执行 mount -t nfs ... 或重新挂载根目录！这可能导致系统死锁。NFS 根文件系统的挂载由内核启动参数（root=/dev/nfs nfsroot=...）完成，不是 rcS 的职责。

    知识点3：给 rcS 添加执行权限 [B] ~400字¶

    3.1 为什么需要设置权限？¶

    Shell 脚本和普通文本文件的区别就在于执行权限。没有 x 权限，/sbin/init 调用 rcS 时会收到 "Permission denied"（权限拒绝）错误，然后系统初始化就此中断——你可能只会看到内核启动信息，之后没有任何反应。

    3.2 设置执行权限¶

    在宿主机（制作根文件系统时）执行：

    chmod +x /mnt/agents/output/part1_book/ch05/rootfs/etc/init.d/rcS

    或在目标设备上（如果已经有其他方式进入系统）：

    chmod +x /etc/init.d/rcS

    3.3 验证权限是否设置正确¶

    ls -l /etc/init.d/rcS

    正确输出应类似：

    -rwxr-xr-x 1 root root 1567 Jan 15 09:30 /etc/init.d/rcS

    注意最左边的 -rwxr-xr-x——第三个字符 x 表示文件拥有者（root）有执行权限。

    3.4 常见权限问题排查¶

    ⚠️ 错误1：从 Windows 复制脚本后权限丢失

    Windows 文件系统（如 FAT32/exFAT）不支持 Linux 的权限位。如果根文件系统是在 Windows 下解压或编辑的，所有文件默认权限会变成 644（无执行权限）。 💡 解决方法：在 Linux 环境下重新执行 chmod +x，或者将根文件系统打包为 tar 格式（能保留权限）再传输。

    ⚠️ 错误2：权限设置了，但仍然提示 "not found"

    可能是 shebang 行指定的解释器不存在。检查 /bin/sh 是否真实存在： 

    ls -l /bin/sh
    如果 /bin/sh 指向 busybox 但 busybox 未启用 ash/hush shell，脚本也无法执行。 💡 解决方法：确保 BusyBox 配置中启用了 CONFIG_ASH 或 CONFIG_HUSH。

    ⚠️ 错误3：脚本有执行权限，但行尾带有 \r（Windows 换行符）

    Windows 使用 \r\n 作为换行，而 Linux 只用 \n。\r 会被当作命令名的一部分，导致 shebang 解析失败。 💡 解决方法：用 dos2unix 转换，或在 Vim 中执行 :set fileformat=unix 后保存。

    本节总结¶

    概念 要点 操作
    rcS 定位 用户空间第一个执行的脚本，由 init 的 sysinit 触发 在 /etc/inittab 中配置 ::sysinit:/etc/init.d/rcS
    挂载 /proc 暴露进程和系统信息 mount -t proc proc /proc
    挂载 /sys 暴露设备和驱动信息 mount -t sysfs sysfs /sys
    挂载 /dev 创建设备节点 mount -t devtmpfs devtmpfs /dev
    设置 hostname 设备在网络中的名字标签 hostname myembedded
    配置网络 启用 lo 和 eth0，分配 IP ifconfig lo 127.0.0.1; ifconfig eth0 ...
    执行权限 脚本必须带 x 权限才能被 init 调用 chmod +x /etc/init.d/rcS
    换行符 Windows \r\n 会导致脚本解析失败 用 dos2unix 转换为 Unix 格式

    本节我们从原理到实践，完成了一个嵌入式 Linux 系统启动脚本 rcS 的编写。你现在拥有的不仅是一个能用的脚本，更是一套可扩展的框架——后续需要添加自启动应用程序、加载内核模块、校准触摸屏等操作时，只需要在这个脚本的末尾追加相应的命令即可。
