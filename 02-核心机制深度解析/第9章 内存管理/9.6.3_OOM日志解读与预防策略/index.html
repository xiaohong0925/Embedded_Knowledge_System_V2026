    9.6.3 OOM 日志解读与预防策略¶

    OOM 发生了，dmesg 跳出来一堆信息。那堆数字到底在说什么？我来逐行翻译。

    知识点 133：dmesg 中 OOM killer 输出的逐行解读 [I][M] ⏱️15min¶

    当 OOM killer 被触发后，内核会在 dmesg（内核环形缓冲区）中输出一段结构化的日志。这段日志是事后排查 OOM 根因的唯一现场证据，每一行都有明确的含义。以下是一段典型的 OOM 日志，我们逐行拆解：

    [  123.456] Out of memory: Kill process 2718 (python3) score 912 or sacrifice child
[  123.457] Killed process 2718 (python3) total-vm:5242880kB, anon-rss:3670016kB, file-rss:8192kB, shmem-rss:0kB
[  123.459] oom_reaper: reaped process 2718 (python3), now anon-rss:0kB, file-rss:0kB, shmem-rss:0kB

    第一行：决策宣告¶

    Out of memory: Kill process 2718 (python3) score 912 or sacrifice child

    这一行是 OOM killer 的"判决书"，包含四个关键信息：

    • 2718：被选中的牺牲进程的 PID。这是 OOM killer 遍历所有进程后，根据 oom_score 选出的"头号目标"。
    • (python3)：进程名。注意这里显示的是进程的可执行文件名，不是命令行参数。如果一个 Java 进程显示为 (java)，你无法直接看出它是哪个服务实例。
    • score 912：该进程的 oom_score 最终值（范围 0~1000）。这个分数越高，越容易被选中。912 意味着该进程占用了系统绝大部分内存，几乎是"必选"。
    • or sacrifice child：这是一句内核的"黑色幽默"——如果内核找不到合适的独立进程来杀，它会考虑杀掉子进程（线程组中的某个线程）来缓解压力。实际场景中很少触发。

    第二行：内存详情¶

    Killed process 2718 (python3) total-vm:5242880kB, anon-rss:3670016kB, file-rss:8192kB, shmem-rss:0kB

    这一行列出了被杀进程的四类内存指标，是反推根因的核心数据：

    指标 全称 含义
    total-vm Total Virtual Memory 进程的虚拟地址空间总量，包括已映射但未实际分配的内存。这个数字通常远大于实际物理内存占用
    anon-rss Anonymous RSS 匿名驻留集——进程的堆、栈、brk/mmap 分配的私有可写内存，无文件 backing。这是导致 OOM 的元凶指标
    file-rss File-backed RSS 文件映射驻留集——可执行文件代码段、共享库、通过 mmap 映射的文件页。这部分可以被内核回收（丢弃后从磁盘重读）
    shmem-rss Shared Memory RSS 共享内存驻留集——tmpfs、POSIX 共享内存、IPC 共享内存。这部分会计入进程 RSS，但可能被多个进程共享

    从上面的例子可以看出，anon-rss:3670016kB（约 3.5GB）占总 RSS 的绝大部分，说明这是一个内存密集型应用（如数据分析脚本、ML 推理进程），其堆内存暴涨导致了 OOM。file-rss 只有 8MB，说明它不是加载了大量文件缓存；shmem-rss 为 0，排除了 tmpfs 泄漏的可能。

    第三行：oom_reaper 回收确认¶

    oom_reaper: reaped process 2718 (python3), now anon-rss:0kB, file-rss:0kB, shmem-rss:0kB

    oom_reaper 是内核 4.7+ 引入的一个专门线程，负责异步回收被杀进程的内存。被杀进程不会立即释放全部内存（其页表和 VMA 需要逐项清理），oom_reaper 会快速回收那些可以直接丢弃的页面，将 RSS 清零。这一行确认回收完成——如果这里长时间不出现，说明进程可能陷入 D 状态（不可中断睡眠），内存回收被阻塞。

    OOM 日志前的内存状态¶

    在"Out of memory"行之前，内核还会输出系统级别的内存快照，格式如下：

    [  123.455] Mem-Info:
[  123.455] active_anon:1234 inactive_anon:5678 isolated_anon:0
[  123.455] active_file:890 inactive_file:123 isolated_file:0
[  123.455] unevictable:0 dirty:12 writeback:0
[  123.455] slab_reclaimable:2345 slab_unreclaimable:3456
[  123.455] Node 0 DMA free:1236kB min:44kB low:56kB high:68kB

    这些信息反映了 OOM 触发瞬间的全局内存状态：

    • active_anon / inactive_anon：活跃与非活跃的匿名页。如果 inactive_anon 很低，说明匿名页几乎都在被使用，swap 回收空间很小。
    • active_file / inactive_file：文件页缓存。如果两者都接近 0，说明 page cache 早已被耗尽。
    • dirty / writeback：脏页和正在回写的页。如果 dirty 很高，可能说明磁盘 I/O 瓶颈导致回收速度跟不上分配速度。
    • slab_reclaimable / slab_unreclaimable：slab 内存。如果 slab_reclaimable 很高但 free 很低，说明内核对象缓存占用了大量可回收内存但尚未被回收。
    • free / min / low / high：空闲内存与三条水位线。如果 free 已经低于 min，说明系统早已进入 direct reclaim 的紧急状态。

    从 OOM 日志反推根因¶

    拿到一段 OOM 日志后，按以下思路分析：

    第一步：看被杀进程是谁。如果是业务进程（如 (python3)、(java)、(nginx)），说明用户空间内存耗尽；如果是内核线程（如 (kworker)、(kswapd)），说明问题更底层，可能是内核内存泄漏或 slab 爆炸。

    第二步：看 anon-rss 占比。如果 anon-rss 占总 RSS 的 80% 以上，根因是进程堆内存暴涨——检查内存泄漏、不合理的缓存策略、未做限流的请求处理。

    第三步：看 file-rss 和 shmem-rss。如果 file-rss 极高，可能是 mmap 了大文件且没有 madvise 释放；如果 shmem-rss 高，检查 /dev/shm 和 tmpfs 的使用。

    第四步：看系统内存状态中的 free 和 slab。如果 free 极低而 slab_reclaimable 很高，说明内核的 slab 缓存（dentry、inode）占据了大量可回收空间，可以通过 echo 3 > /proc/sys/vm/drop_caches 临时缓解，并检查文件系统操作是否过于频繁。

    第五步：看是否有足够 swap。如果系统没有配置 swap，匿名页无法被换出，OOM 会被大大加速。在嵌入式设备上，这是一个非常常见的设计缺陷。

    本节要点回顾¶

    • OOM 日志第一行是"判决书"，包含 PID、进程名和 oom_score
    • 第二行的四列内存指标中，anon-rss 是排查重点，它代表无文件 backing 的堆内存
    • oom_reaper 行确认内存是否被成功回收，长时间不出现说明回收受阻
    • 日志前的 Mem-Info 反映系统全局状态，free 低于 min 水位意味着系统早已进入紧急回收
    • 反推根因的五步法：看进程 → 看 anon-rss → 看 file/shmem → 看 slab → 看 swap

    知识点 134：OOM 三级预防体系 [I] ⏱️10min¶

    OOM 日志解读是事后分析，生产环境更需要事前预防。业界成熟的 OOM 防御体系分三个层级，每层在不同时间窗口介入，形成纵深防御。

    第一层：监控——发现早期信号¶

    监控层在 MemAvailable 跌破阈值时告警。核心指标选 /proc/meminfo 中的 MemAvailable——它估算的是不触发 swap 前提下可立即分配的内存量，考虑了 page cache 和可回收 slab 的贡献。相比之下 MemFree 常误导，可能很低但系统仍有大量可回收缓存。

    推荐阈值：通用服务器低于总内存 15%，嵌入式设备低于 10%，K8s 节点低于 20%。监控层只"看见"问题不解决问题，深夜或高峰时人工响应可能来不及。

    第二层：预警——趋势分析预判¶

    预警层在内存耗尽前数分钟到数十分钟介入。两个关键数据源：

    • /proc/slabinfo：追踪 slab 增长趋势。若某个 slab（如 dentry、inode）持续线性增长，说明内核端存在泄漏。
    • /proc/buddyinfo：追踪伙伴系统碎片。若低阶页面持续减少而高阶请求增加，系统正走向碎片型 OOM。

    实践中定期采样并做线性拟合，连续多点单调下降时触发预警——如清理 page cache 或通知业务降载。

    第三层：自动处理——用户空间工具提前介入¶

    核心思想是在内核 OOM killer 触发前，用户空间工具主动杀掉内存大户，避免进入不可控状态。

    对比维度 earlyoom oomd (Meta) no-oom
    项目定位 轻量级提前 OOM killer 企业级策略引擎 极简嵌入式工具
    依赖要求 几乎零依赖（仅需 /proc） 需 cgroup v2 + Python 3.8+ 零依赖，纯 C
    决策依据 MemAvailable + SwapFree 百分比 PSI 指标 + cgroup 内存统计 + 可编程策略 可配置的内存阈值
    可配置性 中（命令行参数） 极高（JSON 策略文件，规则引擎） 低（仅阈值和信号）
    PSI 支持 否 是（核心依赖） 否
    cgroup 感知 否 是（深度集成 cgroup v2） 否
    资源开销 极低（< 1MB RSS） 中等（Python 运行时） 极低（几十 KB）
    通知机制 SIGTERM/SIGKILL + 可选脚本 D-Bus 信号 + 自定义 action SIGTERM/SIGKILL
    适用场景 桌面/小型服务器 大规模容器集群 极度受限的嵌入式设备
    发行版集成 Fedora 默认启用 Debian/Ubuntu 可选 需手动编译

    earlyoom 在 MemAvailable 与 SwapFree 低于阈值时扫描 VmRSS，向最大进程发 SIGTERM，一个二进制即可运行。

    oomd（Facebook/Meta）基于 PSI 指标决策，支持复杂策略——如"先杀非核心 cgroup 副本保留主节点"，但依赖 cgroup v2。

    no-oom 只做一件事：内存低于阈值时按 RSS 排序杀进程，几十 KB 可烧进 initramfs。

    核心原则：内核 OOM killer 是"核武器"，选择谁死完全由内核决定；用户空间工具把决策权交还给你，在触发前可控地释放内存。

    本节要点回顾¶

    • OOM 预防分三层：监控（看）→ 预警（算）→ 自动处理（杀）
    • 监控核心指标是 MemAvailable，不是 MemFree
    • /proc/slabinfo 和 /proc/buddyinfo 趋势分析可预判内核层面内存危机
    • earlyoom 轻量通用，oomd 功能强但依赖 cgroup v2，no-oom 极简适合嵌入式
    • 用户空间工具在 OOM killer 触发前可控地释放内存
