    9.1.2 ARM64页表结构¶

    主题：ARM64的4级页表结构与页表项标志位详解

    你知道ARM64为什么叫"AArch64"吗？因为地址是64位的——但虚拟地址真的用了64位吗？答案是：没有，只用了48位。那剩下的16位去哪了？原来，ARM64架构采用了一种折中方案：虚拟地址的有效位宽为48位（可在内核配置中扩展至52位），可寻址256TB的地址空间。这48位中，高16位用于区分用户空间与内核空间：当高16位全为0时，表示用户空间；当高16位全为1时，表示内核空间。这种设计被称为"地址标记"（Address Tagging），它不仅简化了地址转换，还为ARM64的MTE（Memory Tagging Extension）内存标记扩展预留了空间。

    知识点103：ARM64的4级页表（PGD→PUD→PMD→PTE）¶

    [I][M] 掌握ARM64的4级页表结构及每级索引的计算方式，理解TTBR0/TTBR1的作用。

    页表的层级架构¶

    ARM64采用4级页表结构，自上而下依次为： PGD（Page Global Directory，全局页目录）→ PUD（Page Upper Directory，上级页目录）→ PMD（Page Middle Directory，中级页目录）→ PTE（Page Table Entry，页表项）。

    每一级页表占用9位作为索引，可寻址512个条目。4级共使用36位索引，加上12位的页内偏移（对应4KB页面），恰好覆盖48位虚拟地址空间。这种设计在寻址能力与页表存储开销之间取得了精妙平衡。

    💡 思考题：为什么每级恰好是9位而不是8位或10位？ 因为页表项大小为8字节（64位），512项 × 8字节 = 4096字节 = 一页。这样每级页表恰好占用一个物理页面，便于内存管理。

    虚拟地址分解¶

    对于48位虚拟地址，其分解如下：

    字段 位数 说明
    页内偏移 [11:0]，12位 4KB页内偏移
    PTE索引 [20:12]，9位 指向PTE表中的512项之一
    PMD索引 [29:21]，9位 指向PMD表中的512项之一
    PUD索引 [38:30]，9位 指向PUD表中的512项之一
    PGD索引 [47:39]，9位 指向PGD表中的512项之一

    地址转换的过程如同一场精密的多级查表：CPU从虚拟地址中提取各级索引，逐级遍历页表，最终从PTE中获取物理页框号，再与页内偏移拼接得到物理地址。

    TTBR0与TTBR1：双页表基址寄存器¶

    ARM64配备两个页表基址寄存器，用于区分用户空间与内核空间：

    • TTBR0_EL1（Translation Table Base Register 0）：存放用户空间的PGD基地址。当CPU访问高16位全为0的虚拟地址时，MMU自动使用TTBR0进行地址转换。

    • TTBR1_EL1（Translation Table Base Register 1）：存放内核空间的PGD基地址。当CPU访问高16位全为1的虚拟地址时，MMU自动使用TTBR1。

    这种双基址设计使得用户进程与内核拥有独立的页表，进程切换时只需更新TTBR0，而TTBR1保持不变（所有进程共享内核页表），极大地提升了上下文切换效率。

    🔍 内核视角：在Linux内核中，current->mm->pgd指向该进程的PGD基址。进程切换时，cpu_switch_mm宏会将新进程的PGD写入TTBR0。

    页表项结构代码¶

    以下代码展示了ARM64 Linux内核中页表项的数据类型与关键操作：

    /* include/asm/pgtable-types.h */
/* 页表项使用64位无符号整数表示 */
typedef u64 pteval_t;
typedef u64 pmdval_t;
typedef u64 pudval_t;
typedef u64 pgdval_t;

typedef struct { pteval_t pte; } pte_t;
typedef struct { pmdval_t pmd; } pmd_t;
typedef struct { pudval_t pud; } pud_t;
typedef struct { pgdval_t pgd; } pgd_t;

/* arch/arm64/include/asm/pgtable.h */
/* 从虚拟地址提取各级索引 */
#define pgd_index(addr)     (((addr) >> 39) & 0x1FF)
#define pud_index(addr)     (((addr) >> 30) & 0x1FF)
#define pmd_index(addr)     (((addr) >> 21) & 0x1FF)
#define pte_index(addr)     (((addr) >> 12) & 0x1FF)

/* 页表项标志位掩码 */
#define PTE_VALID       (_AT(pteval_t, 1) << 0)   /* 有效位 */
#define PTE_TYPE_PAGE   (_AT(pteval_t, 3) << 0)   /* 页描述符类型 */
#define PTE_TABLE_BIT   (_AT(pteval_t, 1) << 1)   /* 下一级表指针 */
#define PTE_ATTRINDX(t) (_AT(pteval_t, (t)) << 2) /* MAIR索引 */
#define PTE_NS          (_AT(pteval_t, 1) << 5)   /* 非安全位 */
#define PTE_AP2         (_AT(pteval_t, 1) << 7)   /* AP[2] */
#define PTE_SH(s)       (_AT(pteval_t, (s)) << 8) /* 共享属性 */
#define PTE_AF          (_AT(pteval_t, 1) << 10)  /* 访问标志 */
#define PTE_NG          (_AT(pteval_t, 1) << 11)  /* 非全局位 */
#define PTE_PXN         (_AT(pteval_t, 1) << 53)  /* 特权执行禁止 */
#define PTE_UXN         (_AT(pteval_t, 1) << 54)  /* 用户执行禁止 */

/* 各级页表操作宏 */
#define pmd_none(pmd)       (!pmd_val(pmd))
#define pmd_table(pmd)      ((pmd_val(pmd) & PMD_TYPE_MASK) == PMD_TYPE_TABLE)
#define pte_present(pte)    (pte_val(pte) & PTE_VALID)
#define pte_page(pte)       (pfn_to_page(__pa(pte_val(pte)) >> PAGE_SHIFT))

    大页（Huge Page）支持¶

    ARM64页表支持多级大页映射，减少对TLB的压力：

    页表级别 映射粒度 覆盖虚拟地址范围
    PUD条目 1GB大页 [38:30]直接指向1GB块
    PMD条目 2MB大页 [29:21]直接指向2MB块
    PTE条目 4KB基础页 [20:12]指向4KB页

    当页表项的type字段指示"块映射"（Block Descriptor）时，该条目直接指向一段连续的物理内存，无需继续遍历下一级页表。Linux内核的透明大页（THP）机制正是利用这一硬件特性，自动将连续的4KB页面提升为2MB大页。

    ⚠️ 注意：实际的大页尺寸依赖于内核配置和处理器实现。部分ARM64处理器支持16KB或64KB基础页大小，对应的大页尺寸也会相应变化。

    知识点104：页表项标志位速查¶

    [I] 理解页表项中关键标志位的含义及其在用户态工具中的映射。

    AF（Access Flag，访问标志）¶

    AF位（bit[10]）记录页面是否被访问过。当硬件首次访问某页面且AF为0时，触发权限故障。Linux内核采用延迟设置策略：首次访问触发page fault后，内核置位AF并重新执行指令。配置CONFIG_ARM64_HW_AFDBM可启用硬件自动更新AF，进一步降低page fault开销。

    AP（Access Permission，访问权限）¶

    AP字段（bits[7:6]）控制页面的读写权限：

    AP[2] AP[1] 内核访问 用户访问 典型用途
    0 1 读写 无权限 内核空间（vmalloc区域）
    0 0 读写 只读 COW共享页面
    1 1 读写 读写 用户数据/堆栈
    1 0 只读 只读 共享库代码段

    AP[1]对应PTE_USER，AP[2]对应PTE_RDONLY。在/proc/pid/maps中，r对应读权限，w对应写权限，p/s对应私有/共享映射。

    UXN/PXN（执行权限）¶

    ARM64通过两级执行权限控制实现DEP（Data Execution Prevention）：

    • PXN（bit[53]）：置位时内核态（EL1）不可执行该页。防止ret2usr攻击。
    • UXN（bit[54]）：置位时用户态（EL0）不可执行该页。实现数据段不可执行。
    场景 UXN PXN maps显示
    用户代码段 0 1 r-xp
    用户数据段 1 1 rw-p
    内核代码段 1 0 内核空间不显示

    🛡️ 安全视角：PXN是ARM64防御ret2usr攻击的关键机制。旧架构（如ARMv7）缺少PXN，需依赖高端内存布局等其他手段隔离内核与用户空间。

    SH（Shareability，共享属性）¶

    SH字段（bits[9:8]）控制缓存一致性策略：

    SH值 共享属性 适用场景
    00 非共享 单核专用内存
    10 外部共享 多核共享、设备内存
    11 内部共享（默认） 同簇CPU共享

    Inner Shareable确保同簇核心间缓存一致，Outer Shareable扩展至多簇或外部总线主设备。DMA缓冲区常配置为Outer Shareable，确保设备与CPU看到一致的数据。

    从页表标志到/proc/pid/maps¶

    /proc/pid/maps的权限列是页表标志位的用户态投影：

    55a3c2d6b000-55a3c2d6d000 r-xp 00000000 08:01 1310734    /usr/bin/cat
55a3c2f6c000-55a3c2f6d000 r--p 00001000 08:01 1310734    /usr/bin/cat
55a3c2f6d000-55a3c2f6e000 rw-p 00002000 08:01 1310734    /usr/bin/cat

    r-xp对应代码段（UXN=0, PXN=1, PTE_RDONLY=1），rw-p对应数据段（UXN=1, PXN=1, PTE_RDONLY=0）。mprotect修改这些标志位动态调整权限，是JIT编译器（如JVM、JS引擎）实现代码生成与执行的基础。

    这些标志位的组合体现了ARM64权限设计的灵活性：读、写、执行权限独立控制，内核与用户态执行权限分离，共享属性精细可调。

    🔍 调试技巧：通过/proc/kpageflags结合/proc/pid/pagemap，可查看每页的标志位（AF是否置位、是否为脏页等）。
