    7.5.4 rest_init与kernel_init：1号进程的诞生¶

    所属：第7章 内核启动流程深度解析 > 7.5 从start_kernel到用户空间 难度：[I→E] | 预计阅读时间：45分钟

    本节导读¶

    start_kernel() 执行了数百项初始化后，终于将控制权交给 rest_init()。这一节追踪从内核初始化到1号进程诞生的最后一公里：PID=1 的 kernel_init 线程如何被创建？它如何从内核线程"蜕变"为用户空间的 /sbin/init？kthreadd (PID=2) 又扮演什么角色？通过将启动日志与内核源码逐行对照，你会获得一种"打开黑盒"的能力——看到每一行启动信息背后精确的代码路径。

    知识点1：rest_init() — start_kernel的终点，新世界的起点 [E] ~1200字¶

    问题场景¶

    当你在内核启动日志中看到：

    [    0.000000] Run /sbin/init as init process

    你有没有想过：这条日志是在哪里打印的？init 进程是如何被创建出来的？ start_kernel() 本身运行在哪个上下文中？如果它结束了，系统会怎样？

    这些问题的答案全部藏在 rest_init() 中。

    机制深入¶

    start_kernel的最后一步¶

    start_kernel() 的末尾调用 rest_init()，然后永不返回。实际上，start_kernel() 运行在初始化上下文（早期的内核线程等价物）中，而 rest_init() 负责将这个原始上下文转化为真正的调度实体，并创建两个关键的内核线程：

    线程函数 PID 最终身份 职责
    kernel_init 1 /sbin/init (用户进程) 所有用户进程的祖先，负责启动用户空间
    kthreadd 2 内核线程 (永不离核) 所有后续内核线程的父进程，管理 kthread_create_list

    🔴 安全提醒：PID=1 的进程是整个系统的信任根。如果 /sbin/init 被恶意替换，攻击者将以 root 权限控制整个系统的进程生命周期。Secure Boot 链的关键验证目标就是确保 init 程序的可信性。

    rest_init() 核心代码¶

    /* init/main.c - Linux 6.6 */
noinline void __init_refok rest_init(void)
{
    struct task_struct *tsk;
    int pid;

    /* (1) 将当前上下文设为 PID=0 的 idle 进程 */
    rcu_read_lock();
    tsk = current;                    // 当前就是 start_kernel 的执行上下文
    tsk->flags |= PF_NO_SETUID;       // idle 进程不允许 setuid
    tsk->flags |= PF_IDLE;            // 标记为 idle task
    rcu_read_unlock();

    /* (2) 关键：启用调度器，从此刻起可以抢占 */
    schedule_preempt_disabled();

    /* (3) 创建 PID=1 的 kernel_init 线程 */
    pid = kernel_thread(kernel_init, NULL, CLONE_FS);
    /* ... 错误处理 ... */
    rcu_read_lock();
    init_task_pid = find_task_by_pid_ns(pid, &init_pid_ns);
    set_cpus_allowed_ptr(init_task_pid, cpumask_of(smp_processor_id()));
    rcu_read_unlock();

    /* (4) 创建 PID=2 的 kthreadd 线程 */
    pid = kernel_thread(kthreadd, NULL, CLONE_FS | CLONE_FILES);
    /* ... */

    /* (5) 关键：从此当前上下文成为真正的 idle 进程 (PID=0) */
    init_idle_bootup_task(current);
    schedule_preempt_disabled();

    /* (6) idle 循环：当没有其他可运行进程时执行 */
    cpu_startup_entry(CPUHP_ONLINE);
}

    关键设计：为什么需要 idle 进程 (PID=0)？¶

    PID=0 的 idle 进程是每个 CPU 都必须有的保底上下文。当 CPU 没有可运行的任务时，调度器必须有一个合法的 task_struct 来切换过去。rest_init() 的精妙之处在于：它将 start_kernel() 的执行上下文"改造"为 idle 进程，而不是创建一个新的。这是最小化设计——复用已有的内核栈和 task_struct。

    kthreadd 的设计意图¶

    kthreadd (PID=2) 的存在解决了一个关键问题：内核线程不能从任意上下文创建。当你调用 kthread_create() 时，实际的创建工作是通过 kthread_add_create_queue() 将请求放入 kthread_create_list，然后由 kthreadd 在安全的进程上下文中完成 do_fork()。

    /* kernel/kthread.c */
int kthreadd(void *unused)
{
    struct kthread_create_info *create;

    set_task_comm(tsk, "kthreadd");
    for (;;) {
        set_current_state(TASK_INTERRUPTIBLE);
        if (list_empty(&kthread_create_list))
            schedule();           // 无请求时睡眠

        __set_current_state(TASK_RUNNING);
        while (!list_empty(&kthread_create_list)) {
            create = list_entry(kthread_create_list.next,
                                struct kthread_create_info, list);
            list_del_init(&create->list);
            create_kthread(create);  // 实际执行 do_fork()
        }
    }
    return 0;
}

    Trade-off：内核线程创建的两种模式¶

    模式 调用方式 实现机制 优点 缺点 适用场景
    直接创建 kernel_thread() 直接调用 do_fork() 简单、同步、立即可用 必须在进程上下文中调用；中断/原子上下文禁止 早期初始化（如 kernel_init、kthreadd 本身）
    委托创建 kthread_create() → kthreadd 通过 kthread_create_list 异步委托 可从任意上下文调用（包括中断上下文） 异步、需要等待 kthreadd 调度 运行时动态创建内核线程（驱动、子系统）

    ⚠️ 常见陷阱¶

    陷阱1：误以为 start_kernel() 返回后系统崩溃

    start_kernel() 调用 rest_init() 后确实有一条 BUG() 语句（保险机制），但实际上执行流通过 rest_init() → cpu_startup_entry() 进入 idle 循环，永远不会回到 start_kernel()。

    陷阱2：kernel_thread() 和 kthread_create() 混用

    kernel_thread() 是早期初始化阶段的"裸"接口，创建的线程需要自己管理退出；kthread_create() 是规范的子系统接口，提供统一的 kthread_should_stop() 机制。

    知识点2：kernel_init线程 — 从内核态到用户空间的蜕变 [E] ~1800字¶

    问题场景¶

    kernel_init 开始时是一个内核线程——它运行在内核地址空间，拥有内核栈，没有用户空间映射。但最终它要变成 /sbin/init——一个用户进程，拥有自己的虚拟地址空间，执行 ELF 文件中的代码。

    这个"蜕变"是如何完成的？ 如果失败会怎样（嵌入式系统常见的 init 崩溃场景）？

    机制深入¶

    kernel_init 的两阶段执行¶

    kernel_init() 分为两个阶段：

    kernel_init()
    ├── kernel_init_freeable()     // 第一阶段：可延迟的内核初始化
    │       ├── wait_for_completion(&kthreadd_done)   // 确保 kthreadd 就绪
    │       ├── kernel_init_freeable()                // 大量子系统初始化
    │       └── try_to_run_init_process("/sbin/init") // 尝试执行 init
    └── 如果上述失败 → 尝试多个 fallback init 路径

    第一阶段：kernel_init_freeable() 的关键工作¶

    /* init/main.c */
static int __ref kernel_init(void *unused)
{
    int ret;

    kernel_init_freeable();           // 完成剩余的内核初始化

    /* ...  various fallback attempts ... */

    /* 尝试执行用户空间 init 程序 */
    if (execute_command) {
        ret = run_init_process(execute_command);
        /* ... */
    }
    /* fallback 链: /sbin/init → /etc/init → /bin/init → /bin/sh */
    /* ... */
    panic("No working init found.");  // 全部失败 → panic
}

    kernel_init_freeable() 内部执行了大量必须在进程上下文中进行的初始化：

    初始化项 关键函数 为什么必须在进程上下文中？
    等待 kthreadd 就绪 wait_for_completion(&kthreadd_done) 需要可睡眠等待
    多处理器初始化 smp_init() 需要为每个 CPU 创建 idle 进程
    驱动模型延迟初始化 driver_init() → deferred_probe_initcall() 部分 probe 需要进程上下文睡眠
    打开控制台设备 console_on_rootfs() 需要在 rootfs 上执行文件操作
    释放初始化内存 free_initmem() 需要修改页表映射
    执行 init try_to_run_init_process() 需要调用 do_execve()

    第二阶段：从内核线程到用户进程的蜕变¶

    这是整个启动链中最关键的转换。核心函数是 run_init_process()：

    /* init/main.c */
static int run_init_process(const char *init_filename)
{
    argv_init[0] = init_filename;
    return do_execve(getname_kernel(init_filename),
                     (const char __user *const __user *)argv_init,
                     (const char __user *const __user *)envp_init);
}

    do_execve() 是系统调用的内核实现，它执行以下关键操作：

    1. 释放旧地址空间：exit_mm() 释放当前内核线程的 mm_struct（内核线程的 mm 指向 init_mm）
    2. 加载 ELF 可执行文件：search_binary_handler() → load_elf_binary()
    3. 建立新地址空间：为新进程创建完整的用户虚拟内存映射（代码段、数据段、BSS、堆栈）
    4. 切换执行上下文：设置新进程的入口点（ELF 的 e_entry），用户态寄存器状态
    5. 返回用户空间：do_execve() 返回时，执行流已经在新进程的上下文中，从用户态入口点开始

    💡 技巧：kernel_init 调用 do_execve() 是一个自替换的过程——同一个 task_struct（PID=1）从内核线程变成了用户进程。pid 不变，task_struct 复用，但 mm、fs、files 全部被替换。

    Fallback 链：当 /sbin/init 不存在时¶

    嵌入式系统中常见的陷阱：rootfs 中缺少 init 程序。内核为此设计了 fallback 链：

    /* init/main.c - kernel_init_freeable() 尾部 */
    if (execute_command) {
        ret = run_init_process(execute_command);
        if (!ret)
            return 0;
        panic("Requested init %s failed (error %d).", execute_command, ret);
    }
    /* 依次尝试以下路径 */
    if (!run_init_process("/sbin/init") ||
        !run_init_process("/etc/init")  ||
        !run_init_process("/bin/init")  ||
        !run_init_process("/bin/sh"))
        return 0;

    panic("No working init found.  Try passing init= option to kernel."
          "See Linux Documentation/admin-guide/init.rst for guidance.");
    Fallback 路径 典型场景 常见失败原因
    init= 参数指定 自定义 init（如 systemd、busybox） 路径拼写错误、文件权限不足
    /sbin/init SysVinit、systemd 的标准位置 rootfs 未正确挂载、init 未包含在 rootfs 中
    /etc/init 早期 Unix 兼容 rootfs 中不存在
    /bin/init 备用位置 同 /sbin/init
    /bin/sh 紧急恢复 shell 连 shell 都不存在 → 必然 panic

    🔴 安全提醒：如果系统 fallback 到 /bin/sh 且没有密码保护，攻击者可直接获得 root shell。嵌入式产品必须在生产环境中禁用这一 fallback（通过配置确保 /sbin/init 始终存在）。

    ⚠️ 常见陷阱¶

    陷阱1：嵌入式开发中 "No working init found" panic

    这是最常见的嵌入式 bring-up 失败。排查清单：

    1. cat /proc/cmdline 检查是否传递了错误的 init= 参数
    2. 确认 rootfs 已正确挂载（mount 命令查看根文件系统）
    3. 确认 init 程序存在且具有可执行权限：ls -l /sbin/init
    4. 如果是静态链接的 busybox，检查 ELF 架构是否与目标 CPU 匹配（file /sbin/init）
    5. 检查 init 依赖的动态库是否存在（ldd /sbin/init，如适用）

    陷阱2：混淆 kernel_thread() 返回值

    kernel_thread() 返回的是子进程的 PID，不是线程函数的返回值。很多开发者误以为它像 pthread_create() 那样返回线程句柄。

    陷阱3：kernel_init 在 do_execve() 之前的 printk 看不到

    kernel_init 线程中调用 printk() 的输出会进入内核日志缓冲区，但如果控制台尚未初始化（console_on_rootfs() 之前），这些消息只能通过 dmesg 在事后查看，不会实时出现在串口上。

    知识点3：启动日志与源码精确对照 [I] ~800字¶

    实践案例：追踪 "Freeing unused kernel memory"¶

    这是启动链中最关键的日志之一，标志着内核初始化阶段的正式结束。

    /* arch/arm64/mm/init.c (以 ARM64 为例) */
void free_initmem(void)
{
    /* (1) 标记 __init 区域为可释放 */
    mark_rodata_ro();           // 先锁定只读数据

    /* (2) 释放 .init.text 段 */
    free_reserved_area(lm_alias(__init_begin),
                       lm_alias(__init_end),
                       -1, "unused kernel");

    /* (3) 打印那行熟悉的日志 */
    /* "Freeing unused kernel memory: XK (virt_start - virt_end)" */

    /* (4) 从这一刻起，__init 函数不可再调用 */
}

    对应的链接脚本中：

    /* arch/arm64/kernel/vmlinux.lds.S */
.init.text : {
    _sinittext = .;
    INIT_TEXT
    _einittext = .;
}
.init.data : {
    INIT_DATA
    INIT_SETUP(16)
    INIT_CALLS
    CON_INITCALL
    SECURITY_INITCALL
}

    free_initmem() 将 .init.text 和 .init.data 所在的物理页返回给 buddy allocator。这是一个重要的内存优化——嵌入式系统通常只有 64MB-256MB 内存，初始化代码可能占用数百 KB 到数 MB。

    Trade-off：init 启动方式对比¶

    init 方案 二进制大小 启动速度 依赖管理 适用嵌入式场景 典型代表
    BusyBox init ~200KB 极快 无 超小型系统（<8MB rootfs） 路由器、传感器
    SysVinit ~100KB + 脚本 中等 手动 传统嵌入式 工控设备
    systemd ~20MB 慢（并行优化后中等） 完整 高端嵌入式/IoT 汽车信息娱乐、边缘网关
    runit/s6 ~100KB 快 轻量 容器化嵌入式 Docker IoT 设备
    自定义 C init 任意 可控 自行实现 极致精简 安全芯片、MCU 级 Linux

    本节总结¶

    rest_init() 是 start_kernel() 与运行中系统的分水岭。它完成了三项关键使命：

    1. 创建 PID=1 的 kernel_init —— 这个内核线程将蜕变为我们熟知的 init 进程，是所有用户进程的祖先
    2. 创建 PID=2 的 kthreadd —— 内核线程的"守护工厂"，确保线程创建始终在安全的进程上下文中进行
    3. 将自身转化为 idle 进程 —— PID=0 成为每个 CPU 的"最后退路"

    kernel_init 的 do_execve() 调用是整个 Linux 启动流程的分水岭：在此之前，所有代码运行在内核态，没有用户空间概念；在此之后，系统进入多进程的用户空间世界。理解这一转换的精确机制，是定位 "No working init found" 类启动故障的核心能力。

    当你下次看到启动日志中那行 Run /sbin/init as init process 时，你看到的不再是一行普通的信息，而是一个内核线程通过 do_execve() 完成自我替换的瞬间。
