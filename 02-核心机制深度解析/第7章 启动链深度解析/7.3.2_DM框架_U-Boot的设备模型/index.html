    7.3.2 DM框架：U-Boot的设备模型¶

    所属：第7章 嵌入式Bootloader深度解析 > 7.3 U-Boot驱动与设备模型 难度：[I→E] | 预计阅读时间：40分钟

    本节导读¶

    U-Boot从2014.10版本引入Driver Model（DM）框架，彻底改变了"每个board目录一堆硬编码驱动指针"的混沌局面。本节回答一个核心问题：U-Boot如何在比Linux内核更严苛的资源约束下，实现驱动的统一管理、设备树的自动绑定、以及平台数据与私有数据的清晰分离？ 理解DM框架，是移植新板卡、调试驱动probe失败、以及实现Driver Sandbox单元测试的必经之路。

    知识点1：为什么需要DM —— 从"散弹枪式"驱动到统一模型 [I] ~900字¶

    问题场景¶

    你接手了一个基于AM335x的老项目，发现board/ti/am335x/evm.c中直接硬编码了如下驱动引用链：

    /* 老版本U-Boot（DM之前）的典型代码 */
extern struct mmc *find_mmc_device(int dev_num);
extern int eth_initialize(bd_t *bis);
extern int spi_init(void);

int board_init(void)
{
    /* 手动初始化每个外设，顺序写死 */
    mmc_initialize(gd->bd);
    eth_initialize(gd->bd);
    spi_init();
    /* 新加一个I2C传感器？在这里再插一行 */
    i2c_init(CONFIG_SYS_I2C_SPEED, CONFIG_SYS_I2C_SLAVE);
}

    这种模式的缺陷在规模化后暴露无遗：

    问题维度 DM前（pre-DM） DM后
    驱动注册 全局链表手动挂接，*_register()散落在各处 U_BOOT_DRIVER()宏自动收集到.u_boot_driver_xxx段
    设备发现 board文件中硬编码struct xxx_dev全局变量 设备树节点自动实例化为struct udevice
    资源管理 时钟、GPIO、pinmux在驱动内直接操作寄存器 通过clk_*、gpio_*、pinctrl_*等DM API统一申请/释放
    代码复用 同IP不同板卡复制粘贴整份驱动 通用驱动 + 设备树compatible绑定，零代码复制
    单元测试 依赖真实硬件，无法mock Driver Sandbox模式可在x86 Linux主机上运行DM驱动

    机制深入：DM的核心设计目标¶

    DM框架借鉴了Linux Kernel Driver Model的思想，但做了显著的简化以适应Bootloader场景：

    1. 没有struct bus_type：U-Boot不需要热插拔、不需要UEvent通知，因此DM省略了Linux中复杂的bus/subsystem层级
    2. 没有module_init依赖解析：所有驱动在编译期通过链接器脚本收集到专用段，启动时一次性遍历
    3. platdata / privdata 强制分离：这是DM最具Bootloader特色的设计
    /* include/dm/device.h — struct udevice 的核心定义 */
struct udevice {
    const struct driver *driver;    /* 指向驱动操作函数表 */
    const char *name;
    void *plat_;                    /* 【平台数据】设备树解析的硬件参数 */
    void *priv_;                    /* 【私有数据】驱动运行时的状态缓存 */
    struct uclass *uclass;          /* 所属设备类 */
    u32 seq_;                       /* 同类设备中的序号（如 mmc0, mmc1） */
    ofnode node_;                   /* 对应的设备树节点 */
    /* ... 其他成员省略 ... */
};

    platdata vs privdata 的语义边界是DM设计的精髓：

    属性 platdata（平台数据） privdata（私有数据）
    来源 设备树自动解析 或 静态platdata结构 驱动probe()中动态分配
    生命周期 与udevice同生共死 probe()分配 → remove()释放
    内容 寄存器基地址、时钟频率、GPIO号等硬件描述 控制器句柄、DMA缓冲、运行时状态等软件上下文
    是否可跨驱动访问 理论上可读（不建议） 完全私有
    示例（MMC） mmc_platdata：bus_width、f_max、capability struct mmc_priv：host控制器指针、current_voltage

    💡 技巧：在驱动代码中始终使用dev_get_plat()和dev_get_priv()宏访问，不要直接操作udevice->plat_——这些宏会在编译开启CONFIG_DEBUG_DEVRES时附加空指针检查。

    关键代码路径：驱动的自注册机制¶

    /* drivers/mmc/mmc-uclass.c — MMC uclass 定义 */
U_BOOT_DRIVER(mmc) = {
    .name   = "mmc",
    .id     = UCLASS_MMC,           /* 所属设备类 */
    .ops    = &mmc_ops,             /* uclass 级别通用操作 */
};

/* drivers/mmc/sdhci.c — SDHCI 驱动定义 */
U_BOOT_DRIVER(sdhci) = {
    .name           = "sdhci",
    .id             = UCLASS_MMC,
    .of_match       = sdhci_ids,    /* compatible 匹配表 */
    .ops            = &sdhci_ops,
    .probe          = sdhci_probe,  /* 生命周期函数 */
    .remove         = sdhci_remove,
    .priv_auto      = sizeof(struct sdhci_host),  /* 自动分配priv空间 */
    .plat_auto      = sizeof(struct mmc_platdata), /* 自动分配plat空间 */
};

    U_BOOT_DRIVER()宏将驱动结构体放入.u_boot_driver_*链接段，由drivers/core/lists.c中的lists_drivers_lookup()在启动时遍历匹配。

    常见陷阱¶

    ⚠️ 陷阱1：priv_auto和plat_auto的大小计算错误——如果sizeof(struct sdhci_host)变了但忘记同步更新宏，会导致静默的内存越界。建议用static_assert(sizeof(...) == EXPECTED_SIZE)做编译期检查。

    ⚠️ 陷阱2：在probe()之前访问dev_get_priv(dev)——此时priv尚未分配，返回野指针。安全做法是先检查dev->flags & DM_FLAG_ACTIVATED。

    知识点2：DM三要素 —— uclass、driver、device 的协作 [E] ~1200字¶

    问题场景¶

    你在移植一款新SoC的eMMC控制器，发现已有drivers/mmc/sdhci.c通用驱动可用。你需要做哪些工作才能让它在新板卡上跑起来？DM三要素的交互关系决定了答案。

    机制深入：uclass → driver → device 的三角关系¶

    要素1：uclass（设备类）— "接口契约"¶

    uclass定义了一类设备的公共行为接口。类比面向对象：uclass是抽象基类，定义了这类设备"能做什么"。

    /* include/dm/uclass-id.h — 预定义的设备类ID */
enum uclass_id {
    UCLASS_ROOT = 0,
    UCLASS_MMC,         /* 块存储：eMMC/SD/SDIO */
    UCLASS_ETH,         /* 以太网 */
    UCLASS_SPI,         /* SPI主控 */
    UCLASS_I2C,         /* I2C总线 */
    UCLASS_PINCTRL,     /* 引脚控制器 */
    UCLASS_CLK,         /* 时钟 */
    UCLASS_BLOCK,       /* 块设备抽象 */
    /* ... 共80+个uclass ... */
};

    每个uclass有对应的uclass_driver，提供该类设备的默认操作和初始化逻辑：

    /* drivers/mmc/mmc-uclass.c — MMC uclass_driver */
UCLASS_DRIVER(mmc) = {
    .id             = UCLASS_MMC,
    .name           = "mmc",
    .flags          = DM_UC_FLAG_SEQ_ALIAS,  /* 支持别名编号 */
    .ops            = &mmc_ops,              /* 类级别通用操作 */
    .init           = mmc_uclass_init,       /* uclass首次使用时 */
    .post_bind      = mmc_post_bind,         /* 设备绑定到uclass后 */
};
    常见uclass 职责 典型驱动 关键API
    UCLASS_MMC SD/MMC/eMMC协议层 sdhci, mtk-sd, omap_hsmmc mmc_init(), mmc_read_blocks()
    UCLASS_ETH 网络MAC/PHY抽象 eqos, fec_mxc, gmac_rockchip eth_init(), eth_send(), eth_recv()
    UCLASS_SPI SPI主控制器 cadence_spi, mtk_spi spi_claim_bus(), spi_xfer()
    UCLASS_I2C I2C总线控制器 i2c_cdns, omap_i2c i2c_read(), i2c_write()
    UCLASS_PINCTRL 引脚复用/配置 pinctrl-single, rockchip_pinctrl pinctrl_select_state()
    UCLASS_CLK 时钟树管理 clk_fixed_rate, ti_sci_clk clk_get_rate(), clk_enable()
    UCLASS_BLOCK 块设备统一层（FAT/EXT4依赖） —（纯uclass层） blk_dread(), blk_dwrite()

    要素2：driver（驱动）— "实现细节"¶

    driver是具体IP的控制逻辑。一个driver通过.id字段声明自己属于哪个uclass，通过of_match声明能兼容哪些设备树compatible字符串。

    关键生命周期回调：

    回调函数 调用时机 典型工作 失败后果
    bind() 设备实例创建时 分配platdata/privdata空间，注册子设备 设备无法实例化
    probe() 设备首次被使用时 映射寄存器、申请时钟/复位、初始化硬件 设备不可用，返回-ENODEV
    remove() 设备被显式卸载（热移除/测试） 释放资源、关闭时钟、unmap寄存器 资源泄漏
    unbind() 设备实例销毁时 释放platdata/privdata 内存泄漏 
    /* drivers/mmc/rockchip_dw_mmc.c — probe函数典型实现 */
static int rockchip_dw_mmc_probe(struct udevice *dev)
{
    struct mmc_platdata *plat = dev_get_plat(dev);
    struct dwmci_host *host = dev_get_priv(dev);
    struct clk clk;
    int ret;

    /* 步骤1：从设备树解析platdata（DM自动完成，这里可补充） */
    /* 步骤2：获取并使能时钟 — 这是DM资源管理的核心优势 */
    ret = clk_get_by_index(dev, 0, &clk);
    if (ret)
        return ret;
    ret = clk_enable(&clk);
    if (ret)
        return ret;

    /* 步骤3：获取reset线 */
    ret = reset_get_by_index(dev, 0, &host->reset_ctl);
    if (!ret)
        reset_deassert(&host->reset_ctl);

    /* 步骤4：映射寄存器（通过设备树reg属性自动解析） */
    host->ioaddr = dev_read_addr_ptr(dev);

    /* 步骤5：调用通用驱动的probe完成后续初始化 */
    ret = dwmci_probe(dev);

    /* 保存时钟句柄到priv供remove时使用 */
    host->clk = clk;
    return ret;
}

    要素3：udevice（设备实例）— "硬件化身"¶

    每个udevice代表一个实际存在的硬件实例。它诞生于U-Boot启动时的设备树扫描：

    /* drivers/core/root.c — 启动阶段的核心初始化路径 */
int dm_init_and_scan(bool pre_reloc)
{
    /* 1. 初始化DM根设备（uclass_root） */
    dm_init(CONFIG_IS_ENABLED(OF_LIVE));

    /* 2. 扫描设备树，为每个compatible匹配的节点创建udevice */
    dm_scan_plat(pre_reloc);   /* 扫描静态platdata */
    dm_extended_scan(pre_reloc); /* 扫描设备树 */

    /* 3. 按需probe核心设备（pre_reloc阶段只probe标记了
     *    UBOOT_DM_PRELOAD的驱动） */
    return 0;
}

    Trade-off 表格：显式probe vs 延迟probe vs 自动probe¶

    策略 实现方式 启动时延影响 内存占用 适用场景
    显式probe 代码中手动调用device_probe(dev) 可控，按需触发 最低，只用才分配 启动路径严格控制的嵌入式产品
    延迟probe 依赖第一次uclass_get_device()时自动probe 首次访问有延迟，后续无感 按需分配 大多数U-Boot驱动的默认行为
    启动期全probe dm_init_and_scan()中遍历所有设备 启动变慢，所有驱动初始化耗时累加 最高，所有设备同时驻留 Driver Sandbox测试、调试阶段
    pre-reloc probe 驱动标记DM_FLAG_PRE_RELOC SPL阶段就初始化，U-Boot proper可直接用 SPL内存承压 Falcon模式、需要SPL阶段访问的设备

    🔴 安全提醒：在SPL阶段使用DM需要特别小心——CONFIG_SPL_DM开启后，SPL的DM框架是精简版，部分uclass操作可能未实现。务必查阅drivers/core/Kconfig中标记SPL的依赖项。

    常见陷阱¶

    ⚠️ 陷阱1：probe()中的资源依赖死锁——驱动A的probe()调用device_probe()去激活驱动B，而驱动B的probe()又依赖驱动A。DM框架不处理循环依赖，会触发ERROR: nested probe call后挂死。

    ⚠️ 陷阱2：忘记在remove()中释放clk_disable()和reset_assert()——这在支持Driver Sandbox的测试中会被dm_leak_check检测到，但在真实硬件上可能导致复位后外设状态混乱。

    知识点3：DM与设备树的绑定 —— compatible到probe的完整链路 [E] ~1200字¶

    问题场景¶

    你移植RK3568板卡时，在设备树中添加了如下eMMC节点，但U-Boot启动后mmc info返回"No MMC device found"。驱动到底绑上了没有？probe走了没有？如何排查？

    机制深入：从dtb节点到活跃udevice的五步链路¶

    设备树节点
    ↓ (1) dm_extended_scan_dt() 遍历/firmware/下所有节点
    ↓ (2) lists_bind_fdt() 匹配 compatible 与 of_match 表
    ↓ (3) device_bind_with_driver_data() 创建 udevice 实例
    ↓ (4) uclass_bind_device() 将 udevice 挂接到对应 uclass
    ↓ (5) device_probe() 调用 driver->probe() 完成硬件初始化
活跃udevice（可使用）

    Step 1-2：compatible 字符串匹配机制¶

    匹配的核心在drivers/core/lists.c：

    /* drivers/core/lists.c — 驱动查找的核心函数 */
const struct driver *lists_driver_lookup_name(const char *name)
{
    /* 遍历 __u_boot_driver_start 到 __u_boot_driver_end */
    struct driver *drv = ll_entry_start(struct driver, u_boot_driver);
    const int n_ents = ll_entry_count(struct driver, u_boot_driver);

    for (entry = drv; entry != drv + n_ents; entry++) {
        if (!strcmp(name, entry->name))
            return entry;
    }
    return NULL;
}

    设备树绑定使用struct udevice_id数组定义匹配规则：

    /* drivers/mmc/rockchip_dw_mmc.c — of_match 表 */
static const struct udevice_id rockchip_dw_mmc_ids[] = {
    { .compatible = "rockchip,rk3568-dw-mshc" },
    { .compatible = "rockchip,rk3399-dw-mshc" },
    { .compatible = "rockchip,rk3288-dw-mshc" },
    { /* sentinel */ }
};

U_BOOT_DRIVER(rockchip_dw_mmc) = {
    .name           = "rockchip_dw_mmc",
    .id             = UCLASS_MMC,
    .of_match       = rockchip_dw_mmc_ids,    /* ← 匹配入口 */
    .probe          = rockchip_dw_mmc_probe,
    .priv_auto      = sizeof(struct dwmci_host),
    .plat_auto      = sizeof(struct mmc_platdata),
};

    Step 3-4：udevice 创建与 uclass 挂接¶

    device_bind_with_driver_data()完成内存分配和关系建立：

    /* drivers/core/device.c — 绑定核心逻辑 */
int device_bind_with_driver_data(struct udevice *parent,
                                 const struct driver *drv,
                                 const char *name, void *driver_data,
                                 ofnode node, struct udevice **devp)
{
    struct udevice *dev;
    int ret;

    /* 1. 分配udevice结构体 */
    dev = calloc(1, sizeof(struct udevice));

    /* 2. 初始化基本关系 */
    dev->driver = drv;
    dev->name = name;
    dev->node_ = node;

    /* 3. 自动分配platdata和privdata空间 */
    if (drv->plat_auto)
        dev->plat_ = calloc(1, drv->plat_auto);
    if (drv->priv_auto)
        dev->priv_ = calloc(1, drv->priv_auto);

    /* 4. 找到或创建对应uclass，将dev挂接上去 */
    ret = uclass_get(drv->id, &dev->uclass);
    uclass_bind_device(dev);

    /* 5. 调用driver的bind()回调（如果有） */
    if (drv->bind) {
        ret = drv->bind(dev);
        if (ret)
            goto fail_bind;
    }

    *devp = dev;
    return 0;
}

    Step 5：platdata 从设备树自动解析¶

    DM框架在device_probe()阶段调用dev_read_u32()系列函数，从设备树节点自动提取参数填充platdata：

    /* arch/arm/dts/rk3568-evb.dtsi — eMMC 设备树节点 */
&sdhci {
    compatible = "rockchip,rk3568-dw-mshc";
    reg = <0x0 0xfe310000 0x0 0x10000>;
    clocks = <&cru HCLK_EMMC>, <&cru CLK_EMMC>,
             <&cru SCLK_EMMC_DRV>, <&cru SCLK_EMMC_SAMPLE>;
    clock-names = "biu", "ciu", "ciu_drv", "ciu_sample";
    resets = <&cru SRST_EMMC>;
    bus-width = <8>;              /* ← 自动解析为 plat->bus_width */
    max-frequency = <150000000>;  /* ← 自动解析为 plat->f_max */
    cap-mmc-highspeed;
    mmc-hs200-1_8v;
    non-removable;
};

    /* drivers/mmc/sdhci.c — probe中从设备树读取platdata */
static int sdhci_probe(struct udevice *dev)
{
    struct mmc_platdata *plat = dev_get_plat(dev);

    /* 从设备树自动解析关键参数 */
    plat->f_max = dev_read_u32_default(dev, "max-frequency", 0);
    plat->bus_width = dev_read_u32_default(dev, "bus-width", 1);
    plat->removable = !dev_read_bool(dev, "non-removable");

    /* capability标志位通过设备树boolean属性设置 */
    if (dev_read_bool(dev, "cap-mmc-highspeed"))
        plat->caps |= MMC_CAP(MMC_CAP_HS);
    if (dev_read_bool(dev, "mmc-hs200-1_8v"))
        plat->caps |= MMC_CAP(MMC_CAP_HS200);

    return 0;
}

    实践案例：RK3568 eMMC probe失败的排查实录¶

    现象：板卡U-Boot启动后，mmc list命令输出为空，mmc info报"No MMC device found"。

    排查过程：

    1. 确认设备树节点存在：检查.config中CONFIG_DEFAULT_DEVICE_TREE="rk3568-evb"正确，dtc编译无报错。

    2. 开启DM调试：在U-Boot命令行设置log level 7（或编译时开启CONFIG_LOG=y、CONFIG_LOG_MAX_LEVEL=7），重启看到：

    DM: device 'mmc@fe310000' has no driver
    1. 检查compatible字符串：对比设备树和驱动的匹配表——
    /* 设备树中写的（错误） */
compatible = "rockchip,rk3568-dwmmc";     /* 少了 "-mshc" */

    /* 驱动中注册的（正确） */
{ .compatible = "rockchip,rk3568-dw-mshc" },   /* 期望 "-mshc" */

    🔴 根因：dwmmc vs dw-mshc的不一致。这是Rockchip平台历史遗留问题——旧版内核使用dwmmc，主线内核统一为dw-mshc。直接复制旧版内核设备树到U-Boot会导致compatible匹配失败。

    1. 修复：修改设备树（推荐，保持与主线一致）：
    compatible = "rockchip,rk3568-dw-mshc";

    或者修改驱动（如果你维护的是闭源老分支）：

    { .compatible = "rockchip,rk3568-dw-mshc" },
{ .compatible = "rockchip,rk3568-dwmmc" },  /* 兼容旧命名 */
    1. 验证：修复后重启，log level 7输出变为：
    DM: probe device mmc@fe310000 (rockchip_dw_mmc)
DM: probe ok
MMC:   mmc@fe310000: 1

    💡 技巧：U-Boot命令行提供dm tree和dm uclass两个强大的诊断命令：

    => dm tree                    # 查看完整的设备树绑定结果
=> dm uclass mmc              # 查看MMC uclass下的所有设备
=> dm dev mmc@fe310000        # 查看特定设备的详细信息

    Trade-off 表格：设备树platdata vs 静态platdata¶

    方案 设备树platdata（推荐） 静态platdata（遗留/特殊场景）
    数据来源 .dts节点属性 C代码中硬编码的struct platdata数组
    可维护性 高，硬件参数与代码分离 低，改参数需重编代码
    多板卡支持 同一驱动 + 不同dts即可 每块板卡可能需#ifdef或独立文件
    启动速度 需解析dtb字符串，微慢 直接内存访问，微快
    SPL适用性 需CONFIG_SPL_OF_CONTROL=y 无需设备树支持，SRAM友好
    调试难度 需检查dtb编译输出 直接gdb跟踪C结构体

    💡 技巧：在SPL阶段，如果SRAM不足以容纳展开的设备树blob，可以使用CONFIG_SPL_OF_PLATDATA——在编译期将设备树节点转换为静态C结构体，兼得设备树的声明式描述和静态数据的零解析开销。

    常见陷阱¶

    ⚠️ 陷阱1：dev_read_u32()在属性不存在时不报错，返回-EINVAL但很多人忽略检查。安全写法：

    u32 freq;
int ret = dev_read_u32(dev, "max-frequency", &freq);
if (ret) {
    log_warning("max-frequency missing, using default\n");
    freq = 52000000;  /* 安全默认值 */
}

    ⚠️ 陷阱2：设备树中的status = "disabled"节点会被DM跳过（不会创建udevice）。如果父节点disabled，所有子节点也不会被扫描。

    ⚠️ 陷阱3：of_match表的最后一项必须是空sentinel{ }，否则lists_fdt_lookup_driver()会越界遍历，导致随机匹配到内存垃圾或触发data abort。

    本节总结¶

    U-Boot的DM框架用约1/10的Linux Driver Model代码量，实现了驱动统一管理的完整能力。核心记住三点：

    1. platdata/privdata分离是DM的设计精髓——plat来自设备树描述硬件，priv来自probe分配描述软件状态
    2. uclass-driver-device三层结构实现了"接口与实现分离"——同一个uclass操作可以透明地对不同driver生效
    3. compatible字符串匹配是dtb到驱动的桥梁，拼写不一致是移植新板卡时最高频的probe失败原因

    掌握dm tree、dm uclass、log level 7这三板斧，能覆盖90%的DM驱动绑定问题。
