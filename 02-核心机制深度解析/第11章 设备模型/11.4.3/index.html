    11.4.3 udev规则与实战¶

    所属章节：第11章 设备驱动进阶话题 > 11.4 驱动与设备模型 难度：[I] | 预计阅读时间：15分钟

    本节导读¶

    本节把udev规则从"黑魔法"拉回到"手写能用的配置"。学完后，你能看懂/etc/udev/rules.d/目录下的规则文件为什么长那样，能自己写两条实用规则，还能在规则不生效时熟练使用调试命令定位问题——而不是对着/dev下乱七八糟的设备名干瞪眼。

    知识点158：udev规则语法——像写"如果…就…"的物业管理条例 [I] ~800字¶

    规则文件放在哪？两个目录要分清¶

    udev规则文件散落在两个地方，跟/etc/profile.d/的脚本逻辑一样——一个管系统默认，一个管用户自定义：

    目录 优先级 用途
    /lib/udev/rules.d/ 低 发行版或根文件系统自带的默认规则，软件包升级时可能被覆盖
    /etc/udev/rules.d/ 高 管理员手写的自定义规则，数字前缀小的先执行，覆盖默认行为

    规则文件名必须以.rules结尾，否则udev直接无视。文件名格式一般是数字-描述.rules，比如50-udev-default.rules、99-my-custom.rules。udevd按数字前缀升序读取，后读到的规则可以覆盖先读到的同名键值。所以自定义规则通常用较大的数字（比如99开头），确保它能"压过"系统默认。

    一条规则 = 匹配条件 + 执行动作¶

    打开任意一个.rules文件，你会看到一行行长得像这样的文本：

    SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", SYMLINK+="myusb%d", MODE="0666"

    别看它长，拆开就是两部分，跟小区物业的条例一个思路——前半句定条件，后半句定动作：

    • 匹配条件（逗号分隔）：SUBSYSTEM=="tty" 且 ATTRS{idVendor}=="1a86" 且 …
    • 执行动作：SYMLINK+="myusb%d"，MODE="0666"

    致命细节：== 和 = 完全不一样¶

    这地方坑过无数人。udev规则的语法偷学了C语言的逻辑，但含义略有不同：

    符号 含义 出现位置
    == 匹配比较，"是不是等于" 只能用于匹配条件（左半部分）
    = 赋值，"设置成什么" 只能用于执行动作（右半部分）
    != 不匹配 匹配条件
    += 追加赋值，在原有值后面追加 执行动作（如SYMLINK+=是加一条软链，不是覆盖）

    如果你把SUBSYSTEM=="tty"写成SUBSYSTEM="tty"，这条规则就变味了——它不再"筛选tty子系统的设备"，而是试图"把子系统改成tty"，udev会直接报错或者行为诡异。反过来，SYMLINK="xxx"是覆盖所有软链，SYMLINK+="xxx"才是追加一条，这个区别在需要多别名时很关键。

    常见匹配键与动作键速查¶

    匹配键——用来"挑设备"：

    键名 匹配目标 示例
    KERNEL 内核设备名 KERNEL=="sd*" 匹配所有SCSI磁盘
    SUBSYSTEM 子系统名 SUBSYSTEM=="usb"
    ATTR{file} sysfs属性值 ATTR{removable}=="1"
    ATTRS{file} 沿设备链向上查找属性 ATTRS{idVendor}=="1a86" 常用于匹配USB设备
    ENV{key} 环境变量 ENV{DEVTYPE}=="partition"
    ACTION 事件类型 ACTION=="add" 只在插入时触发

    动作键——用来"改行为"：

    键名 作用 示例
    SYMLINK+= 创建符号链接 SYMLINK+="myusb%n"
    MODE= 设置设备节点权限 MODE="0666"
    OWNER= / GROUP= 设置属主/属组 OWNER="root"
    NAME= 强制指定节点名（少用） NAME="mydev"
    RUN+= 执行外部脚本/程序 RUN+="/usr/bin/my-script.sh"

    两条实用规则示例¶

    示例1：给USB转串口芯片固定软链名

    CH340是嵌入式开发最常见的USB转串口芯片，每次插拔可能变成/dev/ttyUSB0或ttyUSB1。下面这条规则让它固定出现在/dev/ch340：

    # 代码1：USB转串口固定符号链接规则
# 文件：/etc/udev/rules.d/99-ch340.rules
SUBSYSTEM=="tty", ATTRS{idVendor}=="1a86", ATTRS{idProduct}=="7523", \
    SYMLINK+="ch340", MODE="0666"

    MODE="0666"让普通用户也能读写，省得每次都要sudo。

    示例2：给自定义按键设备加组权限

    假设你写了一个GPIO按键驱动，注册后设备节点在/dev/my_button。想让gpio用户组能访问：

    # 代码2：自定义设备权限规则
# 文件：/etc/udev/rules.d/50-mybutton.rules
SUBSYSTEM=="input", ATTR{name}=="my_gpio_button", \
    MODE="0660", GROUP="gpio"

    写好规则后，必须执行 udevadm control --reload-rules 才能让udevd重新加载，不然新规则只是躺在硬盘上的一行文本。

    知识点159：udev调试三件套——规则不生效时别猜，直接查 [I] ~600字¶

    调试就像修水管：先看、再查、最后试¶

    udev规则写完了却不生效，是最让人抓狂的场景。/dev下面设备名乱变、权限不对、软链没出来……别急着重启，三个命令就能定位大部分问题。

    1. udevadm monitor：实时"监听喇叭"

    这个命令让你亲眼看到内核和用户空间之间在聊什么。开两个终端，一个跑udevadm monitor --kernel --udev --property，另一个插拔设备。你会看到类似这样的输出：

    # 代码3：udevadm monitor实时监控
$ udevadm monitor --kernel --udev --property
KERNEL[1234.567] add      /devices/pci0000:00/0000:00:14.0/usb1/1-2 (usb)
ACTION=add
DEVPATH=/devices/pci0000:00/0000:00:14.0/usb1/1-2
SUBSYSTEM=usb
...
UDEV [1234.589] add      /devices/.../1-2 (usb)

    --kernel看内核发出的原始uevent，--udev看udevd处理后的结果。如果插拔时什么都没打印，说明内核没发出uevent（驱动或硬件问题）；如果有KERNEL但没有UDEV行，说明udevd没在处理（守护进程挂了或规则过滤掉了）。

    2. udevadm info /sys/...：查设备的"户口本信息"

    想知道一个设备有哪些属性可以用来写匹配条件？用udevadm info --attribute-walk --path=/sys/...：

    # 代码4：查看设备完整属性链
$ udevadm info --attribute-walk --path=/sys/class/tty/ttyUSB0

  looking at device '/devices/.../usb1/1-2/1-2:1.0/ttyUSB0':
    KERNEL=="ttyUSB0"
    SUBSYSTEM=="tty"
    ...
  looking at parent device '/devices/.../usb1/1-2':
    KERNELS=="1-2"
    SUBSYSTEMS=="usb"
    ATTRS{idVendor}=="1a86"
    ATTRS{idProduct}=="7523"
    ...

    注意ATTR和ATTRS的区别：ATTR只看当前设备节点本身的sysfs属性，ATTRS会沿着设备链往上爬，找到USB控制器层面的idVendor。写USB设备规则时几乎都得用ATTRS。

    3. udevadm test：模拟执行，看规则怎么跑

    这是最重要的调试命令。不用真的插拔硬件，直接让udevd"假装"某个设备刚插进来，打印它会执行哪些规则、设置哪些属性：

    # 代码5：模拟udevd处理某个设备
$ udevadm test /sys/class/tty/ttyUSB0 2>&1 | less

    输出里会明确显示：匹配到了哪条规则、设置了什么SYMLINK、最终MODE是多少。如果规则没匹配上，检查是不是idVendor写错了，或者用了=而不是==。

    新手最常踩的三个坑¶

    错误现象 根因 解决办法
    规则写了但完全没效果 文件名不是.rules结尾 重命名文件，确保后缀正确
    改完规则依然旧行为 忘记reload 执行udevadm control --reload-rules，然后重新插拔设备
    udevadm test提示语法错误 ==和=写反了 条件用==，赋值用=或+=

    🔴 危险：udevadm control --reload-rules只是让udevd重新读配置文件，不会重新应用已有设备。想让已插入的设备走新规则，必须物理拔插一次，或者用udevadm trigger --action=add /sys/...手动触发。

    本节总结¶

    概念 核心要点 自查操作
    规则文件位置 /etc/udev/rules.d/自定义优先，/lib/udev/rules.d/系统默认 ls /etc/udev/rules.d/
    执行顺序 按文件名数字前缀升序，后覆盖前 用udevadm test验证哪条规则生效
    == vs = ==是匹配条件，=是赋值动作 写规则时默念"左匹配右赋值"
    调试三板斧 monitor看实时流、info查属性、test模拟执行 插拔设备时开三个终端对比
    规则不生效 检查后缀名→reload→trigger，三步走 udevadm control --reload-rules

    你已经学会了udev规则的语法、能写实用规则，也掌握了调试三板斧。下一节（11.4.4）我们将把目光移回内核空间，聊聊devtmpfs与mdev——在极简嵌入式系统里，有时候你根本不想跑完整的udevd，这时候更轻量的方案就该出场了。
